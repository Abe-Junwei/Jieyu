## 结论摘要

当前“utterance 说话人链路”整体是通的：创建、重命名、合并、删除、批量指派的本地状态与持久化基本一致。

主要问题集中在“independent_boundary 语段层”与“说话人 UI/操作入口”之间仍存在 utterance 假设，导致 segment 级说话人信息出现三类断裂：

1. 语段已有 `segment.speakerId`，但时间轴标签不显示。
2. 独立语段层右键菜单中的说话人指派，实际可能改到 owner utterance，或直接无效。
3. 侧栏筛选、聚焦、说话人分组只看 utterance，segment-only 说话人不会进入 UI。

## 发现

### P1：独立语段层右键菜单的说话人指派路由错误，可能改错对象或直接丢失

**影响**

- 在独立边界层上右键语段后执行“指派说话人 / 清空说话人 / 新建说话人并指派”，用户预期修改的是 `layer_segments.speakerId`。
- 当前实现会把 segment id 先映射回 owner utterance id，再调用 utterance 路径。
- 结果有两种错误表现：
  - 若 segment 绑定了 `utteranceId`，实际改的是 utterance，说话人变化会串到继承 utterance 的其它层。
  - 若 segment 没有 `utteranceId`，id 会在映射时被丢弃，操作提示可能出现，但 segment 实际未修改。

**证据链**

- `useTimelineAnnotationHelpers` 在独立层右键时，仍把当前单元 id 放进 `ctxMenu.utteranceId`，这里传入的其实是 segment id。
- `TranscriptionOverlays` 再把该 id 组装为 `targetIds`，并在“说话人管理”子菜单里直接回调上层。
- `TranscriptionPage.Orchestrator` 中 `onAssignSpeakerFromMenu` / `onCreateSpeakerAndAssignFromMenu` 一律调用 `resolveSpeakerActionUtteranceIds(...)`，最终落到 `handleAssignSpeakerToUtterances(...)` 与 `handleCreateSpeakerAndAssignToUtterances(...)`，没有 segment 分支。

**关键位置**

- `src/hooks/useTimelineAnnotationHelpers.tsx:201`
- `src/components/TranscriptionOverlays.tsx:105`
- `src/pages/TranscriptionPage.Orchestrator.tsx:4252-4256`

**建议**

- 右键菜单回调应显式携带 `unit kind`（utterance / segment）或 `layer.constraint`。
- `TranscriptionOverlays` 不应再把所有说话人操作抽象成 `utteranceIds`；应升级为 `unitIds + kind`，或拆成 utterance/segment 两套路由。

### P1：独立语段层的说话人标签没有真正渲染到时间轴单元

**影响**

- `segment.speakerId` 已经可以落库，也能参与重叠轨道布局与聚焦判断，但时间轴 annotation 本身没有对应 badge，用户会看到“轨道布局按说话人分开了”，却看不到语段标签本身是谁。
- 这正好对应你要求重点核对的“说话人标签在语段错误的地方”。当前不是文本错误，而是独立语段层直接不显示。

**证据链**

- `useTimelineAnnotationHelpers` 渲染 annotation 时只从 `speakerVisualByUtteranceId[utt.id]` 取 badge 数据；该 map 是 utterance 维度的。
- `TranscriptionTimelineMediaLanes` 在独立层里把 segment 伪装成 `utt` 传给 `renderAnnotationItem(...)`，但没有额外提供 segment 级 speaker visual。
- `TranscriptionTimelineTextOnly` 更直接，独立层分支把 `speakerVisual` 硬编码为 `undefined`。

**关键位置**

- `src/hooks/useTimelineAnnotationHelpers.tsx:244`
- `src/components/TranscriptionTimelineMediaLanes.tsx:555`
- `src/components/TranscriptionTimelineTextOnly.tsx:359`

**建议**

- 把 speaker visual 的输入从“仅 utterance map”提升为“按 unit id + kind 解析”。
- 独立层应根据 `segment.speakerId` 或 owner utterance 回退值，生成与 utterance 同构的 visual 数据，再传入 `TimelineAnnotationItem`。

### P2：说话人侧栏分组、筛选、聚焦只基于 utterance，segment-only 说话人进入不了 UI

**影响**

- 侧栏的说话人 chip、说话人组、聚焦候选，以及部分摘要，数据源都还是 `utterancesOnCurrentMedia`。
- 因此只存在于独立语段层、未同步到 utterance 的说话人，不会出现在侧栏管理区。
- 用户会看到明显不一致：
  - 批量指派面板在 segment 选择时会显示“说话人轨道编辑”。
  - 但侧栏的说话人列表、筛选和聚焦仍然像“没有这个说话人”。

**证据链**

- `useSpeakerDerivedState` 的 `speakerFilterOptions` 来自 `buildSpeakerFilterOptions(utterancesOnCurrentMedia, speakerVisualByUtteranceId)`。
- `TranscriptionPage.Orchestrator` 只为 `selectedSpeakerSummary` 做了 segment 兼容；传给侧栏的 `speakerFilterOptions` 仍然是 utterance 版本。
- `speakerFocusOptions` 也只从 `utterancesOnCurrentMedia` 收集 speaker key，因此 segment-only speaker 无法被聚焦模式选择或保持。

**关键位置**

- `src/hooks/speakerManagement/useSpeakerDerivedState.ts:24`
- `src/pages/TranscriptionPage.Orchestrator.tsx:3902-3903`
- `src/pages/TranscriptionPage.Orchestrator.tsx:3238-3239`

**建议**

- 将 speaker 管理视图的数据源抽象为“当前媒体上的可见说话人单元集合”，而不是固定 `utterancesOnCurrentMedia`。
- 在独立层激活或 segment 选择上下文下，侧栏和聚焦候选应纳入 segment 级 speaker 统计。

## 已核对通过的链路

- utterance 路径下的创建、重命名、合并、删除、批量指派逻辑基本一致。
- `LinguisticService.assignSpeakerToSegments(...)` 已经存在，说明后端/存储层并不缺 segment 能力，主要问题发生在 UI 路由与展示层。
- 批量指派面板主按钮已经有 routed 分支：当 `selectedSegmentIdsForSpeakerActions.length > 0` 时，会正确走 `assignSpeakerToSegments(...)`。问题主要集中在右键菜单和派生 UI，而不是面板主流程。

这说明当前问题属于“现有测试未覆盖到的 segment 说话人链路缺口”，而不是现成红测。

### 当前覆盖缺口

- 没有测试断言独立语段层 annotation 必须显示 speaker badge。
- 没有测试断言独立层右键菜单的说话人指派必须调用 `assignSpeakerToSegments(...)`。
- 没有测试断言侧栏 speaker chip / focus options 必须包含 segment-only speaker。

## 优先级建议

1. 先修右键菜单路由错误（P1）。这是数据写错对象的问题，风险最高。
2. 再补独立语段层 speaker badge 渲染（P1）。这是最直观的 UI 回归点。
3. 最后统一 speaker 派生数据源，补侧栏筛选/聚焦/分组（P2）。这样可以把 speaker UI 从“utterance 专用”收敛成真正的 unit-aware 设计。
