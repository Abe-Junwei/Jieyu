# 规划：VAD 平台抽象与流式推理落地方案（2026-04-11）

## 0 · 背景与问题

### 0.1 触发场景

导入 400MB 音频文件时页面崩溃（Chromium error code 5 — OOM）。根因链：

1. `VadMediaCacheService.ensureVadCacheForMedia()` 对 blob URL 执行 `fetch().arrayBuffer()`，分配 400MB ArrayBuffer。
2. `decodeAudioData()` 再分配数 GB 高采样率 PCM buffer。
3. 两步叠加超出浏览器 renderer 进程内存限制。

### 0.2 平台方向（本次确定）

> **桌面端为主，浏览器为辅。**

- 浏览器端：≤100MB 走全量 `decodeAudioData`；>100MB 前置门控跳过 VAD。
- 桌面端（Electron/Tauri TBD）：Praat LongSound 路线（`fs.read` 随机定位 + FFmpeg 管道分块解码 + SharedArrayBuffer 零拷贝 → Worker Thread Silero）。
- 不在浏览器端投入分块流式 VAD——等桌面端框架确定后直接走 Praat 路线。

### Phase 2（P1）：Worker 流式消息协议

**vadWorker.ts 新增消息**

| 方向 | 类型 | 说明 |
|---|---|---|
| In | `detect-stream-start` | `{ type, id, sampleRate }` — 开启流式会话，重置 h0/c0 |
| In | `detect-stream-chunk` | `{ type, id, pcm: Float32Array }` — 发送一块 PCM（自动重采样 → 16kHz、跨块帧残留拼接、逐帧 Silero 推理） |
| In | `detect-stream-end` | `{ type, id }` — 结束流式会话，处理剩余帧、frameProbsToSegments 后处理、返回 result |
| Out | `progress` | 每 16 帧报告一次（`totalFrames=0` 表示总帧数未知） |
| Out | `result` | 最终语音段列表 |

内部实现：
- 提取 `runSingleFrame()` 复用于全量/流式两条路径
- `streamProcessQueue: Promise<void>` 串行化 chunk 处理，保证帧状态连续性
- `StreamingSessionState` 管理 leftover buffer + 累积 frameProbs

**WhisperXVadService 新增 API**

```typescript
interface StreamingVadSession {
  sendChunk(pcm: Float32Array): void;  // transfer ownership
  finish(): Promise<SpeechSegment[]>;
  cancel(): void;
}

// 使用方式
const session = vadService.startStreamingDetection(sampleRate, { signal, onProgress });
session.sendChunk(chunk1);
session.sendChunk(chunk2);
const segments = await session.finish();
```

## 4 · 影响四项功能的 VAD 依赖现状

| 功能 | 依赖 | 当前状态 |
|---|---|---|
| 自动分段 | `ensureVadCacheForMedia` → `vadCache` | ≤100MB 正常；>100MB 回退到 `loadAudioBuffer` + 能量 VAD（同样受 OOM 限制） |
| 波形 VAD 叠加 + containsSpeech | `useEnsureVadCache` → waveform bridge | ≤100MB 正常；>100MB 无叠加数据 |
| 语音输入录音 | `WhisperXVadService` 直接使用 | 不受大文件影响（仅处理实时录音流） |
| AI 对话上下文 | 依赖 `vadCache` 是否有数据 | ≤100MB 正常；>100MB 无 VAD 上下文 |

P2 桌面端后端落地后，>100MB 功能将全部恢复。
