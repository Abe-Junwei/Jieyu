# 规划：语言-正字法-字体体系剩余全面落地方案

> 2026-03-31 · 基于当前仓库实况，仅覆盖尚未完成的剩余收口工作

---

## 一、目标定义

本方案只解决一件事：

把当前已经完成主干闭环的 language / orthography / font / bidi / interop 体系，从“主流程已可用”推进到“剩余边界被锁定、可以稳定发版、后续扩展不会再次分叉”。

当前不再重复投入的部分：

- 不再重做 orthography 数据模型扩展
- 不再重做 render policy 主入口
- 不再重做 Builder、层样式菜单、主时间轴、text-only 时间轴的基础接入
- 不再重做 import / AI / assistant 已完成的 transform 合同
- 不再重做 EAF / TextGrid 身份回放与 bidi plain-text 基础能力

本轮只处理真正剩余的四条工作线：

1. transform 扩面到剩余业务写路径
2. 复杂脚本字体验证从 glyph presence 升级到更细门禁
3. provider-neutral interop compatibility contract 与文档收口
4. 发布门禁、测试矩阵、验收脚本与风险清单收口

---

## 二、当前基线

截至今天，仓库的实际状态可视为：

- import 路径已经统一复用 applyOrthographyTransformIfNeeded(...)
- AI 写回在未显式选中来源层时，会回退到首个转写层 orthography
- assistant 语音听写写回，包括 segment timeline 分支，也已补齐相同 fallback source orthography 合同
- TextGrid / FLEx / Toolbox / TRS 的 plain-text bidi isolate 导出导入已统一
- citation / copy / search 的 plain-text normalize 合同已统一
- 字体脚本过滤、document.fonts 校验、localStorage cache、Builder 预览与风险提示已经上线

因此，剩余工作不再是“建体系”，而是“堵最后几个真实分叉点，并把验收和发布门禁补全”。

---

## 三、完成定义

本轮结束后，只有同时满足以下条件，才能把该体系标记为“完全落地”：

1. 所有机器写入、迁移、派生、批处理文本路径都统一走同一套 transform runtime contract。
2. 复杂脚本字体风险不只显示“缺字/已验证”，还具备可解释的高风险分级与更严格 fallback。
3. EAF / TextGrid 当前合同之外，已有 provider-neutral 的 metadata contract 与 compatibility note，可解释哪些字段保真、哪些字段降级。
4. orthography / transform / bidi / font 四类回归测试与手工验收清单齐备，且发布门禁明确。

如果只完成其中一部分，则最多定义为：

> 可发布版本已成立，但尚未达到平台级完全收口。

---

## 四、总体执行策略

执行原则采用严格串行，而不是继续并行散改：

1. 先做 transform 扩面
2. 再做复杂脚本字体门禁细化
3. 再补 interop compatibility contract
4. 最后做测试矩阵、验收文档、发布门禁

原因：

- transform 扩面仍然直接影响真实写路径，优先级最高
- 字体门禁细化依赖 render policy 与当前验证缓存，适合第二阶段推进
- interop 文档应基于最终 runtime 合同和降级策略编写
- 发布门禁必须以最终行为为基线，不能过早冻结

## 五、工作流 1：transform 扩面到剩余业务写路径

### 5.1 目标

把 transform 从“已进入 import / AI / assistant / export fallback”推进到“所有重要文本迁移与派生入口都统一消费同一 runtime helper”。

### 5.2 当前缺口

当前最可能仍未统一的入口类型：

- 派生复制后的批量落库路径
- 批量迁移 / bulk conversion 路径
- 任何把源层文本批量写入目标层的工具入口
- 未来新增的 controller / action hook 内部局部直写路径

### 5.3 技术策略

统一采用以下合同：

- 文本变换入口只允许调用 [src/utils/orthographyRuntime.ts](src/utils/orthographyRuntime.ts)
- 任何业务层不得直接手写 active transform 查找与 preview 调用
- 所有目标写入都显式提供：
  - targetLayerId
  - selectedLayerId
  - fallbackSourceOrthographyId
- 若 source / target 缺失或相同，必须稳定透传原文，不报错、不 silent mutate

### 5.4 实施步骤

#### Step 1：盘点剩余写路径

重点检查：

- [src/hooks](src/hooks)
- [src/pages](src/pages)
- [src/services](src/services)

检索目标：

- saveSegmentContentForLayer
- saveUtteranceText
- saveTextTranslationForUtterance
- clone / derive / migrate / batch convert 等语义入口
- 任何 source text -> target layer text 的写入逻辑

输出物：

- 一份剩余写路径清单
- 每个入口标记为：
  - manual authored，不自动 transform
  - machine write，必须 transform
  - same-layer copy，不 transform
  - cross-orthography migration，必须 transform

#### Step 2：统一 helper 接入

改造原则：

- 业务控制器只负责解析上下文
- 真正文本变换只调用 transformTextForLayerTarget(...)
- 必要时在调用点外补 fallbackSourceOrthographyId

优先检查文件：

- [src/pages/useTranscriptionAiController.ts](src/pages/useTranscriptionAiController.ts)
- [src/pages/useTranscriptionAssistantController.ts](src/pages/useTranscriptionAssistantController.ts)
- [src/hooks/useImportExport.ts](src/hooks/useImportExport.ts)
- 剩余与派生复制、批量迁移相关的 hook / service 文件

#### Step 3：补 transform chain 冲突反馈

需要补两类反馈：

- source -> target 无 active transform 时的稳定提示
- sampleCases / irreversible / chain mismatch 时的管理面提示

建议落点：

- [src/components/OrthographyTransformManager.tsx](src/components/OrthographyTransformManager.tsx)
- [src/services/LinguisticService.ts](src/services/LinguisticService.ts)
- [src/utils/orthographyTransforms.ts](src/utils/orthographyTransforms.ts)

#### Step 4：补回归测试

至少补以下四类测试：

- import 路径 transform 透传 / 命中 / source==target
- AI 写回无 selected source layer 的 fallback 行为
- assistant dictation segment timeline fallback 行为
- 新增批量迁移或派生入口的 transform 命中行为

优先测试文件：

- [src/pages/useTranscriptionAiController.test.tsx](src/pages/useTranscriptionAiController.test.tsx)
- [src/pages/useTranscriptionAssistantController.test.tsx](src/pages/useTranscriptionAssistantController.test.tsx)
- 与批处理入口对应的测试文件

### 5.5 验收标准

- 仓库内不存在新的“直接 previewOrthographyTransform 写业务文本”的重复实现
- 所有机器写入入口都能解释 source / target orthography 的来源
- 未命中 transform 时保持原文透传，不阻塞主流程
- 至少新增 1 组批量或派生路径回归测试

### 5.6 风险与回滚

风险：

- 把 manual edit 路径误接为 auto-transform，造成用户输入被错误改写
- 某些控制器没有 selectedLayerId，容易漏掉 fallback source orthography

控制策略：

- 所有入口先分类，再接 helper
- 分类不清的入口，优先保守为“不自动 transform”
- 每改 1 条入口即补 1 条测试，不最后集中补

## 六、工作流 2：复杂脚本字体验证与门禁细化

### 6.1 目标

把当前“已验证 / 缺字 / 风险高”的字体状态，细化为更可靠的复杂脚本运行时门禁，减少 glyph presence 误判为“可安全渲染”的情况。

### 6.2 当前缺口

当前已具备：

- 基于 document.fonts 的校验
- 基于 orthography exemplar 的 probe text
- localStorage cache
- Builder 与菜单风险标签
- strict-script runtime fallback

当前不足：

- 复杂脚本 shaping 风险仍未独立建模
- “字体可加载”不等于“组合附标/堆叠/连字安全”
- 警告粒度还不足以区分“可用但高风险”和“禁止推荐”

### 6.3 技术策略

在现有 render policy 上增加一个更细的风险层：

- verified
- missing-glyphs
- shaping-risk
- script-mismatch
- unsupported

并将以下脚本纳入高优先级门禁：

- Arabic
- Devanagari
- Tibetan
- Thai
- Khmer
- Myanmar

### 6.4 实施步骤

#### Step 1：扩展验证结果模型

建议优先收口到 [src/utils/layerDisplayStyle.ts](src/utils/layerDisplayStyle.ts)：

- 扩展 FontCoverageVerification 的状态枚举
- 区分 glyph 校验与 shaping 风险信号
- 允许 probe text 按 script 使用更激进样例

#### Step 2：补 shaping-risk probe 策略

按脚本新增最小风险样例，例如：

- Arabic：字母 + 数字 + 标点 + 连写组合
- Devanagari：含上标、下标、matra 组合
- Tibetan：含堆叠组合
- Thai：含上下声调与元音标记

这些样例不要求做像素级渲染比对，但至少要作为高风险脚本的严格探测文本。

#### Step 3：菜单与 Builder 风险标签升级

目标文件：

- [src/components/LayerStyleSubmenu.ts](src/components/LayerStyleSubmenu.ts)
- [src/components/OrthographyBuilderPanel.tsx](src/components/OrthographyBuilderPanel.tsx)
- [src/hooks/useLocalFonts.ts](src/hooks/useLocalFonts.ts)

期望行为：

- 推荐 · 已验证
- 脚本匹配 · 高风险
- 脚本匹配 · 缺字
- 风险高

#### Step 4：strict fallback 再收紧

目标：

- 对高风险脚本，用户选择本地字体后，即使允许保留该字体在栈首，也必须保留脚本安全 fallback
- shaping-risk 状态下不允许把该字体视为“已验证”

#### Step 5：补测试

优先测试文件：

- [src/utils/layerDisplayStyle.test.ts](src/utils/layerDisplayStyle.test.ts)
- [src/hooks/useLocalFonts.test.tsx](src/hooks/useLocalFonts.test.tsx)
- [src/components/LayerStyleSubmenu.test.ts](src/components/LayerStyleSubmenu.test.ts)
- [src/components/OrthographyBuilderPanel.test.tsx](src/components/OrthographyBuilderPanel.test.tsx)

至少覆盖：

- Arabic 高风险但非缺字
- Tibetan 高风险 probe
- Devanagari 缺字
- Latin IPA 已验证

### 6.5 验收标准

- 复杂脚本不再只有“缺字/已验证”二元状态
- 高风险脚本的自定义字体仍保留脚本安全 fallback
- 菜单、Builder、runtime fallback 三处风险语义一致
- 四类核心脚本均有测试覆盖

### 6.6 风险与回滚

风险：

- 风险判定过严，导致用户可用字体被过度标红
- 风险判定过松，继续让复杂脚本假阳性通过

控制策略：

- 初版先只把 shaping-risk 用于提示和 fallback 强化，不直接禁用用户选择
- 真正禁止某字体前，先以告警 + 测试数据验证一轮

## 七、工作流 3：interop compatibility contract 与 provider-neutral 文档收口

### 7.1 目标

把当前已经编码完成的 EAF / TextGrid / plain-text downgrade 行为，整理成一个稳定、可引用、可扩展的 compatibility contract，避免未来新增 provider 时再次各写一套 metadata 规则。

### 7.2 当前缺口

当前已有实现，但缺少统一文档与 neutral contract：

- tier identity 如何保存
- orthography snapshot 哪些字段保真
- 未识别 metadata 如何处理
- transformId 是否保真、何时仅作为提示
- 第三方 provider 不支持 metadata 时的降级策略

### 7.3 技术策略

以现有实现为准，不大改导入导出代码，只补合同与兼容说明。

合同应覆盖三层：

1. internal canonical model
2. format-specific mapping
3. downgrade / ignore / strip policy

### 7.4 实施步骤

#### Step 1：整理 canonical metadata 字段

以 [src/utils/orthographyInteropMetadata.ts](src/utils/orthographyInteropMetadata.ts) 为核心，明确 canonical 字段：

- languageId
- orthographyId
- scriptTag
- regionTag
- variantTag
- transformId

#### Step 2：逐格式写 compatibility note

至少覆盖：

- EAF
- TextGrid
- FLEx
- Toolbox
- TRS

需要说明：

- 哪些字段保真
- 哪些字段只作为 snapshot
- 哪些字段在导入时会被忽略
- 哪些字段在降级后不能 round-trip

#### Step 3：补 provider-neutral 扩展说明

文档层统一定义：

- 新 provider 若支持结构化 metadata，应优先映射 canonical 字段
- 若只支持纯文本标签或名称编码，应明确“这是 downgrade path”
- 未识别字段一律 ignore，不得 fail import

#### Step 4：补测试与文档对应性

优先文件：

- [src/services/TextGridService.test.ts](src/services/TextGridService.test.ts)
- 与 EAF / FLEx / Toolbox / TRS 相关测试文件
- [docs](docs)

要求：

- 文档里写到的 downgrade 行为，测试必须可定位
- 测试里保真的字段，文档必须同步列出

### 7.5 建议新增文档

建议新建两类文档：

- compatibility note
- provider-neutral metadata contract

### 7.6 验收标准

- 外部格式对 orthography metadata 的支持边界有文字说明
- canonical metadata 字段集合固定
- 未识别字段 ignore 策略被文档与测试双重锁定

## 八、工作流 4：发布门禁、测试矩阵、手工验收收口

### 8.1 目标

把当前“功能基本齐”转成“有门禁、有矩阵、有验收脚本”的稳定可发布状态。

### 8.2 当前缺口

虽然最近的 targeted tests 已通过，但还缺：

- 面向 orthography / transform / bidi / font 的完整回归矩阵
- 手工验收场景清单
- 文档化的已知限制与降级说明
- 发布前一次性检查路径

### 8.3 技术策略

不追求新增大量功能测试，而是把已有测试整理成可执行门禁，并补少量缺失高价值回归。

### 8.4 实施步骤

#### Step 1：建立四类测试矩阵

建议按以下四类整理：

- transform runtime
- font verification / fallback
- bidi plain-text / preview / copy
- interop downgrade / round-trip

输出物：

- 一份矩阵文档，列出每个场景对应的测试文件

#### Step 2：补手工验收脚本

至少覆盖语言组：

- Latin IPA
- Arabic
- Devanagari
- Tibetan

每组需验证：

- 层显示
- Builder 预览
- 导入导出
- copy/search snippet
- AI 或 assistant 写回行为

#### Step 3：收口已知限制

必须写明：

- shaping-risk 仍为运行时启发式，不是字体引擎级证明
- provider-neutral 合同存在 downgrade path
- 无 transform 时业务会透传原文，而不是自动失败

#### Step 4：发布命令与门禁固定化

发布前最少执行：

1. build
2. orthography / transform / bidi / interop 相关聚焦测试
3. architecture guard
4. 手工验收记录

### 8.5 验收标准

- 有清晰的测试矩阵文档
- 有手工验收脚本与执行记录模板
- 已知限制与降级路径可对外说明
- 发布前命令序列固定且可重复执行

## 十一、任务依赖关系

依赖顺序如下：

1. transform 扩面
2. transform 测试补齐
3. 字体验证细化
4. interop 文档定稿
5. 测试矩阵与发布门禁

不能倒过来的原因：

- 文档必须反映最终 runtime 合同
- 发布门禁不能在行为未锁定前冻结
- 字体高风险门禁需要建立在当前 render policy 与 cache 模型之上，而不是另起一套

## 十二、风险台账

### R1：manual edit 被误转成 auto-transform

影响：高

应对：

- 先做入口分类
- 无法确定时保守不自动 transform

### R2：复杂脚本风险判断过严

影响：中

应对：

- 初期只用于提示与 fallback，不直接封禁选择

### R3：文档与实现再次漂移

影响：高

应对：

- 每个 compatibility 文档条目必须能在测试中找到映射

### R4：热点文件再次触发 architecture guard

影响：中

应对：

- 优先把新增逻辑抽到 util / helper
- 避免继续堆大 controller 文件

## 十四、建议的直接开工顺序

如果下一轮立刻进入实施，建议按下面顺序落地：

1. 先审计所有剩余 cross-layer / batch 写路径，并列出必须 transform 的入口名单。
2. 先补这些入口，不先碰字体和文档。
3. transform 收口完成后，再推进 shaping-risk 和 strict fallback 细化。
4. 最后再写 compatibility note、测试矩阵、验收脚本与发布门禁。

这是当前收益最高、返工最少的顺序。
