# 规划：首屏拆解与发布风险收口方案（2026-03-28）

## 目标

在当前正确性已经稳定的前提下，完成下一阶段的“首屏负载拆解 + 可选能力隔离 + 发布风险收口”。

本方案只处理以下 3 类问题：

1. 默认进入转写页时，首屏仍携带过重的非必需能力。
2. 语音、PDF、Embedding、Transformers 等可选域虽然已经部分拆分，但仍通过转写主编排层产生较强耦合。
3. 构建与发布层面仍保留明显警告与待确认项，包括大 chunk、`onnxruntime-web` 的 `eval` 提示，以及新增依赖后的 audit 风险。

## 当前基线

以 2026-03-28 当前构建结果为准：

- 主入口构建通过，但仍有大 chunk 警告。
- 当前主要产物体积：
  - `TranscriptionPage`：约 877.78 kB
  - `voice-core`：约 740.85 kB
  - `pdf-vendor`：约 869.43 kB
  - `transformers`：约 824.16 kB
  - `react-vendor`：约 193.07 kB
  - `index.css`：约 156.18 kB
- `onnxruntime-web` 仍在构建中触发 `eval` 警告。
- 当前已有正向基础：
  - 路由级 lazy 已在使用。
  - `TranscriptionPage.AiSidebar` 与 `RecoveryBanner` 已做 lazy。
  - `EmbeddingProviderCatalog` 已改为 provider 级延迟装载。
  - 语音相关 `manualChunks` 已存在于 [vite.config.ts](vite.config.ts)。

## 根因判断

当前问题不是“没有拆分”，而是“拆分粒度还停留在 vendor 或局部组件层，未完成运行时职责层拆解”。

主要根因如下：

1. [src/pages/TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx) 仍然直接静态导入了语音、PDF、AI Embedding、上下文组装、搜索、恢复等多个域的入口。
2. 可选域虽然有 lazy 组件，但很多 hook、service、context value 构造仍发生在主编排层首次渲染路径上。
3. `manualChunks` 已经把语音/PDF/AI vendor 切出去，但“切出去”不等于“首屏不触发依赖图进入主工作流”。
4. `onnxruntime-web` 当前仍处于构建产物可见范围内，意味着本地 embedding / transformers 路径还没有彻底隔离成明确的按需域。

## 拆解原则

1. 先拆运行时职责边界，再调 `manualChunks`；不要先做 bundler 微调。
2. 先让“默认首屏必需能力”和“用户主动打开的可选能力”彻底分层，再做体积优化。
3. 每一批拆解都必须有独立验收标准，不能靠“感觉更轻了”。
4. 每一批都保留回滚点，避免在单次大改里把转写主链路打散后难以定位问题。

## 总体拆解蓝图

### 主轴

把当前转写页拆成 4 层：

1. Shell 层
   - 只负责路由承接、页面骨架、基础数据加载、最小交互闭环。
2. Timeline Core 层
   - 只保留转写主时间轴、选择态、基础编辑、播放控制。
3. Optional Domain Islands 层
   - 语音域、PDF 域、AI/Embedding 域改为独立岛状运行时，按需挂载。
4. Build / Release Guard 层
   - 处理 chunk 约束、CSP 风险、依赖审计、source map 与发布策略。

### 对应目标

1. 默认进入 `/transcription` 时，不再同步初始化语音、PDF、本地 embedding 相关运行时。
2. 用户只有在真正打开语音、PDF、AI/Embedding 相关面板时，才触发对应域的 hook 与依赖树。
3. 构建警告从“多个 > 500 kB 的业务块”收敛为“少量明确且可解释的可选块”。

## 分批拆解方案

## 批次 0：基线固化与观测补强

### 目标

先把优化目标量化，避免后续每批改完都无法判断收益。

### 任务

1. 固化当前构建基线到文档：记录各 chunk 体积、构建耗时、告警内容。
2. 增加一个轻量 build profiling 约定：后续每批执行后都记录变化。
3. 增加“首屏必需能力清单”，明确哪些能力默认必须可用：
   - 转写时间轴
   - 基础播放控制
   - 搜索
   - 基础编辑
4. 同时明确哪些能力属于可选域：
   - 语音 Agent
   - PDF 预览
   - Embedding 检索
   - 本地 transformers provider

### 主要文件面

- [vite.config.ts](vite.config.ts)
- [package.json](package.json)
- [docs/规划-首屏拆解与发布风险收口方案-2026-03-28.md](docs/规划-首屏拆解与发布风险收口方案-2026-03-28.md)

### 验收标准

1. 后续每批都有统一对比口径。
2. 团队对“默认首屏必须包含什么”不再有歧义。

### 回滚风险

无运行时风险，可直接执行。

## 批次 1：转写主编排层瘦身

### 目标

把 [src/pages/TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx) 从“总装配厂”拆成“壳层 + 核心区 + 可选域接入点”。

### 任务

1. 新建一个纯壳层容器，建议命名方向：
   - `TranscriptionPage.Shell.tsx`
   - `TranscriptionPage.Core.tsx`
   - `TranscriptionPage.OptionalDomains.tsx`
2. 壳层只做：
   - route props 承接
   - 基础数据 hook 初始化
   - 基础布局
   - 向核心区传递最小状态
3. 先从 orchestrator 中抽离 3 类非核心逻辑：
   - AI 侧栏上下文装配
   - PDF 预览状态装配
   - 语音 Agent 上下文装配
4. 让 orchestrator 不再直接持有所有可选域的 context value 构造。

### 主要文件面

- [src/pages/TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx)
- [src/pages/TranscriptionPage.tsx](src/pages/TranscriptionPage.tsx)
- [src/pages/index.ts](src/pages/index.ts)

### 预期收益

1. 主编排文件体积和静态依赖数显著下降。
2. 后续语音/PDF/AI 域可以独立 lazy 挂载，而不必继续寄生在 orchestrator 首次渲染路径上。

### 验收标准

1. `TranscriptionPage.Orchestrator.tsx` 不再直接静态导入语音、PDF、Embedding 三域的主要 hook 和 service。
2. 转写主链路不依赖这些可选域也能独立渲染。

### 回滚点

优先按“纯抽壳不改行为”推进；如果出现行为回退，可单独回滚 shell/core 拆分而不影响域逻辑。

## 批次 2：语音域岛化

### 目标

让语音能力彻底变成“用户主动打开后才挂载”的独立运行时，而不是首屏隐式准备好的常驻域。

### 当前问题

虽然 `voice-core` 已单独成块，但 [src/pages/TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx) 仍静态引入了：

- `useVoiceInteraction`
- `useVoiceDock`
- `useVoiceAgentContextValue`
- `VoiceAgentProvider`
- 语音 widget props 组装

这意味着首屏路径仍对语音域高度感知。

### 任务

1. 新建语音域岛组件，建议命名：
   - `TranscriptionVoiceRuntimeIsland.tsx`
2. 将以下逻辑整体迁入该岛：
   - 语音状态 hook
   - 语音 context value
   - 语音 widget props
   - 语音面板展开/收起逻辑
3. 主页面只保留：
   - 一个 `voiceEnabled` 布尔值
   - 一个 lazy 加载的语音运行时入口
4. 在用户触发以下任一动作前，不挂载语音岛：
   - 打开语音面板
   - 点击语音开关
   - 调用语音快捷键

### 主要文件面

- [src/pages/TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx)
- [src/hooks/useVoiceInteraction.ts](src/hooks/useVoiceInteraction.ts)
- [src/hooks/useVoiceDock.ts](src/hooks/useVoiceDock.ts)
- [src/hooks/useVoiceAgentContextValue.ts](src/hooks/useVoiceAgentContextValue.ts)
- [src/contexts/VoiceAgentContext.tsx](src/contexts/VoiceAgentContext.tsx)

### 预期收益

1. `voice-core` 不再通过主编排首屏参与初始化。
2. 首屏 CPU 初始化开销下降，尤其是 context/hook 构造成本下降。
3. 语音问题定位更集中，不再污染转写主编排层。

### 验收标准

1. 未触发语音 UI 前，语音域 hook 不执行。
2. 构建后 `TranscriptionPage` 主块体积下降。
3. 语音打开后的功能行为与当前一致。

### 风险

1. 语音快捷键可能依赖全局注册，需要单独设计“懒挂载前的轻入口”。
2. 首次打开语音面板时会新增一次异步加载，需要可接受的 loading UI。

## 批次 3：PDF 域岛化

### 目标

把 PDF 预览与 citation jump 相关能力从首屏主路径摘出去。

### 当前问题

`pdf-vendor` 已经独立成块，但 [src/pages/TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx) 仍直接感知 PDF 相关状态与逻辑，导致默认首屏要为 PDF 域预留运行时上下文。

### 任务

1. 新建 PDF 域岛组件，建议命名：
   - `TranscriptionPdfRuntimeIsland.tsx`
2. 将以下逻辑从主编排层抽离：
   - `usePdfPreview`
   - `PdfViewerPanel` 相关挂载逻辑
   - citation hash 跳转解析后的 PDF 面板联动
3. 设计 2 段式加载：
   - 第一段：只解析 citation 目标与面板状态，不加载 PDF runtime
   - 第二段：真正打开 PDF 预览时再加载 `pdfjs-dist`
4. 若 citation 只是跳到 note，而非打开 PDF 页面，则不触发 PDF vendor 装载。

### 主要文件面

- [src/pages/TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx)
- [src/hooks/usePdfPreview.ts](src/hooks/usePdfPreview.ts)
- [src/components/PdfViewerPanel.tsx](src/components/PdfViewerPanel.tsx)
- 任何直接引用 `pdfjs-dist` 的实现文件

### 预期收益

1. `pdf-vendor` 与默认转写首屏进一步脱钩。
2. 无 PDF 使用场景的用户首开不再承担 PDF 域初始化成本。

### 验收标准

1. 不打开 PDF 面板时，不加载 `pdf-vendor`。
2. citation 纯文本跳转不触发 PDF runtime。
3. 打开 PDF 后功能与当前一致。

### 风险

PDF 面板首次打开可能出现短暂加载态，需要明确骨架或 loading 文案。

## 批次 4：AI / Embedding 域二次隔离

### 目标

把“AI 聊天 UI”与“Embedding/检索运行时”进一步拆开，避免 AI 面板存在时仍默认初始化检索与 provider 测试链路。

### 当前问题

虽然 provider 本身已经 lazy，但 [src/pages/TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx) 仍静态依赖了：

- `EmbeddingSearchService`
- `createEmbeddingProvider`
- `testEmbeddingProvider`
- `EmbeddingService`
- `useAiEmbeddingState`
- `useEmbeddingContextValue`

这说明 AI 域内部仍是“大一统运行时”。

### 任务

1. 将 AI 聊天 UI 与 Embedding runtime 拆成两个子域：
   - Chat UI 域
   - Retrieval / Embedding 域
2. 默认只保留 Chat UI 可见性；只有在以下场景才挂载 Embedding runtime：
   - 用户打开 embedding 配置
   - 用户触发混合检索
   - 用户执行需要 embedding provider 的 AI 操作
3. provider 测试逻辑改为显式动作触发，不在主编排首次渲染中准备。
4. 若 AI 侧栏只用于普通文本对话，不应强制构造 embedding search service。

### 主要文件面

- [src/pages/TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx)
- [src/pages/TranscriptionPage.AiSidebar.tsx](src/pages/TranscriptionPage.AiSidebar.tsx)
- [src/hooks/useAiEmbeddingState.ts](src/hooks/useAiEmbeddingState.ts)
- [src/hooks/useEmbeddingContextValue.ts](src/hooks/useEmbeddingContextValue.ts)
- [src/ai/embeddings/EmbeddingProviderCatalog.ts](src/ai/embeddings/EmbeddingProviderCatalog.ts)
- [src/ai/embeddings/EmbeddingSearchService.ts](src/ai/embeddings/EmbeddingSearchService.ts)

### 预期收益

1. `transformers` 和 embedding runtime 更明确地变成可选域。
2. 普通对话与基础 AI 面板不再默认带起本地 embedding 能力。

### 验收标准

1. 未触发 embedding 相关操作前，不加载 `transformers`。
2. AI 面板普通文本对话仍可用。
3. 触发 embedding 后运行时行为与当前一致。

### 风险

需要梳理 AI 工具调用里哪些动作强依赖 embedding，避免把必须能力误拆成可选域。

## 批次 5：样式与基础资源收口

### 目标

控制 CSS 与基础共享资源继续膨胀，避免 JS 拆了但 CSS 仍成为首屏负担。

### 当前问题

当前 CSS 产物约 156.18 kB，体积不算失控，但已经足够大，需要避免继续把可选域样式长期固化在首屏公共样式里。

### 任务

1. 梳理以下域的样式是否仍写在全局样式中：
   - 语音面板
   - PDF 面板
   - AI 侧栏
   - FeatureAvailability/占位页
2. 将明显仅在可选域使用的样式迁移到域内样式模块或延迟样式载入策略。
3. 清点是否存在长期未使用但仍被全局引入的样式块。

### 主要文件面

- [src/styles](src/styles)
- [src/styles/global.css](src/styles/global.css)
- 相关组件样式文件

### 验收标准

1. CSS 不再继续随可选域功能增长而无边界膨胀。
2. 可选域样式不污染主工作区首屏样式包。

## 批次 6：构建与发布风险收口

### 目标

把当前“构建能过，但告警很多”的状态，收敛成“构建结果可解释、可发布、可审查”。

### 任务 A：`onnxruntime-web` 风险处理

1. 确认本地 transformers provider 是否必须保留在默认前端发版包中。
2. 若不是必须默认可用：
   - 进一步把本地 provider 放入显式实验能力或高级设置入口。
3. 若必须保留：
   - 形成正式发布口径，说明其 CSP / 安全审查影响。
   - 评估是否仅在特定构建 profile 启用。

### 任务 B：依赖审计收口

1. 审查 `@sentry/react` 引入后的 audit 风险来源。
2. 决定是：
   - 升级依赖链
   - 锁定兼容版本
   - 接受风险并形成记录

### 任务 C：Sentry 发布闭环

1. 既然 [src/observability/sentry.ts](src/observability/sentry.ts) 已接入 lazy init，就继续补齐：
   - release / environment 命名规则
   - source map 策略
   - PII 边界
2. 明确未配置 DSN 时的行为与发布说明。

### 主要文件面

- [vite.config.ts](vite.config.ts)
- [package.json](package.json)
- [src/observability/sentry.ts](src/observability/sentry.ts)
- [src/vite-env.d.ts](src/vite-env.d.ts)
- CI / 发布文档（若仓库中已有）

### 验收标准

1. 大 chunk 留存必须有明确理由。
2. `onnxruntime-web` 的 `eval` 不再是“已知但无人处理”的悬空项。
3. audit 风险有明确处置结论。

## 推荐执行顺序

建议按以下顺序推进，不要反过来：

1. 批次 0：固化基线
2. 批次 1：主编排层瘦身
3. 批次 2：语音域岛化
4. 批次 3：PDF 域岛化
5. 批次 4：AI / Embedding 域二次隔离
6. 批次 5：样式收口
7. 批次 6：构建与发布风险收口

原因很直接：

- 不先拆主编排层，后面所有域拆解都会继续卡在 orchestrator 上。
- 不先拆运行时边界，单调 `manualChunks` 只能得到更碎的包，不能得到更轻的首屏。

## 每批统一验收动作

每批完成后固定执行：

1. `npm run typecheck`
2. `npm test`
3. `npm run build`
4. 记录以下结果：
   - `TranscriptionPage` 主块体积
   - `voice-core` 体积
   - `pdf-vendor` 体积
   - `transformers` 体积
   - 是否仍出现 `onnxruntime-web eval` 警告

## 成功口径

当以下条件同时成立时，本轮拆解视为完成：

1. 默认进入转写页时，不再同步初始化语音、PDF、Embedding 本地运行时。
2. `TranscriptionPage` 主块明显下降，不再接近当前 877 kB 级别。
3. `voice-core`、`pdf-vendor`、`transformers` 都成为更明确的用户触发型可选块。
4. `onnxruntime-web` 风险有明确发布结论。
5. `npm run typecheck`、`npm test`、`npm run build` 持续全绿。

## 一句话结论

这不是一次“继续调包配置”的工作，而是一轮把转写页从“大一统运行时”拆成“核心首屏 + 可选域岛”的结构性收敛；先拆职责边界，再谈 chunk 数字，才是这轮优化能真正落地的路径。
