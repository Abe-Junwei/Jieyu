## 目标

本方案只处理以下 3 类问题：

1. 默认进入转写页时，首屏仍携带过重的非必需能力。

## 根因判断

当前问题不是“没有拆分”，而是“拆分粒度还停留在 vendor 或局部组件层，未完成运行时职责层拆解”。

当前剩余根因如下：

1. [src/pages/TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx) 仍然过度集中，包含 citation 跳转解析、AI prompt context 拼装、搜索与侧栏桥接等大量非时间轴核心逻辑。
2. 语音、PDF、Embedding 虽已岛化，但主编排层仍持有它们的触发判断、请求组装和部分错误汇总，尚未完全降为“纯核心区 + 接口层”。
3. `manualChunks` 已经把 vendor 拆开，但 `voice-core`、`pdf-vendor`、`transformers` 仍然体积过大，说明域内还存在可继续细拆的运行时路径。
4. `onnxruntime-web` 当前仍处于构建产物可见范围内，意味着本地 embedding / transformers 路径虽然已变成可选域，但尚未收敛成更严格的“显式高级能力”。

## 拆解原则

1. 先拆运行时职责边界，再调 `manualChunks`；不要先做 bundler 微调。
2. 先让“默认首屏必需能力”和“用户主动打开的可选能力”彻底分层，再做体积优化。
3. 每一批拆解都必须有独立验收标准，不能靠“感觉更轻了”。
4. 每一批都保留回滚点，避免在单次大改里把转写主链路打散后难以定位问题。

## 总体拆解蓝图

### 主轴

把当前转写页拆成 4 层：

1. Shell 层
   - 只负责路由承接、页面骨架、基础数据加载、最小交互闭环。
2. Timeline Core 层
   - 只保留转写主时间轴、选择态、基础编辑、播放控制。
3. Optional Domain Islands 层
   - 语音域、PDF 域、AI/Embedding 域改为独立岛状运行时，按需挂载。
4. Build / Release Guard 层
   - 处理 chunk 约束、CSP 风险、依赖审计、source map 与发布策略。

### 对应目标

1. 默认进入 `/transcription` 时，不再同步初始化语音、PDF、本地 embedding 相关运行时。
2. 用户只有在真正打开语音、PDF、AI/Embedding 相关面板时，才触发对应域的 hook 与依赖树。
3. 构建警告从“多个 > 500 kB 的业务块”收敛为“少量明确且可解释的可选块”。

## 第二轮完整拆分步骤

本轮后续工作不再按“是否要拆”讨论，而按“从哪个边界继续拆、每一步怎么验收”执行。

建议继续按以下顺序推进，不要打乱：

1. 先压 orchestrator 结构。
2. 再压 voice / embedding 域内体积。
3. 再做发布和 CI 的硬门禁。

原因：

1. orchestrator 仍是当前最大的业务主块，继续拆它的收益最大。
2. 如果先做 vendor 微调而不继续拆 orchestrator，最终只会得到更碎但仍由主编排触发的依赖图。
3. 发布和 CI 门禁必须建立在结构边界已经相对稳定之后，否则很容易因为中途继续大改而反复修改阈值。

   - 调用 jump service
   - 接收 jump result
