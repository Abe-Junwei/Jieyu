# 规划：转写页剩余开发路线（2026-03-29）

## 目标

在当前仓库里，转写页已经承载了互操作、时间轴编辑、AI 助手、语音助手、PDF/RAG、speaker 管理、批量操作、恢复与导出等绝大多数可用能力。后续开发不应再按“多页面并行铺开”推进，而应先把转写页收口成稳定、可拆分、可复用的唯一主工作台。

本路线图的目标是：

1. 把转写页从“大页承载全部能力”推进到“转写内核 + 稳定 runtime/controller 模块”。
2. 补齐真实用户工作流里的剩余断点，而不是继续横向扩页面。
3. 为后续开放 Annotation / Analysis / Writing / Lexicon 独立工作台建立前置条件。

## 当前现状

- `TranscriptionPage.Orchestrator.tsx` 已从约 3000+ 行继续收敛到约 2200 行区间。
- 本轮已新增并接入以下 controller：
	- `useTranscriptionSelectionContextController`

	- `useTranscriptionAiController`

	- `useTranscriptionTimelineContentViewModel`

- 这意味着包 A 的“编排核瘦身”已经完成一轮实质性收口；当前 Orchestrator 主要剩余 JSX 装配与少量页面级状态胶水。
- 后续新增功能默认不应再直接落到 Orchestrator，而应优先落到现有 controller 边界或新增专用 controller。

- `useTranscriptionAssistantSidebarController` 已把 `aiChatContextValue` 与 runtime props 的装配从 Orchestrator 抽离，页面层继续向“薄装配层”收口。
- 已补齐当前主线手工验收资产：`docs/execution/手工验收执行脚本-AI语音RAG主流程-2026-03-30.md` 与 `docs/execution/手工验收执行记录-AI语音RAG主流程-2026-03-30.md`，用于发布前 AI / 语音 / RAG 主流程闭环验收。
- 截至本次盘点，`TranscriptionPage.Orchestrator.tsx` 实际为 2166 行，`useTranscriptionSectionViewModels.ts` 已降到 194 行，`npm run report:architecture-hotspots` 当前无 active hotspot。
- 上述收口已补充定向回归：selection snapshot、assistant sidebar attention、voice interaction、structure tests 均通过，且 `npm run typecheck`、`npm run build` 已重新通过。

### 已有基础

- 转写页是当前唯一开放工作台。
- 导入导出、层管理、波形编辑、搜索替换、批量面板、speaker 管理、user notes、AI chat、embedding/RAG、PDF 预览、语音助手等均已在转写页中可用。
- `npm test` / `typecheck` / `build:guard` 已全部通过，说明当前主问题不是“是否可运行”，而是“是否还适合作为长期主干继续叠加功能”。

### 当前主要约束

- `TranscriptionPage.Orchestrator.tsx` 虽已收敛到约 2200 行区间，但仍承担大量 JSX 装配、跨 runtime props 拼装与少量页面级胶水，离“稳定内核 + 薄装配层”还有一段距离。
- AI / 语音 / RAG / PDF / speaker 等能力虽已落地，但边界仍主要依附在转写页内部，而不是独立 runtime。
- 时间轴、waveform、dependent layer、segment-backed layer、context menu、split/merge 等路径还有少量语义不一致风险。
- F10 对齐链路经最新调研后，判断应先降级为“架构预留项”，而不是近期产品化主线。

## 剩余项盘点（2026-03-30）

### 已完成但需要从待办中剔除的项

以下事项已在代码、结构测试和 architecture guard 中落地，不应继续列为“未完成”：

1. waveform bridge controller 已完成：`useTranscriptionWaveformBridgeController` 已接入 Orchestrator，并成为波形区的统一桥接边界。
2. track persistence controller 已完成：`useTrackEntityStateController` 与 `useTrackEntityPersistenceController` 已承担媒体作用域的 hydration / persist 逻辑。
3. tests / docs / guard 基线已完成一轮同步：结构测试、architecture guard、roadmap 文档与全量验证都已跑通。

### 当前真正剩余的收口项

#### R1. View-model 装配收口

现状：Orchestrator 里大块状态推导已拆出，但 JSX 仍依赖大量页面级 props 拼装与 runtime 间胶水。

剩余内容：

1. 将 toolbar / timeline / dialogs / ai sidebar 的大 props 组装继续下沉到更细的 view-model hook 或 runtime 壳层。
2. 收缩跨 runtime 的 ref 与 callback 透传，优先改成语义化 controller 输出。
3. 把页面里仍然存在的 toast / shortcut / focus 等零散胶水继续归并，避免 Orchestrator 重新长出第二轮“轻逻辑堆积”。

完成标准：

1. Orchestrator 进一步接近“导入 controller -> 组合 section -> 渲染”模式。
2. 新增功能默认不需要改动 Orchestrator 内的大对象装配。

#### R2. Runtime 边界再收紧

现状：assistant / analysis / pdf / voice 已具备 runtime 壳层，但页面仍承担部分跨域数据汇总。

剩余内容：

1. 继续缩小 `TranscriptionPage.AssistantRuntime.tsx`、`TranscriptionPage.AnalysisRuntime.tsx`、`TranscriptionPage.PdfRuntime.tsx` 与 Orchestrator 的接口面。
2. 明确每个 runtime 的“输入 props 合约”，避免将页面内部状态直接暴露给多个 runtime。
3. 对 voice 相关链路补 controller 或 runtime 边界，避免后续语音能力继续直接压回页面层。

完成标准：

1. runtime 的输入稳定且可被结构测试锁定。
2. 语音 / AI / PDF 继续增强时，不需要回到页面层重新拼装状态。

#### R3. 选择语义与编辑语义统一

现状：最危险的 layer fallback 与 waveform bridge 已收口，但 dependent layer / segment-backed layer / timeline context menu 仍有残余复杂度。

剩余内容：

1. 统一 `selectedTimelineUnit`、context menu、split/merge、waveform overlay 的 unit kind 与 layer fallback 语义。
2. 为 dependent layer 与 segment-backed layer 继续补交叉场景回归，尤其是选择切换、批量映射、波形入口与右键菜单。
3. 把当前“页面知道一点、controller 知道一点”的边界继续收成单一解析入口。

完成标准：

1. 同一单元在 timeline / waveform / context menu / batch path 下的目标解析一致。
2. 不再出现“当前层为空、首次打开、未聚焦层”导致的路由分叉。

#### R4. 高频编辑闭环补齐

现状：批量操作、speaker、note、undo 已经有较多回归，但仍偏向“能力存在 + 重点场景锁定”，还没完全达到工作流级闭环。

剩余内容：

1. 继续补批量拆分 / 合并 / 偏移 / 缩放与 speaker / note / undo 的组合回归。
2. 梳理删除、恢复、导入冲突、批量失败回滚等高风险路径的提示与回放体验。
3. 明确“批量编辑后如何确认影响范围、如何撤销、如何恢复”的统一反馈模式。

完成标准：

1. 高频编辑路径具备可重复执行、可撤销、可回放的闭环。
2. 组合场景不再依赖零散回归维持心智稳定。

#### R5. AI / 语音 / RAG 产品化收口

现状：能力已可用，但仍有明显“开发态”特征，尤其是任务可视化、TTS、citation 产品体验与 voice 主流程的一致性。

剩余内容：

1. 完成 TTS / 语音回放辅助 / 分析模式写回反馈的主流程收口。
2. 将 citation 脚注、文献管理和 RAG 模板从“能跳转”提升到“可持续使用”的产品体验。
3. 进一步产品化 AI 任务审计、失败重试与结果回填可见性。

完成标准：

1. AI / 语音 / RAG 明确成为转写主流程的一部分，而不是附着在侧栏里的高级功能。
2. 用户能稳定理解“任务从哪里来、引用来自哪里、结果写回到了哪里”。

#### R6. 开放其他工作台的前置治理

现状：标注 / 分析 / 写作 / 词典页仍是规划说明页，当前不应并行推进。

剩余内容：

1. 基于当前收口结果，定义哪些 controller/runtime 已具备跨页面复用条件。
2. 以 build guard / hotspot report / structure tests 为准，设定“允许解锁独立工作台”的门槛。
3. 仅在转写页稳定后，再从转写页内核抽离独立工作台，而不是平移页面壳层。

完成标准：

1. 其他工作台的开放建立在稳定内核之上，而不是复制当前页面状态。

### Phase 2：语义一致性收口

目标：把“选中什么、编辑什么、播放什么、右键操作什么”统一成一套模型。

目标：将高频编辑与 AI/语音能力从“可用”推进到“工作流稳定”。

任务：

1. 完成 R4 高频编辑闭环补齐。

1. 独立工作台的开放建立在已验证的 controller/runtime 复用上。
2. 不再需要用转写页复制品去支撑新页面。

## 每个 Phase 的具体落地方案

### Phase 1：装配层收口

#### 目标状态

将 `TranscriptionPage.Orchestrator.tsx` 收敛为“controller 装配 + section 渲染”页面，不再承担大块 view-model 拼装和 runtime 胶水。

#### 先改哪些文件

1. `src/pages/TranscriptionPage.Orchestrator.tsx`
2. `src/pages/TranscriptionPage.Toolbar.tsx`
3. `src/pages/TranscriptionPage.TimelineTop.tsx`
4. `src/pages/TranscriptionPage.TimelineContent.tsx`
5. `src/pages/TranscriptionPage.Dialogs.tsx`
6. `src/pages/TranscriptionPage.AiSidebar.tsx`
7. `src/pages/TranscriptionPage.AssistantRuntime.tsx`
8. `src/pages/TranscriptionPage.AnalysisRuntime.tsx`
9. `src/pages/TranscriptionPage.PdfRuntime.tsx`

#### 拆分顺序

##### P1-A. 先下沉 section 级 props 组装

做法：

1. 为 toolbar、timeline top、timeline content、dialogs、ai sidebar 各自建立轻量 view-model hook，优先放在 `src/pages/` 下，命名保持 `useTranscription*ViewModel` 或 `use*SectionController`。
2. Orchestrator 不再直接拼大对象，而是只拿各 section 的最终渲染参数。
3. 将现有 `useTranscriptionRuntimeProps.ts` 中仍能继续聚合的字段，统一回收到更明确的 section/runtime 边界。

建议新增文件：

1. `src/pages/useTranscriptionToolbarViewModel.ts`
2. `src/pages/useTranscriptionTimelineTopViewModel.ts`
3. `src/pages/useTranscriptionTimelineContentViewModel.ts`
4. `src/pages/useTranscriptionDialogsViewModel.ts`
5. `src/pages/useTranscriptionAiSidebarViewModel.ts`

##### P1-B. 再压缩 runtime 接口面

做法：

## 开发包划分

### 包 A：编排核瘦身与运行时拆分

#### 目标

降低 `TranscriptionPage.Orchestrator` 的体量和职责密度，让转写页后续开发不再持续堆到单文件中。

#### 主要任务

1. 继续拆分 UI state、layout state、assistant runtime、analysis runtime、pdf runtime、speaker runtime、waveform runtime。
2. 将页面级派生数据与回调从 Orchestrator 中抽到专用 hook/controller。
3. 为每个 runtime 定义稳定 props 合约，避免继续把“大而全状态对象”层层透传。
4. 增加结构测试与行数/依赖门禁，防止后续回流到单文件膨胀。

#### 验收标准

1. `TranscriptionPage.Orchestrator.tsx` 显著下降到可维护区间。
2. 新功能不再需要默认落在 Orchestrator 中实现。
3. 转写页结构测试能覆盖主要 runtime 边界。

### 包 B：时间轴 / 波形 / 选择语义统一

#### 目标

把转写页里的 layer/unit/waveform 语义收敛成单一心智模型，避免同一操作在不同路径下走不同 fallback 或不同 unit kind。

#### 主要任务

1. 统一 `selectTimelineUnit`、context menu、split/merge、waveform overlay 对 layerId 的 fallback 语义。
2. 继续补 dependent transcription layer 与 segment-backed layer 的统一读写模型。
3. 让 waveform 控件按“当前波形包含的 unit”而不是单一 `selectedUtterance` 决定展示与操作目标。
4. 为 dependent layer / selectedTimelineUnit / waveformTimelineItems 的交叉场景补回归测试。

#### 验收标准

1. 不再出现“当前层为空导致选中/操作失败”的边界行为。
2. waveform、timeline、context menu 对同一选中单元的理解一致。
3. dependent layer 编辑与 segment-backed layer 编辑在用户视角下表现一致。

### 包 C：高频编辑闭环补齐

#### 目标

把目前“有面板但未完全闭环”的高频编辑能力补齐到可长期使用状态。

#### 主要任务

1. 完成批量拆分 / 合并 / 时间偏移 / 缩放 的最终收口与回归。
2. 把 speaker、notes、undo/redo、批量操作之间的组合场景补齐。
3. 把 AI 分析写回、恢复、删除、导入冲突这些高风险路径继续补产品级提示与回放能力。

#### 验收标准

1. 批量操作可以覆盖日常大多数时间轴修订场景。
2. 任意批量编辑后都能稳定撤销或恢复。
3. speaker 与 note 不再是“附加功能”，而是完整地进入批量编辑工作流。

### 包 D：F10 对齐能力预留（暂缓产品化）

#### 目标

为未来可能的单语种/有资源前提下的强制对齐试点预留任务模型、结果模型与 provider 接口，但当前阶段不把它作为近期产品化目标。

#### 推荐顺序

1. 先抽象 provider-neutral 的对齐任务接口、状态模型、provenance 与人工确认边界。
2. 待目标语言具备词典/G2P/评估样本后，再以单语种试点方式评估 WhisperX、MFA 或其他 provider。

#### 主要任务

1. 定义对齐任务类型、输入参数、状态展示与错误分级，但不承诺近期接入具体 provider。
2. 预留“机器建议结果”与“人工确认结果”的分层写回模型，避免未来试点时污染现有真源。
3. 保持与 undo/redo、audit、saveState、speaker / waveform 的接口兼容，但不把 F10 作为近期排期阻塞项。

#### 验收标准

1. 不同对齐 provider 可在不改页面主逻辑的前提下接入或替换。
2. 机器建议结果与人工确认结果的 provenance/写回边界清晰。
3. 在未满足资源前提前，产品路线不会错误承诺“通用低资源强制对齐”。

### 包 E：AI / 语音 / RAG 的转写页产品化

#### 目标

把当前“能力已存在但仍偏内嵌开发态”的 AI 与语音功能，打磨成真正面向转写页主流程的产品能力。

3. 为 RAG 检索增加场景模板（问答 / 审校 / 术语查证）。
4. 补文献管理与 citation 脚注体验，而不是只保留服务层和侧栏底层能力。
5. 将 AI 任务审计、回放、失败重试进一步产品化。

#### 验收标准

1. AI 与语音不再只是“转写页附属实验能力”，而是主工作流的一部分。
2. 用户能清楚看到任务状态、来源引用和结果回填路径。
3. 语音与 AI 不再显著增加编排层耦合度。

### 包 F：性能、包体与开放条件

#### 目标

在继续开发转写页余项时，避免主包与运行时边界重新失控，并明确何时可以开放其余工作台。

#### 主要任务

2. 为转写页建立“开放其他工作台”的准入条件。

1. 转写页 Orchestrator 体量与边界完成一轮实质性收口。
2. F10 至少完成 provider-neutral 预留设计；若要进入实现，需先满足目标语言资源前提并完成单语种试点立项。
3. AI / 语音 / RAG 在转写页内的主流程已稳定，不再频繁回流改架构。
4. 需要迁出的能力具备独立 runtime/controller，而不是只能从转写页复制代码。
