# 代码审计报告：全面盘点（2026-03-27 复核版）

## 审计范围
- 代码范围：前端壳层、转写主链路、AI/语音工具链、构建与运行时入口
- 审计方法：静态代码审查、命令行验证、关键链路对照复核、基础启动烟测
- 已执行验证：
  - `npm run typecheck` 通过
  - `npm run build` 通过
  - `npm test` 通过（120/120 文件，1024/1024 用例）
  - 本地开发服务可启动，首页可打开

## 结论摘要

当前仓库在“可编译、可测试”层面是稳定的，但仍存在一组不会被现有自动化直接拦下的真实问题，主要集中在：

1. AI / 语音工具链声明与执行器不一致，部分已暴露工具实际不可用。
2. App 顶层壳层存在死入口和误导性快捷键提示。
3. 四个主导航页仍是占位实现，但已经作为正式路由暴露。
4. 搜索覆盖层与主题系统割裂，深色模式下会出现明显的 UI 不一致。
5. 默认首屏直达转写页，而转写相关 chunk 仍明显过大，构建也持续出现 `onnxruntime-web` 的 `eval` 风险警告。

以下问题均已做二次核对。

## 发现的问题

### P1. AI / 语音工具面暴露了多条不可执行或半执行链路

证据链：

- 语音工具 schema 对外声明了 `search_segments`、`auto_gloss_segment`、`auto_translate_segment`、`nav_to_time`、`split_at_time`、`zoom_to_segment` 等能力，见 [src/ai/voice/VoiceActionTools.ts](src/ai/voice/VoiceActionTools.ts#L22) 和 [src/ai/voice/VoiceActionTools.ts](src/ai/voice/VoiceActionTools.ts#L248).
- LLM 提示词也把这些工具明确列为允许返回的工具，见 [src/services/VoiceIntentLlmResolver.ts](src/services/VoiceIntentLlmResolver.ts#L18).
- 但执行器中：
  - `search_segments` 只调用 `ctx.executeAction('search')`，完全丢弃 `query` 与 `layers` 参数，见 [src/hooks/useAiToolCallHandler.ts](src/hooks/useAiToolCallHandler.ts#L660).
  - `auto_translate_segment` 直接返回“暂未实现”，见 [src/hooks/useAiToolCallHandler.ts](src/hooks/useAiToolCallHandler.ts#L685).
  - `nav_to_time` 永远返回失败，见 [src/hooks/useAiToolCallHandler.ts](src/hooks/useAiToolCallHandler.ts#L702).
  - `split_at_time` 永远返回失败，见 [src/hooks/useAiToolCallHandler.ts](src/hooks/useAiToolCallHandler.ts#L707).
  - `zoom_to_segment` 忽略 schema 里的 `zoomLevel`，只做导航，不做缩放，见 [src/hooks/useAiToolCallHandler.ts](src/hooks/useAiToolCallHandler.ts#L729).
- `auto_gloss_segment` / `auto_translate_segment` 在 voice schema 中使用 `segmentId`，但 chat schema / helper 又使用 `utteranceId`，见 [src/ai/voice/VoiceActionTools.ts](src/ai/voice/VoiceActionTools.ts#L261)、[src/ai/chat/toolCallSchemas.ts](src/ai/chat/toolCallSchemas.ts#L173)、[src/ai/chat/toolCallHelpers.ts](src/ai/chat/toolCallHelpers.ts#L434)。而 `glossAdapter` 本体也只接受 `utteranceId`，见 [src/hooks/useAiToolCallHandler.ts](src/hooks/useAiToolCallHandler.ts#L528).

影响：

- 模型可能合法选择这些工具，但执行结果是无动作、假成功或直接失败。
- 这类错误不会被类型系统或现有全量测试拦住，属于真实线上链路断裂。

复核结果：

- 当前 `useAiToolCallHandler` 的测试文件未覆盖这些工具名，`rg` 结果为空。

### P1. App 顶层壳层暴露了死入口，且搜索快捷键提示错误

证据链：

- 顶栏搜索按钮和用户按钮都没有 `onClick`，见 [src/App.tsx](src/App.tsx#L147).
- 搜索按钮标题写的是 “搜索 (⌘K)”，见 [src/App.tsx](src/App.tsx#L149).
- 真实默认快捷键却是 `mod+f`，见 [src/services/KeybindingService.ts](src/services/KeybindingService.ts#L26).
- 搜索弹层状态 `showSearch` 归属于转写页对话框状态，而不是 App 壳层全局状态，见 [src/pages/TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx#L433).
- `useKeybindingActions` 也只在转写页使用，见 [src/pages/TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx#L2069).

影响：

- 用户在任意页面都能看到搜索入口和用户入口，但点击后无任何反应。
- 顶栏快捷键提示与真实绑定不一致，会直接误导用户操作。
- 搜索能力实际上是“转写页局部能力”，但 UI 呈现成了“全局壳层能力”。

复核结果：

- 当前没有 App 壳层级测试文件，也没有 `SearchReplaceOverlay` 聚焦测试文件。

### P2. 四个主导航路由仍是占位实现，但已作为正式功能入口暴露

证据链：

- App 已正式注册 `/annotation`、`/analysis`、`/writing`、`/lexicon` 四个路由，见 [src/App.tsx](src/App.tsx#L181).
- 但对应页面文件当前只有标题和一段说明文案：
  - [src/pages/AnnotationPage.tsx](src/pages/AnnotationPage.tsx#L1)
  - [src/pages/AnalysisPage.tsx](src/pages/AnalysisPage.tsx#L1)
  - [src/pages/WritingPage.tsx](src/pages/WritingPage.tsx#L1)
  - [src/pages/LexiconPage.tsx](src/pages/LexiconPage.tsx#L1)

影响：

- 从产品视角，这四个页面已在顶栏导航中暴露，会被理解为可用模块。
- 从工程视角，这不是单纯“未上线”，而是“已有入口、无业务实现”的功能缺口。

复核结果：

- 现有测试未发现这四页的行为验证。

### P2. 搜索覆盖层未接入主题系统，深色模式下会出现明显的视觉割裂

证据链：

- App 明确支持深浅主题切换，见 [src/App.tsx](src/App.tsx#L158).
- 但搜索覆盖层直接写死 `background: '#fff'`、浅色边框和浅色文本，没有读取任何主题变量，见 [src/components/SearchReplaceOverlay.tsx](src/components/SearchReplaceOverlay.tsx#L124).

影响：

- 深色主题下搜索层会以纯白浮层覆盖在深色界面之上，视觉和可读性都会明显出戏。
- 这类问题不会导致测试失败，但属于真实前端实现缺陷。

### P2. 默认首屏直达转写页，但转写首屏资源体积仍明显过大

证据链：

- 默认路由会立即跳转到 `/transcription`，见 [src/App.tsx](src/App.tsx#L182).
- 构建虽通过，但本次审查实测产物仍包含多个超大 chunk：
  - `TranscriptionPage` 约 883 kB
  - `pdf-vendor` 约 869 kB
  - `transformers` 约 824 kB
  - `voice-core` 约 741 kB
- 同一次构建还持续出现 `node_modules/onnxruntime-web/dist/ort-web.min.js` 使用 `eval` 的警告。
- 当前分包策略定义见 [vite.config.ts](vite.config.ts#L40).

影响：

- 首屏就是转写页，意味着这些大体积资源风险会直接体现在主工作区首开体验上。
- `eval` 警告代表后续若接 CSP 或更严格发布环境，存在兼容与安全审查风险。

说明：

- 这是性能/发布风险，不是立即崩溃型 bug，但已经足够影响上线质量。

## 未复现为失败、但已确认存在的系统缺口

1. `initSentryForReleaseStage` 目前仍是占位 gate，没有真实 SDK 初始化，见 [src/observability/sentry.ts](src/observability/sentry.ts#L1).
2. AI / 语音工具链存在明显的测试空白，尤其是本报告中提到的几条失败工具路径。
3. App 壳层级交互没有专门测试，导致死按钮和错误快捷键提示能长期存在而不报红。

## 本轮审计未发现的问题

1. 当前代码不存在编译级错误。
2. 当前全量测试链可稳定通过。
3. 应用可以实际启动，至少不存在首屏即崩溃的问题。

## 建议处理顺序

1. 先收敛 AI / 语音工具面：要么删掉未实现工具暴露，要么把执行器补齐，并统一 `segmentId` / `utteranceId` 参数语义。
2. 修复 App 壳层死入口：搜索按钮接入真实打开逻辑，用户按钮若暂无能力则隐藏，快捷键提示改成真实绑定。
3. 对四个占位页做路由降级策略：隐藏入口、加明确的开发中标识，或补最小可用实现。
4. 把搜索覆盖层改成主题变量驱动。
5. 针对转写首页做进一步按域拆包，并评估 `onnxruntime-web` 的发布策略。
