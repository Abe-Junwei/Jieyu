# 规划：全面缺陷修复方案（2026-03-27）

## 目标

基于 [docs/代码审计报告-全面盘点-2026-03-27-复核版.md](docs/代码审计报告-全面盘点-2026-03-27-复核版.md) 的发现，完成一轮“仓库级质量收口”：

1. 修掉已经暴露给用户或模型、但实际不可用的功能链路。
2. 修掉壳层级死入口、错误提示和前端主题不一致问题。
3. 对已暴露但未上线的页面给出明确降级或最小可用策略。
4. 补齐关键回归测试，防止同类问题再次回流。
5. 对首屏体积和发布风险建立一轮现实可落地的收敛方案。

## 修复原则

1. 先修“假能力”与“死入口”，再修体验与性能。
2. 先让 schema、提示词、执行器三者一致，再谈增强。
3. 不保留“对外暴露但内部未实现”的中间状态。
4. 每一批修复必须同时补对应测试，否则不算收口完成。

## 范围分级

### P0. 必须立即修复

- AI / 语音工具链中已经声明但不可执行、假成功或字段不一致的工具。
- App 顶栏死按钮、错误快捷键提示、壳层与页面能力边界不一致。

### P1. 本轮应完成

- 四个已暴露主导航页的降级或最小可用处理。
- 搜索覆盖层主题适配。
- 针对新增修复点的测试补齐。

### P2. 本轮立项、可分批完成

- 转写首屏大 chunk 收敛。
- `onnxruntime-web` `eval` 发布风险评估与替代策略。
- Sentry 占位 gate 的正式接入方案。

## 执行阶段

## 阶段 A：AI / 语音工具面收口

### A1. 统一工具参数语义

目标：统一 `segmentId` / `utteranceId` 口径，避免 schema、helper、adapter 三套语义漂移。

建议方案：

1. 明确一句段对象在 AI 工具中的 canonical 参数名。
   - 建议统一为 `utteranceId`。
   - 理由：当前执行上下文、AutoGlossService、转写主链路都围绕 utterance 工作。
2. 将以下文件全部统一到同一参数名：
   - `src/ai/voice/VoiceActionTools.ts`
   - `src/ai/chat/toolCallSchemas.ts`
   - `src/ai/chat/toolCallHelpers.ts`
   - `src/hooks/useAiToolCallHandler.ts`
   - 若有文档提示词、tool feedback 文案，也一并改齐。

验收标准：

1. 任意一句段类工具的 schema、contextFill、validateArgs、adapter 实际读取字段完全一致。
2. 不再存在“schema 写 segmentId，adapter 只读 utteranceId”的情况。

### A2. 清理不可执行工具暴露

目标：不能执行的工具不能继续出现在“允许返回工具集合”里。

两种可选策略，必须二选一，不能混用：

1. 真正补齐实现。
2. 从 schema、提示词、helper、tool registry 中完全下线。

建议决策：

1. `search_segments`
   - 若保留：至少要消费 `query` 并把 query 注入搜索 UI 初始值。
   - 若当前做不到：下线 `query/layers` 参数，改成单纯 `open_search` 语义。
2. `auto_translate_segment`
   - 当前直接失败，优先下线暴露，直到真正实现。
3. `nav_to_time`
   - 若页面已有 `player.seekTo` 或秒级 seek 能力，就直接接入。
   - 若没有稳定入口，则下线该工具。
4. `split_at_time`
   - 若已有 `splitUtterance(utteranceId, splitTime)`，则应在当前选中句段上下文里补齐执行。
   - 若没有完整上下文保证，则先下线。
5. `zoom_to_segment`
   - 若保留 `zoomLevel` 参数，就必须真正驱动 zoom 行为。
   - 若暂时只支持导航，则去掉 `zoomLevel` 并改名/改文案，避免假承诺。
6. `auto_segment`、`suggest_segment_improvement`、`analyze_segment_quality`
   - 若当前只是引导用户改走 AI 面板或手动操作，不应继续作为“已支持工具”暴露给模型。

涉及文件：

- `src/ai/voice/VoiceActionTools.ts`
- `src/services/VoiceIntentLlmResolver.ts`
- `src/ai/chat/toolCallSchemas.ts`
- `src/ai/chat/toolCallHelpers.ts`
- `src/hooks/useAiToolCallHandler.ts`
- 可能涉及 `src/hooks/useAiChat.config.ts`、`src/ai/chat/promptContext.ts`、`src/ai/messages/toolFeedback.ts`

测试补齐：

1. 在 `src/hooks/useAiToolCallHandler.test.tsx` 新增以下回归：
   - `search_segments` 会携带 query 打开搜索，或被明确拒绝且不再暴露。
   - `auto_gloss_segment` 参数名统一后可正常命中 adapter。
   - `auto_translate_segment` 若保留则可执行；若下线则 schema/helper 不再出现。
   - `nav_to_time` / `split_at_time` / `zoom_to_segment` 的行为与 schema 一致。
2. 若变更 Voice LLM prompt，应补 `VoiceIntentLlmResolver` 的快照或字符串断言测试。

退出条件：

1. 模型只能选到“真正可执行”的工具。
2. 工具 schema 与执行器不再有语义漂移。

## 阶段 B：App 壳层入口收口

### B1. 修复顶栏死按钮

目标：壳层中展示的按钮必须可点击且行为明确。

建议方案：

1. 搜索按钮：
   - 如果搜索只对转写页有效，则在非转写页隐藏或置灰，并给出清晰提示。
   - 如果要做全局入口，则把壳层按钮接到全局搜索状态，而不是转写页私有状态。
2. 用户按钮：
   - 若暂无账户/设置面板，直接隐藏。
   - 不接受“展示可点击按钮但什么都不做”。

涉及文件：

- `src/App.tsx`
- `src/pages/TranscriptionPage.Orchestrator.tsx`
- 可能涉及 `src/hooks/useDialogs.ts` 或其他对话框状态文件

### B2. 修复快捷键提示与真实绑定不一致

目标：文案必须反映真实 keymap。

建议方案：

1. 顶栏搜索按钮 title 改为真实默认快捷键。
2. 更进一步：不要手写 `⌘K` / `⌘F`，而是从 `DEFAULT_KEYBINDINGS` 或运行时 keymap 派生显示。
3. 若支持用户自定义快捷键，按钮 tooltip 应尽量从当前 keymap 动态读取。

涉及文件：

- `src/App.tsx`
- `src/services/KeybindingService.ts`

测试补齐：

1. 新增 App 壳层测试文件，至少覆盖：
   - 搜索按钮在非转写页的行为。
   - 用户按钮不再是死入口。
   - 搜索 tooltip 与默认 keybinding 一致。

退出条件：

1. 壳层中不再存在无行为按钮。
2. 快捷键提示与真实绑定一致。

## 阶段 C：已暴露占位页收口

目标：路由暴露状态与产品真实状态一致。

两种修法，建议择其一：

1. 降级策略
   - 在导航里隐藏未上线页面。
   - 或保留入口，但页面必须有明确“开发中 / 暂不可用”状态组件，而不是普通内容占位。
2. 最小可用策略
   - 为每页补最小骨架功能，例如空态、数据读取状态、明确的下一步 CTA。

建议优先：

1. 若本轮不准备做完整功能，先走降级策略。
2. 把四页统一改成一个标准 `ComingSoonPage` / `FeatureGatePage` 组件，避免散落硬编码段落。

涉及文件：

- `src/App.tsx`
- `src/pages/AnnotationPage.tsx`
- `src/pages/AnalysisPage.tsx`
- `src/pages/WritingPage.tsx`
- `src/pages/LexiconPage.tsx`
- 可新增统一占位组件文件

测试补齐：

1. 为四页新增最小路由测试或 App 壳层导航测试。
2. 若隐藏入口，则校验导航中不再出现它们。

退出条件：

1. 不再存在“正式导航入口 + 普通段落占位”的中间状态。

## 阶段 D：搜索覆盖层前端实现收口

目标：搜索层遵守应用主题系统和 UI 规范。

建议方案：

1. 将 `SearchReplaceOverlay` 内联样式改为 class + CSS 变量。
2. 使用现有主题变量或在全局样式中补齐 overlay 变量。
3. 同时补 backdrop、焦点态、按钮禁用态，避免仅修背景色。

涉及文件：

- `src/components/SearchReplaceOverlay.tsx`
- `src/styles/global.css` 或相关共享样式文件

测试补齐：

1. 新增 `SearchReplaceOverlay` 组件测试，覆盖：
   - 基本打开/关闭
   - Enter / Shift+Enter 导航
   - replace preview 展示
2. 若仓库已有主题测试惯例，可补 class 或 attribute 断言。

退出条件：

1. 深色模式下不再出现纯白割裂浮层。

## 阶段 E：性能与发布风险收口

### E1. 首屏体积收敛

目标：降低转写首屏的同步负载。

建议方案：

1. 继续拆 `TranscriptionPage`：
   - 把 AI 侧栏、PDF 预览、语音能力、embedding/transformers 相关逻辑进一步懒加载。
2. 明确哪些依赖不应进入默认转写首屏。
   - `pdfjs-dist`
   - `@xenova/transformers`
   - 语音域非首帧必需逻辑
3. 若转写页默认不一定用到 PDF/AI/Voice，则入口处改为按面板打开时动态 import。

涉及文件：

- `src/pages/TranscriptionPage*.tsx`
- `vite.config.ts`
- 相关 AI/PDF/Voice 入口文件

验收标准：

1. `TranscriptionPage` 主 chunk 明显下降。
2. 构建警告数量减少，至少不再同时存在多个接近 1 MB 的首屏相关 chunk。

### E2. `onnxruntime-web` `eval` 风险处理

目标：给出可发布口径，而不是放任警告常驻。

建议方案：

1. 先做技术结论确认：
   - 当前是否必须在主构建产物里带上该依赖。
   - 是否能完全下沉到 worker / 可选模块。
2. 若短期无法移除：
   - 在发布说明中记录 CSP / 安全审查影响。
   - 给出环境限制与替代 provider 建议。
3. 若可以移除：
   - 把本地 embedding / transformers 路径进一步隔离到按需加载域。

## 阶段 F：可观测性与治理补强

### F1. Sentry gate 正式化

目标：不要让 observability 入口长期停留在空壳状态。

建议方案：

1. 若近期不接 Sentry：
   - 将 gate 文档化为明确的“未启用”策略。
   - 避免给人“已启用可观测性”的假象。
2. 若近期要接：
   - 单独立项，补 SDK、release/environment、source map、PII 策略。

### F2. 测试治理补强

必须新增的测试集：

1. `useAiToolCallHandler` 工具面一致性回归。
2. App 壳层按钮/tooltip/导航回归。
3. `SearchReplaceOverlay` 交互测试。
4. 若占位页改为 FeatureGate，也要补路由或导航可见性测试。

## 建议批次切分

### 批次 1：P0 工具链 + 壳层入口

包含：

1. 工具参数统一
2. 下线或补齐不可执行工具
3. 搜索/用户按钮收口
4. 快捷键提示改为真实值
5. 对应测试补齐

验收：

1. `npm run typecheck`
2. `npm test`
3. 人工验证：壳层按钮行为正确，语音/AI 工具不再出现假能力

### 批次 2：占位页 + 搜索覆盖层主题化

包含：

1. 四页降级或最小可用改造
2. 搜索覆盖层改用主题变量
3. 对应测试补齐

验收：

1. `npm run typecheck`
2. `npm test`
3. 人工验证：深色模式下搜索层视觉正常，导航不再误导

### 批次 3：性能与发布风险

包含：

1. 转写页进一步拆包
2. `onnxruntime-web` 风险评估与隔离策略
3. 可观测性 gate 策略定稿

验收：

1. `npm run build`
2. chunk 体积对比记录
3. 发布说明更新

## 最终验收口径

当以下条件同时满足时，本轮修复视为完成：

1. 不再有对外暴露但内部未实现的 AI / 语音工具。
2. App 壳层不再存在死按钮和错误快捷键提示。
3. 导航暴露状态与真实功能状态一致。
4. 搜索覆盖层遵守主题系统。
5. 转写首页体积风险有明确下降或明确隔离方案。
6. 对应测试已补齐，`npm run typecheck`、`npm test`、`npm run build` 全绿。

## 一句话执行建议

先做“去假能力”和“去死入口”，再做页面降级与主题一致性，最后处理性能与发布风险；不要从性能优化开始，因为当前最严重的问题不是慢，而是对外承诺与实际行为不一致。
