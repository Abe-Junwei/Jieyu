# 规划：语言与 ISO 双向自动识别统一方案 - 2026-04-04

## 1. 目标

在所有需要填写“语言名称 + ISO 代码”的入口，统一支持以下能力：

- 输入语言名称时，自动搜索候选并回填对应 ISO 639-3 代码。
- 输入 ISO 代码时，自动识别并回填语言名称。
- 语言名称输入框与 ISO 代码输入框分离，两个字段都允许手动编辑。
- 支持方言、宏语言、脚本、地区、变体等细分场景。
- 支持重名、别名、历史名、弃用码的歧义提示与人工确认。
- 保持现有持久化主键仍以 ISO 639-3 为核心，不破坏现有服务契约。

## 3. 业内通用做法

### 3.1 标准来源分层

业界通常不把“语言名称展示”“代码合法性校验”“标签规范化”混成一层，而是拆成三层：

- 语言代码基线：ISO 639-3
  - 负责语言实体唯一标识。
  - 适合你当前仓库的持久化主键模型。
- 语言标签规范：IANA Language Subtag Registry / BCP 47 / RFC 5646
  - 负责脚本、地区、变体、弃用码、首选值、宏语言、Suppress-Script、Variant Prefix 等规范信息。
  - 适合处理方言、正字法、地区变体、历史拼写等问题。
- 本地化显示名：CLDR
  - 负责多语言展示名、脚本名、地区名、别名展示。
  - 适合做 UI 搜索候选与本地化名称显示。

### 3.2 JavaScript 生态常见方案

- 当前仓库已用 `iso-639-3`
  - 优点：轻量、类型齐全、足够覆盖 ISO 639-3 基础信息。
  - 缺点：没有 BCP 47 规范层，无法直接处理 variant/prefix/preferred-value/macrolanguage 细节。
- `language-tags`
  - 基于 IANA registry，可做 tag/subtag 校验、描述搜索、弃用码 preferred value、macrolanguage 查询。
  - 适合做“标签合法性 + 规范化 + 歧义辅助”。
- CLDR / cldr-json
  - 适合做显示名、脚本/地区本地化名称。
  - 直接全量引入过重，建议只在构建期抽取当前 UI 需要的精简数据。

### 3.3 对本项目的建议

- 短期不建议直接全量引入 CLDR 原始包进运行时。
- 短期建议保留 `iso-639-3`，新增一层构建期生成的语言索引。
- 中期建议引入 `language-tags` 或等价的 IANA/BCP 47 解析能力，补齐方言/变体/弃用码/首选值处理。
- 长期如要做大规模语言选择器、地区/脚本/变体本地化名称，再引入 CLDR 精简产物。

## 4. 关键设计原则

### 4.1 持久化主标识继续使用 ISO 639-3

现有系统大量依赖 `languageId` 为 ISO 639-3：

- 项目主语言
- 层语言
- 正字法语言
- 导入导出互操作元数据

因此本次不建议把 `languageId` 改成任意 BCP 47 标签，否则会波及：

- 约束服务
- 正字法 identity
- 导入导出 round-trip
- AI 工具调用
- 语音/STT provider 适配

建议继续保持：

- `languageId`: ISO 639-3 主语言标识。
- `scriptTag` / `regionTag` / `variantTag` / `localeTag`: 用于表达细分变体。

### 4.2 输入允许“智能识别”，存储必须“结构化收敛”

也就是：

- 用户可以输入：
  - 普通话
  - Mandarin
  - cmn
  - zh-cmn
  - yue-Hant-HK
  - Belarusian Taraskievica
- 但落库时要拆成：
  - languageId
  - scriptTag
  - regionTag
  - variantTag
  - localeTag
  - 或在无法结构化时给出待确认提示

### 4.3 “尽量精确，但不过度编码”

参照 RFC 5646 的原则：

- 如果方言/语言本身已有独立 ISO 639-3，优先存独立代码。
  - 例如普通话用 `cmn`，粤语用 `yue`，客家话用 `hak`，不要一律折叠成 `zho`。
- 如果是同一语言下的书写系统/地区/拼写变体，则保留同一 `languageId`，细节下沉到脚本/地区/变体字段。
- 如果用户输入的是宏语言名，如“中文 / Chinese”，需要提示这可能对应多个更具体语言。

## 5. 建议的数据模型

建议新增统一的语言目录层，例如：

- `src/utils/languageCatalog.ts`
- 或 `src/services/language/LanguageRegistry.ts`

建议暴露的数据结构：

```ts
export interface LanguageCatalogEntry {
  iso6393: string;
  iso6391?: string;
  iso6392B?: string;
  iso6392T?: string;
  canonicalBcp47?: string;
  names: {
    zh?: string[];
    en?: string[];
    native?: string[];
  };
  aliases: string[];
  scope: 'individual' | 'macrolanguage' | 'collection' | 'special';
  type: 'living' | 'historical' | 'extinct' | 'ancient' | 'constructed' | 'special';
  macrolanguage?: string;
  suppressScript?: string;
  scripts?: string[];
  regions?: string[];
  variants?: string[];
  deprecated?: boolean;
  preferredIso6393?: string;
}

export interface LanguageMatch {
  entry: LanguageCatalogEntry;
  score: number;
  matchSource:
    | 'iso6393-exact'
    | 'iso6391-exact'
    | 'alias-exact'
    | 'name-exact'
    | 'bcp47-primary'
    | 'fuzzy-name'
    | 'deprecated-alias';
  warnings: string[];
}
```

建议把现有 `langMapping.ts` 从“工具函数集合”升级为“统一语言解析服务”。

### 6.1 必要 API


### 6.2 核心匹配规则

匹配优先级建议为：

1. ISO 639-3 精确命中。
2. ISO 639-1 / ISO 639-2 精确命中并转换到 ISO 639-3。
3. 已登记别名精确命中。
4. 标准名精确命中。
5. BCP 47 输入解析后命中 primary language。
6. 模糊匹配候选排序。
7. 弃用码或旧别名命中时，返回 preferred value 并带提示。

### 6.3 歧义处理规则

`resolveLanguageQuery(...)` 目前直接返回一个代码，这对于 UI 已不够。

需要改成：

- 非歧义时自动填充。
- 有多个高置信候选时，不自动提交，展示候选列表。
- 如果输入的是宏语言或重名词，展示说明。

示例：

- 输入 `中文`
  - 候选：`cmn` 普通话、`zho` 中文/汉语宏语言、`yue` 粤语、`wuu` 吴语、`nan` 闽南语。
  - 默认不直接写 `zho`，而是提示“请确认更具体语言”。
- 输入 `Mandarin`
  - 直接命中 `cmn`。
- 输入 `zh`
  - 这是 ISO 639-1，不是当前持久化主键。
  - 自动转换候选到 `zho`，但提醒“该代码是宏语言，可能需要更具体的 `cmn` / `yue` / `wuu`”。

### 7.2 当前状态模型

```ts

  languageCode: string;
```

### 7.3 当前交互规则

- 用户输入语言名：
  - 立即搜索候选。
  - 若唯一高置信命中，自动填入 ISO 代码。
  - 若多个候选，弹出候选列表，待用户确认。
- 用户输入代码：
  - 若是 `eng` / `cmn` 这类 ISO 639-3，直接回填语言名。
  - 若是 `en` / `zh` 这类两字符纯字母代码，在输入过程中先保留为草稿，不立即自动推断语言。
  - 若失焦后仍是可识别的 ISO 639-1，则再规范到对应 ISO 639-3，并回填语言名与提示。
  - 若是 `zh-Hant-HK` 这类 BCP 47，解析 primary/script/region，并回填到高级字段。
- 用户手动修改后，不应强制覆盖。
  - 一旦进入手动模式，除非再次点选候选，否则保留用户编辑值。
- 用户从已规范化值继续删除时，应允许删除过程停留在两字符草稿，不得强制回弹到上一规范值。

## 8. 方言、变体、重名的处理策略

### 8.1 方言/宏语言

建议优先遵循“能具体就具体”：

- 普通话 -> `cmn`
- 粤语 -> `yue`
- 吴语 -> `wuu`
- 闽南语 -> `nan`
- 客家话 -> `hak`

如果用户输入宏语言：

- 中文 / Chinese -> 不直接静默落为 `zho`
- 阿拉伯语 / Arabic -> 可先给 `ara`，但若存在更明确上下文可提示细分

### 8.2 脚本/地区/正字法变体

不要把这些信息塞进 `languageId`。

建议拆分：

- `languageId`: 语言实体
- `scriptTag`: 书写系统，如 `Latn` / `Arab` / `Hans` / `Hant`
- `regionTag`: 地区，如 `CN` / `HK` / `TW`
- `variantTag`: 变体，如 `fonipa`、年份拼写变体等
- `localeTag`: 完整 locale 或 BCP 47 兼容标识，用于展示或互操作

### 8.3 重名与歧义

目录项展示格式建议统一：

- `普通话 · cmn · individual`
- `中文（宏语言）· zho · macrolanguage`
- `粤语 · yue · Chinese macrolanguage`
- `Belarusian Taraskievica · be + variant tarask`

对于“同名不同义”或“旧名/弃用名”，显示附加说明：

- `Deprecated: mo -> ro`
- `Alias of Cantonese`
- `Macrolanguage of Chinese`

## 12. 风险与边界

### 12.1 不建议现在就做的事

- 不建议把所有 `languageId` 字段升级成任意 BCP 47 标签。
- 不建议让每个表单各自读 CLDR 原始数据。
- 不建议继续扩充 `COMMON_LANGUAGES` 来冒充通用语言目录。

### 12.2 需要提前确认的边界

- 对“无官方 ISO 639-3 的方言”是否允许保存？
  - 建议第一阶段仍不放开，先要求用户选择最接近的官方代码，并在 notes/locale 中保留补充信息。
- 是否允许直接输入 private-use code？
  - 当前不建议，除非后续明确扩展互操作合同。
- 是否允许宏语言直接保存？
  - 允许，但应在 UI 中提示“可选更具体语言”。

## 13. 最终建议

建议按下面顺序推进：

1. 先做统一语言目录与解析层，不先改 UI。
2. 继续沿用共享 `LanguageIsoInput`，并把解析/候选/歧义能力进一步从组件内抽到统一服务层。
3. 保持现有落库仍是 ISO 639-3，方言细节通过 script/region/variant/locale 表达。
4. 用 IANA/BCP 47 规则补齐规范化和歧义处理，不再依赖单个 `resolveLanguageQuery(...)`。
5. 只在构建期引入更丰富的显示名数据，避免运行时包体和多入口重复逻辑膨胀。

这个方向和当前仓库的正字法、互操作、导入导出架构是兼容的，且不会把复杂度再次散落到多个弹窗里。
