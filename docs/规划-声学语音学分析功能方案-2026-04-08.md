## 1. 背景

Jieyu 当前已经具备较完整的转写工作台能力，包括：

1. 波形与频谱显示
2. 语段切分与局部播放
3. 低置信、重叠、停顿等风险叠加
4. AI prompt context、observer recommendation 与语音智能体 grounding

但在声学语音学分析层面仍有明显空白：

1. 没有 F0 pitch 追踪
2. 没有 intensity 曲线
3. 频谱图只能渲染，不能读取
4. 没有频谱切片、归一化声调图、Formant、元音图
5. AI 不能直接消费声学摘要并给出解释或导航建议

这导致用户在进入更细粒度的语音分析时，仍需要频繁切换 Praat、Audacity、ELAN 或其他外部工具。

本方案目标是在不破坏现有转写主工作流的前提下，把 Jieyu 从“可转写、可审校”推进到“可观察、可解释、可导航”的声学分析工作台。

## 2. 目标

### 2.1 产品目标

1. 在现有波形区内提供最常用的声学观察能力。
2. 在不新增第四种主显示模式的前提下，扩展声学增强层。
3. 为科研级分析和后续导出能力预留稳定接口。
4. 让 AI 与语音智能体具备基于声学现象的解释与定位能力。

### 2.2 体验目标

1. 不破坏现有切分、备注、播放、拖动、套索等高频操作。
2. 不把主波形区做成堆满控件和标签的仪表盘。
3. 重分析能力移到次级分析区，而不是强行堆叠在主时间轴。

### 2.3 工程目标

1. 主编辑层保持编排简单，复杂分析逻辑下沉到 service 和 bridge/controller。
2. 分析结果以结构化对象形式输出，可供 UI、AI、导出与后续扩展复用。
3. 允许分阶段上线，先做轻量只读能力，再做深度分析能力。

## 3. 总原则

本方案遵循三条总原则：

1. 主波形区只做最小增强，维持现有编辑节奏。
2. 重分析功能搬到次级分析区，不把主时间轴做成仪表盘。
3. AI 只消费结构化摘要，不直接消费原始逐帧数据，也不直接写主标注层。

## 6. 主波形区叠加规范

### 6.1 分层模型

新增功能采用固定五层模型：

1. L0：波形 / 频谱底图
2. L1：现有风险 band
3. L2：新增声学叠加层，例如 F0、Intensity
4. L3：lasso、snap、note、readout
5. L4：RegionActionOverlay、popover、上下文菜单

新增声学层只能位于 L1 与 L3 之间，且不承担主要交互命中。

实现约束：

1. L2 优先作为单独声学 overlay surface 存在，内部再区分 F0 与 Intensity 的绘制顺序。
2. 不为 F0 与 Intensity 额外扩张全局 z-index 体系，避免影响现有风险 band、note 与 region action 的层级关系。
3. 若采用 DOM + SVG 或 Canvas 方案，均应保证声学层整体只占用一层全局叠加位。

### 6.2 避让原则

必须先定义禁区，再渲染：

1. 底边安全区避让 RegionActionOverlay。
2. utterance 尾部安全区避让 note indicator。
3. lasso 或 snap 激活时，声学层进入静默态或弱化态。
4. split 模式下，声学层只存在于上半 waveform primary stage。
5. 不改变现有 video top、left、right 布局的几何结构。

挂载约束：

1. 主声学 overlay 挂载在 waveform primary stage 内，而不是挂到整个 waveform shell 或 spectrogram 容器。
2. split 模式下通过主波形区高度裁剪与几何约束控制声学层，不把声学层直接挂入下半频谱容器。
3. 频谱交互 overlay 单独挂载到 spectrogram 容器，不与主声学 overlay 共用命中区域。

### 6.3 降级原则

当空间、交互或性能冲突出现时，按固定顺序降级：

1. 隐藏 intensity
2. 仅保留 F0
3. 关闭 hover readout
4. 关闭 acoustic overlay

性能降级顺序：

1. 逐帧折线改为像素列聚合
2. 降采样密度
3. 极端缩放下暂停局部细节绘制

AI 降级原则：当分析结果为 `missing`、`unstable` 或 `unreliable` 时，只输出不确定性摘要与复核建议，不输出伪确定结论。

### 6.4 表达原则

1. Phase 1 不做常驻标签。
2. 不做峰值 chip。
3. 不做沿线文字说明。
4. 所有数值优先通过单点 readout 呈现。
5. F0 使用青蓝系，Intensity 使用青绿系，避开现有风险色。
6. Intensity 如果带填充，只做低透明度辅助，不盖过原始波形。

## 7. Phase 1：主波形区最小增强

### 7.1 Phase 1 功能边界

主波形区 Phase 1 只增加：

1. `无 / F0 / Intensity / F0+Intensity` 模式切换
2. F0 连续曲线
3. Intensity 轻量包络或细线
4. 统一 hover readout

主波形区暂不增加：

1. 峰值徽章
2. 常驻标签
3. 局部 chip
4. 复杂参数配置
5. Formant 与元音图

### 7.2 数据结构

建议新增声学分析数据结构，至少包括：

1. `AcousticOverlayMode`
2. `AcousticFrame`
3. `AcousticFeatureResult`
4. `AcousticHotspot`
5. `AcousticAnalysisSummary`
6. `AcousticAnalysisConfig`

建议最小字段定义：

```ts
interface AcousticFrame {
	timeSec: number;
	f0Hz: number | null;
	intensityDb: number;
	reliability: number;
}
```

说明：

1. `f0Hz = null` 表示无声或不可稳定估计。
2. `intensityDb` 作为主波形区与 AI 摘要共用的基础能量指标。
3. `reliability` 用于驱动 UI 降级、AI 不确定性表达与 hotspot 过滤。

### 7.3 PCM 获取路径与算法选型

Phase 1 必须先明确音频数据来源与 F0 算法，不把关键实现决策留到编码阶段。

PCM 获取路径分两层：

1. MVP 路径：主线程从现有 WaveSurfer 实例读取已解码音频缓冲区，供分析服务或 worker 消费。
2. 解耦路径：worker 独立拉取媒体并解码，作为后续大音频增量分析与缓存体系的正式形态。

Phase 1 建议执行口径：

1. 1a 阶段允许优先复用现有 WaveSurfer 已解码缓冲区，尽快打通数据层。
2. 1b/1c 阶段开始把分析入口收敛到独立 `AcousticAnalysisService`，逐步切到 worker 自主获取 PCM 的路径。
3. 若走 MVP 路径，则只能在 WaveSurfer `ready` 后、且已解码 PCM 缓冲区实际可用时启动分析；若当前实例尚未 ready 或缓冲区仍为空，则必须延迟分析或返回 `missing` 状态，不允许假定数据必然存在。

算法选型口径：

1. F0 优先采用 YIN 或兼容 pYIN 的路线。
2. 不采用 zero-crossing rate 作为正式 F0 方案。
3. 不采用单纯 FFT peak picking 作为正式浊音基频方案。
4. Phase 1 不做 Formant，但数据结构与服务边界需允许后续接入 LPC 或 WASM 路线。

### 7.4 开源库与技术路线优化

结合现有浏览器端声学分析实践与开源库能力，建议把技术路线进一步收敛为“默认轻量 DSP + 可选重型 backend”的双层策略，而不是一开始就引入单一重库。

摘要：Phase 1 默认采用基于 YIN 原论文的自实现 + RMS/dB，不引入 GPL / AGPL 重依赖。Meyda 采用 MIT 许可证，可作为 Phase 1c 的谱特征补充候选。Essentia 采用 AGPL 路线、CREPE 家族需要额外模型与推理成本，均放到 Phase 2 / 3 再评估。aubio 只作为算法口径参考，不纳入前端正式依赖。

#### 7.4.1 默认基线方案

Phase 1 默认建议：

1. F0 采用以 YIN 原论文为依据的自实现路线；若后续需要 `yinfast` 类优化，也应基于公开算法描述独立实现，而不是直接移植 GPL 代码。
2. Intensity 采用 RMS / dB 计算，不依赖外部大库。
3. reliability 直接复用 YIN 相关置信度或非周期性指标。
4. worker 内优先做纯 DSP 计算，避免首批引入大模型与大体积 WASM。

这样做的优点：

1. 浏览器端可控，首批包体压力最小。
2. 更适合当前语音转写工作台的交互节奏。
3. 便于先把缓存、overlay、AI summary 打通。

#### 7.4.2 借鉴 aubio，但不直接采用 aubio 作为前端依赖

`aubio` 在 pitch tracking、onset detection 与 `yinfast` 方面很值得借鉴，尤其适合作为 Phase 1 的算法参考来源。

但方案上不建议把 aubio 直接作为前端正式依赖，原因包括：

1. 许可证为 GPL，不适合作为当前前端正式依赖默认接入。
2. 工程形态更偏原生 C / Python 生态，而非当前 Vite + React + Worker 的浏览器前端主路线。
3. 当前阶段需要的是可渐进式接入的浏览器端分析底座，而不是一次性引入跨编译与许可证负担。

优化建议：

1. Phase 1 的 F0 实现以 YIN 原论文与公开算法描述为依据，不对照 aubio GPL 源码做移植。
2. aubio 中与 pitch floor、pitch ceiling、threshold 相关的参数口径可作为调参参考，但不把 aubio 本体写进 Phase 1 正式依赖列表。

#### 7.4.3 Meyda 适合作为谱特征补充层，不适合作为主 F0 方案

`Meyda` 适合浏览器端 JavaScript 音频特征提取，适用范围包括：

1. RMS
2. spectral centroid
3. spectral rolloff
4. spectral flatness
5. MFCC

因此它适合：

1. Phase 1c 的频谱 hover readout 补充指标
2. Phase 2 的 spectral inspector 指标层

但不建议把 Meyda 作为主 F0 方案，因为：

1. 它不是以高鲁棒语音基频跟踪为核心设计目标。
2. 它更适合特征描述，而不是科研级主基频追踪。
3. Meyda 采用 MIT 许可证，可作为低风险可选依赖看待，但许可证上的低风险不改变其在主 F0 任务上的能力边界。

优化口径：

1. F0 默认仍走 YIN / pYIN 路线。
2. Meyda 可作为谱特征与 readout 增强的可选依赖。

#### 7.4.4 Essentia.js 更适合 Phase 2 / 3，而不是 Phase 1 起步依赖

`Essentia` 具备非常丰富的频谱、音高、loudness 与描述子能力，并且有 Web 绑定，这使它非常适合：

1. 频谱切片 inspector
2. 更丰富的 spectral descriptors
3. loudness / spectral shape / melody extraction 的后续增强

但当前阶段不建议把它直接纳入 Phase 1 的默认基线，原因包括：

1. 能力过重，容易把首批目标从“打通主路径”拉向“搭建完整音频分析平台”。
2. 更多偏音乐信息检索与通用音频分析，需筛选哪些能力真正服务于语音学工作台。
3. Essentia 核心许可证为 AGPL-3.0，在正式前端依赖中引入前需单独评估合规性。
4. 其更适合作为 Phase 2 / 3 的 inspector backend，而不是首批 F0 / intensity 的最小依赖。

优化口径：

1. Phase 1 不强依赖 Essentia。
2. Phase 2 起评估其作为 spectral / loudness descriptor backend 的可行性。

#### 7.4.5 CREPE 家族不应直接照搬，但可以作为可选神经音高 backend 预留

`CREPE` 的思路值得参考，优点是：

1. 对复杂、含噪或非理想条件下的单音高跟踪更鲁棒。
2. 有时间戳、voicing confidence、平滑等成熟概念可借鉴。

但当前方案不建议把原始 CREPE 项目直接纳入前端实现，原因包括：

1. 公开实现主要是 Python / TensorFlow 工作流，不是现成浏览器前端依赖。
2. 当前仓库虽已有 `onnxruntime-web`，但直接把 CREPE 路线转成前端默认方案会显著提高模型、推理与缓存复杂度。
3. 首批目标仍是工作台主路径，而不是先上神经网络 F0。

更稳妥的优化方式：

1. Phase 1 默认仍走 YIN。
2. Phase 2 或 Phase 3 可在已有 `onnxruntime-web` 基础上预留轻量 ONNX pitch backend。
3. 神经 backend 作为困难音频、噪声环境或评估模式下的可选增强，而不是默认唯一路线。

#### 7.4.6 Praat / Parselmouth 更适合作为互操作与算法对标对象

Praat 及其生态不是当前前端直接可用依赖，但在以下方面应作为对标对象：

1. Pitch / Intensity / Formant 对象化结果结构
2. 查询、可视化与导出语义
3. PitchTier / IntensityTier 等互操作格式

优化建议：

1. 把 Praat 作为结果结构与导出格式的参考标准。
2. 不在 Phase 1 中把 Praat 路线作为前端运行时依赖。

#### 7.4.7 推荐技术栈收口

综合建议如下：

1. Phase 1a：基于 YIN 原论文的自实现 + RMS/dB，worker 化。
2. Phase 1b：主波形区 F0 overlay 与角落 readout，不新增重依赖。
3. Phase 1c：可选引入 Meyda 作为谱特征补充；Meyda 为 MIT 许可证，但不替代主 F0 路线。
4. Phase 2：评估 Essentia.js 作为 spectral inspector / loudness descriptor backend，并单独完成 AGPL 合规性评估。
5. Phase 2 / 3：基于现有 `onnxruntime-web` 预留可选神经 pitch backend，而非默认依赖。
6. Phase 3：以 Praat 互操作格式为导出目标，而不是在前端直接复制 Praat 全量运行时。

#### 7.4.8 本地优先取舍

若后续产品路线进一步明确为本地优先，则技术选型应继续向“浏览器内可运行、可离线、可缓存、可 worker 化、许可证宽松”的方向收口，而不是把核心能力建立在重型依赖或远程分析服务之上。

本地优先下的默认建议：

1. 核心 F0 与 intensity 仍采用基于 YIN 原论文的自实现 + RMS/dB。
2. 结果计算默认放入 worker，避免主线程承担逐帧分析负担。
3. 结果缓存默认优先走内存 + IndexedDB 路线，允许离线复用和配置回退。
4. [src/hooks/useWaveSurfer.ts](../src/hooks/useWaveSurfer.ts) 继续只承担显示、时序与交互壳职责，不把复杂声学分析直接塞入 WaveSurfer 生命周期。

本地优先下可接受的可选增强：

1. Meyda 可作为频谱 hover readout 与 descriptor 补充层，因为其浏览器适配与许可证条件更适合本地优先路线。
2. 当前仓库已有 `onnxruntime-web`，后续可为困难音频或高级模式预留可选本地神经 pitch backend。
3. 本地模型推理必须作为显式开启的增强模式，而不是默认分析路径，避免模型下载、冷启动与缓存体积挤压主路径体验。

本地优先下不建议作为默认依赖的路线：

1. 不把 Essentia.js 作为默认基础设施，除许可证外，其包体、能力范围与初始化成本都偏重。
2. 不把 aubio 作为前端正式依赖，只保留算法口径参考用途。
3. 不把原始 CREPE 工作流直接搬入前端默认分析路径。
4. 不把核心声学分析建立在远程 API 或联网推理前提上，以免破坏离线可用性与本地缓存收益。

因此，本地优先下的推荐收口是：

1. 默认层：自实现 YIN + RMS/dB + worker + IndexedDB 缓存。
2. 轻量增强层：Meyda 一类的本地特征库，仅补谱特征与 readout。
3. 高级增强层：基于 `onnxruntime-web` 的可选本地模型推理，仅用于困难音频或实验模式。
4. 后续评估层：Essentia.js、Praat 互操作与导出能力，放到 Phase 2 / 3 再进入正式选型。

### 7.5 缓存与失效策略

声学分析结果不应只停留在内存态，应明确缓存键与失效规则。

建议：

1. 缓存键为 `mediaId + analysisConfigHash`。
2. `analysisConfigHash` 至少覆盖 pitch floor、pitch ceiling、frame step、smoothing、scale mode 等会影响结果的参数。
3. 参数变更后旧缓存标记为 stale，但不立即清除，允许用户回退配置时直接复用。
4. Phase 1a 可以先做内存缓存，Phase 1c 起转向 IndexedDB 持久缓存。
5. Phase 1a 即应给出总量控制，例如基于结果帧数或估算字节数设置单媒体上限与全局上限，避免长音频分析把页面内存持续顶高。
6. 当缓存触达上限时，优先淘汰最久未访问且非当前媒体的结果；stale 结果优先于 fresh 结果被清理。
7. 若路线明确为本地优先，则持久缓存应优先服务于离线复用，而不是仅作为性能优化附属能力；缓存键设计需允许本地模型版本、算法版本和配置版本独立演进。

## 8. 频谱图交互化改进

### 8.1 当前问题

当前频谱图存在两个关键限制：

1. `.transcription-wave-spectrogram` 当前为只读渲染层，默认不接管交互。
2. [src/hooks/useWaveSurfer.ts](../src/hooks/useWaveSurfer.ts) 中的 spectrogram plugin 主要负责渲染与滚动同步，不适合直接扩展成复杂交互编辑器。

### 8.2 方案方向

终版方案不让 plugin canvas 直接承担交互，而是：

1. 保留 plugin canvas 作为纯渲染层
2. 在 spectrogram 容器上方新增透明交互 overlay
3. 单独管理 cursor、selection、readout 与 click 行为

### 8.3 Phase 1 频谱交互能力

Phase 1 建议先支持：

1. hover readout：时间、频率、近似能量等级
2. click-to-seek：点击频谱跳到对应时间点
3. 固定 readout 区：显示在角落，不跟随鼠标漂浮
4. split 模式下仅下半区启用频谱交互

实现约束：

1. 当前频谱图采用 mel scale 时，`y -> frequency` 的 readout 映射必须与 spectrogram 渲染配置保持一致。
2. click-to-seek 的时间计算以当前滚动位置与可视宽度为准，不直接依赖插件内部 canvas 坐标系。
3. hover readout 先读取缓存帧或近邻统计结果，不走 worker 往返，避免交互延迟。
4. 若 Phase 1c 需要补充 centroid、rolloff、flatness 等谱读数，可优先评估 Meyda，而不是提前引入更重的 descriptor 栈。

### 8.4 后续预留

后续可扩展：

1. `spectrogramInteractionMode = 'read' | 'inspect' | 'select-frequency'`
2. `spectrogramCursor`
3. `spectrogramSelection`
4. vertical zoom
5. linear、log、mel、bark、erb 视图
6. spectral selection
7. Formant 联动

## 10. AI 与语音智能体接入

### 10.1 接入原则

AI 不直接消费逐帧原始数据，而消费结构化摘要，包括：

1. 当前选区 summary
2. hotspots 列表
3. reliability 与异常标记
4. 少量可解释的统计量

### 10.2 建议接入点

可在以下位置扩展：

1. [src/pages/TranscriptionPage.aiPromptContext.ts](../src/pages/TranscriptionPage.aiPromptContext.ts)
2. [src/ai/chat/promptContext.ts](../src/ai/chat/promptContext.ts)
3. [src/services/VoiceAgentService.ts](../src/services/VoiceAgentService.ts)

接入方式应与现有 `waveformAnalysis` 并列，而不是直接混成同一层级的风险对象。

建议补充结构化序列化格式：

```txt
acousticSummary(
	selectionSec=1.20-3.40,
	f0Min=120,
	f0Max=280,
	f0Mean=195,
	contourType=rising,
	intensityPeak=-12.0dB,
	reliability=0.82,
	hotspots=pitch_break@2.10s|intensity_peak@2.80s
)
```

要求：

1. 保持与现有 `waveformAnalysis(...)` 类似的紧凑序列化风格。
2. 不把逐帧数组直接塞进 prompt context。
3. 当结果不稳定时优先输出 `reliability` 与 `unstableReasons`，而非伪精确统计值。

对语音智能体的动作预留：

1. `navigate_to_hotspot(hotspotType, index)`
2. `select_acoustic_range(startSec, endSec)`

### 10.3 AI 可回答的问题

声学摘要接入后，AI 可以回答：

1. 当前片段是否存在明显升降调趋势
2. 哪些区间 pitch break 或 intensity peak 明显
3. 哪些区间不稳定，建议复核
4. 哪些热点值得跳转查看

但 AI 不应直接输出未经验证的音系结论，例如自动断定音类、声调范畴或共振峰类别。

### 11.2 Phase 1b：主波形区只读增强

2. 工具栏入口
3. F0 overlay

### 11.3 Phase 1c：频谱交互与缓存补齐

1. Intensity overlay
2. 频谱只读交互 overlay
3. click-to-seek
4. 更完整的缓存复用
5. 交互回归测试

### 11.4 Phase 2：运行时底座收口

调研并入后的 Phase 2 不再抽象写成“继续做 inspector”，而是明确收口为浏览器本地优先的运行时底座阶段。

继续交付：

1. 统一前端语音运行时入口，继续围绕 `onnxruntime-web` 收口模型加载、缓存、Worker、取消、进度与设备回退策略。
2. 接入 `silero-vad` 或同等 ONNX 方案，形成自动分段、静音检测、chunk 合并与时间轴标注稳定层。
3. 保持当前 acoustic tab 为主要载体，不新增第四种主显示模式，也不把重型 provider 直接塞进默认浏览器主链。
4. 把对齐、diarization、增强等后续能力明确收口到 provider-neutral 接口，而不是在每个功能点各自直连某个库。
5. 在 UI 中逐步补足“算法来源、窗口参数、采样率、平滑/聚合策略”等可解释字段，为后续科研口径校准做准备。

本阶段默认采用：

1. 运行时主线：`onnxruntime-web`。
2. 默认 VAD：`silero-vad`。
3. 结果表达：结构化摘要 + 选区 detail + 导出，不直接暴露全媒体大数组到面板上下文。

本阶段不做：

1. 不默认引入 WhisperX、MFA、pyannote、sherpa-onnx 等重型 provider。
2. 不把强制对齐包装成近期默认稳定能力。
3. 不默认对原始科研音频做不可逆增强。

### 11.5 Phase 3：谱描述与科研口径补齐

调研并入后的 Phase 3 明确拆成“浏览器内可运行的谱描述层”和“科研标尺校准层”，而不是直接跳到完整科研工作台幻想。

再扩展：

1. 引入 `Meyda` 或等价轻量 JS 特征提取层，补齐 spectral centroid、rolloff、flatness、zcr、mfcc、loudness 等 descriptor。
2. 在现有 acoustic tab 内继续扩展 spectral inspector，使 slice、tone、descriptor、导出形成同一工作区。
3. 建立 `Praat / Parselmouth` 校准脚本与固定样本集，把 pitch、intensity、后续 formant 结果对齐到科研口径基线。
4. 在自研 spectral backend 中逐步落地 LPC / Burg 路线的 Formant 候选提取、平滑与可靠度过滤。
5. 在通过校准前，不把 Formant 与元音图包装成确定性结论；在通过校准后，再开放 vowel space、Praat 互操作导出与更丰富科研参数。

本阶段默认采用：

1. 浏览器谱描述：`Meyda` 一类轻量本地特征库，仅补 descriptor 与 readout，不替代主 F0 路线。
2. 科研标尺：`Praat / Parselmouth`，用于离线校准和结果语义对齐。
3. Formant 主线：自研或受控移植的 LPC / Burg spectral backend。

本阶段不做：

1. 不把 Essentia.js 直接作为默认基础设施。
2. 不把神经 pitch 或大型多语言模型直接变成默认分析路径。
3. 不在未校准前承诺跨语言稳定的元音图与共振峰分析质量。

### 11.6 Phase 4：可插拔增强与研究 provider

在默认本地浏览器链路稳定后，再进入 provider 扩展阶段：

1. `Whisper` 小模型 provider：提供本地可选离线转写与粗时间戳。
2. `WhisperX` provider：用于特定语言条件下的词级时间戳增强。
3. `MFA` provider：用于具备词典/G2P/声学资源项目的离线批对齐。
4. `pyannote-audio` 或 `sherpa-onnx` provider：用于多说话人材料的 diarization 预研。
5. `MMS` 或其他 LID/provider：用于语言路由和低资源实验，不进入默认浏览器主链。

进入条件：

1. Phase 2 的 VAD、chunking、缓存与运行时收口已稳定。
2. Phase 3 的谱描述层与科研口径校准已建立基本闭环。
3. UI 已明确区分“默认本地能力”和“外部/增强 provider 能力”。

## 11.7 调研并入后的下一步行动方案

按当前仓库状态，下一步不应直接去做完整 Formant UI，而应按以下顺序推进：

1. 第一优先级：接入 `silero-vad`，形成浏览器端稳定切段、静音检测与 chunk 管线，并把结果接入现有时间轴与缓存链路。
2. 第二优先级：引入 `Meyda` 或等价方案，补齐当前 acoustic tab 缺失的谱描述读数与导出字段。
3. 第三优先级：整理统一的前端模型运行层，围绕 `onnxruntime-web` 统一加载、缓存、取消、进度和 provider 接口。
4. 第四优先级：建立 `Praat / Parselmouth` 校准脚本与小样本基线，为后续 Formant / vowel space 实现设定可验证目标。
5. 第五优先级：在校准基线存在后，再开启自研 spectral backend（LPC / Burg）与 Formant / vowel space 的第一版实现。

对应当前阶段的明确取舍：

1. 先补“底座 + descriptor + 校准”，后补“Formant / 元音图”。
2. 先把默认本地能力做稳，后接外部 provider。
3. 先做可解释与可审计，再做更重的自动化能力。

## 12. 风险与控制

### 12.1 交互风险

风险在于新层和现有 lasso、note、region actions 冲突。

控制方式：

1. 严格限制新增层的 z-index
2. 在关键交互激活时让声学层静默
3. 明确底边与尾部安全区

### 12.2 性能风险

风险在于逐帧绘制与大音频缩放下的渲染压力。

控制方式：

1. worker 化分析
2. 结果缓存
3. 降采样与像素列聚合
4. 极端缩放下局部细节退化

补充约束：

1. 大音频不默认整段一次性分析，优先按可视区间或可视区间加缓冲区做增量分析。
2. hover readout 响应应直接读取本地缓存结果，避免 worker 往返造成抖动。
3. overlay 重绘优先挂在现有 waveform redraw / scroll 同步链路上，不独立开启长期自驱动动画循环。
4. Phase 1 最小预算目标建议为：10 秒单声道语音在常见桌面环境下首次分析冷启动不高于 800ms、同配置热复用不高于 150ms、hover readout 单次响应不高于 16ms、单次可视区 overlay 重绘不高于 32ms。
5. 不在 Phase 1 默认引入模型推理型 pitch backend，避免额外模型下载、冷启动和缓存压力。
6. 本地优先路线下，任何新增高级 backend 都必须单独评估离线包体、首次下载体积、IndexedDB 占用和冷启动开销，不能仅以算法效果决定是否接入。

### 12.3 解释风险

风险在于 AI 或 UI 把不稳定数据表达成过度确定的结论。

控制方式：

1. 引入 reliability 字段
2. 用 `missing`、`unstable`、`unreliable` 明确状态
3. 不把推断性结论直接写入主标注层

## 13. 验收标准

文档层面，本方案对应的实现验收建议包括：

1. 不新增第四种主显示模式。
2. 主波形区新增能力不破坏原有切分、播放、备注与拖动行为。
3. split 模式下波形与频谱布局稳定，且声学层只出现在上半主波形区。
4. 频谱图可 hover 读取时间和频率，并支持 click-to-seek。
5. 新声学能力能以结构化摘要形式提供给 AI。
6. 当分析结果不稳定时，UI 与 AI 都能正确降级。

### 13.1 自动化测试建议

除手工验收外，建议同步补齐：

1. `AcousticAnalysisService` 单元测试，例如在 80Hz、120Hz、200Hz、300Hz 的纯正弦波输入下，F0 估算中位绝对误差不高于 3Hz 或 2%，两者取较宽者作为阈值。
2. 声学 overlay 渲染或交互测试，例如 hover 后 readout 出现时间与频率字段。
3. AI prompt context 测试，确保 `acousticSummary(...)` 正确注入且不携带逐帧数组。
4. 原有波形交互回归测试，确保切分、播放、lasso、note icon 不被破坏。

### 13.2 非 Phase 1 阻塞项

以下议题建议纳入后续阶段，而不作为 Phase 1 阻塞条件：

1. 归一化声调图的完整图表设计
2. PitchTier / CSV 等导出互操作
3. Canvas 层的无障碍补充表达
4. 更细粒度的科研参数面板

## 14. 后续补充议题

后续建议继续跟进但不应抢占当前实现主线：

1. i18n 命名空间收口，例如统一的 `acoustic.*` 词条。
2. 导出格式优先级，建议先 CSV 后 PitchTier。
3. 无障碍策略，例如 readout 的 `aria-live` 与 canvas 的静态描述。
4. 归一化声调图与多语段比较视图的详细产品定义。
5. Essentia.js、可选 ONNX pitch backend 与 Praat 互操作格式的后续评估。

## 15. 结论

Jieyu 的声学语音学能力不应通过堆叠更多主视图模式来实现，而应采用“主编辑层最小增强 + 次级分析层承载重分析 + AI 消费结构化摘要”的方式逐步落地。

这条路线既能保住当前转写工作台的操作节奏，也为后续更成熟的科研级分析能力留下清晰扩展位。
