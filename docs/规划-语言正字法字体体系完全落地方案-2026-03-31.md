# 规划：语言-正字法-字体体系完全落地方案

> 2026-03-31 · 基于当前工作区实际代码状态的剩余盘点与完整实施方案

---

## 一、当前结论

以当前工作区为基线，语言-正字法-字体体系已经完成了第一阶段闭环，且第二阶段的一部分关键骨架也已经落地。

已经确认完成的部分：

- `OrthographyDocType` 已扩展到可承载脚本、方向、示例字符、字体偏好、bidi 策略等元数据
- 项目可写入 `primaryOrthographyId`，层可写入 `orthographyId`
- 已存在统一运行时出口 `resolveOrthographyRenderPolicy(...)`
- 时间轴主渲染、纯文本时间轴、层样式菜单、正字法创建器预览，已开始消费 render policy
- `OrthographyTransformDocType`、transform 预览、样例校验、可逆性标记、创建时校验已经落地
- 字体推荐、脚本过滤、轻量 coverage summary、风险提示、两段式创建确认已经落地

当前不应再把这项工作定义为“未开始”或“只有设计”。更准确的状态是：

> 核心数据模型、创建流程、运行时渲染入口、transform 管理面、互操作 identity、导入期 transform、导出 legacy fallback transform、AI 写回 transform、assistant 语音听写 fallback 写回 transform、字体风险提示、字体验证缓存与复杂脚本 stricter fallback 已经落地；
> 搜索/复制/只读片段主链路的 runtime contract、citation plain-text copy/search 合同，以及 EAF/TextGrid 的 interop downgrade 基础规则也已经完成收口；
> 当前真正剩余的重点已进一步收敛到 transform 在批量迁移/派生复制等更广业务面的统一复用，以及复杂脚本 shaping 风险门禁细化，因此暂时仍不能定义为“平台级完全落地”。

## 三、已确认的当前剩余断点

下面这些断点不是抽象风险，而是当前实现收口后仍然存在的真实剩余项。

### 3.1 搜索 / 复制 / 只读片段的 runtime contract 已基本统一

当前主时间轴、Builder 预览、搜索命中片段、边缘 overlay、批量预览、语音听写只读片段、citation snippet plain-text copy/search 已统一到同一组 preview helper 与 plain-text normalize helper。

当前残余风险：

- 仍可继续抽查更少见的非主路径文本出口，但主链路已不再存在明显合同裂缝

### 3.2 interop downgrade / compatibility contract 已完成基础编码

当前 EAF / TextGrid 身份保真路径已经建立，tier metadata 已补齐 `scriptTag / regionTag / variantTag` 冗余快照；导入端对未知 metadata 字段采取稳定忽略降级策略，并已有 round-trip / downgrade 回归测试锁定。

当前剩余项：

- provider-neutral external mapping contract 仍缺少统一扩展位与兼容性说明

### 3.3 transform 还没有覆盖更广的业务复用面

当前 transform 已经进入导入、导出 legacy fallback、AI 写回、assistant 语音听写 fallback 写回等真实链路，但批量迁移、派生复制后的批处理、批量转换治理仍未统一复用同一 runtime helper 与冲突反馈策略。

直接后果：

- transform 已经不再是“只会 preview”，但仍未做到“所有关键文本迁移路径统一合同”
- 一旦后续再新增迁移/复制入口，仍有回退到局部直写逻辑的风险

### 3.4 字体验证仍以 glyph presence 为主，风险门禁还不够细

当前已经具备 browser `document.fonts` 探测、localStorage coverage cache、本地字体脚本过滤、已验证/缺字标签与 strict-script runtime fallback；
但 shaping 风险、复杂脚本细粒度风险说明、以及更强的验收门禁仍未完全建立。

## 四、完全落地定义

当且仅当以下条件同时满足，才定义为“语言-正字法-字体体系完全落地”：

1. 所有主要文本渲染路径都统一消费 render policy，而不是只在时间轴主视图中生效。
2. 搜索、预览、复制、导出四条链路对 RTL / 混排的行为一致。
3. transform 不只可创建，还能被导入、导出、AI 等真实业务路径调用。
4. 字体层不只提供推荐，还能给出可追踪的验证、风险缓存与严格 fallback。
5. EAF / TextGrid / 内部导出至少有一条稳定的 orthography identity 保留路径。
6. 对未知 orthography / script / variant / transform 的降级策略被明确编码并被测试锁定。

- 采用受控 sidecar 或 tier-name suffix 方案
- 明确“可回放哪些字段、丢弃哪些字段”

#### 重点文件

- 相关导入导出测试文件

#### 验收标准

- 未识别 orthography 时有稳定降级而不崩溃
- EAF / TextGrid round-trip 兼容性说明与测试保持一致

## 六、建议执行顺序

建议采用以下严格顺序，而不是并行散改：

1. 批次 1：搜索 / 复制 / 只读片段 runtime contract 与 bidi 一致性已完成
2. 批次 2：interop downgrade 规则与相关 round-trip 测试已完成基础收口，后续转为 compatibility note / provider-neutral 扩展说明
3. 批次 3：把 transform 继续推进到派生复制、批量迁移、批量转换等真实消费
4. 批次 4：继续细化复杂脚本 shaping 风险分级与字体门禁
5. 批次 5：统一发布门禁、文档与手工验收

这样排的原因：

- export / AI / strict fallback 这一轮已经补完，继续优先做这些方向收益会快速递减
- 先收边缘 runtime contract，再补 interop 说明，能最快降低“看起来已完成但跨边界仍失真”的风险
- transform 扩面与字体细化都应建立在更稳定的兼容合同之上

## 八、本轮之后的明确判断

当前最合理的下一步不是继续补 export / AI / strict fallback 这类基础能力，而是：

> 搜索、复制、边缘只读片段与 interop downgrade 基础规则已经完成收口；
> 下一步应转向 transform 扩面与复杂脚本字体风险门禁细化。

原因很直接：

- Builder、时间轴、层样式菜单、导出 legacy fallback、AI 写回、strict fallback、搜索 / 复制 / citation / interop downgrade 这些主干与跨边界链路已经有了；
- 真正还能继续破坏一致性的，已经缩小到 transform 在更广业务入口的复用与复杂脚本 shaping 风险门禁。

如果这些路径不补齐，现在的体系仍会停留在：

> “主流程已完整，核心运行时与主要跨边界合同已统一，但 transform 扩面与复杂脚本风险门禁仍待继续收口。”

这正是当前剩余工作的核心。

如果从今天继续开发，建议直接从以下子任务开始：
