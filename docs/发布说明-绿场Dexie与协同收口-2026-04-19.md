# 发布说明 — 绿场 Dexie 与协同收口（2026-04-19）

面向 **无存量本地库、不要求旧版整库 JSON 兼容** 的收口批次。详细决策见 [ADR 0008](./adr/0008-greenfield-indexeddb-export-collab-surface.md) 与执行计划 [无历史数据 — 本地库与协同单轨化](./execution/plans/无历史数据-本地库与协同单轨化落地方案-2026-04-19.md)。

## 用户可见变化

1. **IndexedDB 库名**：主库物理名为 `jieyudb_v2`（常量 `JIEYU_DEXIE_DB_NAME`）。旧名 `jieyudb` 不再由应用打开；磁盘上可能残留旧库文件，可按需自行清理。
2. **整库导出**：不再把 `media_items` 中的 `audioBlob` 序列进 JSON；导出对象含 `audioExportOmitted: true`。备份大音频请依赖原始文件或对象存储管线。
3. **整库导入**：`schemaVersion` 必须与当前应用一致，否则拒绝导入。
4. **轨道显示状态**：产品路径仅写入 Dexie `track_entities`，不再与 `jieyu:track-entity-state:v1`（LocalStorage）双写。
5. **托管协同（Supabase）**：部署时在 `001` 之后依次应用 `002_collaboration_rls_identity_bind.sql`（审计列与 `auth.uid()` 绑定）、可选 `003_project_changes_client_op_unique.sql`（`project_changes` 唯一键含 `client_id`）。

## 运维

- 在目标 Supabase 项目执行 `supabase/sql/` 下 **002**（必选）、**003**（若采用多客户端去重键）迁移；勿在无记录情况下改写已应用的 `001` 文件以冒充迁移历史。

## 无 Supabase 时的验收

不配置云端、不执行 SQL 迁移时，可在本机一键跑 **仅本地与 mock 协作契约** 的门禁：

```bash
npm run gate:greenfield-local
```

说明见执行计划 [§8.4](./execution/plans/无历史数据-本地库与协同单轨化落地方案-2026-04-19.md#84-无云端环境验收不连真实-supabase)。具备托管库后再补 `npm run gate:collaboration-cloud`。
