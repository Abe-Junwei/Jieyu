# 发布说明：首屏拆解与发布风险收口（2026-03-28）

## 本次发布口径

本轮改动的目标不是继续微调 bundler，而是把转写页从“大一统运行时”继续收敛为“核心首屏 + 可选域岛”。

截至当前版本：

1. 语音助手运行时已经从转写主编排层抽离到懒加载域。
2. AI Analysis / Embedding 运行时已经从转写主编排层抽离到懒加载域。
3. PDF 预览状态与预览 UI 已从转写主编排层移出，改为独立 PDF runtime。
4. 默认进入转写页时，不再同步初始化语音、Embedding、本地 PDF 预览状态。
5. 默认进入 assistant 标签时，只先挂聊天壳层；语音 runtime 在首次点击语音入口或首次快捷键触发时才激活。
6. 默认进入 analysis 标签时，不再同步构造 embedding provider / service / search，而是走 deferred runtime。
7. PDF 预览已拆为 viewer shell 与 pdfjs renderer 两层，打开预览时先显示稳定壳层，再后置加载 renderer。

## onnxruntime-web 风险结论

当前构建仍会出现 `onnxruntime-web` 的 `eval` 告警。这一项来自上游依赖实现，不是仓库内自定义逻辑。

当前结论如下：

1. 本地 embedding / transformers 能力继续保留，但已经退到 Analysis runtime 的按需路径，不再属于默认首屏能力。
2. 默认进入转写页时，不会因为 AI 聊天 UI 可见而自动初始化本地 embedding 运行时。
3. 如果部署环境要求严格 CSP 审查，需要把 `ai-vendor` 视为“高级可选能力”单独评估，而不是把它当作基础首屏依赖。

## Sentry 发布约定

当前前端仅在以下条件同时满足时初始化 Sentry：

1. `PROD` 构建环境。
2. `VITE_ENABLE_SENTRY=true`。
3. 提供 `VITE_SENTRY_DSN`。

新增环境变量约定：

1. `VITE_SENTRY_RELEASE`
2. `VITE_SENTRY_ENVIRONMENT`
3. `VITE_SENTRY_SEND_DEFAULT_PII`

默认行为：

1. 未提供 `VITE_SENTRY_ENVIRONMENT` 时，回退到 `import.meta.env.MODE`。
2. 未提供 `VITE_SENTRY_RELEASE` 时，不上报 release 字段。
3. 未显式开启 `VITE_SENTRY_SEND_DEFAULT_PII=true` 时，默认不发送默认 PII，并在 `beforeSend` 中移除 `event.user`。

## 依赖审计结论

仓库根目录已有历史 audit 报告 [audit-report.json](/Users/junwei/Documents/Dev/Jieyu/audit-report.json) 与 [audit-report-after.json](/Users/junwei/Documents/Dev/Jieyu/audit-report-after.json)，其中记录的风险集中在旧版 `react-router-dom` 依赖链。

当前仓库中的 `react-router-dom` 版本已是 `6.30.3`，高于历史报告中的受影响区间 `<6.30.2`，因此历史 JSON 报告已不应再作为当前发布结论。

为避免继续依赖过期报告，本轮补充了：

1. `npm run audit:prod`
2. `npm run audit:prod:json`

后续发布前应以实时命令结果为准，而不是沿用旧 JSON 快照。

## 构建体积门禁

当前仓库已补充显式 bundle budget 检查脚本：

1. `npm run check:build-budgets`
2. `npm run build:guard`

当前门禁覆盖以下关键产物：

1. `TranscriptionPage.Orchestrator`
2. `voice-core`
3. `pdf-vendor`
4. `transformers`
5. `index.css`

发布前建议直接执行 `npm run build:guard`，而不是只观察构建输出中的 warning。

## 仍需持续观察的项

以下问题已经从“结构性遗漏”降级为“可继续优化项”：

1. `TranscriptionPage.Orchestrator` 仍然偏大，需要继续拆 citation 解析和部分 AI 侧栏桥接逻辑。
2. `pdf-vendor`、`voice-core`、`transformers` 仍是大块，但已具备清晰的可选域边界。
3. 下一轮如果还要继续压体积，重点应转向更深层的 provider / vendor 策略，而不是继续在页面编排层做纯 helper 抽离。
