# 规划：语言名称矩阵与查询语言覆盖策略 - 2026-04-05

## 1. 直接结论

当前仓库没有“语言名称矩阵”。

更具体地说：

1. 没有 `queryLocale -> targetLanguage -> displayName` 这种全量结构化表。
2. 只有英文主名、部分中文名和手工别名。
3. 没有法语、西语、德语等“某语言对所有语言的名称”矩阵。

因此如果目标是：

- 只做“代码驱动 `local · native · English`，名称命中时按命中语义做最多三段组合展示”的显示规则，当前不一定需要完整矩阵。
- 让用户可以直接输入法语、西语、德语等外语名称来查所有语言，则必须补名称矩阵或等价索引。

## 2. 现状证据

### 2.1 当前语言目录不是矩阵

当前 [src/utils/langMapping.ts](../src/utils/langMapping.ts) 里的 `LanguageCatalogEntry` 只有这些名称相关字段：

- `name`
- `invertedName?`
- `displayNameZh?`
- `aliases`
- `descriptions`

见 [src/utils/langMapping.ts](../src/utils/langMapping.ts#L323-L341)。

这说明：

1. 英文名有一套。
2. 中文名只有部分补丁。
3. 其他 query locale 没有独立显示名集合。

### 2.2 当前搜索索引不是多语言矩阵

当前 `searchLanguageCatalog(...)` 搜索的是：

- `aliases`
- `descriptions`
- `displayNameZh`
- ISO 代码

见 [src/utils/langMapping.ts](../src/utils/langMapping.ts#L634-L689)。

这本质上是“单表多字段搜索”，不是“按 query locale 切换的多语言名称矩阵”。

### 2.3 现有脚本也没产出名称矩阵

当前生成脚本 [scripts/generate-language-orthography-seeds.mjs](../scripts/generate-language-orthography-seeds.mjs#L1-L248) 主要生成：

- top 500 语言种子
- 默认脚本
- 正字法种子

它依赖 CLDR likelySubtags 做脚本推断，但没有生成 `fr -> all language names` 这类名称矩阵。

### 2.4 当前运行时有零散名称派生，但不是稳定矩阵

例如 [src/hooks/useVoiceInteraction.ts](../src/hooks/useVoiceInteraction.ts#L109-L116) 直接使用 `Intl.DisplayNames(['zh-CN', 'en'], { type: 'language' })` 获取显示名。

这说明运行时有“按 locale 派生名称”的能力，但：

1. 不是仓库内显式数据。
2. 不是可离线快照的矩阵。
3. 不是 language input 搜索链路正在使用的索引。

## 3. 要覆盖“所有情况”，到底需要什么矩阵

这里要先把“显示”和“搜索”分开。

### 3.1 显示矩阵

如果需求只是：

- 根据已知 `languageId` 显示代码驱动的 `local · native · English`，以及名称命中后的最多三段组合展示

那么需要的是每个语言的三列名称：

- local
- native
- English

这不是“完整二维矩阵”，更像每个目标语言附带一个多列名称对象；运行时再根据命中语义决定最多展示哪三列。

结构可以是：

```ts
type LanguageDisplayNames = {
  english: string;
  native?: string;
  byLocale: Partial<Record<QueryLocale, string>>;
};
```

### 3.2 搜索矩阵

如果需求是：

- 中文用户输入中文名能搜到
- 英文用户输入英文名能搜到
- 用户输入法语名、西语名、德语名也能搜到

那么需要的是“查询语言维度”的名称索引。

也就是：

`queryLocale -> targetLanguage -> candidateNames[]`

结构可以理解为：

```ts
type QueryLocale = 'zh-CN' | 'en-US' | 'fr-FR' | 'es-ES' | 'de-DE' | ...;

type LanguageNameMatrix = Record<
  QueryLocale,
  Record<string, string[]>
>;
```

其中：

- 外层 key 是“用户用什么语言在查”。
- 内层 key 是目标语言的 ISO 639-3。
- value 是该 query locale 下可用于匹配的名称集合。

## 4. 是否必须有“全量世界矩阵”

不一定。

### 4.1 对显示来说，不需要

要实现你之前定义的显示规则，只需要：

1. 当前 UI locale 名称。
2. native 名称。
3. English 名称。

这不要求法语矩阵、德语矩阵全量存在。

### 4.2 对搜索来说，要看产品范围

如果产品目标是：

- 只支持中文和英文 UI
- 用户常见查询主要是中文、英文、native

那么最小覆盖可以是：

1. `zh-CN`
2. `en-US`
3. `native`

这时不需要完整法语矩阵。

但如果产品目标升级成：

- 任意用户可以输入法语、西语、德语等外语名称来搜任意语言

那就必须引入更完整的 query locale 矩阵。

## 5. 推荐的矩阵分层

建议不要把所有名称都塞成一个大 JSON，而是拆成三层。

### 5.1 Core Display Layer

职责：稳定显示。

每个目标语言保留：

- `englishName`
- `nativeName`
- `zh-CN` 名称
- 未来可扩的 `en-US` / `fr-FR` / `es-ES` 等显示名

这是显示层最小闭环。

### 5.2 Query Index Layer

职责：搜索命中。

每个 query locale 单独生成 token 索引：

```ts
type QueryLocaleIndex = {
  locale: QueryLocale;
  entriesByIso6393: Record<string, Array<{ label: string; kind: 'local' | 'native' | 'english' | 'alias' }>>;
};
```

这样可以按需加载：

- 默认只载入 `zh-CN` 与 `en-US`
- 用户切到法语 UI 或开启高级多语检索时，再载入 `fr-FR`

### 5.3 Alias Layer

职责：人工补洞。

这层保留：

- 业务别名
- 民间俗称
- 旧称
- 高优先级语言的人工校正名

它不应该和 CLDR 默认名混在一层里。

## 6. 最小可落地覆盖策略

建议分三档支持，而不是一步做全世界所有 query locale。

### 档 1：当前即可落地

支持：

- `zh-CN` 名称
- `en-US` 名称
- native 名称

用途：

- 完成代码驱动 `local · native · English` 与名称命中后的最多三段组合展示
- 允许中文、英文、native 检索

优点：

- 成本最低
- 直接解决当前产品主问题

### 档 2：一级国际化查询语言

额外支持：

- `fr-FR`
- `es-ES`
- `de-DE`
- `pt-BR` / `pt-PT`
- `ar`
- `ru`

用途：

- 覆盖主要国际学术 / 多语协作输入场景

### 档 3：全量 CLDR 覆盖

支持 CLDR 中更多 query locale。

这时才接近真正的“名称矩阵”。

但这一步不建议作为第一阶段目标，因为：

1. 体积会快速膨胀。
2. 需要决定 query locale 白名单。
3. 测试矩阵和排序策略复杂度会明显上升。

## 7. 数据来源建议

### 7.1 English

来源：

- 现有 `iso-639-3` 数据

优先级：

- 高

### 7.2 Chinese

来源：

- 当前手工 `displayNameZh`
- 后续建议补 CLDR 中文名称

优先级：

- 高

### 7.3 Native

来源：

- 第一阶段：`Intl.DisplayNames`
- 第二阶段：构建期固化成种子字段

优先级：

- 高

### 7.4 其他 query locale

来源：

- CLDR 精简产物

优先级：

- 中，按产品范围逐步接入

## 8. 推荐的构建产物格式

建议新增长名称矩阵产物，例如：

- `public/data/language-support/language-display-names.core.json`
- `public/data/language-support/language-query-index.zh-CN.json`
- `public/data/language-support/language-query-index.en-US.json`
- `public/data/language-support/language-query-index.fr-FR.json`

### 8.1 core display

```json
{
  "version": 1,
  "generatedAt": "...",
  "languages": {
    "fra": {
      "english": "French",
      "native": "français",
      "byLocale": {
        "zh-CN": "法语",
        "en-US": "French",
        "fr-FR": "français"
      }
    }
  }
}
```

### 8.2 query index

```json
{
  "version": 1,
  "locale": "fr-FR",
  "entriesByIso6393": {
    "deu": [
      { "label": "allemand", "kind": "local" },
      { "label": "Deutsch", "kind": "native" },
      { "label": "German", "kind": "english" }
    ],
    "jpn": [
      { "label": "japonais", "kind": "local" },
      { "label": "日本語", "kind": "native" },
      { "label": "Japanese", "kind": "english" }
    ],
    "fra": [
      { "label": "français", "kind": "local" },
      { "label": "French", "kind": "english" },
      { "label": "法语", "kind": "alias" }
    ]
  }
}
```

## 9. 运行时使用策略

### 9.1 显示

显示不应查 query index，而应查 core display。

原因：

- 显示是稳定问题。
- 搜索索引是召回问题。

### 9.2 搜索

搜索时按用户当前 query locale 选择对应 index。

例如：

- 中文 UI 默认搜索 `zh-CN`
- 英文 UI 默认搜索 `en-US`
- 同时总是并入 `native` 与 `english` fallback

### 9.3 fallback 顺序

建议：

1. 当前 query locale index
2. native names
3. English names
4. aliases
5. 代码

## 10. 对当前项目的建议

最合理的路线不是立刻做“全量法语矩阵”，而是：

1. 先补 `core display`：`zh-CN + en-US + native`
2. 先把三段式显示跑通
3. 搜索先支持 `zh-CN + en-US + native`
4. 再按产品需要增量加 `fr-FR / es-ES / de-DE`

原因是：

1. 当前 UI locale 也只定义了 `zh-CN | en-US`，见 [src/utils/langMapping.ts](../src/utils/langMapping.ts#L323)。
2. 现有搜索和显示体系还没把 native 正式接进来。
3. 先做完整世界矩阵，成本远大于当前产品收益。

## 11. 最终判断

你的判断在“搜索覆盖”这个层面是成立的：

- 如果要正确覆盖所有输入语言名场景，确实需要名称矩阵或等价的多 locale 查询索引。

但在“显示规则”这个层面，它不必一步到位做成完整世界矩阵：

- 显示层先做 `local + native + English` 三列数据，再在运行时按命中语义决定最多展示三段，就足够成立。

因此项目建议采用：

1. `显示层最小三列`
2. `搜索层按 query locale 分索引`
3. `先小矩阵，后扩完整矩阵`

而不是：

1. 一开始就试图把所有 query locale 对所有目标语言的名称一次性打进运行时。
