# 仓库现状与代码地图

> 文档角色：长期有效的当前现状文档。最后更新：2026-04-05。

## 说明

本文档基于当前仓库代码现状整理，目标是给出一份“当前已经实现了什么、代码主要分布在哪里、哪些能力仍是占位/过渡态”的完整盘点。本文以运行时事实为准，不以历史规划文档为准。

## 一、项目定位与技术栈

解语（Jieyu）当前是一个本地优先的濒危语言研究与协作应用，前端采用 React + TypeScript + Vite，数据层采用 Dexie / IndexedDB，围绕“转写主工作区 + 语言资产工作台 + AI/语音辅助 + 多格式互操作”逐步落地。

当前主技术栈与基础设施如下：

- React 19 + React Router 6：应用壳层、路由切分、工作台页面组织。
- TypeScript 5：全仓库主语言。
- Vite 7：开发与构建工具链。
- Vitest：单元测试、结构测试、回归测试与性能基线测试。
- Dexie：浏览器本地数据库与版本迁移。
- Zod：数据入库前校验。
- pdfjs-dist / WaveSurfer：PDF 与音频波形能力。
- @xenova/transformers：本地 AI / embedding 相关能力基础。
- Sentry：发布态异常观测入口。

应用启动入口位于 src/main.tsx，主应用壳层位于 src/App.tsx。

## 二、当前路由与工作台结构

当前应用已经形成“主工作区 + 语言资产工作区”的双层导航结构。

### 1. 主工作区路由

- /transcription：主转写工作区，当前最完整的业务页面。
- /annotation：占位页，展示功能范围与后续方向。
- /analysis：占位页，展示统计、审计、质量分析方向。
- /writing：占位页，展示写作编排、引文、模板方向。
- /lexicon：词典/词条浏览工作区，已具备只读浏览与检索能力。

### 2. 语言资产路由

- /assets/language-metadata：语言元数据工作台。
- /assets/orthographies：正字法工作台。
- /assets/orthography-bridges：正字法桥接工作台。

### 3. 兼容路由

- /lexicon/orthographies：旧入口，当前只做重定向到 /assets/orthographies。

整体上，应用壳层已经把语言资产维护从词典/转写上下文中抽离出来，形成独立的资产维护入口。

## 三、页面实现深度盘点

### 1. 转写工作区：当前核心业务中枢

转写页入口在 src/pages/TranscriptionPage.tsx，真正的业务编排在 src/pages/TranscriptionPage.Orchestrator.tsx。当前该页面已经不是简单展示层，而是仓库内最完整的业务编排中心。

从现有 orchestrator、控制器与配套 hooks 看，转写页已经覆盖以下能力：

- 项目文本、媒体、话语、层、锚点等核心数据读取与写回。
- 音频时间轴与波形联动。
- 多层转写/翻译视图与轨道显示控制。
- 语段创建、拆分、合并、批量偏移、批量缩放、批量删除等编辑操作。
- 图层创建、删除、排序、联动显示与约束处理。
- 语段选择、范围选择、批量选择与焦点切换。
- 自动保存、恢复横幅、撤销/重做历史。
- PDF 预览运行时、AI 侧边栏、分析运行时、弹窗编排。
- 说话人管理、轨道实体持久化、侧栏信息同步。
- 键盘播放控制、JKL shuttle、录音与语音输入接入。
- AI 辅助转写控制器、助手侧边栏控制器、工具调用路由。

从文件组织看，转写页已经形成明显的“编排层 + 控制器层 + 子视图层”分层，而不是单文件页面：

- 页面编排与运行时切片：src/pages/TranscriptionPage.*
- 领域 hooks：src/hooks/useTranscription*.ts、src/hooks/useLayer*.ts、src/hooks/useVoice*.ts
- 时间轴与面板组件：src/components/transcription、src/components/Timeline*.tsx、src/components/SidePaneSidebar*.tsx

### 1.1 声学叠加能力：已进入 Phase 1 最小落地

截至当前代码现状，转写工作区已经开始承载本地优先的声学叠加基础能力，但仍处于 Phase 1 的最小闭环阶段。

当前已存在的声学实现包括：

- `src/utils/acousticOverlayTypes.ts`：声学分析结果、配置与 overlay 模式类型定义。
- `src/services/acoustic/acousticAnalysisCore.ts`：基于 YIN + RMS/dB 的最小浏览器端分析核心。
- `src/services/acoustic/acousticAnalysis.worker.ts`：声学分析 worker 入口。
- `src/services/acoustic/AcousticAnalysisCacheDB.ts`：独立 Dexie 子库，负责声学分析结果的 IndexedDB 持久缓存与 stale/fresh 管理。
- `src/services/acoustic/AcousticAnalysisService.ts`：fetch/decode/downmix/worker 调度、内存缓存与持久缓存复用。
- `src/pages/useWaveformRuntimeController.ts`：声学叠加模式持久化状态。
- `src/components/WaveformToolbar.tsx`：`none / f0 / intensity / both` 工具栏入口。
- `src/pages/useTranscriptionWaveformBridgeController.ts`、`src/pages/OrchestratorWaveformContent.tsx` 与 `src/components/transcription/WaveformReadoutCard.tsx`：主波形区 F0 / intensity 最小叠加渲染、waveform / spectrogram 统一角落 readout，以及频谱 hover readout / click-to-seek。
- `src/pages/TranscriptionPage.aiPromptContext.ts`、`src/ai/chat/promptContext.ts` 与 `src/pages/useTranscriptionAiController.ts`：AI `acousticSummary(...)` 紧凑注入与最小 hotspot 摘要。
- `src/components/AiAnalysisPanel.tsx`、`src/pages/transcriptionAcousticSummary.ts`、`src/contexts/AiPanelContext.tsx` 与 `src/pages/TranscriptionPage.AssistantBridge.tsx`：analysis sidebar 内独立 acoustic tab 的声学摘要、诊断提示与热点导航承载。

当前明确尚未完成的部分包括：

- 主波形区与频谱区统一 hover readout 的进一步打磨；当前已完成最小统一角落 readout。
- 更重的 inspector 形态与跨面板热点复用；当前已具备独立 acoustic tab，但还不是完整科研级分析工作区。
- tone chart、formant、元音图与更重的 inspector 能力。

因此，当前仓库对声学语音学分析的定位应理解为：已具备最小本地分析底座、主波形区只读增强、最小频谱交互、AI 紧凑声学摘要，以及 analysis sidebar 内独立 acoustic tab，但尚未进入完整的科研级重分析工作台阶段。

这部分是当前仓库实现最成熟、最接近生产工作台形态的模块。

### 2. 标注页 / 分析页 / 写作页：路由已建，能力暂为占位

以下页面当前统一通过 FeatureAvailabilityPanel 呈现：

- src/pages/AnnotationPage.tsx
- src/pages/AnalysisPage.tsx
- src/pages/WritingPage.tsx

这三页已经纳入正式导航和国际化文案体系，但当前仍属于“产品方向清晰、具体工作流尚未实装”的阶段。它们更像是明确边界的壳层，而不是已完成业务页。

### 3. 词典页：已具备只读浏览与筛选

src/pages/LexiconPage.tsx 当前不是占位页，而是一个轻量但可用的词条浏览工作区，已有能力包括：

- 从 LinguisticService 读取 lexeme 列表。
- 按 lemma、citation、语言、词类、义项等字段检索。
- 左侧词条列表 + 右侧详情栏布局。
- 展示义项、词形、备注、语言、更新时间等信息。
- 与转写页、正字法工作台建立快速跳转。

当前限制也很明确：它更接近“浏览器/检视器”，不是完整的词典编辑器。

### 4. 语言元数据工作台：已形成独立维护工作流

语言元数据工作台位于以下文件：

- src/pages/LanguageMetadataWorkspacePage.tsx
- src/pages/LanguageMetadataWorkspaceCatalogPanel.tsx
- src/pages/LanguageMetadataWorkspaceDetailColumn.tsx

当前已具备的能力：

- 语言目录列表加载、搜索、选中与新建自定义语言入口。
- 语言主信息维护：languageCode、canonicalTag、ISO 系列字段、family、subfamily、macrolanguage、scope、languageType 等。
- 名称矩阵维护：英文名、本地名、native name、多 locale display name。
- alias 维护与保存。
- visibility 与治理字段维护。
- 备注与变更原因输入。
- 最近历史记录查看。
- 持久化保存、删除 override / custom 记录。
- 侧边栏与正字法、桥接工作台联动跳转。

当前这部分已经不是简单配置页，而是语言资产治理的正式入口。

### 5. 正字法工作台：已具备编辑与资产联动能力

正字法工作台主要位于：

- src/pages/OrthographyWorkspacePage.tsx
- src/pages/OrthographyWorkspaceEditor.tsx
- src/components/OrthographyBuilderPanel.tsx

当前已具备的能力：

- 加载 built-in 与持久化正字法记录。
- 基于语言资产 ID 选择目标语言，支持 user:* 自定义语言资产。
- 编辑正字法核心元数据：名称、缩写、scriptTag、direction、locale/region/variant。
- 编辑目录元信息、字符样本、规范化策略、排序规则、字体偏好、输入提示、bidi 策略、转换规则、备注等。
- 脏状态保护与切换确认。
- 与桥接工作台联动，按 targetOrthographyId 携带上下文跳转。

这部分已经形成“资产列表 + 详情编辑 + 桥接入口”的完整维护闭环。

### 6. 正字法桥接工作台：已从正字法页中独立

正字法桥接工作台位于 src/pages/OrthographyBridgeWorkspacePage.tsx，当前已经成为桥接维护的唯一正式工作区。

当前已具备的能力：

- 以目标正字法为中心浏览桥接对象。
- 左侧正字法列表筛选与选中。
- 右侧桥接管理面板承载桥接规则维护。
- 与语言元数据、正字法工作台互跳。
- 接收 targetOrthographyId 查询参数作为上下文入口。

值得注意的是，桥接管理不再嵌入正字法工作台详情页中，而是通过上下文链接进入独立工作区。这说明语言资产信息架构已经完成一次比较明确的职责拆分。

## 四、服务层能力盘点

### 1. LinguisticService：仓库中的主服务门面

src/services/LinguisticService.ts 仍然是当前仓库中最核心的服务聚合入口，已覆盖：

- 话语、token、morpheme、translation 等转写主数据读写。
- speaker 创建、统计、删除。
- lexeme 列表与搜索。

- 级联删除与图数据收口。

当前仓库大量页面与 hook 都以该服务层为边界，而不是直接操作 Dexie。

### 2. 语言资产相关服务已经独立拆分

以下拆分已经明确：

### 4. 语言目录读模型现状

当前语言目录运行时读取链路已经完成“生成基线退居初始化/回退，Service 投影成为主读模型”的迁移：

## 六、前端结构盘点

### 1. src 目录主干

当前 src 下的关键目录如下：

- pages：页面与页面级控制器。
- components：通用组件、时间轴组件、对话框、AI 面板、正字法管理组件等。
- hooks：仓库中最重的应用逻辑组织层。
- services：领域服务、互操作、AI/语音、层/轨道逻辑。
- db：Dexie schema 与导入导出。
- data：运行时缓存、生成基线、只读数据入口。
- contexts：跨页面/跨组件上下文，如 AI 面板、侧边栏、Toast、语音智能体。
- ai：AI 编排、providers、embeddings、消息与任务系统。
- i18n：国际化词条、typed key、页面与组件文案模块。
- styles：全局样式、基础层、页面样式、面板样式。
- observability：logger、error aggregation、Sentry 初始化。
- tools：当前包含 whisper-server 集成入口。
- types：跨模块共享类型。

### 2. hooks 层现状

当前 hooks 目录规模较大，说明仓库在持续把复杂页面逻辑拆到 hook 层。重点主题包括：

- useTranscription*：转写主工作流与数据流。
- useAi*：AI 会话、工具调用、嵌入状态、面板逻辑。
- useVoice*：语音智能体、语音交互、语音 Dock、唤醒词联动。
- useImportExport*：导入导出编排。
- useOrthographies / useOrthographyPicker / useLanguageCatalogLabelMap：语言资产辅助逻辑。
- usePanelAutoCollapse / usePanelResize / useDraggablePanel：壳层与面板交互。

这说明仓库已经把“页面功能”逐步沉淀为可组合的控制器/状态 hook，而不是把所有逻辑塞回页面本身。

### 3. 组件层现状

components 目录当前包含：

- 时间轴主视图、轨道表头、翻译控制、波形工具栏等转写核心组件。
- NotePanel、LayerActionPopover、SidePaneSidebar、BatchOperationPanel 等工作台组件。
- OrthographyBuilderPanel、OrthographyBridgeManager、LanguageIsoInput 等语言资产组件。
- VoiceAgentWidget、AiAnalysisPanel、PdfViewerPanel 等扩展能力组件。
- ui 子目录中的 PanelSection、PanelSummary、DialogShell、EmbeddedPanelShell 等基础 UI 结构件。

整体上，当前组件层不是简单的视觉组件集合，而是业务组件与壳层组件混合存在，且多数核心组件已有测试覆盖。

## 十二、代码地图（适合新人顺着读）

下面这份“代码地图”不是完整目录镜像，而是按“先看哪里、再看哪里”排序的阅读路径。

### 1. 应用入口层

- src/main.tsx：React 挂载、BrowserRouter、Sentry 初始化、全局样式入口。
- src/App.tsx：应用壳层、主导航、语言资产导航、全局 side pane、路由注册、启动期语言目录读模型刷新。

如果第一次接手仓库，建议先读这两个文件，先搞清楚“页面如何挂进壳层、全局交互框架如何组织”。

### 2. 页面层

pages 目录可以按三类理解：

- 主工作区页：TranscriptionPage.tsx、LexiconPage.tsx。
- 语言资产页：LanguageMetadataWorkspacePage.tsx、OrthographyWorkspacePage.tsx、OrthographyBridgeWorkspacePage.tsx。
- 壳层/占位页：AnnotationPage.tsx、AnalysisPage.tsx、WritingPage.tsx。

其中 TranscriptionPage.* 是一个页面簇，而不是单一页面文件。它已经拆成 orchestrator、toolbar、timeline、dialogs、side pane、analysis runtime、assistant runtime、pdf runtime 等多个子文件。

### 3. 转写页阅读顺序

如果要理解当前主业务链，推荐按下面顺序阅读：

- src/pages/TranscriptionPage.tsx：看到 lazy 入口。

- src/hooks/useTranscriptionData.ts：看到核心数据流与状态收口。

- src/components/transcription：看到布局切片、工具栏、左轨、时间轴 section 组件。
- src/components/Timeline*.tsx 与 src/components/SidePaneSidebar*.tsx：看到主要业务组件。

这条链基本对应“页面编排 -> 状态/动作 -> 具体 UI”的真实运行路径。

这条链大致对应“工作台 UI -> 选择器/映射 -> 服务写入与投影 -> 同步读模型”。

### 5. 服务层地图

services 目录可以按主题理解，而不是按文件名逐个记忆：

- 核心门面：LinguisticService.ts。
- 语言资产：LinguisticService.languageCatalog.ts、LinguisticService.orthography.ts。
- 层/轨道/分段：LayerConstraintService、LayerOrderingService、LayerSegmentGraphService、LayerSegmentationTextService、LayerUnitService、TierBridgeService、TrackEntityStore。
- 互操作：EafService、TextGridService、FlexService、ToolboxService、TranscriberService、JymService、SnapshotService。
- AI/语音：VoiceAgentService*、AudioAnalysisService、VadService、WakeWordService、IntentRouter、CommandResolver、AiHybridRecommendationService。
- 上下文/行为：GlobalContextService、ProjectMemoryStore、VoiceSessionStore、UserBehaviorStore。

如果遇到“这个功能应该去哪改”的问题，优先按主题定位到服务簇，而不是全文搜索后直接改页面。

### 6. 数据层地图

- src/db/index.ts：数据库 schema、迁移、validator、导入导出。
- src/data：运行时缓存、baseline 生成物的读取入口、只读数据 facade。
- public/data/language-support：语言支持基线 JSON、查询索引和 top500 seed 数据。

其中 src/db/index.ts 是“真源结构”，src/data 更偏“运行时读模型与静态基线消费”。

### 7. AI 与语音地图

- src/ai：聊天编排、provider、tasks、embeddings、性能基线。
- src/hooks/useAi*.ts：页面级 AI 面板、会话、工具调用、嵌入状态。
- src/hooks/useVoice*.ts：语音模式、语音 Dock、语音智能体接线。
- src/services/VoiceAgentService*.ts：非 React 运行时主体。
- src/tools/whisper-server：外部语音相关工具接入点。

这条链说明 AI/语音实现不是集中在一个页面组件里，而是分成前端 hook、后台式 service、独立 AI 模块三层。

### 8. UI 基座地图

- src/components/ui：PanelSection、PanelSummary、DialogShell、EmbeddedPanelShell 等基础结构件。
- src/contexts：AiPanel、AppSidePane、Toast、VoiceAgent、TranscriptionEditor 等跨组件上下文。
- src/styles：全局基础样式、页面样式、专题样式。
- src/i18n：统一词条、typed key 与专题 message 模块。
- src/observability：logger、错误聚合、Sentry。

如果是做页面 UI 改造，通常要同时看 components/ui、styles、i18n，而不是只改页面 TSX。

### 9. 脚本与治理地图

- package.json：所有入口脚本的总索引。
- scripts/check-*.mjs / .cjs：架构守卫、i18n 守卫、样式守卫、边界守卫。
- scripts/generate-language-*.mjs：语言数据生成链。
- scripts/profile-build-assets.mjs、check-build-budgets.mjs：构建产物治理。
- scripts/release：发布/提交整理辅助脚本。

这部分决定了仓库“哪些结构不能乱动”，新接手的人应该尽早熟悉。
