# 层关系矩阵与自动排序约束方案

> 2026-03-28
>
> 范围收敛：当前运行时**不设计也不开放** `time_subdivision` 转写层与翻译层。
> 本方案只讨论两类边界约束：
>
> - `independent_boundary` 独立边界
> - `symbolic_association` 依赖边界

---

## 0. 目标

解决两个实际问题：

1. 明确所有层之间允许和不允许的“对应关系”。
2. 让层拖动排序不再只按 `layerType` 分区，而是按“结构依赖组”自动计算、自动调整，必要时抛出警告。

本方案的重点不是继续扩展层类型，而是把已有模型收敛成一个稳定、可解释、可自动修复的排序系统。

---

## 1. 范围与约束

### 1.1 在本方案内，运行时只保留三类层

| 代号 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `TRC-I` | 转写层 | `independent_boundary` | 独立转写根层，唯一可作为结构父层 |
| `TRC-D` | 转写层 | `symbolic_association` | 依赖转写层，必须依赖某个 `TRC-I` |
| `TRL-D` | 翻译层 | `symbolic_association` | 依赖翻译层，必须依赖某个 `TRC-I` |

### 1.2 明确排除的类型

| 类型 | 状态 | 处理方式 |
|---|---|---|
| `translation + independent_boundary` | 不允许新建 | 仅视为历史兼容数据，进入修复/降级路径 |
| `transcription + time_subdivision` | 不纳入当前设计 | UI 不开放，排序不支持 |
| `translation + time_subdivision` | 不纳入当前设计 | UI 不开放，排序不支持 |

### 1.3 设计原则

1. 只有 `TRC-I` 可以作为结构父层。
2. `parentLayerId` 只表达结构依赖，不表达显示顺序。
3. `layerLinks` 只表达辅助语义关联，不参与排序真源。
4. 排序必须由结构依赖组自动推导，不能只靠用户手动把 `sortOrder` 拖对。

---

## 2. 三种关系类型

为了避免后续规则混淆，需要明确区分三类关系。

### 2.1 结构依赖关系

- 真源字段：`parentLayerId`
- 语义：子层边界继承自父层
- 约束：父层必须是 `TRC-I`

### 2.2 语义链接关系

- 真源字段：`layer_links`
- 语义：某个转写层与某个翻译层存在对照/关联
- 作用：用于“链接视图”和辅助工作流，不驱动排序

### 2.3 线性显示关系

- 真源字段：`sortOrder`
- 语义：侧栏、时间轴、导出前可视顺序
- 新要求：不得破坏结构依赖组的连续性

---

## 3. 完整对应关系矩阵

### 3.1 结构父子矩阵

| 父层 \\ 子层 | `TRC-I` | `TRC-D` | `TRL-D` |
|---|---:|---:|---:|
| `TRC-I` | × | ✓ | ✓ |
| `TRC-D` | × | × | × |
| `TRL-D` | × | × | × |

结论：

1. 独立转写层之间只能并列，不能互为父子。
2. 依赖转写层只能依赖独立转写层。
3. 依赖翻译层只能依赖独立转写层。
4. 翻译层不能作为任何层的结构父层。

### 3.2 同层类型之间的对应关系

| 关系对 | 是否允许结构对应 | 实际关系 |
|---|---:|---|
| `TRC-I ↔ TRC-I` | 否 | 并列根组 |
| `TRC-I ↔ TRC-D` | 是 | 父子 |
| `TRC-D ↔ TRC-D` | 否 | 同根兄弟 |
| `TRL-D ↔ TRL-D` | 否 | 同根兄弟 |
| `TRC-D ↔ TRL-D` | 否 | 同根兄弟 |

也就是说，当前模型下不存在“翻译层直接依赖转写依赖层”或“翻译层依赖翻译层”的合法结构。

### 3.3 语义链接矩阵

| 发起层 | 目标层 | 是否允许 `layerLinks` |
|---|---|---:|
| 转写层 | 翻译层 | ✓ |
| 转写层 | 转写层 | 不建议 |
| 翻译层 | 翻译层 | 不建议 |

这里的“不建议”不是数据层绝对禁止，而是当前产品不应把它设计成排序或父子关系规则的一部分。

## 4. 排序模型重设计

### 4.1 现状问题

旧版 [src/hooks/useTranscriptionLayerActions.ts](src/hooks/useTranscriptionLayerActions.ts) 中的 `reorderLayers()` 只做了两件事：

1. 转写层只能在转写区移动
2. 翻译层只能在翻译区移动

这会导致：

- 一个 `TRC-I` 与它的 `TRL-D` 之间可以被另一个 `TRC-I` 插开
- 同一个独立根下的依赖层无法作为“组”保持连续
- `sortOrder` 与 `parentLayerId` 失去一致性

当前实现状态（2026-03-28）补充：

1. 左侧栏已改为按全局 `sortOrder` / Bundle 结果统一排序。
2. 时间轴主渲染已移除“先全部转写层、再全部翻译层”的双分区限制，改为与左侧栏一致，统一按 `allLayersOrdered` 渲染。
3. [src/components/TimelineLaneHeader.tsx](src/components/TimelineLaneHeader.tsx) 的时间轴头部拖拽也已移除按转写/翻译分区裁剪目标 index 的逻辑，改为使用统一全局 index。

因此，当前剩余关注点不再是“翻译层是否被强制压到所有转写层之后”，而是“统一顺序是否始终满足 Bundle 不变量”。

### 4.2 新的排序真源：Bundle

定义 `LayerBundle`：

- 根节点：一个 `TRC-I`
- 子节点：所有 `parentLayerId === root.id` 的 `TRC-D` 与 `TRL-D`

也就是说，显示顺序不再按“转写区/翻译区”展开，而是按 Bundle 展开。

### 4.3 Bundle 内部固定顺序

建议采用以下顺序：

1. `TRC-I` 根层
2. 该根下的全部 `TRC-D`
3. 该根下的全部 `TRL-D`

理由：

1. 保持“翻译层属于某个转写 Bundle，而不是独立全局分区”的主认知模型。
2. 避免翻译层跑出所属根组、跑到根层上方，或插到其他根组中间。
3. 不要求用户手工理解全局位置，只需要理解“我属于哪个根组”。

这里要避免使用含混说法“翻译层必须在转写层之下”。

更准确的表述是：

- `TRL-D` 必须位于其所属 `TRC-I` Bundle 内
- 且位于该 Bundle 的翻译子区
- 而不是位于一个独立的全局“翻译区”

### 4.4 全局顺序规则

全局顺序应变为：

- `Bundle A`
- `Bundle B`
- `Bundle C`

每个 Bundle 内部按 4.3 展开。

这意味着：

1. 独立转写层是“组头”。
2. 任何依赖层都不能跑出自己的组。
3. 任意两个 Bundle 之间不能互相穿插。

## 5. 排序不变量

以下不变量一旦确定，就可以作为拖拽、修复、导入后整理的统一判据。

### 5.1 结构不变量

1. `TRC-D.parentLayerId` 必须指向某个 `TRC-I`。
2. `TRL-D.parentLayerId` 必须指向某个 `TRC-I`。
3. 任意层都不能形成 parent cycle。

### 5.2 顺序不变量

1. 每个 Bundle 必须连续。
2. Bundle 内第一个元素必须是其根 `TRC-I`。
3. `TRC-D` 不得出现在 `TRL-D` 之后的子区之前被其他 Bundle 打断。
4. `TRL-D` 不得出现在其根层之前，且必须位于所属 Bundle 的翻译子区。
5. 一个 Bundle 内不得插入其他 Bundle 的任何层。

### 5.3 用户可见约束

1. 一个独立转写层和它的任一依赖翻译层之间，不允许夹入其他独立层。
2. 一个依赖层离开父根组时，系统必须自动吸附、自动改绑，或直接拒绝。
3. UI 提示不再表述为“翻译层必须排在转写层下面”，而应表述为“翻译层必须位于所属转写 Bundle 的翻译子区”。

## 6. 拖动自动调整规则

### 6.1 拖动 `TRC-I`

行为：移动整个 Bundle。

自动调整：

- 根层移动到目标 Bundle 前后
- 其全部 `TRC-D` / `TRL-D` 跟随重排
- `parentLayerId` 不变

用户反馈：

- 信息提示：`已移动整组层关系。`

### 6.2 拖动 `TRC-D`

分两类处理：

1. 拖到当前根组内部
   - 允许
   - 只能在当前根组的 `TRC-D` 区域内重排

2. 拖到另一个 `TRC-I` 根组附近
   - 可选策略：允许自动改绑
   - 自动把 `parentLayerId` 改为目标根
   - 重新归入目标根组的 `TRC-D` 子区

用户反馈：

- 警告提示：`已将该转写层改为依赖「某某转写层」。`

### 6.3 拖动 `TRL-D`

分三类处理：

1. 拖到当前根组内部
   - 允许
   - 只能在当前根组的 `TRL-D` 区域内重排

2. 拖到另一个 `TRC-I` 根组附近
   - 允许自动改绑
   - 自动更新 `parentLayerId`
   - 自动落到目标根组的 `TRL-D` 子区

3. 拖到无合法根层的位置
   - 拒绝

用户反馈：

- 警告提示：`已将该翻译层改为依赖「某某转写层」。`
- 错误提示：`无法放置到该位置：目标区域没有合法的独立转写层。`

### 6.4 拖到非法间隙时的吸附策略

例如用户把 `TRL-D` 放到两个 Bundle 中间、或放到某个 Bundle 的 `TRC-D` 区域之前。

建议行为：

1. 优先识别最近的合法根层
2. 自动吸附到该根组的合法子区末尾
3. 如果最近合法根不存在，则取消拖动

用户反馈：

- 信息提示：`已自动调整到合法位置。`

## 7. 告警分级

### 7.1 信息提示

适用场景：

- 仅调整了 `sortOrder`
- 未修改 `parentLayerId`

文案建议：

- `已自动整理层顺序。`
- `已移动整组层关系。`

### 7.2 警告提示

适用场景：

- 自动改绑了 `parentLayerId`

文案建议：

- `已将该翻译层改为依赖「{label}」。`
- `已将该转写层改为依赖「{label}」。`

### 7.3 错误提示

适用场景：

- 目标位置不存在合法独立根
- 本次拖动会破坏 Bundle 连续性，且无法自动修正

文案建议：

- `无法放置到该位置：依赖层必须属于某个独立转写组。`

## 8. 实现路线

建议新增：

- `src/services/LayerOrderingService.ts`

建议包含以下 API：

1. `buildLayerBundles(layers)`
   - 按 `TRC-I` 构建根组

2. `computeCanonicalLayerOrder(layers)`

3. `validateLayerOrder(layers)`
   - 返回顺序违规项

4. `resolveLayerDrop(layers, draggedLayerId, dropTarget)`
   - 输入拖动意图

2. `repairExistingLayerConstraints()`
   - 对不合法父层自动降级或改绑到最近合法 `TRC-I`

3. 补充 `validateLayerOrder()` 与 `repairLayerOrder()`
   - 可放在本文件或新的 `LayerOrderingService.ts`

## 9. 验收用例

### 9.1 关系矩阵验收

1. 可创建 `TRC-I`
2. 可创建依赖于某个 `TRC-I` 的 `TRC-D`
3. 可创建依赖于某个 `TRC-I` 的 `TRL-D`
4. 不允许 `TRL-D` 依赖 `TRC-D`
5. 不允许任何层依赖 `TRL-D`

### 9.2 排序验收

1. 拖动 `TRC-I` 时，整组跟随移动
2. 不能把外部 `TRC-I` 插进别人的 Bundle 中间
3. 不能把 `TRL-D` 放到其父根之前
4. 拖动 `TRL-D` 到另一个根组时，允许自动改绑并警告
5. 拖动到无合法根区域时，直接拒绝并提示

### 9.3 修复验收

1. 对历史非法顺序执行一次 repair，可恢复为合法 Bundle 顺序
2. 对历史非法 `parentLayerId` 执行 repair，可恢复为合法独立根依赖
3. repair 后 `sortOrder`、`parentLayerId`、UI 展示一致

## 10. 最终建议

本轮不要继续把层关系做成更通用的树，也不要重新引入 `time_subdivision`。

当前最稳妥的收敛方案是：

1. 只保留 `TRC-I`、`TRC-D`、`TRL-D` 三类运行时层
2. 只允许 `TRC-I` 作为结构父层
3. 用 Bundle 作为排序真源
4. 拖动时允许自动吸附、自动改绑，并配套警告

这样可以同时解决：

1. 关系矩阵不清晰
2. 排序与结构脱节
3. 独立层插进父子层中间
4. 用户拖动后需要手工收残局

这是比“继续补 if”更稳定的长期方案。
