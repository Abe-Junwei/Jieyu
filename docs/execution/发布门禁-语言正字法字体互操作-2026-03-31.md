# 发布门禁：语言-正字法-字体-互操作（2026-03-31）

> 目的：把 orthography / bridge / font / bidi / interop 这条收口线的 go / no-go 条件显式落到仓库资产，避免“代码合了，但没有统一验收口径”。

## 一、适用范围

本门禁只覆盖以下变更面：

- orthography bridge runtime 与机器写回路径
- 复杂脚本字体校验、缓存与 UI 风险提示
- EAF / TextGrid provider-neutral metadata contract
- plain-text bidi 安全导入导出边界

不覆盖：

- AI / voice / RAG 全量产品化主流程
- segmentation v2 历史迁移门禁
- 首屏拆解与页面体量控制的独立发布口径

## 二、发布结论规则

### 直接禁止发布

满足任一项即禁止发布：

1. bridge 机器写回路径出现“目标层被写入原文，但预期应命中 source -> target bridge”的回归。
2. 复杂脚本自定义字体被标成“已验证”，但实际上只证明 glyph presence，没有落入 `shaping-risk`。
3. EAF / TextGrid 任一侧对 provider-neutral metadata 的未知字段降级不稳定，或丢失 `bridgeId` 透传。
4. plain-text bidi 导出导入重新泄漏 isolate 控制字符到搜索、复制或二次导入文本。

### 可有条件发布

满足以下全部条件时可有条件发布：

1. 自动化全部通过。
2. 手工 CASE 全部执行。
3. 若存在 P1 风险，只能是“文档/提示不够清晰”，不能是主流程正确性问题。
4. 已给出证据目录与回滚口径。

## 七、回滚口径

出现以下任一现象时建议回滚：

1. 导入后翻译层或目标层文本被错误保留为未桥接的原文。
2. 复杂脚本自定义字体在 Builder / Layer 菜单中被显示为“已验证”。
3. EAF / TextGrid round-trip 后 tier metadata 丢失或导入报错。
4. plain-text 导出文本在复制、搜索或再次导入时出现双向控制字符污染。

建议动作：

1. 暂停当前发版批次。
2. 保留出错样本文件与项目快照。
3. 先回到对应自动化批次复现，再按 OI-01 ~ OI-05 分类定位。
