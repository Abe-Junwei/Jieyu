**业内对比**：
- **Cursor / Continue.dev**：同样采用多 Provider 抽象层 + 流式输出模式，解语实现水平对齐。

- **差距**：无 Provider 健康检测（heartbeat）、无自动 failover 路由（当首选 Provider 不可用时自动切换）。对单用户桌面工具影响较小。

## 二、计划要求但未实现的功能

按必要性分级（🔴 高 / 🟡 中 / 🟢 低）：

### 🔴 高必要性 — 直接影响产品核心价值

#### 2.1 F32 项目知识库 RAG — ❌ 未实现

**计划要求**：多源向量化管线（PDF + notes + tier 描述）→ 双源 top-k 检索（全文 + 向量）→ 引用溯源（脚注 + 点击跳转）→ 文献管理 UI。

**现状**：EmbeddingSearchService 提供基础的 utterance 相似搜索，但无完整 RAG pipeline：
- ❌ PDF 上传与解析（无 pdf.js 集成）
- ❌ 段落级分块策略（512 token 滑动窗口）
- ❌ user_notes / tier_definitions 向量化管线
- ❌ 双源检索引擎（MiniSearch 全文 + 向量 cosine 合并排序）
- ❌ 引用溯源（AI 回复中脚注标注 + 跳转）
- ❌ 文献管理 UI

**必要性评估**：🔴 高。RAG 是将 AI 从"通用对话"升级为"基于用户数据回答"的关键能力。在学术研究场景中，"AI 引用你的笔记和文献"的价值极高——这是解语区别于 ELAN/FLEx 的核心差异化方向之一。

**业内参考**：
- Obsidian Smart Connections：笔记级 RAG，嵌入 + 语义搜索 + 上下文注入
- Notion AI：文档库级 RAG，支持引用来源
- Cursor：代码库级 RAG，索引整个仓库
- 建议路径：先实现 utterance + notes 双源 RAG（复用已有 EmbeddingService），PDF 管线作为二期

#### 2.2 F12 Leipzig 规则提示 + Auto-Gloss — ❌ 未实现

**计划要求**：缩写合法性提示 + 分隔符一致性检查 + token→lexeme 精确匹配自动填充 + 多义消歧 + 频率词表。

**现状**：
- `ai_tasks` 表定义了 `auto_gloss` taskType，但无对应执行代码
- `abbreviations` 表已有 `isLeipzigStandard` 字段，但无校验逻辑
- `lexemes` 表 + `token_lexeme_links` 表结构完整，但无 auto-gloss 匹配引擎

**必要性评估**：🔴 高。Leipzig 标注规则是语言学行业标准，auto-gloss 是所有 fieldwork 软件（FLEx、Fieldworks、SayMore）的核心能力。**这是解语作为语言学工具的必备功能**，而非 AI 增强。

**业内参考**：
- FLEx（SIL）：auto-gloss 是核心卖点——输入词形自动匹配词典、填充词素分析和 gloss，是 morphological parsing + 词典查找的结合
- Toolbox（SIL）：类似的 lookup + interlinearize 功能
- ELAN：无此能力（ELAN 定位为标注而非分析）
- 建议路径：先做精确匹配（token.form === lexeme.lemma），再做前缀/模糊匹配，复用 EmbeddingSearchService 做语义消歧

#### 2.3 F10 MFA/WebMAUS/WhisperX 音频对齐 — ❌ 未实现

**计划要求**：三通道对齐（MFA 精准 + WhisperX 快速 + WebMAUS 远程备选）+ 差分置信度 UI + ASR 双轨开关。

**现状**：完全未实现。无任何 forced alignment 代码。

**必要性评估**：🔴 高。Forced alignment 是从手工标注到半自动标注的关键技术跳板。计划书引用 Chodroff et al. (2025) 实证数据表明 MFA 在低资源语言上表现最优。**但实现依赖 Python 环境（MFA 基于 Kaldi），纯浏览器无法运行**——需要 Tauri sidecar 或远程服务。

**业内参考**：
- ELAN：集成 WebMAUS（远程 BAS 服务）
- Praat：手动对齐 + 脚本自动化
- ELPIS（ANU + CoEDL）：Kaldi/ESPnet pipeline，专为语言学家设计的 ASR 训练界面
- WhisperX：Python 库，需服务端部署
- 建议路径：Phase 3 优先接入 WebMAUS 远程 API（最小实现成本），Tauri 2 sidecar+MFA 作为桌面版增强

### 🟡 中必要性 — 提升使用效率和专业度

#### 2.4 F31 动态 System Message 自动切换 — ⚠️ 部分完成

**计划要求**：按当前页面自动切换 AI 人设（转写页→语音学家、gloss 页→形态学家、校审页→审校员）。

**现状**：3 种 persona 已定义（`AI_SYSTEM_PERSONAS`），但切换需用户手动操作，未实现 page-aware 自动切换。

**必要性评估**：🟡 中。自动切换可减少用户认知负担，但手动选择不构成功能阻断。

**实现成本**：低。监听当前页面路由 → 映射到 persona → 注入 system prompt。约 20 行代码。

#### 2.5 F8 任务调度队列 — ⚠️ Schema 存在，执行器缺失

**计划要求**：任务状态机（pending→running→success→failed→cancelled）+ 任务队列 + Web Worker 执行器 + 超时/取消/重试。

**现状**：`ai_tasks` 表定义完整（taskType/status/target/modelId/errorMessage），EmbeddingService 已在使用此表进行任务追踪。但无通用的**任务队列调度器**——每种 AI 任务（embedding 构建、未来的 auto-gloss、ASR 等）目前独立管理生命周期。

**必要性评估**：🟡 中。当前只有 embedding 一种长时 AI 任务，独立管理足够。当 auto-gloss、ASR、RAG 索引等任务增多后，统一调度器的价值才会凸显。

#### 2.6 F11 一键回滚 UI — ❌ 缺前端入口

**计划要求**：结合 SnapshotService 提供一键回滚功能的 UI 入口。

**现状**：SnapshotService 基础能力存在，audit_logs 记录完整，但无"回滚到某次 AI 操作前"的前端交互。

**必要性评估**：🟡 中。人工确认机制（破坏性操作双重确认）已经降低了误操作风险。一键回滚是安全兜底而非高频功能。

#### 2.7 多步 Agent Loop — ❌ 未实现

**计划和业内趋势要求**：AI 执行工具后观察结果 → 决定下一步操作 → 循环直至任务完成。

**现状**：当前为单步工具调用（一次对话触发一个工具）。AI 无法在执行工具后根据结果继续推理。

**必要性评估**：🟡 中。单步调用覆盖当前场景（用户说"创建翻译层"→ AI 创建 → 完毕）。当需要复杂操作链（"扫描所有未标注语段，为每个匹配的词自动填充 gloss"）时，agent loop 才成为刚需。

**业内参考**：
- OpenAI Assistants API：原生支持多步 tool use loop
- Claude tool use：支持多步对话
- LangChain Agent：ReAct pattern
- 建议路径：在 auto-gloss（F12）实现时同步引入 agent loop 机制

#### 2.8 Structured Output / Function Calling Schema — ❌ 未实现

**现状**：工具调用通过文本解析（`parseToolCallFromText()` 从回复中提取 JSON），依赖 system prompt 指导模型输出格式。

**业内参考**：
- OpenAI Function Calling：通过 `tools` 参数定义 JSON Schema，模型保证输出符合 schema
- Anthropic Tool Use：类似机制
- 建议路径：当 Provider 支持原生 function calling 时，利用 API 参数而非文本解析。可在 LLMProvider 接口扩展 `tools` 选项。

**现状**：无 pdf.js 集成。对田野语言学家有价值但非核心编辑流程的一部分。

#### 2.10 低资源语言专用模型集成（Glot500-m / ByT5 / MMS）

**现状**：计划中详细调研了这些模型的适用场景，但当前无集成。嵌入层使用 multilingual-e5-small 覆盖主流语言。

**评估**：长期方向。当目标语言不在 e5-small 覆盖范围内时才需要。

#### 2.11 F19 搜索/Concordance KWIC 视图

**现状**：搜索替换已完成，但无 Concordance 视图（KWIC 格式的关键词上下文窗口展示）。

**评估**：语言学研究标准功能，但可独立于 AI 实现。

#### 2.12 Provider 自动 Failover

**现状**：无。当首选 Provider 不可用时需用户手动切换。

**评估**：对单用户桌面工具影响较小。

#### 2.13 对话分支 / Conversation Branching

**现状**：线性对话历史，无分支。

**评估**：学术探索场景有价值（"如果换个角度分析这个语段…"），但非核心。

## 六、结论

解语 AI 功能在 **LLM 多 Provider 接入、对话式交互、工具调用安全、嵌入与语义搜索** 方面已达到甚至超越业内水平（相对于语言学工具领域），部分功能（ProjectObserver 阶段探测、Prompt Lab）属行业首创。

核心缺口集中在三个方向：
1. **语言学基础能力**（Auto-Gloss、Leipzig 规则）——这是"语言学工具"身份的必备项
2. **RAG 知识库**——这是"AI 分析伙伴"定位的关键能力
3. **音频对齐**（Forced Alignment）——这是从"手工标注"到"半自动标注"的技术跳板

建议优先补齐 Auto-Gloss + minimal RAG + 动态 Persona 切换，使 AI 功能从"能对话的编辑器"进化为"能分析数据的语言学伙伴"。
