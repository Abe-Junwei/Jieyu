# 用户建议评估 — 闭环负样本数据飞轮 + 降级路径 UX 细化

> **评估日期**：2026-05-05  
> **建议来源**：用户反馈（路线图 v2 审查后的补充建议）  
> **评估维度**：产品价值、技术可行性、与路线图契合度、工作量、风险

## 建议一：闭环负样本数据飞轮

> **原建议**：将用户标记为 👎 的线上真实样本（如幻觉引用），自动脱敏并沉淀到 P1 扩充的 30+ 条测试集中的"对抗样本"或"错配样本"中，实现自动化测试集的有机生长。

### 1.1 评估结论：✅ **强烈推荐吸收**

| 维度 | 评分 | 说明 |
|------|------|------|
| 产品价值 | ★★★★★ | 工业级 AI 系统的标准做法（OpenAI、Anthropic、Google 均用用户反馈改进模型和评测集） |
| 技术可行性 | ★★★★☆ | 脱敏逻辑需要仔细设计，但 pipeline 本身不复杂 |
| 与路线图契合度 | ★★★★★ | 自然衔接 P1（评测 30+ 条）和 P2（用户反馈闭环 👍/👎） |
| 工作量 | ★★★☆☆ | 中等（脱敏规则 + 自动化 pipeline + 分类模型） |
| 风险 | ★★☆☆☆ | 低（隐私合规可控，失败不影响主链） |

### 1.2 为什么值得做

**真实 > 人工**：
- 当前路线图规划的对抗样本（越狱、提示注入、越权调用）是"人工设计"的，覆盖面有限。
- 用户 👎 的样本来自真实使用场景，比人工设计的 case 更贴近实际故障模式。
- 对于语言资料工作平台，专业领域的幻觉（如编造不存在的 gloss、错误的时间码引用）很难被通用对抗样本覆盖。

**数据飞轮的复利效应**：
- 第 1 个月：30 条人工 case + 5 条用户反馈 case。
- 第 3 个月：30 条人工 case + 40 条用户反馈 case。
- 第 6 个月：30 条人工 case + 120 条用户反馈 case，形成"测试集有机生长"。

**与竞品的差异化**：
- NotebookLM、Cursor 等产品也有用户反馈，但很少将反馈自动沉淀为领域专用评测集。
- Jieyu 如果在语言学领域率先建立「真实用户反馈 → 自动脱敏 → 领域评测集」的飞轮，是独特的技术品牌资产。

### 1.3 落地方案

#### 数据流设计

```
用户点击 👎 
  → 触发 "反馈采集" 对话框（可选填原因：幻觉引用 / 错误结论 / 无关回答 / 其他）
  → 样本进入 "反馈队列"（IndexedDB / Supabase，带 projectId、conversationId、messageId）
  → 夜间批处理：
      1. 脱敏引擎运行（去除 languageCode、projectName、具体语段文本、说话人名称）
      2. 分类模型运行（判断属于：hallucination / wrong_scope / missing_evidence / contradiction / other）
      3. 人工审核（可选，研究员每周 review 一次）
      4. 通过的样本写入 agent-evals suite（suite.v2.json 的 `user_feedback` 分区）
  → CI 下次运行时自动包含新样本
```

#### 脱敏规则（关键）

| 敏感字段 | 脱敏方式 | 示例 |
|----------|----------|------|
| 项目名称 | 替换为 `[PROJECT_NAME]` | "Navajo 语料库" → "[PROJECT_NAME]" |
| 语言名称/ISO | 替换为 `[LANGUAGE]` | "Navajo (nav)" → "[LANGUAGE] (iso-xxx)" |
| 具体语段文本 | 替换为 `[SEGMENT_TEXT]` | "Shí éí..." → "[SEGMENT_TEXT]" |
| 说话人名称 | 替换为 `[SPEAKER]` | "Speaker_A" → "[SPEAKER]" |
| 时间码 | 保留相对格式，替换绝对值 | "00:03:24.500" → "[TIMESTAMP]" |
| 词典词条 | 替换为 `[LEXEME]` | "shí" → "[LEXEME]" |
| EvidencePacket.sourceId | 替换为随机 UUID | 保留结构，不保留真实 ID |

**约束**：脱敏后样本必须仍可复现原始 bug（即保留 prompt 结构、tool call 序列、模型响应格式，只替换具体值）。

#### 自动分类模型（轻量）

不需要训练专用模型，用规则 + 低成本 LLM 即可：

```typescript
// src/ai/eval/userFeedbackClassifier.ts
export async function classifyUserFeedback(
  message: AiMessage,
  feedbackReason: string,
): Promise<UserFeedbackCategory> {
  // 规则层（快路径）
  if (feedbackReason === 'hallucinated_citation') return 'hallucination';
  if (feedbackReason === 'wrong_scope') return 'wrong_scope';
  
  // LLM 层（慢路径，低成本模型）
  const prompt = `判断以下 AI 回答被用户标记为 👎 的原因类别：
  用户问题：${message.userQuery}
  AI 回答：${message.assistantResponse}
  用户反馈：${feedbackReason}
  可选类别：hallucination / wrong_scope / missing_evidence / contradiction / unrelated / other`;
  
  const result = await callCheapModel(prompt);
  return parseCategory(result);
}
```

## 建议二：细化降级路径的 UX（手动接管 / 单步执行）

> **原建议**：当发生服务降级（如复杂的组合任务降级为单一检索）时，不仅要提供文案解释，还应在侧边栏提供一个明确的"手动接管（Manual Override）"或"单步执行"按钮，赋予用户最高控制权。

### 2.1 评估结论：✅ **值得吸收，但建议放在 P2 并与 Plan-and-Execute 深化一起交付**

| 维度 | 评分 | 说明 |
|------|------|------|
| 产品价值 | ★★★★★ | 与 Jieyu「可审批、可复盘」定位高度契合，是差异化体验 |
| 技术可行性 | ★★★☆☆ | 需要 UI 状态机支持暂停-恢复，workflow 引擎支持 step-by-step |
| 与路线图契合度 | ★★★★☆ | 与 P2 Plan-and-Execute 深化和用户可解释性闭环自然衔接 |
| 工作量 | ★★★★☆ | 高（需改 UI 状态机 + workflow 执行引擎 + 新增交互组件） |
| 风险 | ★★★☆☆ | 中（交互复杂度高，若设计不当可能让用户困惑） |

### 2.2 为什么值得做

**Jieyu 的产品定位是「证据型、可审批」**：
- 通用 AI 聊天产品（ChatGPT、Claude）降级时只需要一句"我降级了"的文案。
- Jieyu 处理的是语言资料工作流，用户是专业研究者，**需要精确控制每一步**。
- 当组合 workflow 降级时，研究者可能仍希望看到分步结果（如"虽然系统降级了，但我仍想先看标注 QA 的结果"）。

**"手动接管"是 Human-in-the-Loop 的最高形态**：
- 当前路线图已有 `allow` / `ask` / `deny` 三态，但缺少 `pause_and_review`（暂停并审阅）。
- 手动接管填补了 "On-the-Loop"（默认自主运行，人类可干预）模式的最后一环。

### 2.3 为什么建议放在 P2（而非 P1）

**依赖前提**：
1. **Plan-and-Execute 需要先有真正的 step 结构**：P1 的 Plan-and-Execute 若按保守方案降级为"伪组合"（顺序调用两个 workflow），则没有真正的"step"可让用户接管。手动接管的前提是系统内部有显式的 step 状态机。
2. **TaskRunner 需要支持暂停-恢复**：当前 TaskRunner 的 `cancel()` 只是 abort，没有 `pause()` / `resume()` 能力。
3. **UI 需要支持异步暂停态**：当前 AI 侧边栏的交互模型是"请求 → 等待 → 结果"，没有"请求 → step1 完成 → 等待用户 → step2"的状态。

**如果在 P1 强行实现**：
- 工作量爆炸：需要同时做 Plan-and-Execute + TaskRunner 重构 + UI 状态机 + 手动接管交互。
- 交互设计不成熟：在没有真实用户验证 step-by-step 工作流之前，"手动接管"按钮的交互可能设计错误。

### 2.4 落地方案（P2）

#### 降级场景与手动接管映射

| 降级场景 | 系统行为 | 手动接管选项 | 用户可执行操作 |
|----------|----------|-------------|---------------|
| Plan-and-Execute 组合任务降级为单检索 | 跳过组合，直接回答 | 🎮 "分步执行" | 强制启用组合 workflow，逐 step 运行 |
| Reflection 检测到 findings 矛盾 | 显示警告但输出结果 | 🎮 "审查矛盾" | 打开 findings 列表，逐条查看矛盾点 |
| LLM-as-Judge citation 评分 <3 | 输出结果但标记低质量 | 🎮 "手动验证引用" | 打开 EvidencePacket 列表，逐条验证 source |
| Workflow 成本偏离 >50% | 继续执行但标记异常 | 🎮 "简化查询" | 缩小 scope（project → current_track）后重试 |
| RAG 无结果降级为通用回答 | 显示"来源不足" | 🎮 "扩大检索范围" | 增加 RAG topK 或扩展 source set 后重试 |

#### UI 设计草案

```
┌─────────────────────────────────────────┐
│  🤖 AI 助手                              │
│                                         │
│  由于当前任务复杂度超出预期，系统已降级   │
│  为单一检索模式。                        │
│                                         │
│  [🎮 分步执行]  [📋 查看原计划]         │
│                                         │
│  ── 或接受降级结果 ──                   │
│                                         │
│  [以下是基于项目语料的回答...]            │
│                                         │
└─────────────────────────────────────────┘
```

#### 状态机设计

```typescript
// 新增状态
export type AiChatWorkflowState = 
  | 'idle'
  | 'planning'
  | 'running_step'      // 正在执行某个 step
  | 'step_completed'    // step 完成，等待用户决策
  | 'user_reviewing'    // 用户正在审阅 step 结果
  | 'user_overridden'   // 用户手动接管，修改参数后重试
  | 'completed'
  | 'failed';

export interface StepReviewAction {
  kind: 'proceed' | 'retry_with_scope' | 'skip' | 'abort' | 'manual_override';
  scope?: 'project' | 'current_track' | 'current_scope';
  reason?: string;
}
```

| 模块 | 人天 | 说明 |
|------|------|------|
| AI 侧边栏状态机扩展（暂停/恢复/step 展示） | 3.0 | 核心工作 |

| TaskRunner pause/resume 支持 | 2.0 | 需改 TaskRunner 核心 |

### 2.5 与路线图的衔接

**P2 交付项 4（用户可解释性闭环）增加子项**：
```
4f. 降级路径手动接管：
    - 所有降级场景（Plan 降级、Reflection 警告、Judge 低分、成本偏离、RAG 无结果）
      在文案解释之外，提供「手动接管」入口。
    - 用户可选择：分步执行 / 审查矛盾 / 手动验证引用 / 简化查询 / 扩大检索范围。
    - 接管操作不绕过 policy / sandbox / confirm / audit，仍走现有门禁链路。
```

**P2 交付项 7（Plan-and-Execute 深化）增加约束**：
```

      支持用户暂停、审阅、修改参数后继续。
```

## 3. 两项建议的总体评估

| 建议 | 是否吸收 | 放在哪个阶段 | 关键依赖 | 新增工作量 |
|------|----------|-------------|----------|-----------|
| 闭环负样本数据飞轮 | ✅ **强烈推荐** | P2（与 LLM-as-Judge 深化并行） | 👍/👎 UI + 脱敏引擎 + 分类器 | ~7 人天 |
| 降级路径 UX 细化（手动接管） | ✅ **值得吸收** | P2（与 Plan-and-Execute 深化 + 用户可解释性闭环一起交付） | Plan-and-Execute 真实 step 结构 + TaskRunner pause/resume | ~12 人天 |

**对总周期的影响**：
- 飞轮建议：~7 人天，可融入 P2 现有交付项，不显著延长周期。
- 手动接管建议：~12 人天，若 P2 已有 27 人天工作量，新增后变为 ~39 人天，8 周刚好满负荷（无缓冲）。
- **建议**：如果资源允许，两项都吸收；如果资源紧张，**优先飞轮**（工作量低、价值高、无阻塞依赖），手动接管可延后到 P2 后期或 P3。
