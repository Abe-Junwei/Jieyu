> **审查日期**：2026-05-05

## 1. 执行摘要

**总体判断**：路线图在「治理收口」维度（证据、门禁、安全矩阵）设计合理且优先级正确；但在「智能体认知架构」维度（规划能力、长期记忆、协议标准化、评测深度）与工业级标杆仍有结构性差距。

| 维度 | 路线图评分 | 核心结论 |
|------|-----------|----------|
| 治理与证据 | ★★★★☆ | P0→P1→P2 节奏合理，不做项清醒 |
| 运行安全 | ★★★★☆ | F4 矩阵 + sandbox + confirm/audit 已对齐工业「最小权限+人在环」原则 |
| 产品化闭环 | ★★★☆☆ | EvidencePacket + VerticalWorkflowRegistry 方向正确，但规划层（Planning）缺失 |
| 评测体系 | ★★☆☆☆ | 6 条 case 远不足；缺少对抗样本、LLM-as-Judge、边界覆盖 |
| 架构演进 | ★★★☆☆ | ReAct 单步循环（maxSteps=6）胜任当前任务，但垂直工作流需要 Plan-and-Execute 分层 |
| 协议与生态 | ★★☆☆☆ | 未对接 MCP / A2A，工具生态可能在未来 6-12 个月出现兼容债务 |
| 可观测性 | ★★★☆☆ | release evidence 体系完整，但缺少产品化 AI runtime dashboard 和输出质量评估 |

## 2. 项目现状速览（代码级）

### 2.1 核心执行架构

| 构件 | 文件锚点 | 当前能力 | 工业对标状态 |
|------|----------|----------|--------------|
| Agent Loop | `src/ai/chat/agentLoop.ts` | ReAct 风格，maxSteps=6，tokenBudgetWarningThreshold=12000 | 基础 ReAct ✅ |
| Durable Task | `src/ai/tasks/TaskRunner.ts` | IndexedDB 持久化、checkpoint、heartbeat、cancel、指数退避重试 | 轻量 Temporal-lite ✅（但缺少跨会话精确恢复） |

| 安全矩阵 | `src/ai/policy/aiToolPolicyMatrix.ts` | 20+ 工具的风险分级（low/medium/high）、destructive 标记、confirm 模式、auditReasonCodes | 最小权限 ✅ |

### 2.2 已关闭里程碑（T1–T4 + F4 Batch A–C）

- **T1**：Durable agent loop 可操作闭环（取消/重试/冷启动 reconcile）
- **T2**：执行层 policy 与权限隔离 v1
- **T3**：变更事务化（`AiChangeTransactionPreviewV1` + dry-run + commit/审计顺序）

**结论**：地基打得扎实。当前处于「治理框架就绪 → 产品化扩展」的转折期。

## 4. 工业级架构对标：八大差距

### 差距 1：规划层（Planning）缺失 — 中高风险 ⚠️

**现状**：`agentLoop.ts` 是 ReAct 单步循环，每步至少调用一次 LLM，`maxSteps=6`。垂直工作流（如 annotation_qa → lexeme_candidates 的链式任务）没有分层规划。

**工业标杆**：
- LangGraph：支持 Planner → Executor → 反思 → 重规划的子图
- CrewAI：任务分解（Task Decomposition）是核心抽象
- OpenAI Agents SDK：Handoff 原语支持多 Agent 路由

**影响**：当用户请求跨越多个 workflow（如"帮我检查这段转写的标注质量，并提出词典候选"），当前架构依赖 LLM 在 6 步内即兴编排，没有显式的计划-执行分离。

**建议**：
1. **P1 增量**：在 `VerticalWorkflowRegistry` 中增加 `composedWorkflow` 类型，支持「plan → [workflow A, workflow B] → synthesize」模式
2. **P2 深化**：对长周期 workflow（田野笔记、批量 QA）引入显式 `AgentRunState` 状态机中的 `planning` 子状态
3. 不引入通用 planner，但为高频组合 workflow 写死编排模板（如 `annotation_qa → lexeme_candidates`）

### 差距 2：长期记忆（Long-term Memory）不完整 — 中风险 ⚠️

**现状**：
- 短期记忆：`sessionMemory`（当前会话上下文）
- 外部知识：RAG / `EmbeddingSearchService`（项目语料）
- 长期记忆：`backgroundMemoryExtractor`（会话内后台抽取）+ `userDirectiveExtractor`（用户指令提取）

**工业标准三层记忆**：
| 层级 | 项目现状 | 工业标杆 |
|------|----------|----------|
| 短期 | ✅ `sessionMemory` | 同 |
| 长期 | ⚠️ 仅当前会话；跨会话记忆未结构化 | GraphRAG / 向量数据库 + 结构化用户画像 |
| 外部 | ✅ RAG / EmbeddingSearchService | 同 |

**问题**：`backgroundMemoryExtractor` 只写入当前会话的 `sessionMemory`，用户下次打开新项目时记忆不携带。研究显示：无持久记忆的 Agent 多步任务失败率高达 **67%**。

**建议**：
1. **P1**：`backgroundMemoryExtractor` 输出同时写入 `project` 级别的持久存储（如 IndexedDB 中的 `project.ai_memory` 表），带 TTL 和遗忘策略
2. **P2**：引入「记忆整合」机制——定期将散落的 user directive、采纳/拒绝记录总结为结构化「用户偏好卡片」，在 RAG 时作为 system prompt 注入
3. 不做云端多租户记忆（与路线图「不做项」一致）

### 差距 3：评测体系深度不足 — 高风险 🔴

**现状**：`scripts/agent-evals/suite.v1.json` 只有 **6 条 case**（修订记录多次提到 `agent-evals 6/6`），覆盖 build / policy / runtime / release-evidence 四个类别。

**工业标杆**：
- LangSmith：在线评估（online eval）+ LLM-as-Judge + 程序化评估
- Maxim AI：对抗提示集、边界覆盖、多轮会话漂移检测
- 行业数据：67% 的生产 Agent 故障由用户发现，而非监控系统

**差距量化**：
| 维度 | 当前 | 工业级建议 |
|------|------|-----------|
| 总 case 数 | ~6 | 50–100（P1），200+（P2） |
| 对抗样本 | 0 | 至少 20 条（越狱、提示注入、越权工具调用） |
| 负样本（误阻断/误放行） | 2 条线（destructive / no-silent-write） | 每类 capability 至少 3 条 allow/ask/deny |
| 跨语言/长会话 | 0 | 至少 10 条多轮漂移 case |
| LLM-as-Judge | 0 | 对 QA 回答、evidence citation 质量做自动评分 |
| 输出质量指标 | 0 | citation accuracy、hallucination rate、adoption rate |

**建议**：
1. **P0 即刻**：将 agent-evals 从 6 条扩展到 **30 条**（新增：对抗 5 条 + 负样本 10 条 + 跨语言长会话 5 条 + 垂直 workflow 4 条）
2. **P1**：引入轻量 LLM-as-Judge——用低成本模型（如 Gemini Flash）对 Agent 输出的 evidence citation 准确性和回答相关性做 1–5 分评分
3. **P2**：建立「误放行率 / 误阻断率 / 恢复成功率」的自动趋势图，纳入 release evidence

### 差距 4：未对接 MCP / A2A 协议 — 中长期风险 ⚠️

**现状**：工具系统完全自定义（`toolCallSchemas.ts` + `aiToolPolicyMatrix.ts` + 本地执行器）。没有 MCP Client/Server 形态，也没有 Agent-to-Agent 通信能力。

**行业趋势（2025–2026）**：
- MCP：SDK 月下载量超 **9700 万**，OpenAI/Google/Microsoft/AWS 均已支持，Anthropic 捐赠给 Linux Foundation
- A2A：Google 发起，150+ 企业支持，已捐赠给 Linux Foundation
- **双协议栈**正在成为事实标准：MCP 负责工具/数据垂直接入，A2A 负责 Agent 水平协作

**对 Jieyu 的影响**：
- 短期内无直接风险（桌面端封闭生态）
- 中期风险：语言学界可能出现 MCP Server 形态的工具生态（如 ELAN/FLEx 插件、语言数据库接口），不兼容将丧失扩展性
- 长期风险：若解语想开放插件或与其他学术工具链集成，需重写工具接入层

**建议**：
1. **P1（轻量）**：在 `AiToolRegistry` shadow 阶段，将工具 schema 输出兼容 MCP Tool 格式（即 JSON Schema 结构对齐），为未来留接口
2. **P2（可选）**：评估是否将 `VerticalWorkflowRegistry` 暴露为 MCP Server（供外部学术工具调用解语 Agent）或 MCP Client（调用外部语言数据库）
3. **不纳入阻塞项**：当前路线图正确地将此列为「不做项」范畴，但建议在「不做项」中增加注释：「暂不实现，但 schema 层预留兼容」

### 差距 5：可观测性缺少「输出质量」维度 — 中风险 ⚠️

**现状**：
- 工程可观测：aiTrace、provider health、tool failure、cost guard、latency
- 发布证据：release evidence 体系（gate_script / pass/fail / trend）
- **缺少**：Agent 输出的质量评估（正确性、相关性、引用准确性）

**工业标杆**：
- LangSmith：在线评估 + 人类反馈 + LLM-as-Judge 三轨并行
- Arize Phoenix：RAG 引用准确率、幻觉检测
- Maxim AI：毒性检测、越狱检测、策略违规

**建议**：
1. **P1**：对 RAG 检索增加「引用准确率」指标——统计 EvidencePacket 中 sourceId 与实际 RAG snippet 的匹配率
2. **P2**：建立「用户反馈闭环」——在 UI 中为每条 AI 回答增加 👍/👎 按钮，反馈数据进入评测集和模型选择决策
3. 与「用户可解释性闭环」（路线图 P2）结合：不仅解释「为什么阻断」，还要解释「为什么给出这个回答」（引用了哪些 source）

### 差距 6：Agent Loop 缺少 Reflection / 自我改进 — 中风险 ⚠️

**现状**：`agentLoop.ts` 没有显式的反思节点。工具执行失败后进入 `error` 状态，但没有「分析失败原因 → 调整策略 → 重试」的循环。

**工业标杆**：
- LangGraph：Reflection 是核心模式之一（生成 → 批评 → 改进）
- CrewAI：内置任务回顾（Task Review）机制
- 研究显示：Reflection 可减少长周期任务错误累积 **30–50%**

**建议**：
1. **P1（轻量）**：对 `annotation_qa` 和 `lexeme_candidates` workflow，在输出 findings 后增加一个「自检」步骤：用低成本模型检查 findings 之间是否有矛盾、evidence 是否充分
2. **P2（深化）**：在 `TaskRunner` 的 retry 路径中，不仅指数退避重试，还要将前一次失败原因注入 prompt，让 LLM 调整参数后重试

### 差距 7：Cost Guard 未进入阻塞性门禁 — 低风险 ⚠️

**现状**：`featureFlags` 中有 cost guard 相关逻辑，perf 脚本存在，但路线图明确说「不调整阈值到阻塞模式」（PR-5 说明）。

**问题**：工业级部署中，成本失控是常见故障源。当前策略是「趋势报告接入发布评审」（P1），但不阻塞发布。

**建议**：
1. 保持当前非阻塞策略（桌面端个人用户场景合理）
2. **P1 增加**：当单次请求 token 超过历史 P95 的 2 倍时，自动触发 `warning` 级告警并记录到 release evidence
3. **P2 增加**：为每个 workflow 建立「预期成本基线」（如 annotation_qa 平均 2k token/次），偏离 >50% 时触发审计标记

### 差距 8：并发与并行执行 — 低风险 ⚠️

**现状**：`TaskRunner` 支持 `concurrency` 参数，但 Agent Loop 是单步串行。没有并行工具调用（如同时查询多个 RAG source）。

**工业标杆**：
- LangGraph：原生支持并行节点执行
- OpenAI Agents SDK：支持并行工具调用

**建议**：
1. **P1（轻量）**：对 `CorpusSourceSet` 的多个 source 类型（segment + document + note），支持并行 RAG 检索
2. **P2（可选）**：多个独立 workflow 的并发执行（如同时运行 annotation_qa 和 timeline_hygiene）

### 6.4 长期记忆跨会话持久化（2 周）

扩展现有 `backgroundMemoryExtractor`：

```typescript
// src/ai/memory/backgroundMemoryExtractor.ts 增量
async function persistCrossSessionMemory(
  projectId: string,
  memory: ExtractedMemory,
): Promise<void> {
  const db = getDb();
  await db.projectAiMemory.put({
    projectId,
    id: memory.id,
    content: memory.content,
    category: memory.category, // 'user_preference' | 'workflow_pattern' | 'rejected_suggestion'
    createdAt: new Date().toISOString(),
    ttlDays: memory.category === 'user_preference' ? 90 : 30,
  });
}
```

**遗忘策略**：TTL 到期自动清理；每周一次「记忆整合」将散落记忆总结为结构化卡片。

## 7. 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 评测集扩容导致 pre-merge 时间变长 | 中 | 中 | 分级执行：P0 跑全量 30 条（<2min），P1 跑对抗 5 条（<30s） |
| Plan-and-Execute 增加 token 消耗 | 中 | 低 | 只对组合 workflow 启用，单 workflow 保持原路径；synthesizer 用低成本模型 |
| MCP 兼容预留过度设计 | 低 | 低 | 只做 schema 转换，不引入 MCP runtime 依赖 |
| 长期记忆持久化导致 IndexedDB 膨胀 | 中 | 中 | TTL + 整合 + 用户可手动清空；设置单项目 1000 条上限 |
| 资源有限，P1 项过多无法完成 | 高 | 高 | 严格按「建议前置」表执行，其余维持原路线图；每两周 review 一次优先级 |
