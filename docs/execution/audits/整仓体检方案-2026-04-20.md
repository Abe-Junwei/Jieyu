# 整仓体检方案（运行手册）

目标：在**不依赖单次大 diff / simplify** 的前提下，用仓库已有脚本与门禁，对全仓做**可重复、可分级、可留痕**的健康检查。

---

## 1. 分级策略（按频率与成本）

| 级别 | 名称 | 典型耗时 | 适用场景 |
| --- | --- | --- | --- |
| **L0** | 极轻量 | 约 1–3 分钟 | 每次保存 / 预提交（可选 hook） |
| **L1** | 日常 PR | 约 5–15 分钟 | 每个 PR、每日开发主干 |
| **L2** | 标准全量 | 约 20–60 分钟 | 合并日前、周节奏、大重构后 |
| **L3** | 发版 / 深度 | 约 1–3+ 小时 | 版本发布、架构大改、安全审计 |

原则：**默认 CI 至少跑到 L1；发布前跑到 L2；L3 按版本计划排期**。同一目录不必每天跑满 `npm test` 中的全部 CSS 子项，可按团队约定把 L1 收窄为 `typecheck + vitest`（见下「可选瘦身」）。

---

## 2. L0 — 极轻量（本地最快反馈）

在改动涉及 TS/文案/文档时优先跑：

```bash
npm run typecheck
npm run check:docs-governance
```

若本次提交只动文案与 i18n，可再加（比全量 i18n guard 快）：

```bash
npm run check:i18n-hardcoded:changed
```

**通过标准**：命令 exit code 为 0。

---

## 3. L1 — 日常 PR（推荐 PR 门禁底线）

在 L0 基础上增加**全量单测**（不跑 `npm test` 里嵌套的全套 CSS 闸门时，可用下面「瘦身版」）：

**完整 L1（与当前 `npm test` 前大半部分对齐的意图）** — 若团队已在 CI 使用 `npm test`，则 L1 可直接等同：

```bash
npm test
```

**瘦身版 L1**（当 `npm test` 过长、且本 PR 未改样式时临时采用）：

```bash
npm run typecheck
npm run check:docs-governance
npm run check:architecture-guard
npx vitest run
```

**通过标准**：全部 0 exit；Vitest 无失败用例。

---

## 4. L2 — 标准「整仓」体检（周 / 合并日前）

在 L1 基础上补齐**构建体积与生产依赖审计**（仓库已编排）：

```bash
npm run validate
```

等价于（便于理解成本）：

```bash
npm run typecheck
npm run gate:acoustic
npm run test
npm run build:guard
```

其中 `build:guard` = `build` + 产物画像 + `check:build-budgets` + `audit:prod`（见 `package.json`）。

**可选加跑**（按需启用，不必每周全跑）：

```bash
npm run report:docs-link-debt
npm run report:architecture-hotspots
npm run check:i18n-hardcoded:summary
```

**通过标准**：`validate` 成功；可选报告中的**新增**致命问题需登记 issue 或记入本目录简短附录。

---

## 5. L3 — 发版 / 专题深度

在 L2 已通过的前提下，按**本次发布触及的域**叠加「域门禁」（`package.json` 中已有 `gate:*`）：

| 域 | 示例命令 | 说明 |
| --- | --- | --- |
| 时间轴 / CQRS | `npm run gate:timeline-cqrs-full-migration` | 重、仅大版本或迁移相关时 |
| 协作云 | `npm run gate:collaboration-cloud` | 动到 Supabase / 同步协议时 |
| 面板 / CSS 架构 | `npm run gate:panel-phase1` | 大 UI  refactor 时 |
| 扩展 | `npm run gate:m7-extension` 或 `:shadow` | 扩展宿主相关变更时 |
| 声学 | `npm run gate:acoustic` | 声学链路与性能门禁 |

**端到端（可选）**：

```bash
npm run test:e2e
```

**性能基线（可选）**：

```bash
npm run perf:ai
npm run perf:track
```

**通过标准**：所选 `gate:*` 与 E2E 全部 green；性能基线若出现回归，需在 `docs/execution/audits/` 留一条日期化说明或 ADR 备注。

---

## 6. 人工烟测（无法被脚本替代）

与产品路径强相关的手动清单，建议继续引用各功能方案中的「放行前验证」小节（例如转写对照方案 §14），每次发版至少抽 **30–60 分钟** 覆盖：

- 转写：横向 / 纵向对照 / 纯文本壳关键路径；
- 协作：若本版含云同步，走冲突与恢复；
- 地图代理：若动到环境变量，验证代理与回退。

结果可简短记在 `docs/execution/release-gates/` 或本目录一条「烟测记录」markdown（非强制）。

---

## 7. 留痕与索引

- **自动化结论**：CI 日志为主；必要时将失败摘要复制到 issue。
- **阶段性盘点**：长报告仍放在 `docs/execution/audits/`（与本手册同级）。
- **长期约束**：从体检中提炼的规则进入 `docs/adr/` 或 `docs/architecture/`。

---

## 8. 与 `simplify` / AI 审查的关系

- **整仓体检**以本手册的 **npm 脚本门禁**为主，保证**可重复、可机器判定**。
- **`/simplify` 或人工审查**适合在「某次 PR 的 diff」或「某子目录」上做代码味道与架构讨论，**不替代** L1/L2。
- 推荐工作流：**开发中 L0 → PR L1 → 合主干前/周 L2 → 发版 L3**。

---

## 9. 一键速查表（复制用）

```bash
# L0
npm run typecheck && npm run check:docs-governance

# L1（瘦身）
npm run typecheck && npm run check:docs-governance && npm run check:architecture-guard && npx vitest run

# L1（完整测试套件，等同 npm test 意图）
npm test

# L2
npm run validate
```
