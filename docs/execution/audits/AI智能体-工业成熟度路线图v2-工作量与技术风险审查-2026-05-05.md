## 1. 执行摘要

**核心结论**：路线图 v2 的方向正确、覆盖全面，但 **P1（4 周 / 15 项交付项）严重超载**，在现有资源下不可能按期完成。若不调整，必然导致质量下降或大面积延期。

| 维度 | 评估 | 说明 |
|------|------|------|
| P0（2 周 / 9 项） | ⚠️ 偏紧但可行 | 需 2–3 人并行，1 人串行会超时 |
| P1（4 周 / 15 项） | 🔴 严重超载 | 仅重量级项就需要 8–10 周，4 周不可能 |

| 次高技术风险 | 🟡 MCP SDK 浏览器兼容性 | `@modelcontextprotocol/sdk` 为 Node.js 设计，浏览器环境需验证 |

## 3. 技术风险矩阵（基于代码验证）

### 3.1 🔴 最高风险：Plan-and-Execute 与 TaskRunner 的结构性冲突

**代码验证**：`src/ai/tasks/TaskRunner.ts`（445 行）

**关键发现**：

1. **不支持子任务嵌套**：
   ```typescript
   // TaskRunner.enqueue() 只是将任务加入队列，run 函数内部无法调用 enqueue
   // run 函数的 context 只提供 heartbeat/checkpoint，没有子任务 enqueue 能力
   run: (context: TaskRunContext) => Promise<TResult>
   ```

2. **checkpoint 不是断点恢复**：
   ```typescript
   // checkpoint 只是将 JSON 存入 IndexedDB
   async checkpoint(taskId: string, checkpoint: TaskRunnerCheckpoint): Promise<void> {
     await db.collections.ai_tasks.update(taskId, {
       checkpointJson: serializeCheckpoint({ ...checkpoint }),
     });
   }
   ```
   任务失败后 `retry()` 会重新 `enqueue` 同一个 input，**重新运行整个 `run` 函数**，不会从 checkpoint 的中间步骤恢复。

3. **没有"继续执行"机制**：
   ```typescript
   // retry() 实现：重新 enqueue 同一个 input
   async retry(taskId: string): Promise<string | null> {
     const retryInput = this.retryInputs.get(taskId);
     // ... 重新 enqueue
   }
   ```

**影响**：路线图 v2 中 P1 交付项 9 描述的「每个 step 是一个子任务，有独立 checkpoint；step 失败时从 checkpoint 恢复，不从头重跑」**在现有 TaskRunner 上无法实现**，除非：

- **方案 A**：重构 TaskRunner，增加子任务支持和断点恢复（估计 2–3 周，影响 T1 里程碑稳定性）。
- **方案 B**：应用层 workaround——组合 workflow 自己维护状态机，用 `localStorage`/`sessionStorage` 存 step 状态，不依赖 TaskRunner checkpoint（估计 3–5 天，但不够"durable"）。
- **方案 C**：降级设计——组合 workflow 只是顺序调用两个单 workflow，失败时整体重试，不做 step 级恢复（估计 1–2 天，但不符合路线图描述）。

**建议**：P1 采用 **方案 C（降级）**，P2 再评估是否做方案 A 或 B。

### 3.2 🟡 高风险：Dexie Schema 升级与 IndexedDB 性能

**代码验证**：`src/db/engine.ts`（1499 行，schema v46）

**关键发现**：

1. **新增表需要完整的 schema 升级链**：
   ```typescript
   this.version(46).stores({ /* ... 所有表定义 ... */ });
   ```
   新增 `project_ai_memories` 需要：
   - 在 `src/db/types.ts` 中定义 `ProjectAiMemoryDocType`
   - 在 `src/db/schemas.ts` 中写 `validateProjectAiMemoryDoc`
   - 在 `engine.ts` 中增加 `project_ai_memories!: Table<ProjectAiMemoryDocType, string>`
   - 在 `engine.ts` 中增加 `this.version(47).stores({ ...project_ai_memories: 'id, projectId, category, createdAt' })`
   - 在 `engine.ts` 末尾增加 collection adapter
   - 升级 `JIEYU_DEXIE_TARGET_SCHEMA_VERSION` 从 46 到 47
   - 可能写 upgrade hook（如果已有数据需要迁移）

2. **IndexedDB 性能历史问题**：代码中有大量 `adapterDexieQueryErrors.ts` 的 fallback 逻辑，说明 IndexedDB 查询降级是已知问题。

3. **启动时 schema 升级可能阻塞 UI**：Dexie 的 `open()` 在 schema 升级时会执行 migration，如果数据量大可能阻塞。

**影响**：长期记忆的 Dexie schema 升级不是简单工作，需要完整走一遍版本升级流程，且要测试大量数据下的启动性能。

**建议**：
- P1 可以先不复用 Dexie，而是用简单的 `localStorage` + JSON 存储项目级记忆（几百条数据的量，localStorage 够用）。
- 如果必须上 Dexie，需要在 dogfood 环境先做 schema 升级测试，确认启动时间无回归。

### 3.3 🟡 高风险：MCP SDK 浏览器兼容性

**代码验证**：`package.json`（无 `@modelcontextprotocol/sdk` 依赖）

**关键发现**：

- `@modelcontextprotocol/sdk` 主要是为 Node.js 设计的，使用 `stdio` 或 HTTP Server 模式。
- MCP Server 在浏览器环境中运行可能遇到：
  - CORS 限制（如果 HTTP 模式）
  - Node.js 内置模块缺失（如 `fs`、`path`、`child_process`）
  - `stdio` 模式在浏览器中不可用

**影响**：Jieyu MCP Server 的实现在浏览器端可能比想象中复杂。可能需要：
- 用 Vite 的 `resolve.alias` 做 polyfill
- 或自己实现 MCP 协议的子集（不引入完整 SDK）
- 或只在 Electron/桌面端运行 MCP Server（如果 Jieyu 有桌面壳）

**建议**：P1 先不引入 `@modelcontextprotocol/sdk`，而是自己实现轻量 MCP Server（HTTP JSON-RPC 子集），只支持 `tools/list` 和 `tools/call` 两个方法。等验证稳定后再决定是否引入完整 SDK。

### 3.4 🟡 高风险：LLM-as-Judge 的 token 成本与延迟

**估算**：
- 每次 AI 回答后调用 Judge：额外 500–1500 token（取决于回答长度）。
- 如果用户一天 50 轮对话，Judge 额外消耗 25k–75k token/天。
- 低成本模型（Gemini Flash）价格虽低，但累积成本不可忽视。
- 延迟目标 <500ms，但网络波动时可能达到 1–2s。

**建议**：
- 增加 feature flag `aiLlmAsJudgeEnabled`，默认只在 dogfood/staging 开启，prod 逐步放量。
- 增加 `aiLlmAsJudgeMaxTokensPerDay` 配额限制，超限后自动降级为跳过 Judge。
- Judge 结果异步写入，不阻塞用户输出。

### 3.5 🟢 低风险项

| 项 | 风险 | 原因 |
|----|------|------|
| 评测 30+ 条 | 低（技术）高（工作量） | 只是写 case 定义和 fixture，无技术难题 |
| Reflection 自检 | 低 | 只是多一次低成本模型调用，逻辑简单 |
| 引用准确率指标 | 低 | P0 已埋点，P1 只是统计和展示 |
| Workflow 成本基线 | 低 | 只是统计和比较，逻辑简单 |
| MCP Client | 中 | 依赖外部 MCP Server 稳定性，但实现本身不复杂 |

## 5. 调整建议（两档方案）

### 方案 A：保守调整（推荐）— 确保按期交付

**核心原则**：P1 只做「必做 + 轻量」，重量级认知架构项部分延后到 P2。

**P0 不变**（9 项，2 周）。

**P1 调整为 10 项（原 15 项削减 5 项）**：

| 原编号 | 原交付项 | 调整 | 原因 |
|--------|----------|------|------|
| 1 | VerticalWorkflowRegistry 闭环 | ✅ 保留 | 产品化核心 |
| 2 | 契约版本治理 | ✅ 保留 | 工业级必选项 |
| 3 | 指标可信度治理 | ✅ 保留 | 轻量 |
| 4 | 运行安全统一策略 | ✅ 保留 | 安全基础 |
| 5 | 性能与成本接入 | ✅ 保留 | 轻量 |
| 6 | 语音 release-stable | ✅ 保留 | 维护性 |
| 7 | 评测 12→30+ | ⚠️ **改为 12→20 条** | 18 条新 case 太多；P2 再补到 30+ |
| 8 | LLM-as-Judge | ⚠️ **改为 P2** | 重量级，且依赖评测集稳定后才能校准 |
| 9 | Plan-and-Execute | ⚠️ **改为「伪组合」**：顺序调用两个 workflow，失败整体重试，不共享 checkpoint | 避免 TaskRunner 重构 |
| 10 | 长期记忆 | ⚠️ **改为 localStorage 版** | 避开 Dexie schema 升级风险 |

| 12 | 引用准确率指标 | ✅ 保留 | 轻量 |
| 13 | Workflow 成本基线 | ✅ 保留 | 轻量 |
| 14 | MCP Client | ⚠️ **只接 Zotero，不接 OpenAlex** | 降低范围 |
| 15 | MCP Server | ⚠️ **改为 P2** | 浏览器兼容性风险高，需更多验证时间 |

**P2 调整为 14 项（原 9 项 + P1 延后 5 项）**：
- 原 9 项全部保留
- 新增：评测 20→30+、LLM-as-Judge、Plan-and-Execute 深化（含 checkpoint 恢复）、长期记忆 Dexie 迁移、Reflection、MCP Server

**P1 工作量重新估算**：

### 方案 B：激进调整 — 维持 v2 范围，延长 P1 周期

**核心原则**：不削减范围，但承认现实，延长 P1 从 4 周到 **8 周**。

**调整**：
- P0 不变（2 周）。
- P1 延长为 8 周（40 人天），保留全部 15 项。
- P2 顺延为 12 周（60 人天），保留全部 9 项 + 深化项。

**风险**：
- 总周期从 2+4+8=14 周延长到 2+8+12=22 周。
- 团队可能因长周期而失去紧迫感。
- 竞品（NotebookLM、Cursor + MCP）可能在 14 周内发布新功能，拉开差距。

**建议**：除非团队能从其他项目抽调 2 名全职工程师，否则不推荐此方案。

## 6. 具体修改建议（针对路线图文件）

如果采纳方案 A（保守调整），建议对路线图 v2 做以下修改：

### 6.1 新增 "P1 扩展" 阶段（原 P2 拆分为 P1 扩展 + P2）

将原 P1 的 15 项拆分为：
- **P1 核心（4 周）**：10 项，必须完成。
- **P1 扩展（额外 4 周）**：5 项，与 P2 并行或前置。

这样总周期变为 2+4+4+8=18 周，比方案 B 的 22 周短，但比原 v2 的 14 周现实。

### 6.2 对 Plan-and-Execute 的描述修正

将 P1 交付项 9 从：
> "每个 step 是一个子任务，有独立 checkpoint；step 失败时从 checkpoint 恢复"

改为：
> "组合 workflow 顺序调用两个单 workflow（annotation_qa → lexeme_candidates），应用层维护 step 状态（成功/失败）。若第一步失败，整体失败并重试；若第二步失败，可只重试第二步（不重新运行第一步）。不依赖 TaskRunner 子任务支持。"

### 6.3 对长期记忆的描述修正

将 P1 交付项 10 从：
> "扩展 backgroundMemoryExtractor：输出同时写入 project 级别的 IndexedDB 持久存储"

改为：
> "P1 先用 localStorage 实现跨会话记忆（project 级 JSON 存储，TTL + 上限 500 条）。P2 评估后迁移到 Dexie（需 schema v47 升级）。"

### 6.4 对 MCP Server 的描述修正

将 P1 交付项 15 从：
> "新增 src/ai/mcp/server/ 目录，实现只读 MCP Server（@modelcontextprotocol/sdk）"

改为：
> "P1 不引入 @modelcontextprotocol/sdk，先实现轻量 HTTP JSON-RPC 子集（只支持 tools/list 和 tools/call）。P2 验证浏览器兼容性后再决定是否引入完整 SDK。"

### 6.5 增加技术风险章节

在路线图 §7 风险与缓解矩阵中，增加以下条目：

| 风险 | 概率 | 影响 | 缓解 |
