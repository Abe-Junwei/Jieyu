# 第三方路线图评估 — 吸收意见与 P0/P1 压缩执行方案

> **评估来源**：外部专家意见（基于代码级现状审查）  
> **评估对象**：路线图 v2 + 审查报告 v2  
> **输出**：吸收结论 + 压缩后的 P0/P1 执行清单（按周粒度 + 每周门禁）

---

## 1. 一句话结论

**全部吸收**。这份评估与我们的代码审查结论高度一致，且在三个关键点上比我们走得更准：① 对 P0「硬收口」的定调更坚决；② 对 P1「主次分离」的拆分更清晰；③ 对 MCP「不急 runtime」的判断更务实。

| 评估维度 | 专家评级 | 我方审查评级 | 一致性 |
|----------|----------|-------------|--------|
| 战略正确性 | A | 方向正确 ✅ | 一致 |
| 工程可落地性 | A- | 地基扎实，P1超载 ⚠️ | 一致 |
| 执行复杂度控制 | B（P1过大） | P1严重超载 🔴 | 一致，专家更温和 |

## 2. 逐项吸收评估

### 2.1 成熟度判断 — 全部同意 ✅

| 判断项 | 专家结论 | 我方验证 | 吸收方式 |
|--------|----------|----------|----------|
| 运行与恢复能力：中高（优势） | TaskRunner 已具备工业雏形 | 确认：attempt/timeout/backoff/checkpoint/stale recovery 齐全 | 直接吸收，作为 P0 不重构 TaskRunner 的依据 |
| 语音与主链一致性：中高（优势） | VoiceAgentService 已拆分并共享分发 | 确认：ADR-0028 单事实源已落地 | 直接吸收 |
| 发布治理：中（关键风险） | dry-run/enforce 混用是现实风险 | 确认：`release-evidence-governance-gate.json` 存在 `mode: enforce + dryRun: true` | 直接吸收，作为 P0 最高优先级项 |
| 评测体系：中低（最大缺口） | 6 条全绿只能说明基础链路可用 | 确认：suite.v1.json 6/6，`requiredGoldenTasksMin: 30` 但远未达标 | 直接吸收 |
| 产品化闭环：中低（应优先补） | EvidencePacket/VerticalWorkflowRegistry 仍以规划为主 | 确认：shadow registry 就绪但未接入主链 | 直接吸收 |

### 2.2 P0 建议 — 全部同意 ✅

专家建议："只做硬收口，把可争议项变成不可争议项"

| 建议项 | 与我方 P0 对比 | 差异 | 吸收方式 |
|--------|---------------|------|----------|
| 发布证据 SoT：彻底消除 dry-run/enforce 混用 | 我方 P0 交付项 1–2 | 一致 | 保持，提升为 P0 **最高优先级** |
| 评测 6→12：优先负样本 | 我方 P0 交付项 7 | 一致 | 保持 |
| EvidencePacketV0 + shadow registry | 我方 P0 交付项 5 | 一致 | 保持 |
| 新写入口强制矩阵登记 + 自动检测失败即拦截 | 我方 P0 交付项 6 | 一致 | 保持 |
| MCP 先做 schema 兼容，不急 runtime | 我方 P0 交付项 9 | 一致 | 保持 |
| **不做项** | 我方 P0 有引用准确率埋点（交付项 8） | 专家建议 P0 **不做埋点**，放到 P1 | **采纳**：P0 确实应该聚焦收口，埋点可延后 2 周 |

**关键调整**：将 P0 交付项 8（引用准确率埋点）后移到 P1，P0 从 9 项减为 **8 项**，更聚焦。

### 2.3 P1 建议 — 大部分同意，一处微调 ⚠️

专家建议拆主次：
- **主线 A（必须）**：3 个 workflow 闭环 + evidence 可跳转 + eval 30+
- **主线 B（必须）**：长期记忆跨会话 + reflection 轻量化
- **次线（可后移）**：MCP Client/Server 放到 P1 后半或 P2 前半

与我方保守方案的对比：

| 项 | 专家建议 | 我方保守方案 | 差异 |
|----|---------|-------------|------|
| workflow 闭环 + evidence | 主线 A | ✅ 保留 | 一致 |
| eval 30+ | 主线 A | 改为 20 条 | **专家更激进，建议恢复 30+** |
| 长期记忆 | 主线 B | localStorage 版 | 一致（专家也认可先轻量） |
| reflection | 主线 B 轻量化 | 延后到 P2 | **专家建议 P1 做轻量版** |
| MCP Client/Server | 次线，后移 | Client 保留、Server 延后 | **专家建议 Client 也后移** |
| LLM-as-Judge | 未提及 | 延后到 P2 | 一致（双方都认为不急） |
| Plan-and-Execute | 未提及（P2 深化） | 伪组合降级 | 一致 |

**关于 eval 30+ 的争议**：
- 专家建议 P1 做到 30+（主线 A 必须）。
- 我方保守方案改为 20 条，担心工作量超载。
- **折中判断**：30 条是工业级底线，不能妥协。但 18 条新 case 在 4 周内写不完。
- **解决方案**：P1 交付 30 条，但其中 10 条可以是「半自动化」的（复用相同 fixture，只改 prompt 参数），真正需要精心设计的只有 8 条（对抗 4 + 垂直 4）。这样工作量从 4 天降到 2.5 天。

**关于 MCP Client 后移**：
- 专家建议 Client 也后移到 P1 后半或 P2。
- 我方保守方案保留 Client（仅 Zotero）。
- **判断**：专家更务实。MCP Client 虽然有价值，但 Zotero 的集成需要处理用户认证、文献格式转换、EvidencePacket 包装，工作量不小。且 Zotero MCP Server 的稳定性不由 Jieyu 控制。
- **采纳**：MCP Client 和 Server 统一后移到 P1 扩展或 P2。

## 3. 压缩后的 P0/P1 执行方案

基于专家意见 + 我方审查，重新压缩后的方案：

### P0（2 周 = 10 人天）— 只做硬收口

**目标**：把「可争议项」变成「不可争议项」，从 M2.5 升到 M3。

| # | 交付项 | 人天 | 验收命令 | 退出条件 |
|---|--------|------|----------|----------|
| 1 | **发布证据 SoT 硬收口**：彻底消除 `mode: enforce + dryRun: true` 混用；新增 `gate:release-evidence:governance:enforce` | 2.0 | `gate:release-evidence:governance:enforce` | 任何 gate 报告中不存在 dry-run skipped |
| 2 | **评测 6→12 条**：destructive 误触发 2 + no-silent-write 2 + policy deny 2 + 基础对抗 2 | 1.5 | `check:agent-evals:trace` | 12/12 pass，falseAllowRate <5% |
| 3 | EvidencePacketV0 + VerticalWorkflowRegistry shadow | 1.5 | `typecheck` + registry 单测 | 3 个 workflow 可枚举，i18n key 无漂移 |
| 4 | 新写入口强制矩阵登记 + `check:ai-session-sidecar-entrypoints` 拦截 | 1.5 | `check:ai-session-sidecar-entrypoints` | 未登记写入口触发门禁失败 |
| 5 | 边界守卫测试纳入 pre-merge | 1.5 | `check:architecture-guard` | pre-merge 默认执行 |
| 6 | 故障注入集（timeout / executor throw / 审计写入失败） | 1.0 | 相关 Vitest 子集 | 3 case 全部通过且有恢复信号 |
| 7 | 运行安全 taxonomy | 1.0 | `check:ai-session-sidecar-entrypoints` | 6 类 capability 各 1 个 allow/ask/deny 样本 |
| 8 | MCP schema 兼容预留 | 0.5 | `check:mcp-schema-compatibility` | shadow registry 所有工具可转 MCP schema |
| **合计** | **8 项** | **10.0 人天** | | |

**P0 不做项（明确排除，避免分心）**：
- 引用准确率埋点（→ P1）
- 任何 MCP runtime 集成（→ P1 扩展/P2）
- 任何 UI 改动 beyond VerticalWorkflowRegistry shadow

### P1 核心（4 周 = 20 人天）— 价值闭环

**目标**：形成端到端「可见价值闭环」，从 M3 升到 M3.5~M4。

**主线 A：产品化闭环（必须，~12 人天）**

| # | 交付项 | 人天 | 说明 |
|---|--------|------|------|
| A1 | VerticalWorkflowRegistry 最小闭环 | 3.0 | 3 个 workflow 接入统一注册，UI 可展示 evidence 并跳转 |
| A2 | EvidencePacket 可跳转原始来源 | 1.5 | segment / layer_text / document 三类 source 均可点击跳转 |
| A3 | 契约版本治理 + CI schema 漂移阻断 | 2.0 | 审计字段 schema 引入版本迁移规则 |
| A4 | 评测 12→30 条 | 2.5 | 对抗 4 + 跨语言 3 + 垂直 4（半自动化）+ RAG 错配 3 + 用户场景 4 = 18 条新 case |
| A5 | 指标可信度治理 | 1.0 | 样本阈值与窗口要求，partial/stable 降级 |
| A6 | 引用准确率指标产品化 | 1.0 | P0 未做，P1 补埋点 + 产品化进 C5 |
| A7 | Workflow 成本基线 | 1.0 | 3 个 workflow 预期 token/延迟，偏离 >50% 触发审计 |

**主线 B：认知架构轻量（必须，~6 人天）**

| # | 交付项 | 人天 | 说明 |
|---|--------|------|------|
| B1 | 长期记忆跨会话（localStorage 版） | 3.0 | project 级 JSON 存储，TTL + 上限 500 条，P2 再迁 Dexie |
| B2 | Reflection 轻量化（自检节点） | 2.0 | annotation_qa / lexeme_candidates 输出后低成本模型自检矛盾，不阻塞 UI |
| B3 | 运行安全统一策略 | 1.0 | capability taxonomy + allow/ask/deny 样本 |

**次线：性能与语音（维护性，~2 人天）**

| # | 交付项 | 人天 | 说明 |
|---|--------|------|------|
| C1 | 性能与成本接入 | 1.0 | latency/token 趋势报告 |
| C2 | 语音 release-stable | 1.0 | manifest health 周期采样 |

**P1 核心合计：10 项，~20 人天**

**P1 核心验收命令**：
```bash
npm run -s gate:release-evidence:session-sidecar-sandbox
npm run -s check:release-evidence:governance:strict
npm run -s gate:voice-maintenance
npm run -s gate:release-evidence:core
npm run -s perf:ai

npm run -s check:workflow-cost-baseline
```

**P1 核心退出条件**：
- 3 个 workflow 可在不修改核心执行器情况下扩展。
- 30 条 case 全部通过，falseAllowRate <5%，falseBlockRate <10%。
- EvidencePacket 覆盖 segment/layer_text/document，UI 可跳转。
- 长期记忆跨会话写入/读取/遗忘单测通过。
- Reflection 自检可检测人为注入的矛盾 finding。
- Workflow 成本基线 3 个 workflow 均有数据。

### P1 扩展（额外 4 周）— 生态与深化

**目标**：补全 MCP、LLM-as-Judge、Plan-and-Execute 伪组合。

| # | 交付项 | 人天 | 说明 |

**为什么拆出 P1 扩展**：
- P1 核心 20 人天已是 4 周满负荷，再加 8 人天必然延期。
- P1 扩展的项（MCP、Judge、Plan）是「加分项」而非「生死项」。
- 如果 P1 核心按期完成，团队士气高，P1 扩展可以加速执行。
- 如果 P1 核心延期，P1 扩展可以整体顺延，不影响主线价值闭环。
