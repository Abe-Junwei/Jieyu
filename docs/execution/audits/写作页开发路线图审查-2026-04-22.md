# 写作页开发路线图审查报告（2026-04-22）

> **审查范围**：`docs/execution/plans/写作页开发路线图-2026-04-22.md` 及其直接依赖文档（G0 定稿、开放门槛、ADR 0010–0012）。
> **审查方法**：对照当前仓库代码事实 + 检索外部技术资料（Pandoc 3.9 WASM、Typst WASM、CM6+Yjs 生态、unified/remark 现状）。

## 二、确认合理的部分（无需改动）

1. **CM6 首发、PM/TipTap 仅作第二编辑面**：合理。CM6 直接编辑 Markdown 源码，与「md 为唯一持久化真源」的 G4 硬约束天然一致；Milkdown/BlockNote 等方案或 React 集成简陋，或 bundle 过大，或需付费导出，均不适合首发。
2. **G4「md 为唯一持久化真源」**：关键且正确。避免 Yjs/PM 成为第二真源，防止后续数据模型混乱。
3. **G1 垂直切片顺序（A→B）**：正确。先搞定存盘真源（CM6 + Dexie + Zod），再叠加 remark/rehype 预览，避免预览管道拖累存盘路径。
4. **`corpusRef` 必填 + 跨项目只读 resolve**：正确。防止 GrammarDoc 段落树嵌入 md 成为第二真源。
5. **NO-GO / lab flag / layoutGuard 三重约束**：正确。在转写主线封账前不替换占位页，符合项目当前工程纪律。
6. **ADR 0011 `jieyu:corpus:v1` URI 前缀**：合理。版本化 URI 便于后续演进。

## 三、发现的可改进之处

### 🔴 改进 1：Pandoc 3.9 原生 WASM 进展未纳入导出运行时评估

**现状**：
- 路线图中 [2.3 导出运行时决策树](docs/execution/plans/写作页开发路线图-2026-04-22.md#23-导出运行时决策树须在-g0-收口g1-前成文) 与 [ADR 0010](docs/adr/0010-writing-workspace-export-runtime.md) 将「纯 Web 用户」默认降级为「下载 md+yaml+asset manifest zip + 本地脚本」。
- ADR 0010「被放弃的备选方案」仅记录「默认承诺浏览器内完整 LaTeX」。

**新事实（2026-02-03）**：
- Pandoc 3.9 官方发布原生 WASM 构建（`make pandoc.wasm`），附 JS bridge (`wasm/pandoc.js`) 与完整 GUI demo（https://pandoc.org/app）。
- Pandoc WASM 的 **主要限制**：不能产 PDF（需外部命令），但 **可以运行 Lua filters**，支持 md → docx / HTML / LaTeX 源码 / 等多种格式。
- Pandoc WASM GUI 甚至内嵌了 **Typst WASM** 来在浏览器内产 PDF。

**影响**：
- 「纯 Web 用户只能下载 zip」不再是唯一技术路径。浏览器内「md → docx / HTML / TeX 源码」已成为可行选项。
- 虽然完整 CTAN 仍无法在浏览器内默认提供，但「无 CLI 用户」的体验可以显著好于「下载 zip」。
- 若愿意引入 Typst WASM，浏览器内 PDF 也非完全不可能（Pandoc 官方 demo 已验证）。

**建议修改**：
1. 在 ADR 0010 中新增 **「新观察项」** 段落，记录 Pandoc 3.9 WASM 的能力与限制：
   - 可行：md → docx / HTML / TeX 源码（含 Lua filters）。
   - 限制：PDF 需外部命令（除非引入 Typst WASM）。
   - 不能 fetch HTTP 资源、不能运行系统命令（JSON filters 不可用）。
2. 将「被放弃的备选方案」改为 **「已重新评估的备选方案」**，注明：浏览器内 md→docx/HTML/TeX 已可通过 Pandoc WASM 实现，不再是「放弃」项，但需在 G1.5 前做 spike 验证。
3. [G0 定稿](docs/execution/plans/写作页导出运行时已采纳默认-G0-定稿-2026-04-22.md) 的「纯 Web 用户默认」一列增加脚注：「Pandoc WASM 浏览器内编译为 2026-02 后的新选项，详见 ADR 0010 后续回顾点」。

### 🟡 改进 2：Typst WASM 作为浏览器内 PDF 的潜在路径未评估

**现状**：
- 路线图中 Typst 被列为「可选」排版引擎（第四节），与 Quarto 并列。
- 路线图中明确「不承诺浏览器内一键真 PDF」。

**新事实**：
- Typst 编译器已有成熟的 WASM 构建（typst.app 即纯浏览器运行）。
- Pandoc 3.9 WASM GUI 已内嵌 Typst WASM，实现浏览器内 md → PDF 的完整链路。
- Typst 的编译速度（毫秒级增量编译）远快于 LaTeX，用户体验显著优于「下载 zip + 本地运行」。

**影响**：
- 若项目的目标用户群体中「纯 Web 用户」占比较高，引入 Typst WASM 可以提供一个比「下载 zip」好得多的降级体验。
- 但 Typst 的生态（尤其语言学专用包：expex、forest、vowel 等）远不及 LaTeX，对于需要这些包的用户，Typst 路径可能不可用。

**建议修改**：
1. 在路线图的 **附录 D（图码供应链降级矩阵）** 或 **第八节（风险与待决 ADR）** 中增加一条：
   - **「Typst WASM 浏览器内 PDF」**：技术可行（Pandoc 3.9 已验证），但语言学包生态不足；建议作为 **G1.5 后的技术 spike**，与 LaTeX 路径并行评估。
2. 在 ADR 0010 的「后续回顾点」中增加：
   - 「评估 Typst WASM 作为纯 Web 用户 PDF 降级的可行性（不晚于 G2）」。

### 🟡 改进 3：缺少「技术预研（spike）」里程碑与构建预算

**现状**：
- 路线图从 G0（文档）直接跳到 G1（CM6 + Dexie + Zod 实现），中间没有预留技术验证时间。
- 项目已有 `check-build-budgets.mjs` 等构建预算门禁，但路线图中未为写作页各阶段设定 bundle size 目标。

**风险**：
- CM6 + @codemirror/lang-markdown + remark/rehype + KaTeX + Yjs + 可能的 Pandoc WASM 叠加后，bundle size 可能远超当前转写页。
- CM6 与 mdast 的集成（尤其是自定义 `kind` 块的解析/序列化）存在未知复杂度，G1 切片 A 的「回滚点」如果触发，代价较高。

**建议修改**：
1. 在 G0 与 G1 之间增加一个 **G0.5「技术预研（spike）」** 阶段（时间盒 1–2 周，不写入产品代码）：
   | 验证项 | 通过标准 |
   |--------|----------|
   | CM6 + @codemirror/lang-markdown 最小集成 | 可输入、可保存 md 字符串、bundle 增量可测 |
   | remark-parse + 自定义 `kind` 块插件 | 可解析/序列化 `corpus-example`、`syntax-tree` 等块 |
   | Pandoc WASM spike（可选） | 浏览器内 md → docx/TeX 成功，记录 wasm 体积与首载时间 |
   | Typst WASM spike（可选） | 浏览器内 md → PDF 成功，记录 wasm 体积 |
2. 在 [第七节 验收指标](docs/execution/plans/写作页开发路线图-2026-04-22.md#七验收指标汇总须可测) 中增加构建预算项：
   - **CM6 核心 bundle**：增量 ≤ X KB（建议根据转写页当前 budget 设定，如 200KB gzipped）。
   - ** remark/rehype 管道**：增量 ≤ Y KB。
   - **Pandoc/Typst WASM（若引入）**：首载 ≤ Z MB，懒加载。

## 四、其他细微观察（非必须，供参考）

| # | 观察 | 建议 |
|---|------|------|
| 4 | `@chenglou/pretext` 的引用 | 路线图第三节提到用于「DOM 外折行与高度」。但该库主要用于虚拟列表的预渲染测量，而 Markdown 编辑器通常不需要像富文本编辑器那样复杂的虚拟列表。建议在 G1 spike 中验证其实际必要性，避免过早引入。 |
| 5 | `kind` 枚举中的 `embed-whiteboard` | 第四节的 `kind` 枚举包含 `embed-whiteboard`，但图码降级矩阵（附录 D）未提及白板。建议在 G3 前明确白板的持久化格式（SVG？JSON？）与导出映射。 |
| 6 | G3 `writingDocAdapter` 的 mock 策略 | 路线图中提到「tool_call → CM6 集成测（可 mock）」，但未说明 mock 层级。建议明确：mock 的是 `useAiChat` 的 tool_call 响应，还是 mock `writingDocAdapter` 本身？ |

## 六、一句话总结

> 写作页路线图整体严谨、可执行；但 **2026 年 Q1 Pandoc 官方 WASM 与 Typst WASM PDF 的成熟，使得「纯 Web 用户只能下载 zip」的导出降级策略过于保守**。建议在 G0 收口前更新 ADR 0010 与 G0 定稿，将 Pandoc WASM 纳入观察项；在 G1 前增加技术 spike 阶段验证 CM6 集成与 WASM 导出可行性；并为写作页设定明确的构建预算。
