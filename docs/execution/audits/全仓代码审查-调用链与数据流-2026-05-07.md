## Findings

### P1：Tool decision runtime report 写入/读取字段不匹配，rollup 实际读不到生产审计

入口与写入链路：`useAiChat` 的工具执行 / 确认路径调用 `writeToolDecisionAuditLog`，该函数写入 `audit_logs`，字段为 `ai_tool_call_decision`。写入的 `metadataJson` 类型是 `ToolDecisionAuditMetadata`，核心字段是 `outcome` 与 `toolCall.name`。

读取链路：`generateAiRuntimeReportFromAudit()` 调用 `collectToolDecisionRecords()` 后再进入 `rollupToolDecisions()`。但 collector 读取的是 metadata 顶层 `decision` 和 `toolName`。

破损路径：生产写入 `{ phase: 'decision', outcome, toolCall: { name } }`，报告读取 `{ decision, toolName }`。因此真实 tool decision 审计即使持久化成功，也不会进入 `toolDecisionRollup`，`denied` / `confirmRequired` / `byToolName` 会漏数。

为什么现有检查会漏掉：`typecheck` 只验证各自局部类型；现有 runtime report 测试只覆盖 adoption outcome，没有覆盖 `collectToolDecisionRecords()` 对真实写入 metadata 的读回。

证据：

- 写入端：[src/hooks/useAiChat.toolAudit.ts](../../../src/hooks/useAiChat.toolAudit.ts#L50-L74)
- metadata 定义：[src/ai/chat/toolCallHelpers.ts](../../../src/ai/chat/toolCallHelpers.ts#L63-L88)
- 读取端：[src/ai/eval/aiRuntimeReportDimensionalAudit.ts](../../../src/ai/eval/aiRuntimeReportDimensionalAudit.ts#L73-L114)
- 汇总端：[src/ai/eval/aiRuntimeReport.ts](../../../src/ai/eval/aiRuntimeReport.ts#L385-L404)

需要的修复验证：用真实 `buildToolDecisionAuditMetadata(...)` 生成的 `metadataJson` 写入 mock / Dexie `audit_logs`，执行 `generateAiRuntimeReportFromAudit()`，断言 `toolDecisionRollup.byToolName[call.name]`、`denied` / `confirmRequired` 按真实 outcome 映射出现。

### P1：Reflection failed-check rollup 读取 `checks[]`，写入端只保存 `failedCheckNames[]`

为什么现有检查会漏掉：现有 runtime report 测试里 reflection mock metadata 是 `{}`，没有断言 failed-check rollup；typecheck 不会发现 JSON schema 约定漂移。

### P1：AI chat thread 使用未定义短 CSS token，状态/置信度样式声明会被浏览器丢弃

入口与渲染链路：AI chat 的 adoption actions、source scope、evidence confidence 等组件依赖 `src/styles/panels/ai-chat-thread.css` 中的 `color-mix(... var(--success|--warning|--danger) ...)` 来呈现成功、警告、危险状态。

破损路径：`tokens.css` 定义的是 `--state-success-*`、`--state-warning-*`、`--state-danger-*` 系列；没有 `--success`、`--warning`、`--danger`。CSS custom property 未定义时，包含该 var 的整个 `color-mix(...)` 声明无效。结果是 adoption 接受/忽略、evidence confidence 高中低、warning/danger 边框背景等状态视觉可能回落到默认值或完全缺失。

为什么现有检查会漏掉：`npm run build` 不会校验 CSS var 是否已定义；当前 CSS token guard 没有拦截短 token；视觉回归若只看结构/文本也发现不了颜色语义丢失。

证据：

- 未定义短 token 使用：[src/styles/panels/ai-chat-thread.css](../../../src/styles/panels/ai-chat-thread.css#L198-L202)、[src/styles/panels/ai-chat-thread.css](../../../src/styles/panels/ai-chat-thread.css#L536-L592)、[src/styles/panels/ai-chat-thread.css](../../../src/styles/panels/ai-chat-thread.css#L939-L947)
- tokens 文件没有 `--success:` / `--warning:` / `--danger:` 定义，现有状态 token 为 `--state-*` 系列：[src/styles/tokens.css](../../../src/styles/tokens.css)

需要的修复验证：把短 token 替换为语义 token（如 `--state-success-solid/text/bg` 等），补或扩展 CSS token usage guard，跑 `npm run check:css-architecture`；再用 AI chat evidence/adoption UI 的截图或 computed style 验证状态颜色不再为空。

### P2：AdoptionQueue 自动过期审计 callback 未接入生产，且当前实现会重复上报历史过期项

入口与状态链路：AI card mount 时调用 `pruneExpiredItems({ items: prev }, Date.now())` 自动过期 adoption items。`pruneExpiredItems()` 新增了 `_auditCallback`，但生产调用未传入，TODO 也说明尚未接线。

实现风险：`pruneExpiredItems()` 先 map 全量 items，再用 `pruned.filter(item.status === 'expired' && reasonCode === 'auto_expired')` 取 expired items。该筛选会包含之前已经是 `auto_expired` 的历史项，而不只是本轮从 pending 变成 expired 的项。未来一旦接入 audit callback，重复进入页面 / 重复 prune 会重复写入 expire outcome。

为什么现有检查会漏掉：测试只断言 old pending 会变 expired、recent 不变；没有 callback 调用次数、只上报新过期项、重复 prune 幂等性测试。

证据：

- 生产调用未接 callback：[src/components/ai/AiChatCard.tsx](../../../src/components/ai/AiChatCard.tsx#L486-L494)
- callback 与筛选逻辑：[src/ai/vertical/adoptionQueue.ts](../../../src/ai/vertical/adoptionQueue.ts#L144-L174)
- 审计写入 helper 只覆盖用户动作：[src/ai/vertical/adoptionOutcomeAuditPersist.ts](../../../src/ai/vertical/adoptionOutcomeAuditPersist.ts#L13-L46)
- 测试缺口：[src/ai/vertical/adoptionQueue.test.ts](../../../src/ai/vertical/adoptionQueue.test.ts#L121-L136)

需要的修复验证：补两个用例：一是 production callback 接入后 expire action 会写 `ai_adoption_outcome` 并可被 report 读回；二是同一 expired item 连续 prune 两次不会重复 callback。

### P2：MCP tool-call timeout 使用 `Promise.race + setTimeout` 未清理定时器，成功调用也会留下 30 秒 timer

入口与调用链路：`POST /messages` 路由到 `routeMethod('tools/call')`，执行 handler 时用 `Promise.race([handler(...), new Promise(setTimeout 30_000)])` 实现超时。

资源风险：当 handler 很快成功时，timeout promise 里的 timer 不会被 clear。高频 MCP 调用会累积短期 timer；在脚本 / 测试进程中也可能导致进程额外等待或资源噪声。现有测试覆盖成功、失败、timeout 行为，但没有检测 timer cleanup。

为什么现有检查会漏掉：功能断言只看 JSON-RPC 结果和 audit outcome；资源生命周期泄漏不会被 typecheck / build / 当前单测发现。

证据：

- `Promise.race` timeout：[src/ai/mcp/server/McpServer.ts](../../../src/ai/mcp/server/McpServer.ts#L286-L293)
- 成功后直接 persist 并 return，未清 timer：[src/ai/mcp/server/McpServer.ts](../../../src/ai/mcp/server/McpServer.ts#L294-L302)
- 测试未覆盖 timer cleanup：[src/ai/mcp/server/McpServer.test.ts](../../../src/ai/mcp/server/McpServer.test.ts#L177-L227)

需要的修复验证：将 timeout 封装为可清理 helper，在 success / handled error 后 `clearTimeout`；用 fake timers 或 active handle 观测补回归。

### P2：移动端 AI 侧栏 collapsed 状态被后置 media rule 覆盖，折叠语义在小屏失效

入口与状态链路：桌面侧栏通过给面板加 `.transcription-ai-panel-collapsed` 来设置 `transform: translateX(100%)` 与 `pointer-events: none`。移动端 `@media (max-width: 640px)` 后置规则对同一 `.transcription-ai-panel` 设置 `position: static`、`transform: none`、`pointer-events: auto`，且隐藏 toggle/resizer。

破损路径：当应用状态仍认为侧栏是 collapsed 时，小屏 CSS 会用后置同 specificity 规则把面板恢复为可见、可交互，并且折叠/展开控制也被隐藏。用户在窄屏或窗口缩小时可能无法维持已折叠状态。

为什么现有检查会漏掉：CSS 架构检查只看规则治理，不跑状态组合；桌面截图不会触发 `max-width: 640px`；组件测试通常不计算 cascade 后的 transform/pointer-events。

证据：

- collapsed 规则：[src/styles/pages/ai-sidebar-shell.css](../../../src/styles/pages/ai-sidebar-shell.css#L119-L123)
- 移动端覆盖规则：[src/styles/pages/ai-sidebar-shell.css](../../../src/styles/pages/ai-sidebar-shell.css#L288-L309)

需要的修复验证：明确移动端产品语义。如果小屏不支持折叠，需要状态层禁用 collapsed 并给用户可见入口；如果支持折叠，需要补 `.transcription-ai-panel-collapsed` 在 media block 内的覆盖规则，并用 Playwright mobile viewport 验证 transform/pointer-events 与可恢复入口。

### P2：独立 AI ChatWindow 从 localStorage 恢复布局时未做 viewport clamp，窗口可能离屏或超宽

入口与持久化链路：`TranscriptionPage.ChatWindow.tsx` 从 `jieyu.aiChatWindow.v1` 读取 `open/minimized/x/y/width/height`，直接 `setPosition({ x, y })`，尺寸只按全局 `MIN/MAX` clamp，未按当前 viewport clamp；后续也没有 window resize 监听主动重夹。

破损路径：用户在大屏拖到右下角并保存布局，之后换到小屏/移动宽度打开页面，恢复出的 `x/y/width/height` 可以超过当前 viewport。最终 `style={{ left, top, width, height }}` 直接应用像素，导致 dialog 部分或全部离屏，且入口按钮隐藏后可能难以恢复。

为什么现有检查会漏掉：类型、构建、CSS guard 都看不到 localStorage 历史状态；手动测试若不跨 viewport 或不预置旧缓存也不会复现。

证据：

- 恢复布局未按 viewport clamp：[src/pages/TranscriptionPage.ChatWindow.tsx](../../../src/pages/TranscriptionPage.ChatWindow.tsx#L139-L158)
- clamp helper 存在但恢复路径未使用：[src/pages/TranscriptionPage.ChatWindow.tsx](../../../src/pages/TranscriptionPage.ChatWindow.tsx#L224-L246)
- 像素布局直接写入 style：[src/pages/TranscriptionPage.ChatWindow.tsx](../../../src/pages/TranscriptionPage.ChatWindow.tsx#L343-L343)

需要的修复验证：恢复 localStorage 时先按当前 viewport 执行 `clampSize` 和 `clampPosition`，并增加 resize listener 或 `ResizeObserver` 重新夹取；用 Playwright 预置大屏缓存后在 375px/640px viewport 打开，断言 dialog bounding box 完全在 viewport 内。

### P3：Knip 当前未配置入口，死代码扫描不能作为删除依据

### P3：CSS inline style guard 只匹配 `style={{`，漏掉 `style={identifier}` / `style={expr}` 形态

为什么现有检查会漏掉：脚本自身定义了扫描盲区；阈值又刚好等于当前 debt，新增 `style={someObject}` 可以绕过 guard。

证据：

- guard 正则：[scripts/css-inline-style-governance.mjs](../../../scripts/css-inline-style-governance.mjs#L1-L9)
- 当前被匹配的一例对象字面量 inline style：[src/pages/TranscriptionPage.ChatWindow.tsx](../../../src/pages/TranscriptionPage.ChatWindow.tsx#L343-L343)

需要的修复验证：用 TS/JSX parser 或增强正则扫描 `JSXAttribute name=style` 的所有表达式形态；补 fixture 覆盖 `style={{...}}`、`style={panelStyle}`、`style={condition ? a : b}`，并重新生成 baseline/whitelist。
