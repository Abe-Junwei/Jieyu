# 算法资产与热点审计（2026-05-13）

审计目标：为下一轮「算法优化」建立可执行候选清单，而不是直接进入无证据微优化。本轮仅做只读审计；不改变产品行为。

范围：`src/` 中与时间轴 / 批量变更 / 波形、AI embedding / PDF / RAG、语言目录 / 搜索 / 正字法相关的算法路径；同时记录已有测试和可验证命令。

## 0. 结论摘要

| 优先级 | 候选 | 主要收益 | 风险 | 建议动作 |
| --- | --- | --- | --- | --- |
| P1 | 时间轴 read-model：`buildTimelineUnitViewIndex` 与 `transcriptionLaneReadScope` 扫描 | 大文档 / 多层时间轴下减少重复扫描与排序成本 | 高：涉及 segment shadowing、语义合并、ReadyWorkspace 装配边界 | 先补性能与一致性基线，再做索引/分组优化 |
| P1 | AI embedding / RAG 检索：全表向量扫描 + 每次构建 MiniSearch | 大 embedding 集合下明显降低查询延迟 | 高：召回、排序、citation 行为可能变化 | 先做 candidate-set / rerank parity，再考虑缓存或近似检索 |
| P2 | 语言目录建议：每次 query 重建候选集合并全量打分 | 减少语言搜索输入过程中的冷启动和每键成本 | 中：候选缓存失效与排序稳定性 | memoize candidate index，保留 ranking 测试 |
| P2 | 批量时间偏移 overlap 校验：克隆全媒体单元并排序 | 批量操作更接近 selection-size 成本 | 中：边界与 overlap 语义敏感 | 改为邻居/局部区间校验前先补大样本测试 |
| P3 | 波形 region presentation pass：`forEach` + `find` | region 多时从近似 O(R²) 降到 O(R) | 中：拖拽、skipProcessing 样式和 effect 时序 | 先 Map 化查找，配合 E2E 回归 |
| P3 | 正字法列表 substring scan | 大 builtin + DB 目录时降低全量扫描 | 中低：匹配语义和排序需稳定 | 数据量上来后再做索引化 |

总判断：第一轮不应从「换算法」开始，而应先做 **P1 两条的基线固化**：时间轴索引与 embedding/RAG。这两处既是用户可感知路径，也已有较多测试或性能脚手架可承接。

## 2. 主要复杂度与风险模式

1. **全量扫描仍是主模式**：时间轴 index、语言目录候选、embedding 检索、正字法列表都以全量扫描为基础；目前可接受，但需要用数据规模阈值来决定何时优化。
2. **排序成本集中在时间轴与批量变更**：`buildTimelineUnitViewIndex`、batch selection、overlap 校验均可能出现 O(n log n)；优化必须守住确定性排序。
3. **MiniSearch 多处按 query 构建**：embedding hybrid、project memory、lexicon / language catalog 中都有动态建索引模式；适合统一评估缓存边界，但不宜过早抽象。
4. **缓存失效是主要正确性风险**：语言目录 runtime snapshot、embedding invalidation、RAG candidate cache 若做错，会产生 stale result，比慢更危险。
5. **大列表 DOM 与算法耦合**：纵向视图、横向 lanes 和 side pane 列表不能只看计算复杂度，还要看渲染节点数量和 React effect churn。

## 4. 建议执行顺序

### Phase A：基线补强（只加证据，不改算法）

1. 为 `buildTimelineUnitViewIndex` 增加覆盖 `transcriptionLaneReadScope` 多层 / 多媒体 / segment shadowing 的性能样本，避免只测理想线性路径。
2. 为 `EmbeddingSearchService.searchMultiSourceHybrid` 增加大 candidate-set 的 ranking parity fixture，记录 vector-only / hybrid / MiniSearch rerank 的耗时拆分。
3. 为 `LanguageCatalogSearchService.searchLanguageCatalogSuggestions` 增加冷启动与连续 keystroke 场景的 micro-benchmark 或 vitest perf baseline。

验收建议：

```bash
npm run typecheck
npm run test:timeline-cqrs-phase9
npm run perf:embedding-candidates
npx vitest run src/services/LanguageCatalogSearchService.test.ts src/utils/langMapping.test.ts
```

### Phase B：低风险局部优化

1. `src/hooks/media/useWaveSurfer.ts`：region presentation pass 建一次 `Map<id, region>`，替换循环中的 `regions.find`。
2. `src/services/LanguageCatalogSearchService.ts`：memoize candidate index，cache key 必须包含 locale / catalogScope / runtime snapshot signature。
3. `src/services/ProjectMemoryStore.ts`：若 memory docs 规模增长，缓存 MiniSearch 与 `Map<id, doc>`，避免每次 query 重建。

验收建议：

```bash
npm run typecheck
npm run lint:ci
npx vitest run src/services/LanguageCatalogSearchService.test.ts src/hooks/ai/useAiChat.memoryBroker.test.ts
npm run test:e2e:chromium
```

### Phase C：高收益但高风险优化

1. `src/hooks/transcription/timelineUnitView.ts`：重构 `transcriptionLaneReadScope` 下的 lanes × units 扫描，改为预分组 / 预索引 canonical rows。
2. `src/hooks/transcription/useTranscriptionUnitActions.batchActions.ts`：batch overlap 校验从全媒体 clone + sort 逐步改为邻居区间校验。
3. `src/ai/embeddings/EmbeddingSearchService.ts`：从全表向量扫描演进到 candidate-set 优先、缓存候选或近似检索；任何排序变化都必须有 golden parity。

验收建议：

```bash

npm run check:architecture-guard
npm run test:timeline-cqrs-phase9
npm run perf:embedding-candidates
npm run test:e2e:chromium
```

## 5. 不优化规则

以下情况本轮不建议动：

1. 没有基线数据、只能凭直觉判断的微优化。
2. 仅在小样本变快、但引入跨模块缓存失效复杂度的方案。
3. 改变排序 / 召回 / overlap 语义，却没有 golden parity 或边界测试的方案。
4. 在页面 / orchestrator 中新增复杂缓存或业务算法；算法应优先落在纯函数、service 或已有 controller 边界内。
5. 为了追求单次查询速度，引入长期后台索引而没有写入、删除、导入、reload 后 readback 验证。

## 6. 下一步推荐

建议先执行 **Phase A-1 + A-2**，产出两条可比较基线：

- 时间轴：`buildTimelineUnitViewIndex` 在 1k / 5k / 10k units + 多层 scope 下的耗时与正确性快照。
- RAG：`searchMultiSourceHybrid` 在 2.5k / 10k embeddings + mixed PDF/unit candidate 下的 vector scan、full-text rerank、final sort 耗时。

完成后再决定 Phase B 或 Phase C。若目标是最快见效，先做 `useWaveSurfer` 的 O(R²) → O(R) 局部优化；若目标是长期收益，先做时间轴 read-model 与 embedding 检索的基线固化。
