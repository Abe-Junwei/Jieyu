## 一、审计结论摘要

本轮核查覆盖历史规划文档 A8 中的 M1-M5 里程碑，并逐项对照当前代码实态。

结论分三类：

| 结论 | 含义 |
|---|---|
| 已落地 | 已有明确运行时代码，且能力已接入当前产品链路 |
| 已收口 | 适用于评估/选型项；已完成真实评测并形成明确结论，即使结论是不切换默认链路 |
| 部分落地 | 有评估、脚手架或局部接入，但尚未形成稳定默认链路 |
| 未落地 | 当前仅停留在规划、评估或外围准备，未进入真实产品闭环 |

本次核查结果如下：

| 里程碑项 | 结论 | 备注 |
|---|---|---|
| CRAG 检索质量评估 | 已落地 | 已进入 useAiChat RAG 注入链路 |
| Distil-Whisper 本地 STT 替换 | 已落地 | 已成为本地 STT 默认模型 |
| Late Chunking 阶段 A | 已落地 | 已用于长笔记与 PDF 嵌入分块 |
| Self-RAG 检索前反思判断 | 已落地 | 已有 skip / force / retrieve 判定 |

## 四、UI 入口影响结论

本次审计顺带核查了这些研究项是否伴随“UI 入口”变化。

结论是：

1. 没有看到新增独立 AI 工作台或独立首页入口。
2. 当前 AI 入口仍然是转写页助手运行时里的 `AiChatCard` 和 `VoiceAgentWidget`。
3. 真正发生变化的是既有入口后的能力增强，包括 AI prompt context 增强、citation 跳转，以及 PDF 预览与 snippet 高亮。
4. 虽然 `/analysis` 路由已存在，但当前 `AnalysisPage` 仍按治理文档保持说明页，因此不能算“独立 AI 工作台已开放”。

## 六、当前收口建议（不含 F10）

基于本轮核查结果，Arctic-Embed-XS 默认化决策、CrisperWhisper 时间戳评估，以及 AI 独立工作台入口都已经形成明确结论；当前仍未收口的主要只剩 F10 产品链路。

这里的“收口”不是继续泛化调研，而是把它们各自推进到“可做决策”的状态。

| 项目 | 当前建议 | 直接落点 | 完成信号 |
|---|---|---|---|
建议执行顺序如下：

1. 若未来重开 CrisperWhisper，只在研究环境下复跑时间戳 benchmark，不直接推进产品接入。
2. Arctic 只需在未来重新跑一次真实 A/B，即可复核是否需要改判默认模型。
3. Analysis/Annotation/Writing/Lexicon 四个独立工作台继续维持 `NO-GO` 说明页，直到转写主工作台封账完成并重新复核开放门槛。
4. F10 继续保留在后续考虑项中，但不作为本轮阻塞条件。

### 6.1 Arctic-Embed-XS：从“有评测脚手架”推进到“有默认化结论”

本轮已完成该闭环，当前结论如下：

1. 已补 `npm run perf:embedding-model-ab:real` 作为真实模型 A/B 评测入口。
2. 已把迁移规则收敛到 `embeddingModelDecision.ts`，不再散落在测试注释里。
3. 本轮固定语料实测结论为：Arctic 的延迟门槛满足，但质量门槛未满足，因此默认模型继续保留 E5-Small。
4. 本地 provider、service 与 UI 文案中的默认模型硬编码已统一收敛到 `localEmbeddingModelConfig.ts`。

这意味着该项当前已不再是“未收口”问题，而是一个未来可复核的已决事项：

1. 若后续真实评测跨过门槛，再迁移默认模型；
2. 在此之前，继续保留 E5-Small 默认链路。

### 6.3 AI 独立工作台入口：明确维持 `NO-GO`

这项当前也已经形成明确结论，但结论不是“开放”，而是“继续不开放”：

1. `/analysis` 路由已存在，但当前页面仍是 `FeatureAvailabilityPanel` 说明页。
2. 独立工作台治理文档已经把 Analysis 明确标为 `NO-GO`，并要求在转写主工作台封账完成前继续维持说明页。
3. 因此，当前对外口径应是“有占位路由，不代表能力已开放”，不能再把它计入 AI 产品化落地项。

这项后续只有两种合法状态：

1. 继续维持 `NO-GO`，直到 go 条件满足；
2. 先更新独立工作台开放门槛文档，再进入真实页面开放。

### 6.2 CrisperWhisper：保持评估态，不提前产品化

本轮已把这项收口为明确的“暂不接入”结论，原因不是运行时难度，而是产品条件当前不成立：

1. 文档已明确其许可带有非商用限制，不适合作为当前产品默认候选。
2. 仓库中仍无 CrisperWhisper provider/runtime，可接入对象本身不存在。
3. 因此本轮只补“评测工具 + 评估报告”，不引入 `timestampProvider` 抽象，不往产品链路写入新代码。

当前保留的 reopen 条件是：

1. 仅在研究环境中确认许可边界可接受；
2. 且已有稳定服务形态；
3. 且基于真实 10 组样本跑出足够好的 MAE / F1 / 覆盖率后，再重新讨论接入。

## 七、审计结论

“AI 智能体调研改进方案都落地了吗”的答案是否定的。

更准确的说法是：

1. 已进入真实产品链路的研究项，主要是 CRAG、Distil-Whisper、Late Chunking 阶段 A、Self-RAG 和 WhisperX 风格 VAD。
2. 研究型改进项中，Arctic、CrisperWhisper 与 AI 独立工作台入口均已形成明确结论；当前唯一仍未收口的产品链路主要是 F10。
3. UI 层目前仍没有新增可交付的独立 AI 工作台入口；当前变化主要体现为转写页既有 AI 入口和其后续 citation / PDF 预览链路的增强。

这意味着后续文档、发布说明和对外口径仍不应笼统宣称“调研方案已全部落地”，但已经可以把 Transformers.js v4 闭环从“未收口”调整为“已落地”，同时继续明确区分“运行时已接入”“评估中”和“未产品化”三种状态。
