# 无历史数据场景 — 本地库与协同层单轨化落地方案

## 0. 前提与适用范围（必须显式满足）

本方案 **仅** 在以下前提同时成立时采用；任一不成立则应回退到「带兼容与迁移回放」的通用路线。

| 前提 | 含义 |
|------|------|
| **无本地历史数据** | 目标环境不要求保留既有 IndexedDB 内容；允许 `Dexie.delete(dbName)`、清空表、或一次性抬升 major 版本后等价于空库启动。 |
| **无向后兼容义务** | 不向用户承诺「安装新版后仍能读取 N 个月前旧版导出的快照」；旧快照可 **直接拒绝** 并提示重新建工程或重新导出（若未来再引入导出，则仅支持当前 schema）。 |
| **无 deprecated / legacy 维护** | 代码与文档中 **删除** 仅为旧 schema 存在的分支、占位键、双字段名映射与「静默吞错式回退读」。 |

**非目标（本方案不解决）**

- 跨大版本 **存量用户** 的无损升级路径（另立 ADR / 单独迁移项目）。
- 声学缓存库、行为库以外的 **全仓静默 catch 治理**（见 `docs/execution/governance/静默catch分级台账-2026-03-23.md`，可并行但不在此文档展开）。

---

## 1. 设计原则（结束态）

1. **单轨真源**：每种持久化实体只有一个正式表名与一套字段；业务路径不读「影子表」或占位 `textId`。
2. **校验先于写入**：导入与协作入站均以 **当前 Zod / 类型** 为准；非法数据 **失败可见**（抛错或结构化错误），不在热路径做「猜旧格式」。
3. **主键带作用域**：凡可能被多文档复用的外显 id（如 `mediaId`），在本地表中与 **`textId` / `projectId` 等租户键** 组合保证唯一，避免跨文档覆盖。
4. **身份列不可伪造（云端）**：凡审计向列（`created_by`、`actor_id`、`uploaded_by`、`author_id`），由 RLS `with check` 与 `auth.uid()` **强绑定**；客户端不得写入他人 UUID。
5. **迁移可粗暴**：允许「删表 / 删库 / 单版本 stores 终态」；但必须 **文档化** 且 **测试覆盖空库打开 + 当前 schema CRUD**。

## 3. WP-A — 导入导出（`io.ts`）

### 3.1 当前问题（基线）

- `normalizeImportedDoc` 对 `unit_tokens` / `unit_morphemes` 在 `typeof d.unitId === 'string'` 时抛错，与 `UnitTokenDocType` / `UnitMorphemeDocType` 的 **`unitId: string` 必填** 冲突，导致 **合法快照被拒绝**。

### 3.2 结束态行为

1. **删除** `normalizeImportedDoc` 中与「legacy `unitId`」相关的整个 `case` 分支（或整函数若其余分支仅为 provenance 补全则保留最小逻辑）。
2. **导入路径**：对每个集合文档执行 **现有 `validate*`**；不通过则 **中止导入** 并返回明确错误（保持现有 `ImportResult` 或统一抛错策略，二选一并文档化）。
3. **`schemaVersion`**：仅支持 **当前** `SNAPSHOT_SCHEMA_VERSION`（`io.ts` 常量）；更低或更高版本 **拒绝** 并提示「请使用当前应用导出」或「请升级应用」，**不做** 自动升级映射。
4. **媒体内嵌**：[x] 导出 **不** 将 `audioBlob` 转为 data URL；从导出 JSON 中移除 Blob 并标记 `audioExportOmitted`；导入仍支持 `audioDataUrl` 灌回（见 ADR 0008）。

### 3.3 验收检查清单

- [x] 含合法 `unit_tokens` / `unit_morphemes` 的快照可完成导入（`normalizeImportedDoc` 已改为仅 `ensureImportProvenance`；`importDatabaseFromJson` 套件已绿）。
- [x] 全仓检索无 `legacy field "unitId"` 错误文案路径（除非测试断言负例）。
- [x] `schemaVersion` **必须等于** `SNAPSHOT_SCHEMA_VERSION`（`io.ts`）；否则抛错（含 Zod 要求字段存在）。
- [x] `npx vitest run src/db` 相关套件已绿（含 io 导入路径）。

## 4. WP-B — 轨道实体（`TrackEntityStore` + Dexie）

### 4.1 当前问题（基线）

- 主键 `id = \`track_${mediaId}\`` 导致不同 `textId` 共用媒体时 **后写覆盖**。
- `loadTrackEntityStateMapFromDb` 在无行时回退 **`textId === '__unknown__'` 全表**，占位多时产生 **错误合并**。

### 4.2 结束态设计

1. **文档主键**：`id = \`te:${textId}:${trackKey}\``（实现见 `src/db/trackEntityIds.ts` 的 `trackEntityDocumentId`；禁止仅使用 `track_${trackKey}` 形式作为主键）。
2. **索引**：在 `engine.ts` 的 `track_entities` store 上保证可按 `textId` 查询与 bulkDelete（与现有 `saveTrackEntityStateMapToDb` 事务一致）；若 Dexie 需复合索引，使用 `[textId+mediaId]` 或文档化查询模式。
3. **删除** `__unknown__` 回退分支；**删除** 任何依赖占位 `textId` 的测试数据路径（若有）。
4. **LocalStorage**：[x] 产品路径 **不再** `saveTrackEntityStateMap` 写 `TRACK_ENTITY_STORAGE_KEY`；`load`/`save` LocalStorage API 仍保留在 `TrackEntityStore.ts` 供单测等调用。Dexie **v26** 对 `jieyu:track-entity-state:v1` → `track_entities` 的 upgrade 已改为 **no-op**（ADR 0008）。

### 4.3 验收检查清单

- [x] 同一浏览器中两个 `textId` + 同一轨道键后缀写入后，分别读取互不覆盖（`src/services/TrackEntityStore.db.test.ts`）。
- [x] `loadTrackEntityStateMapFromDb` 在无数据时返回 **空对象**，不查询 `__unknown__`。
- [x] Dexie **v42**：`upgradeM42TrackEntityDocumentIds` 将遗留 `track_*` 主键重写为 `te:` 前缀（`src/db/migrations/m42TrackEntityDocumentIds.ts` + 单测）。

## 5. WP-C — Dexie `engine.ts` 迁移链

### 5.1 目标

在 **无历史数据** 前提下，迁移链应 **可读、可维护**，避免「仅为修复历史误删表」而存在的 **空窗删表版本** 给未来维护者造成误判。

### 5.2 可选策略（择一，写入 ADR 或本计划 §8 执行记录）

| 策略 | 做法 | 优点 | 风险 |
|------|------|------|------|
| **C1 — 新逻辑库名** | 将主库 `name` 改为新常量（如 `jieyudb_v2`），旧名不再打开 | 代码最干净；旧键留在磁盘但不使用 | 需确认无硬编码 `jieyudb` 散落 |
| **C2 — Major 版本 + 启动清库** | 单次 `version(N).stores({...终态...})` + `upgrade` 内 `await db.tables.forEach(t => t.clear())`（谨慎：仅当确认无数据） | 保留库名 | 误发版会清空 **有** 数据的环境 |
| **C3 — 保留版本号但注释「仅历史」** | 不删版本块，在块顶注明「绿场勿模仿」 | 改动最小 | 链仍长，认知负担高 |

**推荐**：优先 **C1** 或 **C2** 之一，并在 `docs/adr/` 增加短 ADR 说明「绿场假设与不可用于存量升级」。

### 5.3 与 `v32` / `v40` 的关系

- 若采用 C1/C2：**评估**是否可折叠或移除「`layer_units: null` 再恢复」的仅历史语义块；若工具链要求保留单调版本号，则至少 **注释** 该段在绿场下等价于 no-op，且 **测试** 仅覆盖「从空库到当前版本」。

### 5.4 验收检查清单

- [x] 空 IndexedDB 下首次打开：`JieyuDexie` 可 `open()`，核心表存在（`src/db/engineGreenfieldOpen.test.ts`）；完整 UI 路径仍以手工 smoke 为准。
- [x] v30–v40 在 `engine.ts` 已用 **集中注释** 折叠叙事；版本块未删除（Dexie 单调性约束）。

## 6. WP-D — 恢复快照（`SnapshotService`）

### 6.1 结束态语义

- 明确为 **进程内崩溃恢复 / 单次覆盖**，不承诺版本历史与跨设备同步。
- `RECOVERY_SCHEMA_VERSION` 变更时：**删除** 旧行或返回 null（保持当前「损坏即删」可接受），但须在本文档与代码注释中写明 **不兼容即丢弃**。

### 6.2 可选增强（非必须）

- [x] 限制 `units` / `translations` / `layers` 合并序列化 **UTF-8 字节**（默认 8MiB；`saveRecoverySnapshot` 第三参 `maxSerializedUtf8Bytes` 供测试收紧），超限则 **跳过写入** 并 `console.debug`（`SnapshotService.ts`）。

### 6.3 验收检查清单

- [x] 行为与文档描述一致；无「兼容多版 recovery」的隐含承诺；体积上限有单测（`SnapshotService.test.ts`）。

## 7. WP-E — Supabase 协同 SQL

### 7.1 结束态（RLS）

对以下表的 **INSERT** `with check` 追加（或与现有条件 **AND**）：

| 表 | 列 | 约束 |
|----|-----|------|
| `project_snapshots` | `created_by` | `created_by = auth.uid()` |
| `project_changes` | `actor_id` | `actor_id = auth.uid()` |
| `project_assets` | `uploaded_by` | `uploaded_by = auth.uid()` |
| `project_comments` | `author_id` | `author_id = auth.uid()`（与 update 策略对称） |

### 7.2 工程交付形式

- **已部署环境**：在 `supabase/sql/` 下新增 **`002_*.sql`**（或项目约定的下一序号）使用 `drop policy ...` + `create policy ...`，**不要**在无迁移记录的情况下直接改写他人已应用的 `001` 文件语义（除非仓库流程明确允许 force reset）。
- **客户端**：插入 payload **仍传** 上述列，但值必须等于会话用户；或由 RPC 默认 `auth.uid()`（若采用服务端写入）。

### 7.3 `project_changes` 唯一键（可选第二阶段）

- [x] 已提供追加迁移 [`../../supabase/sql/003_project_changes_client_op_unique.sql`](../../supabase/sql/003_project_changes_client_op_unique.sql)：删除 `unique (project_id, client_op_id)`，改为 **`unique (project_id, client_id, client_op_id)`**（绿场/空表或协调停机后执行）。

### 7.4 验收检查清单

- [ ] 本地或 CI 中 SQL 静态检查 / `supabase db` 流程（若已接入）通过。
- [ ] 尝试以成员身份插入 `actor_id != auth.uid()` 时被拒绝（集成测试或手动 SQL）。
- [x] 协作客户端：`registerProjectAsset` / `createProjectSnapshot` 在 `useTranscriptionCollaborationBridge` 中 **强制** `uploadedBy` / `createdBy` 为 `getSupabaseUserId()`；`npm run check:collaboration-cloud-foundation` 与 `npm run test:collaboration-supabase-contract` 在本批次变更后已跑通。

## 8. WP-F — 测试与文档联动

### 8.1 必改测试方向

- **删除或改写**：依赖 `__unknown__`、`track_${mediaId}` 唯一、或错误 `unitId` 抛错的测试。
- **新增**：双 `textId` 同 `mediaId` 轨道状态隔离；导入含子图 token/morpheme 的合法快照。

### 8.2 文档联动

- 发版或协作相关：`docs/architecture/collaboration-cloud.md` 若涉及身份写入假设，在合并后 **更新 `last_reviewed`** 与一小节「RLS 身份绑定」。
- 若引入新库名或 major 清库：**新增 ADR**（建议 `docs/adr/000x-greenfield-dexie-reset.md`），并在 `docs/发布说明-*.md` 中写一句 **数据清空范围**（若对外发布）。

### 8.3 合并前命令（仓库约定）

```bash
npm run check:docs-governance
npm run check:architecture-guard
```

（若改动 SQL 或协作路径，再跑协作相关 gate，以 `package.json` 脚本为准。）

### 8.4 无云端环境验收（不连真实 Supabase）

以下命令 **不要求** Supabase 项目、环境变量或外网调用真实 Postgres；协作相关单测使用 **mock / fake-indexeddb**，仅验证客户端协议与桥接逻辑。

```bash
npm run gate:greenfield-local
```

等价串联：`check:docs-governance` → `check:architecture-guard` → Dexie/轨道/恢复快照相关 Vitest → M2 导入导出幂等 → `check:collaboration-cloud-foundation`（静态文件与片段契约）→ `test:collaboration-supabase-contract`（协作云 Vitest 套件）。

**不包含**：在 Supabase 控制台执行 `001`/`002`/`003`、RLS 手工插入负例、`supabase db` 等。具备云端项目后，再补跑 `npm run gate:collaboration-cloud`（会额外写 `collaboration-cloud-gate-report-auto.md` 并二次跑部分 Vitest）。

**CI**：`.github/workflows/ci.yml` 中 **`greenfield-local-gate`** job 对每个 PR / `main`·`dev` push 执行 `npm run gate:greenfield-local`（不依赖 Supabase Secret）。

## 9. 建议实施顺序（依赖关系）

1. **WP-A**（导入导出）— 解除当前硬错误，测试基线恢复。  
2. **WP-B**（轨道实体）— 数据形状与主键一次到位。  
3. **WP-C**（Dexie 链）— 与 WP-B 的索引变更可同 PR 或紧接 PR，避免多次破坏性打开。  
4. **WP-E**（Supabase）— 可与 1–3 并行，由负责后端的成员交付迁移文件。  
5. **WP-D**（恢复库）— 可并行，优先级低于 1–3。  
6. **WP-F** — 贯穿各 PR。
