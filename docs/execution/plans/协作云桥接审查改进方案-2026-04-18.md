# 协作云桥接审查改进方案

> 本文档为执行清单：只描述「做什么 + 验收句」，业务事实源见 `docs/architecture/` 与代码。

## 背景

- 审查依据：`docs/architecture/仓库现状与代码地图.md`、`docs/execution/plans/托管实时协同-Supabase完整落地方案-2026-04-17.md`。
- 范围：**Supabase 托管协作云桥接**（`src/collaboration/cloud/`、`useTranscriptionCloudSyncActions`、`useTranscriptionCollaborationBridge`）。

## P0（两周内）

| ID | 行动 | 验收 |
|----|------|------|
| P0-1 | 在转写 ready 阶段露出协议守卫与同步徽标（`CollaborationCloudReadOnlyBanner`、`CollaborationSyncBadge`），消费 `derivedApi` | 进入 `/transcription` ready 态可见 banner（写禁用时）与 badge |
| P0-2 | 抽取 `realtimeSubscription.ts`，`CollaborationSyncBridge` 与 `CollaborationPresenceService` 共用 subscribe | 两处无重复 `subscribeChannel` 实现 |
| P0-3 | `project_presence` / `projects` / `project_members` 的 Supabase 直连下沉到 `collaboration/cloud/` 服务 | `grep` `src/hooks/useTranscriptionCloudSyncActions.ts` 无 `getSupabaseBrowserClient` |
| P0-4 | `CollaborationCloudPanel.directory`：在 ReadyWorkspace 传入 `listAccessibleProjects` / `listProjectMembers` 或移除分支 | 无「冷冻」directory UI 分支 |

## P1（一个月）

| ID | 行动 | 验收 |
|----|------|------|
| P1-1 | 新增 `docs/architecture/collaboration-runtime-map.md` | 每个 `collaboration*Runtime.ts` 有生产/门禁/历史标签 |
| P1-2 | 新增 `useTranscriptionCloudSyncActions.applyRemote.test.tsx` | Vitest 覆盖入站→冲突→日志路径 |
| P1-3 | CI 独立 job：`gate:collaboration-cloud`；`gate:m14-collaboration-promotion` 显式依赖 | workflow 绿；M14 脚本链包含 cloud gate |
| P1-4 | 刷新 `docs/execution/plans/README.md`、`docs/execution/release-gates/README.md` 的 `last_reviewed`；更新 Supabase 方案 `status` | 元数据与内容阶段一致 |

## P2（两个月）

| ID | 行动 | 验收 |
|----|------|------|
| P2-1 | `resolveCollaborationConflicts` 内发 `aiTrace`/metrics，冲突解决日志带 `traceId` | 域日志对象含 `traceId`；指标或 observer 可断言 |
| P2-2 | `architecture-guard`：禁止 `src/hooks/**` 直接 import `src/integrations/supabase/**` | `npm run check:architecture-guard` 通过 |
| P2-3 | 稳定后新增 `docs/architecture/collaboration-cloud.md` 现状基线 | ADR/方案可引用该文件 |

## 不做

- 不重写 `collaborationBetaRuntime` 等旧 runtime 业务逻辑。
- 不改 `supabase/sql/001_collaboration_foundation.sql` schema。
- 不把 `CollaborationAuditLogService` 升级为独立审计表。

## 外部依赖

- OTel 根 span / `fetch` 级传播：走 `OTel-Trace透传与ConflictResolved五阶段执行清单`，本方案 P2-1 仅预留钩子。
