## 0. 文档定位

本方案把「结论—证据索引」方法论、AI 工具权限策略单一事实源、会话与项目记忆统一编排、工具契约可 diff、扩展与观测可证明性、性能门限与发布联动、语音运行时渐进平台化等事项，收敛为**分阶段、可验收、可回滚**的执行规格。与下列既有能力对齐，避免重复造轮子：

- 工具审计与回放：[src/ai/auditReplay.ts](src/ai/auditReplay.ts)、[src/hooks/useAiChat.toolAudit.ts](src/hooks/useAiChat.toolAudit.ts)
- 指标聚合：[src/observability/metrics.ts](src/observability/metrics.ts)（`summarizeMetrics`）
- 工具决策管道与闸门：[src/hooks/useAiChat.toolDecisionPipeline.ts](src/hooks/useAiChat.toolDecisionPipeline.ts)、[src/hooks/useAiChat.destructiveGate.ts](src/hooks/useAiChat.destructiveGate.ts)
- 转写页风险检查：[src/pages/transcriptionAiToolRiskCheck.ts](src/pages/transcriptionAiToolRiskCheck.ts)
- 扩展运行时与审计尾：[src/extensions/extensionRuntime.ts](src/extensions/extensionRuntime.ts)、[src/extensions/extensionRegistry.ts](src/extensions/extensionRegistry.ts)
- Schema 与校验：[src/ai/chat/toolCallSchemas.ts](src/ai/chat/toolCallSchemas.ts)
- 会话循环与 token 预算：[src/hooks/useAiChat.agentLoopRunner.ts](src/hooks/useAiChat.agentLoopRunner.ts)
- 项目记忆检索：[src/services/ProjectMemoryStore.ts](src/services/ProjectMemoryStore.ts)（`getRagContextVector`）
- 会话记忆归一化：[src/ai/chat/sessionMemory.ts](src/ai/chat/sessionMemory.ts)
- 发布与观测脚本：[package.json](package.json)（`report:m5-trend`、`gate:m6-release`、`perf:*` 等）
- 语音服务：[src/services/VoiceAgentService.ts](src/services/VoiceAgentService.ts)

## 1. 总目标与成功画像

### 1.1 总目标（约 6 个月滚动达成）

1. **发布决策可证据化**：里程碑或 CI 能产出固定版式的发布资产，含结论、证据索引、未通过项与复现命令。
2. **工具与扩展策略可审计、可扩展**：单一策略源描述工具与扩展能力与确认规则；拒绝原因码稳定，可统计。
3. **记忆召回在预算内可解释**：统一编排层输出带「为何命中」元数据的上下文块。
4. **契约与脱敏可证明**：工具参数契约可自动生成并与前版 diff；敏感字段脱敏有覆盖率与回归清单。

### 1.2 非目标（本方案不承诺）

1. 替换现有全部 `gate:m*` 链为单一脚本（仅在其上增加统一呈现与索引层）。
2. 一次性重写语音栈或引入未讨论的新引擎。
3. 将本文件升格为产品架构唯一真源；架构事实仍以 [docs/architecture/](docs/architecture/) 与代码为准。

## 2. 分阶段路线图（A → B → C）

执行顺序：**必须先完成阶段 A 的可运行最小闭环**，再进入 B；C 仅在触发条件满足时启动。

| 阶段 | 时间盒（建议） | 主题 | 核心交付物 |
|------|----------------|------|-------------|
| A | 4–8 周 | 证据化主链路 | 统一发布报告生成器 + AI 证据卡 + 时间轴 perf 证据卡 + 文档符号守卫 |
| B | 6–10 周（可与 A 尾部重叠） | 策略 SSOT | 工具/扩展策略表 + 渐进迁移 + 发布报告中的策略摘要 |
| C | 触发式 | 记忆 broker 与语音/扩展加深 | 见第 8 节触发条件 |

## 3. 阶段 A：证据化主链路（P0）

### 3.1 工单 A1 — 统一发布报告生成器（Release Evidence Bundle）

**目标**：每次 `npm run report:release-evidence`（新脚本，名称可调整）产出**一份**主报告（建议默认 JSON，可选 Markdown 摘要），并写入约定目录（建议：`docs/execution/release-gates/` 下按日期或 CI run id 子目录，与 [docs/execution/release-gates/](docs/execution/release-gates/) 治理一致）。

**报告必选章节**：

1. **元数据**：git 短 SHA、时间戳、`package.json` 中与应用相关的版本字段、运行环境（ci/local）。
2. **摘要**：通过/未通过/跳过门限的数量表。
3. **证据索引表**（核心）：列包括 `结论 id`、`结论简述`、`证据类型`（audit_replay | golden_diff | metric_summary | perf_baseline | gate_script | doc_guard）、`产物路径或命令`、`关联模块`。
4. **子进程输出引用**：对已存在的 `report:m5-trend`、`gate:m6-release`（或 shadow）、`report:m7-extension-gate` 等，只存**退出码 + 日志路径 + 关键一行摘要**，不重写其内部逻辑。

**实现要点**：

1. 新增 `scripts/generate-release-evidence-bundle.mjs`（或 `.ts` 与现有脚本风格一致）， orchestrate 子脚本；失败时仍写出部分报告并标记 `degraded: true`。
2. 在 [package.json](package.json) 增加 `report:release-evidence` 与可选 `gate:release-evidence`（enforce 模式调用 m6 等）。
3. 与 [docs/execution/release-gates/](docs/execution/release-gates/) 约定：产物可机器引用，人工可读摘要可同目录 `.md`。

**验收标准**：

1. 本地一条命令生成完整 JSON，且 `证据索引表` 至少包含 5 行有效引用。
2. CI 中失败时报告仍生成，且能定位到首个失败子步骤。
3. `npm run check:docs-governance` 通过。

**建议验证命令**：

1. `node scripts/generate-release-evidence-bundle.mjs`（实现后）
2. `npm run check:docs-governance`

**风险与回滚**：低风险；回滚为移除脚本入口与 CI job，不影响业务代码。

---

### 3.2 工单 A2 — AI 工具调用「证据卡」

**目标**：在发布报告中嵌入结构化「AI 工具证据卡」片段（可多条），便于发布评审与回归对照。

**字段建议（v1）**：

- `requestId`、`toolName`、`replayable`、`latestDecision.decision`、`diff.matches`（若有 baseline 文件路径则填）
- `goldenSnapshotPath`（可选，CI 产出路径）
- `auditLogQueryHint`：例如 `collection=ai_messages` + `field=ai_tool_call_decision` + `requestId`

**数据来源**：

1. 回放装配：`loadAiToolReplayBundle`、`buildAiToolGoldenSnapshot`、`serializeAiToolGoldenSnapshot`、`diffAiToolSnapshot`（[src/ai/auditReplay.ts](src/ai/auditReplay.ts)）。
2. 审计写入：`writeToolDecisionAuditLog`（[src/hooks/useAiChat.toolAudit.ts](src/hooks/useAiChat.toolAudit.ts)）。

**实现要点**：

1. 在 A1 脚本中增加可选步骤：从最近一次 E2E 或指定 fixture 导出 sample `requestId` 列表（或由环境变量注入）；无 DB 时跳过并标记 `skipped`。
2. 不在热路径增加同步 IO；发布流水线专用。

**验收标准**：报告中至少 1 张证据卡字段齐全；与手工从 DB 导出对照一致。

**风险**：fixture 与真实环境差异 —— 用 `skipped` 显式表达，不伪造数据。

---

### 3.3 工单 A3 — 时间轴性能「证据卡」

**目标**：将时间轴与轨道布局 perf 门限结果结构化写入发布报告。

**数据来源**：

1. `npm run perf:track` → [src/perf/TrackLayoutPerformanceBaseline.test.ts](src/perf/TrackLayoutPerformanceBaseline.test.ts)
2. `vitest run src/perf/TimelineUnitViewIndexPerformanceBaseline.test.ts`（或并入 `perf:timeline` 新脚本）
3. 可选：`perf:ai` → [src/ai/perf/AiPerformanceBaseline.test.tsx](src/ai/perf/AiPerformanceBaseline.test.tsx)

**实现要点**：

1. v1：解析 vitest verbose 输出中的关键行（已有 `console.info` 前缀的可作为稳定锚）；v2：改为 `--reporter=json` 写入临时文件再解析。
2. 报告中记录：`medianMs` 或 `elapsed`、门限 `maxMs`、是否通过。

**验收标准**：同机两次运行报告数值可复现（允许微小浮差，文档说明容差）。

**风险**：CI 时长 —— 默认放入 `gate:release-evidence:full`，日常 PR 用 `lite` 跳过 perf 全量。

---

### 3.4 工单 A4 — 架构文档与代码符号一致性守卫

**背景**： [docs/architecture/timeline-unit-governance.md](docs/architecture/timeline-unit-governance.md) 提及 `LEGACY_LOCAL_CONTEXT_TOOL_ALIAS_MAP` 与 legacy 工具名；当前 [src/ai/chat/localContextTools.ts](src/ai/chat/localContextTools.ts) 为 canonical 集合与 `normalizeToolName`，仓库内无该常量符号。

**目标**：二选一或组合，必须闭环：

1. **守卫**：新增 `scripts/check-doc-code-symbol-parity.mjs`（名称可调整），对允许列表中的架构文档扫描反引号内 `src/...` 路径与标识符，校验符号在目标文件存在（或反向：禁止引用不存在符号）。
2. **修订文档**：若别名阶段已结束，更新 governance 文档删除虚构移除闸门，或改为「当前实现为 canonical-only，无 LEGACY map」。

**验收标准**：`npm run check:architecture-guard` 与新建检查脚本在 CI 中通过；文档与代码无矛盾陈述。

**风险**：误杀合法文档 —— 用允许列表与豁免注释标记。

## 4. 阶段 B：统一 AI 工具与扩展策略（P0）

### 4.1 工单 B1 — 策略 SSOT 数据模型与存放位置

**目标**：引入单一结构化策略源（建议：`src/ai/policy/aiToolPolicyMatrix.json` + 生成的 `aiToolPolicyMatrix.types.ts`，或纯 TS `const` 满足 typecheck），描述每一工具：

- `toolName`（与 [src/ai/chat/toolCallSchemas.ts](src/ai/chat/toolCallSchemas.ts) 中 `AiChatToolName` 对齐）
- `riskTier`：`low` | `medium` | `high`
- `destructive`：boolean
- `requiresExplicitTarget`：boolean 或更细粒度字段集
- `confirmationMode`：`none` | `pending_propose_changes` | `host_modal` 等枚举
- `auditReasonCodes`：与 `writeToolDecisionAuditLog` 中 `newValue` / metadata 可映射的稳定码列表
- `extensionCapabilityHints`：可选，映射到 [src/extensions/extensionRuntime.ts](src/extensions/extensionRuntime.ts) 中 `ExtensionCapability`

**权威规则**：以**用户可见最终结果**（拒绝文案、待确认状态、任务会话状态）反推各层校验；SSOT 为规范，现有模块为执行器。

**验收标准**：新增工具时仅改 SSOT + 测试即可通过 pipeline。

---

### 4.2 工单 B2 — 渐进迁移执行器

**迁移顺序（建议）**：

1. [src/pages/transcriptionAiToolRiskCheck.ts](src/pages/transcriptionAiToolRiskCheck.ts) 从 SSOT 读取「需显式 target」类规则，删除重复硬编码列表。
2. [src/hooks/useAiChat.destructiveGate.ts](src/hooks/useAiChat.destructiveGate.ts) 对齐 `destructive` 与 `confirmationMode`。
3. [src/hooks/useAiChat.toolDecisionPipeline.ts](src/hooks/useAiChat.toolDecisionPipeline.ts) 仅保留编排，不保留分散的魔法字符串比较（逐步收）。

**兼容策略**：保留一期 `LEGACY_TOOL_POLICY_OVERRIDE` 环境变量或特性开关，默认关闭；对照测试覆盖 `ambiguous_target`、`invalid_proposed_changes`、扩展 capability deny。

**验收标准**：现有 vitest 全绿 + 新增「策略表与运行时一致性」单测文件。

**风险**：迁移遗漏 —— B1 中 `auditReasonCodes` 与日志解析脚本对齐，发布报告统计异常码突增可报警。

**验收标准**：无扩展加载时章节为空数组且不失败。

## 8. 阶段 C：触发式加深（P0/P1/P2）

### 8.1 触发条件（满足任一即启动 C 对应工单）

1. `report:m5-trend` 或发布报告中时间轴 perf 卡连续 **2 个里程碑** 退化超约定容差。
2. `ambiguous_target` 或 `blocked_write_without_explicit_target_total`（见 [src/pages/transcriptionAiToolRiskCheck.ts](src/pages/transcriptionAiToolRiskCheck.ts)）在聚合指标中持续高位。
3. 语音场景确认需 **2+ 引擎** 并行灰度（产品决策）。

### 8.2 工单 C1 — Memory Broker（统一检索层）

**目标**：新增 `src/ai/memory/memoryBroker.ts`（路径可调整），统一 API 形如：

```ts
// 规格说明用伪代码；实现时以实际类型为准
retrieveMemoryContext(input: {
  query: string;
  tokenBudget: number;
  session: AiSessionMemory;
  projectMemoryStore: ProjectMemoryStore | null;
}): MemoryChunk[];
interface MemoryChunk {
  text: string;
  source: 'session_summary' | 'session_preferences' | 'project_rag' | ...;
  score?: number;
  why: string; // 人类可读：命中原因
}
```

**约束**：与 [useAiChat.agentLoopRunner.ts](src/hooks/useAiChat.agentLoopRunner.ts) 中 token 预算逻辑协同；broker 必须支持「截断后仍返回 why」。

**验收标准**：单测覆盖预算截断与空 store；不增加默认上下文大小（默认行为与现网一致，开关启用 broker）。
