## 1. 术语、命名空间与发布证据契约

### 1.1 命名空间（实现与文档须一致）


| 类别                   | 含义                                                       | 示例                                                  |
| -------------------- | -------------------------------------------------------- | --------------------------------------------------- |
| **审计 reason / 事件标签** | 写入工具决策审计或门控路径的**字符串原因**，用于回放与发布卡                         | `ambiguous_target`、`auto_failed:*:ambiguous_target` |
| **指标 id**            | `src/observability/metrics.ts` 注册的 **metric id**，用于聚合与趋势 | `blocked_write_without_explicit_target_total`       |
| **报告字段名**            | `release-evidence.json` 等产物中的 JSON 键                     | `progressiveDisclosure.cards[]`、`perf.cards[]`      |


触发条件、告警与工单描述中，**禁止混用**「审计 reason」与「metric id」而不加标注。

### 1.2 Release Profile（发布证据 bundle 的运行档）

A1  orchestrate 的子步骤必须声明本次使用的 **profile**，并写入报告元数据，避免读者默认「全仓库所有 gate 均已执行」。


| Profile        | 典型包含                                                                                    | 适用场景       |
| -------------- | --------------------------------------------------------------------------------------- | ---------- |
| `lite`         | `check:docs-governance`、`check:architecture-guard`、核心 vitest 子集、跳过或 shadow 部分重门限        | 日常 PR、快速反馈 |
| `core`         | 在 `lite` 基础上纳入 `gate:m6-release`（或 shadow）、`report:m5-trend`、扩展 gate 等与 AI/观测主链路直接相关的条目 | 常规里程碑      |
| `collab-cloud` | 在 `core` 基础上串联 `gate:collaboration-cloud` 及团队约定的协作里程碑子集                                 | 协作能力发布周    |
| `full`         | 在选定基线上纳入完整 perf、shadow 对比、Week 6 DoD 所列必检项等                                             | 发布候选、重大版本  |


**首版 A1 落地约束**：首版 bundle 至少串联 `**lite` 或 `core` 之一** 的可运行子集并产出有效证据索引；`full` 为演进目标，避免第一版即绑定全部 `gate:m`* 导致不可维护。

### 1.3 编排器行为契约（对齐无头/CI 产品的工程习惯）

`generate-release-evidence-bundle` 建议约定：

1. **退出码**：任一 enforce 子步骤失败 → 非零退出；若配置为「失败仍写报告」，则同时设置报告根字段 `degraded: true` 并列出失败步骤（与「静默成功」区分）。
2. **部分产物**：子进程失败时仍写出 JSON/Markdown 摘要，且 `证据索引表` 中失败行含**退出码、日志路径、首条错误摘要**。
3. **可复现**：报告元数据含 git 短 SHA、profile 名、主要 npm script 版本（与 A1 正文一致）。

## 2. 总目标与非目标

### 2.1 总目标（约 6 个月滚动达成）

1. **发布决策可证据化**：里程碑或 CI 能产出固定版式的发布资产，含结论、证据索引、未通过项与复现命令。
2. **工具与扩展策略可审计、可扩展**：单一策略源描述工具与扩展能力与确认规则；拒绝原因码稳定，可统计。
3. **记忆召回在预算内可解释**：统一编排层输出带「为何命中」元数据的上下文块。
4. **契约与脱敏可证明**：工具参数契约可自动生成并与前版 diff；敏感字段脱敏有覆盖率与回归清单。

### 2.2 非目标（本方案不承诺）

1. 替换现有全部 `gate:m`* 链为单一脚本（仅在其上增加统一呈现、索引层与 profile 调度）。
2. 一次性重写语音栈或引入未讨论的新引擎。
3. 将本文件升格为产品架构唯一真源；架构事实仍以 [docs/architecture/](docs/architecture/) 与代码为准。

## 3. 路线图、渐进披露与近 6 周节奏

### 3.1 分阶段路线图（A → B → C）

执行顺序：**必须先完成阶段 A 的可运行最小闭环**，再进入 B；C 仅在触发条件满足时启动（见第 7.1 节）。


| 阶段  | 时间盒（建议）           | 主题                 | 核心交付物                                      |
| --- | ----------------- | ------------------ | ------------------------------------------ |
| A   | 4–8 周             | 证据化主链路             | 统一发布报告生成器 + AI 证据卡 + 时间轴 perf 证据卡 + 文档符号守卫 |
| B   | 6–10 周（可与 A 尾部重叠） | 策略 SSOT            | 工具/扩展策略表 + 渐进迁移 + 发布报告中的策略摘要               |
| C   | 触发式               | 记忆 broker 与语音/扩展加深 | 见第 7 节触发条件与工单                              |


### 3.2 渐进披露（Progressive Disclosure）约束

为避免“功能都在但首屏过载”，本方案补充以下治理约束（对齐 NNGroup / IxDF 的通用定义）：

1. 首层仅承载高频任务必需能力；低频或高风险能力按需展开，不默认灌入。
2. 次层入口必须可发现、可预期（文案明确：展开后会看到什么）。
3. 核心任务应可在首层完成；仅在信息不足或风险升高时进入澄清/确认。
4. 默认不超过 2 级披露；如需更多层级，必须提供分组依据与可用性测试证据。
5. 渐进披露不是“隐藏能力”；需提供可追溯路径（入口、提示、回退）。

### 3.3 当前实现判定（2026-04-21 快照）

本仓当前状态（用于后续验收对照）：

1. **已满足**：阶段化落地与触发式加深（A→B→C），见第 3.1 节与第 7 节。
2. **已满足**：运行时具备澄清分流与风险门控（`execute/clarify/ignore/cancel`、`gray/rollback`）。
3. **部分满足**：检索编排仍以单层召回为主，尚缺“候选召回→重排选择→预算注入”完整链。
4. **部分满足**：可发现性主要依赖运行时澄清，缺“前置高级能力引导”统一规范。

### 3.4 近 6 周执行节奏

为确保增量落地且不扰动现网，执行节奏由 4 周扩展为 6 周，纳入跨仓对照形成的 7 项新增借鉴（与第 6 节、第 7 节工单对齐）：

1. **第 1 周：意图多候选 + 低置信澄清触发（基于现有 intent 分流）** — 对齐 `7.5 工单 C4`；周验收：`clarify_loop_rate` 不劣化；`clarify_then_success_rate` 建立首版基线。
2. **第 2 周：工具语义后验检查与 reason code** — 对齐 `5.2 工单 B2` + `4.2 工单 A2`；周验收：新增后验失败原因码，发布报告可见并可追溯到 requestId。
3. **第 3 周：Memory Broker 最小版 + 检索形态遥测** — 对齐 `7.2 工单 C1` + `6.6 工单 P1-RagShapeTelemetry`；周验收：空 store、预算截断、两阶段检索；`why` 全覆盖；召回形态指标可入报告。
4. **第 4 周：context 自动压缩触发 + 失败熔断 + 状态回注一致性** — 对齐 `6.4 工单 P1-ProgressiveDisclosure`（门限追踪）+ agent loop 预算逻辑；周验收：压缩失败熔断；关键状态可回注。
5. **第 5 周：成本治理硬阈值 + 输出 token 封顶与升级重试** — 对齐 `6.5 工单 P1-CostGuard`；周验收：会话预算硬停止、默认输出封顶、escalate retry 可审计。
6. **第 6 周：后台记忆抽取子代理 + 工具沙箱 + 轻量协调模式** — 对齐 `7.6–7.8 工单 C5–C7`；周验收：后台不阻塞主链路；沙箱可证明；相位与并行策略可审计。

**执行约束**：每周只引入一个主能力面，跨周能力默认 behind-flag；必须在 `report:release-evidence` 中留下对应证据卡与指标快照（profile 由第 1.2 节约束）。

### 3.5 跨仓对照新增的 7 项借鉴（代码证据导向）

1. 后台记忆抽取子代理（与主链路互斥，避免重复写入）。
2. 后台任务工具权限沙箱（只读优先、受限写入范围）。
3. 检索形态遥测（含空选、候选规模、抑制率等分母指标）。
4. 成本治理硬阈值（会话预算上限与可解释终止原因）。
5. 输出 token 封顶 + 升级重试（capped default + escalation）。
6. 上下文压缩后的状态回注一致性（任务状态、摘要、最近工具上下文）。
7. 轻量协调模式与并行策略规范（research / synthesis / implementation / verification）。

## 4. 阶段 A：证据化主链路（P0）

### 4.1 工单 A1 — 统一发布报告生成器（Release Evidence Bundle）

**目标**：每次 `npm run report:release-evidence`（新脚本，名称可调整）产出**一份**主报告（建议默认 JSON，可选 Markdown 摘要），并写入约定目录（建议：`docs/execution/release-gates/` 下按日期或 CI run id 子目录，与 [docs/execution/release-gates/](docs/execution/release-gates/) 治理一致）。

**报告必选章节**：

1. **元数据**：git 短 SHA、时间戳、`releaseProfile`（见第 1.2 节）、`package.json` 中与应用相关的版本字段、运行环境（ci/local）。
2. **摘要**：通过/未通过/跳过门限的数量表。
3. **证据索引表**（核心）：列包括 `结论 id`、`结论简述`、`证据类型`（audit_replay | golden_diff | metric_summary | perf_baseline | gate_script | doc_guard）、`产物路径或命令`、`关联模块`；可选列：`symbolOrSchemaVersion`（稳定符号或契约版本，便于跨发布 diff）。
4. **子进程输出引用**：对已调用的 `report:m5-trend`、`gate:m6-release`（或 shadow）、`report:m7-extension-gate` 等，只存**退出码 + 日志路径 + 关键一行摘要**，不重写其内部逻辑。若 profile 包含协作链，则同样只引用对应 `gate:m8`…或 `gate:collaboration-cloud` 的摘要行，**不在此方案中改各 gate 脚本语义**。
5. **渐进披露指标卡**：至少包含 `first_pass_resolution_rate`、`clarify_then_success_rate`、`advanced_entry_reach_rate`、`token_budget_checkpoint_rate` 四项（允许 `skipped`，但必须显式标注原因）；指标**数据源**须在实现工单中落到具体审计字段或 metric id（见第 6.4 节）。

**实现要点**：

1. 新增 `scripts/generate-release-evidence-bundle.mjs`（或 `.ts` 与现有脚本风格一致），按 profile orchestrate 子脚本；行为符合第 1.3 节。
2. 在 [package.json](package.json) 增加 `report:release-evidence` 与可选 `gate:release-evidence`（内部再分 `:lite` / `:core` / `:full` 或环境变量切换）。
3. 与 [docs/execution/release-gates/](docs/execution/release-gates/) 约定：产物可机器引用，人工可读摘要可同目录 `.md`。

**验收标准**：

1. 本地一条命令生成完整 JSON，且 `证据索引表` 至少包含 5 行有效引用。
2. CI 中失败时报告仍生成，且能定位到首个失败子步骤。
3. `npm run check:docs-governance` 通过。

**风险与回滚**：低风险；回滚为移除脚本入口与 CI job，不影响业务代码。

### 4.4 工单 A4 — 架构文档与代码符号一致性守卫

**背景**：[docs/architecture/timeline-unit-governance.md](docs/architecture/timeline-unit-governance.md) 提及 `LEGACY_LOCAL_CONTEXT_TOOL_ALIAS_MAP` 与 legacy 工具名；当前 [src/ai/chat/localContextTools.ts](src/ai/chat/localContextTools.ts) 为 canonical 集合与 `normalizeToolName`，仓库内无该常量符号。

### 6.4 工单 P1-ProgressiveDisclosure — 渐进披露专项门限

**目标**：将“渐进披露是否有效”从描述性结论转为可追踪门限。

**指标定义（v1）**：

1. `first_pass_resolution_rate`：无需澄清即可完成的会话占比。
2. `clarify_then_success_rate`：进入 `waiting_clarify` 后一次补充即成功的占比。
3. `advanced_entry_reach_rate`：进入次层能力（如预算继续、展开摘要、高级工具确认）的会话占比。
4. `clarify_loop_rate`：连续 2 次及以上澄清仍未完成的占比（应受控）。

**数据源落点（实现强制）**：每条比率须注明**分子分母来源**（例如：来自 `writeToolDecisionAuditLog` 的某 metadata 字段、`metrics.ts` 的某 id、或 `full` profile 下采样的 requestId 列表）。若某环境无法采集，该卡片的对应字段显式 `skipped` 并附 `skipReason`，避免报告仅具空壳字段。

**验收标准**：

1. A1 报告中新增 `progressiveDisclosure.cards[]`。
2. 支持 `lite/full` 两档：`lite` 仅统计比率，`full` 增加样本 requestId。
3. 至少 1 个发布周期出具趋势图（可复用 `report:m5-trend` 输出机制）。

### 6.5 工单 P1-CostGuard — 成本阈值与输出上限治理

**目标**：将现有 token 预算预警升级为成本与输出双阈值治理。

**交付**：

1. 会话级预算上限（超限时产生稳定审计 reason，并在结果层可见；与第 1.1 节命名一致）。
2. 输出 token capped default（默认封顶），超限场景触发一次升级重试。
3. 报告新增成本治理卡：预算触发次数、重试触发率、重试成功率。
4. 子代理上下文瘦身策略：按任务类型裁剪上下文（如只读研究代理默认省略高成本上下文块），并保留 kill switch。

**多厂商 / 多模型注意**：不同 provider 的 token 计数与计费口径可能不一致；成本卡元数据须记录**采用的估算规则版本**（例如统一用本地 tokenizer 估算或供应商返回值），否则跨发布对比易失真。

**验收标准**：

1. 不影响正常短会话成功率。
2. 预算超限与升级重试均可被审计回放。
3. 上下文瘦身开启后，token 消耗下降且核心成功率不劣化（至少一个发布周期对照）。

### 7.2 工单 C1 — Memory Broker（统一检索层）

**目标**：新增 `src/ai/memory/memoryBroker.ts`（路径可调整），统一 API 形如：

```ts
// 规格说明用伪代码；实现时以实际类型为准
retrieveMemoryContext(input: {
  query: string;
  tokenBudget: number;
  session: AiSessionMemory;
  projectMemoryStore: ProjectMemoryStore | null;
}): MemoryChunk[];
interface MemoryChunk {
  text: string;
  source: 'session_summary' | 'session_preferences' | 'project_rag' | ...;
  score?: number;
  why: string; // 人类可读：命中原因
  freshnessDays?: number; // 可选：距今天数，用于陈旧提示
  suppressedBy?: 'already_surfaced' | 'recent_tools' | 'budget' | null;
}
```

**约束**：与 [useAiChat.agentLoopRunner.ts](src/hooks/useAiChat.agentLoopRunner.ts) 中 token 预算逻辑协同；broker 必须支持「截断后仍返回 why」。

**检索编排要求**：

1. 两阶段检索：`候选召回`（低成本）→ `重排选择`（限额 topN）。
2. 默认抑制重复暴露：支持 `already_surfaced` 过滤。
3. 对近期工具文档噪声抑制：支持 `recent_tools` 过滤。
4. 预算不足时允许降级，但需返回 `suppressedBy='budget'` 的可解释标记。

**快照同步协议**：

1. 增加 `snapshotTimestamp` 与 `syncedFrom` 元数据，用于判断本地记忆与项目快照是否一致。
2. 同步动作三态：`initialize`（首次同步）、`prompt_update`（提示是否更新）、`none`（无需动作）。
3. 进行 replace 同步前先清理本地孤儿记忆文件，避免旧内容残留误召回。

**验收标准**：单测覆盖预算截断与空 store；不增加默认上下文大小（默认行为与现网一致，开关启用 broker）。

4. 命令语义级 fail-closed：变量展开、花括号扩展、process substitution、`cd + 写操作` 复合链路默认不自动放行。
5. 只读命令采用「命令 + 参数白名单」验证；对 deny/ask 输出稳定审计 reason（与第 1.1 节一致）。

1. 沙箱拒绝具备稳定 reason。
2. 非授权写入在回归测试中全部被拦截。
3. 安全绕过样例（变量注入、重定向绕过、`cd && write`）全部落入 ask/deny，不出现静默放行。

### 7.8 工单 C7 — 轻量协调模式与并行策略规范（P2）

**前置**：C5/C6 稳定运行。

**目标**：引入轻量协调语义层，不引入全量多智能体编排复杂度（与「全 swarm」方案刻意保持距离）。

**交付**：

1. 任务相位：`research` / `synthesis` / `implementation` / `verification`。
2. 并行策略：只读并行、写入串行、验证可并行。
3. 任务通知协议最小版（完成/失败/中止 + usage 摘要）。
4. 任务通知 schema：`taskId`、`status`、`summary` 必填；`result`、`usage` 选填；不合规通知进入隔离通道。
5. 子代理生命周期收尾清单：任务结束后清理临时上下文、挂起任务、会话级遗留状态，避免僵尸任务与状态泄漏。
6. **可观测性**：相位切换与并行决策写入审计元数据；若存在子代理身份，建议增加稳定 `actorId`/`taskId` 维度（与 C7 通知 schema 对齐），避免仅依赖自然语言描述。

**验收标准**：

1. 相位切换与并行决策可观测（审计元数据可追踪）。
2. 复杂任务平均往返轮次下降或持平。
3. 生命周期收尾完成率可统计，且残留后台任务数在会话结束后归零。
