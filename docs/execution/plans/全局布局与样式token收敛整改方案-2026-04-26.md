# 全局布局与样式 token 收敛整改方案

**状态**：已拍板；Phase A–E 已分步落地（持续压降例外）  
**日期**：2026-04-26  
**范围**：桌面端 `src/styles/**` 与语言资产/壳层合约（`.la-shell`）相关的布局与视觉一致性；不涵盖产品功能需求变更。

---

## 1. 目标

- **形状语言统一**：圆角、阴影、卡片外轮廓尽量从 `tokens.css` 语义档位派生，减少页面内硬编码 `18px` / `24px` / 长串 `box-shadow`。
- **间距节奏统一**：节与节、表单行与行之间优先使用 `--space-*` / `--space-stack-*` / `--la-section-gap`，减少「10px/14px/6px 双轨」。
- **控制件心智单一化**：在应用壳内明确 `.input` / `.btn` 与 `.panel-input` / `.panel-button` 的边界，避免同屏微差。
- **合约层上收**：语言资产与重型对话框已部分落在 `language-asset-section-contract.css`；继续将「多页重复的壳/节间距」上收到合约或 foundation，减少 LM/OM/OB 三处平行覆盖。
- **可验证**：每阶段有 `npm` 检查与 E2E/视觉基线，不增加 silent regression。

---

## 2. 现状摘要（审计结论复用）

| 问题域 | 表现 |
|--------|------|
| 圆角 | `--radius-*` 与 `18px`/`24px` 硬编码并存（矩阵行、聊天气泡、桥接面板、`.la-shell` 等） |
| 间距 | `gap: 10px` / `8px 14px` 等与 token 混用；部分 `var(--space-3, 10px)` 与主值不一致 |
| 双轨控件 | `control-primitives` 与 `panel-primitives` 并存；`.la-shell` 内对 `btn` 有额外着色统一 |
| 节间距 | `ws-form-stack`、`ws-subsection`、`dialog-body` 等多来源 gap |
| 阴影 | `tokens` 的 `--shadow-*` 与页面内自写长阴影并存（如顶栏） |
| 行内样式 | 受治理的 inline 债与白名单；属长期压降项 |

## 3. 分阶段整改（建议顺序）

### Phase A — 形状与 token 映射（低风险、高收益）

- 在 `tokens.css` 中**可选**增加语义别名（如 `--radius-card: var(--radius-2xl)`），或统一把高频硬编码改为现有 `--radius-*`。
- 覆盖优先顺序：`language-metadata-workspace.css`、`ai-chat-thread.css`、`orthography-bridge-workspace.css` 中与审计一致的硬编码圆角；`.la-shell` 的 `24px` 与 token 对齐。
- **验证**：`npm run build:guard`；语言资产/元数据 E2E 子集；人工 spot-check 语言资产与 OM/OB 壳。

### Phase B — 间距魔数收束

- 统计顶栏、壳层、大表单中最高频的 `10px/14px/6px`，能映射则映射到 `--space-*` 或新增**具名** token（如已在用的 `--space-row-padding-y` 模式）；避免再引入无文档的魔数。
- **验证**：`npm run check:css` 与 `check:css-debt-thresholds`（若当时未恶化）。

### Phase C — 控制件双轨裁定与迁移

- 依据**拍板项 C** 选择策略后，在 `docs/adr/` 或本方案附录中记录「壳内唯一入口」。
- 小步：新代码只走选定入口；旧代码按模块迁（不动无关模块）。

### Phase D — 阴影与 elevation

- 将工具栏/下拉里重复的长 `box-shadow` 收束为 `--shadow-sm` / `--shadow-popover` 等；保留极少数主题级例外并注释原因。
- **验证**：转写顶栏、下拉在 Chromium/Firefox 的对比截图（可用手动或现有 E2E 附属）。

### Phase E — 类名与可发现性

- 语言元数据页内，将二级小节标题区从 `lm-subgroup-header` / `lm-subgroup-title` / `lm-subgroup-description` 收束为 **`panel-section__copy` + `panel-title-secondary` + `panel-subsection__description`**（`panel-subsection__description` 定义在 `panel-primitives.css`，较 `panel-section__description` 更小一档，对应原 12px 正文说明）。
- **不改动** `lm-subsection` / `lm-subsection-header`（`<details>`/折叠行仍属 LM 域命名）；与 Phase A 分 PR 的约束可解除，本轮仅触及 LM 详情/地理三处 TSX。

## 4. 需要你拍板的内容（**拍板后补入本节前「决议」子节再执行**）

### 拍板 A：圆角「卡片」档位

- **选项 A1（推荐）**：不新增 token，将 `18px` 统一为现有 `--radius-2xl`（14px 设计 token，与当前部分硬编码 18px 有视觉差时需接受 2px 级调整或 B2）。
- **选项 A2**：在 `tokens.css` 增加 `--radius-card: 18px`（与现有阶梯并存，语义化「大卡片/矩阵行」）。
- **选项 A3**：保持现状，仅文档化「三类圆角语汇」、不做机械替换（技术债可接受时）。

**建议默认**：A2（若产品强依赖 18px 视觉）或 A1（若可接受用 design token 阶梯统一）。

---

### 拍板 B：语言资产 / 大表单「节间距」真源

- **选项 B1（推荐）**：以 `--la-section-gap`（及 `--space-stack-page` / `--space-stack-section`）为唯一真源；`ws-subsection` 的 `gap: 12px` 改为引用同一 token（可能微调 12→与 token 一致）。
- **选项 B2**：保留 12px 与 16px 两档，但在合约层为「折叠小节」与「大节」起不同变量名，避免同一层级混用。

**建议默认**：B1，减少读者心智负担。

---

### 拍板 C：`.input` / `.btn` 与 `.panel-input` / `.panel-button` 双轨

- **选项 C1（推荐）**：**壳与对话框内**（`.la-shell` 及同合约面板）以 `control-primitives`（`.input` / `.btn`）+ 合约为准；`panel-*` 仅用于无壳独立面板，并逐步缩小使用面。
- **选项 C2**：反向以 `panel-input` / `panel-button` 为壳内主入口，`control-primitives` 仅用于非 panel 页。
- **选项 C3**：不合并心智，只写清文档「新代码优先用哪一套」与现有代码不强制迁移（最低成本、长期双轨）。

**建议默认**：C1 与当前 `language-asset-section-contract` 的 `.la-shell` 选择器方向一致。

---

### 拍板 D：顶栏/波形等高密度区是否纳入本方案

- **选项 D1**：纳入 Phase D，与全局 shadow token 一起收束（工作量大、需回归转写全路径）。
- **选项 D2（推荐）**：**排除**高密度区，仅对「管理型表单、对话壳、语言资产三页」收束；顶栏/波形单独小工单与性能测试挂钩。

**建议默认**：D2，降风险、易阶段性交付。

---

### 拍板 E：是否新增/修改 ADR

- **选项 E1（推荐）**：补一篇短 ADR（「布局与 token 壳层策略」），本执行计划作落地清单。
- **选项 E2**：仅保留本 `docs/execution/plans` 文档，不新增 ADR。

**建议默认**：E1，利于长期与 `check:architecture-guard` 类治理一致。

1. 各 Phase 开始前列出：受影响文件 glob、回滚方式（revert 单 commit）。
