## 2. M0-M4 里程碑门禁矩阵

统一门禁采用“两轮制”：
1. 首轮：`typecheck + 聚焦测试 + build`
2. 第二轮（放行前）：并入全量 `npm test` 阻断

| 里程碑 | 首轮必跑命令 | 聚焦测试口径 | 第二轮放行前 | 手工模板与证据 |
|--------|--------------|--------------|--------------|----------------|
| M0（规格冻结） | `npm run -s typecheck`、`npm run -s check:docs-governance`、`npm run -s build` | `npx vitest run <changed-tests>`（深链/事件合同相关用例） | `npm test` | `docs/execution/templates/三页联动-M0-M4-手工验收模板-2026-04-25.md` |
| M1（标注 MVP） | `npm run -s typecheck`、`npm run -s build` | `npx vitest run <annotation + deep-link changed-tests>` | `npm test` | 同上，回填 M1 小节 |
| M2（形态标注） | `npm run -s typecheck`、`npm run -s build` | `npx vitest run <token/morpheme/lexeme-link changed-tests>` | `npm test` | 同上，回填 M2 小节 |
| M3（词典编辑导出） | `npm run -s typecheck`、`npm run -s build` | `npx vitest run <lexicon/export changed-tests>` | `npm test` | 同上，回填 M3 小节 |
| M4（协作收口） | `npm run -s typecheck`、`npm run -s build` | `npx vitest run <collaboration/conflict changed-tests>` | `npm test` | 同上，回填 M4 小节 |

补充：
1. `npm test` 被仓库既有守卫阻断时，必须先记录阻断项，再补跑 `npx vitest run <changed-tests>` 作为功能回归证明。
2. 每个里程碑都要同步附上 R1-R8 联评勾选结果（见第 6 节）。

## 3. 跨页事件合同（v1）

### 3.1 当前已落地的 app-shell 事件

| 事件名 | payload | 现状锚点 |
|--------|---------|----------|
| `jieyu:open-search` | `AppShellOpenSearchDetail`（`query`、`scope`、`layerKinds`） | `src/utils/appShellEvents.ts`、`src/pages/useTranscriptionShellController.ts` |
| `jieyu:open-collaboration-cloud-panel` | 无 | `src/App.tsx`、`src/components/SidePaneSidebar.tsx` |

### 3.2 三页数据同步事件（M0 定稿，M1+ 实装）

| 事件名 | 触发时机 | 最小 payload |
|--------|----------|--------------|
| `jieyu:workspace.unit-updated.v1` | 标注/转写写入 unit 相关内容后 | `eventId`、`occurredAt`、`unitId`、`layerId`、`revision`、`idempotencyKey` |
| `jieyu:workspace.lexeme-updated.v1` | 词条新增/更新后 | `eventId`、`occurredAt`、`lexemeId`、`revision`、`idempotencyKey`、`textId?` |
| `jieyu:workspace.lexeme-deleted.v1` | 词条软删/删除后 | `eventId`、`occurredAt`、`lexemeId`、`deletionMode`、`idempotencyKey`、`textId?` |
| `jieyu:workspace.context-sync.v1` | 深链往返需要同步上下文时 | `eventId`、`occurredAt`、`sourcePage`、`targetPage`、`unitId?`、`layerId?`、`lexemeId?`、`contextKeys`、`idempotencyKey` |

补充字段约定：
1. lexeme 事件中的 `textId` 为可选字段（缺省合法）。
2. 存在 `textId` 时优先用作增量刷新范围约束；缺省时按 `lexemeId` 命中集反查。

### 3.3 幂等键、冲突与回放顺序

幂等键约定：
1. unit 事件：`unit:${unitId}:rev:${revision}`
2. lexeme 事件：`lexeme:${lexemeId}:rev:${revision}`
3. context 事件：`context:${sourcePage}->${targetPage}:${eventId}`

冲突策略：
1. 同 `idempotencyKey` 重复到达：丢弃。
2. 同目标不同版本：以 `revision` 新者为准；版本无法比较时按 `occurredAt`。
3. 若目标页存在未提交草稿：仅打脏标记，不直接覆盖草稿。

回放顺序：
1. `context-sync`
2. `unit-updated` / `lexeme-updated` / `lexeme-deleted`
3. 读模型增量刷新

刷新退化策略（已拍板）：
1. 正常路径：按 `unitId` 或 `lexemeId` 命中集增量刷新。
2. 命中集反查失败时，允许退化为 `textId` 级刷新。
3. 退化刷新属于诊断兜底，不得作为默认路径常驻。

`context-sync` 约束（已拍板）：
1. `context-sync` 仅同步导航态与视图态。
2. `context-sync` 严禁触发任何写操作（含隐式写入与二次提交）。

### 3.4 分词与覆盖治理口径（M1+）

1. 分词执行链固定为：首次自动分词 → 手动分词修正 → 二次自动分词。
2. 二次自动分词默认“保守模式”：不得覆盖人工确认内容。
3. 覆盖策略分三级：
   - 保守模式（默认）：仅补齐未确认区域
   - 审核模式：差异候选逐条采纳
   - 强制模式：允许覆盖，但必须先生成快照并支持回滚
4. 分词相关批处理必须记录审计字段（触发来源、批次、影响范围、回滚点）。

### 3.5 Validator 模板治理口径（语言资产）

1. LeipzigValidator 执行模型固定为“内置规则 + 语言资产模板扩展”。
2. 模板作用域优先级固定：系统默认 → 语言模板 → 项目模板 → 用户覆盖。
3. 允许非英文规则与本地术语模板，不强制英文缩写体系。
4. 模板切换与启停不得改变历史已发布结果；历史结果按当时模板版本回放。

## 4. Lexeme 数据迁移与索引计划（v1）

### 4.1 当前基线（代码事实）

1. `LexemeDocType` 当前字段以 `lemma/senses/forms/notes/examples/...` 为主，尚未包含 `ipa/audioUrl/etymology/semanticDomains/dialect`（`src/db/types.ts`）。
2. Dexie 当前目标版本为 `43`（`src/db/engine.ts`），`lexemes` 现索引基线仍为 `id, updatedAt`（v1 起沿用）。
3. `token_lexeme_links` 已有 `[targetType+targetId]`、`lexemeId`、`[lexemeId+targetType]` 索引（v17+）。

### 4.2 目标变更（建议在 v44 落地）

1. 扩展 `LexemeDocType` 可选字段：`ipa`、`audioUrl`、`etymology`、`semanticDomains`、`dialect`。
2. 索引增强建议：
   - `language`
   - `lexemeType`
   - `dialect`
   - `*semanticDomains`（multiEntry）
   - `[language+updatedAt]`
   - `[lexemeType+updatedAt]`

### 4.3 迁移步骤

1. 类型层先加可选字段，不改旧字段语义。
2. Dexie 升版（建议 v44）并补 `.stores(...)` 索引声明。
3. `upgrade(tx)` 做轻量回填：仅规范化新增字段类型，不做破坏性重写。
4. 回放验证：
   - `src/db/migrations/jieyuDexieOpenReplay.test.ts`
   - 变更范围内 `npx vitest run <db related changed-tests>`

### 4.4 回滚策略

1. 功能回滚：UI/服务先停用新增字段读写，只保留核心词条字段。
2. 查询回滚：若索引异常，临时降级为已存在索引查询 + `toArray().filter()`（并记录技术债）。
3. 数据安全：新增字段为可选，不删除旧字段，不执行 destructive migration。
4. 触发条件：出现 `SchemaError`、升级失败率超阈值、或词典主路径查询明显退化。

## 5. 交付口径（alpha vs go）

| 档位 | 目标 | 最低门槛 | 证据要求 |
|------|------|----------|----------|
| `alpha` | 面向内测/灰度验证可用性 | 首轮门禁通过（`typecheck + 聚焦测试 + build`）+ 手工模板关键路径回填 | PR 需附聚焦测试结果与手工记录链接 |
| `go` | 面向产品开放评审/放行 | `alpha` 全部满足 + 第二轮 `npm test` 通过 + 开放门槛文档 go 条件通过 | 发布证据按 `core` profile 封账，含门禁结果、风险与回退结论 |

补充：
1. `alpha` 不是“对外开放”；外部口径必须以 `go` 判定为准。
2. `go` 前必须完成 R1-R8 联评勾选（含 N/A 说明）。

### 6.2 三页与语料库联动边界（已拍板）

必须（Must）：
1. 深链互跳（`unitId/layerId/lexemeId`）
2. 上下文回传（排序/选中/滚动/筛选状态）
3. 只读 concordance 联动
4. 最小粒度事件增量刷新

禁止（Must Not）：
1. 语料库直接执行 lexeme CRUD
2. 语料库复制标注写链成为第二真源
3. 语料库绕过事件合同触发隐式写入
