# 标注页与词典页开发路线图

## 1. 目标与现状

### 1.1 目标

1. 对标 FLEx 的标注与词典核心能力，落地最小闭环
2. 打通转写→标注→词典数据流，避免并行真源
3. "每语段一行 + 多层可多实例 + 词汇可选择性入词典"

## 2. 数据流合同

### 2.1 SSOT（单一真源）

- 标注数据真源 = `layer_units` + `unit_tokens` + `unit_morphemes`
- 词典数据真源 = `lexemes` + `token_lexeme_links`
- 标注页 **不得** 自建第二套表
- 所有写操作走 `LinguisticService` 统一入口

### 2.2 跨页刷新事件

| 事件 | 粒度 | 触发方 |
|------|------|--------|
| token gloss 变更 | `unitId` | 标注页 write → 转写页 refresh |
| lexeme 入库 | `lexemeId` | 词典页 write → 标注页 refresh |
| lexeme 删除 | `lexemeId` | 词典页 write → 标注页/转写页 降级展示 |

事件名、payload、幂等键、冲突与回放顺序统一见《标注词典语料-治理补充规范-2026-04-25.md#cross-page-event-contract-v1》。

### 2.3 防御性引用

- 所有跨页引用持 `targetId` + 快照文本
- 删除使用软标记，不做静默级联
- 断裂态展示诊断信息，不可白屏

### 2.4 入词典链路

```
标注 token → AutoGlossService 匹配 → 用户确认 → saveLexeme → token_lexeme_link
```

### 3.1 直接对标

### 3.2 不嵌入但参考

| 工具 | 原因 |
|------|------|
| **INCEpTION** | 语言学标注功能最全（UIMA/Relations/ML 推荐）——但 JVM 架构，仅做 UX 参考 |
| **leipzig.js** | 行间标注 HTML 渲染——建议 fork 为 React 组件用于 IGT 展示 |
| **Label Studio** | 通用标注平台——太重，不适合 interlinear gloss 场景 |
| **DictPress** | Go 词典服务器——考虑做未来独立后端参考 |
| **brat / WebAnno** | 支持 standoff、relation、attribute、normalization；不作为依赖，但借鉴非连续 span 的 relation/link-feature 建模 |

### 3.2.1 非线性对齐与 edge case 借鉴结论

1. **FLEx** 的强项是词典驱动的形态分析：手动断词、同词形历史建议、parser 建议、lexeme/allomorph/sense 联动。Jieyu 借鉴“候选分析 → 人工采纳 → 写库”，不做静默覆盖。
2. **ELAN** 的强项是 tier hierarchy：时间对齐主层 + dependent tiers；`Symbolic Association` 适合译文/gloss 一对一，`Symbolic Subdivision` 适合词/词素派生层，`Included In` 适合允许空隙的语音片段。Jieyu 的标注页应保留宿主 unit 与依赖层的边界。
3. **UD / CoNLL-U** 区分 surface token 与 syntactic word，并提供 multiword token 与 empty node。Jieyu 应支持表面 token、分析 word、零形式节点分离，导出 CoNLL-U 时再投影。
4. **LAF / GrAF** 使用 stand-off graph：源文本不被改写，node/edge/region 描述标注与非连续区间。Jieyu 的非连续词素、多词表达、词根模板应走 typed relation，不塞入 gloss 字符串。
5. **brat / WebAnno / INCEpTION** 的经验是：复杂不连续 span 常用 span + relation / chain / link feature 表达。Jieyu M2 可先做有限 relation set，而不是一次性提供任意复杂图编辑器。
6. **TypeCraft / Ligt** 表明 IGT 结果应可检索、可链接、可互操作。Jieyu 内部模型应比导出格式更强，导出时生成 projection/diagnostic，说明哪些关系被完整表达、哪些被降级为 note/custom field。

### 3.3 导出格式优先级

|------|--------|------|

### M0 规格冻结拍板结果（2026-04-25）

#### A. LayerType ↔ 标注页 UI 字段对照（最终）

| 维度 | `transcription` | `translation` |
|------|------------------|---------------|
| 层定义 | 主句段层，可作为时间轴主承载 | 宿主仅 `layer_links`，持久化不得含 `parentLayerId` |
| 主编辑区字段（M1） | 原文文本、gloss、token POS、morpheme、note/tag/selfCertainty、词典链接操作 | 译文文本、note/tag/selfCertainty |
| 只读信息区字段 | `name/languageId/orthographyId/constraint/displaySettings` | 同左 |
| 词典联动 | 允许 token → lexeme 链接与解除链接 | 只读回显，不直接入词典 |
| 刷新最小粒度 | `unitId`（必要时 `unitId+layerId`） | `unitId`（必要时 `unitId+layerId`） |

补充：`gloss` 在实现语义上属于内容角色，不新增为第三个 `LayerType`。

#### B. 键盘状态机（最终）

| 当前状态 | 按键 | 条件 | 结果状态 | 动作 |
|---|---|---|---|---|
| 行聚焦（非输入） | `Space` | 光标不在输入框 | 播放/暂停切换 | 触发行内段播放或暂停 |
| 输入中 | `Space` | 光标在输入框 | 保持输入中 | 永远作为空格输入，不劫持播放 |
| 输入中 | `Tab` | 有可接纳 AI 建议 | 保持输入中 | 先接纳建议，写入当前草稿 |
| 输入中 | `Tab` | 无可接纳 AI 建议 | 下一行聚焦 | 轻量保存后换到下一行 |
| 输入中 | `Enter` | 无 `Ctrl` | 当前行输入中 | 保存当前行并停留当前行 |
| 输入中 | `Ctrl+Enter` | `Ctrl` 生效 | 下一行聚焦 | 保存当前行并跳到下一行 |

失败约束：
1. 保存失败时保持当前行输入态，不允许跳下一行。
2. `Ctrl+Enter` 的跳行仅在保存成功后发生。

#### C. 波形显示方案（最终）

1. 标注页不使用顶部共享波形。
2. 仅使用行内微波形（micro-waveform），每个转写行只显示与该行相关的语段波形。
3. 行内微波形默认显示在对应转写行上方。
4. 行内微波形支持折叠/展开，折叠态不影响该行标注输入与播放快捷键。
5. 波形交互优先级服从键盘状态机：输入框聚焦时 `Space` 永远作为空格输入，不劫持播放。

#### D. 分词机制（最终）

1. 分词采用“自动 + 手动 + 二次自动”的三阶段机制：
	- 首次自动分词：按语言配置执行规则分词与可选算法分词。
	- 手动分词：用户可在行内执行切分、合并、边界调整、批量修正。
	- 二次自动分词：允许基于已标注语料再次自动分词。
2. 手动分词必须是一等入口，至少包含：
	- 光标处分割（split at caret）
	- 相邻合并（merge prev/next）
	- 边界拖拽微调
	- 同词形批量修正（批量操作）
3. 二次自动分词默认采用“保守模式”：
	- 只补齐未确认区域
	- 不覆盖人工确认结果（`manualConfirmed=true`）
4. 覆盖策略分三级：
	- 保守模式（默认）：禁止覆盖人工确认结果
	- 审核模式：生成差异候选，逐条人工采纳
	- 强制模式：允许覆盖，但必须先生成快照并支持回滚
5. 自动分词与覆盖行为必须可审计：记录触发来源、批次、影响范围与回滚点。

#### E. LeipzigValidator 配置化（最终）

1. LeipzigValidator 不再仅依赖内置规则，升级为“内置规则 + 模板扩展”双层模型。
2. 新增“语言资产”面板子模块：`Validator 模板`，支持：
	- 新建/编辑/启停模板
	- 模板导入导出
	- 规则即时校验（sandbox）
3. 模板作用域与优先级：
	- 系统默认模板
	- 语言模板
	- 项目模板
	- 用户会话覆盖
4. 非英文与本地术语规则必须可配置，不强制英文缩写体系。
5. 规则应用顺序固定：基础 Leipzig → 语言模板 → 项目模板 → 用户覆盖。

#### F. 三页与语料库联动边界（最终）

结论：三页（标注/词典/转写）与语料库页“必须联动”，但仅允许弱耦合联动。

必须联动（Must）：
1. 深链互跳（`unitId/layerId/lexemeId`）
2. 上下文回传（排序/选中/滚动/筛选状态）
3. 只读 concordance 联动（语料命中跳转）
4. 按最小粒度事件增量刷新

禁止联动（Must Not）：
1. 语料库直接执行 lexeme CRUD
2. 语料库复制标注写链成为第二真源
3. 语料库绕过事件合同触发隐式写入

#### G. 非线性对齐与 typed relation（M2+ 目标）

结论：标注页 UI 可以先采用线性 IGT 矩阵，但底层不得只保存线性字符串。非连续、跨词、多词表达、词根模板、重叠、依存、共指等 edge case 应通过 typed relation 表达。长期架构边界以 [ADR 0022：标注 analysisGraph 与 typed relation](../../adr/0022-annotation-analysis-graph-typed-relations.md) 为准；fixture 验收以 [标注 analysisGraph fixture 基线](../audits/标注-analysisGraph-fixture基线-2026-04-25.md) 为准。本节仅保留执行摘要。

**节点类型（候选）：**

| 类型 | 说明 |
|------|------|
| `token` | 表面文本切片，来自分词结果 |
| `word` | 语言学分析词；可与 token 不一致 |
| `morpheme` | 词素分析节点，可有多个 surface parts |
| `zero` | 零形式、空节点、省略谓词等无表面字符串节点 |
| `mwe` | 多词表达、固定搭配、light verb construction |
| `root` / `pattern` | 词根模板、非连接形态分析 |
| `exponent` | 表达一个或多个语法 feature 的形式或过程；覆盖 cumulative / portmanteau / multiple exponence |
| `featureBundle` | case/number/gender/person/tense/aspect 等结构化 feature 集合 |
| `process` | addition / deletion / substitution / reduplication / templaticMapping / toneOverlay / stressShift / lengthening / suppletion 等形态过程 |
| `underlyingForm` / `surfaceForm` | 底层形式与表面形式分离，承载 allomorphy、morphophonology 与 suppletion |
| `prosodicFeature` | tone、stress、length 等非音段形态标记 |
| `gloss` / `pos` / `note` | 结构化标注值；不再仅作为自由文本 |

**关系类型（M2 最小集 + 后续扩展）：**

| 关系 | 用途 | 阶段 |
|------|------|------|
| `glosses` | gloss 节点指向 token / morpheme / zero | M1b-M2 |
| `hasPos` | POS 节点指向 token / morpheme / word | M1b-M2 |
| `linksLexeme` | 分析节点指向 lexeme / sense / allomorph | M2 |
| `partOfMwe` | 多个 token/word 组成一个 MWE | M2 |
| `discontinuousPartOf` | 一个 morpheme/root/pattern 对应多个非连续片段 | M2 |
| `alternativeAnalysis` | 同一 token/word 的候选分析集，支持 pending/accepted/rejected | M2 |
| `reduplicates` | 重叠成分指向被复制 stem/root | M2+ |
| `realizesFeature` | exponent/process 指向被表达的 featureBundle | M2+ |
| `hasUnderlyingForm` / `hasSurfaceForm` | 分析节点连接底层与表面形式 | M2+ |
| `derivedByProcess` | surfaceForm 或 exponent 由某个 process 生成 | M2+ |
| `hasAllomorph` | lexeme/morpheme 连接 allomorph 集合 | M2+ |
| `suppletes` | 表面形式与词元之间为 suppletion，而非规则派生 | M2+ |
| `deletesSegment` / `substitutesSegment` / `overwritesTone` | truncation、mutation、tonal morphology 等非连接过程 | M2+ |
| `dependsOn` / `corefersTo` | 依存、共指等更高层关系 | M3+ |

**数据约束：**
1. 节点可持有 `surfaceParts: [{ tokenId, startOffset?, endOffset? }]`；非连续单位用多个 part 表示。
2. 自动/二次自动分词只产生 `alternativeAnalysis` 候选，不直接替换已 `manualConfirmed` 的节点或关系。
3. `token_lexeme_links` 保持为高频索引表；复杂链接与跨词链接进入 typed relation 层，避免强行扩散专用表语义。
4. 导出到 FLEx/LIFT/CoNLL-U/ELAN/LaTeX 时通过 projection 层降级；降级项必须进入导出诊断报告。
5. Leipzig 是展示/出版规则，不是完整形态理论模型；内部模型需用 `exponent`、`process`、`underlyingForm/surfaceForm` 覆盖 Oxford/Cambridge morphology handbooks 常见问题域。
6. M2 实现遵循 fixture-first：先冻结 [analysisGraph fixture 基线](../audits/标注-analysisGraph-fixture基线-2026-04-25.md)、projection 与 diagnostic，再开放复杂 relation 编辑。

**形态学综述覆盖评估（Oxford/Cambridge/WALS/GGG 对齐）：**

| 问题域 | 典型难点 | Jieyu 表达策略 |
|--------|----------|----------------|
| allomorphy / stem alternation | 同一词素多个变体，表面形式与底层形式不一致 | `underlyingForm` / `surfaceForm` + `hasAllomorph` + `derivedByProcess` |
| suppletion | `go/went` 类非规则替换，无法线性切分 | `suppletes` relation，避免伪造 morpheme boundary |
| zero exponence | feature 存在但无表面形式 | `zero` node + `realizesFeature` |
| cumulative / portmanteau exponence | 一个形式同时表达 case+number+gender 等 | `exponent` node + 多个 `realizesFeature` |
| multiple exponence | 同一 feature 在多个位置重复表达 | 多个 `exponent` 指向同一 `featureBundle`，projection diagnostic 标记重复表达 |
| non-concatenative morphology | infixation、reduplication、mutation、truncation、tonal overwriting、templatic morphology | `process` node + `derivedByProcess` / `reduplicates` / `deletesSegment` / `substitutesSegment` / `overwritesTone` |
| clitic / affix / word 边界 | 边界状态不稳定 | `word` 与 `morpheme` 分离，role `clitic` + `cliticizesTo` |
| compounding / incorporation / polysynthesis | 单个 orthographic word 内部结构深、论元/谓词并入 | `mwe` / `word` / `morpheme` 多层节点 + `contains` / `partOfMwe` / `linksLexeme` |
| morphophonology | 形态边界与音系变化交织 | `underlyingForm` → `process` → `surfaceForm` |
| syncretism / ambiguity | 同一形式对应多个范畴或分析 | `alternativeAnalysis` + accepted/pending/rejected 生命周期 |
| non-segmental morphology | tone、stress、length 表达语法意义 | `prosodicFeature` + `overwritesTone` / `derivedByProcess` |

**Leipzig structural parser + relation projection（M2 子任务）：**

当前 `LeipzigValidator` 只能做文本级缩写与简单分隔符检查；M2 需新增结构化解析层，把 Leipzig 符号解析为可映射到节点/关系的结构，而不是继续扩展单一正则校验。

| Leipzig 结构 | 解析目标 | relation / node 投影 |
|---------------|----------|----------------------|
| `-` morpheme boundary | source 与 gloss 的 morpheme 段数对齐 | `morpheme` nodes + `glosses` |
| `.` feature composition | 一个 morpheme 的多个语法 feature | `gloss.features[]`，不拆成多个 morpheme |
| `=` clitic boundary | clitic 与 host 分离 | `word` / `morpheme` role `clitic` + `cliticizesTo`（或 `contains(role=clitic)`） |
| `<...>` infix | token 内部插入成分 | `morpheme.surfaceParts` offset + role `infix` |
| `\` morphophonological alternation | 词形/音系交替标记 | `realizes` relation + `features.alternation`，不得默认报错 |
| `[]` non-overt / supplied element | 非显性或补足成分 | `zero` node 或 `features.supplied=true` |
| zero morpheme | 无表面字符串的语法意义 | `zero` node + `glosses` |
| reduplication | 复制成分与 base/stem 的关系 | `reduplicates` relation |
| discontinuous/bipartite morpheme | 一个形态单位多个非连续片段 | 多个 `surfaceParts` + `discontinuousPartOf` |

解析输出至少包含：`segments`、`boundaries`、`features`、`warnings`、`projectionDiagnostics`。UI 使用 warnings 做即时提示；写库只保存人工采纳后的 relation projection。

4. IGT 布局：Row-based + `@tanstack/react-virtual`（不引入 pretext）
5. AI 深度融入 M0-M4，但仅作为建议层，写入必须人工确认
6. 不引入 Yjs/CRDT；协同主干维持 Dexie + Supabase
7. LIFT 导出进入 M3 同批
8. M1 产品口径保留更多字段进入首个标注 MVP，但工程交付拆为 M1a/M1b，M1a 先锁定 gloss/POS/保存/键盘/播放最小闭环
9. 输入框内 `Space` 永远保留“输入空格”语义
10. `Tab` 优先级固定为“先接纳建议，再换行”
11. `Enter` 与 `Ctrl+Enter` 职责分离（保存留位 vs 保存跳行）
12. 波形显示仅采用行内微波形：每行上方默认显示对应语段，且可折叠；不采用顶部共享波形
13. 分词采用三阶段（自动 + 手动 + 二次自动），默认不覆盖人工确认结果
14. LeipzigValidator 升级为语言资产可配置模板体系，支持非英文规则
15. 三页与语料库必须联动，但仅允许弱耦合联动，禁止语料库进入写链真源
16. 非线性形态、MWE、零词素、词根模板、重叠、依存/共指等不塞进字段字符串；M2 起以 typed relation + projection/diagnostic 表达
17. Leipzig 仅作为输入/展示/导出 projection；内部形态模型需额外覆盖 `exponent`、`process`、`underlyingForm/surfaceForm`，以表达 suppletion、mutation、truncation、tone、cumulative/multiple exponence
