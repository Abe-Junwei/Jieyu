# 对照模式与多壳时间轴段块统一收口规划

> **2026-04-19 修订**：对照 `TranscriptionPage.TimelineContent`、`timelineShellMode`、`TranscriptionTimelineTextTranslationItem` / `TranscriptionTimelineMediaTranslationRow` 引用关系修正 §2、§3、§9；增补 **§3.4** 多行对照 vs 单行轨之产品约束，并反映于 §8；补记 **comparisonFocus 页面层持有事实**、**source / target 非对称保存路径** 与 **更具体的 Phase B/C 验收门槛**；**§10** 为代码落地核对与风险状态清单，并反映 **`TimelineDraftEditorSurface` 三路径接入** 与已闭合项。

- **本文目标**：在「左右对照」已具备 P0/P1 能力的前提下，把**段块级 UI 与交互**与「非对照下的两条壳」对齐，降低长期分叉成本；不写具体逐行实现补丁。

## 1. 问题陈述

当前对照模式**不是**完全另起一套，也**不是**完全复用波形轨 UI，而是：

- **底层**：读写、自动保存、草稿、分组读模型、部分录音控件等已部分共用；
- **表层**：左右语段渲染、高亮/焦点语义、部分 CSS、编辑反馈仍与时间轴壳存在分叉。

若不收口，后续每次改「译文条/段块」都要改多处，且易出现**看起来一致、行为不一致**（尤其是右键、区域 overlay、键盘路径）。

## 2. 架构前提：三条壳 + 一条入口（必须写进同一规划）

转写时间轴内容区并非「对照 vs 波形」二元，而是：


| 壳        | 典型组件                                | 说明                                                       |
| -------- | ----------------------------------- | -------------------------------------------------------- |
| **对照**   | `TranscriptionTimelineComparison`   | 双列骨架 + 组级读模型 + 页面层持有的 `comparisonFocus`（组件内仅保留 fallback） |
| **纯文本**  | `TranscriptionTimelineTextOnly`     | 文献时间映射、虚拟列表、轨上拖建/改时等                                     |
| **媒体波形** | `TranscriptionTimelineMediaLanes` 等 | 波形锚定、媒体轨布局                                               |


**路由事实**（实现入口）：`TranscriptionPage.TimelineContent` 中分支顺序为：

1. `**comparisonViewEnabled && effectiveLayersCount > 0`** → 仅渲染 `TranscriptionTimelineComparison`，且 `**{...textOnlyProps}` 原样展开**（与纯文本壳同源 props）。
2. 否则 `shell === 'waveform'` → `TranscriptionTimelineMediaLanes`；
3. 否则 `shell === 'text-only'` → `TranscriptionTimelineTextOnly`。

其中 `**effectiveLayersCount`** 取 `layersCount` 与 `textOnlyProps` 内转写/翻译层列表长度的 **max**，避免轨条计数与层列表短暂不一致时挡住对照（见 `TimelineContent` 内注释）。

**与壳枚举的交叉事实**：`resolveTimelineShellMode` 在 **对照开启且有层** 时固定返回 `**shell: 'text-only'`**（`src/utils/timelineShellMode.ts`），即其它依赖 `shell` 的 UI 不应假设「有声学 URL 就一定 waveform」。但底部主内容区在对照开启时已在第 1 步 return，**不会挂载 `TranscriptionTimelineMediaLanes`**。

因此：

- 任何在 `textOnlyProps` 上增减的能力，须同时考虑 **「关对照 → 纯文本」** 与 **「开对照」** 两条分支；
- `**comparisonFocus` 当前主存于页面级 UIState**，并经 ReadyWorkspace / orchestrator 向下透传；`TranscriptionTimelineComparison` 中的 internal state 主要用于 fallback / 隔离场景，不应在后续重构中被误判为第二套长期真源；
- **「译文条」实现本身在非对照世界已是两套**：见 §3.0；收口目标应是 **三条路径**（或抽第四套共用核），而非默认「媒体轨 = 唯一真源」。

## 3. 当前复用结论（摘要）

### 3.0 非对照下的两条「译文条」实现（规划须显式包含）

与「对照 vs 时间轴」正交，仓库里 **未开对照** 时译文/段块 UI 已分叉为两条：


| 路径        | 入口                                                                                             | 样式与 DOM                                                    |
| --------- | ---------------------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| **纯文本壳**  | `TranscriptionTimelineTextOnly` → `TranscriptionTimelineTextTranslationItem`                   | `timeline-text-item.css`，单行 `input`，含 `saveStatus` 脏/存/错点等 |
| **媒体波形壳** | `TranscriptionTimelineMediaTranslationRow` → `renderAnnotationItem` → `TimelineAnnotationItem` | `timeline-annotation.css`，波形轨上的标注条模型                       |


媒体译文行路径 **不使用** `TranscriptionTimelineTextTranslationItem`；与纯文本译文条是两条并行实现。

### 3.1 已较好共用

- 文本读写与保存链路、自动保存与草稿（对照侧接 `TranscriptionEditorContext` 等）；
- **source 列已可直接编辑**，并已进入共享 draft / autosave 主链路（现状不是只读按钮壳）；
- **target 列已复用 layer / segment 级保存能力**，但其持久化语义与 source 侧并不完全对称；
- 录音控件（如 `TimelineTranslationAudioControls`）；
- 分组与层来源解析（`buildComparisonGroups`、segment 源层解析等）。

### 3.2 仍明显分叉（收口对象）

- **表面渲染**：对照在组件内直接渲染 source/target（**多行** `textarea` + `normalizeComparisonText`），与纯文本 **单行** `input`、与媒体 `**TimelineAnnotationItem`** 均为不同路径；视觉与状态点对齐策略见 **§3.4**；
- **样式**：至少三套段块皮肤并存——`timeline-text-item.css`、`timeline-annotation.css`、`timeline-comparison.css`；若其它材料写「以 `timeline-text-item` 为主源」，须注意 **媒体轨主用 `timeline-annotation`**，避免只迁一侧；
- **状态语义**：`comparisonFocus`、`activeUnitId`、时间轴自身的 active/editing 语义需有**明确派生规则**，避免隐式漂移；
- **保存职责**：source 侧当前偏向 unit 级文本写入，target 侧则走 layer / segment adapter 与组级保存语义；若后续抽共用输入内核，必须保证「统一表面 ≠ 混同持久化职责」；
- **非编辑类交互**：右键菜单、区域 overlay、侧栏/审查队列与 `activeUnitId` 的同步，样式统一后仍可能分叉，须单独列入验收。（**现状**：对照已对 source/target 接 `handleAnnotationContextMenu`，与「完全未接菜单」不同；Phase C 侧重 **与时间轴/纯文本同一套菜单构建与 unit 解析**，而非从零接线。）

### 3.3 不建议强行共用（保留对照专属）

- 双列布局、对照组信息架构、bundle/speaker 汇总、**组级 note 能力（入口是否外露按产品策略显隐）**、紧凑列模式等；
- 纯文本壳专属的 **文献时间映射、虚拟化、轨上拖建** 等（与对照 UI 无必要 1:1 合并）。

### 3.4 输入形态：多行对照 vs 单行轨（产品约束）

**当前实现差异（事实）**

- **对照**：经 `TimelineDraftEditorSurface`（`multiline` 时为 `textarea`）+ `normalizeComparisonText`（多行、段落规则与组级编辑一致）；状态点样式与 `TimelineDraftEditorSurface` 共用 `timeline-text-item.css` 中的 `.timeline-text-item-status-dot*`（已部分落实 Phase B 的「表面统一」）；
- **纯文本 / 媒体译文行**：单行语义（如 `normalizeSingleLine`、`TimelineAnnotationItem` 的单行输入模型），与波形条宽度强绑定。

**本规划采纳的约束**

1. **对照视图默认保留多行编辑**，不将「与轨条共用同一 input 类型」列为 Phase A–B 的硬验收项；统一收口以 **颜色/间距/状态点/焦点环/脏存错反馈** 等可参数化层为主，而非先合并控件类型。
2. **若未来要改为单行或共用同一 DOM 控件**：须**单独立项**，前置评审至少包括：换行在导出/交换格式中的语义、检索与 AI 管线、窄屏与紧凑列、IME 与无障碍、与组级译文的编辑模型是否冲突。
3. **CSS / token 层可对齐**：多行与单行可共用 **设计 token、hover/active 语义、保存状态指示**；不要求像素级「框高一致」。

## 4. 目标形态（一句话）

**布局与组级信息架构可不同，段块级外观与输入反馈应尽量同源；对照侧用 adapter 承接组级持久化与多 anchor 策略，不强行把组塞进单行媒体组件。**

---

## 5. 技术原则

1. **三路径回归**：涉及段块 class/组件时，按触及面至少覆盖 **对照 / 纯文本译文条（TextTranslationItem）/ 媒体译文条（TimelineAnnotationItem）** 的组合；不得默认「只测波形」或「只测 TextTranslationItem」。
2. **CSS 去耦**：优先抽 **共用 modifier / `data-*` 状态** 或极薄的「segment surface」样式层，由多壳共同 import；避免让对照 DOM 长期「冒充」媒体轨 class 名。
3. **焦点与无障碍**：列间 Tab、焦点可见样式、`aria-*` 与「从波形/侧栏回到哪一列」与状态机一并设计；不单靠视觉对齐。
4. **状态单一事实或显式派生**：文档化 `activeUnitId` 与 `comparisonFocus` 的优先级（谁覆盖谁、是否允许短暂不一致及如何收敛）。
5. **保存链路按职责抽象**：允许共用输入表面、状态点与 autosave 外壳，但 **source → unit 文本**、**target → layer / segment / group adapter** 的职责边界必须保留；禁止为了“统一”引入语义不清的单一 save 黑箱。
6. **性能**：对照列表与纯文本虚拟列表在共用大组件时须划定 **memo 边界**（按组/按单元），避免统一后重渲染退化。

## 6. 分阶段路线与验收

### Phase A — 视觉状态同源（低风险优先）

- **内容**：常态 / hover / active / focus / editing / empty 等 class 或 token 与主时间轴段块对齐；`timeline-comparison.css` 逐步收缩为**布局与组级 chrome**，段块皮肤迁出或引用共用层。（**不**要求对照 `textarea` 与轨条单行控件同高或同控件类型，见 §3.4。）
- **验收**：对照左右卡与「纯文本/媒体轨」下同态下的译文条在视觉上一致；窄屏紧凑模式不破坏焦点环。

### Phase B — 行为与反馈同源（中风险）

- **内容**：保存中/脏态/重试等反馈与 **纯文本 `TranscriptionTimelineTextTranslationItem`** 及 **媒体 `TimelineAnnotationItem` 路径上已有行为**对齐（两条路径能力未必一致，收口前需先列差异表）；能复用则抽 **与布局无关的输入+状态点内核**，**外层**仍保留对照 **组级 persist、多 target、多行**（adapter，非整颗替换 `TextTranslationItem`）。同时把**source 直接编辑**视作已落地能力，后续收口不得回退成只读/假编辑态。
- **验收**：关对照后纯文本行与开对照前同一数据的表现一致；**有媒体 URL 时**关对照后的波形译文条与改动前一致；组级保存与单条保存边界用例通过；source 直接编辑、target 保存、dirty/saving/error 点与重试路径行为一致。

### Phase C — 命令入口与选择模型（高风险，可拆子阶段）

- **内容**：右键菜单、区域 overlay、与侧栏/审查队列的选中对象与 `activeUnitId` / `comparisonFocus` 同步策略落地。
- **验收**：对照与两壳在「同一条用户路径」下操作同一语段时，菜单项与 overlay 目标一致、无静默错对象。

### Phase D — 文档与长期维护

- **内容**：若状态机或壳层职责发生结构性变化，在 `docs/architecture/` 或 `docs/adr/` 补一篇短 ADR；本文 `last_reviewed` 更新。
- **验收**：`npm run check:docs-governance` 通过；关键行为有 Vitest 或约定 smoke 覆盖。

**可选**：每阶段是否使用功能开关由实现者按发布节奏决定；若拆 PR，每阶段须自带可合并的验收子集，避免「半套皮肤半套行为」长期滞留主分支。

## 7. 测试与质量门槛

- **自动化**：已有 `TranscriptionTimelineComparison.test.tsx`；Phase B/C 须与 `TranscriptionTimelineTextOnly.test.tsx`、媒体轨/标注相关用例 **互补补差**，不得仅扩对照单测。至少补齐以下高风险断言：**source 直接编辑可保存**、`activeUnitId` ↔ `comparisonFocus` 同步、**target 侧右键/菜单锚定正确源 unit**、**multi-anchor 组保存路径不串写**。
- **手工清单**（在原有对照手工项基础上增加）：关对照 ↔ 纯文本列表、开对照 ↔ 双列、媒体轨译文条三态切换；键盘穿越列与 Esc 行为；宽屏与紧凑列切换下的高亮、焦点环与滚动稳定性。
- **发布前**：重大改动合入发布分支前，可对改动文件集追加一轮 `line-by-line-closure-auditor` 式逐行审计（与本文规划正交）。

## 8. 非目标（本文不承诺）

- 不为了统一而合并「文献时间轴」与「媒体解码时间轴」语义；
- 不在此阶段新建持久化对齐表（与对照产品方案中非目标一致）；
- 不把纯文本虚拟轨整套 UI 搬进对照视图；
- **本阶段不**把对照侧 **多行 `textarea`** 强行改为与纯文本/媒体轨一致的 **单行 `input`**，也不把「三处同一控件类型」作为交付定义（与 §3.4 一致）；若要做，须另开 ADR 与专项。

## 10. 代码落地核对（2026-04-19）

以下按规划章节对照 **当前仓库** 实现；用于修正认知与排障，**不**代替 Vitest / 手工回归。

### 与 §2、§3、§9 一致或已部分落地的项

| 规划断言 | 代码证据 |
| --- | --- |
| 对照优先分支 + `textOnlyProps` 展开 | `TranscriptionPage.TimelineContent.tsx` 先判 `comparisonViewEnabled && effectiveLayersCount > 0` 再渲染 `TranscriptionTimelineComparison` |
| `effectiveLayersCount` | 同上，`max(layersCount, transcriptionLayers.length, translationLayers.length)` |
| 对照开启时 `shell === 'text-only'` | `resolveTimelineShellMode` 中 `comparisonViewEnabled && hasLayers` 分支 |

| target 组级保存 + segment pick | `persistGroupTranslation` + `pickTranslationSegmentForPersist` |
| 纯文本 vs 媒体译文双路径 | `TextTranslationItem` 仅见于 `TranscriptionTimelineTextOnly`；媒体为 `MediaTranslationRow` → `renderAnnotationItem` |

### 与规划表述存在偏差的项（§3.4 / §9 已上修）

- 对照侧输入 **已不**是散落的裸 `textarea`，而统一经 **`TimelineDraftEditorSurface`**；与「Phase A 只做 CSS」相比，**输入壳 + 状态点**已部分收口。
- 纯文本壳与媒体译文行也已接入相同的 **saveStatus / retry 表面层**，说明 Phase B 的“共用输入壳 + 保存适配器”已从规划进入代码现实。

### 风险状态（2026-04-19 复验后）

1. **已闭合：segment 目标层无 `pick` 时的错误回退写入**
   - 当前在 `persistGroupTranslation` 中，若目标层使用 segment 且当前组无可对齐 segment，已改为 **toast 提示 + 阻断保存**，不再回退写入宿主 unit 层。
   - 已补回归：对照 target 保存阻断场景会断言 **不调用** `saveSegmentContentForLayer` / `saveUnitLayerText`，并显示错误状态点。

2. **已闭合：多锚点组右侧点击只打到主锚点**
   - 当前 target 列交互已改为优先解析 **context menu anchor → 当前 activeUnit → primary anchor**，不再固定只用 `primaryUnitId`。
   - 已补回归：合并组右侧点击会验证 `handleAnnotationClick` 命中解析后的锚点。

3. **已缓解：录音删除固定主锚点的风险**
   - 当前译音 scope 已先按已有译音挂载键、当前上下文锚点与 activeUnit 解析，再回落主锚点；删除/录音不再天然只绑第一条源语段。
   - 仍建议后续在产品上手工回归一次“多锚点 + 非主锚点已有录音”的真实交互。

4. **继续观察：target 侧右键菜单的 layer / unit 组合语义**
   - 当前逻辑已用更稳定的源 anchor 解析，并有回归用例覆盖“锚点正确”；
   - 但若未来 `useTimelineAnnotationHelpers` 内部更强依赖 `utt.layerId` 与行层严格一致，仍需保留该回归，避免后续重构时静默偏移。

## 11. 结论

本规划在「对照 vs 时间轴」之上显式并入 **纯文本壳与 `textOnlyProps` 路由事实**，并据代码核对补充：**媒体译文不经 `TranscriptionTimelineTextTranslationItem`**、**对照开启时底部不挂载 MediaLanes**、**壳枚举在对照下为 `text-only`**、**`comparisonFocus` 已以页面层为主持有**、**source / target 保存职责不可误合并**。截至本轮复验，**Phase B 的共享输入表面、保存状态点与重试反馈已在对照 / 纯文本 / 媒体三路径落地**，此前最突出的 **segment 无 pick 回退写错** 与 **多锚点右侧点击只认主锚点** 也已闭合。无障碍、命令入口、CSS 去耦、三路径测试与分阶段验收仍作为**与实现同级的约束**。按 Phase A→D 渐进执行，可在不推翻对照组信息架构的前提下，以「**继续消除剩余分叉**」而非「推倒重写」为原则，把段块分叉收束到可维护边界内。
