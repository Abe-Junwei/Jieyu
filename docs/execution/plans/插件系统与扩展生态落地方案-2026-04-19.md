# 插件系统与扩展生态落地方案

> **2026-04-19 首版**：结合当前仓库已落地的扩展运行时基础（Manifest / Capability / Lifecycle / Compatibility v1）与 Obsidian 之外的成熟产品经验，明确本项目插件能力的**产品定位、实施边界、分阶段路线与门禁**。本文目标不是立即开放“社区市场”，而是先把“预留插件功能”收敛为**可控扩展内核**。

---

## 0. 文档定位与一句话结论

- **本文目标**：为当前项目的“插件 / 扩展能力”提供一条可执行、可验证、可发布的落地路径。
- **适用阶段**：当前仓库仍处于产品主干快速收口阶段，适合先做**官方扩展 + 受控扩展位**，不适合直接开放完全自由的第三方生态。
- **一句话结论**：**先做可控扩展内核，再做扩展中心，最后才考虑开放社区生态。**

---

## 1. 为什么不能一开始就做“完全开放插件系统”

成熟产品几乎都不会直接把“任意代码执行”暴露给生态方，而是先建立以下四个基础：

1. **稳定宿主 API**：插件只能通过公开契约访问能力，而不是直接依赖页面私有实现。
2. **权限模型**：插件必须声明“要做什么”，宿主决定“允不允许”。
3. **生命周期治理**：安装、启用、停用、卸载、升级与失败降级都要可控。
4. **发布与审计机制**：用户可以看到来源、版本、权限与错误状态，必要时可一键停用。

如果缺这四项，就会出现：

- 插件碰内部实现，宿主一升级就大面积失效；
- 任意写库 / 任意写文件，数据一致性与安全性不可控；
- 一个扩展报错拖垮整个主界面；
- 无法审核来源、回滚版本与定位问题。

---

## 2. Obsidian 之外的成熟产品是怎么做的

不同产品的开放策略不同，但底层规律高度一致：


| 产品类型         | 代表产品                   | 核心策略                                     | 对本项目的启发                            |
| ------------ | ---------------------- | ---------------------------------------- | ---------------------------------- |
| **开发工具型**    | VS Code、JetBrains      | 扩展点清晰、宿主 API 稳定、常配合进程隔离与市场治理             | 适合借鉴 **命令 / 面板 / provider 注册点** 模式 |
| **设计工具型**    | Figma、Sketch           | 插件多在受控沙盒中运行，只能通过文档 API 读写画布              | 适合借鉴 **文档读写由宿主代行、插件不能直碰内部状态**      |
| **浏览器型**     | Chrome、Edge            | 权限申明最严格，安装时明确提示 scope，商店审核与下架机制成熟        | 适合借鉴 **最小权限原则 + 用户显式授权**           |
| **SaaS 集成型** | Notion、Slack、Zapier 生态 | 更偏 OAuth / Webhook / API 集成，而非在主应用内部任意执行 | 适合借鉴 **“集成优先于本地任意脚本”** 的开放路线       |


### 共性结论

无论是哪类产品，成熟形态最终都会落在同一组要素上：

- **Manifest**：身份、版本、入口、兼容范围；
- **Capability / Permission**：声明式权限；
- **Lifecycle**：load / activate / deactivate / unload；
- **Isolation**：崩溃、超时、资源泄漏不能伤及宿主；
- **Review / Distribution**：来源可信、更新可控、问题可回滚；
- **Observability**：日志、错误、熔断、安全模式。

## 4. 本项目推荐的产品定位

### 4.1 目标定位

本项目的插件体系建议分为三层，而不是一步到位做完全开放：


| 层级     | 对用户可见名称     | 开放程度 | 当前建议       |
| ------ | ----------- | ---- | ---------- |
| **L1** | 内建扩展 / 官方扩展 | 最保守  | **立即可做**   |
| **L2** | 受信本地扩展      | 中等开放 | **第二阶段再做** |
| **L3** | 社区扩展生态      | 高开放  | **后续单独立项** |


### 4.2 当前最适合先开放的扩展类别

优先开放那些**边界天然清晰**、且能明显提升产品能力的类型：

1. **AI Provider 扩展**
  - 新接一个 LLM / STT / Embedding provider；
  - 宿主掌握 UI、状态与审计，扩展只提供能力实现。
2. **导入 / 导出扩展**
  - 针对非标准格式的自定义转换器；
  - 风险边界清晰，适合通过 manifest + capability 收口。
3. **分析器扩展**
  - 声学分析、统计分析、批量校验器；
  - 以“输入数据 → 输出结果”的纯函数或准纯函数模式最稳。
4. **工作流脚本扩展**
  - 批量清洗、批量替换、项目检查；
  - 必须在宿主事务 / 审计壳之内执行。

### 4.3 当前不建议先开放的类别

- 任意 DOM 注入型插件；
- 可直接读写 Dexie / 本地文件系统内部结构的插件；
- 长期常驻并可绕开宿主状态机的后台脚本；
- 不受版本兼容与权限约束的“开发者模式全开”插件。

## 5. 核心架构原则

### 5.1 宿主拥有最终控制权

插件不是主系统的“第二宿主”，只能通过宿主提供的注册点与能力处理器工作。任何真正的数据写入、模型切换、任务派发、导出执行，都应由宿主完成。

### 5.2 统一 Capability，而不是散落特权口子

新增扩展能力时，优先在统一 capability 枚举里增补，不在业务代码里偷偷加“特判白名单”。这样才能保证：

- 权限提示清晰；
- 审计链路统一；
- 测试可以稳定补齐。

### 5.3 统一失败语义：可禁用、可降级、可回退

扩展失败不应直接导致主页面报废，而应至少支持以下一种结果：

1. 自动禁用该扩展；
2. 降级为宿主默认能力；
3. 保留只读模式；
4. 给出错误提示与诊断入口。

### 5.4 插件“能做什么”与“看起来能做什么”必须一致

UI 中所有扩展入口都应来源于真实 capability，不允许界面暴露按钮但运行时因权限或宿主版本不兼容而静默失败。

### 5.5 事务、审计与资源清理由宿主统一托管

特别是涉及：

- 转写内容修改；
- 语言资产写入；
- AI 调用；
- 导入导出。

这些都应走宿主统一壳层，以便记录：

- 调用方扩展 ID；
- 时间；
- 输入摘要；
- 输出状态；
- 错误信息；
- 是否回滚。

## 6. 推荐的 V1 → V3 分阶段路线

### Phase A — 官方扩展位（当前阶段优先）

**目标**：把“预留插件功能”从概念变成**受控扩展槽位**。

**范围**：

- 仅支持官方或仓库内受信扩展；
- 不开放任意第三方安装；
- 只使用当前 Manifest / Capability / Lifecycle / Compatibility 契约；
- 提供最小化扩展管理面板。

**建议交付物**：

1. **扩展中心 v1**
  - 已安装列表；
  - 扩展名称 / 版本 / 状态；
  - capability 展示；
  - 启用 / 停用按钮；
  - 错误状态与重试。
2. **扩展注册点 v1**
  - AI provider slot
  - import / export slot
  - analysis provider slot
  - workflow action slot
3. **审计与观测 v1**
  - 记录扩展加载失败；
  - 记录 capability 调用失败；
  - 为后续安全模式预留诊断入口。

**验收门槛**：

- 未声明能力调用拦截率 = 100%；
- 生命周期超时可被识别并禁用；
- 加载失败不影响主应用核心页面；
- 可视化看到每个扩展的状态与权限。

### Phase B — 受信本地扩展（中期）

**目标**：允许用户安装“本地扩展包”，但仍在宿主边界内受控运行。

**新增能力**：

- 本地扩展包导入 / 删除；
- 来源与签名信息展示；
- 首次安装权限确认；
- 版本不兼容时阻断启用；
- 安全模式：一键禁用全部非官方扩展。

**建议额外约束**：

- 只允许声明过的 capability；
- 对 AI / 导出 / 文件访问类能力做更强提示；
- 需要配额与节流（如连续失败熔断、调用频率限制）。

### Phase C — 社区生态与分发治理（后续单独立项）

**目标**：把“可控扩展框架”升级为“可审核的生态系统”。

**需要新增的不是功能点，而是治理能力**：

- 官方扩展仓库 / 注册表；
- 扩展审核流程；
- 签名、来源校验、撤回机制；
- 更新渠道与版本回滚；
- 社区评分、问题反馈与兼容公告。

**判断是否进入 Phase C 的前提**：

1. 宿主 API 在至少一个发布周期内稳定；
2. Phase A/B 中的扩展错误已可观测、可定位、可回退；
3. 已有 3–5 个官方 / 内部扩展证明注册点设计合理；
4. 主要业务主线已收口，不再频繁重构宿主内部模型。

## 7. 能力矩阵建议（V1 起步）

建议继续沿用 capability 白名单思路，并按业务域扩展：


| 能力                      | 含义                 | 风险级别 | V1 是否建议开放 |
| ----------------------- | ------------------ | ---- | --------- |
| `read.transcription`    | 读取转写数据、层与语段        | 低    | 是         |
| `write.transcription`   | 修改转写文本或结构          | 高    | 仅官方扩展     |
| `read.language-assets`  | 读取语言资产数据           | 低    | 是         |
| `write.language-assets` | 修改语言资产数据           | 高    | 仅官方扩展     |
| `invoke.ai`             | 调用 AI 能力与任务管线      | 中到高  | 是         |
| `export.interop`        | 触发格式导出             | 中    | 后续建议新增    |
| `import.interop`        | 触发受控导入             | 高    | 后续建议新增    |
| `register.panel`        | 注册只读面板 / inspector | 中    | 后续建议新增    |
| `run.workflow`          | 批处理或规则检查           | 中到高  | 后续建议新增    |


**原则**：所有高风险能力默认不开，按“官方扩展 → 受信扩展 → 社区扩展”逐层放开。

## 8. 扩展中心（用户向界面）建议

成熟产品不会只提供一个“启用插件”开关，而会把管理能力显式做出来。建议扩展中心至少包含：

### 8.1 列表页

- 名称、版本、来源；
- 状态：active / disabled / incompatible / error；
- 权限摘要；
- 最近错误或最后成功运行时间。

### 8.2 详情页

- Manifest 信息；
- Capability 列表；
- 与当前宿主版本的兼容性结果；
- 错误日志与诊断建议；
- 启用、停用、卸载、重试按钮。

### 8.3 安全模式

当用户遇到异常时，应能快速进入：

- **禁用所有非官方扩展**；
- **只保留核心能力**；
- **导出扩展故障报告**。

这是成熟产品中极其重要但常被低估的一环。

---

## 9. 安全与治理要求

### 9.1 最小权限原则

扩展只声明自己真正需要的能力。任何“为了以后方便先全开”的设计都应避免。

### 9.2 宿主代行写入

扩展尽量不直接写库，而是调用宿主提供的 handler，让宿主完成：

- 参数校验；
- 事务包裹；
- 冲突处理；
- 审计日志；
- 撤销 / 回滚接入。

### 9.3 超时与熔断

对扩展加载、初始化、任务执行都应有 timeout 与失败计数。超过阈值后自动停用，避免错误扩展在启动期反复拖死界面。

### 9.4 来源可信与版本兼容

- 本地扩展必须显示来源；
- 官方扩展可带签名或内置白名单；
- 不兼容宿主版本的扩展只允许显示，不允许激活。

### 9.5 数据与隐私边界

若未来开放更强 AI / 网络型扩展，需在 capability 层显式区分：

- 本地计算；
- 网络请求；
- 外部模型 / 服务调用；
- 数据是否允许离开本地。

---

## 10. 非目标（本阶段不承诺）

- 不在当前阶段建设完整“社区插件市场”；
- 不开放可绕开宿主状态机的任意代码执行；
- 不允许扩展直接依赖页面私有 DOM / 未文档化内部状态；
- 不把“插件系统”做成一个新的核心复杂度来源，反过来拖慢主产品收口。

---

## 11. 建议的近期实施顺序

### Step 1：把当前运行时基础接到用户可见层

- 做扩展中心 v1；
- 把 manifest / capability / 状态显示出来；
- 接好启停、错误态与兼容提示。

### Step 2：先落 2–3 个官方扩展示例

建议优先挑这几类：

1. 一个 AI provider 扩展；
2. 一个导出格式扩展；
3. 一个分析器 / 批量检查扩展。

如果这三类都能稳定挂载，说明扩展点设计基本成立。

### Step 3：补门禁与文档

- 扩展契约测试继续补齐；
- 增加安全模式与故障日志；
- 更新本文与 release gate 文档。

---

## 12. 验收口径

当满足以下条件时，可认为“插件功能已从预留进入可用态”：

1. 至少有 **2–3 个官方扩展** 能被真正加载、启停与回退；
2. Manifest / Capability / Compatibility / Lifecycle 全链路有自动化覆盖；
3. 用户可在扩展中心看见状态、权限与错误；
4. 任一扩展失败不会阻断核心转写 / 语言资产主流程；
5. 扩展相关门禁可在本地与 CI 中稳定运行。

---

## 13. 结论

对当前项目而言，最优路径不是直接复制某一个成熟产品的表层形态，而是抽取这些产品共同验证过的底层规律：

- **Manifest 管身份**；
- **Capability 管边界**；
- **Lifecycle 管启停**；
- **宿主管数据与事务**；
- **扩展中心管用户可见治理**；
- **门禁与审计管发布风险**。

因此，当前最推荐的战略不是“马上做社区插件平台”，而是：

> **把现有扩展运行时继续收敛为一套可控、可观测、可发布的官方扩展框架；等主产品与宿主 API 稳定后，再决定是否向受信本地扩展与社区生态演进。**

## 14. 执行进展与实现记录（2026-04-20）

### 14.1 与本文对齐的代码增量（Phase A 局部）

以下项在仓库中落地，用于缩小「文档 Phase A」与「仅单元测试运行时」之间的差距：

1. **宿主版本号单一来源**：与构建注入的 `__APP_VERSION`__ / `VITE_APP_VERSION` 对齐，供 `negotiateManifestCompatibility` 与设置 UI 共用。
2. `**ExtensionRegistry`（多扩展）**：在单例 `ExtensionHost` 之上增加注册表，支持按扩展 id 注册 / 卸载官方扩展、列举状态与兼容结果。
3. `**invokeCapability` 治理**：可选调用超时、try/catch 包裹宿主 handler、可注入审计回调（为后续安全模式与故障导出预留）。
4. **激活失败语义**：激活抛错时清理 manifest/hooks，生命周期进入 `**error`**（与「用户停用」的 `disabled` 区分）；兼容性拒绝仍为 `disabled` 且不挂载半残状态。
5. **扩展中心 v1（设置内）**：设置面板新增「扩展」页：展示宿主版本、已注册扩展列表（名称、版本、状态、能力、兼容性说明）、最近能力调用审计摘要；**不**开放任意第三方安装。
6. **内置官方契约探针**：`jieyu.official.host-contracts` 在用户**首次打开**「扩展」设置页时注册，用于验证全链路；`manifest.entry` 仍为契约字段，实际 hooks 由仓库内建模块注入（见下节）。

### 14.2 `manifest.entry` 与真实加载路径（避免误解）

- **当前阶段**：运行时仍由宿主在注册扩展时传入 `ExtensionHooks`；`entry.activate` / `entry.deactivate` 字符串为 **Manifest 契约与未来动态加载器**预留，**不等于**已在浏览器内按路径动态 `import()`。
- **后续 Phase B**：若支持「本地扩展包」，再实现 entry 解析、校验与沙箱策略；在此之前，门禁与 UI 以 manifest + 实际注册来源为准。

### 14.3 仍待后续迭代（未在本轮承诺）

- 将真实业务（如 AI Provider、导入导出）逐步迁入 `invokeCapability` 宿主 handler，而非仅桩实现；
- 安全模式、一键禁用非官方扩展、故障报告导出；
