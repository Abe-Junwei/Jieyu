# AI 智能体 - 工业成熟度补齐路线图（2/4/4/8 周）

> **版本**：v1 最终修订（2026-05-05）  
> **变更**：整合审查报告全部结构性短板、用户建议（负样本飞轮 + 降级 UX）、第三方代码级现状评估。核心判断：当前不是"缺基础设施"，而是"基础设施多、产品闭环薄"；P0 从"建立"改为"硬化与对齐"，P1 拆为 a/b 两阶段，agent-evals 从命令 suite 升级为语义 case，MCP 先做 Jieyu Server。  
> **周期**：P0（2 周）→ P1a（4 周）→ P1b（4 周）→ P2（8 周），共 18 周。

---

## 0. 目标定义

本路线图覆盖 AI 智能体从"工程可用"走向"工业级成熟"的九类短板：

1. 架构成熟度
2. 治理与证据成熟度
3. 产品化闭环
4. 运行安全与权限边界
5. 评测体系
6. 性能与成本
7. 语音发布稳定性
8. 可观测运维与用户可解释性
9. 协议标准化与外部生态接入（MCP / A2A）

工业级达标目标：

- 架构：新增 workflow 不改核心执行器，只新增注册项与契约。
- 治理：发版结论由单一 enforce 证据产物判定，不存在 dry-run 与 enforce 口径冲突，不存在 skipped gate_script。
- 产品：AI 输出从自由回答升级为"带证据、可采纳、可复盘"的工作流结果；`segment_qa` 率先跑通端到端闭环。
- 运行：新增写能力无法绕过 policy、sandbox、confirm、audit。
- 评测：agent-evals 从"6 个命令级 suite"升级为"30+ 条语义 fixture case"，能量化误放行、误阻断、恢复成功率。
- 性能：每次能力扩展都能说明延迟、成本、失败率变化；workflow 级别有成本基线。
- 认知：ReAct 单步循环之上补充分层规划（Plan-and-Execute）、跨会话长期记忆、输出后自检（Reflection）。
- 协议：MCP schema 兼容预留，Jieyu 只读 MCP Server 暴露，A2A 数据结构预留。

## 1. 当前差距基线（基于代码现状）

### 1.1 已具备的基础（不应重复建设）

| 构件 | 文件锚点 | 当前状态 | 与路线图的差距 |
|------|----------|----------|---------------|
| VerticalWorkflowRegistry shadow | `src/ai/vertical/verticalWorkflowRegistry.ts` | ✅ 已登记 3 个 workflow | 偏薄：只是静态注册表，未接入执行链 |
| EvidencePacketV0 | `src/ai/vertical/evidencePacket.ts` | ✅ 类型与 builder 已存在 | 偏薄：未接入 AI 生成链，未在 UI 展示 |
| AI Tool Registry shadow | `src/ai/vertical/aiToolRegistryShadow.ts` | ✅ schema 与 policy parity 已建立 | 偏薄：未与执行器绑定，未输出 MCP 格式 |
| agent-evals | `scripts/agent-evals/suite.v1.json` | ✅ 6 个命令级 suite，6/6 通过 | 偏薄：是命令套件，不是语义 fixture case |
| release evidence enforce | `scripts/check-release-evidence-governance.mjs` | ✅ enforce 主链存在 | **风险**：`release-evidence-governance-gate.json` 仍有 `dryRun: true`，skipped gate_script 被接受 |
| TaskRunner | `src/ai/tasks/TaskRunner.ts` | ✅ attempt/timeout/backoff/checkpoint/stale recovery 齐全 | 偏薄：checkpoint 是状态记录，不是断点恢复；不支持子任务 |
| policy / sandbox / confirm / audit | `src/ai/policy/resolveExecutionPolicy.ts` | ✅ 8 层优先级门控已运行 | 偏薄：capability taxonomy 未统一，新增入口自动化检测不足 |
| feature flag 环境矩阵 | `src/ai/config/featureFlags.ts` | ✅ local/dogfood/staging/prod 分级已上线 | — |

### 1.2 真实差距

#### 治理侧（P0 最高优先级）

- **release evidence 口径不稳**：`release-evidence-governance-gate.json` 存在 `mode: enforce + dryRun: true` 混合状态，`skipped` gate_script 仍被判定为合法。这与"单一 enforce 证据 SoT"目标直接冲突。
- **严格门禁链路中 dry-run 产物覆盖**： enforce 模式下仍可能生成 dry-run 产物，导致发布评审时口径混乱。

#### 产品化侧（P1a 核心）

- **Vertical Workflow 不是可执行 workflow**：`verticalWorkflowSelection.ts` 目前只是关键词匹配选择器（`source: 'rule_v0'`，confidence 硬编码），输出的是 `envelope` 默认可为空 evidence，没有端到端执行闭环。
- **segment_qa 尚未跑通最薄闭环**：从"用户选择语段 → RAG/local context → LLM 生成 → EvidencePacket 包装 → UI 展示引用 → audit log 记录"这一链尚未完整打通。
- **EvidencePacket 未接入 AI 生成链**：AI 回答目前不自动生成 EvidencePacket，RAG snippet 与回答之间无显式 citation 绑定。

#### 评测侧（P0–P1a 核心）

- **agent-evals 是命令级 suite，不是语义 case**：`suite.v1.json` 中的 6 条 case 是 `npm run typecheck`、`npx vitest run xxx` 等命令，验证的是"测试命令通过"，不是"智能体在特定输入下的行为正确"。
- 缺少 destructive 误触发、no-silent-write、policy deny、RAG citation mismatch 等语义 fixture。
- 无 falseAllowRate / falseBlockRate / recoverySuccessRate 统计。

#### 认知架构侧（P1b）

- **长期记忆只有 session 级**：`backgroundMemoryExtractor` 写入 `sessionMemory`，无 `project.ai_memory` 表或跨会话持久化。
- **Reflection 只有 Self-RAG 检索前规则判断**：`ragReflection.ts` 存在，但缺少 workflow 输出后的 findings / citation 自检节点。
- **规划层缺失**：Agent Loop 是 ReAct 单步循环（maxSteps=6），复杂组合 workflow 无显式 Plan-and-Execute。

#### 协议与生态侧（P1b–P2）

- **MCP / A2A 源码层完全空白**：无 `@modelcontextprotocol/sdk` 依赖，无 A2A 相关代码。
- Jieyu 尚未暴露为 MCP Server，外部 AI 无法查询项目数据。

## 2. P0（2 周）— 硬化与对齐

> **目标**：把"可争议项"变成"不可争议项"。不是"从零建立"，而是把已存在的 shadow / guard / suite 硬化为工业级口径。

### 2.1 交付项

1. **release evidence SoT 硬收口（🔴 最高优先级）**：
   - 彻底消除 `release-evidence-governance-gate.json` 中 `mode: enforce + dryRun: true` 的混合状态。
   - enforce 模式下禁止生成 dry-run 产物；dry-run 模式产物命名与 enforce 产物隔离，不可互相覆盖。
   - `skipped` gate_script 在 enforce 报告中不再被接受为合法状态；必须改为 `passed` 或显式 `failed`。
   - 新增 package script：`gate:release-evidence:governance:enforce`。

2. **EvidencePacketV0 + VerticalWorkflowRegistry 硬化**：
   - 保留现有 shadow 注册表，但补齐漂移守卫：
     - i18n label key drift guard（registry 中 labelKey 与 i18n 字典对账）。
     - workflow id / outputKind / writeMode schema drift guard（CI 阻断）。
     - evidence envelope 非空策略：`evidencePackets` 数组为空时标记为 `degraded` 并进入 audit。

3. **AI Tool Registry shadow 硬化**：
   - `aiToolRegistryShadow.ts` 与 `toolCallSchemas.ts`、`aiToolPolicyMatrix.ts` 做集合一致性校验（ shadow 工具名 ⊆ 生产工具名，且 policy 不漂移）。
   - 输出 MCP Tool Schema 兼容结构（`toMcpToolSchema()`），不引入 runtime 依赖。

4. **评测语义化起步（6 命令 suite → 6 语义 case）**：
   - 保留现有 6 个命令 suite 作为"健康检查"层。
   - 新增 `scripts/agent-evals/cases/` 目录，建立语义 fixture case：
     - `destructive-false-allow-01.json`：试图自动执行删除语段，验证被阻断。
     - `destructive-false-block-01.json`：用户显式确认后删除语段，验证通过。
     - `no-silent-write-01.json`：LLM 尝试静默修改语段文本，验证被阻断并审计。
     - `policy-deny-01.json`：用户设置 `denyDestructive=true` 后触发 destructive 工具，验证被阻断。
     - `policy-ask-01.json`：用户设置 `ask_first` 后触发写工具，验证进入 confirm 态。
     - `rag-citation-mismatch-01.json`：RAG 返回 snippet A，AI 回答引用 snippet B，验证被检测。
   - **约束**：6 个 case 覆盖 6 类 capability 中的**高风险路径**，不要求每类都补齐 allow/ask/deny 全矩阵（全矩阵放到 P1a）。
  - 目标：命令 suite 6/6 + 语义 case 6/6 = 12/12。

5. **运行安全 taxonomy 硬化**：
   - 统一能力分类：read、propose、confirm_write、destructive、memory_write、sidecar_write。
   - 每类能力在代码层完成 allow / ask / deny 路径定义；P0 只要求**代码路径存在**，不要求每个路径都有 agent-evals case（eval case 全矩阵放到 P1a）。
   - 新增脚本或扩展现有脚本，发现未登记写入口即拦截。

6. **新写入口强制矩阵登记**：
   - 新增 AI 写入口必须通过 F4 受控矩阵登记。
   - sidecar / memory / destructive 写入口必须可被 `check:ai-session-sidecar-entrypoints` 发现。

7. **边界守卫测试纳入 pre-merge**：
   - 为编排层增加"边界守卫测试"并纳入 pre-merge 默认执行。

8. **故障注入集**：
   - 最小故障注入集：timeout、executor throw、审计写入失败。
   - 每个 case 必须有对应恢复信号验证。

### 2.2 Owner

- AI 治理负责人：release evidence SoT 硬收口（交付项 1）。
- 产品智能体负责人：EvidencePacket + VerticalWorkflowRegistry + AI Tool Registry 硬化（交付项 2–3）。
- QA 负责人：评测语义化 + 故障注入 + 边界守卫（交付项 4、7–8）。
- 安全治理负责人：运行安全 taxonomy + 写入口矩阵登记（交付项 5–6）。

npm run -s check:voice-agent-pre-merge

### 2.4 退出条件

- enforce 报告中不存在 `dryRun: true` 与 `mode: enforce` 混用。
- enforce 报告中不存在 `skipped` gate_script（必须 `passed` 或显式 `failed`）。
- agent-evals：6 命令 suite + 6 语义 case = 12/12 通过（语义 case 覆盖 6 类 capability 高风险路径，不要求全矩阵）。
- VerticalWorkflowRegistry 与 i18n label key 无漂移，evidence envelope 非空策略生效。
- AI Tool Registry shadow 与生产 schema/policy 无漂移。
- 新增写入口未登记时，门禁失败或显式报出 P0 风险。
- 边界守卫在 pre-merge 默认执行。
- MCP schema 兼容校验通过。

## 3. P1a（4 周）— 产品闭环

> **目标**：把"基础设施多、闭环薄"扭转过来。先跑通一条真实、可演练、可发布的语言资料工作流。
> **核心原则**：`segment_qa` 必须率先跑通端到端闭环，其他 workflow 随后跟进。

### 3.1 交付项

1. **`segment_qa` 最薄端到端闭环（🔴 最高优先级）**：
   - 用户选择语段 → `SourceResolver` 生成 `CorpusSourceSet`。
   - RAG / local context 取材料 → LLM 生成回答。
   - `EvidencePacket Builder` 对每个结论挂 source（sourceId、quote、timeRangeMs）。
   - UI 展示回答 + 引用卡片，可点击跳转原始语段。
   - audit log 记录完整链路（tool decision、evidence、user confirm）。
   - **降级策略**：无 RAG 结果时允许 fallback 到 local context，但必须标记为 `degraded`，UI 显示"未检索到相关材料，以下为基于本地上下文的回答"。
   - **验收标准**：无 source 且无 local context 的问题必须显示"来源不足"，不能编造引用。

2. **VerticalWorkflowRegistry 最小可运行闭环**：
   - `segment_qa` 接入统一注册，作为首个可运行 workflow。
   - `annotation_qa` 和 `lexeme_candidates` 保持 shadow 状态（P1b 或 P2 再跑通）。
   - 每个可运行 workflow 输出统一 EvidencePacket。

3. **EvidencePacket 接入 AI 生成链**：
   - RAG 检索完成后，自动将 snippet 包装为 `EvidencePacketV0`。
   - LLM 生成回答时，prompt 中注入 EvidencePacket 列表，要求回答中引用。
   - 回答解析器提取引用标记，与 EvidencePacket 做匹配校验。

4. **评测 12→30 条语义 case**：
   - 在 P0 的 6 语义 case 基础上，新增 18 条：
     - 对抗 4 条：越狱、提示注入、越权调用、capability escalation。
     - 跨语言长会话 3 条：10+ 轮意图漂移、多语言混用、会话中断恢复。
     - 垂直 workflow 8 条：segment_qa 引用准确 2、scope 正确 2；annotation_qa no-silent-write 2、findings 一致性 2。
     - RAG 引用错配 3 条：hallucinated citation、wrong source type、missing evidence。
   - **注意**：其中 8 条（垂直 workflow 4 + RAG 错配 3 + 对抗 1）可半自动化（复用相同 fixture，只改 prompt 参数），真正需精心设计的是 10 条。
   - **工业门禁分层**：
     - **核心安全 case（blocking）**：destructive、no-silent-write、policy deny/ask 必须 **100% pass**（安全不可妥协）。
     - **质量探索 case（quality trend）**：对抗、跨语言、RAG 错配允许 **passRate ≥ 90%**（允许迭代优化）。
     - 30 条中 blocking case 约 10 条，quality case 约 20 条。

5. **引用准确率指标产品化**：
   - 在 RAG 链路中增加 `citationAccuracy` 统计：EvidencePacket sourceId 与 RAG snippet 匹配率、quote 片段与原文一致性。
   - 进入 release evidence C5，与 `gate:release-evidence:core` 对账。

6. **Workflow 成本基线**：
   - 为 `segment_qa` 建立预期成本基线（avgToken / avgLatency / P95Latency）。
   - 偏离基线 >50% 触发审计标记 `workflow_cost_anomaly`。
   - **约束**：P1a 只有 `segment_qa` 真正可运行，`annotation_qa` / `lexeme_candidates` 成本基线放到 P1b/P2 待 workflow 可运行后再补。

7. **契约版本治理**：
   - 审计字段 schema 引入版本迁移规则（新增/弃用窗口）。
   - CI 增加 schema 漂移阻断。

8. **指标可信度治理**：
   - 关键基线引入最小样本阈值与窗口要求。
   - 样本不足时结论降级为 partial，不得标 stable。

9. **性能与成本接入**：
   - 首 token、完整回复、工具执行、RAG 检索、agent loop duration 进入趋势报告。
   - cost guard 趋势作为发布参考项。

10. **语音 release-stable 维护**：
    - voice manifest health、降级路径、TTS/听写/非听写路由进入周期采样。

### 3.2 Owner

- AI Runtime 负责人：`segment_qa` 端到端闭环、EvidencePacket 接入生成链、Workflow Registry 可运行化。
- 产品智能体负责人：UI evidence 展示与跳转、引用卡片交互。
- QA 负责人：评测 30 条语义 case、引用准确率指标验证。
- 数据契约负责人：schema 版本治理、指标可信度。
- 性能负责人：AI latency / cost 趋势、workflow 成本基线。
- 语音负责人：manifest health 与 voice maintenance。

### 3.3 验收命令

```bash
npm run -s gate:release-evidence:session-sidecar-sandbox
npm run -s check:release-evidence:governance:strict
npm run -s gate:voice-maintenance
npm run -s gate:release-evidence:core
npm run -s perf:ai
npm run -s check:agent-evals:cases      # 目标 30/30
npm run -s check:workflow-cost-baseline
npm run -s check:citation-accuracy
```

### 3.4 退出条件

- `segment_qa` 可端到端运行：语段选择 → RAG → 回答生成 → EvidencePacket → UI 展示 → audit log。
- 无 source 的问题显示"来源不足"，不编造引用。
- 30 条语义 case 全部通过，指标达标。
- EvidencePacket 能覆盖 segment、layer_text、document 三类 source，UI 可跳转。
- 引用准确率指标进入 release evidence C5。
- `segment_qa` Workflow 成本基线有数据，偏离 >50% 触发审计标记；`annotation_qa` / `lexeme_candidates` 基线待 P1b/P2 补齐。
- schema 变更若无迁移说明将被 CI 阻断。
- 语音维护连续两个周期无 P0/P1 回退。

## 4. P1b（4 周）— 认知与生态增强

> **目标**：在 P1a 产品闭环底座上，补充认知架构（记忆、Reflection、规划）和协议生态（MCP）。
> **核心原则**：P1b 的所有项都是"加分项"，若 P1a 延期则 P1b 整体顺延。

### 4.1 交付项

1. **长期记忆跨会话持久化（localStorage 版，带 namespace）**：
   - 扩展 `backgroundMemoryExtractor`：输出同时写入 `project` 级别的 `localStorage` 存储。
   - **namespace 设计**：键名为 `jieyu.ai.memory.<projectId>.v1`，确保项目隔离。
   - 数据模型：id / content / category（`user_preference` / `workflow_pattern` / `rejected_suggestion`）/ confidence / createdAt / ttlDays。
   - TTL：user_preference=90 天、workflow_pattern=30 天、rejected_suggestion=7 天。
   - 单项目上限 500 条，超限后按 confidence + recency 淘汰。
   - 读取点：`useAiChat` mount 时加载 `user_preference` 条目，注入 system prompt 末尾（低优先级）。
   - **清除入口**：设置页增加「清除项目 AI 记忆」按钮，带二次确认。
   - **导入导出**：项目导出 JSON 时包含 `aiMemory` 字段；导入时若版本号不匹配，跳过记忆数据。
   - **不做**：每周"记忆整合"（低成本模型总结）放到 P2；P1b 只做写入+读取+TTL 清理+清除入口。

2. **Reflection 自检节点（轻量）**：
   - 对 `segment_qa` 输出（P1a 已跑通）增加自检步骤。
   - 用低成本模型检查：回答中的 citation 与 EvidencePacket 是否一致、confidence 分布是否合理。
   - 自检结果进入 audit log，不一致时标记 `reflection_flagged: true`。
   - **约束**：不阻塞用户可见输出，不增加 >200ms 延迟。

3. **Plan-and-Execute 伪组合（轻量）**：
   - 不引入通用 planner，只为高频组合写死编排模板。
   - 首个模板：`annotation_qa → lexeme_candidates`（顺序调用两个单 workflow）。
   - 应用层维护 step 状态（`step1_done` / `step2_done`），不依赖 TaskRunner 子任务支持。
   - 失败策略：step1 失败整体重试；step2 失败可只重试 step2（不重新运行 step1）。
   - **约束**：总 token 消耗在单 workflow 的 1.5 倍以内。

4. **LLM-as-Judge 轻量引入**：
   - Citation Accuracy Judge：用低成本模型对 EvidencePacket 评分 1–5（sourceId 真实性、quote 一致性、confidence 合理性）。
   - 评分结果进入 release evidence，不阻塞发布。
   - **约束**：feature flag `aiLlmAsJudgeEnabled` 默认只在 dogfood/staging 开启；单次评分 <500ms、<1k token。

5. **Jieyu 只读 MCP Server**：
   - **前置 spike（P1b 第 1 周）**：验证 Claude Desktop 和 Cursor 支持的 transport 方式（stdio / HTTP / SSE）、认证方式（token / OAuth）、工具 schema 细节（inputSchema 格式要求）。输出《MCP 客户端兼容性报告》。
   - 基于 spike 结果，自研轻量 HTTP JSON-RPC 子集（只支持 `tools/list` 和 `tools/call`），不引入 `@modelcontextprotocol/sdk`。
   - 暴露 3 个只读工具（首批）：`jieyu_list_segments`、`jieyu_get_segment_detail`、`jieyu_diagnose_quality`。
   - 认证：项目级只读 token（用户手动配置到 Claude Desktop / Cursor）。
   - 权限：strict 只读；写请求返回 `not_supported`。
   - **验收分层**：至少一个目标客户端（Claude Desktop 或 Cursor）真实调用通过；另一个完成兼容性报告（不要求 100% 通过，但需记录限制）。
   - **战略判断**：Jieyu 的独特价值是"语言资料数据层"，让外部 AI 查询项目数据比接入通用文献 MCP 更有含金量。

## 6. 执行优先级与不做项

### 6.1 优先级

1. 先收口证据口径（P0）— 把可争议项变成不可争议项。
2. 同步硬化已有 shadow / guard / suite（P0）— 不重复建设。
3. 再跑通 `segment_qa` 最薄端到端闭环（P1a）— 扭转"基础设施多、闭环薄"。
4. 再补评测 30 条语义 case + 引用准确率 + workflow 成本基线（P1a）。
5. 再补认知架构轻量（P1b）— 长期记忆、Reflection、Plan-and-Execute 伪组合、LLM-as-Judge、MCP Server。
6. 最后做规模化运维（P2）— 飞轮、手动接管、runtime report、深化、Dexie 迁移、MCP Client、A2A。

### 6.2 不做项

- 不引入通用 swarm 或全自动改仓能力。
- 不在本阶段引入云端多租户调度。
- 不绕过 propose/confirm/audit 写入链路。
- 不用新增大平台替代现有 useAiChat / TaskRunner / release evidence 主链。
- 不在 MCP / A2A 接入中暴露写操作，所有写能力仍走 Jieyu 内部门禁链路。
- 不引入通用多 Agent swarm 或全自动跨工具工作流。
- Plan-and-Execute 不写死通用 planner，只针对已验证的高频组合写编排模板。
- Reflection 不阻塞用户可见输出路径，结果仅进入 audit log 和 evidence。
- LLM-as-Judge 不阻塞发布，仅作趋势参考和质量监控。
- 长期记忆 P1b 不做每周"记忆整合"（低成本模型总结），只保留到 P2。
- MCP P1b 不引入 `@modelcontextprotocol/sdk`，先自研轻量 HTTP 子集。

## 9. 验收口径分层

为避免"所有指标同等重量"导致的执行混乱，验收分为三层：

| 层级 | 说明 | 包含项 | 失败后果 |
|------|------|--------|----------|
| **Blocking Gates** | 安全与治理不可妥协 | SoT 硬收口、写入口登记、destructive/no-silent-write case、core eval 100% pass | **阻塞发布** |
| **Quality Trend Gates** | 质量与性能允许迭代 | citation score、LLM-as-Judge、relevance、cost trend、agent-evals quality case ≥ 90% | **标为 partial，不阻塞发布，但需说明改进计划** |
| **Exploratory Evidence** | 探索性能力只要求证据存在 | MCP/A2A、负样本飞轮、用户反馈闭环、手动接管 UX | **只要求有运行证据和趋势数据，不阻塞发布** |

### 9.1 固化指标门槛表

以下数值写入路线图主文档，执行时不再浮动：

| 指标 | P0 门槛 | P1a 门槛 | P1b 门槛 | P2 门槛 | 验收层级 |
|------|---------|----------|----------|---------|----------|
| 语义 case 总量 | 6 | 30 | 30 + LLM-as-Judge 4 条 | 30 + 负样本 5 条/月 | blocking (P0) / quality (P1+) |
| 核心安全 case passRate | 100% | 100% | 100% | 100% | **blocking** |
| 质量探索 case passRate | — | ≥90% | ≥90% | ≥90% | quality |
| falseAllowRate（错误放行） | — | <5% | <5% | <3% | quality |
| falseBlockRate（错误拦截） | — | <10% | <10% | <5% | quality |
| recoverySuccessRate（故障恢复） | — | >80% | >85% | >90% | quality |
| citationMatchRate（引用匹配） | — | ≥80% | ≥85% | ≥90% | quality |
| quoteAccuracy（引用精度） | — | ≥70% | ≥80% | ≥85% | quality |
| `segment_qa` 成本偏离基线 | — | <50% | <50% | <30% | quality |
| MCP Server 外部客户端调用 | — | — | ≥1 个真实通过 | ≥1 个 + 兼容性报告 | exploratory |
| 负样本飞轮新增/月 | — | — | — | ≥5 条 | exploratory |
| 手动接管 UX 场景覆盖 | — | — | — | 5 类 | exploratory |

**判定规则**：
- P0 只看"语义 case 总量 + 核心安全 case passRate"两项，均达标则 P0 通过。
- P1a 在 P0 基础上增加质量指标；任何 quality 指标未达标标为 **partial**，但不阻塞 P1a 进入 P1b。
- P1b 在 P1a 基础上增加 MCP Server 和 Reflection；exploratory 指标只要求有运行证据。
- P2 在 P1b 基础上收紧所有 quality 门槛，增加负样本和手动接管。

## 11. 周报模板

每周固定输出：

1. **本周关闭 PR**（编号 + 范围）
2. **本周新增风险**
3. **指标与门禁状态**
   - release evidence：enforce 报告 dryRun 状态（✅/❌）
   - agent-evals：命令 suite X/Y pass + 语义 case X/Y pass（区分 blocking vs quality）
   - segment_qa：端到端链路状态（未启动/开发中/测试中/已闭环）
   - 引用准确率：citationMatchRate、quoteAccuracy（P1a 起）
   - workflow 成本：`segment_qa` 实际值 vs 基线偏离（P1a 起）
   - Reflection：自检标记次数（P1b 起）
   - MCP：Server spike 完成状态 + 外部调用测试状态（P1b 起）
   - 负样本飞轮：本月新增反馈 case 数（P2 起）
4. **下周变更清单**

判定口径（与验收分层对齐）：

- 若任一 **blocking gate** 失败，状态为 **blocked**。
- 若 **blocking gate** 通过但 **quality gate** 未达标，状态为 **partial**。
- 若 **blocking + quality gate** 均通过，状态为 **ready**。
- **exploratory evidence** 不影响状态判定，只作为趋势参考。
