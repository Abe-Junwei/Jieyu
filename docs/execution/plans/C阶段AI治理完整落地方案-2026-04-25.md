# C 阶段 AI 治理完整落地方案（2026-04-25）

## 0. 目标与边界

本方案将《证据化发布与 AI 工具治理完整方案（2026-04-20）》中的 C1-C7 全部落地为可执行工程任务：

1. C1 Memory Broker：统一 session memory、project memory 与 RAG 的召回、重排、预算裁剪和解释。
2. C2 扩展信任等级、配额与健康度：把 capability audit 升级为按信任等级治理。
3. C3 Voice Provider Adapter Manifest：只描述现有 voice provider，不重写语音栈。
4. C4 意图多候选与置信门控：降低误调用和 ambiguous target。
5. C5 后台记忆抽取子代理：异步抽取高价值记忆，不阻塞主链路。
6. C6 后台任务工具沙箱：后台任务能力 fail-closed，先只读、再受限写。
7. C7 轻量协调模式：定义 research / synthesis / implementation / verification 相位与任务通知协议。

非目标：

1. 不把未接热路径的治理能力宣称为运行时保护；默认值以 `src/ai/config/featureFlags.ts` 为准，dogfood 已开启的能力必须有单测与 release evidence 支撑。
2. 不引入重型 swarm 或全量多智能体编排。
3. 不以 fixture-only 证据替代运行时证据；release evidence 必须能区分 `ready`、`skipped`、`degraded`。

## 1. Feature Flags

新增或使用以下 flags 作为唯一默认开关：

1. `aiMemoryBrokerEnabled`：启用 C1 Memory Broker。
2. `aiMemoryRecallShapeTelemetryEnabled`：启用运行时 memory/RAG shape metadata。
3. `aiIntentConfidenceGateEnabled`：启用 C4 多候选与置信门控。
4. `aiBackgroundToolSandboxEnabled`：启用 C6 后台工具沙箱。
5. `aiBackgroundMemoryExtractorEnabled`：启用 C5 后台记忆抽取。
6. `aiExtensionTrustGovernanceEnabled`：启用 C2 扩展信任治理。
7. `aiVoiceProviderManifestEnabled`：启用 C3 voice provider manifest。
8. `aiCoordinationLiteEnabled`：启用 C7 轻量协调协议。

当前默认值（2026-04-25，以 `src/ai/config/featureFlags.ts` 为准）：

1. 默认开启：`aiMemoryRecallShapeTelemetryEnabled`、`aiMemoryBrokerEnabled`、`aiIntentConfidenceGateEnabled`、`aiBackgroundMemoryExtractorEnabled`、`aiExtensionTrustGovernanceEnabled`、`aiVoiceProviderManifestEnabled`、`aiCoordinationLiteEnabled`。
2. 默认关闭且未接运行时热路径：`aiBackgroundToolSandboxEnabled`。
3. `aiCoordinationLiteEnabled` 当前为 audit-only，不作为写入并发控制的强制执行层。

约束：已 dogfood 开启的 C 阶段能力仍必须保持可回滚 flag；未接热路径的能力不得在 release sign-off 中作为已生效运行时保护。

## 2. 实施阶段

### Phase 0：启动基线与发布证据口径

交付：

1. 写入本方案并加入 `docs/execution/plans/README.md`。
2. 修复 release evidence fail-open：关键证据缺失、显式输入读取失败时，`enforce` 必须 `degraded=true` 且非零退出。
3. 更新 CostGuard trend gate 产物，使根 `summary` 统计完整 `evidenceIndex`。

验收：

```bash
npm run -s check:architecture-guard
npm run -s check:docs-governance
npm run -s gate:release-evidence:cost-guard-trend
```

### Phase 1：运行时 Telemetry 生产者

交付：

1. 在 RAG enrichment 路径产出 `memoryRecallShape`：
   - `candidateCount`
   - `selectedCount`
   - `duplicateSuppressedCount`
   - `budgetSuppressedCount`
   - `freshnessBucket`
2. 将该 metadata 带入后续工具决策审计，使 `release-evidence` 的 `memoryRecallShape` 卡可以读取真实运行时数据。
3. 保留 fixture 测试，但不把 fixture 作为完成证据。

验收：

```bash
npx vitest run src/hooks/useAiChat.rag.test.ts src/hooks/useAiChat.test.tsx scripts/generate-release-evidence-bundle.test.ts
npm run -s typecheck
```

### Phase 2：C1 Memory Broker

交付：

1. 新增 `src/ai/memory/memoryBroker.ts`。
2. API 输入：`query`、`tokenBudget`、`sessionMemory`、`projectMemoryStore`、`alreadySurfacedRefs`。
3. API 输出：`MemoryChunk[]`，每项含 `source`、`score`、`why`、`freshnessDays`、`suppressedBy`。
4. 两阶段召回：candidate recall → budget-aware rerank/select。
5. 关闭 flag 时保持现有 `enrichContextWithRag` 行为。

验收：broker 单测覆盖空 store、预算裁剪、重复抑制、why 保留。

### Phase 3：C6 后台任务工具沙箱

交付：

1. 后台 capability profile：`readonly`、`restricted_write`、`deny_by_default`。
2. Shell 只允许命令 + 参数白名单的只读集合。
3. 写入工具校验 workspace root、cwd 与授权目录。
4. 拒绝变量展开、重定向、process substitution、`cd && write` 等绕过。
5. deny/ask 输出稳定 audit reason。

验收：安全绕过样例全部落入 ask/deny。

### Phase 4：C5 后台记忆抽取子代理

前置：C6 只读沙箱完成。

交付：

1. 主响应完成后触发后台抽取，失败不影响主链路。
2. 主链路已写记忆时跳过后台抽取。
3. 抽取中收到新上下文时合并 trailing run。
4. 审计记录 `taskId`、`actorId`、输入范围、写入数量、skipped reason、duration。
5. transcript/schema version 与恢复一致性校验。

验收：重复触发不重复写，恢复后无孤立 tool result。

### Phase 5：C4 意图多候选与置信门控

交付：

1. 工具意图结果新增 `intentCandidates`、`confidence`、`margin`。
2. top1/top2 margin 低于阈值时进入最小澄清。
3. 审计 metadata 写入候选和阈值。
4. `progressiveDisclosure.cards[]` 使用真实事件计算。

验收：ambiguous target 不直接执行，明确 target 不新增澄清。

### Phase 6：C2 扩展信任、配额与健康度

交付：

1. 扩展 manifest 增加 `trustLevel`、`declaredCapabilitiesVersion`、quota profile。
2. registry 聚合 success rate、failure rate、p95 duration、deny reason distribution。
3. release evidence 增加 extension health card。

验收：低 trust 扩展无法请求高风险 capability，配额超限有稳定 reason。

### Phase 7：C3 Voice Provider Manifest

交付：

1. 描述现有 provider：engine id、display name、capabilities、load function key、health check、local/remote。
2. `VoiceAgentService` 读取 manifest，但保留现有 lazy runtime。
3. release evidence 记录启用 engine 与 health 状态。

验收：manifest 缺失或 health fail 时可降级，不显著增加 bundle。

### Phase 8：C7 Coordination Lite

交付：

1. 任务相位：`research`、`synthesis`、`implementation`、`verification`。
2. 并行策略：只读并行、写入串行、验证可并行。
3. 通知 schema：`taskId`、`status`、`summary` 必填；`result`、`usage` 选填。
4. 不合规通知进入隔离通道。
5. 会话结束清理临时上下文、pending task 与后台任务。

验收：会话结束后残留后台任务数为 0，release evidence 有 coordination 卡。
