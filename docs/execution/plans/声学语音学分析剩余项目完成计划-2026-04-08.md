# 声学语音学分析剩余项目完成计划（2026-04-08）

> 文档角色：执行方案文档。用于把当前未闭合的 Phase 2 / 3 / 4 事项压成可执行的交付顺序、验收目标与依赖关系。

## 1. 当前判断

基于当前代码状态，剩余项目不应再按“从零开始做声学功能”来规划，而应按以下现实前提推进：

1. Phase 1 主线已经完成，剩余只属于 polish 与约束收口。
2. Phase 2 已具备本地 runtime、VAD warmup、live inspector 基础，但缺重型 inspector 收口与统一运行时语义。
3. Phase 3 已具备 descriptor、近似 Formant、元音空间摘要与 CSV / JSON 导出，但缺成熟 backend、校准闭环与科研互操作导出。
4. Phase 4 仍应作为最后阶段，不应抢跑到默认主链之前。

## 2. 总目标

完成计划的总目标是把剩余项目按“先稳定当前本地能力，再补科研口径，最后开放 provider 扩展”的顺序收口。

对应总原则：

1. 先收口 inspector 和参数层，再做更重的 Formant / 导出。
2. 先把本地结果做到可解释、可校准，再谈外部 provider。
3. 先形成稳定的数据契约，再扩 UI 和导出格式。

## 3. 分阶段完成计划

### 3.1 Wave A：Phase 2 Inspector 收口

目标：把现有 acoustic tab 从“轻量统计面板”提升为“可停留、可比较、可继续扩展的 inspector 工作区”。

本波次交付：

1. 增加 inspector 冻结能力，允许把当前 hover 点固定为 snapshot。
2. 增加双点比较能力，至少支持当前点与冻结点并排对比。
3. 为 inspector 增加独立的信息容器，不再完全复用统计行布局。
4. 为热点与 inspector 建立最小统一选中态。

明确不做：

1. 不在本波次引入新 backend。
2. 不在本波次重做 acoustic analysis 核心算法。
3. 不在本波次做科研导出。

验收标准：

1. 鼠标离开后，冻结点数据仍可保留在右侧 panel。
2. 当前点与冻结点能并列展示 F0、intensity、descriptor、formant。
3. 从热点列表进入时，主波形区与 panel 共享同一选中状态。

建议优先级：最高。

### 3.2 Wave B：Phase 3 参数层收口

目标：把现有只读 runtime 参数升级为研究级配置层，并与缓存键、重算语义闭环。

本波次交付：

1. 增加参数面板，开放 pitch floor / ceiling、window、frame step、silence threshold 等控制项。
2. 定义 preset 与 custom 模式，避免直接暴露原始参数集合而缺少语义。
3. 参数变更后触发新的 analysis config、缓存键与局部重算。
4. 在 UI 中明确展示当前图表使用的参数集与算法版本。

明确不做：

1. 不在本波次引入外部 provider。
2. 不在本波次做 PitchTier 导出。

验收标准：

1. 任意参数调整都会生成新的配置语义，而不是只改展示文本。
2. 用户能看出当前结果对应哪套 preset 或 custom 参数。
3. 缓存回退、stale 标记与重算提示行为一致。

建议优先级：高。

### 3.3 Wave C：Phase 3 Formant 与校准闭环

目标：把当前“近似 Formant / vowel space”推进到“有明确校准基线的实验级能力”。

本波次交付：

1. 为 Formant 增加更稳定的 backend 路线，例如 LPC / Burg 候选提取、平滑与可靠度过滤。
2. 建立固定校准样本集，不只保留示例 JSON。
3. 固化校准门槛，让脚本输出可稳定用于 gate，而不只是一次性人工验证。
4. 在 UI 中把未达标状态明确标为 exploratory。

明确不做：

1. 不在校准未通过前把 Formant / 元音图包装为确定性分析结果。
2. 不在本波次开放跨语言稳定性承诺。

验收标准：

1. `formantF1Hz` / `formantF2Hz` 误差与覆盖率有固定门槛。
2. UI 能根据校准状态决定使用 exploratory 还是正式标签。
3. 元音空间图不再只是直接显示已有摘要，而是能绑定新的 backend 结果。

建议优先级：高。

### 3.4 Wave D：Phase 3 科研互操作导出

目标：把当前导出从前端调试格式升级为研究工作流可复用格式。

本波次交付：

1. 保留当前 CSV / JSON，并补充更稳定的元数据契约。
2. 增加 `Praat` 方向的 PitchTier 或等价互操作导出。
3. 支持整段媒体或批量选区导出，而不只是一段 selection。
4. 导出结果带上算法版本、参数、采样率与校准版本。

明确不做：

1. 不追求一次性覆盖所有外部研究工具格式。
2. 不在本波次重做主 UI。

验收标准：

1. 导出结果可以被外部工具读取，而不是只适合前端回看。
2. 导出文件可追溯具体分析配置和校准基线。

建议优先级：中高。

### 3.5 Wave E：Phase 4 Provider 扩展准备

目标：在本地主链稳定后，为外部增强 provider 打基础，但不让其污染默认链路。

本波次交付：

1. 抽象 acoustic provider-neutral 契约，与默认本地链路并列。
2. 区分默认本地能力与增强 provider 能力的 UI 命名与状态表达。
3. 先做 provider contract、配置入口、可达性检测与降级策略。
4. 在此基础上再评估 WhisperX、MFA、pyannote、MMS 等具体接入次序。

明确不做：

1. 不把任何外部 provider 升级为默认能力。
2. 不在本地主链未稳定时并行推进多个重型 provider。

验收标准：

1. 用户能清楚区分“默认本地分析”与“增强 provider”。
2. provider 失败不会破坏默认本地链路。
3. provider 契约可复用于至少两个增强方向，而不是一次性硬编码某一个库。

建议优先级：最后。

## 4. 推荐时间顺序

若按最小返工路径推进，建议顺序如下：

1. 先做 Wave A，解决 inspector 不能停留、不能比较的问题。
2. 再做 Wave B，把参数与缓存语义收口。
3. 再做 Wave C，补 Formant backend 与校准闭环。
4. 然后做 Wave D，把导出升级到科研互操作层。
5. 最后做 Wave E，开启 provider-neutral 扩展。

原因：

1. Wave A 决定后续所有重分析 UI 的承载面。
2. Wave B 决定后续校准、缓存和导出的数据契约。
3. Wave C 决定 Formant 与元音空间能否从 exploratory 进入正式实验层。
4. Wave D 依赖前面三波形成稳定结果语义。
5. Wave E 不应反向影响默认本地链路设计。

## 5. 阻塞关系

1. Wave B 依赖 Wave A 至少完成 inspector 容器与冻结态，否则参数调整没有稳定承载面。
2. Wave C 依赖 Wave B 至少完成参数契约与缓存键收口，否则校准结果无法稳定复跑。
3. Wave D 依赖 Wave C 至少完成校准门槛，否则导出只会放大不稳定结果。
4. Wave E 依赖 Wave B 到 Wave D 的主链稳定，否则 provider 接口会提前固化错误边界。

## 6. 每波次建议拆分

### 6.1 Wave A 建议拆分

1. A1：冻结态与 pinned inspector 数据模型。
2. A2：双点比较 UI 与热点联动。
3. A3：主波形区 / 侧栏统一选中态与回归测试。

### 6.2 Wave B 建议拆分

1. B1：参数 schema 与 preset 定义。
2. B2：参数 UI、缓存键和重算逻辑。
3. B3：参数切换的提示、降级和测试。

### 6.3 Wave C 建议拆分

1. C1：Formant backend 候选提取与平滑。
2. C2：校准样本集与脚本固化。
3. C3：UI exploratory / calibrated 状态接线。

### 6.4 Wave D 建议拆分

1. D1：稳定元数据导出契约。
2. D2：PitchTier 或等价互操作导出。
3. D3：批量 / 整段导出与测试。

### 6.5 Wave E 建议拆分

1. E1：acoustic provider contract。
2. E2：配置与 reachability 检测。
3. E3：首个增强 provider 的受控试接入。

## 7. 当前最推荐的下一步

如果现在立刻开始做，最推荐起手的是 Wave A。

原因：

1. 它最贴近当前已有代码结构，不需要先引入新 backend。
2. 它能直接提升现有 acoustic tab 的可用性。
3. 它会为后面的参数层、Formant 和导出提供稳定承载面。

一句话总结：先把现有能力变成稳定工作区，再继续补科研口径和 provider 扩展。
