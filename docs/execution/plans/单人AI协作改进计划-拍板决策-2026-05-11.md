# 单人 + AI 协作改进计划（拍板决策）

本文档记录 **2026-05-11** 与改进路线相关的八项决策，并据此收敛执行优先级。约束背景：单人开发、主要依赖 AI 实现、无历史兼容包袱、无固定排期；**但已选择「半年内可能对外/多用户」与「主动治理 i18n」**，故在计划中预留对应增量。

---

## 一、已拍板决策（宪法）

| 项 | 选项 | 含义 |
|----|------|------|
| **1** | **B** | 已有或 **半年内可能对外/多用户**：改进计划中 **预留** 隐私/发布相关项与更完整的国际化准备，不按「永久自用工具」缩范围。 |
| **2** | **A** | **合并前验证**：必跑 `typecheck` + 与本次改动相关的定向 `vitest`；E2E 仅在改动交互/装配/侧栏等相关路径时跑 `npm run test:e2e:chromium`（或等价子集）。 |
| **3** | **A** | **Knip/删除**：unused file 在确认无静态引用、无已登记动态入口后 **快删**；动态入口集中 knip 白名单 + 简短注释。 |
| **4** | **A** | **E2E**：失败用例以 **修稳** 为主（helper、业务稳态等待、必要时 `data-testid`），不设长期 `skip` 债。 |
| **5** | **A** | **ReadyWorkspace 合同化**：按域 **渐进**（建议先 side pane / overlays，再 waveform / layout）；中间允许 `unknown` + 窄 type guard。 |
| **6** | **B** | **i18n**：在维持 `check:i18n-hardcoded:guard` **不新增债** 的前提下，**计划性消减基线**；**UI 文案** 与 **AI prompt/内部协议类字符串** 分 catalog（目录或命名约定清晰即可，不必一步到位）。 |
| **7** | **A** | **依赖**：升级以 **手动** 为主；按需或安全公告；knip 报告的 unlisted dependency 经确认为直接 `import` 后再写入 `package.json`。 |
| **8** | **A** | **文档**：本计划 **落盘** `docs/execution/plans/`；重大删除或装配边界变更用 **commit message + 必要时更新 `docs/architecture/`**；不强制每条 ADR。 |

## 二、对总路线的修正（相对「纯自用」版）

1. **1B**：在阶段 F 中，将「上线前再打开」里的 **隐私最小集、发布检查单、观测是否满足对外最低线** 提升为 **半年内可启动的 backlog**，与功能开发并行排优先级（仍无固定截止日期，但不无限期推迟）。
2. **6B**：在阶段 F 中，**i18n 不是仅防新增**，需设 **基线消减里程碑**（例如按模块：hooks/pages → components，或与对外时间表挂钩）；**AI 消息 catalog** 与 UI `dictKeys` 分离，避免混用同一套 key 命名空间。

其余阶段（ReadyWorkspace 合同、E2E 可信、Knip 收缩、热点按需拆）**不变**，仅验证门槛与合并习惯遵循 **2A**。

## 三、执行阶段（与决策对齐）

### A. 基线与个人默认验证

- 将 **2A** 写成仓库内可引用的一句话（例如 `AGENTS.md` 或 `package.json` 脚本注释旁），避免每次会话重议。
- 快照当前 knip 输出，作为阶段 D 对照。

### B. ReadyWorkspace 装配合同（**5A**）

- 分域替换 `any` / 无意义 `as any`；与 `docs/architecture/ReadyWorkspace-数据域与壳层装配边界.md` 保持一致。
- 每域结束：`typecheck` + 相关定向 vitest。

### C. E2E 可信化（**4A**）

- 创建层等流程收口 helper；web-first 断言；结构选择器处补稳定 `data-testid`。
- 修因而非长期 skip；本地可对脆弱 spec 做连续多次复跑后再认为结案。

### D. Knip 与代码面（**3A**）

- unused files 逐个确认后删除；动态入口白名单；unlisted dependency 按 **7A** 处理。

### E. 热点文件按需降温

- 仅在「改动频繁 + 上下文过大」时拆；遵守现有 controller/service 落位纪律。

### F. 对外与 i18n（**1B + 6B**）

- **i18n**：制定「基线消减顺序 + 完成定义」（例如某目录 hits 归零或下降阈值）；新建 UI 文案只走字典；AI 侧新建字符串走 **独立 catalog**，并在 `copilot-instructions` 或架构说明中写清边界。
- **对外预留**：数据流/协作/云同步相关的一页「最小隐私与导出说明」、发布前 smoke 清单；与现有 `docs/execution/release-gates/` 衔接，不另起大而全体系。

### G. 依赖（**7A**）

- 保持手动节奏；若未来感到痛苦再评估低频自动化。

## 四、刻意不做（除非决策变更）

- CODEOWNERS、多角色评审路由、正式 on-call / SLO 看板。
- 高频 Dependabot/Renovate（与 **7A** 冲突除非另议）。
- 为形式而做的「全量 E2E 每次合并」（与 **2A** 冲突除非另议）。
