## 一、背景与现状诊断

经过两轮代码级审查与核实，当前 AI 子系统呈现"**工程骨架扎实、产品路径未全部跑通**"的状态：

### 1.1 已成熟的能力
- **Send-turn 管道**：4 阶段拆分（Preflight → Persist+Primary Stream → Stream Phase → Completion），含首块超时、输出 cap 重试、session token budget。
- **Agent Loop 恢复**：支持 token 预算预警、断点续跑、`pendingAgentLoopCheckpoint` 跨刷新恢复。
- **工具决策治理**：8 层过滤（rollback → intent → gray → dedup → validation → user directive → destructive gate → auto execute），每层有审计。
- **Provider 多模降级**：10 种 LLM 提供商统一接口，主备自动降级。
- **本地上下文工具**：20+ 种只读工具（list_units、search_units、get_project_stats、diagnose_quality 等），grounding 极强。
- **审计与证据**：工具决策审计、intent 审计、agent loop 步骤审计、background memory 审计，全链路覆盖。

### 1.2 未跑通的瓶颈
- **MCP Server 已接真实门面**：`tools.ts` 经 `segmentReadQueries` 读 Dexie；**空 runtime scope 硬失败**（`SEGMENT_READ_SCOPE_REQUIRED`），避免未绑宿主时的静默全库读取。外部宿主仍需正确注入 `McpServerRuntimeContext`。
- **垂直工作流失衡**：仅 `segment_qa` 有完整闭环（reflection、source resolver、composed workflow retry）；`annotation_qa`、`lexeme_candidates` 有注册但无深度闭环。
- **CorpusSourceSet 仍缺最小产品面**：当前已有 `EvidencePacketV0`、source resolver 与 prompt context 投影，但用户还不能清晰看到、保存或调整“本次 AI 可读哪些语段 / 层 / 笔记 / 文档 / 词典范围”。这会限制 NotebookLM 式来源限定能力。
- **AdoptionQueue 部分进入主线**：`AiChatCard` 已承载待审阅队列与 outcome 审计；垂直审计条目可汇入队列；`elan_flex_compatibility` 流式结束项经 `onPushAdoptionItems` 与宿主 ref 贯通至同一队列。**仍缺**：`annotation_qa` / `lexeme_candidates` 的等价 push、以及队列 **采纳 → `propose_changes` / confirm** 的产品闭环（见 P4 checklist）。
- **ELAN / FLEx 互通检查未纳入本方案主线**：当前 P4 聚焦 `annotation_qa` 与 `lexeme_candidates`，但语言资料产品最贴近用户价值的互通报告（EAF tier 映射、FLEx 词条候选、时间码缺口、受控词表不一致）仍未进入阶段验收。
- **领域评测样本偏工程化**：现有 eval 与 release evidence 能证明工具链和治理链路，但低资源语言、断句边界、gloss / POS 冲突、词条重复、引用越界、无证据拒答等语言资料 golden cases 仍不足。
- **用户可解释运行状态不足**：治理侧已有 audit / report，但用户侧还缺“本次读了哪些 source、为什么无证据、为何低置信、为什么不能写、下一步可怎么扩大范围”的明确反馈。
- **Release evidence 关键 AI 卡片 skipped**：以 `release-evidence-20260506-182733-core.json` 为例，22 张卡片中 7 张 skipped（约 31.8%），关键 AI 模块如 `c5.background-memory-extraction`（`no_background_memory_extraction_rows`）、`f3.durable-orchestration`（`no_ai_task_snapshot_source`）、`c7.coordination-lite`（`no_coordination_lite_rows`）、`c8.user-directive-governance`（`no_user_directive_governance_rows`）仍缺真实审计数据源。
- **主 hook 层边界过厚**：`src/hooks/useAiChat*` 共 69 个 TS/TSX 文件（生产文件 44 个 + 测试 25 个）；`useAiChat.ts` 审查快照为 **430** 行且承载 runtime wiring（历史审查曾记录 484/508 行）；architecture-guard 当前 `useAiChat` 上限 **1100** 行、`ReadyWorkspace` 上限 2600 行。

### 1.3 总体判断
当前系统是一个**本地优先、证据驱动、受控工具执行的科研 AI 助手平台雏形**。工程成熟度（安全、审计、恢复、门禁）已超越一般前端项目，但距离成熟工业级垂直 AI 产品仍缺五个产品闭环：**来源限定、专业输出对象、人在环采纳、领域评测、工作流嵌入**。解语不应追通用 autonomous agent，而应产品化为“证据型语言资料智能体”。

调研 NotebookLM、Glean、Harvey、Lexis+ with Protégé、Abridge、Looppanel、Microsoft Foundry Agent Service、LangGraph / LangSmith、CrewAI、LlamaIndex 与 MCP 后，本方案只吸收与解语定位相关的能力：

## 二、改进原则

1. **做深而非扩宽**：先把已有骨架上的关键路径做深做实，不新增协议或 workflow。
2. **数据优先**：让 release evidence 中关键 AI 卡片从"缺真实审计数据 skipped"变成"有真实运行数据支撑"。
3. **结构减压**：新增能力落入 `src/ai/` 纯逻辑层，停止向 `useAiChat` 派生文件堆积状态。
4. **以点带面**：先打样 `segment_qa`，形成可复制模板，再补齐 `annotation_qa` / `lexeme_candidates`。
5. **来源先行**：证据型回答必须先明确 source scope，再生成答案；无来源时优先解释 scope，而不是只提示 degraded。
6. **采纳优先于自动写入**：成熟形态不是让 AI 静默替代研究者，而是生成可审阅候选并进入 `AdoptionQueue`。
7. **Flow 控制状态，Agent 处理不确定性**：确定性流程由 workflow / service 管理，只有检索、总结、候选生成等不确定步骤交给模型；不为展示复杂度而引入多 agent。
8. **调研证据驱动取舍**：每个新增能力需注明对应对标样本与“为何适配解语”，避免复制他业态的非必要复杂度。
9. **契约先于界面扩张**：凡新增 source set、evidence、adoption、compatibility report，都先固化字段契约、状态机、审计 schema、eval 样本与 runtime 指标，再扩 UI 复杂度。

## 三、阶段规划（复核后顺序：P0 ∥ P2a 起步，P1 产品闭环，P2b/P3/P4/P5 递进）

### 阶段一：MCP Server 真实化（P0，2–3 周）

**目标：** 将 `src/ai/mcp/server/tools.ts` 从 static fixtures 迁移到真实 Dexie 查询，使外部 AI 客户端能读取项目上下文。

**行动项：**

1. **抽只读查询门面（前置步骤）**
   - 新增 `src/ai/queries/segmentReadQueries.ts`，暴露 `listSegmentSummaries`、`getSegmentDetail`、`diagnoseProjectQuality` 三个纯函数。
   - 门面内部消费 `SegmentMetaService`、`WorkspaceReadModelService`、`db.dexie`，不依赖 React 上下文。
   - `localContextTools.ts` 和 MCP handler **共同复用**该门面，避免 MCP 直接耦合 `localContextTools` 这个大文件。

2. **MCP Server 接入真实查询**
   - `jieyu_list_segments` → 调用 `listSegmentSummaries`，返回当前 project/media/layer scope 内的真实语段。
   - `jieyu_get_segment_detail` → 调用 `getSegmentDetail`，返回 layers、annotations、translations 等真实字段。
   - `jieyu_diagnose_quality` → 调用 `diagnoseProjectQuality`，返回基于 `WorkspaceReadModelService.summarizeQuality()` 的真实诊断。

3. **当前项目绑定（Runtime Context）**
   - 定义 `McpServerRuntimeContext`：至少包含 `projectId`、`textId`（映射到 `AiPromptContext.shortTerm.workspaceTextId`）、`currentMediaId`、`currentLayerId`。
   - Server 启动时注入只读 context provider（复用 `AiPromptContext` 的 shortTerm 投影）。
   - `tools/call` **持久化审计**（已实现）：每条调用写入 Dexie **`mcp_tool_call_audits`**（schema **v48**，**独立表**、**非** `audit_logs`）— **`arguments` 全量 JSON**、**`runtimeContextJson`**（`McpServerRuntimeContext` 全量，用于事后复核宿主绑定）、outcome（成功与 **validation / 未知名称 / 执行错误 / 超时** 等失败路径均写）；插入失败 **best-effort**（打日志，不阻断 MCP）。成功时返回体 JSON 超长截断存储。
   - 作用域规则与 `localContextTools` 的 `segmentMetaStorageLayerId` 等读模型对齐。

4. **信任边界与背压**
   - 复用 `src/ai/mcp/server/auth.ts` 的 Bearer token 校验；未授权时硬拒绝（`401 Unauthorized`），不返回空结果。
   - 明确接入范围：**仅本机 loopback**（默认 `127.0.0.1`），不承诺任何远程 MCP transport。
   - 第一版仅单线程串行处理 `tools/call`；`limit` 上限 100，`offset` 上限 1,000（无分页快照句柄前保守值）；超时 30s；与现有 session 成本护栏不冲突。

### 阶段五：垂直工作流复制与采纳队列（P4，3–4 周，依赖 P2）

**目标：** 以 P1 固化的 `segment_qa` 模板（reflection、fallback、eval case shape、evidence binding）为契约，复制到 `annotation_qa` 和 `lexeme_candidates`，并补齐成熟垂直产品所需的 `AdoptionQueue` MVP。

**本季度执行顺序（已拍板）：** 先 **AdoptionQueue MVP**（人环采纳/忽略/证据跳转闭环），再 **`annotation_qa` / `lexeme_candidates` 深度复制**（reflection、eval suite 与 AdoptionQueue 接入可随第二梯队并行，但以队列骨架为先）。

**行动项：**

1. **模板契约固化（P1 衔接）**
   - 在 P1 完成后，明确写出 `VerticalWorkflowTemplate` 契约：
     - reflection 结果类型（`reflectionFlagged`、`checks[]`、`summary`）
     - fallback UX 模式（无证据、低置信、反射失败）
     - evidence 绑定方式（`EvidencePacketV0` → UI citation）
     - eval case 最小字段集（id、category、goldenTaskCount、trajectorySignals）
   - 该契约已以 **ADR-0030** 落档（`docs/adr/0030-vertical-workflow-template-contract.md`）；后续迭代修订该 ADR 与代码契约，不新建平行占位 ADR。

2. **补齐缺失的闭环**
   - `annotation_qa`：新增 `annotationQaReflection.ts`（检查标注一致性、空标注、语言匹配）。
   - `lexeme_candidates`：新增 `lexemeCandidatesReflection.ts`（检查词形有效性、POS 合法性、与现有词典冲突）。
   - 两者接入 `sourceResolver.ts`，获取真实 annotation / lexeme 数据源。

3. **AdoptionQueue MVP**
   - 新增统一候选审阅对象：`AdoptionCandidateV0`，字段至少包含 `id`、`workflowId`、`outputKind`、`title`、`summary`、`evidencePackets[]`、`recommendedAction`、`writeMode`、`status`、`requestId`。
   - 支持 `accept`、`ignore`、`copy`、`jumpToEvidence` 四个动作；`accept` 只允许走 `propose_changes` / confirm / audit，不允许直接写库。
   - `ignore` 必须记录 outcome，可选记录 reason；采纳率、忽略率进入 eval / runtime report。
   - 状态机固定为：`pending -> accepted | ignored | copied | expired`；`accepted` 进入 `proposed` / `confirmed` / `failed` 的写入链路由 `propose_changes` 与 confirm 审计承接，不在 AdoptionQueue 内部静默完成。
   - 审计字段至少包含：`candidateId`、`workflowId`、`outputKind`、`action`、`previousStatus`、`nextStatus`、`reasonCode`、`requestId`、`evidencePacketIds`、`proposalId`（如有）。
   - P4 首版只覆盖 `annotation_qa` findings 与 `lexeme_candidates` candidates；不做完整多角色协作审阅。

4. **Composed Workflow 健壮性**
   - 为 `annotation_qa_then_lexeme_candidates`、`segment_qa_then_annotation_qa_then_lexeme_candidates` 每步增加 reflection gate。
   - reflection flagged 时的行为策略：自动 retry（最多 1 次）→ 仍 flagged 则进入用户确认分支；或 reflection flagged 时直接进入用户确认分支，不自动 retry。具体策略在实现前由开发者在 ADR 中明确。

5. **Eval Suite 复制**
   - 使用 P1 固化的 eval case 模板，为 `annotation_qa` 和 `lexeme_candidates` 各增加 4+ cases，接入 `suite.v1.json`。
   - 增加 AdoptionQueue 相关 cases：候选必须带 evidence、采纳必须 confirm、忽略必须 audit、无 evidence 候选不得进入可采纳态。

**验收标准：**
- [ ] `VerticalWorkflowTemplate` 契约 ADR 已发布。
- [ ] `annotation_qa` 和 `lexeme_candidates` 拥有与 `segment_qa` 同等级别的 reflection、fallback、eval suite。
- [ ] `AdoptionCandidateV0` 契约存在，`annotation_qa` 与 `lexeme_candidates` 输出可进入 AdoptionQueue。
- [ ] AdoptionQueue 支持 accept / ignore / copy / jumpToEvidence；accept 必须复用 `propose_changes` / confirm / audit。
- [ ] AdoptionQueue 状态机与 outcome audit schema 已有单测覆盖。
- [ ] AdoptionQueue outcome 进入 audit 与 `AiRuntimeReport`。
- [ ] Composed workflow step1 reflection flagged 时，按既定策略正确处理（自动 retry 后进入用户确认，或直接进入用户确认），UI 展示当前状态与下一步动作。
- [ ] 三个 `evalSuiteId` 均有对应 cases 在 CI 中运行。

### 阶段六：语料库 Source Set 与 ELAN / FLEx 互通检查（P5，3–5 周，依赖 P1/P4）

**目标：** 补齐与解语产品定位最相关、但不适合挤进 P0/P1 的工业级垂直能力：语料库 NotebookLM 化 source set 与 ELAN / FLEx compatibility report。P5 不复制 ELAN / FLEx 编辑器，只做检查、候选、导出辅助。

**行动项：**

1. **CorpusSourceSet 完整化**

   - Source set 可绑定会话，不污染长期记忆；用户可在提问前切换当前 source set。
   - RAG / read model 按 source set 限定检索，EvidencePacket 必须可回溯到 source set 成员。

4. **导出辅助与外软桥接**

   - 报告可导出为 markdown / JSON bundle，保留 `corpusRef`、时间码与 project metadata。

**验收标准：**
- [ ] Source set 可保存、复用、切换，且不会污染长期记忆。
- [ ] Source set 生命周期与失效 fallback 已有测试覆盖。

- [ ] 互通报告可导出为 markdown / JSON bundle。
- [ ] EAF / FLEx 最小 fixture 已进入 eval 或单测；round-trip golden validation 已登记为 P5 后续项。

## 四、明确边界（不列入本方案的事项）

以下事项同 §一 1.4“明确不吸收”列表，仅合并同类项并补充垂直工作流边界：

| 远程 MCP transport | P0 仅 in-proc MCP Server（默认 loopback）；不承诺任何远程 transport |
| 新增与语言资料无关的第 4 个垂直工作流 | 先把现有 3 个做深；P5 只允许补与语料库 / ELAN / FLEx 互通直接相关的工作流 |

## 十、补充治理：统一体验契约与边界复审机制

### 10.1 跨 workflow 统一体验契约（UX Contract）

目的：避免 `segment_qa`、`annotation_qa`、`lexeme_candidates`、`elan_flex_compatibility` 各自演进后出现交互割裂，影响用户学习成本与信任感。

**统一契约（所有垂直 workflow 必须满足）**

1. **Source Scope 可见**
   - 每条结果必须显示本次 `sourceScopeSummary`（最小字段：scopeType、对象数量、关键上下文 id）。
2. **Evidence 口径一致**
   - 统一由 `EvidencePacket` 驱动 citation 展示与跳转；不允许 workflow 私有证据字段绕过主口径。
3. **低置信与无证据状态一致**
   - `lowConfidence`、`noEvidence`、`reflectionFailed` 三类状态使用统一文案结构：
     - 发生了什么
     - 为什么
     - 下一步可执行动作
4. **写入门禁一致**
   - 所有写相关动作统一经过 `propose_changes -> confirm -> audit`，禁止任何 workflow 例外直写。
5. **结果动作一致**
   - 结果区动作统一支持子集：`copy`、`jumpToEvidence`，候选型结果额外支持 `accept/ignore`。
6. **失败可解释性一致**
   - 统一暴露 `fallbackReason` / `toolDeniedReason` / `confirmRequiredReason`（至少其一），避免“只显示失败状态码”。

**建议落地文件**
- `src/ai/chat/workflowExplainability.ts`（统一 explainability DTO）
- `src/components/ai/AiChatAssistantMessage.tsx`（统一渲染协议）
- `src/components/ai/AiAdoptionQueuePanel.tsx`（统一动作协议）
- `docs/adr/0030-vertical-workflow-template-contract.md`（契约文档）

**契约 → 阶段映射**

| 契约 | 主要落地阶段 | 说明 |
|------|------------|------|
| Source Scope 可见 | P1（最小可解释面）、P5（完整化） | P1 做 **`sourceScopeSummary`** 可见；P5 做 source set 保存/复用/切换 |
| Evidence 口径一致 | P1（绑定契约） | `EvidencePacketV0` 与 `SourceRef` 类型统一 |
| 低置信与无证据状态一致 | P1（fallback UX）、P4（annotation/lexeme 复制） | P1 在 segment_qa 打样；P4 复制到 annotation_qa / lexeme_candidates |
| 写入门禁一致 | 已有（T3/T4）、P4（AdoptionQueue 复用） | `propose_changes -> confirm -> audit` 路径不变；P4 AdoptionQueue 的 `accept` 必须复用该路径 |
| 结果动作一致 | P1（copy/jump）、P4（accept/ignore） | P1 实现 copy/jumpToEvidence；P4 增加 accept/ignore |
| 失败可解释性一致 | P1（reflection 面板）、P3（explainability DTO） | P1 在 UI 展示 reflection 结果；P3 统一 `workflowExplainability.ts` DTO |

**验收标准**
- [ ] 4 类 workflow 使用同一 explainability DTO（字段名称与语义一致）。
- [ ] 低置信/无证据/反射失败三类状态在 UI 上具备一致结构与交互。
- [ ] 任一 workflow 不再定义绕过 `EvidencePacket` 的私有 citation 通路。
- [ ] 候选采纳类 workflow 全部复用统一动作协议与审计字段。

### 10.2 边界复审触发器（Boundary Revisit Triggers）

目的：当前方案明确“不做多租户控制平面/通用 swarm/重型协作平台”，但应定义何时复审，防止边界长期僵化或被临时需求击穿。

**触发器（满足任一即进入复审）**

1. **协作复杂度触发**
   - 同一项目出现稳定多角色审阅诉求（例如 >2 角色、持续 4 周以上）且 AdoptionQueue 冲突率上升。
2. **规模触发**
   - workflow 周运行量持续增长并导致人工审阅瓶颈（例如 pending 候选积压显著、处理时延连续上升）。
3. **治理触发**
   - release evidence 连续两个周期出现同类缺陷性 skipped 或关键质量指标持续劣化。
4. **生态触发**
   - ELAN/FLEx 往返出现高频冲突（导出后回流修订无法稳定对齐）并影响主流程可用性。
5. **产品目标触发**
   - roadmap 明确新增“跨团队/跨项目治理”目标，超出当前本地优先单项目边界。

**复审输出（必须产出）**
- 边界是否维持/放宽/分层引入（结论）
- 影响范围（架构、数据、权限、审计、成本）
- 最小可行增量（不做大爆炸重构）
- 风险与回滚策略
- 对应 ADR 或执行计划更新

**执行机制**
- 建议在每个里程碑结束时增加一次“边界复审”检查项（10-15 分钟快速审议）。
- 触发后必须在 `docs/adr/` 或 `docs/execution/plans/` 留痕，不接受口头决策。

## 十一、交付风险排序（必须优先执行的 5 个 PR）

排序口径：按“失败成本”而非开发难度。失败成本 = 对可信性/数据边界/可回滚性的破坏程度 × 影响范围 × 修复代价。

### 11.1 Top 5（高到低）

1. **PR-P0-2：MCP tools 门面与 scope 门禁**
   - **为何优先**：对外工具必须 **可证明** 绑定真实宿主上下文；空 scope 硬失败避免信任级「静默全库读」。
   - **延误后果**：外部调用链看似可用但结果失真；后续 eval 与 runtime evidence 失去真实性基础。
   - **前置依赖**：PR-P0-1。

2. **PR-P1-1：source scope 最小可解释面 + evidence 绑定契约**
   - **为何高优先**：这是“证据型智能体”最核心的产品契约；无此项则引用可信性与来源限定无法被用户验证。
   - **延误后果**：用户无法判断 AI 到底读了什么；citation、audit、runtime report 口径漂移。
   - **前置依赖**：无硬依赖（可并行于 P0-1，与查询门面共享 scope 概念）。

3. **PR-P4-3：AdoptionQueue MVP**
   - **为何高优先**：人在环采纳是“建议变结果”的主通路；无此项将导致 annotation/lexeme 输出停留在文本层，难复盘。
   - **延误后果**：写入门禁虽存在，但产品端无法形成可审阅闭环，采纳率与忽略原因不可统计。
   - **前置依赖**：PR-P1-1、PR-P4-1、PR-P4-2。

4. **PR-P3-1：release evidence 缺陷 skip 收敛（第一批）**
   - **为何高优先**：缺陷性 skipped 不清零，意味着“看起来有治理，实际上观测断链”。
   - **延误后果**：上线前无法证明关键路径可观测；异常定位成本高且容易误判“功能稳定”。
   - **前置依赖**：P0/P1 基础审计字段落地。

5. **PR-P5-2：elan_flex_compatibility workflow**
   - **为何高优先**：这是语言资料垂直定位的差异化能力；缺失会导致产品与通用 AI 助手同质化。
   - **延误后果**：ELAN/FLEx 互通价值无法兑现，用户高频痛点仍靠手工流程。
   - **前置依赖**：PR-P1-1、PR-P4-3（建议依赖）。

### 11.2 延误容忍度分层

- **不可延误（阻断级）**：PR-P0-2、PR-P1-1  
- **可短延（1 个里程碑内）**：PR-P4-3、PR-P3-1  
- **可中延（2 个里程碑内）**：PR-P5-2（前提是 P1/P4 主闭环已跑通）

### 11.3 资源冲突时的取舍规则

当同一里程碑资源不足时，按以下顺序裁剪，不反向：

1. 先保“真实数据与证据契约”（P0-2、P1-1）
2. 再保“人在环闭环与可观测证明”（P4-3、P3-1）
3. 最后扩“垂直差异化能力”（P5-2）

> 禁止为追求“workflow 数量”而压缩 P0-2 / P1-1 的验收深度；这会在后续阶段放大返工成本。
