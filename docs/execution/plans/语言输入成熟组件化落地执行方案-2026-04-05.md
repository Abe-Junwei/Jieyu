# 执行：语言输入成熟组件化落地执行方案 - 2026-04-05

## 1. 执行目标

基于 [docs/规划-语言输入成熟组件化状态模型方案-2026-04-05.md](../../规划-语言输入成熟组件化状态模型方案-2026-04-05.md)，将现有语言名称 / ISO 输入从“多 helper 协作”升级为“显式 reducer + committed value 驱动”的实现。

本执行方案只解决语言输入组件化收敛，不顺带改动语言目录数据源、持久化契约或宿主业务流程。

本轮落地后的最小结果应满足：

1. 名称输入、代码输入、候选选择、失焦提交都由统一 reducer 驱动。
2. 宿主组件不再自己推测“名称和代码哪个是准值”，而是消费统一 committed value。
3. 两位短码延迟提交、标签拆分、旧值失效这些边界行为由 reducer 明确建模。
4. 组件、工具层、页面层都有回归覆盖。

补充状态：以上目标均已完成，本文档现作为执行记录保留。

#### 具体改动

1. 新增 [src/utils/languageInputReducer.ts](../../../src/utils/languageInputReducer.ts)
   - 定义：
     - `LanguageInputModel`
     - `LanguageInputEvent`
     - `createLanguageInputModel(...)`
     - `reduceLanguageInput(...)`
     - `selectCommittedValue(...)`

2. 新增 [src/utils/languageInputReducer.test.ts](../../../src/utils/languageInputReducer.test.ts)
   - 先覆盖核心状态转移：
     - 名称输入导致旧 code 失效
     - 唯一名称匹配进入 selected
     - 两位短码进入 deferred-code
     - blur 后 deferred-code 规范化
     - 非法代码进入 invalid
     - external-sync 不覆盖本地草稿
3. 保留 [src/utils/langMapping.ts](../../../src/utils/langMapping.ts) 作为纯解析层
   - 不再承载“谁来改 committed”这类状态职责

#### 完成标准

满足以下任意场景都由 reducer 测试直接覆盖，而不是只靠组件测试间接覆盖：

1. `English/eng` 改输 `P` 时 committed 清空。
2. `P` 继续输成 `Portuguese` 时 committed 变成 `por`。
3. `en` 保持 deferred-code，blur 后变成 `eng`。
4. `zh-Hant-HK` 能进入 selected，并带出 locale/script/region。

### 批次 B：组件改为 reducer 驱动

#### 目标

让 [src/components/LanguageIsoInput.tsx](../../../src/components/LanguageIsoInput.tsx) 从“自己拼事件逻辑”变成“只负责 UI + 发事件 + 取 selector”。

#### 具体改动

1. 组件内部新增 model state：
   - `const [model, dispatch] = useReducer(...)`
2. 删除或收缩当前这些逻辑的直写职责：
   - `codeDraft`
   - `lastCommittedCodeRef`
   - 名称 change 直接调用 state helper
   - 代码 change / blur 直接改值
3. 组件只做：
   - `nameChanged`
   - `codeChanged`
   - `nameSuggestionHovered`
   - `nameSuggestionCommitted`
   - `codeBlurred`
4. 保留现有 suggestion UI 结构，但数据全部来自 reducer selector
5. 对外 `onChange` 只在 committed value 变化时通知宿主

#### 完成标准

1. 输入名称时，名称框始终显示草稿。
2. 只有明确选中候选或唯一明确匹配时，宿主收到 committed 更新。
3. 两位短码输入中，不会被宿主回填打断。
4. 键盘和鼠标选候选行为一致。

建议不要一次性大改，按下面 4 个提交切：

1. `feat(language-input): add reducer model and reducer tests`
2. `refactor(language-input): drive LanguageIsoInput from reducer`

## 5. 风险与防守点

### 风险 1：宿主同步重新覆盖草稿

防守：

- 所有外部回填统一经 `externalValueSynced`。
- reducer 测试单独覆盖“外部回填不覆盖本地 deferred draft”。

### 风险 2：名称框自动匹配过早提交

防守：

- 明确只有唯一明确匹配或用户确认候选时才写 committed。
- 组件测试覆盖“模糊名称只出候选，不直接提交”。

### 风险 3：代码框两位短码语义回退

防守：

- reducer 保留 `deferred-code` 状态。
- 页面级测试覆盖 `en -> blur -> eng` 与 `en` 编辑中不改名。

### 风险 4：宿主层测试大量旧选择器失效

防守：

- 保持共享输入的 accessible name 不变。
- 避免在同一批同时改业务语义和文案标签。
