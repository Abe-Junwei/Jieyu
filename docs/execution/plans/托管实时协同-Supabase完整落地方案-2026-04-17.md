# 托管实时协同 Supabase 完整落地方案

日期：2026-04-17  
适用范围：解语当前 React + TypeScript + Dexie 本地优先架构  
目标：在不自建和长期运维传统后端的前提下，为项目增加正式可扩展的多人协同能力。

---

## 1. 结论与总原则

本方案采用：

- 前端继续保留本地优先架构（Dexie / IndexedDB）
- Supabase 作为托管协同中枢（Auth + Postgres + Realtime + Storage）
- 以“快照 + 增量变更 + 在线状态”三层同步模型实现多人协同

核心原则：

1. 本地库仍然是工作副本，不直接废弃。
2. 云端不直接镜像整个浏览器数据库，而是维护项目真源、快照和变更日志。
3. 同步协议必须显式版本化，避免未来 schema 演进时互相写坏数据。
4. 现有 M11-M13 协作运行时作为冲突与一致性内核继续复用。

---

## 2. 当前代码基线与可复用资产

当前仓库已经具备以下基础：

1. 本地数据库引擎与迁移链：
   - src/db/engine.ts
   - src/db/io.ts

2. 跨设备冲突与摘要一致性逻辑：
   - src/collaboration/collaborationCrossDeviceRuntime.ts
   - src/collaboration/collaborationMultiReplicaRuntime.ts
   - src/collaboration/collaborationTransactionSyncRuntime.ts

3. 快照导入导出与项目恢复能力：
   - exportDatabaseAsJson
   - importDatabaseFromJson

因此本次落地不需要推翻现有本地模型，而是新增一个“云同步桥接层”。

---

## 3. 目标边界

### 3.1 本期目标

1. 同一项目支持多人在线进入。
2. 项目成员、权限、资源与版本可托管管理。
3. 文本、层、语段、备注等结构化数据支持实时同步。
4. 新设备打开项目时可自动恢复到最近一致状态。
5. 出现冲突时可自动合并或进入人工审查。

### 3.2 明确非目标

1. 首期不做逐字级 CRDT 富文本体验。
2. 首期不做音频流媒体实时协同编辑。
3. 首期不做跨项目组织级复杂工作流。
4. 首期不做离线数周后无限制自动收敛；需要定义最大漂移窗口与人工介入策略。

---

## 4. 总体架构

```text
用户操作
  ↓
页面控制器 / 现有服务层
  ↓
本地 Dexie 写入（即时反馈）
  ↓
Collaboration Sync Bridge
  ├─ 出站：编码为变更事件写入 Supabase
  ├─ 入站：订阅 Realtime 接收远端变更
  ├─ 合并：复用现有 M11-M13 冲突治理逻辑
  └─ 恢复：快照回放 + 增量补齐
  ↓
Supabase
  ├─ Auth
  ├─ Postgres
  ├─ Realtime
  └─ Storage
```

设计重点：

- 写操作先落本地，保证编辑手感与离线可用。
- 云端只承担协同真源、权限控制、在线广播和资产托管。
- 远端变更回放到本地时必须经过统一协议，不允许组件私自直接写云端。

## 5. 云端数据模型

### 5.1 核心表

#### A. projects

用途：项目主记录与顶层治理。

建议字段：

- id
- name
- owner_id
- visibility
- protocol_version
- app_min_version
- latest_snapshot_id
- latest_revision
- created_at
- updated_at
- archived_at

#### B. project_members

用途：成员关系与权限。

建议字段：

- project_id
- user_id
- role
- invited_by
- joined_at
- disabled_at

角色建议：

- owner
- editor
- commenter
- viewer

#### C. project_snapshots

用途：项目一致性恢复点与回滚基线。

建议字段：

- id
- project_id
- version
- schema_version
- created_by
- snapshot_storage_bucket
- snapshot_storage_path
- checksum
- size_bytes
- change_cursor
- created_at
- note

说明：

- 快照正文默认进入 Storage，而不是直接把整包 JSON 长期塞进 Postgres 大字段。
- 数据库只保留元数据、校验值、体积与游标，便于备份、迁移和归档。
- change_cursor 用于标记该快照覆盖到了哪一条变更日志。

#### D. project_changes

用途：增量协同事件总线。

建议字段：

- id
- project_id
- actor_id
- client_id
- client_op_id
- session_id
- protocol_version
- project_revision
- base_revision
- entity_type
- entity_id
- op_type
- payload
- payload_ref_path
- vector_clock
- source_kind
- created_at

说明：

- 这是实时协同的核心表。
- `client_op_id` 用于幂等去重，建议在云端建立 `UNIQUE(project_id, client_op_id)`。
- `project_revision` 作为项目级单调顺序字段，用于稳定回放与补拉。
- 大 payload 不直接塞进 Realtime 消息体，超限内容改走 Storage 引用。

#### E. project_presence

用途：在线状态与“谁在编辑”。

建议字段：

- project_id
- user_id
- display_name
- state
- focused_entity_type
- focused_entity_id
- cursor_payload
- last_seen_at

说明：

- 高频在线状态、焦点与轻量光标信息优先走 Supabase Realtime Presence。
- 数据表只承担低频持久状态，例如最后在线时间、最近关注对象与审计摘要，避免高频心跳写放大。

#### F. project_assets

用途：音频、导出包、附件的元数据索引。

建议字段：

- id
- project_id
- asset_type
- storage_bucket
- storage_path
- mime_type
- size_bytes
- checksum
- uploaded_by
- created_at

实际文件进入 Supabase Storage，数据库只存索引与校验信息。

## 6. 同步协议设计

### 6.1 统一事件模型

所有同步变更统一收敛为 change event：

- protocolVersion
- projectId
- clientId
- clientOpId
- actorId
- entityType
- entityId
- operation
- payload
- payloadRefPath
- vectorClock
- baseRevision
- projectRevision
- sourceKind
- createdAt

### 6.2 操作类型建议

第一批只支持以下操作：

1. upsert_text
2. upsert_layer
3. upsert_unit
4. upsert_unit_content
5. upsert_relation
6. delete_entity
7. batch_patch
8. asset_attached
9. comment_added

这样可以直接映射现有 Layer / Unit / Relation 模型，避免一开始做过细粒度事件爆炸。

### 6.3 顺序、幂等与乱序治理

建议把以下规则写成协议硬约束：

1. 每个本地操作必须生成唯一 `client_op_id`。
2. 云端以 `UNIQUE(project_id, client_op_id)` 去重，允许安全重试。
3. 项目内必须存在单调递增的 `project_revision`，供客户端稳定回放。
4. 事件若 `base_revision` 落后过多，客户端不得盲目硬套本地合并，应先补拉缺口或回退到快照重基。
5. 普通回放按 `project_revision` 顺序应用，不以 `created_at` 作为唯一真序。

### 6.4 现有 M11-M13 runtime 的使用边界

现有协作运行时继续复用，但边界必须明确：

- **常规情况**：按 `project_revision` 顺序回放日志，不额外触发复杂合并。
- **并发或漂移场景**：出现版本向量并发、删除/更新冲突、时钟漂移超预算时，再调用现有 M11-M13 runtime。
- **高风险场景**：当缺口过大、事件乱序严重或运行时要求回滚时，直接退回最近稳定快照并重基，而不是继续自动写入。

### 6.5 协议版本治理

建议固定协议字段：

- protocol_version：协同协议版本
- schema_version：数据快照版本
- app_min_version：最小允许写入版本

老客户端如果版本低于最低要求，只允许只读进入，不允许写共享状态。

**实现进展（客户端）**：`useTranscriptionCollaborationBridge` 在启动桥接前读取 `projects.protocol_version` 与 `projects.app_min_version`，经 `evaluateCollaborationProtocolGuard`（`src/collaboration/cloud/collaborationProtocolGuard.ts`）判定后：不满足时抑制 `project_changes` 出站、`enqueueMutation` 及资产/快照等云端写；应用版本取自 `getMetricVersionTag()`（与 `__APP_VERSION__` / `VITE_APP_VERSION` 对齐）。

### 6.6 payload 与 Realtime 约束

1. 单条 change 的 payload 大小应设置上限。
2. 大 patch、附件、长文本包与整包快照默认走 Storage 引用，不直接塞进 Realtime。
3. Realtime 只承担事件广播与 Presence，不承担长期对象存储。

## 7. 前端代码落位方案

建议新增如下模块，不把云协同逻辑直接塞进页面组件：

### 7.1 新增目录

```text
src/
  collaboration/
    cloud/
      CollaborationSyncBridge.ts
      CollaborationOutboundQueue.ts
      CollaborationInboundApplier.ts
      CollaborationPresenceService.ts
      CollaborationSnapshotService.ts
      ProjectChangeCodec.ts
      syncTypes.ts
  integrations/
    supabase/
      client.ts
      auth.ts
      storage.ts
```

### 7.2 各文件职责

#### CollaborationSyncBridge.ts

- 项目打开时初始化同步
- 建立 Realtime 订阅
- 协调快照恢复、增量回放、在线状态心跳

#### CollaborationOutboundQueue.ts

- 接收本地变更
- 去抖、分批、重试
- 保证断网后重新发送
- 对 `source_kind=sync|migration` 的写入做忽略或旁路，防止回声重复出站

#### CollaborationInboundApplier.ts

- 应用远端 change event 到本地 Dexie
- 调用现有冲突与一致性运行时
- 对高风险冲突生成人工审查记录
- 在本地写入时打上同步世代或来源标记，避免再次触发 outbound

#### CollaborationSnapshotService.ts

- 拉取最近快照
- 从本地导出快照并定时上传
- 执行脱敏、压缩、校验值生成与 `change_cursor` 对齐
- 管理恢复点和回滚入口

#### CollaborationPresenceService.ts

- 优先接管 Realtime Presence 的高频在线状态
- 低频持久状态按需落表
- 广播当前关注实体
- 维护协作者列表、编辑提示

#### ProjectChangeCodec.ts

- 负责本地业务对象与 change event 的双向编解码
- 防止页面层直接拼 payload

### 7.3 现有代码的最小改动入口

优先从以下位置接入：

- 项目加载入口与页面 orchestrator
- 领域写服务统一出口
- 导入导出服务

原则：

- 页面层不直接请求 Supabase。
- 只有服务层和协同桥接层能访问云端 SDK。

## 8. Supabase 安全与权限设计

### 8.1 Auth

首期使用最简单模式：

- 邮箱魔法链接登录
- Google 登录可后补

### 8.2 RLS 基本规则

1. 只有成员才能读项目。
2. owner / editor 可写结构化协同变更。
3. commenter 若参与评论协同，只允许写 `comment_added` 等被白名单允许的操作类型；不得修改核心结构化实体。
4. viewer 只能读，不可写。
5. `project_changes` 的 INSERT 策略建议按 `role + op_type` 细分，而不是单一“能写 / 不能写”。

### 8.3 Bucket 规划

建议至少三个 bucket：

- project-audio
- project-exports
- project-attachments

路径规范：

- project-audio/{projectId}/{assetId}/...
- project-exports/{projectId}/{snapshotId}.json
- project-attachments/{projectId}/{assetId}/...

### 8.4 Storage Policy 与上传治理

1. Storage 的访问控制要与表 RLS 一起设计，不能只配数据库表权限。
2. 上传时应校验：
   - 最大文件大小
   - 允许的 mime type
   - 所属项目路径前缀
3. 对外下载优先使用签名 URL 或受控公开策略，不建议裸暴露桶路径。
4. 后续可预留病毒扫描、媒体转码或校验任务位。

## 9. 具体实施分期

### Phase 0：设计冻结与准备（1 周）

目标：把协议与表结构定死，避免开发中反复改模型。

任务：

1. 确认云端表结构与字段命名。
2. 确认角色模型、RLS 与 Storage policy 原则。
3. 确认 change event 枚举、payload 上限、幂等键与顺序字段规范。
4. 定义最小支持的协同实体范围。
5. 评审“本地字段 → 云端事件”的映射关系。
6. 明确快照脱敏规则、压缩方式与 `change_cursor` 提交顺序。
7. 明确回声抑制与日志压缩归档策略。

输出：

- SQL + RLS 初稿
- syncTypes.ts 草案
- Storage policy 草案
- 风险清单

### Phase 1：基础设施接入（1 周）

目标：项目可以登录、识别成员、拿到项目列表。

任务：

1. 接入 Supabase SDK。
2. 新建 client.ts / auth.ts。
3. 完成登录、退出、当前用户读取。
4. 完成项目列表页或项目上下文初始化。
5. 实现 members 读取与角色判断。

验收标准：

- 登录成功后可拉到本人项目。
- viewer / editor / owner 的只读和可写行为区分明确。

### Phase 2：快照层（1 到 2 周）

目标：任何设备首次进入项目时都能恢复到可编辑状态。

任务：

1. 复用 exportDatabaseAsJson 生成规范化项目快照。
2. 实现 cloud snapshot 上传与下载，正文默认进 Storage。
3. 定义定期快照策略：
   - 大操作后
   - 间隔保存
   - 手动生成版本点
4. 明确快照脱敏清单与校验值生成。
5. 新设备进入时先拉快照恢复本地库，并按 `change_cursor` 补拉缺口。

验收标准：

- 新设备首次进入项目恢复成功。
- 可从最近快照回滚到稳定版本。
- 快照恢复后不会出现重复回放或遗漏回放。

### Phase 3：增量同步层（2 周）

目标：把本地结构化编辑转成远端 change event，并能被其他客户端稳定回放。

任务：

1. 建立 ProjectChangeCodec。
2. 接入统一出站队列。
3. 为每条变更生成 `client_op_id`、`base_revision` 与 `source_kind`。
4. 对以下实体实现事件编码：
   - text
   - layer
   - layer_unit
   - layer_unit_content
   - unit_relation
5. 实现 project_changes 的补拉或订阅回放逻辑。
6. 远端回放后写回 Dexie，并抑制同步回声。

验收标准：

- 两个浏览器窗口下结构化编辑可同步。
- 断网重连后可补发未发送事件。
- 重试不会造成重复落地。

### Phase 4：在线状态与协作体验（1 周）

目标：在线用户能看到彼此状态和轻量协作提示。

任务：

1. 接入 Supabase Realtime Presence。
2. 对低频持久 presence 做按需落表。
3. 展示在线成员列表。
4. 展示“某用户正在编辑某对象”的轻提示。

验收标准：

- 在线状态断开后在合理时间内清理。
- 焦点提示稳定，不因高频心跳导致明显写放大。

### Phase 5：冲突治理与人工审查（1 到 2 周）

目标：出现写冲突时不默默覆盖。

任务：

1. 复用 M11-M13 runtime 处理向量时钟与 drift。
2. 自动合并低风险冲突。
3. 对 delete-update / concurrent-vector-clock 触发人工审查。
4. 为高风险冲突提供：
   - 暂存
   - 对比
   - 回滚
   - 重新应用

验收标准：

- 常见低风险冲突自动收敛。
- 高风险冲突有清晰 UI 和可恢复路径。

### Phase 6：资产托管与版本历史（1 周）

目标：项目附件和历史版本也进入正式治理。

任务：

1. 音频与附件进入 Storage。
2. project_assets 元数据和校验值落表。
3. 提供快照历史列表与恢复入口。
4. 提供基础审计日志与变更时间线。

验收标准：

- 附件在新设备可复用。
- 用户可查看和恢复历史快照。

## 11. 风险清单与预案

### 风险 1：本地模型过大，直接快照过重

预案：

- 快照仅作恢复点，不作为每次实时同步载体。
- 高频更新只发送增量 change。

### 风险 2：老客户端写坏共享状态

预案：

- 增加 protocol_version 和 app_min_version 守卫。
- 老版本只允许只读进入。

### 风险 3：二进制音频体积大、同步慢

预案：

- 音频不走 Realtime，统一走 Storage。
- 实时层只同步元数据与引用。

### 风险 4：页面层各处都有写逻辑，难统一接管

预案：

- 先梳理“统一写出口”，把协同挂在服务层。
- 避免在 JSX 组件里直接发云请求。

### 风险 5：冲突过多导致体验差

预案：

- 先缩小首批协同对象范围。
- 对高风险操作加轻量锁或编辑占用提示。

### 风险 6：入站回放触发再次出站，形成同步回声

预案：

- 为本地写入增加 `source_kind` 或同步世代标记。
- 出站队列忽略来自 sync / migration 的派生写入。

### 风险 7：变更日志无限增长，回放成本持续升高

预案：

- 在稳定快照覆盖后定期做 change squash 与归档。
- 对超长历史优先恢复快照，再补拉近窗口增量。

- 多客户端模拟同步一致率

## 13. 建议的首批开发顺序

按以下顺序推进，风险最低：

1. 先做 Supabase 表与 RLS
2. 再做 Auth 和项目成员
3. 再做快照同步
4. 再做 change event 编解码
5. 再做 Realtime 与 presence
6. 最后做冲突 UI 与历史版本

不要反过来从“实时广播”先做，否则后续协议和安全边界容易返工。

## 14. 里程碑交付定义

### M1：云端项目可见
- 用户可登录并看到自己的项目与角色

### M2：新设备可恢复
- 通过快照可以在另一台设备打开同一项目

### M3：双端实时同步可用
- 两端对结构化数据编辑可在秒级同步

### M4：冲突与回滚闭环
- 冲突可自动治理或人工审查，不出现静默丢数据

### M5：版本与资产治理闭环
- 成员、历史、附件、审计进入稳定态

## 15. 最终建议

对当前仓库，最佳实现路线不是“把 Dexie 换成云数据库”，而是：

- 保留 Dexie 的本地编辑体验与离线韧性
- 用 Supabase 承担协同真源、成员权限、在线广播和资产托管
- 用现有 M11-M13 协作 runtime 作为冲突合并内核

这条路线对当前代码改动最可控，也最容易演进到正式产品。
