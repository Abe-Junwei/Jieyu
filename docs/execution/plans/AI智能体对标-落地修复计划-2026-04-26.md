# AI 智能体工业级对标 — 落地修复计划（2026-04-26）

本文档承接《AI智能体工业级对标改进计划书》与 2026-04-25/26 落地审查结论，将已证实的**缺口**整理为**可排期、可验收**的修复项。优先级按用户风险与可逆性排列。

## 1. 目标与成功标准

| 阶段 | 目标 | 成功标准（验收） |
|------|------|------------------|
| **A — 指令治理正确性** | 停用/控制台操作与 `responsePreferences` / `toolPreferences` / `safetyPreferences` / `terminologyPreferences` 及 prompt 一致 | Vitest 覆盖：仅 ledger、仅 preferences、session+长期偏好；停用后 `buildUserDirectivePrompt` 与 tool policy 读路径与预期一致 |
| **B — 多步写操作语义** | `propose_changes` 失败路径可解释；在具备快照时**可逆序回滚**已成功子步；整批成功后才标记父级已执行 | 单测覆盖「部分子调用成功后失败」与 rollback/审计；用户可见进度与回滚结果；`markExecutedRequestId` 仅整批成功 |
| **C — 评测与门禁** | agent-evals 从「元数据加总」演进为可验证的治理信号 | 报告或 CI 中至少一类**基于导出/审计的断言**（非仅 exit code）；`goldenTaskCount` 语义在 README 中写清或改名 |
| **D — 发布证据与 CI 口径** | 运维/发布对「以谁为准」无歧义；strict 下审批样本量有下限 | ADR-0024 固定 **core + governance** 双闸与顺序；strict 默认 `--min-approval-total=2`（非 strict 不传） |

**非目标（本计划不展开，另立史诗）：** 完整 durable orchestration（checkpoint/resume/handoff）、全链路工具沙箱、**任意工具**的生产级 inverse patch 自动回滚。**已采纳的 MVP 子集（B2）：** 仅 `propose_changes` 确认路径下，对白名单写工具（segment adapter 的 `set_transcription_text` / `set_translation_text` 等）在提供读模型回调时附带 per-step `rollback`，批失败时逆序调用；无快照时用户文案明确为「无可自动回滚快照」。

## 2. 工作项

### P0 / P1 — A：Directive 停用与偏好回滚

**问题摘要（已收口）：** 曾出现「仅 `directiveLedger` 接受的长期偏好、无 `sessionDirectives` 行时停用 no-op」以及停用后 preferences 未与 prompt 对齐的问题。当前 `useAiChat.deactivateSessionDirective` 在 **ledger `accepted` 或 session 命中** 时均调用 `sessionMemory.deactivateSessionDirective`；`stripPreferencesForDeactivatedEntry` 覆盖 `response` / `tool` / `safety` / `terminology`。回归见 `sessionMemory.test.ts` 与 `useAiChat.test.tsx` 中 **`deactivateSessionDirective`** / **`deactivateSessionDirective integration`**（含 **仅 ledger** 的 `responsePreferences` 与 **`toolPreferences.preferLocalReads`**）。

**实现引证：** `src/hooks/useAiChat.ts`，`src/ai/chat/sessionMemory.ts`，`buildUserDirectivePrompt`（`userDirectivePrompt.ts`）读归一化后的 `sessionMemory`。

### P1 — B：`propose_changes` 多步执行：失败语义与幂等

**问题摘要：** 子调用顺序执行，中间失败后 `appliedChildCount > 0` 即部分生效；`hasPersistedExecutionForRequest` 对多种 `confirm_failed` 仍可能视为已执行，与「重试同一次 propose」的期望易冲突。

**分档交付（建议两迭代）：**

1. **B1 — 产品语义与可观测性（已落地）：**
   - 部分失败时，助手消息/审计明确写出**已成功子调用数、失败子调用名与原因**（`executionProgress`）。
   - 子调用使用独立 `requestId`（`genRequestId`）；父级 `propose_changes` 的幂等与 `hasPersistedExecutionForRequest(parentCall.requestId)` 对齐（见 B2：`markExecutedRequestId` 仅在整批成功后调用，避免「部分成功却标父级已执行」阻断重试）。
2. **B2 — per-child 补偿（MVP，已落地）：**
   - `useAiToolCallHandler.segmentAdapter`：对 `set_transcription_text` / `set_translation_text` / `clear_translation_segment`（segment 与 unit 路径）在写前通过 `readSegmentLayerText` / `readUnitLayerText` 取快照，成功结果附带 `rollback`（再调对应 `save*` 恢复）。
   - `useAiToolCallHandler.annotationAdapters`（`tokenAdapter`）：`set_token_pos`（按 `tokenId` 与 **unitId+form** 批量）与 `set_token_gloss` 在具备 `readTokenPos` / `readTokenGloss` 与 `update*` 时附带 `rollback`；批量 POS 在 `ctx.units` 上按与 `LinguisticService.batchUpdateTokenPosByForm` 一致的 form 匹配收集快照后逆序恢复。
   - **`auto_gloss_unit`**（`glossAdapter`，`auto_gloss_segment` 经 `voiceAdapter` 复用同路径）：`AutoGlossMatch` 记录新建 **`token_lexeme_links.id`**（`linkId`）；在具备 `updateTokenGloss` 时 `rollback` 先 **`LinguisticService.removeTokenLexemeLinksByIds`**，再按匹配逆序对各 token 的各 gloss 语言键调用 `updateTokenGloss(..., null, lang)`（与「仅对原先无 gloss 的 token 赋 gloss」语义一致）。无 `updateTokenGloss` 时不挂 `rollback`（无法同步读模型）。
   - `useTranscriptionAiController`：注入读回调（`segmentContentByLayer`、`getUnitDocById` + `getUnitTextForLayer` / scope units；**以及** `readTokenPos` / `readTokenGloss` 扫 `timelineUnitViewIndex.allUnits` + `getUnitDocById`）。
   - `executeConfirmedProposedChangeBatch`：收集各子步 `rollback`，在 `invalid_child_args` / `child_failed` / `exception` 失败时**逆序**执行；用户文案追加「无可自动回滚快照 / 已回滚已应用的修改 / 回滚失败：…」；审计 `ToolDecisionAuditMetadata.proposeRollback`（`attempted` / `ok` / `errorCount`）。
   - **未纳入本 MVP：** 全工具 preview 单一 changeset、非文本类工具的补偿。

**主要涉及文件：** `src/hooks/useAiChat.confirmExecution.ts`，`src/hooks/useAiToolCallHandler.segmentAdapter.ts`，`src/hooks/useAiToolCallHandler.annotationAdapters.ts`，`src/hooks/useAiToolCallHandler.ts`，`src/hooks/useAiToolCallHandler.types.ts`，`src/pages/useTranscriptionAiController.ts`，`src/ai/chat/chatDomain.types.ts`（`AiChatToolResult.rollback`），`src/ai/chat/toolCallHelpers.ts`（`proposeRollback`），相关测试。

### P2 / 史诗 — F：Durable orchestration 与全链路沙箱

**问题摘要：** `TaskRunner` 过期标失败、agent loop checkpoint 偏续写；`aiBackgroundToolSandboxEnabled` 仍为 false。

**建议：** 单独开「Phase C+ 史诗」，本修复计划**仅**在 A/B 中避免与将来 checkpoint 模型冲突（数据结构预留可在 ADR 中记一笔）。

**2026-04-26 首切片（F0/F1，已落地）：**

1. **范围建档：** 基线文件限定为 `src/ai/tasks/TaskRunner.ts`、`src/ai/tasks/TaskRunner.test.ts`、`src/db/types.ts`、`src/db/schemas.ts`、`src/services/WorkspaceReadModelService.ts`；只读扩圈确认 `useAiChat.agentLoopRunner.ts` 与 `chatDomain.types.ts`，本轮不迁移 chat loop。
2. **持久编排元数据：** `AiTaskDoc` 增加 `attempt` / `maxAttempts` / `timeoutMs` / `startedAt` / `completedAt` / `lastHeartbeatAt` / `checkpointJson` / `resumable` / `handoffReason`；`TaskRunner` 在入队、尝试开始、heartbeat/checkpoint、终态写入这些字段。
3. **运行时契约：** `TaskRunContext` 暴露 `heartbeat(checkpoint?)` 与 `checkpoint(checkpoint)`，让长任务在不迁移为完整后台 worker 的前提下先具备可恢复状态记录；现阶段仍不承诺自动重启后继续执行闭包任务。
4. **读模型：** `ai_task_snapshots` 追加 `attempt` / `maxAttempts` / `lastHeartbeatAt` / `hasCheckpoint` / `resumable` / `handoffReason`，供后续 UI 与 release evidence 消费。

**后续 F 阶段：**

- **F2a runtime bridge（2026-04-27 已落地）：** `pendingAgentLoopCheckpoint` 增加可选 `taskId`；新增 `agent_loop` 类型 `ai_tasks` 行与 `agentLoopCheckpoint` 编解码/回填工具；`runAgentLoop` 在 token budget warning 时可持久化 checkpoint，并把返回的 `taskId` 写回 session memory。该切片只完成 **session memory ↔ ai_tasks.checkpointJson** 绑定，不承诺自动重启后继续执行。
- **F2b runtime integration（2026-04-27 最小续跑切片已落地）：** 当 `pendingAgentLoopCheckpoint.taskId` 存在时，`useAiChat` 在 resume 入口优先从 `ai_tasks.checkpointJson` 重新加载 checkpoint；成功消费后将对应 `ai_tasks` 行标记为 `done` 且 `resumable=false`。剩余 F2b 范围仍包括 cancel/retry UI/runtime 入口、任务面板 listing/selection，以及不依赖 session memory 的完整重启后恢复边界。
- **F3 handoff/observability（2026-04-27 最小可见切片已落地）：** 现有 AI 任务列表先展示 `resumable` / checkpoint / `handoffReason`，让 `agent_loop` token-budget handoff 在任务面板雏形中可见；发布证据包可通过 `--ai-task-snapshots` / `RELEASE_EVIDENCE_AI_TASK_SNAPSHOTS` 聚合 `ai_task_snapshots` 的 durable orchestration 指标（长任务完成率、人工介入率、checkpoint 恢复率）。真实导出入口已补 `npm run export:release-evidence:ai-task-snapshots`，并可用 `npm run gate:release-evidence:durable-orchestration` 串联生成证据包。剩余 F3 范围仍包括审批中心深度联动。
- **F4 sandbox 联动：** 与 `aiBackgroundToolSandboxEnabled` 独立推进，避免把 durable orchestration 与 runtime capability isolation 混成一次高风险迁移。

  - `useAiChat.deactivateSessionDirective`：在 `sessionDirectives` 或 `directiveLedger` 中存在对应 `accepted` 项时即可停用。
