# AI 智能体工业级对标 — 落地修复计划（2026-04-26）

本文档承接《AI智能体工业级对标改进计划书》与 2026-04-25/26 落地审查结论，将已证实的**缺口**整理为**可排期、可验收**的修复项。优先级按用户风险与可逆性排列。

## 1. 目标与成功标准

| 阶段 | 目标 | 成功标准（验收） |
|------|------|------------------|
| **A — 指令治理正确性** | 停用/控制台操作与 `responsePreferences` / `toolPreferences` / `safetyPreferences` / `terminologyPreferences` 及 prompt 一致 | Vitest 覆盖：仅 ledger、仅 preferences、session+长期偏好；停用后 `buildUserDirectivePrompt` 与 tool policy 读路径与预期一致 |
| **B — 多步写操作语义** | `propose_changes` 失败路径可解释；在具备快照时**可逆序回滚**已成功子步；整批成功后才标记父级已执行 | 单测覆盖「部分子调用成功后失败」与 rollback/审计；用户可见进度与回滚结果；`markExecutedRequestId` 仅整批成功 |
| **C — 评测与门禁** | agent-evals 从「元数据加总」演进为可验证的治理信号 | 报告或 CI 中至少一类**基于导出/审计的断言**（非仅 exit code）；`goldenTaskCount` 语义在 README 中写清或改名 |
| **D — 发布证据与 CI 口径** | 运维/发布对「以谁为准」无歧义；strict 下审批样本量有下限 | ADR-0024 固定 **core + governance** 双闸与顺序；strict 默认 `--min-approval-total=2`（非 strict 不传） |

**非目标（本计划不展开，另立史诗）：** 完整 durable orchestration（checkpoint/resume/handoff）、全链路工具沙箱、**任意工具**的生产级 inverse patch 自动回滚。**已采纳的 MVP 子集（B2）：** 仅 `propose_changes` 确认路径下，对白名单写工具（segment adapter 的 `set_transcription_text` / `set_translation_text` 等）在提供读模型回调时附带 per-step `rollback`，批失败时逆序调用；无快照时用户文案明确为「无可自动回滚快照」。

## 2. 工作项

### P0 / P1 — A：Directive 停用与偏好回滚

**问题摘要：** `deactivateSessionDirective`（`useAiChat.ts`）仅在 `sessionDirectives` 中存在同 `id` 时继续，长期偏好类指令常只写入 `directiveLedger` + preferences，导致**停用 no-op**；`sessionMemory.deactivateSessionDirective` 将 ledger 标为 `superseded` 但**不撤销**已写入的 preferences，而 `buildUserDirectivePrompt` 仍直接读 preferences（`userDirectivePrompt.ts`）。

**设计要点：**

1. **入口条件：** 若 `sessionDirectives` 中无该 `id`，但在 `directiveLedger` 中存在 `action === 'accepted'` 且 `id` 匹配，**仍应执行停用**（与控制台展示来源对齐）。
2. **副作用撤销：** 依据 ledger 条目的 `targetPath` / `value` / `category`（与 `applyUserDirectivesToSessionMemory` 对称）实现 `revertUserDirectiveFromMemory(memory, directiveId)` 或扩展现有 `deactivateSessionDirective`：
   - `response* / tool* / safety*`：按 `targetPath` 删除对应键或恢复为与「未设该条目前」的归一化一致（与 `normalizeResponsePreferences` 等一致，避免残留空对象语义问题）。
   - `terminology`：确认 `AiUserDirectiveLedgerEntry` 是否存够 `sourceTerm`/`targetTerm`；若不足，在 `createLedgerEntry` 中**补充**可逆字段（小范围类型扩展 + 归一化），再按源术语移除 `terminologyPreferences` 中对应项。
3. **session 类仅文本指令：** 从 `sessionDirectives` 移除 + ledger supersede 保持现有行为。
4. **测试：** 扩展 `sessionMemory.test.ts`、补充 hook 级或 `userDirectiveRegistry` 集成用例；覆盖「仅 ledger 可停用」「停用后 prompt 行消失」。

**主要涉及文件：** `src/hooks/useAiChat.ts`，`src/ai/chat/sessionMemory.ts`，`src/ai/memory/userDirectiveRegistry.ts`，`src/ai/chat/userDirectivePrompt.ts`（仅必要时），`src/ai/chat/chatDomain.types.ts`（若扩展 ledger）。

### P1 — B：`propose_changes` 多步执行：失败语义与幂等

**问题摘要：** 子调用顺序执行，中间失败后 `appliedChildCount > 0` 即部分生效；`hasPersistedExecutionForRequest` 对多种 `confirm_failed` 仍可能视为已执行，与「重试同一次 propose」的期望易冲突。

**分档交付（建议两迭代）：**

1. **B1 — 产品语义与可观测性（已落地）：**  
   - 部分失败时，助手消息/审计明确写出**已成功子调用数、失败子调用名与原因**（`executionProgress`）。  
   - 子调用使用独立 `requestId`（`genRequestId`）；父级 `propose_changes` 的幂等与 `hasPersistedExecutionForRequest(parentCall.requestId)` 对齐（见 B2：`markExecutedRequestId` 仅在整批成功后调用，避免「部分成功却标父级已执行」阻断重试）。  
2. **B2 — per-child 补偿（MVP，已落地）：**  
   - `useAiToolCallHandler.segmentAdapter`：对 `set_transcription_text` / `set_translation_text` 在写前通过 `readSegmentLayerText` / `readUnitLayerText` 取快照，成功结果附带 `rollback`（再调对应 `save*` 恢复）。  
   - `useTranscriptionAiController`：注入读回调（`segmentContentByLayer`、`getUnitDocById` + `getUnitTextForLayer` / scope units）。  
   - `executeConfirmedProposedChangeBatch`：收集各子步 `rollback`，在 `invalid_child_args` / `child_failed` / `exception` 失败时**逆序**执行；用户文案追加「无可自动回滚快照 / 已回滚已应用的修改 / 回滚失败：…」；审计 `ToolDecisionAuditMetadata.proposeRollback`（`attempted` / `ok` / `errorCount`）。  
   - **未纳入本 MVP：** 全工具 preview 单一 changeset、非文本类工具的补偿。  

**主要涉及文件：** `src/hooks/useAiChat.confirmExecution.ts`，`src/hooks/useAiToolCallHandler.segmentAdapter.ts`，`src/hooks/useAiToolCallHandler.ts`，`src/hooks/useAiToolCallHandler.types.ts`，`src/pages/useTranscriptionAiController.ts`，`src/ai/chat/chatDomain.types.ts`（`AiChatToolResult.rollback`），`src/ai/chat/toolCallHelpers.ts`（`proposeRollback`），相关测试。

### P1 — C：Agent evals：从标签覆盖到可验证断言

**问题摘要：** `run-agent-evals.mjs` 仅跑命令并汇总 `goldenTaskCount` / `trajectorySignals`（`suite.v1.json` 元数据），不解析真实工具/审计轨迹。

**建议步骤：**

1. **已落实**：`scripts/agent-evals/README.md` 写清 `goldenTaskCount` 为**权重/代表度**（非独立用例条数）、`trajectorySignals` 为声明式覆盖标签；将字段改名为 `weight` 仍为可选后续。
2. 新增 **shadow 或 opt-in 模式**（`--assert-audit-trace`）：对固定 fixture 跑一轮 `export:release-evidence:ai-audit` 或读 Dexie/导出 NDJSON 子集，断言最小字段集（如 `ai_tool_call_decision` 行数、某类 `newValue` 前缀）。
3. 将 1–2 条高价值断言**接入** `check:agent-evals` 的 enforce 模式（或独立 `npm run check:agent-evals:trace`）以免拖慢 PR。

**主要涉及文件：** `scripts/run-agent-evals.mjs`，`package.json` 脚本，可选新 fixture 与 `scripts/agent-evals/` 下小模块。

### P2 — D：CI 与「权威 gate」表述（已落地）

**问题摘要：** CI 中 `release-evidence:core` / `full` 与 `gate:release-evidence:governance` 为**组合顺序**，与「单一权威入口」的口头表述易冲突。

**行动（已做）：** ADR [`0024-release-evidence-core-governance-dual-gate.md`](../../adr/0024-release-evidence-core-governance-dual-gate.md) 固定 **core（enforce）** 与 **governance（dry-run + check）** 的职责、顺序与「两段均须通过」的发布语义；runbook [`release-evidence/README-2026-04-25.md`](../release-gates/release-evidence/README-2026-04-25.md) 增加双闸表与 ADR 链接。

**主要涉及文件：** `docs/adr/0024-release-evidence-core-governance-dual-gate.md`，`docs/adr/README.md`，`docs/execution/release-gates/release-evidence/README-2026-04-25.md`。

### P2 — E：Release evidence governance 样本量（已落地）

**问题摘要：** `check-release-evidence-governance.mjs` 对 `actionApprovalCenter.summary` 无最小 `total`；dry-run 报告可能仅有少量样本仍通过。

**行动（已做）：**

- `check-release-evidence-governance.mjs` 支持 **`--min-approval-total=<n>`**（`n>=1`）；`check:release-evidence:governance:strict` 默认 **`--min-approval-total=2`**；非 strict 不传参保持兼容。  
- `release-evidence-full-gate` 中 strict 分支的内联 `node scripts/check-release-evidence-governance.mjs ...` 同步追加 `--min-approval-total=2`（core 分支走 `npm run gate:release-evidence:governance:strict`，随 `package.json` 已覆盖）。  
- runbook 写明 **何时** workflow_dispatch 开 strict（与 E 合并于 README）。

**主要涉及文件：** `scripts/check-release-evidence-governance.mjs`，`scripts/check-release-evidence-governance.test.ts`，`package.json`，`.github/workflows/ci.yml`，`docs/execution/release-gates/release-evidence/README-2026-04-25.md`。

### P2 / 史诗 — F：Durable orchestration 与全链路沙箱

**问题摘要：** `TaskRunner` 过期标失败、agent loop checkpoint 偏续写；`aiBackgroundToolSandboxEnabled` 仍为 false。

**建议：** 单独开「Phase C+ 史诗」，本修复计划**仅**在 A/B 中避免与将来 checkpoint 模型冲突（数据结构预留可在 ADR 中记一笔）。

## 6. 实施记录

- **2026-04-26 — 阶段 A（核心）已落地**  
  - `useAiChat.deactivateSessionDirective`：在 `sessionDirectives` 或 `directiveLedger` 中存在对应 `accepted` 项时即可停用。  
  - `sessionMemory.deactivateSessionDirective`：对 `responsePreferences` / `toolPreferences` / `safetyPreferences` / `terminologyPreferences` 在「当前值仍与 accepted ledger 的 `value` 一致」时撤销；否则仅将 ledger 标为 `superseded`；并从 `pinnedDirectiveRefs` 去掉该 id。  
  - 测试：`src/ai/chat/sessionMemory.test.ts` 补充 ledger-only、术语、被覆盖值、pinned ref 用例。  
- **2026-04-26 — 阶段 B1 已落地**  
  - `executeConfirmedProposedChangeBatch` 在 `invalid_child_args` / `child_failed` / `exception` 失败路径写出 **已执行 x/y 项** 与失败子步骤原因（基础 `buildProposeChangesFailureDetail`）；用户可见文案与审计 `message` 对齐。  
  - `ToolDecisionAuditMetadata` 新增 `executionProgress`（`appliedCount` / `totalCount` / `partial`），用于区分“零执行失败”与“部分执行失败”，且保持原有 `newValue` reason code 不变。  
  - 测试：`src/hooks/useAiChat.test.tsx` 新增 `propose_changes` 两步执行中途失败用例（后与 B2 合并断言 rollback 元数据）。  
  - release evidence 汇总：`scripts/generate-release-evidence-bundle.mjs` 的 `actionApprovalCenter.summary` 新增 `partialFailureCount`，并在 key summary 中展示，支持 CI/报告直接观察“半致状态”趋势。  
  - 脚本测试：`scripts/generate-release-evidence-bundle.test.ts` 增加 `confirm_failed:propose_changes:child_failed` + `executionProgress.partial=true` 样本并校验统计结果。  
- **2026-04-26 — 阶段 B2（MVP）已落地**  
  - `AiChatToolResult` 可选 `rollback?: () => Promise<void>`；segment adapter 在具备 `readSegmentLayerText` / `readUnitLayerText` 与对应 `save*` 时为文本写成功结果附带 `rollback`。  
  - `executeConfirmedProposedChangeBatch`：逆序执行已收集的 `rollback`；失败文案由 `appendProposeRollbackStatus` 区分「无可自动回滚快照」「已回滚已应用的修改」「回滚失败：…」；审计写入 `proposeRollback`（`attempted` / `ok` / `errorCount`）。  
  - **`markExecutedRequestId(parentCall.requestId)` 仅在整批子调用全部成功后调用**，避免部分成功后仍标父请求已执行、与重试语义冲突。  
  - `useTranscriptionAiController` 注入 `readSegmentLayerText` / `readUnitLayerText`。  
  - 测试：`useAiChat.test.tsx` 中途失败用例为第一子步提供 `rollback` mock，断言逆序调用一次及 `metadata.proposeRollback`。  
- **2026-04-26 — 阶段 C（最小可用）已落地**  
  - `scripts/run-agent-evals.mjs` 新增可选参数 `--assert-audit-trace=<ndjson>`：解析真实 audit 导出，要求至少存在 `ai_tool_call_decision` 且 metadata 含 `phase=decision` 与非空 `outcome`。  
  - 报告新增 `summary.auditTracePassed` 与 `auditTrace` 区块（行数、解析错误数、失败原因），并将该断言纳入 `thresholdPassed`。  
  - 新增脚本入口：`npm run check:agent-evals:trace`（基于 `docs/execution/audits/ai-tool-decision-audit-export-v1.ndjson`）。  
  - 测试：新增 `scripts/run-agent-evals.test.ts`，覆盖 trace 断言通过/失败两条路径。  
  - CI：`.github/workflows/ci.yml` 的 `workflow_dispatch` 增加 `run_agent_evals_audit_trace`；仅在手动勾选时于 `agent-evals-gate` 中追加 `npm run check:agent-evals:trace`，默认 PR/push 不跑以免受审计样本波动影响。  
  - 文档：`scripts/agent-evals/README.md` 说明 `goldenTaskCount` / `trajectorySignals` 语义与 `check:agent-evals:trace`、CI 手动入口。  
- **2026-04-26 — 阶段 D（文档）已落地**  
  - ADR `0024-release-evidence-core-governance-dual-gate.md`：`gate:release-evidence:core/full`（enforce 整包）与 `gate:release-evidence:governance`（dry-run + 策略脚本）职责划分、CI 顺序、**两段均须通过**。  
  - `docs/adr/README.md` 编入 ADR-0024；`docs/execution/release-gates/release-evidence/README-2026-04-25.md` 增加双闸表与 ADR 链接。  
- **2026-04-26 — 阶段 E（样本量门槛）已落地**  
  - `check-release-evidence-governance.mjs`：`--min-approval-total=<n>`（`n>=1`），校验 `actionApprovalCenter.summary.total >= n`。  
  - `check:release-evidence:governance:strict` 默认附带 `--min-approval-total=2`；`release-evidence-full-gate` 的 strict 内联命令同步。  
  - 测试：`scripts/check-release-evidence-governance.test.ts` 覆盖低于阈值失败路径。
