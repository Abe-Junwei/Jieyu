# 独立工作台开放门槛与复用边界（2026-03-30）

> 目的：把 Annotation / Analysis / Writing / Lexicon 四个独立工作台的开放条件从“口头约束”收敛为可执行的 go / no-go 规则，并明确哪些能力必须复用转写内核，哪些实现禁止直接复制。

## 一、当前结论

当前统一结论为：`NO-GO`。

原因：

1. 当前唯一稳定工作台仍是转写页。
2. 四个独立页面目前都只是说明页，不构成已交付功能证据。
3. 转写页主线虽然已恢复自动化绿线，但手工 CASE 01-10 与最终封账尚未完成。

## 三、允许复用的内核边界

以下能力可作为未来独立工作台的优先复用边界：

1. runtime contract 与 runtime props 装配：`src/pages/TranscriptionPage.runtimeContracts.ts`、`src/pages/TranscriptionPage.runtimeProps.ts`、`src/pages/useTranscriptionRuntimeProps.ts`
2. AI sidebar / assistant 装配边界：`src/pages/useTranscriptionAssistantSidebarController.ts`、`src/pages/useTranscriptionSidebarSectionsViewModel.ts`
3. 选择语义与目标解析：`src/pages/useTranscriptionSelectionSnapshot.ts`、`src/pages/transcriptionSelectionSnapshot.ts`、`src/pages/transcriptionUnitTargetResolver.ts`
4. speaker / waveform / shell 页面级 controller：`src/pages/useTranscriptionSpeakerController.ts`、`src/pages/useTranscriptionWaveformBridgeController.ts`、`src/pages/useTranscriptionShellController.ts`
5. AI / voice / citation 核心能力：`src/hooks/useAiChat.ts`、`src/hooks/useVoiceAgent.ts`、`src/pages/TranscriptionPage.citationJump.ts`

原则：先抽 controller/runtime 内核，再开放页面壳层；不允许先造新页面再回填内核。

## 四、禁止复制的页面级实现

以下内容不允许直接复制到新工作台：

1. `src/pages/TranscriptionPage.Orchestrator.tsx` 的 page-level 装配与 JSX 结构
2. 转写页内部临时胶水、局部 `useMemo` 拼装与仅服务于当前页面布局的局部状态
3. 任何为通过现有结构测试而写死的转写页专有渲染细节

原则：独立工作台若需要同类能力，应复用既有 controller/runtime，或先抽出新的共享边界，再接页面。

## 五、开放 go / no-go 清单

### 必须同时满足的 go 条件

1. 自动化基线通过：`npm run typecheck`、`npm run build`、`npm test`
2. 转写主工作台已完成当前批次封账：`docs/execution/archive/manual-validation/手工验收执行记录-AI语音RAG主流程-2026-03-30.md` 中 CASE 01-10 已回填并完成最终签署
3. architecture guard、structure tests、hotspot report 对相关核心文件无新增超线项
4. 新工作台具备明确的用户工作流定义，而不是仅有占位路由或说明文案
5. 新工作台至少复用一组已验证 controller/runtime 内核，不允许以复制 `TranscriptionPage.Orchestrator.tsx` 为起点
6. 新工作台有对应的聚焦自动化与最小手工验收脚本

### 任一命中即 `NO-GO`

1. 转写页主线仍存在未封账阻塞项
2. 新工作台仍依赖转写页页面级 props 拼装或复制 page-level 逻辑
3. 没有专属手工验收与聚焦自动化
4. 开放理由仅是“页面已经有路由/壳层”

## 七、解锁前的最小动作

当转写主线封账完成后，若要解锁任一工作台，按以下顺序推进：

1. 定义该工作台的最小用户工作流与数据入口
2. 从转写页抽出对应 controller/runtime 内核，禁止复制 page-level 代码
3. 为该工作台新增聚焦自动化与结构守卫
4. 编写独立手工验收脚本与执行记录模板
5. 再做一次 go / no-go 复核，结论写回路线图与 README

## 八、当前建议

在转写页手工封账未完成前，继续维持四个独立工作台为说明页；后续如需推进，先更新本文件的判定，再进入实际开放。
