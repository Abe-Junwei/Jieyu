# 语言资产左侧模态工作区视觉基线（2026-04-09）

## 范围

只覆盖三页语言资产工作区：

1. 正字法工作区
2. 语言元数据工作区
3. 正字法桥接工作区

不覆盖词典页、分析页、标注页、写作页，以及转写页内 AI / 语音 / 批量 / 搜索替换等重面板。

- 设计文件：[面板—极简.pen](./面板—极简.pen)

## 设计结论

### 1. 不再沿用当前三页的“页面工作台”视觉语义

三页进入左侧模态后，不应继续维持：

1. 页面 hero 区
2. 页面级双栏大布局
3. 页面式大面积留白

应改为“工作区卡片”语义：

1. 壳层更紧凑
2. 重点信息前置
3. 列表和详情统一落在单列滚动区内

### 2. 正字法工作区作为视觉母版

原因：

1. 当前它已经最接近 host + embedded panel 结构
2. 信息密度中等，最适合先校准视觉语言
3. 语言元数据和桥接都能在它的基础上换内容块，而不是重做壳层

### 3. 语言元数据必须从“页面双栏表单”收缩为“单列治理流”

重点不是把所有字段缩小，而是把编辑节奏重排为：

1. 当前对象摘要
2. 搜索与切换
3. 详情分组块顺序编辑

地图、自定义字段、分类路径不再主导整体版面宽度。

### 4. 桥接工作区应表现为“规则编辑器”而不是“第二个目录页”

视觉重点应放在：

1. 当前 target 上下文
2. source 选择
3. rule block
4. preview / validation

不再强调大目录浏览感。

## CSS 落地合同

### 1. 共同壳层

语言资产三页在代码里应统一抽出共享壳层，至少统一：

1. modal 外层容器宽度策略
2. 白色工作区卡片圆角、边框、阴影
3. 标题栏高度与关闭区布局
4. summary 卡片样式
5. 主滚动区和 footer 的分区方式

### 2. 布局原则

建议统一为：

1. 左侧固定宽度工作区卡片
2. 卡片内部为单列纵向布局
3. header、summary、footer 固定
4. 中间 body 为主滚动区
5. 各业务块使用统一 section 样式，而不是继续沿用页面级 layout class

### 3. 样式迁移原则

从页面到面板落地时应同步做：

1. page hero 相关样式下线或不再用于 modal 形态
2. page layout 样式改为 panel layout 样式
3. 把三页通用规则抽到共享语言资产样式文件
4. 仅把业务差异留在各自页面或面板样式文件中

### 4. 状态块统一

三页统一至少保留以下状态样式：

1. 当前对象摘要
2. 空态
3. 错误态
4. 未保存提醒
5. 主动作按钮

## 并入实施工作流

本视觉基线应并入语言资产三页左侧模态化工作流，顺序如下：

1. 统一 route-modal 宿主
2. 固定共同视觉母版
3. 用正字法工作区先落一版 CSS
4. 语言元数据接入同一壳层
5. 正字法桥接接入同一壳层
6. 统一补布局与交互回归

不要把视觉改造拖到三页都接入之后再做，否则会形成第二轮大返工。

## 下一步实施建议

1. 先从正字法工作区抽共享语言资产 modal shell
2. 把当前正字法页样式拆成共享壳层样式 + 正字法差异样式
3. 再让语言元数据和桥接复用同一壳层

## 展开态效果图补充

本次已在 [面板—极简.pen](./面板—极简.pen) 新增总板“语言资产-展开态方案-2026-04-09”，用于补足三页 section 的展开态设计。

### 1. 正字法工作区

#### A1 默认工作态

默认展开：

1. 当前对象摘要
2. 浏览 / 搜索
3. Identity
4. Render

默认折叠：

1. 语言名称与样例
2. Advanced catalog / 治理字段

原因：默认进入时先完成对象识别、检索切换与渲染参数判断，不让长矩阵和低频治理字段挤掉首屏。

#### A2 语言名称与样例展开态

默认展开：

1. Localized names
2. Examples

默认折叠：

1. Render（仅保留摘要）
2. Advanced

原因：当用户进入深编辑时，应把名称矩阵与样例验证视作同一工作流连续展开。

### 2. 语言元数据工作区

#### B1 主编辑态

默认展开：

1. 目录搜索 / 新建入口
2. Identity
3. Standards
4. Classification

默认折叠：

1. Geography
2. Population / Vitality
3. Documentation
4. Notes
5. Custom fields
6. History

原因：语言元数据的默认首屏首先解决“这是什么语言、它的编码和分类是什么”，地图和人口信息不应先压住识别流。

#### B2 地理与人口深入态

默认展开：

1. Geography
2. Population / Vitality

默认折叠：

1. Documentation（仅保留摘要）
2. Notes
3. Change reason
4. Custom fields
5. History

原因：地图、行政区和人口活力是连续决策，适合在第二工作流里合并展开；其余治理型区块继续下沉。

### 3. 正字法桥接工作区

#### C1 默认建规则态

默认展开：

1. Target 摘要
2. Source 选择
3. Rule meta
4. Rule text
5. 单条 Preview

默认折叠：

1. Sample cases
2. Validation

原因：默认创建规则时应让用户先完成来源绑定、规则定义和即时预览，而不是先看到大段错误列表。

#### C2 验证与调试态

默认展开：

1. Rule text
2. Sample cases
3. Validation
4. Preview output

默认折叠：

1. Source / Meta（收成摘要条）

原因：进入调试流后，核心目标变成逐条校验和修正规则，来源信息应退到摘要层，不再占据主视口。
