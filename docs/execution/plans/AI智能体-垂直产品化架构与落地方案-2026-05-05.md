# AI 智能体 — 垂直产品化架构与落地方案（2026-05-05）

## 0. 文档定位

本文承接《[AI智能体-战略规划与下一步-2026-05-07.md](./AI智能体-战略规划与下一步-2026-05-07.md)》§3.6，定义解语 AI 从“治理增强型聊天/工具执行”继续产品化为 **证据型语言资料智能体** 的架构、分期与验收。

**本轮假设**：当前仓库事实以 `useAiChat` 主链、`resolveExecutionPolicy`、`TaskRunner`、`propose_changes`、F4 sandbox、ADR-0027/0028/0029 与语料库一期定位为准；本方案先补规划与架构，不直接实现代码。

**成功标准**：后续每个垂直 agent 的输出都能回答四个问题：引用了哪些材料、建议了什么变更、用户是否采纳、事后如何复盘。

---

## 1. 外部垂直范式结论

调研样本包括 NotebookLM、Loom / Grain、Looppanel、Harvey / Lexis+ Protégé / Spellbook / Ironclad、Abridge / Nabla / Heidi / DeepScribe / Ambience / Suki，以及 ELAN / FLEx / Language Forge。对解语有用的不是“通用 autonomous agent”，而是以下共性：

| 范式 | 成熟产品共性 | 解语迁移口径 |
|------|--------------|--------------|
| 来源限定 | NotebookLM / Lexis 将回答限定在用户信任材料、内部文档或权威库内 | 以项目语料、语段、层、词典、笔记、导入文档组成 `CorpusSourceSet` |
| 精确引用 | 研究/法律产品把引用和原文片段作为可信基础 | AI 输出必须携带 `EvidencePacket`，能回到语段时间码、层、词条或文档片段 |
| 模板化专业产出 | 医疗 scribe、会议 agent、合同 agent 输出领域模板，而非自由闲聊 | 输出标注 QA、词典候选、导出清单、田野笔记、研究摘要等固定工作流 |
| 人在环采纳 | Looppanel / Ironclad 强调 AI works with you，not instead of you | 写操作默认 proposal / adoption queue；不静默写转写主链 |
| 工作流嵌入 | Loom / Grain / Suki 把结果推送到真实工作系统 | 解语优先嵌入语料库、转写时间轴、术语备忘、ELAN/FLEx 往返 |
| 审计与合规 | 法律/医疗 agent 强调审计、来源、隐私、治理 | 沿用 `ai_tool_call_decision`、release evidence、F4 矩阵与本地优先策略 |

**核心判断**：解语应对标 `NotebookLM + ELAN/FLEx + Looppanel + Lexis Protégé` 的交集，成为“有证据、有层级、有审阅队列的语言资料工作流智能体”。

## 2. 产品边界

### 2.1 做什么

1. **语段问答 Agent**：围绕选中语段、相邻语段、说话人、层和时间码回答问题。
2. **标注 QA Agent**：检查断句、缺翻译、缺 gloss、低置信度、层间不一致。
3. **词典候选 Agent**：从语段与 gloss 中提出候选 lexeme、义项、例句、冲突项。
4. **语料导出 Agent**：为论文、课堂材料、交付包生成可追溯导出清单。
5. **田野笔记 Agent**：从录音上下文、语段、元数据整理采集前/中/后 notes。
6. **证据化写作 Agent**：生成短段研究说明、例句说明、脚注草稿，并保留引用。
7. **语音采集助手**：沿用语音主链，服务采集准备、现场质量提示、采后复盘。

### 2.2 不做什么

| 非目标 | 边界 |
|--------|------|
| 通用 agent 平台 / swarm | 不做多 agent 自主分工与全自动执行链 |
| 自动替代研究者判断 | AI 只给候选、证据与理由，采纳权在用户 |
| 站内重做 ELAN / FLEx 全量能力 | 做互通、检查、候选与导出，不复制成熟工具全部编辑器 |
| 静默写入转写主链 | 与语料库定位一致：默认 proposal / copy / export，写入需确认 |
| 云端多租户知识库 | 当前保持桌面本地与用户项目内 source set |

## 3. 目标架构

### 3.1 总览

```text
User Request / Voice Request
  -> Send Turn Pipeline (useAiChat)
  -> VerticalWorkflowRegistry
  -> SourceResolver (CorpusSourceSet)
  -> Tool / Read Model Registry
  -> AgentRunState (TaskRunner / checkpoint when needed)
  -> EvidencePacket Builder
  -> Preview / AdoptionQueue / Export
  -> Audit / Eval / Release Evidence
```

### 3.2 核心构件

| 构件 | 职责 | 现有锚点 | 落地方式 |
|------|------|----------|----------|
| `CorpusSourceSet` | 描述一次 agent run 可读取的来源集合：语段、层、词典、笔记、PDF、导入文档 | 语料库定位、RAG、`EmbeddingSearchService` | P0 先以 DTO + builder 文档化，后续接语料库选择集 |
| `EvidencePacket` | AI 输出的证据基础单位：source、quote、location、confidence、reason | RAG snippet、tool audit、preview | P0 新增类型与格式化 helper，先覆盖只读回答与 QA |
| `AiToolRegistry / Workbench` | 工具 schema、scope、trust、preview、dry-run、executor 的单一目录 | `toolCallSchemas.ts`、`resolveExecutionPolicy.ts`、tool adapters | P0 shadow registry，不替换执行；P1 收敛主链读写 |
| `VerticalWorkflowRegistry` | 定义语段问答、标注 QA、词典候选等 workflow 的输入/输出/门禁 | `useAiChat` pipeline、CommandResolver、planner | P1 从 2-3 个高频 workflow 起步 |
| `AgentRunState` | 长任务状态图：created/planning/running/waitingForUser/applying/done/failed/cancelled | `TaskRunner`、`ai_tasks`、checkpoint | P0 写架构状态图；P1 在长 QA/导出任务接入 |
| `AdoptionQueue` | 用户审阅候选：采纳、忽略、复制、导出、跳转证据 | `propose_changes`、pending confirm | P1 先复用确认卡，P2 产品化列表 |
| `VerticalAgentEval` | 领域任务评测：引用准确、scope 正确、无静默写、采纳率 | `agent-evals`、release evidence | P0 seed 任务集，P1 接 trace 断言 |

## 4. 数据契约草案

### 4.1 EvidencePacket v0

```typescript
type EvidencePacketV0 = {
  schemaVersion: 0;
  id: string;
  sourceType: 'segment' | 'layer_text' | 'lexeme' | 'note' | 'document' | 'audio_region';
  sourceId: string;
  projectId?: string;
  mediaId?: string;
  segmentId?: string;
  layerId?: string;
  timeRangeMs?: { startMs: number; endMs: number };
  quote?: string;
  summary?: string;
  confidence?: number;
  reasonCode?: string;
};
```

**约束**：`EvidencePacket` 是证据指针，不是复制整段数据的容器；UI 可以显示 quote，但存储和审计优先存 id + location。

### 4.2 VerticalWorkflow v0

```typescript
type VerticalWorkflowV0 = {
  id: string;
  labelKey: string;
  /** v0.1：补充 §10 表中跨页/声学/PDF 类 workflow 所需粒度 */
  inputScope:
    | 'current_segment'
    | 'selection'
    | 'corpus_source_set'
    | 'project'
    | 'pdf_document_region'
    | 'acoustic_selection';
  outputKind:
    | 'answer'
    | 'qa_findings'
    | 'lexeme_candidates'
    | 'export_plan'
    | 'fieldwork_note'
    | 'comparison_brief'
    | 'token_tier_findings'
    | 'acoustic_brief'
    | 'orthography_findings'
    | 'timeline_hygiene_findings'
    | 'next_steps_checklist';
  writeMode: 'read_only' | 'propose_only' | 'confirm_required';
  /** 须与 F4 受控矩阵 capability id 对齐；占位用 `TBD_*` 仅在 P0 shadow 阶段 */
  requiredCapabilities: string[];
  evalSuiteId?: string;
};
```

### 4.3 AgentRunState v0

```text
created -> resolving_sources -> planning -> running
running -> waiting_for_user -> applying -> done
running -> done
running -> failed
running -> cancelled
running -> checkpointed -> running
```

**约束**：短请求不必入 `ai_tasks`；跨语料库 QA、批量候选、导出计划、长语音复盘等超过单 turn 的 workflow 才进入 durable state。

## 5. 关键工作流

### 5.1 只读问答：语段 / 语料库 NotebookLM 化

1. 用户选择语段、集选或 source set。
2. `SourceResolver` 生成 `CorpusSourceSet`。
3. RAG / read model 取材料。
4. LLM 生成回答。
5. `EvidencePacket Builder` 对每个结论挂 source。
6. UI 展示回答 + 引用，可跳转原语段 / 文档片段。

**验收**：无来源的问题必须显示“来源不足 / 需要扩大范围”，不能编造引用。

### 5.2 写建议：标注 QA / 词典候选

1. workflow 读取选区 / 项目范围。
2. 产出 findings 或 candidates。
3. 每条候选绑定 `EvidencePacket[]` 与 reason。
4. 进入 `AdoptionQueue`，用户逐条采纳 / 忽略 / 复制。
5. 真写入时走 `propose_changes` / confirm / audit。

**验收**：候选被忽略也应可复盘原因；采纳率进入 eval / evidence 可选指标。

### 5.3 语音田野工作流：采集前 / 中 / 后

| 阶段 | Agent 职责 | 写入边界 |
|------|------------|----------|
| 采集前 | 检查项目元数据、目标词表、上一轮待补问题、设备/TTS/STT 状态 | 只读 + checklist |
| 采集中 | 质量提示、当前语段上下文、低置信度提醒、必要澄清 | 不静默改主链；听写仍走既有独立路径 |
| 采集后 | 汇总新语段、待复核列表、词典候选、导出下一步 | proposal / export |

**对齐 ADR**：非听写语音请求仍经 AI 聊天主链；听写独立；`VoiceAgentService` 继续按 Phase B/C 提取后退役生产路径。

### 5.4 ELAN / FLEx 互通辅助

1. 导入/导出前检查层、tier、语言、gloss、时间码一致性。
2. 输出“可互通性报告”，而不是自动修所有问题。
3. 允许用户按 finding 跳转修复或加入导出说明。

**验收**：至少覆盖 EAF tier 映射、FLEx 词条候选、时间码缺口、受控词表不一致四类检查。

### 5.5 补充工作流（跨页与转写日课缺口）

下列条目来自「转写主链 + 语料库/标注路线图 + 语言资产」中的高频任务，**尚未**在 §2.1 / §6 中全部显式命名；实现时须与《语料库-产品定位与执行方案》**D 区分轨**、《标注词典语料-治理补充规范》**R4**（`corpusBridgeAdapter` / `annotationAdapters` 分区）一致。登记草案见 **§10**。

## 6. 分期落地

### P0 — 证据底座与 registry 影子层（1-2 个 PR）

| 交付 | 范围 | 验收 |
|------|------|------|
| `EvidencePacketV0` 类型与 builder | 只读回答、QA finding 先可生成证据包 | 单测覆盖 segment/layer/document 三类 source |
| `VerticalWorkflowRegistry` 最小表 | 登记语段问答、标注 QA、词典候选三个 workflow | registry 与 i18n label key 无漂移 |
| `AiToolRegistry` shadow map | 汇总现有 schema、scope、writeMode、policy reason，不替换执行 | 与 `toolCallSchemas.ts` 的工具名集合一致 |
| eval seed | 10-15 条垂直任务：引用、scope、no silent write | `check:agent-evals:trace` 可扩展或新增子脚本 |

### P1 — 两个可见垂直 agent（2-4 个 PR）

| 交付 | 范围 | 验收 |
|------|------|------|
| 语段问答 Agent | 当前语段 / 选区 / source set 问答，引用可跳转 | 回答中每个事实性结论至少 1 个 evidence |
| 标注 QA Agent | 缺翻译、低置信、层间不一致、时间码异常 | findings 不写主链；采纳需 confirm |
| AdoptionQueue MVP | 复用确认卡或轻量列表，支持采纳/忽略/跳转 | 审计记录 outcome 与 requestId |

### P2 — 语料库 NotebookLM 化与互通检查（3-6 个 PR）

| 交付 | 范围 | 验收 |
|------|------|------|
| Source set UI/状态 | 项目材料、语段集、笔记、导入文档选择 | source set 可保存到会话，不污染长期记忆 |
| 词典候选 Agent | lexeme/gloss/example/conflict candidates | 采纳默认进入确认；可导出为 FLEx 友好清单 |
| ELAN/FLEx compatibility Agent | EAF/FLEx 往返前检查 | 报告可导出，至少 4 类 finding |

### P3 — 田野语音生命周期与协作审阅（后续）

| 交付 | 范围 | 验收 |
|------|------|------|
| 采集前/中/后语音工作流 | checklist、质量提示、采后复盘 | 与 `check:voice-agent-pre-merge` 和 rollout E2E 对齐 |
| 多角色审阅 | 研究者/审校者/社区贡献者的候选反馈 | 不引入云多租户；本地/协作策略另行 ADR |
| release evidence 垂直指标 | evidence coverage、adoption rate、citation failure | governance gate 可消费或显式 skip |

## 7. 与当前代码的映射

| 当前模块 | 承载的新职责 | 注意事项 |
|----------|--------------|----------|
| `useAiChat` send-turn pipeline | 继续作为文本与非听写语音统一入口 | 不把 workflow 逻辑塞回 hook，落 service/registry |
| `toolCallSchemas.ts` | 工具参数 schema 继续为执行契约 | P0 shadow registry 只读生成，不改执行路径 |
| `resolveExecutionPolicy.ts` | capability / sandbox / quota 判定 | 新 write workflow 必须登记 F4 矩阵 |
| `useAiChat.confirmExecution.ts` | confirm、dry-run、`propose_changes` commit | AdoptionQueue 真写复用该路径 |
| `TaskRunner` / `ai_tasks` | 长 workflow 状态与 checkpoint | 短只读问答不入 durable task，避免过度工程化 |

| `corpusBridgeAdapter`（语料库路线） | 语料库材料与导出桥 | AI 默认不静默写转写主链 |

## 9. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 证据包变成又一套 RAG 结果格式 | P0 只做 source pointer + location，不复制全文 |
| workflow registry 与 tool schema 漂移 | shadow registry 加集合一致性测试，后续再合并真源 |
| 过早产品化长任务 UI | 短问答保持同步；只有长 QA/导出进入 `TaskRunner` |
| 候选采纳绕过 confirm | 所有写入 workflow 标记 `propose_only` 或 `confirm_required` |
| 垂直 eval 变成形式化标签 | 至少断言引用存在、scope 正确、无静默写、policy outcome |
| 与 ELAN/FLEx 互通目标膨胀 | 只做检查、候选、导入导出辅助；不复制完整编辑器 |

| `dictation.segment_cleanup` | `verticalWorkflow.dictationSegmentCleanup` | `current_segment` | `qa_findings` | `propose_only` | `TBD_transcription_read`, `TBD_propose_changes` | P2 | 听写/STT 填句后的错误清理候选；**不**替代听写独立链路与 ADR-0027 口径 |

### 10.2 与既有工作流的合并原则

- **能并入不写新 id**：若产出形态与 §5.1 / §5.2 相同，仅在 registry 增加一行变体（不同 `id`、相同 `outputKind`），避免 UI 与 eval 混测。
- **写入统一**：凡 `propose_only` / `confirm_required`，真写入仍只走 §7 表中 `propose_changes` / confirm / audit。
- **能力占位符**：`TBD_*` 仅用于 P0–P1 shadow registry；合入 F4 矩阵后 **必须** 改为稳定 capability id（见《F4-扩展入口-受控矩阵》登记模板）。
