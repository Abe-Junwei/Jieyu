# AI 与代码库可治理性综合整改方案

## 一、背景与目标

本方案综合以下输入，形成**可排期、可验收、与现有治理体系兼容**的整改路线：

| 来源 | 核心结论（吸收进本方案） |
| --- | --- |
| [AIHero：codebases AI agents love](https://www.aihero.dev/how-to-make-codebases-ai-agents-love) | **反馈环**、**可导航文件系统**、**深模块 + 清晰对外接口**、**灰盒（测试锁行为）**；TS 难硬隔离 → 用脚本 + 文档补位。 |
| [IBM：RAG 演进](https://www.youtube.com/watch?v=JB2P5Gk23VI) | 检索从线性管线走向 **agentic**；工程侧需 **可观测、可审计、边界清晰**；与「代码可治理性」交叉点在 **工具链、遥测、装配合同**，不单是算法。 |
| 仓库内两次评估 | **Orchestrator 已薄**与 **`src/pages` 仍密**可同时为真**；**宽接口 + `unknown`** 时不宜硬拆；**说明书口径**须与 `.cursor` / `AI_QUICKSTART` 一致。 |
| 既有拍板 | [单人AI协作改进计划-拍板决策-2026-05-11.md](./单人AI协作改进计划-拍板决策-2026-05-11.md) **5A**：ReadyWorkspace **渐进合同化**（先侧栏/overlays，再波形/layout）。 |

**总目标**：在 **不引入 Effect / 不大搬 `src/features`** 的前提下，使 **人 + AI agent** 均能 **快速定位、安全修改、有证据闭环**；并与 **Wave / ratchet / `check:architecture-guard`** 对齐。

---

## 二、原则（硬约束）

1. **编排层只做装配**：重逻辑进 controller / service / 纯函数（见根目录 `copilot-instructions.md`）。
2. **ReadyWorkspace `data` 域与壳层域分离**：壳层 API **不得**从 `useTranscriptionData` 的 `data` 误接；以 [ReadyWorkspace-数据域与壳层装配边界.md](../../architecture/ReadyWorkspace-数据域与壳层装配边界.md) 与 `npm run audit:ready-workspace-timeline-host` 为准。
3. **合同化先于大拆分**：`UseReadyWorkspaceSurfacePropsInput` 等宽输入在 **`unknown` 未收窄前**，优先 **分域 type guard + 文档化消费方**，避免「拆成多块仍全 `any`」。
4. **结构性改动必带证据**：`typecheck`、定向 `vitest`、触及交互/装配时 `test:e2e:chromium`（与拍板 **2A** 一致）。
5. **禁止根级 mega barrel**：若新增域级 `index.ts`，仅允许 **白名单 re-export**，并与 `copilot-instructions.md` 目录纪律一致。
