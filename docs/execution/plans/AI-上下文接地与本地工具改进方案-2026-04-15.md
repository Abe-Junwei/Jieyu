# AI 上下文接地与本地工具改进方案（对照现状代码）

**状态**：**已交付**（Phase A–E + 续步 payload 预算收口至 `useAiChat.config.ts`）  
**日期**：2026-04-15  
**实施**：Phase A–E 已于 2026-04-15 落地（E：`list_units` `resultHandle` 内存快照分页 + ADR `adr-ai-list-units-snapshot-handle.md`）；agent loop 续步 `tool_result_payload` 与首轮工具通道对齐（结构化压缩 + `ai.local_tool_result_truncated` 的 `scope: agent_loop`）；预算常量见 `src/hooks/useAiChat.config.ts`。  
**溯源**：工作区 `HEAD` 短哈希 **`1e675e8`**（合并/推送时请替换为实际 PR 或远端 commit）。  
**范围**：转写页 `useAiChat` 链路、`AiPromptContext`、本地只读工具（`localContextTools`）、RAG 与待确认工具 epoch；不覆盖语音助手 `VoiceAgentGroundingContext`（可后续对齐模式）。

## 1. 目标与非目标

### 1.1 目标

1. **本地只读工具使用「调用时刻」的读模型**（JIT），消除「用户发消息瞬间」快照与「模型输出 local tool JSON」之间的时间差导致的陈旧数据。
2. **工具结果可审计**：在 JSON 中稳定携带 `timelineReadModelEpoch`、`unitIndexComplete`、（适用时）`query`/分页信息，便于 UI、日志与后续 `groundingMetadata` 扩展。
3. **大结果可控**：在保持「不编造截断内容」前提下，减少 2000 字符硬截断对 agent loop 的误伤；可选引入句柄/分页协议。
4. **与现有门控一致**：`timelineReadModelEpoch` 已在 `buildTranscriptionAiPromptContext`、`resolveDestructiveGate`、`validateChangeSetEpoch` 路径上使用；本方案不破坏该契约，只补齐本地工具侧缺口。

### 1.2 非目标（本迭代不做或单列）

- 把 `list_units` 改为直接 `getDb()` SQL（可作为 Phase 2+ 备选；需与 `TimelineUnitView` 单口径对齐后再做）。
- 重做完整 Gemini 式 UI `groundingSupports`（先做结构化字段与内部消费，UI 可二期）。
- 修改远程 provider 的 function-calling 协议（当前转写 AI 以文本内 JSON 为主）。

## 2. 现状代码地图（事实基线）

| 环节 | 文件与行为 |
|------|------------|
| 每轮用户发送时构造「对话用」上下文 | `useAiChat.ts`：`getContextRef.current?.()` → `aiContext` → `buildPromptContextBlock` → RAG → `buildAiSystemPrompt`（约 554–604 行）。 |
| 转写页 `getContext` | `useTranscriptionAiController.ts`：`buildAiPromptContext` → `buildTranscriptionAiPromptContext`（含 `localUnitIndex`、`timelineReadModelEpoch`、`worldModelSnapshot` 等）。 |
| `localUnitIndex` 语义 | `chatDomain.types.ts`：`AiShortTermContext.localUnitIndex` 注释为 **不序列进 prompt 文本**，仅供工具；`SHORT_TERM_TEMPLATES` 应已排除该字段（若未排除需在 prompt 侧确认，避免把全量 unit 打进 system）。 |
| 本地工具执行 | `useAiChat.streamCompletion.ts`：`resolveAiChatStreamCompletion` 收到 **参数 `aiContext`**，在 `executeLocalContextToolCall(stepCall, aiContext, …)` 中使用（约 172–176、203–206 行）。**首轮与 `send()` 内捕获的 `aiContext` 为同一引用快照。** |
| Agent loop 续步 | `useAiChat.ts`：`loopAiContext = getContextRef.current?.() ?? aiContext`（约 776 行），**仅续步刷新**；首轮本地工具仍用旧 `aiContext`。 |
| 工具实现 | `localContextTools.ts`：`list_units` / `search_units` / `get_unit_detail` 从 `context.shortTerm?.localUnitIndex` 读取；`get_current_selection` 等读 `shortTerm`/`longTerm`。 |
| 工具结果展示 | `formatLocalContextToolResultMessage`：**整段 JSON `payload.length > 2000` 时硬截断**（约 466–478 行），与 `buildLocalContextToolGuide` 中「可能截断」说明一致。 |
| 意图类工具 | `intentTools.ts`：`findIncompleteUnits`、`diagnoseQuality`、`batchApply` 等通过 `getUnits(context)` 依赖同一 `localUnitIndex`。 |
| 远程工具规划 | `useAiChat.streamCompletion.ts`：`planToolCallTargets(parsedToolCall, userText, aiContext)`（约 232–234 行）；`resolveToolDecisionPipeline` 同样接收 `aiContext`。 |
| 幻觉启发式 | `appendHallucinationWarningIfSuspicious` 使用 `context?.shortTerm?.projectUnitCount` 等（约 290–330 行）。 |

**核心缺口**：`resolveAiChatStreamCompletion` 内本地工具路径使用的 `aiContext` 在首轮固定为 `send()` 开始时对象；长流式或用户快速编辑时，**工具读到的 `localUnitIndex` 可能落后于当前 `timelineUnitViewIndex`**。

## 3. 改进设计总览

### 3.1 原则（与成熟产品模式对齐）

- **摘要进 prompt，权威在工具边界**：保持 `worldModelSnapshot` 等为展示层；枚举/计数以工具在 **JIT `AiPromptContext`** 上跑出的结果为准。
- **元数据与内容并列**：工具 JSON 顶层增加 `_readModel`（或 `meta`）对象，避免模型把 epoch 当业务字段误用。

### 3.2 JIT 上下文（必选，Phase A）

在 **`resolveAiChatStreamCompletion` 入口处**（或在「仅本地工具分支」内第一次 `executeLocalContextToolCall` 之前）：

1. 新增可选参数 **`resolveFreshAiContext?: () => AiPromptContext | null`**（命名可改为 `getAiContextForTools`，与 `getContext` 区分语义：`getContext` 仍用于 prompt + 远程工具规划的首轮一致性是否要保持——见 3.3）。
2. `useAiChat` 在调用 `resolveAiChatStreamCompletion` 时传入：  
   `resolveFreshAiContext: () => getContextRef.current?.() ?? null`  
   本地工具执行统一使用：  
   `const toolContext = resolveFreshAiContext?.() ?? aiContext`  
   若 `toolContext` 为 null，行为与今日「`context` unavailable」一致。

**注意**：`appendHallucinationWarningIfSuspicious`、以及 **非本地工具** 路径是否仍用首轮 `aiContext`：

- **建议**：`appendHallucinationWarningIfSuspicious` 对「最终 assistant 自然语言」可继续用 **首轮 `aiContext`**（与用户问题同一时空），或改为 `toolContext` 仅在存在本地工具成功时合并校验——细节见 Phase A 任务拆分。

### 3.3 首轮 `aiContext` 与 `planToolCallTargets` 是否改用 JIT（需决策）

| 选项 | 行为 | 优点 | 风险 |
|------|------|------|------|
| **A. 仅本地工具 JIT**（推荐先做） | `planToolCallTargets` / `resolveToolDecisionPipeline` 仍用调用 `resolveAiChatStreamCompletion` 时传入的 `aiContext` | 与「用户发话时意图」一致；改动面小 | 若用户在流式期间改选区，远程工具 materialize 仍可能略旧 |
| **B. 全路径 JIT** | 上述 pipeline 也改用 `toolContext` | 远程工具与用户确认预览更新 | 与用户首轮问题时空不一致，可能改变 risk 文案；需回归测试量大 |

**建议迭代顺序**：先 **A**，观测 `ai.timeline_unit_count_mismatch` 与人工反馈后再评估 B。

### 3.4 工具结果结构化元数据（Phase B）

对 `list_units`、`search_units`、`get_unit_detail`、`find_incomplete_units`、`diagnose_quality` 等返回对象统一增加（示例形状）：

```json
{
  "_readModel": {
    "timelineReadModelEpoch": 42,
    "unitIndexComplete": true,
    "capturedAtMs": 1713000000000
  },
  ...
}
```

- `timelineReadModelEpoch`：来自 `context.shortTerm?.timelineReadModelEpoch`（JIT 上下文下即为当前 epoch）。
- `unitIndexComplete`：来自 `context.shortTerm?.unitIndexComplete`（缺省视为 true 或与 `localUnitIndex` 存在性一致——在实现文档中写死规则）。
- **不在** `get_project_stats` 重复编造；若 `projectStats` 与 `localUnitIndex.length` 不一致，保留现有 `list_units` 内 metric，并在 `_readModel` 中加 `indexRowCount?: number` 便于调试（可选）。

同步更新 `buildLocalContextToolGuide` 中的一行说明，避免模型把 `_readModel` 当用户数据。

### 3.5 大结果与截断（Phase C）

**现状**：`formatLocalContextToolResultMessage` 对整段 JSON 截 2000 字符，易导致 agent loop 中模型「看不到」`matches` 尾部。

**改进方向（二选一或组合）**：

1. **提高工具通道预算**（低风险）：例如 `LOCAL_TOOL_RESULT_CHAR_BUDGET = 8000`（常量置于 `localContextTools.ts`），与 `buildAgentLoopContinuationInput` 的「只用工具结果」指令一起评估 token；配合 metric `ai.local_tool_result_truncated`。
2. **句柄 + 分页**（中成本）：`list_units` 返回 `resultHandle`（uuid）+ 首屏 `matches`；新增本地工具 `get_unit_list_page` 或扩展 `list_units(arguments: { resultHandle, offset })`，结果存 `sessionMemory` 或短期 `Map`（注意内存上限与 tab 关闭清理）。与 Anthropic「大结果外置」思路一致。

**建议**：先做 **1** + metric；若仍频繁截断再上 **2**。

### 3.6 RAG 与读模型对齐（Phase D，可与 M18 索引工作并行）

- `useAiChat.rag.ts`：在写入 `citations` 或 context 块时，若 citation 含 `utteranceId`/`unitId`，附加当前 `aiContext.shortTerm?.timelineReadModelEpoch`（首轮）或 JIT epoch（若 RAG 在工具后触发则取最新——需梳理 RAG 仅在 `send` 前执行的事实：**RAG 仍在首轮前**，与工具 JIT 独立）。
- 可选：embedding 元数据写入 **索引 epoch**（依赖索引管道存储），回答生成后做「id 仍存在于 `localUnitIndex`」的过滤（二次校验）。

### 3.7 UI / 产品（Phase E，可选）

- `AiChatAlertsPanel` 已对比 `readModelEpochCaptured` 与 `timelineReadModelEpoch`；可在本地工具成功消息旁展示「读模型版本：epoch 42」供高级用户排查。
- Debug：`isAiContextDebugEnabled()` 路径增加「工具 JIT 前后 epoch」差值日志（仅 dev）。
- **本迭代**：上述 UI / dev 差分日志 **未实现**，保留为后续 backlog（见 §10）。

## 6. 风险与缓解

| 风险 | 缓解 |
|------|------|
| `getContext()` 副作用或成本过高 | `getContext` 已为纯函数组装；若未来变重，再拆 `getToolContext()` 只含 `localUnitIndex` + meta |

### 10.1 建议观测的指标（M5 / 自建看板）

| Metric ID | 用途 |
|-----------|------|
| `ai.local_tool_result_truncated` | 按 `tags.scope`（`single` / `batch` / `agent_loop`）观察工具 JSON 是否频繁触顶；若 `agent_loop` 占比高，考虑再收紧 `list_units` 的默认 `limit` 或引导模型分页。 |

### 10.2 方案 3.3「选项 B」（远程规划也 JIT）

**暂不实施**。启用条件建议写死为：连续两个发布周期内 `ai.timeline_unit_count_mismatch` 或用户反馈表明「首轮 `aiContext` 与工具 JIT」系统性不一致，且 A/B 评估后接受「与用户发话时空略错位」的风险文案变化。

### 10.3 已知产品/工程 backlog

- **`search_units` 空 query → `list_fallback`**：每次可新建 `list_units` 快照（新 `resultHandle`）；高频率时空占与指标噪声——待观测后再决定是否复用会话级 handle。  
- **§3.7 UI**：工具成功旁展示 epoch、dev JIT 差分日志 — **未做**，保留 backlog。
