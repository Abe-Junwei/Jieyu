# 声学语音学分析 Phase2-3 收尾清单（2026-04-08）

> 文档角色：执行方案文档。用于把“Phase 2 / 3 真正还缺什么”单独收口，避免把已落地能力重复列为待办。

## 1. 目标

本清单只回答一个问题：

在当前仓库已经具备 acoustic runtime、live inspector、slice、tone、descriptor、近似 Formant / 元音空间摘要与 CSV / JSON 导出的前提下，Phase 2 / 3 还剩哪些真正未闭合的差距项。

## 2. 已完成项，不再重复列入待办

以下能力已经进入主线，不应再作为 Phase 2 / 3 的新增目标重复立项：

1. 本地 acoustic runtime、进度状态与 VAD runtime 状态。
2. waveform / spectrogram 的统一 readout 与最小 live inspector。
3. local slice 与归一化 tone / intensity contour。
4. spectral centroid、rolloff、zero-crossing、flatness、loudness、MFCC 等 descriptor 摘要。
5. 近似 Formant、元音空间摘要与最小散点表达。
6. 当前选区 CSV / JSON 导出。
7. IndexedDB 持久缓存、stale / fresh 分层与校准脚本基础骨架。

对应代码落点可优先查看：

1. `src/components/AiAnalysisPanel.tsx`
2. `src/utils/acousticPanelDetail.ts`
3. `src/pages/useWaveformAcousticOverlay.ts`
4. `src/services/acoustic/AcousticAnalysisCacheDB.ts`
5. `scripts/evaluate-acoustic-calibration.mjs`

## 3. 真正未完成的差距项

### 3.1 Phase 2-A：把当前 acoustic tab 收口成重型 inspector

当前缺口不是“有没有 inspector”，而是 inspector 仍停留在单点只读与轻量图表阶段。

仍待补齐：

1. 冻结当前观测点与固定 inspector snapshot，避免 hover 离开后信息消失。
2. 多点比较视图，至少支持当前点 vs 冻结点两列对比。
3. 更细粒度的谱分析参数控制，而不是只展示当前 `YIN` / `RMS` 参数只读值。
4. 明确的 spectrogram inspector 工作区，而不是把所有能力都堆在统计面板样式中。

当前判定依据：

1. `AiAnalysisPanel` 中存在 runtime 参数展示，但没有冻结、对比或参数编辑入口。
2. `useWaveformAcousticOverlay` 只提供 hover / leave / click-to-seek 的瞬时交互，没有 pinned inspector 状态。

验收建议：

1. 用户可以把当前观测点冻结为固定卡片。
2. 用户可以在 acoustic tab 内同时查看至少两个时间点的 F0、intensity、descriptor 与 formant 对比。
3. 参数调整能显式影响分析请求或局部重算，而不只是展示现有配置。

### 3.2 Phase 2-B：跨面板热点复用还未闭环

当前热点已经存在，但仍主要停留在 acoustic tab 内导航和单点跳转。

仍待补齐：

1. 热点在 waveform、spectrogram、acoustic tab、AI insight 卡片之间的统一选中态。
2. 热点与当前选区、当前 utterance、当前 AI 解释之间的稳定关联模型。
3. 热点筛选、排序与复核状态，而不是仅显示列表后直接跳转。

验收建议：

1. 点击任一热点后，其状态可在主波形区与侧栏共同高亮。
2. AI 面板能够稳定引用“当前选中热点”，而不是只消费摘要文本。

### 3.3 Phase 3-A：Formant / 元音空间仍缺成熟 backend 与校准闭环

当前仓库已经有近似 Formant 与 vowel-space 摘要，但这还不是规划里承诺的科研口径版本。

仍待补齐：

1. 基于成熟谱分析 backend 的 Formant 候选提取，而不是当前近似估计。
2. Formant 平滑、可靠度过滤与异常帧剔除。
3. 与 `Praat / Parselmouth` 参考结果对齐的持续校准样本集与通过门槛。
4. 在通过校准前，继续把当前 Formant / 元音图标记为 exploratory，而不是确定性结论。

验收建议：

1. 校准脚本不只存在样例数据，还应形成固定样本集与可复跑门槛。
2. `formantF1Hz` / `formantF2Hz` 的误差与覆盖率达到预设阈值后，才把 UI 文案提升为正式分析能力。

### 3.4 Phase 3-B：科研导出互操作仍未完成

当前导出只覆盖当前选区 CSV / JSON，这对调试足够，但对科研互操作仍不够。

仍待补齐：

1. 面向 `Praat` 的 PitchTier / 相关互操作导出。
2. 比当前 CSV / JSON 更稳定的研究导出契约，包括元数据、参数、样本率、算法版本与校准版本。
3. 对整段媒体或批量选区的导出能力，而不只是一段 selection 的轻量下载。

验收建议：

1. 导出文件可在外部分析工具中复用，而不是只服务前端调试。
2. 导出结果可追溯到具体算法版本、参数与校准基线。

### 3.5 Phase 3-C：分析参数层仍偏只读，缺研究级配置面

当前 runtime 区域能展示窗口、步长与阈值，但还没有形成研究级配置层。

仍待补齐：

1. pitch floor / ceiling、窗长、步长、静音阈值等参数的可控入口。
2. preset 与自定义参数之间的切换语义。
3. 参数变更后的缓存键、重算、降级与 UI 提示策略。

验收建议：

1. 参数调整能驱动新的 analysis config 与缓存键。
2. 用户能明确知道当前图表对应的是哪套参数集。

## 4. 明确不在本清单内的内容

以下内容属于 Phase 4 或更后阶段，不应混入本清单：

1. provider-neutral 的外部声学 backend 适配层。
2. `MFA`、`pyannote`、外部 WhisperX 对齐或远程研究 provider 的正式接入。
3. 把默认浏览器本地声学链路替换为外部服务。

## 5. 推荐实现顺序

如果只按最小返工路径推进，建议顺序如下：

1. 先做 Phase 2-A：冻结 + 双点比较 + inspector 工作区收口。
2. 再做 Phase 3-C：把只读 runtime 参数升级为可控研究参数。
3. 再做 Phase 3-A：补 Formant backend、平滑与校准闭环。
4. 最后做 Phase 3-B：把导出升级到 Praat / 科研互操作层。

原因：

1. 先收口 inspector，才能让后续 Formant 与导出有稳定承载面。
2. 先把参数与缓存语义做清楚，后续校准与导出才不会反复返工。
3. Formant 和科研导出都依赖校准口径，不适合在当前近似版本上直接做重 UI。

## 6. 当前一句话结论

Phase 2 / 3 当前真正缺的不是“slice、tone、descriptor、导出有没有”，而是“重型 inspector、成熟 Formant backend、科研级校准和互操作导出是否闭环”。
