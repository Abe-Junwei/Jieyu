# 如是我闻本地版改进计划书（中文内容优先）

## 0. 结论摘要

如是我闻应先作为独立的本地中文转写校对工具落地，在线版仅保留接口余地，不进入第一阶段主线。

本计划对原《如是我闻_技术方案规划书》做三项关键收敛：

1. 从“云端平台 + 本地兜底”收敛为“本地优先 + 可选在线扩展”。
2. 从“多语通用 ASR + 微调驱动”收敛为“中文专用 ASR + 热词/术语/后处理/人机确认优先”。
3. 从“一期同时做实时、微调、协同、主题摘要、Word 智能排版”收敛为“导入、转写、校对、导出、项目保存”的最小闭环。

第一版成功标准不是“模型会自我训练”，而是：用户能在本地完成中文禅修音视频的高质量转写、同步校对、术语修正、低置信度确认和可交付文档导出。

**「第一阶段」与阶段表对齐（修订）**：文档中的「第一阶段」**产品交付口径**指 **§8 中 P0–P3 全部完成**（离线转写 + 校对闭环 + 中文领域增强 + 三种格式导出）。**P1 结束**即须具备 **TXT/SRT 最小导出**（不含 DOCX 固定模板与讲稿模式）；DOCX、讲稿/逐字稿双模式、低置信度与 LLM 建议写入导出等，按 P3 交付。避免 §1.3 与 §8 被误读为「P1 即含全套导出」。

**工期与人力假设（修订）**：§8 周数为**方向性估算**，默认约 **2 名全职等效**（桌面 + 推理服务 + 产品）且工具链熟练；人手不足或需从零搭建评测集时，应整体后移并显式更新里程碑，不得将周数当作对外固定承诺。

## 1. 产品定位

### 1.1 一句话定位

如是我闻是面向中文禅修课程音视频的本地转写、校对与文稿生成工作台。

### 1.2 第一阶段用户

1. 禅修机构助教、义工、编辑。
2. 需要整理课程录音、带练音频、开示问答的内容维护者。
3. 不具备机器学习背景，但能进行人工校对和术语确认的普通用户。

### 1.3 第一阶段核心场景

1. 导入中文音频或视频。
2. 本地完成离线转写。
3. 在音频时间轴上同步校对文本。
4. 管理禅修术语、讲师专属词库和常见错词。
5. 导出 TXT、SRT、DOCX 文稿。
6. 保存本地项目包，便于备份、迁移和继续编辑。

**与 §8 的对应关系**：场景 5 的「可交付文档」在 **P3** 达到完整形态（TXT + SRT + DOCX 模板与模式）；**P1** 即提供 **TXT/SRT 最小导出**，以满足「能交出字幕/纯文本」的助教工作流，并与「本地项目包」一并验收。

### 1.4 暂不作为第一阶段目标

1. 在线 SaaS 平台。
2. 多用户实时协同。
3. K8s / API 网关 / 多租户 / 云端灰度发布。
4. 自动 LoRA 微调与自动模型替换。
5. 实时直播字幕。
6. 多语言识别与多语言文档规范。
7. 全自动主题图谱、深度摘要和范文学习式复杂排版。

## 2. 原计划书主要问题

### 2.1 范围过大

原计划同时覆盖本地客户端、云端 GPU、实时转写、离线批处理、LLM 校对、模型训练、多租户、Word 模板、主题摘要、协同权限。作为长期愿景可以保留，但不适合作为第一版执行计划。

### 2.2 过早依赖模型微调

中文禅修音频的早期精度提升不应首先押注 LoRA 微调。更现实的收益来自音频预处理、VAD 分段、中文专用 ASR、热词注入、拼音近音纠错、术语表和低置信度人工确认。

### 2.3 在线架构过早

如果第一版就引入云端服务、多租户、同步和权限，会把核心风险从“转写是否好用”转移成“平台是否能运维”。本地版更适合先验证真实工作流。

### 2.4 指标缺少验收口径

原计划提到 CER、术语识别率、实时延迟、LLM 校正覆盖率，但没有定义测试集、音频质量分层、样本量、统计方法和失败判定。

### 2.5 LLM 校对边界不足

禅修文稿涉及教义、引文和讲师表达风格，LLM 不应静默改写正文。第一版只能作为批注建议器，而不是自动定稿器。

## 3. 总体技术路线

### 3.1 本地优先架构

建议采用“桌面应用 + 本地推理服务”的松耦合结构：

1. 桌面应用：负责导入、播放、时间轴、校对、术语管理、导出和项目保存。
2. 本地 ASR 服务：Python 进程提供 FunASR / SenseVoice / Whisper HTTP API。
3. 本地数据层：见 **§7.7**（默认推荐 **Tauri + SQLite**；纯 Web + 本地 Python 原型可用 IndexedDB，但有配额与大文件限制）。
4. 本地文件目录：保存音频副本、导出文件、项目包、缓存和模型配置。
5. 本地 ASR 服务仅绑定 **回环地址**（如 `127.0.0.1`），对外不可达；见 **§10.8**。

推荐第一版桌面方案：Tauri + React。若团队更熟 Python，也可先用纯 Web + 本地 Python 服务原型，但不建议优先 PyQt，因为后续校对体验、复杂时间轴和前端生态复用成本更高。

### 3.2 中文专用 ASR 主线

只处理中文内容后，主引擎应优先选择中文优化模型：

1. 主引擎：FunASR / SenseVoice / Paraformer。
2. VAD：FunASR VAD 或 Silero VAD。
3. 标点：FunASR 标点模型或中文标点恢复模块。
4. 兜底：Whisper / faster-whisper 仅用于噪声、口音或低置信度片段重跑。
5. **LLM（默认不进入安装包）**：P0–P3 **不**将本地 LLM 作为默认依赖；若做实验，仅允许以 **可选插件 / 用户自备路径** 接入，且遵守 §4.7。正式「LLM 保守校对」留在 **P5 候选**（与 §12 一致）。

### 3.3 第一版处理流水线

```text
音频/视频导入
-> FFmpeg 转码、抽音轨、音量归一化
-> VAD 切分，保留前后上下文
-> FunASR / SenseVoice 中文识别
-> 热词 / 术语偏置
-> 中文标点与文本规整
-> 术语纠错、拼音近音候选
-> 低置信度标注
-> 人工校对确认
-> TXT / SRT / DOCX 导出
-> 本地项目包保存
```

### 3.4 技术栈与部署形态（调研摘要）

以下为 **2026-05 公开资料** 层面的归纳，用于支撑 §3.1–§3.2 选型；**许可证与商用边界**以各项目 LICENSE 及权重发布条款为准，见 §10.7。

**桌面与 UI**

| 选项 | 说明 |
|------|------|
| **Tauri + React**（计划首选） | 小包体、系统 WebView；Rust 侧便于管理子进程与文件系统权限。 |
| **Electron + React** | 生态成熟，包体与内存占用通常更大。 |
| **纯 Web + 本地 Python 服务** | 原型快；大文件与持久化受浏览器存储配额约束，见 §7.7。 |

**推理与服务形态**

| 组件 | 业界常见做法 | 与如是我闻关系 |
|------|----------------|----------------|
| **FunASR** | 提供 **ONNX**（如 `funasr-onnx`）、**Docker / WebSocket** 等 runtime 路径；Paraformer、FSMN-VAD 等与流水线拆解成熟。[FunASR ONNX runtime 说明](https://github.com/modelscope/FunASR/blob/main/runtime/python/onnxruntime/README.md)、[runtime quick start](https://github.com/modelscope/FunASR/blob/main/runtime/quick_start.md) |
| **SenseVoice** | 多语言语音理解模型（ASR + 语种 / 情感 / 音频事件等），支持 **ONNX / libtorch** 与多语言客户端示例；适合中文主线或与 FunASR 组合，但 **附加头（情感、掌声等）** 对禅修校对可能是噪声，第一版宜默认关闭或仅诊断日志。[SenseVoice 仓库](https://github.com/FunAudioLLM/SenseVoice) |
| **Whisper / faster-whisper** | 通用 ASR；作 **低置信度片段兜底** 与多引擎对照，与 [Buzz](https://github.com/chidiwilliams/buzz) 等多后端桌面工具路线一致。 |
| **FFmpeg** | 抽轨、重采样、归一化；流水线标配。 |

**未来在线 Provider（非 MVP）**

| 示例 | 说明 |
|------|------|
| **火山引擎大模型流式语音识别** | WebSocket API、双向流式等，适合日后实现为 `TranscriptionProvider` 的远程实现，与 §11 预留一致。[官方文档入口](https://www.volcengine.com/docs/6561/1354869) |

## 4. 非微调精度提升方案

### 4.1 音频预处理

目标：降低输入音频质量对 ASR 的影响。

必须支持：

1. 视频抽音轨。
2. 统一转为 16kHz、mono、wav 或模型推荐格式。
3. 音量归一化。
4. 静音检测与裁剪。
5. 超长音频分段。

可后置：

1. 神经降噪。
2. 去混响。
3. 自动判断录音质量并推荐重录。

验收：同一批样本预处理前后，静音幻觉减少，超长音频不因单段过长导致失败。

### 4.2 VAD 与智能分段

目标：避免长静默、超长上下文和段落漂移造成识别错误。

必须支持：

1. 按语音活动切段。
2. 每段保留前后 0.3-1 秒上下文。
3. 禅修长静默单独标记，不送入 ASR 或以空段保存。
4. 过短噪声段过滤。

验收：长静默不产生无意义文字；用户可看到分段依据并手动合并或拆分。

### 4.3 热词和术语偏置

目标：不微调模型也能提升禅修专业词识别率。

第一版术语来源：

1. 用户手动录入。
2. 预置禅修核心术语。
3. 用户校对修改中推荐生成。
4. 讲师或课程系列专属词库。

术语字段建议：

1. 标准写法。
2. 别名 / 异体 / 繁简。
3. 拼音。
4. 常见误识别。
5. 适用讲师或课程。
6. 是否进入热词。

**能力分级（修订）**：FunASR、SenseVoice、Whisper 等对「热词 / 偏置」的 API **并不统一**，部分引擎可能 **不支持** 服务端热词。验收口径调整为：

1. 若当前 `TranscriptionProvider` 声明支持 `hotwordBias`，则术语表变更后，**下一次**走该引擎的识别请求应携带热词或等价偏置；系统记录术语命中与疑似漏识别。
2. 若引擎不支持，则热词仅作用于 **后处理 / 拼音近音 / 人工提示**，不得假造「已注入模型」；UI 须标明该引擎「术语偏置：仅后处理」。

详见 §6.1 `supportsHotwordBias` 等扩展字段。

### 4.4 拼音近音纠错

目标：处理中文 ASR 常见同音、近音错误。

示例：

1. 安波那那 -> 安那般那。
2. 舍摩他 -> 奢摩他。
3. 四念住 / 四念处的标准化提示。

处理原则：

1. 高置信度、明确错词对可自动替换。
2. 中置信度以批注或黄色建议展示。
3. 教义、引文、人名不确定时必须人工确认。

验收：每条自动替换都有来源规则；用户可撤销并禁用该规则。

### 4.5 中文标点与文本规整

目标：提升文稿可读性，而非改变原意。

第一版支持：

1. 中文标点恢复。
2. 数字、日期、课程编号规整。
3. 问答换行。
4. 长停顿作为段落候选。
5. 逐字稿模式与讲稿模式分离。

验收：用户可选择导出模式；讲稿模式的规则不会覆盖原始逐字稿。

### 4.6 多引擎兜底

目标：对低置信度片段重跑，而不是全文使用重模型。

策略：

1. 清晰普通话：FunASR / SenseVoice。
2. 噪声、口音、异常低置信度：Whisper / faster-whisper 重跑片段。
3. 多引擎结果差异大：提示人工确认。

验收：低置信度片段可以单独重跑；重跑结果与原结果并排比较。

### 4.7 LLM 保守校对

目标：让 LLM 做助理，而不是定稿者。

允许：

1. 错字疑点提示。
2. 术语替换建议。
3. 标点和断句建议。
4. 标题、段落和摘要建议。
5. Word 批注。

禁止：

1. 静默改写正文。
2. 自动补充模型没有听到的内容。
3. 自动统一教义表达。
4. 删除讲师风格化表达而不留痕。

验收：所有 LLM 建议均可接受、拒绝、撤销，并记录输入、输出、模型和时间。

## 5. 自我学习闭环

### 5.1 第一阶段：无需训练的学习

第一版的自我学习应优先体现为“系统记得用户怎么改”：

1. 纠错记忆：记录修改前、修改后、时间段、讲师、课程、置信度。
2. 术语推荐：从高频纠错中推荐加入术语库。
3. 错词规则：把确认过的错词对变成后处理规则。
4. 讲师词库：按讲师或课程系列加载不同热词。
5. 导出偏好：保存用户对逐字稿、讲稿、问答、偈颂的格式偏好。

### 5.2 第二阶段：训练数据准备

当纠错记录足够多后，才进入训练数据准备：

1. 自动切出纠错片段音频。
2. 使用 WhisperX / MFA / FunASR token timestamp 对齐。
3. 生成训练 manifest。
4. 标记数据质量：音质、对齐置信度、文本是否润色、是否多人重叠。
5. 建立 held-out 测试集。

### 5.3 第三阶段：可选本地微调

微调只作为高级功能，并必须手动触发。

启用门槛：

1. 至少 10 小时高质量对齐音频-文本对。
2. 有独立测试集，不参与训练。
3. 本机具备足够 GPU 或用户明确接受长时间训练。
4. 训练后 CER、术语准确率和回归样本均优于旧模型。
5. 支持一键回滚旧模型。

不满足以上条件时，不应开放“自动微调并替换模型”。

## 6. 可复用 Jieyu 的内容

如是我闻是独立项目，但可以复用或借鉴 Jieyu 中的成熟设计。**独立新 Git 仓库** 场景下，规范包、代码白名单、禁止照搬项与初始化顺序，以架构真源 [`如是我闻-独立新仓库与-Jieyu-对齐策略.md`](../../architecture/如是我闻-独立新仓库与-Jieyu-对齐策略.md) 为准（与本章互补）。

### 6.1 STT provider registry

参考：[`src/services/stt/index.ts`](../../../src/services/stt/index.ts)

可复用点：

1. 将不同 ASR 引擎统一成 provider。
2. 本地、商业、未来在线服务走同一选择器。
3. 支持 provider 可用性探测。

建议在如是我闻中建立 `TranscriptionProvider`（**独立仓库内维护契约**，可借鉴 Jieyu 思路，避免反向依赖 Jieyu 整仓）：

```ts
interface TranscriptionProvider {
  id: string;
  label: string;
  /** 引擎是否支持将术语以模型可消费的方式注入识别请求 */
  supportsHotwordBias: boolean;
  /** 可选：情感/事件等多余 ASR 头，禅修场景可全 false */
  capabilities?: {
    emotion?: boolean;
    audioEvents?: boolean;
  };
  isAvailable(): Promise<boolean>;
  transcribe(input: TranscriptionInput): Promise<TranscriptionResult>;
}
```

### 6.8 多产品对标（公开信息摘要）

下列对标仅用于 **能力清单与差异化**，不构成实现指令；各产品功能与合规以官方为准。

| 产品 / 形态 | 数据路径（摘要） | 与如是我闻差异化 |
|-------------|------------------|------------------|
| **[Buzz](https://github.com/chidiwilliams/buzz)**（开源，MIT） | 本机离线，Whisper / whisper.cpp / Faster Whisper 等 | 形态最接近「本地转写 + 导出」；**缺**禅修术语体系、教义安全编辑流、项目包追溯 |
| **MacWhisper 等**（商业/免费增值，多见于 macOS） | 宣传为本机 Whisper | 导出格式多；**中文专模与词库**弱于本计划主线 |
| **Descript** | 以 **云** 为主，用户长期反馈「真离线」有限（见 [功能请求讨论](https://feedback.descript.com/feature-requests/p/true-offline-mode)） | 强在音视频剪辑 + 脚本编辑一体化；**非**「数据默认不出机」 |
| **Otter.ai** 等会议产品 | 企业云管线；公开资料侧重实时会议与集成 | **非**长课程本地包体与垂直校对场景 |
| **[讯飞听见](https://www.iflyrec.com/index)** | 云端按分钟计费；机器快转 + 人工精转 | **功能清单参考**：专业领域、热词、说话人、语篇规整等；实现路径为 **本地引擎 + 自管词库**，而非复刻其云 API |
| **[飞书妙记](https://www.feishu.cn/product/minutes)** | 上传本地文件，处理在服务商侧 | **对标**「上传即转、多人分轨」体验；隐私模型与本产品不同 |
| **通义听悟** | 标准用法为云；企业私有化/离线能力以 **阿里云合同与文档** 为准 | **对标**「听记 + 摘要助理」；本计划 **LLM 仅建议、不定稿**（§4.7） |
| **火山引擎豆包语音**（[流式 ASR 文档](https://www.volcengine.com/docs/6561/1354869)） | 云端 WebSocket API | **未来在线 `TranscriptionProvider`** 候选，与 §11 一致 |

**如是我闻的定位摘要**：在「**默认本地、中文禅修、术语与纠错可追溯、教义不可被模型静默改写**」上形成差异化；对标商用产品时取其 **能力维度**，不复制其云端架构。

### 7.7 本地数据层选型（SQLite / IndexedDB）

| 维度 | **SQLite（推荐，Tauri 桌面默认）** | **IndexedDB / Dexie（纯浏览器 + 本地服务原型）** |
|------|-----------------------------------|--------------------------------------------------|
| 大项目与全文检索 | 成熟，可配合 FTS | 能力受限，检索需自建 |
| 与媒体大文件同目录备份 | 易与「项目包」目录结构一致 | 浏览器沙箱与配额风险高 |
| 架构迁移与版本 | 成熟方案（迁移脚本） | 需严格版本化升级策略 |
| 离线包体与权限 | 与 Tauri 文件 API 一致 | 依赖下载与用户授权 |

**结论**：**正式产品默认 SQLite + 文件系统媒体引用**；仅当团队选择「无桌面壳、浏览器即 UI」原型阶段时采用 IndexedDB，并在 §8 中单独标注为 **原型分支**，避免两种存储长期并存无主从。

## 8. 分阶段落地计划

### 8.0 阶段命名与依赖

- **P0–P4** 为顺序主线的推荐次序；**P5** 为候选集合，逐项 Go / No-Go。
- **P1 末尾** 必须交付 **TXT/SRT 最小导出**（见下文 P1），与 §1.3、§0 口径一致。
- **P3** 完成「三种格式 + 模式」后，§1.3 场景 5 方视为完整达成。

### P0：技术原型（1-2 周）

目标：验证本地中文 ASR 服务和前端/桌面调用链路。

交付：

1. 本地 FunASR / SenseVoice 服务可启动。
2. 支持单文件音频上传并返回分段 JSON。
3. 支持 FFmpeg 抽音轨和格式规范化。
4. 定义 `TranscriptionProvider` 与 `TranscriptionResult` 契约。

不做：

1. 完整编辑器。
2. Word 模板。
3. 微调。
4. 在线同步。

验收：

1. 10 个中文样本可批量转写成功。
2. 每个样本有文本、时间段、置信度或可降级置信度。
3. 失败样本有明确错误原因。

### P1：本地校对闭环（3-5 周）

目标：形成最小可用工作台。

交付：

1. 项目创建、音频导入、音频播放。
2. 语段列表与时间轴联动。
3. 文本编辑、保存、撤销。
4. 语段拆分、合并、时间戳调整。
5. 本地项目保存和重新打开。
6. **TXT、SRT 最小导出**（单项目一键导出当前稿；编码与换行规则写进用户文档；不含 DOCX、不含讲稿模式）。

验收：

1. 用户能完整校对一段 30-60 分钟中文课程录音。
2. 关闭应用后重新打开，文本、时间戳和状态不丢失。
3. 所有编辑操作可追溯到修改记录。
4. 同一项目可导出 **TXT 与 SRT**，导出文件与 UI 当前正文一致且不反向改写项目内正文。

### P2：中文领域增强（6-8 周）

目标：不微调模型，先提升禅修术语和文稿可读性。

交付：

1. 术语库。
2. 热词导出与识别请求注入（**受 `supportsHotwordBias` 约束**；不支持引擎走 §4.3 后处理路径并在 UI 标明）。
3. 拼音近音候选。
4. 中文标点和文本规整。
5. 低置信度高亮。
6. 纠错记忆和错词规则推荐。

验收：

1. 术语表新增词能进入下一次识别。
2. 用户确认过的错词对能在后续文稿中提示。
3. 自动替换和建议替换有清晰区分。

### P3：导出交付（9-10 周）

目标：让转写结果可以直接交付编辑或发布团队。

交付：

1. TXT 导出。
2. SRT 导出。
3. DOCX 固定模板导出。
4. 逐字稿模式与讲稿模式导出。
5. 低置信度标记导出为批注或样式；**LLM 建议写入导出** 仅当产品已启用 **可选 LLM 插件**（默认关闭，见 §0 / §4.7），否则本项验收只覆盖低置信度。

验收：

1. 同一项目可导出三种格式。
2. DOCX 保留段落、标题、问答和低置信度标记。
3. 导出结果不反向修改项目正文。

### P4：稳定化与评测（11-12 周）

目标：建立可持续改进的证据体系。

交付：

1. 中文禅修音频评测集。
2. CER、术语准确率、低置信度命中率统计。
3. 批量任务与失败恢复。
4. 本地安装包。
5. 崩溃日志和可导出诊断包。

验收：

1. 至少覆盖清晰录音、远场、噪声、问答、术语密集五类样本。
2. 每次引擎或后处理规则更新都有对比报告。
3. 批量转写失败后可从失败文件继续。

### P5：高级能力候选（12 周后）

进入条件：P0-P4 可稳定使用，且真实用户已经完成多轮校对。

候选能力：

1. Whisper / faster-whisper 低置信度片段兜底。
2. WhisperX / MFA 强制对齐。
3. pyannote 说话人分离。
4. LLM 保守校对批注。
5. 本地 LoRA 微调。
6. 在线 provider。
7. 多用户协同。

每项高级能力必须独立做 Go / No-Go，不默认进入主线。

## 9. 验收指标

### 9.1 转写质量

1. CER：按固定中文评测集统计。
2. 术语准确率：按术语密集片段统计。
3. 静音幻觉率：长静默片段中错误输出字数。
4. 低置信度召回：人工发现错误中有多少被系统标记。

### 9.2 工作效率

1. 每小时音频的初转写耗时。
2. 每小时音频的人工校对耗时。
3. 用户接受/拒绝建议比例。
4. 导出后返工次数。

### 9.3 本地稳定性

1. 30、60、120 分钟音频转写成功率。
2. 断点恢复成功率。
3. 项目重新打开一致性。
4. 导出格式快照一致性。

## 10. 风险与缓解

### 10.1 中文模型仍可能误识别术语

缓解：热词、拼音近音、术语批注和人工确认先行；微调后置。

### 10.2 长静默导致幻觉

缓解：VAD 跳过静默；对长静默生成空段或练习段，不送入 ASR。

### 10.3 LLM 过度改写

缓解：LLM 默认只生成建议和批注；正文修改必须由用户确认。

### 10.4 本地机器性能差异大

缓解：提供模型档位；默认轻量模型；重模型只用于片段复核。

### 10.5 数据未来无法微调

缓解：从第一版开始保存时间戳、纠错记录、音频片段和数据质量标记。

### 10.6 在线版回头改造成本高

缓解：第一版即抽象 `TranscriptionProvider`、`ProjectStorage`、`FeedbackStore`，但实现保持本地。

### 10.7 分发、依赖与许可证风险

**现象**：安装包捆绑 **FFmpeg**、**模型权重**、**ONNXRuntime / CUDA** 与各 **ASR 仓库** 时，许可证组合复杂，易影响「能否商用分发」。

**缓解**：

1. 维护 **第三方与模型许可证清单**（含再分发条款、署名要求），在 P4 安装包交付前完成法务可发布评审。
2. 默认 **用户首次启动时下载模型**（减小安装包、权责清晰），与「一键离线包」分支分轨发布。
3. 对 AGPL 等强 copyleft 组件避免与闭源壳 **不当静态链接**；优先进程边界或替代实现。

### 10.8 本地 HTTP 服务安全与稳定性

**现象**：Python 子进程若监听 `0.0.0.0` 或未处理 OOM，存在局域网暴露与桌面假死风险；转写长音频时进程崩溃会导致前端状态不一致。

**缓解**：

1. 仅监听 **127.0.0.1**；固定或注册端口冲突策略；可选 **本地 token**（防止本机多用户恶意进程抢连）。
2. 定义 **心跳与健康检查**、**请求超时**、**断点续转** 与 **崩溃自动拉起** 策略；错误信息对用户可理解、对诊断可导出（与 P4 诊断包一致）。

### 10.9 热词与多引擎 API 不齐套

**现象**：各引擎热词/偏置接口不一致，易导致「术语表已更新但识别未生效」的假象。

**缓解**：严格执行 §6.1 `supportsHotwordBias` 与 §4.3 能力分级；集成测试覆盖「支持 / 不支持」两路径。

### 10.10 人力资源与排期承诺

**现象**：§8 周数在人力不足时被当作对外合同日期，引发交付纠纷。

**缓解**：以 §0 **工期与人力假设** 为前置；里程碑变更需同步修订本文档 `last_reviewed` 与修订说明。

## 11. 在线版预留边界

在线版不是第一阶段目标，但可以保留以下接口：

1. `TranscriptionProvider`：本地服务与远程服务共享契约；远程实现示例可包括 **火山引擎大模型流式 ASR**（[文档](https://www.volcengine.com/docs/6561/1354869)）等，**不得**进入 P0–P3 默认依赖。
2. `ProjectStorage`：本地项目保存与未来云端同步分离。
3. `FeedbackStore`：纠错记录默认本地，未来可选择上传。
4. `ModelProfile`：本地模型与远程模型统一描述。

不得在第一版为了在线版提前引入：

1. 多租户数据库。
2. 云端权限模型。
3. 实时同步协议。
4. 云端模型灰度发布。
5. 服务器依赖的核心编辑流程。

## 13. 计划判定

本改进版计划的核心判断是：

1. 先本地，后在线。
2. 先中文专用，后通用扩展。
3. 先非微调增强，后模型训练。
4. 先校对闭环，后智能自动化。
5. 先证据化评测，后宣称精度提升。

若 P0-P4 无法证明本地工作台能减少人工校对成本，则不应进入在线平台化和模型微调阶段。
