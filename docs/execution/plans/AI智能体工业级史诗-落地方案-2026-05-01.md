# AI 智能体工业级史诗 — 规划落地方案（2026-05-01）

## 0. 文档定位

本方案把《[AI 智能体工业级对标改进计划书（2026-04-25）](./AI智能体工业级对标改进计划书-2026-04-25.md)》中 **Phase C+ / 专项能力** 与《[AI 智能体对标 — 落地修复计划（2026-04-26）](./AI智能体对标-落地修复计划-2026-04-26.md)》已声明的 **后续史诗** 收敛为 **四条可排期轨道**，每条给出：目标、与现状差距、切片交付、验收、主要代码锚点与依赖。

**非目标（本方案不展开细节设计）：** 云端多租户编排、非桌面端形态、与解语核心业务无关的通用 Agent 平台。

---

## 1. 总览：四条轨道与推荐顺序

| 轨道 | 解决的核心问题 | 依赖 | 建议顺序 |
|------|----------------|------|----------|
| **T1 — Durable 任务可操作闭环（F2b/F3）** | 长任务有状态、可取消/可重试、刷新后可理解 | `ai_tasks`、session memory、现有任务 UI 雏形 | **1（优先）— 主切片已收口（见 §2.6）** |
| **T2 — 执行层权限隔离 v1（capability policy）** | 工具 × 范围 × 信任 × 额度在执行层可断言 | F4 后台记忆 sandbox 范式、[F4 扩展史诗](./F4-扩展入口-capability-isolation-epic-2026-05-01.md) | **2** |
| **T3 — 变更事务化（preview / dry-run / commit）** | 多步写有一致语义与统一回滚/解释边界 | T1 部分（任务与审计关联）、B2 rollback 边界 ADR | **3** |
| **T4 — 失败分流与运维信号** | 可分类处置 + 发布/证据可消费指标 | 现有 audit、release evidence、agent-evals trace | **4（可与 T1 后半并行）** |

**原则：** 先让「任务看得见、停得了、续得上」（T1），再扩「谁能写什么」（T2），再谈「一笔变更单」（T3），最后把「出事怎么办、线上看什么」产品化（T4）。

## 2. 轨道 T1 — Durable 任务可操作闭环（F2b / F3）

### 2.1 目标

用户在 **刷新、重开、长时间等待** 后仍能：**看到** agent 循环 / 长任务状态，**取消**或**重试**，并在允许范围内 **从 durable checkpoint 恢复**（与当前「仅 session 绑 taskId」能力对齐并超出）。

### 2.2 现状基线（代码事实）

- `AiTaskDoc` 元数据、`TaskRunner` heartbeat/checkpoint API（修复计划 F0/F1）。
- `pendingAgentLoopCheckpoint` 与 `ai_tasks.checkpointJson` 绑定；resume 入口可从 DB reload（修复计划 F2a/F2b 最小切片）。
- 任务列表已部分暴露 `resumable` / checkpoint / `handoffReason`（修复计划 F3 最小可见切片）。

### 2.3 规划切片（建议 3 个迭代内可验收）

| 切片 | 交付物 | 验收 |
|------|--------|------|
| **T1-a** | 任务面板：对 `agent_loop` pending 行提供 **取消**（写终态 + 清理 session 悬挂 checkpoint） | Vitest + 手工：取消后不再出现可误触 resume；**2026-05-01**：`useAiEmbeddingState.handleCancelAiTask` 成功返回后经 `embedding.navigation.onAgentLoopTaskCancelledFromTaskList` 调用 `useAiChat.clearPendingAgentLoopCheckpointIfTaskIdMatches`；`AssistantBridge` deferred `aiChat` 补齐 `dismissPendingAgentLoopCheckpoint` 供侧栏 Hub |
| **T1-b** | **重试**入口：`agent_loop` **复用同一 `ai_tasks.id` 行**（`retryDurableAgentLoopTask`）；重试成功后 **清空** 会话内与该 `taskId` 匹配的 `pendingAgentLoopCheckpoint`，后续新一轮 handoff 再由运行时写入**同 taskId** 的新 checkpoint；**任务列表**续跑/重试按钮文案与聊天 composer 的 send/resume 提示**分流**（`msg.embCard.resumeAgentLoopTaskRow` / `msg.embCard.retryAgentLoopTaskRow`） | 与 `hasPersistedExecutionForRequest` / 父级 propose 语义无冲突单测；`onAgentLoopTaskRetriedFromTaskList` 接线同 T1-a cancel 桥 |
| **T1-c** | **冷启动**：无 session 时从 `ai_tasks` 列出可恢复项并可选恢复（能力上限在 ADR 写清） | **2026-05-01**：`reconcilePendingAgentLoopCheckpointFromDexie` + `useAiChat` 挂载水合；**多 tab**：过期 `taskId` 清除后递归再拉「最新可续跑」；**deferred 桥**：`aiChatPendingAgentLoopFingerprint` 使 handoff 水合立刻进 `deferredAiRuntime`；Vitest（reconcile / 双 hook / `ReadyWorkspace.runtime` 切片）；Playwright `tests/e2e/aiAgentLoopHandoffAfterReload.spec.ts`（刷新 + 第二 tab）；`docs/architecture/ai-durable-agent-loop-recovery-scope.md` |

### 2.4 代码锚点

- `src/ai/chat/agentLoopCheckpoint.ts`、`src/ai/tasks/TaskRunner.ts`
- `src/hooks/useAiChat.ts`（resume / `persistAgentLoopCheckpoint`）
- 任务列表 UI（如 `AiChatAlertsPanel` / embedding 卡片侧任务展示，以当前实现为准）

### 2.5 风险

- **双写**：session memory 与 `ai_tasks` 终态不一致 → 每切片定义「权威源」与收敛顺序。
- **架构热点**：`useAiChat.ts` 已接近 guard 上限 → 新逻辑优先 **controller / service**，避免继续堆叠。

### 2.6 T1 收口（工程化）

- **状态**：T1-a / T1-b / T1-c 主交付已完成；执行与验收命令见归档 **[T1 执行记录（2026-05-01）](../archive/milestone-records/T1-Durable任务可操作闭环-执行记录-2026-05-01.md)**。
- **合并前最小集**（与 §6 一致，且含 E2E）：`typecheck` → `check:docs-governance`（若动文档）→ `check:architecture-guard` → `check:transcription-text-telemetry-contract`（若动遥测键）→ 相关 Vitest → **`npm run build` + `npm run test:e2e:chromium`**（本地快速）或 **`npm run test:e2e`**（三引擎，与 CI `e2e-critical-paths` 对齐）→ `gate:release-evidence:core`（若触发布证据）。
- **说明**：Playwright 与 `dist` 一致；仅改 TS 未 build 时请先 `npm run build` 再跑 E2E（见 `tests/e2e/aiAgentLoopHandoffAfterReload.spec.ts` 文件头注释）。

## 3. 轨道 T2 — 执行层权限隔离 v1（tool × scope × trust × quota）

### 3.1 目标

所有 **写路径与高风险路径**（含后台旁路）走统一 **policy 判定**，输出可审计的 **allow / ask / deny**（及 reason），并可挂 **配额**（频率、批量大小、每会话写入上限等可渐进）。

### 3.2 现状基线

- 主链工具门控：`useAiChat.toolDecisionPipeline.ts`、`useAiChat.confirmExecution.ts`、destructive gate。
- F4 最小收口：后台记忆抽取 sandbox + 审计（修复计划 F4）；扩展见 [F4-扩展入口-capability-isolation-epic-2026-05-01.md](./F4-扩展入口-capability-isolation-epic-2026-05-01.md)。

### 3.3 规划切片

| 切片 | 交付物 | 验收 |
|------|--------|------|
| **T2-a** | **策略矩阵 v0 文档**（scope/trust/quota 维度 + 与现有 `toolPreferences` / safety 的优先级） | **[ai-execution-capability-strategy-matrix-v0.md](../../architecture/ai-execution-capability-strategy-matrix-v0.md)**（2026-05-01）；评审以架构 README 与合并前门禁为准 |
| **T2-b** | **统一决策入口**：主链 `onToolCall` 前与 F4 扩展入口共用同一 `resolveXxxPolicy` 函数族 | **`src/ai/policy/resolveExecutionPolicy.ts`**（`resolveUserDirectivePolicyDecision` + `resolveAiChatBackgroundMemorySandboxPolicy`）；`resolveExecutionPolicy.test.ts` + 既有 `toolDecisionPipeline` / `backgroundMemory` 回归 |
| **T2-c** | **quota 最小实现**（每 `conversationId` 后台记忆成功 flush 计数，behind-flag） | `featureFlags.aiBackgroundMemorySessionWriteQuota*`、`BackgroundMemoryExtractor.flushQuotaGate`、审计 `session-write-quota-exceeded`；release evidence `quotaSkipped` |

### 3.4 代码锚点

- `src/hooks/useAiChat.backgroundMemory.ts`、`src/ai/sandbox/`（现有 sandbox 类型与 profile）
- `src/hooks/useAiToolCallHandler.ts` 及各类 adapter（写入前 policy 钩子落位点）

### 3.5 依赖

- 与 **T1** 弱耦合；可与 F4 Batch A/B **并行**，但需避免两处各写一套策略字符串。

## 4. 轨道 T3 — 变更事务化（preview / dry-run / commit）

### 4.1 目标

多步写操作具备 **用户可理解的单笔变更单**：先展示将发生什么（preview），可选演练（dry-run），确认后提交（commit），失败语义与 **B2 rollback 边界**一致且不无限扩张 inverse patch。

### 4.2 现状基线

- `propose_changes` + `executeConfirmedProposedChangeBatch` + B2 白名单 rollback（见 [ADR 0025](../../adr/0025-ai-propose-changes-b2-rollback-boundary.md) 与 `docs/architecture/ai-chat-tool-confirm-idempotency.md`）。

### 4.3 规划切片

| 切片 | 交付物 | 验收 |
|------|--------|------|
| **T3-a** | **统一变更描述结构**（与 UI 预览行对齐的 DTO，可先仅覆盖 propose 子集） | [`ai-change-transaction-preview-v1.md`](../../architecture/ai-change-transaction-preview-v1.md) + `src/ai/changeset/aiChangeTransactionPreviewV1.ts`；单测；默认 UI 路径未改 |
| **T3-b** | **dry-run 钩子**：对只读校验与参数校验集中调用（不写库） | `src/ai/chat/toolCallDryRun.ts`（`dryRunToolCallForConfirm`）+ `useAiChat.confirmExecution`；单测；[`ai-change-transaction-preview-v1.md`](../../architecture/ai-change-transaction-preview-v1.md) §T3-b |
| **T3-c** | **commit 与审计**绑定：与 `markExecutedRequestId`/审计顺序策略一致 | `finalizeHumanToolCallConfirmOutcome` / `finalizeHumanProposeChangesParentConfirmSuccess`（`useAiChat.confirmExecution.ts`）；单测顺序 + 既有 propose 回归；[`ai-chat-tool-confirm-idempotency.md`](../../architecture/ai-chat-tool-confirm-idempotency.md) §实现锚点 |

### 4.4 依赖

- **强依赖** B2 边界文档；**建议**在 T1-a 后再上大 UI，避免「任务卡死 + 大额 preview」双重不确定。

## 5. 轨道 T4 — 失败分流与运维信号

### 5.1 目标

失败可 **分类**（重试 / 澄清 / 人工接管 / 放弃），关键比率进入 **release evidence 或 audits**，与 agent-evals **trace 断言**可增量衔接。

### 5.2 现状基线

- 工具决策审计、`executionProgress`、`proposeRollback`、agent-evals `--assert-audit-trace`（修复计划 C）。
- `export:release-evidence:ai-task-snapshots` 等（修复计划 F3）。

### 5.3 规划切片

| 切片 | 交付物 | 验收 |
|------|--------|------|
| **T4-a** | **`toolDecisionFailureReason.ts`**（`NON_PERSISTED_*` / 元数据 reason 枚举 / `TOOL_DECISION_REASON_FAILURE_TRIAGE`）+ [`ai-chat-tool-confirm-idempotency.md`](../../architecture/ai-chat-tool-confirm-idempotency.md) §T4-a | 单测 `src/ai/chat/toolDecisionFailureReason.test.ts`；`useAiChat.toolAudit` 导入白名单无漂移 |
| **T4-b** | **`toolDecisionFailureSignals`**：`generate-release-evidence-bundle.mjs` + `toolDecisionFailureReason.bundle.mjs`；含 triage 分布、部分失败行数、`proposeRollback.errorCount` 分桶及 durable handoff 率 | `scripts/generate-release-evidence-bundle.test.ts`；`evidenceIndex` 含 `t4.tool-decision-failure-signals.v1` |
| **T4-c** | **`aiToolCallExecutorAutoRetryEnabled`** + `shouldRetryToolCallExecutorThrow`；仅人工确认单工具路径、非破坏性、抛错/超时一次重试 | `useAiChat.confirmExecution.autoRetry.test.ts`；默认 `false` |

### 5.4 依赖

- 与 **T1** 的终态字段、`handoffReason` 一致；避免与 T3 的 commit 语义打架。
