# 现场执行顺序：语言-正字法-字体-互操作（2026-03-31）

> 目的：把 CASE 01-07 按最省切换成本的顺序排好，便于一次性完成手工验收并同步留证。

## 一、执行原则

1. 先确认自动化与构建基线已通过。
2. 优先按“同一项目、同一媒体、同一正字法配置”连续执行，减少环境切换。
3. 每完成一个 CASE 立即保存证据，不要最后补录。

## 二、推荐执行顺序

### Step 0 基线确认

1. 打开 [archive/manual-validation/手工验收执行记录-语言正字法字体互操作-2026-03-31.md](../archive/manual-validation/%E6%89%8B%E5%B7%A5%E9%AA%8C%E6%94%B6%E6%89%A7%E8%A1%8C%E8%AE%B0%E5%BD%95-%E8%AF%AD%E8%A8%80%E6%AD%A3%E5%AD%97%E6%B3%95%E5%AD%97%E4%BD%93%E4%BA%92%E6%93%8D%E4%BD%9C-2026-03-31.md)
2. 确认以下结果已存在：
   `npm run typecheck`
   `npm run build`
   orthography/font/interop 门禁批次
3. 在证据目录下新建本次会话子目录，例如：
   `docs/execution/archive/manual-validation/evidence/orthography-interop-2026-03-31/session-01/`
4. 建议在会话目录下先建立以下占位结构：
   `CASE-01-import/`
   `CASE-02-ai-assistant/`
   `CASE-03-builder-font-risk/`
   `CASE-04-layer-menu-label/`
   `CASE-05-eaf-roundtrip/`
   `CASE-06-textgrid-downgrade/`
   `CASE-07-plaintext-bidi/`

### Step 1 先跑写回链路

建议先做：

1. CASE 01 import 路径 source -> target bridge 命中
2. CASE 02 AI / assistant fallback source orthography 正确

原因：

1. 两条都依赖“源/目标 orthography 不同”的同类样本。
2. 一次准备好 source/target 对照项目后可以连续执行，不必反复改配置。

建议证据：

1. 导入前后文本对照截图
2. AI 写回前后截图
3. assistant segment timeline 写回前后截图

### Step 2 再跑字体风险链路

建议继续做：

1. CASE 03 Builder 默认字体复杂脚本高风险标签
2. CASE 04 Layer 菜单与 Builder 标签一致

原因：

1. 两条都围绕同一个复杂脚本草稿和同一套字体选择。
2. 做完 Builder 后直接切到 Layer 菜单即可，不必重新建数据。

建议证据：

1. Builder 截图
2. LayerStyleSubmenu 截图
3. 刷新后缓存命中截图

### Step 3 最后跑 interop 与 plain-text 边界

建议最后做：

1. CASE 05 EAF metadata round-trip
2. CASE 06 TextGrid downgrade 忽略未知字段
3. CASE 07 plain-text bidi 安全边界

原因：

1. 这三条需要导入导出样本和文本文件留档，单独一组最清晰。
2. EAF / TextGrid / plain-text 可以共用一套最终项目数据做导出检查。

建议证据：

1. 导出文件副本
2. 导入成功界面截图
3. metadata 或文本内容比对截图

## 三、建议耗时

1. Step 0：5 分钟
2. Step 1：10-15 分钟
3. Step 2：10 分钟
4. Step 3：15-20 分钟

总计：约 40-50 分钟

## 四、证据命名规范

建议格式：

`CASE-xx-序号-动作-结果.扩展名`

示例：

1. `CASE-01-01-before-import.png`
2. `CASE-01-02-after-import.png`
3. `CASE-02-01-ai-before.png`
4. `CASE-02-02-ai-after.png`
5. `CASE-05-01-export.eaf`
6. `CASE-06-01-textgrid-with-unknown-fields.TextGrid`
7. `CASE-07-01-plain-text-export.txt`

命名要求：

1. 同一 CASE 内按时间顺序递增编号。
2. 截图、导出文件、文本副本都使用同一 CASE 前缀。
3. 若存在失败复测，追加 `-retry-01`、`-retry-02` 后缀。

## 五、执行后动作

1. 回填 [archive/manual-validation/手工验收执行记录-语言正字法字体互操作-2026-03-31.md](../archive/manual-validation/%E6%89%8B%E5%B7%A5%E9%AA%8C%E6%94%B6%E6%89%A7%E8%A1%8C%E8%AE%B0%E5%BD%95-%E8%AF%AD%E8%A8%80%E6%AD%A3%E5%AD%97%E6%B3%95%E5%AD%97%E4%BD%93%E4%BA%92%E6%93%8D%E4%BD%9C-2026-03-31.md)
2. 生成 go / no-go 结论：
   使用 [封账结论-语言正字法字体互操作-2026-03-31.md](%E5%B0%81%E8%B4%A6%E7%BB%93%E8%AE%BA-%E8%AF%AD%E8%A8%80%E6%AD%A3%E5%AD%97%E6%B3%95%E5%AD%97%E4%BD%93%E4%BA%92%E6%93%8D%E4%BD%9C-2026-03-31.md)

## 六、快速判定

1. 任一写回路径出现“应桥接却写原文”：直接 no-go。
2. 复杂脚本自定义字体出现“已验证”而非“高风险”：直接 no-go。
3. EAF / TextGrid 导入因 metadata 异常失败：直接 no-go。
4. plain-text 导出/二次导入出现 bidi 控制字符污染：直接 no-go。
