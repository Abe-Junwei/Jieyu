## 背景

- 审查依据：`docs/architecture/仓库现状与代码地图.md`，当前 git status 中的新增文件与 `docs/execution/plans/托管实时协同-Supabase完整落地方案-2026-04-17.md`。
- 范围聚焦：**Supabase 协作云桥接**。其余主线（OTel/trace、父 unit、文献轨）本轮不在行动项内，只在"外部依赖"里提一下。

## 产出物

**唯一新增文件**：`docs/execution/plans/协作云桥接审查改进方案-2026-04-18.md`

- 遵循 `.cursor/rules/jieyu-docs-governance.mdc`：放入 `docs/execution/plans/`、中文命名带日期。
- front matter：`doc_type: execution-plan`，`status: proposed`，`owner: collaboration-cloud`。
- 本方案只写"做什么 + 验收句"，不承担业务事实源职责（保留给 `docs/architecture/`）。

## 主要问题（来自本轮审查）

- **胖 hook**：[`src/hooks/useTranscriptionCloudSyncActions.ts`](src/hooks/useTranscriptionCloudSyncActions.ts) 约 1390 行，内嵌 `project_presence` / `projects` / `project_members` 的 Supabase 直连、自动快照 `setInterval`、冲突票据、LinguisticService 调度。
- **UI 孤岛**：[`src/components/transcription/CollaborationSyncBadge.tsx`](src/components/transcription/CollaborationSyncBadge.tsx) 与 [`src/components/transcription/CollaborationCloudReadOnlyBanner.tsx`](src/components/transcription/CollaborationCloudReadOnlyBanner.tsx) 无任何调用方；`CollaborationCloudPanel.directory` 未在 `TranscriptionPage.ReadyWorkspace.tsx` 被传入。
- **重复实现**：`subscribeChannel` 在 [`src/collaboration/cloud/CollaborationSyncBridge.ts`](src/collaboration/cloud/CollaborationSyncBridge.ts) 与 [`src/collaboration/cloud/CollaborationPresenceService.ts`](src/collaboration/cloud/CollaborationPresenceService.ts) 各写一份。
- **旧 Runtime 语义模糊**：`collaborationBetaRuntime` / `CrossDevice` / `MultiReplica` / `Promotion` / `TransactionSync` 在 cloud 生产路径中**未被调用**，仅自身测试持有引用。
- **测试缺口**：无覆盖"入站 → 冲突检测 → 审计落库 → state commit"完整路径的 hook 级测试。
- **门禁割裂**：`gate:collaboration-cloud` 未出现在 `.github/workflows/ci.yml`；`gate:m14-collaboration-promotion` 未显式依赖它。

## 行动项

### P0（两周内，低风险高收益）

- **P0-1 露出协议守卫 UI**：在 [`src/pages/TranscriptionPage.ReadyWorkspaceLayout.tsx`](src/pages/TranscriptionPage.ReadyWorkspaceLayout.tsx) 或 `transcription-toolbar` 上挂入 `CollaborationSyncBadge` + `CollaborationCloudReadOnlyBanner`，消费 `useTranscriptionData().derivedApi.{collaborationSyncBadge, collaborationProtocolGuard}`。
- **P0-2 抽 `subscribeChannel`**：新建 `src/collaboration/cloud/realtimeSubscription.ts`，`CollaborationSyncBridge` 与 `CollaborationPresenceService` 改为共用。
- **P0-3 下沉 Supabase 直连**：从 `useTranscriptionCloudSyncActions` 中把 `project_presence` upsert 移到 `CollaborationPresenceService.persistPresencePatch`，把 `projects` / `project_members` 相关调用合并到新增的 `CollaborationDirectoryService`（或并入 `CollaborationAssetService`，如规模小）。
- **P0-4 directory 能力二选一**：要么在 ReadyWorkspace 将 `directory` prop 传入 `CollaborationCloudPanel`，要么在该组件暂时去除 `directory` 分支，避免冷冻代码。

**验收**：`grep -R "from 'src/integrations/supabase/client'" src/hooks` 返回 0；badge 与 banner 出现在转写工作区；既有测试全绿。

### P1（一个月窗口，结构治理）

- **P1-1 Runtime 角色地图**：新增 `docs/architecture/collaboration-runtime-map.md`，逐一标注 `collaboration*Runtime.ts` 的当前角色（生产 / 门禁 / 历史），明确哪些可被 `cloud/` 取代。
- **P1-2 端到端 hook 测试**：新增 `src/hooks/useTranscriptionCloudSyncActions.applyRemote.test.tsx`，覆盖 "入站 → `collaborationConflictRuntime.detectCollaborationConflicts` → `createConflictResolutionLog` → 审计写入 → commit revision" 一条 happy / 一条 conflict 路径。
- **P1-3 CI 收口**：在 `.github/workflows/ci.yml` 新增独立 job 运行 `gate:collaboration-cloud`；`gate:m14-collaboration-promotion` 中显式串联该脚本。
- **P1-4 文档时效**：刷新 `docs/execution/plans/README.md` 与 `docs/execution/release-gates/README.md` 的 `last_reviewed`；把 `托管实时协同-Supabase完整落地方案-2026-04-17.md` 的 `status` 由 `in_progress` 更新到实际阶段。

**验收**：CI 上出现绿色 `gate:collaboration-cloud` job；新测试覆盖率报告可见入站-冲突路径；Runtime 地图中每个文件都有状态标签。

### P2（两个月窗口，深入集成）

- **P2-1 ConflictResolved 观测钩子**：在 `collaborationConflictRuntime.resolveCollaborationConflicts` 内部发一条观测事件（现阶段仅 `aiTrace` / metrics，等 OTel 主线到位后再升级为真实 span），并将 `traceId` 字段写入 `audit_logs` —— 为 `OTel-Trace透传与ConflictResolved五阶段执行清单` 第四阶段铺垫。
- **P2-2 结构守卫**：在 `scripts/architecture-guard.config.mjs` 增加规则：禁止 `src/hooks/**` 直接引用 `src/integrations/supabase/**`，强制 cloud 层作为唯一桥。
- **P2-3 现状入库**：将 P0–P1 稳定下来的接线规则从本执行方案**抽离**到 `docs/architecture/collaboration-cloud.md`，本方案仅保留"下一步"，符合仓库 governance 单一事实源原则。

**验收**：`architecture-guard` 覆盖 hooks→supabase 规则；`docs/architecture/collaboration-cloud.md` 成为引用基线。

## 外部依赖/边界

- OTel 真正贯通 `fetch` 与 root span 的任务**不在本方案**，P2-1 只留钩子，正式纳入 `OTel-Trace透传与ConflictResolved五阶段执行清单`。
- 父 unit 写路径治理与文献伪时长轨不会直接改动 cloud 模块，但会共用 ReadyWorkspace props builder；P0-1 露出 badge 时需避免与父 unit 方案对 toolbar 的并发改动。

## 不做的事

- 不重写 `collaborationBetaRuntime` 等旧 runtime 的业务逻辑，只给角色标签与取舍建议。
- 不调整 Supabase schema（`supabase/sql/001_collaboration_foundation.sql`）。
- 不升级 `CollaborationAuditLogService` 到独立审计表。
