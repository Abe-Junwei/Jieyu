# 标注词典路线改进计划

## 主要结论

- 当前主方向正确：`/annotation` 复用转写读写链、`/lexicon` 承担项目词典、`/corpus` 保持只读集选与出站，符合 SSOT 与开放门槛约束。
- 需要先收口文档事实：`docs/execution/plans/标注页与词典页开发路线图-2026-04-25.md` 已成为 L0 真值，但多个 L1/语料文档仍引用旧版 `§4.2.1`、`§4.7`、`§6.5` 等章节号。
- 需要修正落地方案中的代码 API 假设：`docs/execution/plans/标注页与词典页-落地方案-2026-04-25.md` 提到 `LeipzigValidator.validateGlossLine()`、`AutoGlossService.matchToken()`、`FlexService.serializeLexemeEntriesAsFlexText()`，当前代码实际分别是 `validateGloss()`、`glossUnit()`、`exportToFlextext()`。
- 依赖策略应保持克制：一期优先使用已有 `minisearch`、`@tanstack/react-virtual`、Dexie、wavesurfer、zod；`leipzig` 可作为 IGT 展示 Spike 或参考实现，不建议阻塞 M1；LIFT 暂无成熟 JS 包证据，建议自建小型 exporter/importer 并用 SIL 标准样例测试。

## 建议改进顺序

1. 文档事实收口：把旧路线图 `docs/execution/plans/标注页与词典页开发路线图-2026-04-22.md` 的 YAML 改为 `superseded`，并把 `README`、语料库路线图、产品定位文中的旧章节引用改到 2026-04-25 路线图 `§2.2` 或治理补充规范 `#cross-page-event-contract-v1`。
2. 落地方案 API 对齐：修正 `docs/execution/plans/标注页与词典页-落地方案-2026-04-25.md` 中不存在的 API，明确若需要 token 级建议 UI，应新增只读候选方法，而不是直接复用会写库的 `AutoGlossService.glossUnit()`。
3. M1 范围降风险：将“首版直接放开更多编辑字段”拆成 M1a/M1b，先完成原文/译文只读、token gloss/POS、保存、键盘与段播放，再接 morpheme 边界调整、Validator 模板和二次自动分词。
4. 词典页优先升级已有能力：基于 `src/pages/LexiconPage.tsx` 已有只读列表、详情和命中语段跳转，先补 `minisearch` hook、列表虚拟滚动、返回状态，再做编辑表单和导出。
5. 事件合同先落公共工具：在 `src/utils/appShellEvents.ts` 增加 workspace 事件类型和派发/订阅 helper，避免各页自造 `CustomEvent` 名称、payload 和幂等策略。
6. 增加验收护栏：为 API 文档一致性、R1-R8 联评、深链 query 键、词典搜索、lexeme jump targets、标注 token gloss 保存各补聚焦测试或脚本检查。

## 外部对标取舍

- FLEx：继续作为标注/词典核心闭环与 LIFT/FLExText 互操作参照，但不要照搬桌面配置复杂度。
- Webonary：适合借鉴语义域浏览、多语言搜索、headword 优先排序，不适合作为编辑器架构参照。
- INCEpTION：适合借鉴关系标注、推荐/人工确认、协作审阅 UX；JVM/UIMA 架构不适合直接嵌入。
- Leipzig.js：适合 IGT 只读渲染 Spike；编辑态仍应保留当前计划的 React 受控 input 矩阵。
- FlexSearch/Fuse/Orama：当前不建议替换 `minisearch`；只有在项目级语料索引出现可复现性能瓶颈后再做时间盒 Spike。
