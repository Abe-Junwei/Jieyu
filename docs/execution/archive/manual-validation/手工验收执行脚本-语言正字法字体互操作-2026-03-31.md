## CASE 01：import 路径 source -> target bridge 命中

目标：确认导入文本进入已有目标层时，会按 source orthography -> target orthography 桥接，而不是直接落原文。

步骤：

1. 准备一个带 source orthography metadata 的导入样本。
2. 在项目中选中目标层，其 orthography 与样本 source orthography 不同。
3. 执行导入。
4. 观察导入后的文本是否为变换后结果。

通过标准：

1. 文本为预期 target orthography 形式。
2. 导入未报错。
3. 未命中 bridge 时保留原文，不出现空文本或异常中断。

## CASE 02：AI / assistant fallback source orthography 正确

目标：确认未显式选中来源层时，AI 写回与 segment timeline 听写写回都回退到首个转写层 orthography。

步骤：

1. 创建一个含转写层和翻译层的项目，两层 orthography 不同。
2. 清空当前显式 source layer 选择。
3. 触发 AI 翻译写回一次。
4. 触发 assistant segment timeline 听写写回一次。

通过标准：

1. 两条路径写入结果一致遵守 source -> target bridge。
2. 不出现“一条变换、一条原文”的分叉。

## CASE 03：Builder 默认字体复杂脚本高风险标签

目标：确认复杂脚本自定义字体在 Builder 里不会被误显示为“已验证”。

步骤：

1. 打开一个 Arabic / Devanagari / Tibetan 类 orthography 草稿。
2. 把默认字体切到一个非 preset 的本地或自定义字体。
3. 等待字体校验结果出现。

通过标准：

1. 若 glyph check 通过，显示“高风险”而不是“已验证”。
2. 若字体缺字，显示“缺字”。
3. 最终字体栈仍保留脚本安全 fallback。

## CASE 04：Layer 菜单与 Builder 标签一致

目标：确认字体风险语义在 LayerStyleSubmenu 与 Builder 中一致，不存在一边“已验证”、一边“高风险”。

步骤：

1. 对同一个复杂脚本层选择同一个自定义字体。
2. 分别查看 Builder 与 Layer 字体菜单的标签。

通过标准：

1. 两处标签一致。
2. 刷新页面后缓存命中时标签仍一致。

## CASE 05：EAF metadata round-trip

目标：确认 EAF 导出导入后 provider-neutral metadata 仍可回放。

步骤：

1. 准备一个带 languageId / orthographyId / scriptTag 的项目。
2. 导出为 EAF。
3. 重新导入该 EAF。
4. 检查 tier 对应语言/正字法身份是否保留。

通过标准：

1. 导入成功，无 metadata 解析错误。
2. tier identity 与导出前一致或按合同允许的方式降级。

## CASE 06：TextGrid downgrade 忽略未知字段

目标：确认 TextGrid metadata 中出现未知字段时，导入稳定忽略，同时允许 `bridgeId` 保留。

步骤：

1. 准备一个带 `__jieyu_meta_` suffix 的 TextGrid 样本。
2. 在 metadata JSON 中加入一个未知字段，并保留 `bridgeId`。
3. 执行导入。

通过标准：

1. 导入成功。
2. 未知字段未污染业务层。
3. `bridgeId` 如预期保留。

## CASE 07：plain-text bidi 安全边界

目标：确认 plain-text 导出复制和再次导入时，不会把 isolate 控制字符带进业务文本。

步骤：

1. 准备一个包含 RTL / mixed-direction 内容的项目。
2. 导出 plain-text 族格式或执行复制文本操作。
3. 把结果重新用于搜索、复制检查或再次导入。

通过标准：

1. 显示顺序正确。
2. 搜索与复制结果不含可见或不可见的 isolate 污染。
3. 二次导入文本内容与预期一致。
