# AI 智能体工业级对标改进计划书（2026-04-25）

## 0. 目的与范围

本计划将两类输入合并为一份统一执行方案：

1. 前一轮 `AI Agent Industrial Gap Assessment` 与既有改进路线；
2. 对业内成熟智能体体系的调研结论（OpenAI / Anthropic / Google / Microsoft）。

本计划用于指导 `Jieyu` 智能体从“治理增强型助手”升级为“可托付、可审计、可回放的工业级研究协作智能体”。

---

## 1. 现状判断（Jieyu 当前基线）

### 1.1 已具备能力

- 已有结构化偏好与指令治理：`responsePreferences` / `toolPreferences` / `safetyPreferences` / `sessionDirectives` / `directiveLedger`。
- 已有执行链路门控：`intent confidence gate`、`destructive gate`、`ask_first`/`never` 等策略接入执行路径。
- 已有可观测闭环雏形：审计日志、`release evidence`、协调协议审计（C7）。
- 已有领域上下文优势：本地工具与时间轴/语段读模型深度耦合，具备场景优势。

### 1.2 关键短板

- 缺少任务级 durable orchestration（checkpoint / resume / rollback / handoff）。
- 缺少 trajectory-level 评测体系（不仅测最终回答，还测工具选择与中间步骤）。
- 缺少用户可控治理界面（查看/编辑/失效 memory 与 directives）。
- 缺少事务化多步执行语义（preview -> dry-run -> commit）。
- 缺少更强 runtime capability isolation（tool × scope × trust × quota）。

---

## 2. 行业对标结论（成熟产品共性）

> 下述结论用于“架构方向校准”，不是对外合规声明。

### 2.1 OpenAI 体系（Agents / Guardrails / Tracing / Evals）

- 强调 control loop 与工具契约化调用。
- 强调 guardrails 与 tracing、eval-driven 的上线方式。
- 结论：Jieyu 需补强“评测优先 + 轨迹可观测”。

### 2.2 Anthropic 体系（可信 agent 框架）

- 强调 human-in-control、透明执行、最小权限、隐私与注入防护。
- 结论：Jieyu 需补强“审批中心 + 可解释动作计划 + 权限隔离”。

### 2.3 Google Gemini Agents（agentic workflows）

- 强调多步规划、长上下文、显式系统指令对复杂任务的稳定增益。
- 结论：Jieyu 需补强“规划模板化 + 长任务状态管理 + 工具链一致性”。

### 2.4 Microsoft（Semantic Kernel / AutoGen）

- 强调企业级支持、事件驱动编排、长流程、人机协同与治理。
- 结论：Jieyu 需补强“生产级流程编排 + 审批与运维一体”。

---

## 3. 目标架构（统一五层）

1. **Instruction & Memory Layer**
  事实、长期偏好、会话指令、术语映射、pinned directive digest。
2. **Policy Layer**
  输出策略、工具策略、安全策略、扩展信任策略与优先级模型。
3. **Planner & Orchestrator Layer**
  任务分解、阶段推进、依赖调度、失败恢复、人工接管。
4. **Execution Layer**
  工具沙箱、权限边界、事务执行（preview/dry-run/commit/rollback）。
5. **Evidence & Operations Layer**
  评测、审计、发布证据、调试可视化、线上质量监控。

---

## 4. 统一改进路线图（四阶段）

## Phase A（0-6 周）：评测与发布基线（P0）

### 目标

将“可迭代”升级为“可验证可发布”。

### 关键工作

- 建立 `agent eval harness`（首批 30-50 条黄金任务）。
- 引入 trajectory-level 评分：工具选型、门控正确性、恢复路径。
- 将评测与 release evidence 接入发布门禁。

### 交付物

- `agent-eval` 脚本与报告规范。
- CI 门禁规则（PR 阶段）。
- 审计字段字典 v1。

### 验收

- 黄金任务通过率 >= 90%（可分层阈值）。
- 关键场景（语言一致性/安全阻断）回归可视化。

## Phase B（6-12 周）：用户可控治理（P0/P1）

### 目标

把“内部治理能力”开放为“用户可见可控能力”。

### 关键工作

- 上线 `Directive & Memory Console`（查看/编辑/禁用/过期）。
- 上线 `Action Approval Center`（高风险动作前置确认）。
- 上线“为何执行/为何阻断”解释面板。

### 验收

- 用户可在 3 步内修正偏好。
- 高风险动作 100% 进入可解释确认流。

## Phase C（12-20 周）：执行工业化（P1）

### 目标

多步工具执行具备事务语义与恢复能力。

### 关键工作

- 执行语义升级：`preview -> dry-run -> commit`。
- 多步写操作支持 partial rollback。
- runtime policy enforcement（scope/trust/quota）下沉到执行层。

### 验收

- 多步任务成功率提升（目标 +15%）。
- destructive 误执行接近 0。

## Phase D（20-36 周）：自治与运维增强（P2）

### 目标

从“可控执行”进化为“可靠自治”。

### 关键工作

- durable orchestration：checkpoint/resume/handoff。
- 失败智能分流：retry/clarify/rollback/handoff。
- 线上质量看板与告警机制。

### 验收

- 长任务完成率持续上升。
- 人工干预率下降且安全指标不恶化。

## 5. 关键专项（跨阶段推进）

1. **评测专项**：黄金任务 + 回放集 + 发布阈值。
2. **策略可控专项**：Memory/Directive 控制台。
3. **审批专项**：高风险动作审批中心。
4. **事务执行专项**：多步写操作事务化。
5. **权限隔离专项**：tool × scope × trust × quota。
6. **恢复专项**：失败自动恢复策略。
7. **证据专项**：审计与 release evidence 深化。

---

## 6. 指标体系（统一 KPI）

### 6.1 质量指标

- task success rate（分场景）
- preference adherence rate（语言/格式/术语）
- clarification usefulness rate

### 6.2 安全指标

- policy block precision / recall
- unauthorized action rate
- destructive incident count

### 6.3 执行可靠性指标

- multi-step completion rate
- rollback success rate
- mean time to recovery (MTTR)

### 6.4 工程发布指标

- eval pass rate
- evidence completeness
- regression escape rate

## 7. 优先级与里程碑

### M0（立即）

- 锁定黄金任务集 v1。
- 将 eval 接入 PR 门禁。

### M1（4 周内）

- Action Approval Center MVP（仅高风险）。
- Directive/Memory Console MVP（至少支持停用）。

### M2（8-12 周）

- 事务执行链路与 rollback v1。
- runtime capability policy v1。

### M3（>12 周）

- durable orchestration 与长任务恢复。

---

## 8. 风险与缓解

### R1 规则冲突膨胀

- 缓解：统一策略优先级矩阵；冲突写入 `directiveLedger` 并可解释。

### R2 可用性下降（过度阻断）

- 缓解：分级阻断（block/confirm/clarify）与误判回放评测。

### R3 记忆污染

- 缓解：可编辑、可过期、可停用、可追溯。

### R4 回归漏检

- 缓解：强制 trajectory-level eval + release gate。

---

## 10. 执行原则

1. 先评测后扩展：无 eval 不放量。
2. 先可控后自治：先有人类控制，再提升自动化。
3. 先可解释后优化：先能解释每一步，再追求效率。
4. 所有高风险能力默认 behind-flag。

本计划是执行方案，不替代 `docs/architecture/` 与代码作为长期事实源。

## 11. 可执行任务包（M0 / M1）

### 11.1 M0 任务包（立即，1-2 周）

#### WP-M0-1：黄金任务集与评测脚手架

- **目标**：完成 Agent 黄金任务集 v1（30-50 条）与可重复执行脚本。
- **负责人占位**：`ai-governance-owner`
- **核心文件**：
  - `scripts/agent-evals/suite.v1.json`
  - `scripts/run-agent-evals.mjs`
  - `src/hooks/useAiChat*.test.ts`*（补轨迹断言）
  - `scripts/generate-release-evidence-bundle.test.ts`（补评测映射）
- **完成定义（DoD）**：
  - 可本地一键跑完整评测；
  - 输出包含 pass/fail、错误分类、回归 diff。

#### WP-M0-2：PR 门禁接入（Eval + Evidence）

- **目标**：把评测结果接入 PR 阶段门禁，低于阈值即失败。
- **负责人占位**：`release-governance-owner`
- **核心文件**：
  - `package.json`（新增 `report:agent-evals` / `check:agent-evals` 命令）
  - `scripts/check-release-evidence-*.mjs`
  - CI 配置文件（若仓库已有 workflow，则在原文件扩展）
- **完成定义（DoD）**：
  - PR 可见门禁结果；
  - 失败时能定位到具体任务样本与责任模块。

#### WP-M0-3：审计字段字典 v1

- **目标**：统一 `ai_user_directive`_*、`ai_response_policy_resolution`、`ai_tool_decision_`* 字段口径。
- **负责人占位**：`observability-owner`
- **核心文件**：
  - `docs/execution/release-gates/release-evidence/README-2026-04-25.md`
  - `scripts/generate-release-evidence-bundle.mjs`
- **完成定义（DoD）**：
  - 字段定义、来源、聚合逻辑、门禁规则可追溯；
  - 无歧义字段被 evidence 消费。
