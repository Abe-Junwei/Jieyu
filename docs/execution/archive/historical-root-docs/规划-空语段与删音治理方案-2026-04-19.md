# 空语段与删音治理方案

## 结论

本项目最适合采用“**先标记跳过，后按需删除**”的双层方案：

1. **主路径：非破坏性空段标记**
   - 对语段新增专门的处置语义：`normal | silence | noise | skip`
   - 一旦标记为 `silence/noise/skip`，后续转写、AI 推荐、批处理、听写导航默认跳过
   - 时间坐标保留，不改动后续段的时间轴

2. **副路径：可选删除能力**
   - 保留现有“整条音频删除但不丢时间线”的能力
   - 新增“删除选中音频区间”时，优先做成**逻辑删除/导出时裁切**，不要直接重写原始 Blob

## 为什么这是本仓最合适的方案

### 1. 现有数据模型天然偏向“标记而非重写音频”

仓库已经把“时间单元”和“媒体是否存在”分开建模：

- 整条音频删除后，文本时间线和语段仍被保留
- 占位媒体会顶上，继续维持 media/document 时间模式

这意味着本项目的核心资产是“**语段坐标与标注状态**”，不是对原始音频做频繁 destructive edit。

### 2. 与成熟工具一致

- Praat / TextGrid 以区间标注为核心，静音和非语音首先是“带标签的 interval”
- Audacity 也是先用 label 标注噪音/静音区域，再决定是否连同音频一起删除

因此最稳妥的产品路径不是“先删音频”，而是“**先把非语音区域作为一等对象标出来**”。

### 3. 破坏性删音会引入大量连锁复杂度

如果直接删掉中间一段音频，需要同时处理：

- 后续所有语段时间整体左移
- 段内引用、对齐、导出、回放范围重映射
- 协作同步与回滚
- 原始素材可追溯性

这对当前仓库来说成本高、风险大，而且不符合田野语言资料“原始录音尽量保真”的原则。

## 建议的产品形态

### A. 语段级新增“非语音处置”

在选中语段时提供三个动作：

- **标为空白**
- **标为噪音**
- **跳过后续处理**

显示效果：

- 波形和时间线弱化显示
- 右上角 badge 显示“空白 / 噪音 / 已跳过”
- 侧栏支持按该状态筛选

### B. 后续流程默认跳过

以下能力统一忽略被标记语段：

- 自动听写导航
- AI 补写/推荐
- 批量处理
- “下一条未处理语段”跳转
- 完成率统计（单独计为“已处置但不需转写”）

### C. 删除能力分三级

1. **删语段**：仅删标注行，不改音频
2. **删整条音频**：已有能力，继续保留
3. **删选中音频区间**：后续增强项，默认只生成逻辑掩码或导出裁切结果

## 数据设计建议

不要复用已有 `annotationStatus` 表达空段/噪音，因为它当前承载的是“转写进度”。

建议新增独立字段，例如：

```ts
segmentDisposition?: 'normal' | 'silence' | 'noise' | 'skip';
skipProcessing?: boolean;
```

如需更细原因，可附加：

```ts
skipReason?: 'blank' | 'background_noise' | 'overlap' | 'unusable';
```

这样可以避免把“是否需要处理”与“处理到什么阶段”混在一起。

## 实施优先级

### Phase 1（建议立即做）

- 语段标记：空白 / 噪音 / 跳过
- 导航、AI、批处理默认跳过
- 时间线与侧栏显示 badge

### Phase 2

- 批量标记空段
- 结合 VAD/声学结果自动建议“疑似静音区”

### Phase 3

- 选区逻辑删音
- 导出时选择“保留原时间轴”或“压缩时间轴”

## 最终推荐

**本项目最优方案：以“非破坏性空段标记 + 流程跳过”为主，整条音频删除为辅，区间删音作为后续增强而不是主流程。**

这条路线最符合当前仓库的数据结构、回滚需求、协作需求和语言资料保真原则。
