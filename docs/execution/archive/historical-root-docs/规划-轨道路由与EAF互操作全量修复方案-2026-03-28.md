# 规划：轨道路由与 EAF 互操作全量修复方案（2026-03-28）

## 背景

本方案用于承接 2026-03-28 这轮“Speaker 管理、Segment 路由、EAF Import/Export、Layer Constraint、状态管理、数据模型”复核结果。

当前仓库基线仍然稳定：

1. `npm run typecheck` 通过。
2. `npm test` 通过。
3. `npm run build` 通过。

但这轮复核确认，仍有一组不会被现有自动化直接拦住、且已经影响互操作正确性与编辑一致性的真实问题，重点集中在：

1. Speaker 指派路由与批量状态同步。
2. segment 创建 / 投影时的 speaker 继承与孤立段控制。
3. EAF 导入导出中的 PARTICIPANT 与 speaker 标识往返一致性。
4. time_subdivision 约束在 V2 CRUD 中的收口。
5. 少量“原计划列为 bug、但实际并未复现或只属于代码整洁性”的项目清理。

本方案只把“已经确认成立的问题”和“必须先缩小复现的问题”纳入执行批次，不再把误报项混进同一条修复链路。

## 范围分级

### P0：本轮必须完成

1. EAF speaker resolution 大小写 / 规范化不一致。
2. EAF 导出 PARTICIPANT 对自由字符串 speakerId 的丢失。
3. 独立转写层在无 inserted utterances 时被跳过。
4. time_subdivision 新建 segment 未继承 parent utterance speakerId。
5. V2 投影 segment 未继承 utterance.speakerId。
6. 独立层在无重叠 utterance 且无当前 speaker focus 时静默创建孤立段。
7. batchSpeakerId 的 utterance/segment 双 effect 覆盖。
8. SpeakerAssignPanel 标题与当前操作对象不一致。
9. split/merge 后未重新收敛 time_subdivision 父边界。
10. createSegment / createSegmentWithContentAtomic 缺少 layer constraint 校验。

### P1：本轮必须完成的验证项

1. `handleAssignSpeakerToSegments()` 仅依赖 `reloadSegments()` 是否足够。
2. EAF 导出筛选问题的真实根因到底是：
   - transcription `time_subdivision` tier 漏导出；
   - 导入阶段 independent tier 在无 utterance 时被跳过；
   - 还是 translation/export filter 不一致。

### P2：本轮做决策，不强制改码

1. `layer_utterances` 是继续物化、还是正式删除。
2. 若继续保留 number 时间戳风格，是否统一把相关条目从“schema bug”降级为“风格一致性清理”。

### 不纳入本轮 bug 批次

以下项目在复核后不再按“功能 bug”执行：

1. [src/contexts/SpeakerRailContext.tsx](../../../../src/contexts/SpeakerRailContext.tsx) 与旧 LayerRailSidebar 侧栏实现现已拆分到 [src/components/SidePaneSidebar.tsx](../../../../src/components/SidePaneSidebar.tsx) / [src/components/SidePaneSidebarLayerRow.tsx](../../../../src/components/SidePaneSidebarLayerRow.tsx)；其 utterance-only 暴露面不是 bug，而是设计边界。
2. “activeUtteranceUnitId 为 segment ID 时误路由到 utterance-only 逻辑”按当前实现未复现；真正的选择模型已经区分 `kind`，不应再用字符串前缀猜类型。
3. “Tier PARTICIPANT 只落到第一个 utterance”未在当前导入实现中成立。
4. `handleVoiceAnalysisResult` 依赖整个 `data` 对象属于依赖精度问题，不是 stale closure 功能 bug。
5. `toggleVoice` 的空依赖数组在 ref 回调模型下是合理写法。
6. `undo/redo` 的 `canUndo/canRedo/undoLabel` 当前已经会因 `_undoRedoVersion` 重渲染，不是非响应式 bug。
7. 多处 `Date.now()` 写入 number 字段不是“number 写进 ISO string schema”的 bug；若要处理，应以“时间戳表示法一致性”单列 cleanup。

## 修复原则

1. 先补最小复现测试，再改代码，避免在路由 / 导入导出链路上盲修。
2. 先修跨层数据正确性，再修面板文案与局部体验。
3. 对于 speaker 相关问题，统一遵循“segment 显式 speakerId 优先，回退 owner utterance.speakerId”的既有模型。
4. 对于 time_subdivision，任何 create / split / merge / import 路径都必须收敛到父 utterance 边界。
5. 对于 EAF，任何 PARTICIPANT 往返问题都必须同时覆盖 import、export、re-import 三段验证，不能只看单向导出。

## 批次 A：先补回归与最小复现

### 目标

把已确认问题先转成可失败的自动化断言，把“待验证项”缩小到可重复场景。

### 需要新增或补齐的测试

1. Speaker 路由与状态：
   - segment 批量指派后，`batchSpeakerId` 不应被 utterance effect 回写覆盖。
   - utterance-only 批量指派路径保持不回退到 segment 写接口。
2. Segment 创建 / 投影：
   - time_subdivision 新建 segment 继承 parent utterance speakerId。
   - `syncUtteranceTextToSegmentationV2()` 投影后保留 speakerId。
   - independent layer 在无重叠 utterance 且无 speaker focus 时不再静默创建孤立段。
3. EAF 往返：
   - import 时 speaker 名称大小写不同仍映射到同一 speaker。
   - export 时 `utterance.speakerId='John'` 这类自由字符串仍能写出 PARTICIPANT。
   - 无 inserted utterances 但有 independent transcription tier annotations 时，仍能导入该层 segments。
4. Layer constraint：
   - `splitSegment()` 后子段仍在 parent utterance 范围内。
   - `mergeAdjacentSegments()` 后合并结果仍在 parent utterance 范围内。
   - 对不允许自由创建 segment 的 layer，`createSegment()` / `createSegmentWithContentAtomic()` 拒绝非法写入。
5. 缩小复现项：
   - EAF export 是否遗漏 transcription `time_subdivision` tier。
   - `reloadSegments()` 是否足以驱动 SpeakerAssignPanel 当前 summary / 选择态刷新。

## 批次 B：Speaker 管理与面板路由修复

### 目标

### B1. 收口 batchSpeakerId 同步逻辑

建议方案：

2. 统一以“当前操作对象”为判断条件：
   - 有 selected segments 时，只根据 segment 集合派生。
   - 否则只根据 utterance 集合派生。
3. 避免在两个独立 effect 中对同一 `setBatchSpeakerId()` 做竞争写入。

### B2. 动态标题与摘要语义

建议方案：

1. 为当时的 SpeakerAssignPanel 入口补 `mode: 'segment' | 'utterance'` 或 `selectionKind` 入参；现行收口点对应 [src/components/SidePaneSidebarSpeakerManagement.tsx](../../../../src/components/SidePaneSidebarSpeakerManagement.tsx) 与 [src/pages/useSpeakerActionRoutingController.ts](../../../../src/pages/useSpeakerActionRoutingController.ts)。
2. 标题规则：
   - segment 模式：`说话人轨道编辑`
   - utterance 模式：`说话人批量编辑`
3. 若 summary 文案里含“语段 / 句段”混用，也一并按当前 mode 调整。

### B3. 验证 `reloadSegments()` 是否足够

建议方案：

1. 不预设这里一定是 bug。
2. 先用测试证明 segment 指派后：

   - `selectedSpeakerSummaryForActions`

3. 只有当 UI 派生状态仍滞后时，才补局部更新逻辑；补也应只补当前选择快照，不要重新引入双写状态。

## 批次 C：Segment 创建与投影链路修复

### 目标

保证新建 / 投影得到的 segment 在 speaker 与归属关系上可用，不再生成静默孤立数据。

### C1. time_subdivision 新建继承父 speaker

建议方案：

1. 在 [src/pages/TranscriptionPage.Orchestrator.tsx](../../../../src/pages/TranscriptionPage.Orchestrator.tsx) 的 time_subdivision 创建分支里，在调用 `createSegmentWithParentConstraint()` 前补继承逻辑。
2. 规则：仅当 `newSeg.speakerId` 为空且 `parentUtt.speakerId` 存在时继承。

### C2. V2 投影保留 utterance.speakerId

建议方案：

1. 在 [src/services/LayerSegmentationTextService.ts](../../../../src/services/LayerSegmentationTextService.ts) 构建 `segmentDoc` 时追加 `speakerId` 投影。
2. 只在 `utterance.speakerId` 有值时写入，避免显式写 `undefined`。

### C3. 禁止静默创建孤立 independent segment

建议方案：

1. 在 independent layer 创建路径中，如果既找不到 overlapping utterance，又没有当前 `speakerFocusTargetKey`，直接拒绝创建。
2. UI 层给出明确提示：
   - 需要先选定说话人；或
   - 需要在已有 utterance 范围内创建。
3. 不要继续把这种情况落成既无 `utteranceId` 又无 `speakerId` 的 segment。

### 涉及文件

### 退出条件

## 批次 D：EAF Import / Export 核心修复

### 目标

把 speaker 标识的 import/export 往返和 independent transcription tier 导入补齐到可互操作状态。

### D1. import 阶段统一 speaker key 规范化

建议方案：

1. 在 [src/hooks/useImportExport.ts](../../../../src/hooks/useImportExport.ts) 中，为 `speakerIdMap` 增加统一的 normalize helper。
2. 写入和读取都走同一 normalized key，而不是“写时 normalized、读时原串”。
3. 对 TRS / EAF / 未来其他导入器统一复用相同 normalize 逻辑。

### D2. export 阶段兼容 UUID 和自由字符串 speakerId

建议方案：

1. 在 [src/services/EafService.ts](../../../../src/services/EafService.ts) 中抽出 `resolveSpeakerDisplayName()`。
2. 规则：
   - 若 `speakerId` 命中 `speakers.id`，按实体名导出。
   - 若未命中，则按自由字符串或 speaker.name 回退查找。
   - 若仍找不到，再放弃 PARTICIPANT。
3. 该规则同时用于默认转写 tier、translation tiers、independent transcription tiers。

### D3. independent transcription tier 在无 inserted utterances 时仍导入

建议方案：

1. 在 [src/hooks/useImportExport.ts](../../../../src/hooks/useImportExport.ts) 中拆开“是否有 inserted utterances”和“是否允许创建 independent tier segments”两个判断。
2. 若源数据本身就是独立转写层，且 tier annotations 已给出时间边界，应允许直接创建该层 canonical segment graph。
3. 媒体与 textId 的来源应从导入上下文解析，而不是强绑 `insertedUtterances[0]`。

### D4. 缩小 3.5 的真实影响范围

建议方案：

1. 用最小 fixture 分别验证：
   - transcription `independent_boundary`
   - transcription `time_subdivision`
   - translation `independent_boundary`
   - translation `time_subdivision`
2. 明确究竟是 export filter 问题，还是 import 阶段掉了 segment graph。
3. 只有定位到唯一根因后，再把它并入 D1-D3 或单独立一个补丁批次。

### 涉及文件

2. [src/services/EafService.ts](../../../../src/services/EafService.ts)

1. EAF import → export → re-import 后 PARTICIPANT 保持不丢失。
2. 大小写不同的 speaker 名称不再生成重复 speaker 或映射失败。
3. independent transcription tier 在无 utterance 场景下仍能导入。
4. export filter 问题有唯一结论，不再留模糊描述。

## 推荐执行顺序

建议严格按以下顺序推进：

1. 批次 A：先补回归与最小复现。
2. 批次 B：Speaker 管理与面板路由。
3. 批次 C：Segment 创建与投影。
4. 批次 D：EAF Import / Export。
5. 批次 E：Layer Constraint。
6. 批次 F：决策与清理项。

原因：

1. Speaker、segment、EAF 三条链路彼此耦合，必须先把数据正确性锁住。
2. Constraint 修复依赖前面批次先稳定 parent / speaker / import-export 基线。
3. `layer_utterances` 与时间戳风格属于收尾决策，不应阻塞主链路修复。
