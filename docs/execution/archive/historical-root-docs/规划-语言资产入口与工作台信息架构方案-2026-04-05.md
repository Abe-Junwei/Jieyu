# 语言资产入口与工作台信息架构方案（2026-04-05）

## 历史说明
- 本文保留 2026-04-05 立项时的信息架构判断与原始拆解。
- 当日落地结果已收口到 `docs/发布说明-语言资产工作台收口-2026-04-05.md` 与 `docs/architecture/仓库现状与代码地图.md`。
- 下文中的“现状与约束”均按文内日期理解，不再作为当前事实源。

## 目标
- 将“语言元数据、正字法、正字法桥接”等能力从词典工作台中解耦，收口为独立的“语言资产”入口。
- 复用现有左侧壳层结构，在主导航下方建立可扩展的二级入口区，避免后续继续把资产类能力塞进 `/lexicon/*`。
- 为后续“用户可编辑目录”能力建立稳定的信息架构，使“内置基线 + 用户覆盖 + 用户自定义条目”三层模型有明确承载页面。

## 现状与约束
- 现有主导航仅覆盖核心工作台：转写、标注、分析、写作、词典。
- 左侧栏已预留底部区域 `left-rail-bottom-slot`，适合承载未来次级入口。
- 正字法页当前仍挂在 `/lexicon/orthographies`，属于历史过渡，不宜继续作为长期路由命名。
- 语言名称元数据当前来源于生成产物与运行时读取层，不应允许用户直接修改生成 JSON。
- 仓库已存在 `languages` 主实体，但当前语言名称目录并未以该表作为唯一事实源。
- 当前语言名称目录更接近“统一读取接口，分散上游来源”，而不是“统一底层表”。

## 现状事实：当前语言元数据如何维护

### 当前来源链路
- 生成脚本：`scripts/generate-language-name-indexes.mjs`
- 生成产物：`public/data/language-support/*.json`
- 运行时生成模块：`src/data/generated/languageNameCatalog.generated.ts`
- 运行时读取入口：`src/data/languageNameCatalog.ts`
- 运行时补丁：`src/data/languageNameOverrides.ts`
- 查询与解析逻辑：`src/utils/langMapping.ts`

### 当前数据形态判断
- 语言名称存在“显示名称矩阵”，但它是隐式矩阵：`english` / `native` / `byLocale` 仍附着在生成产物中。
- 语言名称也存在“检索名称矩阵”：query index 与 alias 文件已存在，但仍是生成结果，不是可编辑底层表。
- 现有 `languages` 表更多承载语言实体基础资料（名称、语族、濒危等级等），尚未正式接管语言名称目录治理。

### 当前问题
- 名称、别名、locale 展示名、运行时覆盖没有统一写入路径。
- 生成产物适合构建，不适合作为用户编辑真源。
- 页面未来若直接依赖生成文件，会导致“编辑能力”和“展示能力”脱节。

## 命名决策

### 一级概念
- 左侧底部区域命名为：`语言资产`
- 英文可用：`Language Assets`

该名称覆盖范围明确，既能容纳目录/元数据，也能容纳正字法、桥接、模板、映射等长期资产，不会把入口误导为“词典附属功能”。

### 子入口命名
- `语言元数据`
- `正字法`
- `正字法桥接`

不建议使用：
- `桥接规则`

原因：
- “桥接规则”过于泛化，未来层桥接、导入桥接、provider bridge、格式映射都可能使用“桥接规则”这一表述。
- 当前能力本质上是 orthography-to-orthography 的 source -> target 转换，“正字法桥接”更清晰，也更利于未来扩展其他 bridge 概念。

### 页面标题命名
- `语言元数据工作台`
- `正字法工作台`
- `正字法桥接工作台`

导航按钮保持短名，页面标题使用完整名称，避免左侧栏过长。

## 路由决策
- `/assets/language-metadata`
- `/assets/orthographies`
- `/assets/orthography-bridges`

不再新增或继续依赖以下长期路由：
- `/lexicon/orthographies`

说明：
- `assets` 比 `lexicon` 更准确，表达“系统资产与目录治理”而非“词条工作流”。
- 后续如果还要接入模板、导入映射、提示词、字体策略等，也可继续扩展为 `/assets/*`，避免二次迁移。

## 页面职责边界

### 1. 语言元数据工作台
职责：
- 查看内置语言名称目录的最终生效值。
- 编辑用户覆盖层。
- 新建用户自定义语言条目。
- 维护各 locale 展示名、native 名、英文名、aliases、状态、备注。
- 预览查询命中与最终显示效果。

不负责：
- 正字法详情编辑。
- 正字法之间的桥接规则维护。

### 2. 正字法工作台
职责：
- 维护语言下的正字法记录。
- 编辑 identity、scriptTag、direction、fontPreferences、inputHints、catalogMetadata 等。
- 管理 built-in reviewed / built-in generated / user 三类来源。

不负责：
- 语言名称元数据治理。
- 正字法之间的转换规则本体编辑。

### 3. 正字法桥接工作台
职责：
- 维护 source orthography -> target orthography 的 bridge 规则。
- 管理规则状态（draft / active / deprecated）。
- 预览样例输入输出。
- 后续扩展批量校验、规则测试、冲突检测。

不负责：
- 语言层级名称目录维护。
- 正字法基础身份元数据编辑。

## 数据模型规划

## 设计取舍

### 采纳的建议
- 采纳“语言元数据不是简单名称表，而是可治理目录资产”的方向。
- 采纳“正字法必须与语言实体解耦”的原则。
- 采纳“基线 + 用户覆盖 + 用户自定义条目”的维护模型。
- 采纳“审计历史必须记录原因、来源、字段差异”的治理要求。
- 采纳“语言名称矩阵应从隐式 byLocale 对象升级为正式底层实体”的方向。

### 不直接采纳的建议
- 不采用“`languageId` 直接作为业务唯一键”的表述。
  - 内部应使用稳定 `id` 作为主键。
  - `languageCode` / `canonicalTag` / `iso6393` 属于领域标识，不直接等同内部主键。
- 不强制要求全部语言记录都必须具备 `glottocode`。
  - `glottocode` 作为强推荐的外部参考字段，而不是普适硬约束。
- 不采用 `political_correct` 这类主观 display context 枚举。
  - 名称角色采用可审计、可解释的 `preferred / autonym / exonym / historical / menu / academic / search`。
- 不采用 `SourceWeight × ReviewStatus` 一类打分公式来决定 canonical 字段。
  - canonical 标识应走规则驱动与字段级 precedence，而非打分竞争。
- 不把 aliases 简化为 `languages` 表上的字符串数组作为唯一真源。
  - 数组索引可作为缓存或搜索加速，但 alias 真源仍应独立管理。

### 总体原则
- 不把生成文件当作最终真源，只把它们当作内置基线与构建缓存。
- 不把所有语言元数据塞进一张大 JSON 表，而是采用“主实体 + 子表 + 物化读模型”的结构。
- 不让页面直接感知数据来自 Dexie 还是生成模块，由统一 Service 聚合与投影。

### 语言元数据三层模型
为支持“编辑已有内容 + 新建完整内容”，语言元数据采用三层架构：

1. 内置基线
- 来源：生成产物
- 属性：只读

2. 用户覆盖层
- 面向已有 `languageId`
- 只存差量字段

3. 用户自定义条目层
- 面向全新语言记录
- 存完整内容
- 支持标准 ISO 639-3 语言，也支持项目私有语言

### 用户自定义条目策略
- 若输入合法 ISO 639-3，则按标准语言处理。
- 若非标准码，则进入“项目私有语言”模式，使用 `user:` 前缀本地 id。
- 自定义条目默认仅本地生效，不直接回写生成 seeds。

## 统一底层模型

### 核心目标
- 以现有 `languages` 为核心主表扩展，而不是另起一套平行语言主实体。
- 将语言名称矩阵、aliases、审计历史从生成产物中剥离，收口到底层存储。
- 保留生成脚本，但把它降级为 baseline seed 导入器与快照构建器。

## 运行时读取与合并策略

### 统一读取原则
- 页面、面板、搜索、解析逻辑统一通过 Service 层读取。
- 页面不应知道当前数据来自生成模块、Dexie，还是两者的 Merge 结果。

### 运行时优先级
按字段级 precedence 合并，而不是使用打分公式：

1. baseline built-in
2. reviewed override
3. user override
4. user custom entry

### 字段级规则
- `canonicalTag`：规则驱动，只允许标准基线或显式审核后的覆盖改写。
- `display name`：显式 override 优先于 built-in。
- `alias`：显式增加与显式屏蔽并存。
- `custom entry`：独立命名空间，不与 built-in 直接竞争同一主键。

### 生成链的新角色
- `generate-language-name-indexes.mjs` 不再是直接供业务代码依赖的长期真源。
- 它改为：
  - baseline seed 生成器
  - 初始化数据包生成器
  - snapshot/index 构建器

## Dexie/IndexedDB 落地建议

### 不建议的做法
- 不建议所有字段完全高度分表后在页面层多次手工 join。
- 不建议只靠一张大表承担名称矩阵、alias、历史、来源、审核全部职责。

### 推荐方案
- 写模型：适度规范化
  - `languages`
  - `language_display_names`
  - `language_aliases`
  - `language_catalog_history`
- 读模型：物化快照或内存索引
  - `language_catalog_snapshot` 或运行时 rebuilt map

### 索引建议
- `languages`
  - `id`
  - `languageCode`
  - `canonicalTag`
  - `iso6393`
  - `glottocode`
  - `family`
  - `sourceType`
  - `visibility`
- `language_display_names`
  - `id`
  - `languageId`
  - `[languageId+locale]`
  - `[languageId+role]`
  - `[locale+role]`
- `language_aliases`
  - `id`
  - `languageId`
  - `normalizedAlias`
  - `[languageId+kind]`
  - `[normalizedAlias+enabled]`
- `language_catalog_history`
  - `id`
  - `languageId`
  - `createdAt`

### 关于 `multiEntry`
- 可在 snapshot 或 search cache 层保留 `multiEntry` 风格的 `searchTokens` 以提高检索效率。
- 但 `aliases` 真源不应退化成主表上的字符串数组。

## 与现有仓库结构的衔接

### 可直接复用的基础
- 现有 `languages` 主实体可作为扩展基础，而不是被废弃。
- 现有正字法与桥接实体已接近理想结构，可作为语言资产区的后续页面。
- 当前 `languageNameCatalog` 读取入口可逐步改造成 Service facade，而不必一次性重写全部调用方。

### 迁移原则
1. 先引入新表与 Service。
2. 再让运行时读取层从“生成模块直读”过渡到“Service 聚合”。
3. 最后将生成脚本从“业务依赖”降级为“种子与缓存构建”。

## 交互策略

### 语言元数据工作台
- 入口支持两种模式：
  - 编辑现有语言
  - 新建自定义语言
- 详情页展示三列视图：
  - 基线值
  - 用户覆盖值
  - 最终生效值
- alias 编辑必须带冲突检测与歧义警告。

### 正字法与桥接工作台
- 允许从语言元数据页跳转到正字法页。
- 允许从正字法页跳转到对应桥接页。
- 不建议在第一期实现跨页面复杂联动，只保留明确跳转入口。

## 实施注意事项
- 不允许用户直接编辑 `public/data/language-support/*.json` 生成产物。
- 语言元数据页的保存必须走持久化层，不能在页面内直接构造运行时临时全局覆盖。
- alias 写入前必须检查是否引入高频歧义词，尤其是 `Chinese`、`中文` 这类泛称。
- `glottocode` 作为推荐外部引用，不作为新增条目的普适硬门槛。
- `canonicalTag` 不通过打分公式竞争产生，而由规则与审核后的字段级 precedence 决定。
- 正字法数据仍独立维护，不应回塞进 `languages` 主表。
- Phase 1 不要顺手把正字法页和桥接页一起全量迁移，避免范围膨胀。

## 推荐的首批落地顺序
1. 左侧底部二级入口框架
2. `/assets/language-metadata` 路由与骨架页
3. 语言元数据底层表与 Service
4. 语言元数据列表/详情/编辑
5. 运行时读取迁移
6. 旧正字法入口迁移与重定向

## 当前结论
- “语言资产”作为底部入口分组名可直接拍板。
- “语言元数据 / 正字法 / 正字法桥接”作为子入口命名可直接拍板。
- 路由统一收口到 `/assets/*`，不再继续扩展 `/lexicon/*`。
- 用户完全输入新内容需要依赖“三层模型”，即基线、覆盖、自定义条目并存。
- 当前已经存在“隐式语言名称矩阵”，后续应升级为正式 `language_display_names` 底层实体。
- 最推荐的落地方式不是“全量重做”，而是“扩展现有 `languages` + 新增名称/alias/历史表 + 统一 Service + 生成链降级”。

补充：
- 上述决策已全部落地，不再处于待实施状态。
- 若后续继续扩展，应从“语言资产区的新增能力”出发，而不是回退到 `/lexicon/*` 或页面内临时拼接的旧模式。
