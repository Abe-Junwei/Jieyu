# 规划：语言名称三段式显示与输入优先策略 - 2026-04-05

## 1. 结论

当前仓库对这次目标的支持情况可以概括为：

1. `local 语言名称`：部分已支持。
2. `native 语言名称`：当前数据模型未显式支持，但现有运行时大概率可以派生出来。
3. `英文语言名称`：已支持。
4. `输入语言名优先`：当前架构已可支持，但前提是保留“用户命中的名称种类”，而不是只保留命中字符串本身。

因此结论不是“底层完全不支持”，而是：

- 持久化层和 reducer 架构已经足够承载这套规则。
- 当前真正缺的是“显示名三元信息”和“名称命中来源”的建模；其中“名称命中来源”必须细化到 local / native / English / alias / code 级别。
- 这次需求可以在不改持久化主键、不改 `languageId = ISO 639-3` 契约的前提下落地。

## 2. 目标定义

目标显示规则分两类：

### 2.1 代码驱动显示

当用户通过 ISO 代码、BCP 47 标签、外部回填值或宿主默认值命中语言时，名称显示采用：

`local · native · English`

示例：

- 中文界面，输入 `fra` 后显示：`法语 · français · French`
- 英文界面，输入 `fra` 后显示：`French · français`
- 中文界面，输入 `uig` 后显示：`维吾尔语 · ئۇيغۇرچە · Uyghur`

### 2.2 名称驱动显示

当用户直接通过语言名命中语言时，名称显示采用：

`输入语言名优先 · 其余名称按命中语义补位`

示例：

- 中文界面，输入 `法语` 后显示：`法语 · français · French`
- 中文界面，输入 `français` 后显示：`français · 法语 · French`
- 中文界面，输入 `French` 后显示：`French · 法语 · français`
- 英文界面，输入 `法语` 后显示：`法语 · français · French`

注意：

- 显示时必须去重，避免出现 `French · français · French` 这类重复片段。
- 不强制显示三个片段；去重后如果只剩两个有效名称，就只显示两个。
- 这里的“输入语言名优先”指的是“用户实际命中的规范名称或别名”，不是任意部分输入片段。

## 3. 当前数据层调研

### 3.1 现有目录数据结构

当前 [src/utils/langMapping.ts](../../../../src/utils/langMapping.ts) 中的 `LanguageCatalogEntry` 字段如下：

```ts
export type LanguageCatalogEntry = {
  iso6393: string;
  iso6391?: string;
  iso6392B?: string;
  iso6392T?: string;
  name: string;
  invertedName?: string;
  displayNameZh?: string;
  aliases: string[];
  scope: ...;
  type: ...;
  descriptions: string[];
  deprecated: boolean;
  preferredIso6393?: string;
  suppressScript?: string;
  macrolanguage?: string;
};
```

现状判断：

1. `name` 和 `descriptions[0]` 本质上是英文名称。
2. `displayNameZh` 只覆盖一部分高频语言，不是完整中文表。
3. `aliases` 主要来自手工别名表，重点覆盖中文别名和业务高频称呼。
4. 没有 `nativeName` / `autonym` 字段。
5. 没有结构化 `displayNames` 字段，也没有“命中的具体名称 token”。

### 3.2 当前搜索索引

[src/utils/langMapping.ts](../../../../src/utils/langMapping.ts) 里的 `searchLanguageCatalog(...)` 当前搜索这些内容：

- `aliases`
- `descriptions`
- `displayNameZh`
- `iso6393`
- `iso6391`
- `iso6392B`
- `iso6392T`

这意味着：

1. 中文名称检索：部分支持。
2. 英文名称检索：支持。
3. native 名称检索：当前不支持。

### 3.3 当前显示函数

当前显示链路核心是：

- [src/utils/langMapping.ts](../../../../src/utils/langMapping.ts) 中的 `getLanguageDisplayName(...)`
- [src/utils/langMapping.ts](../../../../src/utils/langMapping.ts) 中的 `formatLanguageCatalogMatch(...)`

当前行为：

1. `getLanguageDisplayName(...)` 只返回单个名称。
2. `formatLanguageCatalogMatch(...)` 当前格式是：`displayName · iso6393 · scope?`
3. 不支持三段式名称组合。
4. 不支持“输入名称优先”。

### 3.4 native 名称的现实可得性

当前仓库依赖中没有 CLDR 数据包，也没有自建 native 名称表：

- `iso-639-3`
- `language-tags`

但在当前运行环境里，`Intl.DisplayNames` 已经能为不少代表语种直接生成 native 名称。对当前环境的抽样验证结果如下：

| ISO 639-3 | local(zh-CN) | native | English |
| --- | --- | --- | --- |
| `fra` | 法语 | français | French |
| `deu` | 德语 | Deutsch | German |
| `spa` | 西班牙语 | español | Spanish |
| `cmn` | 中文 | 中文 | Chinese |
| `yue` | 粤语 | 粵語 | Cantonese |
| `bod` | 藏语 | བོད་སྐད་ | Tibetan |
| `uig` | 维吾尔语 | ئۇيغۇرچە | Uyghur |
| `jpn` | 日语 | 日本語 | Japanese |

这说明：

1. native 名称不是完全拿不到。
2. 现有运行时对高频语种已经具备相当可用的 autonym 派生能力。
3. 但它还不是仓库内“稳定、显式、可测试”的数据资产。

## 4. 当前状态架构调研

### 4.1 当前架构优点

当前语言输入已经完成 reducer 化，核心层包括：

- [src/utils/languageInputReducer.ts](../../../../src/utils/languageInputReducer.ts)
- [src/utils/languageInputHostState.ts](../../../../src/utils/languageInputHostState.ts)
- [src/components/LanguageIsoInput.tsx](../../../../src/components/LanguageIsoInput.tsx)

这套架构已经具备以下能力：

1. 区分草稿态和 committed 值。
2. 区分名称编辑和代码编辑。
3. 支持 suggestion 列表、键盘选择、两字符代码延迟提交。
4. 支持 `lastChangeSource`。
5. 所有宿主已经收口到共享输入链路。

这意味着它在架构层是支持新增显示策略的。

### 4.2 当前真正的阻塞点

阻塞点不在 reducer 框架本身，而在以下实现细节：

1. [src/utils/languageInputReducer.ts](../../../../src/utils/languageInputReducer.ts) 中 `buildCommittedValueFromMatch(...)` 会把名称重新规范成 `getLanguageDisplayName(code, locale)`。
2. [src/utils/languageInputReducer.ts](../../../../src/utils/languageInputReducer.ts) 中 `buildCommittedValueFromResolvedCode(...)` 也只保留单个 locale 名称。
3. [src/utils/languageInputHostState.ts](../../../../src/utils/languageInputHostState.ts) 中 `buildLanguageInputSeed(...)` 和 `syncLanguageInputWithExternalCode(...)` 都是 locale-only 单名称模式。
4. `LanguageIsoInputValue` 只有一个 `languageName: string`，没有“命中名称来源”或“三段式显示元信息”。

结果就是：

- 用户输入 `French` 命中 `fra` 后，系统会把名称重新写回 locale 名称，而不是保留 `French` 作为优先展示名。
- 用户输入 `français` 即便未来能搜索命中，也会在 commit 时被覆盖成 locale 名。

## 5. 支持性判断

### 5.1 现在已经支持的部分

1. `languageId = ISO 639-3` 持久化主键，不需要变。
2. 宿主侧结构化字段 `localeTag/scriptTag/regionTag/variantTag`，不需要变。
3. reducer 模型足以承载“代码命中”和“名称命中”不同显示策略。
4. `Intl.DisplayNames` 在当前环境下可以为不少语言生成 native 名称。

### 5.2 现在还不支持的部分

1. native 名称没有显式字段，也没有稳定索引。
2. 搜索结果没有“命中的具体名称 token”。
3. committed 值里没有“名称显示策略来源”。
4. 所有显示 helper 仍然假定“一个 code 只有一个当前展示名”。

### 5.3 最终判断

结论是：

- 数据层：部分支持，但需要补一层显式名称派生/缓存。
- 架构层：支持，且改造成本可控。
- 持久化层：无需改 schema。

## 6. 推荐策略

## 6.1 总体原则

这次不要把“三段式显示字符串”直接当持久化值或领域值。

应该坚持：

1. `languageCode` 仍是唯一权威语言主键。
2. `languageName` 仍是 UI 草稿文本，不是持久化真相。
3. 三段式显示应由 `entry + display provenance + locale` 动态派生。
4. “输入名称优先”应该建模为显示策略，不应该改写数据主键。

## 6.2 数据模型扩展策略

建议扩展 `LanguageCatalogEntry` 的显示能力，但不改宿主持久化结构。

建议新增一层派生显示数据：

```ts
export type LanguageDisplayNames = {
  local?: string;
  native?: string;
  english: string;
};

export type LanguageResolvedDisplay = {
  primary: string;
  secondary: string[];
  composed: string;
};
```

建议在 `langMapping.ts` 中新增统一能力：

```ts
getLanguageDisplayNames(languageId, locale): LanguageDisplayNames
formatLanguageResolvedDisplay(params): LanguageResolvedDisplay
```

其中：

- `local`：当前 UI locale 下的名称。
- `native`：语言自己的 autonym。
- `english`：稳定英文名。

## 6.3 native 名称来源策略

建议分两层：

### Phase A：先用现有运行时派生

优先使用 `Intl.DisplayNames([candidateLocale], { type: 'language' }).of(candidateLocale)` 生成 native 名称。

候选 locale 选择顺序：

1. `iso6391`
2. 项目已有 `toBcp47(languageId)`
3. `iso6393`

### Phase B：再做构建期固化

为了避免不同浏览器 / ICU 数据版本造成差异，建议后续加一个构建脚本，在种子文件中固化：

- `englishName`
- `displayNameZh`
- `nativeName`

运行时优先读取静态种子，缺失时再回退 `Intl.DisplayNames`。

这样可以兼顾：

1. 先快速落地。
2. 后续结果可测试、可快照、可稳定。

## 6.4 搜索索引扩展策略

`searchLanguageCatalog(...)` 需要把 native 名称纳入索引。

建议把搜索 token 拆成结构化集合：

```ts
type LanguageSearchTokens = {
  localeNames: string[];
  englishNames: string[];
  nativeNames: string[];
  aliases: string[];
  codes: string[];
};
```

匹配优先级建议改成：

1. ISO 639-3 精确命中
2. ISO 639-1 / 639-2 精确命中
3. alias 精确命中
4. locale 名精确命中
5. native 名精确命中
6. english 名精确命中
7. prefix
8. contains

同时建议 `LanguageCatalogMatch` 增加：

```ts
matchedLabel: string;
matchedLabelKind: 'locale' | 'native' | 'english' | 'alias' | 'code';
```

这是“输入名称优先”落地的关键字段。

## 6.5 显示规则算法

建议统一收口到一个组合函数：

```ts
formatLanguageLabelByContext({
  entry,
  locale,
  mode,
  matchedLabel,
}): string
```

其中 `mode`：

- `locale-first`
- `input-first`

算法：

### locale-first

顺序：

1. local
2. native
3. english

### input-first

顺序不是固定的，而是取决于命中的名称语义：

1. 若命中的是 English：`matchedLabel -> local -> native`
2. 若命中的是 native：`matchedLabel -> local -> English`
3. 若命中的是 local 或普通 alias：`matchedLabel -> native -> English`

补充约束：

1. 排序前要先判断 `matchedLabel` 在语义上是否等于 local / native / English，而不能只看原始 `matchedLabelKind`。
2. 命中 alias 时，如果 alias 文本恰好等于 English 或 native，也应按对应语义排序，而不是按普通 alias 处理。
3. 去重后最多保留三段，不强制凑满三段。

### 去重规则

按归一化文本去重：

- trim
- case-insensitive
- Unicode NFKC

示例：

- `法语 · français · French`
- `français · 法语 · French`
- `French · 法语 · français`
- `French · français`
- `中文 · English` 不成立，因为 `English` 与 `中文` 不是同语种名称组合结果

## 6.6 UI 渲染策略

### 不建议的做法

不建议始终把完整三段式字符串直接塞进可编辑 input 中，因为用户继续编辑时会很别扭。

### 推荐做法

名称输入框分“编辑显示”和“已选显示”两种状态：

1. 当 `activeField === 'name'` 时，输入框显示可编辑主名称。
2. 当 `activeField !== 'name'` 且当前处于 `selected` 状态时，输入框显示组合展示名；该展示名最多三段，去重后允许只剩两段。

这样用户体验会变成：

- 选中后看到完整名称信息。
- 一旦重新点进名称框，就回到可编辑的 primary label，而不是在 `法语 · français · French` 里硬删点号。

这正好可以复用当前 reducer 中已有的 `draft.activeField`。

## 6.7 名称来源策略

建议按来源区分：

### 代码 / 外部同步命中

以下场景使用 `locale-first`：

- `codeChanged` 成功解析
- `codeBlurred` 成功规范化
- `externalValueSynced`
- `buildLanguageInputSeed(...)`
- `syncLanguageInputWithExternalCode(...)`

### 名称命中

以下场景使用 `input-first`：

- `nameChanged` 产生明确 exact match 并自动提交
- `nameSuggestionCommitted`

但这里有一个重要约束：

- 若用户只是输入 `Fr`、`Portu` 这类不完整前缀，不应把原始草稿直接当主显示名。
- 只应在匹配结果能提供稳定 `matchedLabel` 时，才使用 `input-first`。

因此 `matchedLabel` 应来自“候选命中的标准 token”，而不是裸 `draft.nameInput`。

## 7. 推荐落地方案

### Phase 1：能力补齐，不改宿主协议

目标：先在 `langMapping.ts` 和 reducer 内补齐能力，不改外部持久化结构。

改动点：

1. 在 [src/utils/langMapping.ts](../../../../src/utils/langMapping.ts) 中新增：
   - `getLanguageDisplayNames(...)`
   - `getNativeLanguageDisplayName(...)`
   - `formatLanguageLabelByContext(...)`
2. `LanguageCatalogMatch` 增加 `matchedLabel` 和 `matchedLabelKind`
3. `searchLanguageCatalog(...)` 纳入 native tokens

### Phase 2：状态机接入显示来源

目标：让 reducer 明确知道当前名称显示是 `locale-first` 还是 `input-first`。

建议在 reducer 内部状态增加：

```ts
displayMode: 'locale-first' | 'input-first';
matchedLabel?: string;
```

触发规则：

- 代码命中：`locale-first`
- 名称命中：`input-first`

### Phase 3：组件层切换“编辑视图 / 显示视图”

目标：真正把组合展示做出来，又不破坏编辑体验。

改动点：

1. `selectDisplayedLanguageInputValue(...)` 改为 focus-aware。
2. 名称字段非编辑态显示组合展示名。
3. 名称字段进入编辑态时回退到可编辑主名称。

### Phase 4：宿主种子与外部同步全量接入

目标：所有宿主通过 code 回填时都统一看到 `local · native · English`。

改动点：

1. [src/utils/languageInputHostState.ts](../../../../src/utils/languageInputHostState.ts)
2. [src/pages/OrthographyBridgeWorkspacePage.tsx](../../../../src/pages/OrthographyBridgeWorkspacePage.tsx)
3. [src/components/ProjectSetupDialog.tsx](../../../../src/components/ProjectSetupDialog.tsx)
4. [src/components/LayerActionPopover.tsx](../../../../src/components/LayerActionPopover.tsx)
5. [src/components/OrthographyBuilderPanel.tsx](../../../../src/components/OrthographyBuilderPanel.tsx)
6. [src/components/OrthographyBridgeManager.tsx](../../../../src/components/OrthographyBridgeManager.tsx)

## 8. 不建议改动的部分

以下部分不建议动：

1. 不把 `languageId` 改成任意 BCP 47。
2. 不把三段式显示字符串写入持久化层。
3. 不把 `languageName` 当成权威数据来源。
4. 不在第一阶段就引入重量级 CLDR 运行时包。

## 9. 风险与边界

### 9.1 native 名称稳定性风险

如果完全依赖 `Intl.DisplayNames`，不同环境的结果可能有轻微差异。

策略：

- 第一阶段接受运行时派生。
- 第二阶段补构建期固化表。

### 9.2 组合字符串进入编辑态的体验风险

如果直接把三段式文本塞进 input 并允许继续编辑，用户体验会差。

策略：

- 用 `activeField` 切换展示态与编辑态。

### 9.3 搜索排序回归风险

加入 native tokens 后，候选排序可能变化。

策略：

- 先为 `zh-CN` / `en-US` 各补一批精确回归。
- 特别覆盖 `French / français / 法语`、`Chinese / 中文 / 粵語 / Mandarin`、`Tibetan / 藏语 / བོད་སྐད་`。

## 10. 验收用例

最少要覆盖这些行为：

1. 中文界面输入 `fra`，失焦后显示 `法语 · français · French`。
2. 中文界面输入 `法语`，命中后显示 `法语 · français · French`。
3. 中文界面输入 `français`，命中后显示 `français · 法语 · French`。
4. 英文界面输入 `fra`，失焦后显示 `French · français`。
5. 英文界面输入 `法语`，命中后显示 `法语 · français · French`。
6. 中文界面输入 `French`，命中后显示 `French · 法语 · français`。
7. 英文界面输入 `French`，命中后显示 `French · français`。
8. 去重后允许最终只显示两个片段，而不是强制凑三个。
9. native 名称和英文名相同的情况下去重正确。
10. 两字符短码延迟提交逻辑不回退。
11. 外部同步不覆盖本地未提交草稿。

## 11. 最终建议

建议采用“先派生、后固化；先显示、后扩索引”的策略：

1. 先用当前架构直接实现三段式显示和 `input-first` / `locale-first` 双模式。
2. native 名称第一阶段用 `Intl.DisplayNames` 派生。
3. 搜索索引同步纳入 native token。
4. 等行为跑稳后，再把 high-frequency 语言的 native/local/english 名称固化到构建产物中，消除环境差异。

这条路径的好处是：

- 不动持久化协议。
- 不推翻当前 reducer 架构。
- 只增强目录显示与 suggestion/commit 元数据。
- 能精确满足“代码查找时 locale-first、名称查找时 input-first”的产品规则。
