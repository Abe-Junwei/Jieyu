## 一、设计目标

在现有 AI 对话（11 tool calls + 3 persona + RAG + streaming）和快捷键体系（20 action, 2 scope）基础上，新增一个 **应用级语音智能体（Voice Agent）**，提供三大核心能力：

| # | 能力 | 说明 |
|---|------|------|
| **V1** | 语音/文字指令 → UI 操作 | 用户口头说"删除当前句段""播放下一个"等，Agent 解析意图后调用已有 action/tool 执行 |
| **V2** | 实时语音输入 → 语言分析建议 | 持续监听目标语言语音，实时提供 POS/gloss/翻译等建议 |
| **V3** | 语音转文字 → 翻译/标注层填充 | 对选中语段录制口述翻译或标注，STT 后直接写入对应 layer 的 utterance_texts |

### 4.1 VoiceInputService — 语音采集与识别


**职责**: 统一管理麦克风权限、音频流、STT 转写，屏蔽底层差异。

#### 4.1.1 STT 策略（三层 fallback）

| 优先级 | 引擎 | 场景 | 延迟 | 成本 |
|--------|------|------|------|------|
| 1️⃣ | **Web Speech API** (`SpeechRecognition`) | V1 指令识别、V3 口述翻译 | ~200ms | 免费 |
| 2️⃣ | **Whisper.cpp Local**（whisper-server，端口 3040） | V3 高精度转写 | ~3-8s（本地 GPU） | **免费 · 离线可用** |
| 3️⃣ | **Commercial STT**（插拔式商业模型） | V2/V3 高精度转写，V1 指令增强 | ~1-5s（取决于提供商） | API 计费（groq 低价 · minimax 免费额度 · volcengine 按量） |

#### 4.1.x 商业 STT 插拔式架构（v2.4+）

采用 **Provider Pattern**：接口 → 实现 → 注册表 → 工厂 → UI，分层清晰，扩展容易。

|--------|---------|------|
| **Groq** | `groq` | Groq ASR API，低延迟，价格低廉 |
| **OpenAI Audio** | `openai-audio` | OpenAI Audio API (gpt-4o-transcribe) |
| **Gemini** | `gemini` | Google Gemini 多模态 ASR |
| **MiniMax** | `minimax` | MiniMax T2A/ASR API，国内直连，有免费额度 |
| **VolcEngine** | `volcengine` | 字节火山引擎 ASR，使用 X-App-Id + AccessToken |
| **Custom HTTP** | `custom-http` | 任意 OpenAI 兼容 ASR 接口 |

| 置信度范围 | 颜色 | 行为 |
|------------|------|------|

## 五、V1 — 语音指令 → UI 操作

### 5.1 指令覆盖范围

基于现有 20 个 ActionId + 11 个 AiChatToolName，语音可触发的全部操作：

#### 直接 Action（本地执行，零延迟）

| 语音指令示例 | ActionId | 说明 |
|-------------|----------|------|
| "播放" / "暂停" / "play" | `playPause` | 切换播放 |
| "下一个" / "next" | `navNext` | 导航到下一句段 |
| "上一个" / "previous" | `navPrev` | 导航到上一句段 |
| "删除" / "delete" | `deleteSegment` | 删除选中句段 |
| "撤销" / "undo" | `undo` | 撤销 |
| "重做" / "redo" | `redo` | 重做 |
| "合并上一个" | `mergePrev` | 合并上一句段 |
| "合并下一个" | `mergeNext` | 合并下一句段 |
| "分割" / "split" | `splitSegment` | 在播放头位置分割 |
| "标记" / "mark" | `markSegment` | 标记句段起止 |
| "搜索" / "search" | `search` | 打开搜索 |
| "全选" / "select all" | `selectAll` | 全选句段 |

#### LLM Tool Call（通过 AI Chat 执行）

| 语音指令示例 | Tool Name | 说明 |
|-------------|-----------|------|
| "转写这句为'...'" | `set_transcription_text` | 设置转写文本 |
| "翻译成'...'" | `set_translation_text` | 设置翻译文本 |
| "新建一个日语转写层" | `create_transcription_layer` | 创建层 |
| "自动标注这句" | `auto_gloss_utterance` | 自动标注 |
| "关联翻译层" | `link_translation_layer` | 关联层 |

### 5.2 安全策略

- **破坏性操作确认**: `deleteSegment`, `delete_layer` 等仍需用户确认（复用已有 `allowDestructiveToolCalls` 机制）
- **误触防护**: action 类意图 `confidence < 0.8` 时，显示确认 toast 而非直接执行
- **撤销能力**: 所有通过语音触发的写操作记录到 undo 栈

### 14.2 P0 改进详细设计

#### I1: TTS 语音反馈输出

**问题**: 当前方案是"哑巴 Agent"——只能听指令、不能说话回复。所有成熟语音助手都有语音输出。对于语言学分析场景，TTS 尤其关键：用户可以在操作波形时听到 AI 回复，无需转头看屏幕。

**方案**: 利用 Web Speech Synthesis API（浏览器原生，零成本）+ 可选云端 TTS。

**集成到 useVoiceAgent**:

**用户可配置**: TTS 开关、语速、声音偏好都放入 `VoiceAgentSettings`。

---

#### I2: 回声消除（AEC）

**问题**: 解语是音视频标注工具，用户在操作波形时大量播放音频。如果 STT 同时激活，WaveSurfer 回放的声音会被麦克风拾取，导致误识别。

**方案（三层防御）**:

**推荐默认值**: `aecMode: 'both'`（硬件 AEC + 播放暂停双保险）。

**v2.1 新增: AEC 自检（嵌入 Phase 1 D1-D2）**

---

#### I4: 多轮对话上下文

**问题**: 原方案每次语音输入都是独立的，无法理解指代关系：
- "删除这句" → ok
- "也删除下一句" → ？（"也"和"下一句"需要理解上一轮执行了什么）
- "把刚才删的撤销" → ？（需要知道"刚才"做了什么）

**对标**: OpenAI Realtime API 维护完整 conversation items 列表；Alexa 的 session attributes。

**方案**: 在 useVoiceAgent 中维护一个 **短期操作栈**（VoiceSession）：

**效果**: LLM 收到上下文后能正确理解 "也删除下一句"（看到上一轮是 deleteSegment）、"撤销刚才的"（看到上一轮操作）。

**v2.1 新增: VoiceSession 操作回放（Phase 3 启用）**

---

#### I6: languageId → BCP-47 自动映射

**问题**: 数据库中 `languageId` 是自由文本（如 `'zho'`、`'eng'`、`'jpn'`），而 Web Speech API / Whisper 需要 BCP-47（`'zh-CN'`、`'en-US'`、`'ja-JP'`）。当前设计要求用户手动设置 `sttLang`。

**方案**: 建立映射表 + LanguageDocType 扩展。

**同时扩展 LanguageDocType**（可选字段，不破坏兼容）:

**使用**: `toBcp47(layer.languageId)` → 作为 STT `lang` 参数。

---

#### I14: 快捷键冲突修正

**问题**: `Ctrl+Shift+V` 在 macOS/Windows/Linux 上均为"粘贴纯文本"系统快捷键。

**修正**: 改为 `mod+shift+.`（句号键，暗示"开始说话"）或 `F6`（功能键不冲突）。

#### I9: LLM 全量意图识别

放弃正则 → 所有语音 transcript 统一发给 LLM，利用 function calling 识别意图。类似 GitHub Copilot Voice 的做法：

#### I12: 离线 STT（Whisper WASM）

对田野语言学家（无网络环境）至关重要：

**权衡**: 模型下载 ~40MB（首次），推理速度取决于设备 CPU，适合低资源语言（这些语言通常 Web Speech API 不支持）。

#### I15: TextGrid 导出/导入（MFA 互操作）🆕

**问题**: MFA（Montreal Forced Aligner）是语音学研究的"金标准"强制对齐工具（HMM-GMM 架构，时间分辨率 10ms），但延迟 2-5s 不适合实时集成。语音学家几乎均已本地安装 MFA。

**方案**: 不在解语内集成 MFA 运行时，而是提供 **TextGrid 导出/导入工作流**：

**价值**:
- 满足语音学家核心刚需（音位边界 10ms 精度）
- 与竞品形成差异化（豆包/千问无此能力）
- 实现成本低（仅 TextGrid 导出/导入，~100 行代码）
- 与 F10 MFA 对齐功能天然补充（F10 做自动对齐，I15 做手动精标往返）

### 1. LLM 幻觉防护体系（v3.1 新增）

现有三层防护机制，覆盖工具参数验证、RAG 检索质量、数据注入安全：

#### 防护层 1：工具参数实体 ID 验证（`planToolCallTargets`）

**文件**：`src/hooks/useAiChat.ts` · `ensureUtteranceId()` 函数

**机制**：LLM 返回工具调用时，可能虚构 `utteranceId` / `translationLayerId` / `transcriptionLayerId`。验证逻辑：

```
if (currentUtteranceId && existing !== currentUtteranceId)
  → 强制替换为上下文 ID（选中的句段）

if (!currentUtteranceId && LLM 给出了 ID)
  → 接受但记录警告（无选中时无法验证，真实无效 ID 将在执行层失败）
  → 执行层失败 = 安全（数据库查找返回空，不破坏数据）
```

**防护范围**： utteranceId、translationLayerId、transcriptionLayerId 三个字段的虚构/错配。

#### 防护层 2：RAG 检索质量阈值（`EmbeddingSearchService`）

**文件**：`src/ai/embeddings/EmbeddingSearchService.ts`

**机制**：混合检索（向量相似度 + MiniSearch 全文）结果设置 `minScore: 0.3` 下限过滤：

```typescript
const minScoreMatches = scored.filter((m) => m.score >= minScore);
return { query: normalizedQuery, matches: minScoreMatches.slice(0, topK) };
```

**应用函数**：`searchSimilarUtterances` / `searchMultiSource` / `searchMultiSourceHybrid` 三个检索入口统一应用阈值，低质量匹配（ hallucinated 知识）不会注入 LLM 上下文。

**默认值**：`DEFAULT_MIN_SCORE = 0.3`，可按调用方覆盖。

#### 防护层 3：RAG 数据注入括号转义

**文件**：`src/hooks/useAiChat.ts` · `buildPromptContextBlock()` 函数

**机制**：RAG snippet 注入 LLM 前，将 `[` / `]` 替换为 `【` / `】`：

```typescript
const safeSnippet = snippet
  .slice(0, 300)
  .replace(/[[\]]/g, (c) => c === '[' ? '【' : '】');
ragLines.push(`[${contextTag}] ${safeSnippet}`);
```

**防止**：snippet 内容中的 `[[` `]]` 被 LLM 误解为工具调用边界。

## 附：分析模式重构 — ConversationAgent 方案【已废弃】

> **背景**：以下旧方案（v2.6 规划的自建 ConversationEngine + CorpusAnalyzer + LinguisticsKnowledgeBase 三件套）已被废弃。

### 废弃原因

ConversationAgent 三件套的问题：**自建了一套 LLM 编排层**。

| 旧方案 | 问题 | 成熟替代 |
|--------|------|---------|
| `ConversationEngine` 意图分类 | 手写分类器，维护成本高 | LLM function calling 天然理解意图 |
| `ConversationContext` 多轮状态 | 重复状态，与 LLM message history 冗余 | LLM message history 自然继承 |
| `CorpusAnalyzer.findSimilarSegments` | 关键词 Jaccard 精度低 | 已有 `EmbeddingSearchService`（语义+关键词混合） |
| `LinguisticsKnowledgeBase` | 硬编码 prompt 工厂 | 应为 tool definition 的一部分 |

### 正确路径：Tool-based 架构

**核心思路**：不要建 ConversationAgent，而是把语料库能力暴露为 LLM tools，接入已有 ChatOrchestrator + function calling 体系。

```
用户（语音/文字）
    ↓
VoiceInputService / 文字输入
    ↓
现有 ChatOrchestrator（已有 system prompt + function calling）
    ↓
LLM（function calling）─────────── tool results
    ↓
┌─ Corpus Tools ────────────────────────────────────┐
│  search_corpus(query, filters)   → EmbeddingSearch │
│  get_lexeme_stats(lexeme)        → LinguisticService │
│  compare_speakers(a, b)          → 新建 ~80 行      │
│  find_contextual_usages(word)    → 新建 ~60 行      │
│  get_segment_detail(id)          → 复用现有           │
│  suggest_glosses(id)             → AutoGlossService  │
│  batch_update_glosses(items)     → LinguisticService  │
│  get_project_stats()              → ProjectObserver   │
└────────────────────────────────────────────────────┘
```

## 附：学术前沿技术调研与落地评估（2026-04）

> **调研范围**：过去半年（2024.10 – 2025.04）已发表于顶会/顶刊或被业界广泛采用的学术成果，侧重 ASR、嵌入检索、RAG、浏览器端 ML、TTS 五大方向。每项技术均评估与解语项目的适用性（⭐1–5）。

### A1 语音识别 / ASR

#### A1.1 Distil-Whisper — 知识蒸馏加速 Whisper ⭐⭐⭐⭐⭐

- **论文**：arXiv:2311.00430 (HuggingFace)
- **核心**：对 Whisper large-v2/v3 做知识蒸馏，保留 2 层 decoder → **5.8× 加速、51% 参数量缩减**，OOD 数据 WER 仅增 1%，且幻觉更少；支持 speculative decoding 可再提速 2×
- **许可**：MIT
- **落地路径**：项目已集成 Whisper.cpp Local（`LocalWhisperSttProvider`），可直接切换为 Distil-Whisper ONNX/GGUF 版本。对 T1A/T3B 引擎组合矩阵中的 Whisper.cpp 层直接替换，预期本地 STT 延迟降至当前 1/5
- **关联改进项**：I12（离线 STT）、引擎组合矩阵 T1A/T1B

#### A1.2 CrisperWhisper — 逐字转录 + 精确词级时间戳 ⭐⭐⭐⭐

- **论文**：arXiv:2408.16589 (INTERSPEECH 2024, nyrahealth)
- **核心**：OpenASR Leaderboard #1（verbatim 赛道），WER 6.66%（vs Whisper v3 的 7.7%）；通过调整 tokenizer + DTW cross-attention heads 自定义损失获得精确词级时间戳；支持填充词（um/uh）与口吃检测
- **许可**：CC-BY-NC-4.0（非商用限制）
- **落地路径**：对 `SpeechAnnotationPipeline` 尤其有价值——精确词级时间戳可直接回写 `utterance_texts` 的 word boundary；verbatim 模式有助于田野调查场景下保留原始语言特征。需评估非商用许可对项目的影响
- **关联改进项**：V3（层填充）、`SpeechAnnotationPipeline` dictation 精度

#### A1.3 WhisperX — VAD + 强制音素对齐 ⭐⭐⭐⭐

- **论文**：arXiv:2303.00747 (INTERSPEECH 2023)，2024 年仍为业界事实标准
- **核心**：VAD Cut & Merge 预分段 → 12× 批量推理加速；phoneme-level forced alignment 提供词级时间戳
- **落地路径**：`SttStrategyRouter` 可集成 WhisperX 的 VAD 分段策略；其 forced alignment 思路与项目已有的 `WebMaService`（MAUS API）互补——WhisperX 可提供本地版对齐
- **关联改进项**：I3（VAD）、`vadWorker.ts`（Phase 4）

#### A1.4 Whisper-Medusa — 多头并行解码 ⭐⭐⭐

- **论文**：arXiv:2409.15869 (Sep 2024, IBM Research + BIU)
- **核心**：扩展 Whisper 架构，每次迭代预测多个 token → **50% 延迟降低**，WER 影响极小
- **许可**：CC-BY-4.0
- **落地路径**：可叠加在 Distil-Whisper 之上进一步降低本地推理延迟。需 ONNX 导出支持
- **关联改进项**：引擎组合矩阵延迟列

#### A1.5 Moonshine — 超轻量实时 ASR ⭐⭐

- **论文**：arXiv:2410.15608 (Useful Sensors, Oct 2024)
- **核心**：RoPE 编码器、无零填充，5× 计算量降低（vs Whisper tiny-en），同等 WER。专为实时/资源受限场景设计
- **许可**：MIT
- **限制**：仅英语。如果项目有英语低延迟 VAD 实时反馈场景可考虑

#### A1.6 Whistle — 音素监督跨语言 ASR ⭐⭐⭐

- **论文**：arXiv:2406.02166 (IEEE-TASLP)
- **核心**：通过弱音素监督（IPA）实现跨语言 ASR 迁移，在 CV-Lang10 10+2 语言测试中表现优异；音素模型在 few-shot 跨语言场景和数据效率方面具有优势
- **落地路径**：与项目的 IPA 转写功能天然匹配。可为 `langMapping.ts`（I6）中的低资源语言提供额外 ASR 路径，拓展 STT 语种覆盖
- **关联改进项**：I7（STT 语言自动切换）、低资源语言支持

### A2 文本嵌入 / 检索

#### A2.1 ModernBERT — 现代化 Encoder-Only Transformer ⭐⭐⭐

- **论文**：arXiv:2412.13663 (Dec 2024, Answer.AI + LightOn + HuggingFace)
- **核心**：2T token 训练，原生 8192 上下文，Flash Attention + RoPE + unpadding；分类与检索（单/多向量，含代码检索）均 SOTA；速度/内存效率最优的 encoder
- **许可**：CC-BY-4.0
- **落地路径**：项目当前用 E5-Small (33M) ONNX。ModernBERT-base (149M) 可作为嵌入模型升级候选，**但目前仅英语**，需等待多语言版本。8192 上下文长度对长段标注文本嵌入有直接收益
- **关联模块**：`EmbeddingSearchService`、`embedding.worker.ts`

#### A2.2 E5-Mistral — LLM 驱动的跨语言嵌入 ⭐⭐

- **论文**：arXiv:2401.00368 (ACL 2024, Microsoft)
- **核心**：用 LLM 生成 93 种语言的合成训练数据 → 标准对比损失微调开源 decoder-only LLM；无需人工标注数据即达 BEIR/MTEB SOTA
- **许可**：CC-BY-4.0
- **落地路径**：模型 7B 无法浏览器端运行，但其"合成多语言训练数据"方法论可借鉴——用 Gemini API 为项目自有的 E5-Small 生成低资源语言微调数据

#### A2.3 NV-Embed — MTEB 排行榜冠军 ⭐⭐

- **论文**：arXiv:2405.17428 (ICLR 2025 Spotlight, NVIDIA)
- **核心**：Latent attention 池化层 + 移除因果注意力掩码 + 两阶段对比指令微调；NV-Embed-v1/v2 分别在 2024.05 和 2024.08 登顶 MTEB（56 任务）
- **许可**：CC-BY-4.0，模型开源
- **落地路径**：模型太大无法浏览器端，但"latent attention pooling"范式和"两阶段微调"可指导项目嵌入模型选型与微调策略

#### A2.4 Arctic-Embed — 轻量高性能嵌入族 ⭐⭐⭐⭐

- **论文**：arXiv:2405.05374 (Snowflake, May 2024)
- **核心**：22M–334M 参数族，各尺寸段 MTEB Retrieval SOTA，超越 Cohere embed-v3 和 OpenAI text-embed-3-large
- **许可**：Apache-2.0
- **落地路径**：**Arctic-Embed-XS (22M) 可直接替代 E5-Small 作为浏览器端嵌入模型**。Apache-2.0 许可友好；ONNX 转换后体积与 E5-Small 相当。需在 `embedding.worker.ts` 中加入 Arctic 模型路径并做 A/B 评测
- **关联模块**：`EmbeddingSearchService`、`LocalEmbeddingProvider`

#### A2.5 Late Chunking — 上下文感知分块嵌入 ⭐⭐⭐

- **论文**：arXiv:2409.04701 (Sep 2024, Jina AI)
- **核心**：利用长上下文嵌入模型先编码完整文档所有 token，在 mean pooling 前按分块切割，使每个 chunk embedding 保留完整上下文语义；无需额外训练
- **许可**：CC-BY-NC-SA-4.0
- **落地路径**：项目 RAG 当前采用固定分块策略，Late Chunking 可显著提升长文档（如导入的 PDF、参考文本、标注材料）检索质量。需配合长上下文嵌入模型（如 ModernBERT 8192 / jina-embeddings-v3 8192）
- **关联模块**：`EmbeddingSearchService.indexDocuments()`

#### A2.6 ColPali — 视觉语言文档检索 ⭐⭐

- **论文**：arXiv:2407.01449 (ICLR 2025)
- **核心**：直接将文档页面图像嵌入多向量表示 → late interaction 匹配，跳过传统 OCR→文本→嵌入管线
- **许可**：CC0，模型/数据/代码全开源
- **落地路径**：如果项目未来增加文档/PDF 标注功能可考虑；当前优先级不高

### A3 RAG 改进

#### A3.1 Self-RAG — 自反思检索增强生成 ⭐⭐⭐

- **论文**：arXiv:2310.11511 (ICLR 2024, U of Washington)
- **核心**：训练 LM 通过"反思 token"（reflection tokens）按需检索 → 生成 → 自评；7B/13B 在 Open-domain QA、推理、事实验证上超越 ChatGPT 和 RAG-Llama2-chat
- **许可**：CC-BY-4.0
- **落地路径**：项目 AI 聊天使用 Gemini API，Self-RAG 的按需检索思路可集成到 `IntentRouter` 决策链——在 system prompt 中添加"需要引用标注数据才回答"的反思指令，仅当 LLM 判断"需要语料库上下文"时才触发 `EmbeddingSearchService` 检索，减少无关检索开销
- **关联模块**：`ChatOrchestrator`、`useAiChat.send()`、`IntentRouter` LLM fallback

#### A3.2 CRAG — 修正式检索增强生成 ⭐⭐⭐⭐

- **论文**：arXiv:2401.15884 (Jan 2024)
- **核心**：轻量检索评估器评估文档质量 → 置信度分级 → 触发不同策略（直接用 / 精炼 / Web 搜索增补）；Plug-and-play，可嵌入任何 RAG 框架
- **落地路径**：在 `searchMultiSourceHybrid` 检索后加一层"结果质量评估"——当检索结果 `minScore < threshold` 或命中数过少时，回退到更大范围搜索或跳过检索直接回答。实现成本低（~50 行评估函数），可显著减少 LLM 幻觉
- **关联模块**：`EmbeddingSearchService.searchMultiSourceHybrid()`、LLM 幻觉防护体系（v3.1）

### A4 浏览器端 ML / 前端推理

#### A4.1 Transformers.js v4 — 全新 C++ WebGPU Runtime ⭐⭐⭐⭐⭐

- **发布**：2026-02-09 (HuggingFace)
- **核心亮点**：
  - 全新 C++ 重写的 WebGPU runtime → **BERT 嵌入模型 4× 推理加速**
  - 支持 Node/Bun/Deno WebGPU 加速
  - 独立 `@huggingface/tokenizers` 包（8.8kB gzipped，零依赖）
  - `ModelRegistry` API：加载前查文件列表/大小/缓存状态，`progress_total` 事件
  - `env.useWasmCache`：首次加载后完全离线运行
  - 支持 >8B 模型（GPT-OSS 20B q4f16 实测 ~60 tok/s on M4 Pro Max）
- **许可**：Apache-2.0
- **落地路径**：**项目当前仍在用 `@xenova/transformers` 2.17.2（v2），已滞后两个大版本**。迁移至 `@huggingface/transformers` v4 的具体收益：
  1. `embedding.worker.ts` 中 E5 嵌入推理 4× 提速（WebGPU 路径）
  2. `ModelRegistry.is_pipeline_cached()` 实现优雅的"首次下载 / 后续离线"体验
  3. `env.useWasmCache` 保证离线可用（田野调查场景，对应 I12）
  4. 独立 `@huggingface/tokenizers` 可为分词模块减小 bundle（当前 `transformers` 824KB 告警）
  5. per-module dtypes 支持：对 Whisper 模型可分别设置 encoder fp16 + decoder q4，降低内存占用
- **关联模块**：`embedding.worker.ts`、`EmbeddingRuntime.ts`、`LocalEmbeddingProvider.ts`；构建体积治理（`transformers` 824KB 告警）

#### A4.2 WebGPU 降级策略 ⭐⭐⭐

- **状态**：截至 2024.10 全球 WebGPU 支持约 70%（Chrome 稳定、Safari/Firefox 实验性），2025 年覆盖率预计升至 80%+
- **落地路径**：在 `EmbeddingRuntime.ts` 中增加设备能力检测——有 WebGPU 时启用 `device: 'webgpu'` GPU 加速嵌入计算，否则回退 `device: 'wasm'`。与 Transformers.js v4 升级同步实施
- **关联模块**：`embedding.worker.ts`、`AmbientObserver.ts`（设备能力检测）

### A5 语音合成 (TTS)

#### A5.1 Matcha-TTS — 最优流匹配 TTS ⭐⭐⭐

- **论文**：arXiv:2309.03199 (ICASSP 2024)
- **核心**：基于最优传输条件流匹配（OT-CFM），非自回归，从零学习对齐；最小内存占用 + 最高 MOS 评分
- **许可**：MIT
- **落地路径**：项目 TTS 服务（I1）尚在规划中（P0 待实现），Matcha-TTS 可作为本地 TTS 引擎候选。需通过 ONNX 导出后集成到 `TtsProvider` 接口，或部署 Python 后端服务
- **关联改进项**：I1（TTS 语音反馈输出）、`src/services/tts/`

#### A5.2 StyleTTS 2 — 人类水平 TTS ⭐⭐

- **论文**：arXiv:2306.07691 (NeurIPS 2023)
- **核心**：Style diffusion + WavLM adversarial 训练，在 LJSpeech 上超越人类录音 MOS，在 VCTK 上匹配；支持 zero-shot speaker adaptation
- **许可**：CC-BY-NC-SA-4.0（非商用限制）
- **落地路径**：质量极高但需 GPU 推理 + 非商用限制，可作为服务端 TTS 备选方案

#### A8.2 P0-2：CRAG 检索质量评估层

**目标**：RAG 检索后加入质量评估，避免低质量结果注入 LLM 上下文导致幻觉

**当前状态**
- `enrichContextWithRag()`（`src/hooks/useAiChat.rag.ts` L74）两阶段搜索：minScore 0.3 → 0.1 fallback
- 无质量评估：搜到就全部注入 `contextBlock`

**CRAG 三路分支设计**

```
搜索结果 matches[] ──→ evaluateRagQuality(matches, queryText)
                         │
              ┌───────────┼───────────┐
              ▼           ▼           ▼
          CORRECT     AMBIGUOUS    INCORRECT
       (maxScore≥0.55   (0.35≤max<0.55  (max<0.35
        且 gap≥0.1)      或 gap<0.1)     或 matches=0)
              │           │             │
              ▼           ▼             ▼
         直接使用      精炼查询后     跳过检索
         matches      重搜一次       contextBlock 不注入
```

**改动清单**

| # | 文件 | 变更内容 |
|---|------|----------|
| 1 | 新建 `src/ai/embeddings/ragQualityEvaluator.ts` (~60 行) | 导出 `evaluateRagQuality(matches, query)` → `{ verdict, maxScore, scoreGap, refinedQuery? }` |
| 2 | `src/hooks/useAiChat.rag.ts` L86-100 | `activeMatches` 获取后插入评估调用；`incorrect` → early-return；`ambiguous` → query expansion 重搜 |
| 3 | `src/ai/embeddings/searchFusionProfiles.ts` | 新增 `queryExpansion` 场景（keyword 权重 0.45） |

**核心逻辑**：`evaluateRagQuality()` 计算 `maxScore`、`scoreGap`（top1 - top2 距离），按阈值判定三路分支。`ambiguous` 路径提取查询关键词重组后以 `keywordWeight: 0.45` 重搜一次。

**测试策略**：新增 `ragQualityEvaluator.test.ts`（三路各 2+ 用例）；修改 `useAiChat.rag.test.ts` 验证低分不注入、ambiguous 触发重搜；`npm run perf:ai` 回归
