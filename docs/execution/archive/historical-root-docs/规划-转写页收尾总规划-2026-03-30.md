## 一、规划结论

未来 3-4 周，研发主线继续严格收敛到“转写主工作台收尾”，不再平均推进标注页、分析页、写作页、词典页。近期目标不是继续铺页面，而是把当前已存在的转写、时间轴、AI、语音、RAG/PDF、批量编辑能力收束成一个可持续维护、可解释、可验收、可发布的产品内核。

这轮收尾的核心判断有四个：

1. `TranscriptionPage.Orchestrator.tsx` 虽已从 4956 行收缩到约 2187 行，但仍是转写页最主要的编排密度风险源。
2. 语音与 AI 主链路已可用，但 `VoiceAgentService.ts`、`useVoiceAgent.ts`、`useAiChat.ts` 仍然承担过多状态协调，产品化可见性也未完全闭环。
3. 架构热点台账当前无 active hotspot，不代表可以继续扩写大模块；相反，这意味着当前阶段应该把“未超线但仍偏重”的模块继续往稳定 controller/runtime contract 收口。
4. 自动化基线已较完整，但手工验收记录仍为空白模板，说明当前最大缺口之一已经从“代码不存在”转为“证据链和发布闭环未完成”。

## 二、范围边界

### 2.1 本轮必须完成

1. 转写页编排核继续收口，降低页面层职责密度。
2. 选择语义、时间轴语义、波形语义统一，消除 fallback 分叉。
3. 高频编辑链路补成完整闭环，尤其是批量编辑与 undo/speaker/note 的组合场景。
4. AI / 语音 / RAG 的主路径产品化，做到“用户看得见状态、看得懂来源、知道写回去了哪里”。
5. 自动化回归、手工验收、执行记录、发布封账形成完整闭环。

### 2.2 本轮明确不做

1. 不重新开放标注页、分析页、写作页、词典页的真实交付承诺。
2. 不把 F10 强制对齐做成近期产品目标，继续只保留 provider-neutral 预留设计。
3. 不为“页面看起来完整”引入新的空壳 UI 或半完成工作台。

## 三、当前基线与主风险

### 3.1 当前重点模块体量

1. `src/pages/TranscriptionPage.Orchestrator.tsx`：2187 行
2. `src/services/VoiceAgentService.ts`：1220 行
3. `src/hooks/useAiChat.ts`：1154 行
4. `src/hooks/useVoiceAgent.ts`：915 行
5. `src/components/BatchOperationPanel.tsx`：764 行
6. `src/pages/useTranscriptionWaveformBridgeController.ts`：497 行
7. `src/pages/useTranscriptionTimelineInteractionController.ts`：488 行
8. `src/pages/TranscriptionPage.AssistantRuntime.tsx`：470 行

### 3.2 当前剩余风险

1. 页面层仍偏重：Orchestrator 虽收缩，但仍是新需求最容易回流的默认落点。
2. 语音链路偏复杂：command、dictation、analysis 三模式共享状态，仍有继续膨胀的风险。
3. 交互语义尚未完全统一：timeline、waveform、context menu、split/merge 对 layerId 与 unit kind 的理解仍存在残余差异。
4. 产品可见性仍不均匀：citation、决策日志、任务状态、失败重试、写回提示已部分落地，但还没有完全统一的用户心智模型。
5. 发布证据不完整：`docs/execution/archive/manual-validation/手工验收执行记录-AI语音RAG主流程-2026-03-30.md` 仍为空白。

## 四、执行总策略

本轮采用“4 个执行包 + 1 个封账包”的方式推进，每个执行包都必须同时产出代码、测试、文档、手工验收结果。禁止只做其中一项。

### 包 0：基线封口与门禁补齐

目标：先把后续 3 周开发的门槛、验收和回归入口锁定，避免收尾阶段一边开发一边改标准。

必须完成：

1. 冻结本轮收尾范围与 out-of-scope。
2. 把 AI / 语音 / RAG 主流程手工脚本映射到真实执行记录。
3. 为本轮涉及的重模块补齐“结构守卫 + 聚焦回归 + 手工用例”三层校验映射。
4. 明确出包 gate：`npm run typecheck`、`npm run build`、`npm test`、对应手工验收记录、缺陷清零。

建议代码/文档落点：

1. `docs/规划-转写页收尾总规划-2026-03-30.md`
2. `docs/execution/archive/manual-validation/手工验收执行记录-AI语音RAG主流程-2026-03-30.md`
3. `docs/execution/governance/架构热点台账.md`
4. `scripts/architecture-guard.config.mjs`

完成标准：

1. 研发与验收对“本轮要做什么、不做什么、做到什么算完成”无歧义。
2. 发布前的证据模板与执行入口不再缺失。

### 包 1：编排核与运行时边界收紧

目标：继续缩小转写页页面层职责，避免新功能回流到单一编排文件。

主要任务：

1. 把 Orchestrator 剩余的页面组装逻辑继续下沉到 controller/runtime 层。
2. 细化 assistant、analysis、pdf、waveform、workspace layout 的 contract，减少大对象透传。
3. 对语音链路执行第二轮拆分，把状态机、设备控制、执行管线边界再拉开一层。
4. 对 `useAiChat.ts` 中纯展示派生、provider 协调、task 可见性逻辑进行进一步收束，避免继续当“万能总线”。

优先落点：

1. `src/pages/TranscriptionPage.Orchestrator.tsx`
2. `src/pages/TranscriptionPage.AssistantRuntime.tsx`
3. `src/pages/TranscriptionPage.AnalysisRuntime.tsx`
4. `src/pages/TranscriptionPage.PdfRuntime.tsx`
5. `src/pages/useTranscriptionWaveformBridgeController.ts`
6. `src/pages/useTranscriptionWorkspaceLayoutController.ts`
7. `src/services/VoiceAgentService.ts`
8. `src/hooks/useVoiceAgent.ts`
9. `src/hooks/useAiChat.ts`

自动化门禁：

1. 相关 structure test 更新，防止 runtime/controller 回流页面层。
2. 聚焦回归覆盖 assistant、voice、waveform、pdf、task 状态。
3. `npm run typecheck` 与 `npm test` 全绿。

完成标准：

1. Orchestrator 行数继续下降到 1600-1800 区间。
2. 新增功能默认落点不再是 Orchestrator。
3. 语音和 AI 的核心状态不再跨页面层、hook 层、组件层重复拼接。

### 包 2：选择语义与高频编辑闭环

目标：把“选中什么、操作什么、写回哪里、撤销如何恢复”统一成单一模型。

主要任务：

1. 统一 `selectedTimelineUnit`、waveform 当前单元、context menu 操作目标、batch preview 目标的语义。
2. 清理 dependent layer、segment-backed layer、default transcription layer、空选中 fallback 的残余分叉。
3. 完成批量拆分 / 合并 / 偏移 / 缩放的最终收口。
4. 打通批量编辑与 speaker、note、undo/redo、导入冲突恢复的组合路径。

优先落点：

1. `src/pages/useTranscriptionTimelineInteractionController.ts`
2. `src/pages/useTranscriptionTimelineController.ts`
3. `src/pages/useTranscriptionSelectionContextController.ts`
4. `src/pages/useTranscriptionSegmentMutationController.ts`
5. `src/components/BatchOperationPanel.tsx`
6. `src/hooks/useTranscriptionLayerActions.ts`
7. `src/services/LayerOrderingService.ts`

自动化门禁：

1. 聚焦回归补齐 dependent layer / segment-backed / waveform / context menu 交叉场景。
2. 批量操作与 undo/speaker/note 组合场景形成稳定测试矩阵。
3. 不允许新增“空 layerId / fallback 分支不一致”类回归。

完成标准：

1. 同一选中单元在 timeline、waveform、context menu、批量面板中的目标理解一致。
2. 高频编辑后的撤销恢复稳定。
3. speaker 与 note 不再是批量编辑链路之外的附属能力。

### 包 3：AI / 语音 / RAG 产品化收口

目标：把当前已经存在的 AI / 语音 / RAG 能力，从“开发态可用”推进到“主流程可依赖”。

主要任务：

1. 补完 dictation、analysis、command 三种语音模式的统一状态摘要与错误反馈。
2. 落地 TTS / 回放辅助的最小可用路径，不追求大而全，只追求主路径闭环。
3. 将 AI task / clarify / pending confirm / retry / metrics / decision logs 收束到统一可见模型。
4. 完成 citation 脚注体验与 bibliography 最小闭环，让“引用 -> 跳转 -> 修订 -> 复核”成为可走通主路径。
5. 把 Prompt Lab / RAG 模板从“会用的人知道怎么触发”提升到显式场景化入口。

优先落点：

1. `src/hooks/useAiChat.ts`
2. `src/components/AiAnalysisPanel.tsx`
3. `src/components/ai/AiChatCard.tsx`
4. `src/components/ai/AiChatPromptLabModal.tsx`
5. `src/pages/TranscriptionPage.AssistantRuntime.tsx`
6. `src/pages/TranscriptionPage.citationJump.ts`
7. `src/services/VoiceAgentService.ts`
8. `src/hooks/useVoiceAgent.ts`

自动化门禁：

1. AI 任务状态、澄清、待确认、失败重试的 UI 回归。
2. citation 跳转与降级路径回归。
3. dictation / analysis / command 三模式聚焦测试。

完成标准：

1. 用户可以在 UI 上完成一次完整的“提问 -> 引用 -> 跳转 -> 修订 -> 再验证”流程。
2. 语音模式切换、失败、取消、写回后状态稳定且可见。
3. AI / 语音 / RAG 不再依赖“开发者知道内部状态”的前提才能使用。

### 包 4：发布封账与开放条件确认

目标：在功能收尾后完成真实验收，封住当前版本，并明确后续是否允许打开其他工作台。

主要任务：

1. 按现有脚本执行 10 条 AI / 语音 / RAG 主流程手工验收。
2. 回填执行记录、证据文件路径、自动化结果。
3. 重新运行全量门禁并封账。
4. 根据收尾结果评估是否具备开放其他工作台的前置条件。

优先落点：

1. `docs/execution/archive/manual-validation/手工验收执行记录-AI语音RAG主流程-2026-03-30.md`
2. `docs/execution/archive/manual-validation/手工验收执行脚本-AI语音RAG主流程-2026-03-30.md`

4. `docs/规划-转写页剩余开发路线-2026-03-29.md`

1. 手工 10 条用例全部执行完毕并有证据。
2. P0/P1 缺陷为 0。
3. 自动化与手工记录一致。
4. 明确 go / no-go 结论：继续只开放转写页，或开始准备独立工作台解锁。

## 六、每批次固定产物

每个批次必须同时交付以下 5 项，缺一项视为未完成：

1. 代码改动
2. 聚焦测试或结构测试
3. 文档更新
4. 手工验收记录或证据补充
5. 批次结论与下一批入口

## 七、每批次固定门禁

1. `npm run typecheck`
2. `npm run build`
3. `npm test`
4. 本批影响链路的聚焦手工验收
5. 执行记录同步回填

任何一个门禁未完成，本批不得算收口完成。

## 八、go / no-go 条件

### 可以继续解锁其他工作台的条件

1. 转写页编排核已明显降到稳定区间，新增功能不再依赖页面层堆逻辑。
2. 选择语义、批量编辑、AI / 语音 / RAG 主链路已稳定。
3. 手工验收与自动化门禁完整可复跑。
4. 需要迁出的能力已经具备独立 controller/runtime，而不是只能复制转写页代码。

### 不可解锁的条件

1. Orchestrator 仍然是默认需求落点。
2. 语音与 AI 仍频繁回流改架构。
3. 手工验收记录仍停留在空模板。
4. citation / task 状态 / 批量编辑任一主路径仍存在高频回归。

## 九、建议的开工顺序

若按收益与风险排序，建议直接按以下顺序执行：

1. 先做包 0，确保后续收尾有统一门禁。
2. 再做包 1，继续降低页面层与语音层复杂度。
3. 然后做包 2，解决最容易产生“看起来能操作、实际语义不一致”的问题。
4. 接着做包 3，把 AI / 语音 / RAG 主路径真正做成可交付能力。
5. 最后做包 4，完成真实封账与开放条件判断。

这比“继续均匀推进多个页面”更稳，因为当前产品瓶颈已经不是功能名义覆盖率，而是转写主工作台是否真的收口。
