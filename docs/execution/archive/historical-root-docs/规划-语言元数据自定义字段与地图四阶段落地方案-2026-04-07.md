# 规划：语言元数据自定义字段与地图四阶段落地方案 - 2026-04-07

## 1. 目的

在现有语言元数据工作台已可用、数据库与服务层已基本稳定的前提下，把“用户自定义字段”和“地图/地名搜索”两条能力线，从可工作的业务实现，继续收敛到更接近业内成熟方案的工程形态。

本方案不推翻当前 Dexie、MapLibre、多 provider 路线，也不直接引入重型动态表单平台，而是在现有拆分基础上分四个阶段补足：

- 自定义字段统一 schema 与校验收口。
- 自定义字段 UI 从录入层升级到治理层。
- 地图搜索从轻量 wrapper 升级为可抽象的 geocoder 能力层。
- 地图交互从坐标输入器升级为完整位置编辑闭环。

当前主要差距：

1. 自定义字段仍然是“定义表 + 手工渲染 + 保存时按类型还原”，缺少统一 schema 和单一验证源。
2. 自定义字段 UI 还偏录入层，缺少 required、默认值、约束、字段分组、内联错误等治理能力。
3. geocoder 仍是单一函数封装，缺少 debounce、缓存、bias、reverse geocode、provider capability 等成熟能力。
4. 地图交互仍是单 marker 单坐标流程，缺少拖拽、视图联动、结果与地图状态统一等高级编辑闭环。

## 3. 总体策略

### 3.1 采纳的成熟做法

- 动态表单参考 React Hook Form 的低重渲染状态管理模式。
- 定义与值校验参考 Zod 的 schema 收口方式。
- schema-driven 方向参考 JSON Forms / Formily 的“定义、校验、渲染分离”原则，但本阶段不直接引入整套平台。
- 地图与搜索参考 Mapbox Search JS 的能力分层思路：UI 层、Core 层、Provider 层分离。
- 地理编码接口能力参考 Nominatim 的 structured query、countrycodes、viewbox、accept-language 等查询参数模型。

### 3.2 不采纳的做法

- 不在当前阶段直接引入 JSON Forms、Formily 作为运行时核心依赖。
- 不在当前阶段切换到单一地图 provider 绑定实现。
- 不把 geocoder UI 直接外包给单一厂商控件。

### 3.3 交付原则

1. 每阶段都必须独立可发布，不能依赖“大爆炸式重构”。
2. 每阶段都必须补回归测试、类型检查和构建验证。
3. 自定义字段和地图能力都保持 provider-neutral / renderer-neutral 的扩展方向。
4. 仅在新能力被证明需要时，才继续上升到更重的框架层。

## 4. 阶段一：自定义字段统一 Schema 与值协议收口

### 4.1 阶段目标

把当前散落在 shared helper、保存路径、service 输入约束中的字段规则，统一收敛为：

- 字段定义 schema。
- 字段值 schema。
- 字段渲染注册表。
- 草稿值与持久化值的显式转换适配层。

阶段一结束后，自定义字段不再依赖“多处 switch + 多处 parse/serialize”才能保证正确性。

### 4.2 主要改动面

建议新增：

- `src/pages/languageMetadataWorkspace.customFieldSchema.ts`
- `src/pages/languageMetadataWorkspace.customFieldRegistry.ts`
- `src/pages/languageMetadataWorkspace.customFieldAdapters.ts`

建议改造：

- [src/pages/languageMetadataWorkspace.shared.ts](../../../../src/pages/languageMetadataWorkspace.shared.ts)
- [src/pages/LanguageMetadataWorkspacePage.tsx](../../../../src/pages/LanguageMetadataWorkspacePage.tsx)
- [src/pages/languageMetadataWorkspace.customFieldController.ts](../../../../src/pages/languageMetadataWorkspace.customFieldController.ts)
- [src/pages/LanguageMetadataWorkspaceCustomFieldsSection.tsx](../../../../src/pages/LanguageMetadataWorkspaceCustomFieldsSection.tsx)
- [src/services/LinguisticService.languageCatalog.ts](../../../../src/services/LinguisticService.languageCatalog.ts)

### 4.3 具体任务

1. 定义字段定义 schema。
内容：`fieldType`、`name`、`options`、`description`、`required`、`defaultValue`、`placeholder`、`helpText`、`min`、`max`、`pattern`、`sortOrder`。

2. 定义字段值 schema。
内容：按 `text / number / boolean / select / multiselect / url` 生成值验证器，并明确空值语义。

3. 引入统一 adapter。
内容：把“持久化值 -> 草稿值”“草稿值 -> 持久化值”“字段定义 -> renderer props”三类转换从页面里剥离。

4. 建立 renderer registry。
内容：页面只根据字段定义选择 renderer，不再在主保存路径和 section 中重复按类型分支。

5. 把 service 的输入检查与 UI 的定义检查统一到同一套 schema。

### 4.4 验收标准

1. 新增字段类型约束后，只改 schema 和 renderer registry，不需要同时修改 3 到 4 个文件。
2. `buildDraft`、页面保存、section 渲染都走共享 adapter，不再各自维护独立协议。
3. 自定义字段定义无效时，UI 层和 service 层报错来源一致。
4. 现有 multiselect、number、boolean、select 全部回归通过。

### 4.5 风险与控制

- 风险：schema 落得太重，导致阶段一就出现大面积 UI 改动。
- 控制：阶段一先不替换整页表单状态管理，只收敛协议和验证源。

### 4.6 时间盒

- 预计 2 到 3 个工作日。

## 5. 阶段二：自定义字段 UI 从录入层升级到治理层

### 5.1 阶段目标

把当前“添加字段 / 管理字段 / 输入值”的 UI，升级为接近成熟内部平台的字段治理界面。

阶段二结束后，自定义字段不仅能用，还能稳定维护和扩展。

### 5.2 主要改动面

建议新增：

- `src/pages/LanguageMetadataWorkspaceCustomFieldDefinitionCard.tsx`
- `src/pages/LanguageMetadataWorkspaceCustomFieldValueField.tsx`
- `src/pages/languageMetadataWorkspace.customFieldForm.ts`

建议改造：

- [src/pages/LanguageMetadataWorkspaceCustomFieldsSection.tsx](../../../../src/pages/LanguageMetadataWorkspaceCustomFieldsSection.tsx)
- [src/pages/languageMetadataWorkspace.customFieldController.ts](../../../../src/pages/languageMetadataWorkspace.customFieldController.ts)
- [src/i18n/index.ts](../../../../src/i18n/index.ts)
- 相关样式文件

### 5.3 具体任务

1. 引入 React Hook Form 管理字段定义编辑态与字段值编辑态。
目标：减少大组件重复 rerender，并让字段级 dirty / error / touched 可见。

2. 补字段治理属性的 UI。
内容：required、defaultValue、placeholder、helpText、min/max、pattern、description。

3. 拆字段定义编辑器与字段值渲染器。
目标：管理态和录入态彻底分离，不再由一个 section 同时承载复杂 if/else。

4. 升级 multiselect 与 select 交互。
目标：支持可搜索候选、空态提示、选项说明，逐步淘汰原生 multiple select。

5. 增加字段分组与排序。
目标：为未来字段数量增长做准备。

6. 补字段级错误展示与说明文案。

### 5.4 验收标准

1. 字段定义编辑与字段值录入都有独立错误提示和 dirty 提示。
2. 新增 required / min / max / pattern 后能实际影响值输入与保存。
3. 字段 section 的职责拆分后，单文件不再重新膨胀。
4. 对 10 到 20 个自定义字段并存场景，交互无明显卡顿。

### 5.5 风险与控制

- 风险：一次性把所有字段治理能力都做完，拖慢交付。
- 控制：优先级按 `required -> defaultValue -> placeholder/helpText -> min/max/pattern -> 分组排序` 推进。

### 5.6 时间盒

- 预计 3 到 5 个工作日。

## 6. 阶段三：地图搜索能力抽象与 geocoder 收口

### 6.1 阶段目标

把当前地图搜索从“一个 geocodeSearch 函数 + 一个 controller”升级为能力明确、可替换 provider 的搜索层。

阶段三结束后，地图搜索具备成熟方案常见的基础能力：

- debounce
- request cache
- provider capability
- bias / filter
- reverse geocode
- query result normalization

### 6.2 主要改动面

建议新增：

- `src/components/languageGeocoder.ts`
- `src/components/languageGeocoder.providers.ts`
- `src/components/languageGeocoder.types.ts`
- `src/pages/languageMetadataWorkspace.geocodeCache.ts`

建议改造：

- [src/components/languageMapEmbed.shared.ts](../../../../src/components/languageMapEmbed.shared.ts)
- [src/pages/languageMetadataWorkspace.mapController.ts](../../../../src/pages/languageMetadataWorkspace.mapController.ts)
- [src/pages/LanguageMetadataWorkspaceGeographySection.tsx](../../../../src/pages/LanguageMetadataWorkspaceGeographySection.tsx)
- [src/components/LanguageMapEmbed.tsx](../../../../src/components/LanguageMapEmbed.tsx)

### 6.3 具体任务

1. 引入 geocoder provider capability model。
内容：`supportsForwardGeocode`、`supportsReverseGeocode`、`supportsBias`、`supportsCountryFilter`、`supportsStructuredQuery`。

2. 统一 geocode result shape。
内容：display label、lat/lng、provider metadata、bbox、confidence、raw payload。

3. 加 debounce 与短期 cache。
目标：减少重复请求和输入期间的无效流量。

4. 加 bias / filter 参数。
内容：国家过滤、视窗偏置、语言偏置、结果数量和 feature type。

5. 增加 reverse geocode。
目标：支持“点地图后反查地名”，形成编辑闭环。

6. 统一错误态。
目标：区分 abort、空结果、网络失败、provider key 缺失、provider 不支持。

### 6.4 验收标准

1. geocoder 请求不再每个按键都直发。
2. 同一查询短时间内重复搜索可命中 cache。
3. provider 不支持某能力时，UI 能显式降级而不是静默失败。
4. reverse geocode 能把地图点击结果回填为坐标和地点标签。

### 6.5 风险与控制

- 风险：provider capability 设计过度，导致引入一层复杂抽象但只服务一个 provider。
- 控制：只抽 forward、reverse、bias、filter 四个已明确需要的能力，不提前做路线规划、行政区下载等扩展。

### 6.6 时间盒

- 预计 3 个工作日。

## 7. 阶段四：地图交互与位置编辑闭环升级

### 7.1 阶段目标

把当前地图能力从“输入坐标 + 看一个 marker”，升级为位置编辑、搜索和地图视图联动的一套完整闭环。

阶段四结束后，地图能力应达到成熟内部工具的常见水平，而不是只作为附属预览。

### 7.2 主要改动面

建议新增：

- `src/components/LanguageMapMarker.tsx`
- `src/components/LanguageMapSearchResultLayer.tsx`
- `src/pages/languageMetadataWorkspace.mapViewState.ts`

建议改造：

- [src/components/LanguageMapEmbed.tsx](../../../../src/components/LanguageMapEmbed.tsx)
- [src/pages/languageMetadataWorkspace.mapController.ts](../../../../src/pages/languageMetadataWorkspace.mapController.ts)
- [src/pages/LanguageMetadataWorkspaceGeographySection.tsx](../../../../src/pages/LanguageMetadataWorkspaceGeographySection.tsx)
- 样式与测试文件

### 7.3 具体任务

1. 支持 draggable marker。
目标：直接拖拽调整坐标，并同步回填输入框。

2. 选中搜索结果后做相机联动。
目标：`flyTo` 或 `fitBounds`，而不是只写回经纬度。

3. 地图点击、marker 拖拽、搜索选择、反查结果共用同一套 view state。

4. 搜索结果列表与地图高亮联动。
目标：用户能清楚看到“当前选中的是哪个候选”。

5. 增加地图交互保护。
内容：bounds 约束、缩放边界、重复点击防抖、provider key 缺失降级提示优化。

6. 视情况增加行政区或 bbox 预览。
目标：为后续语言分布区域、行政区元数据维护做准备。

### 7.4 验收标准

1. 点击地图、拖 marker、选 geocode 候选三条路径最终都能稳定更新同一份位置状态。
2. 搜索结果选中后地图视图会自动移动到合理位置。
3. 无 provider key、provider 不支持某能力、网络失败三种情况的 UI 降级路径都清晰可见。
4. 地图交互新增后不引入 resize、popup、marker 生命周期回归。

### 7.5 风险与控制

- 风险：地图交互增强导致组件重新复杂化。
- 控制：view state、marker 逻辑、结果图层逻辑分组件拆分，避免重新把所有能力塞回 `LanguageMapEmbed`。

### 7.6 时间盒

- 预计 3 到 4 个工作日。

## 8. 阶段顺序与依赖关系

### 8.1 建议顺序

1. 阶段一：统一 schema 与协议。
2. 阶段二：自定义字段治理 UI。
3. 阶段三：geocoder 能力抽象。
4. 阶段四：地图交互升级。

### 8.2 原因

- 自定义字段的协议和 schema 若不先收口，阶段二会继续在不稳定地基上迭代 UI。
- 地图搜索能力若不先抽象，阶段四会把交互、provider、搜索、错误处理继续耦合在一起。
- 两条能力线彼此相对独立，可按阶段插空开发，但每条线都要先做协议层，再做体验层。

1. 阶段一完整落地。
2. 阶段三中的 debounce、cache、reverse geocode 子集。

原因：

- 阶段一决定自定义字段是否会继续在协议层重复出错。
- 阶段三决定地图搜索是否继续停留在轻量 wrapper 层。

阶段二和阶段四更偏体验与治理，可在协议层稳定后稳步推进。

2. 以阶段一为边界做第一轮实现与回归。
