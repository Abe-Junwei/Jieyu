## 一、已完成功能

### 1.1 AI Chat（对话功能）
- ChatOrchestrator 核心流式引擎，支持 primary/fallback 双 provider
- 9 个 LLM Provider：OpenAI、Anthropic、DeepSeek、Qwen、MiniMax、Ollama、Gemini、CustomHttp、OpenAICompatible
- SSE 流式解析、think 标签剥离（`<think>`/`</think>` 跨 chunk 处理）
- 工具调用计划与执行（4 个 adapter：segment、layer、gloss、token）
- 幻觉防护：ID 校验后执行
- FNV1A32 请求幂等性（去重）
- 3 种模式：enabled / gray（跳过工具，口述）/ rollback（跳过全部）
- Audit log 记录与回放

### 1.2 语音代理（Voice Agent）
- 状态机：idle / listening / routing / executing / ai_thinking
- 3 种模式：command（语音命令）/ dictation（听写）/ analysis（分析）
- 唤醒词检测（WwakeWordDetector）
- STT 引擎路由（区域检测 + 电量 + 噪声自适应）
- 意图 LLM 解析（VoiceIntentLlmResolver）
- 语音标注流水线（SpeechAnnotationPipeline）+ VAD 静默检测

### 1.3 RAG / Embedding
- Hybrid 检索：向量相似度 + BM25 关键词 + 全文重排
- EmbeddingSearchService 多源检索
- TaskRunner 并发队列 + 指数退避重试 + 过期任务恢复

### 1.4 工具调用处理器（useAiToolCallHandler）
- 4 个 adapter，幻觉防护
- 补偿/回滚机制（TTL 60s）
- 上下文自动填充

### 🔴 高优先级（会导致数据错误或功能失效）

#### B-01：compensationRef 共享覆盖问题 ✅ 已修复
**文件**：`src/hooks/useAiToolCallHandler.ts:623`
**问题**：`compensationRef` 是单个 `CompensationEntry | null`对象，两个工具调用在 60s 内创建层时，第二个的补偿条目覆盖第一个，第一个层创建失败时无法回滚
**证据**：
```typescript
// line 623: 单一共享 ref
const compensationRef = useRef<CompensationEntry | null>(null);
// line 346: 写入时直接覆盖
compensationRef.current = { layerId: newLayer.id, layerType: 'translation', createdAt: Date.now() };
```
**修复**：改用 `Map<callId, CompensationEntry>`，以调用 ID 为 key，不共享 ref

#### B-02：Hybrid 搜索加载全部 utterance_texts ✅ 已修复
**文件**：`EmbeddingSearchService.ts:374-385`（原 `hybridSearchForRerank` 函数）
**问题**：当 `sourceTypes` 包含 `'utterance'` 且 `candidateSet` 未覆盖所有来源时，无条件执行 `.find().exec()` 加载全部 utterance_texts，大项目内存爆炸
**修复**：在加载前增加 `sourceTypes` 过滤条件 + 时间窗口限制，无 candidateSet 时跳过 utterance 分支
**证据**：
```typescript
// line 374-385: 无任何过滤条件的全量加载
const utteranceRows = await db.collections.utterance_texts.find().exec();
const merged = new Map<string, string[]>();
for (const row of utteranceRows) {
  // 即使传入了 candidateSet，BM25 候选阶段仍然全量加载
```
**修复**：在执行 `find().exec()` 前按 `candidateSet` 或 mediaId 过滤，或分页加载

#### B-03：TaskRunner status 更新用 insert 而非 update ✅ 已修复
**文件**：`src/ai/tasks/TaskRunner.ts:336-343`
**问题**：`findOne` 后 `insert`（非 `update`），status 可能不持久化（若 Dexie 对相同 id 的 insert 行为是追加而非覆盖）
**证据**：
```typescript
// line 333-343
const prev = row.toJSON();
await db.collections.ai_tasks.insert({  // ❌ 应为 update
  ...prev,
  status,
  updatedAt: nowIso(),
  ...(errorMessage ? { errorMessage } : {}),
});
```
**修复**：改用 `db.collections.ai_tasks.update(taskId, { status, updatedAt: nowIso(), ... })`

#### B-04：AutoGlossService 重复 insert ✅ 已修复
**文件**：`src/ai/AutoGlossService.ts:145-149`
**问题**：`insert` 而非 `update`，同一 token 已存在 gloss 时会创建重复条目（而非更新）
**证据**：
```typescript
// line 145-149
await db.collections.utterance_tokens.insert({
  ...token,
  gloss,
  updatedAt: now,
});  // ❌ token 已存在 gloss 时应 update
```
**修复**：改用 `db.collections.utterance_tokens.update(token.id, { gloss, updatedAt: now })`

#### 🆕 B-14：applyAssistantMessageResult 使用 insert 造成重复记录 ✅ 已修复
**文件**：`src/hooks/useAiChat.ts:2236-2246`、`db.ts`
**问题**：`finalizeAssistantMessage` 调用 `applyAssistantMessageResult` 时，对已存在的 ai_message 记录使用 `insert` 而非 `update`（Dexie 中 insert 对相同 id 的行为是覆盖而非追加，但若记录已被其他流并发修改可能丢数据）
**证据**：
```typescript
// line 2240-2246
await dbRef.collections.ai_messages.insert({
  ...row,
  content,
  status,
  ...(errorMessage ? { errorMessage } : {}),
  updatedAt: nowIso(),
});  // ❌ 应为 update
```
**修复**：改用 `db.ai_messages.update(messageId, { content, status, ... })`

#### 🆕 B-15：confirmPendingToolCall 使用 insert 覆盖状态 ✅ 已修复（随B-14）
**文件**：`src/hooks/useAiChat.ts`
**问题**：在 `confirmPendingToolCall` 的错误处理和成功处理中也使用 `insert` 而非 `update`
**证据**：同 B-14，`insert` 用于更新已存在的记录

#### 🆕 B-17：ChatOrchestrator streamWithFallback 非重试错误后不自动降级 ✅ 已修复
**文件**：`src/ai/ChatOrchestrator.ts:91-113`
**问题**：`streamWithFallback` 只在 `catch(retryableError)` 时降级，若主 provider 抛出**非重试性错误**（如 401/403/格式错误），fallback 不会触发（因为错误被向上抛出而非被捕获）
**证据**：
```typescript

```
**修复**：对非重试性错误也考虑降级，或在 useAiChat 层面捕获并切换

#### 🆕 B-18：useAiChat 历史记录水合使用 insert 而非 update
