# 规划：面板化语言资产按需搜索与分片加载方案（2026-04-07）

## 1. 背景

未来语言元数据工作区、正字法工作区、正字法桥接工作区将从独立页面收敛为面板功能。面板场景下不再适合“开箱即用加载数百条语言/正字法列表并直接铺给用户”，因为这会同时带来三个问题：

1. 首次打开面板时加载过重，影响转写主工作流。
2. 用户并不需要浏览全部 700+ / 500+ 条目，真正诉求是“在输入框里准确找到需要的那一项”。
3. 当前全量基线语言名数据被打入运行时 JS，已经形成超大语言映射 chunk。

本方案目标不是削弱语言资产能力，而是把“全量可查”与“默认全量可见”彻底分离。

## 2. 当前现状

### 2.1 语言名称数据链路

当前语言名称基线由 [scripts/generate-language-name-indexes.mjs](../../../../scripts/generate-language-name-indexes.mjs) 生成，产物包括：

1. `public/data/language-support/language-display-names.core.json`
2. `public/data/language-support/language-query-aliases.json`
3. `public/data/language-support/language-query-index.{locale}.json`
4. `src/data/generated/languageNameCatalog.generated.ts`

运行时主要通过以下链路消费：

1. [src/data/languageCatalogRuntimeCache.ts](../../../../src/data/languageCatalogRuntimeCache.ts) 直接导入 `languageNameCatalog.generated.ts` 作为 baseline。
2. [src/data/languageNameCatalog.ts](../../../../src/data/languageNameCatalog.ts) 读取 runtime cache。
3. [src/utils/langMapping.ts](../../../../src/utils/langMapping.ts) 基于 runtime cache 做语言显示名解析、别名解析、查询匹配。
4. [src/components/LanguageIsoInput.tsx](../../../../src/components/LanguageIsoInput.tsx) 通过 reducer + `langMapping` 走同步建议列表。

这意味着“只要任何语言输入框需要搜索”，整个大语言名数据就会进入前端运行时。

### 2.2 三个工作区当前行为

1. 语言元数据工作区在打开时会加载目录结果集，并默认呈现左侧语言列表。
2. 正字法工作区在打开时会拉取 `listOrthographies({ includeBuiltIns: true })`，默认可见完整正字法目录。
3. 正字法桥接工作区在打开时也会拉取完整正字法目录，并在桥接编辑器中继续展开源/目标选择。

这些行为适合独立管理页，但不适合“嵌入式面板”。

## 3. 目标

### 3.1 产品目标

1. 面板初开时只展示与当前上下文强相关的少量条目。
2. 用户在输入框中键入名称、别名、ISO 代码、BCP 47 标签时，仍能准确定位目标条目。
3. 搜索结果以建议列表形式出现，不要求用户浏览完整目录。

### 3.2 性能目标

1. 不再把全量语言名称基线直接打入主运行时 JS。
2. 面板首开不应请求全量 700+ / 500+ 条目列表。
3. 搜索请求与候选渲染应走 debounce + 缓存，避免每次按键全量扫描。

### 3.3 架构目标

1. 将“同步 baseline 大对象”改为“按需加载的搜索索引 + 详情读取”。
2. 保留当前 `LanguageIsoInput` 的成熟交互语义，但替换其建议源。
3. 让语言元数据、正字法、桥接三类面板共享同一套“搜索驱动选择器”基础设施。

## 4. 总体整改思路

### 4.1 原则

1. 不再默认展示全量目录，只展示上下文相关子集。
2. 搜索与详情分离：搜索只返回轻量候选，选中后再加载完整详情。
3. baseline 数据改为静态 JSON / 分片索引，运行时只在需要时 fetch。
4. 用户覆盖数据继续保留在 IndexedDB / localStorage runtime cache，并在搜索结果合并时优先。

### 4.2 目标运行模型

运行时拆成四层：

1. `Panel bootstrap subset`
   只包含当前条目、项目使用中条目、最近使用条目、固定推荐条目。
2. `Search index shards`
   用于输入框搜索，返回轻量候选，不含完整详情。
3. `Detail lookup`
   用户选中候选后，再按 ID 读取完整语言/正字法/桥接详情。
4. `User override merge`
   将用户自定义语言、覆盖条目、工作区内新增正字法/桥接结果叠加到候选与详情视图上。

## 5. 语言资产整改方案

### 5.1 baseline 数据从 TS 大对象切到 JSON 分片

已有脚本已经生成 JSON，因此整改不需要推翻生成器，只需要调整产物组织与运行时入口。

建议将语言名数据拆成三类文件：

1. `language-core.min.json`
   仅保留按 ID 读取详情所需的最小核心字段：`languageCode`、`english`、`native`、`byLocale`、`aliases`、可选坐标。
2. `language-query-index.{locale}.{bucket}.json`
   按 locale + bucket 分片的候选索引，只存轻量匹配项。
3. `language-alias-map.{bucket}.json`
   仅用于 exact alias / ISO / tag 命中的快速映射。

其中 `bucket` 建议优先按归一化前缀分桶，而不是每个 locale 一个整文件。推荐两级：

1. 一级：首字符 bucket
2. 二级：高频字符下再按前 2 个字符细分

这样用户输入 `en`、`zh`、`hak`、`客家` 时，只需拉取极少量索引文件。

### 5.2 运行时不再以 `languageCatalogRuntimeCache` 持有全量 baseline

当前 [src/data/languageCatalogRuntimeCache.ts](../../../../src/data/languageCatalogRuntimeCache.ts) 会在模块初始化时把 `GENERATED_LANGUAGE_DISPLAY_NAME_CORE` 全部组装进 baseline cache。整改后应改为：

1. baseline cache 默认只含空对象或极小 subset。
2. `readLanguageCatalogRuntimeCache()` 只负责读取用户覆盖层和已加载 subset。
3. baseline 详情通过 `LanguageCatalogSearchService` 懒加载并写回内存缓存。

即：runtime cache 只缓存“已访问过的条目”，而不是“整个世界”。

### 5.3 搜索从同步 `langMapping` 扫描改成异步 search service

新增服务建议：

1. `LanguageCatalogSearchService.search(query, locale, options)`
2. `LanguageCatalogSearchService.lookupById(id, locale)`
3. `LanguageCatalogSearchService.lookupMany(ids, locale)`

搜索策略：

1. exact code / exact alias / exact locale label 命中优先。
2. prefix 命中次之。
3. token contains / normalized contains 最后。
4. 用户覆盖条目优先于 baseline。
5. 项目使用中 / 最近使用的条目在同分时前排。

返回结果不要直接暴露完整 `LanguageCatalogEntry`，只返回建议展示足够字段：

1. `id`
2. `languageCode`
3. `primaryLabel`
4. `secondaryLabel`
5. `matchSource`
6. `rank`

### 5.4 `LanguageIsoInput` 改造方向

[src/components/LanguageIsoInput.tsx](../../../../src/components/LanguageIsoInput.tsx) 当前交互语义成熟，不建议重写；应保留 reducer 模型，只替换建议来源。

建议新增一层 adapter：

1. reducer 继续管理编辑态、provisional ISO、blur 语义、键盘高亮。
2. 建议结果不再由同步 `langMapping` 直接生成，而由异步 hook 注入。
3. hook 负责 debounce、取消过时请求、结果缓存。

建议接口：

1. `useLanguageSearchSuggestions(query, locale, context)`
2. `LanguageIsoInput` 接收 `suggestionProvider` 或 `searchAdapter`

### 5.5 语言元数据面板行为调整

面板化后，语言元数据不再默认展示全量左侧目录。建议改为三段：

1. `当前条目`
   当前选中的语言或当前上下文推导出的语言。
2. `项目使用中 / 最近编辑`
   仅展示少量高相关语言。
3. `搜索结果`
   当用户输入时出现，不输入则为空。

加载策略：

1. 面板初开：`lookupMany(projectLanguageIds + recentIds + currentId)`
2. 用户搜索：走 `LanguageCatalogSearchService.search`
3. 用户选中候选：再 `lookupById`

这样可以保留“准确找语言”的能力，而去掉“默认灌一整页目录”的行为。

## 6. 正字法工作区整改方案

### 6.1 正字法列表不再默认全量展示

当前正字法工作区默认 `listOrthographies({ includeBuiltIns: true })`。改造后建议默认只展示：

1. 当前语言已绑定正字法
2. 项目中实际使用的正字法
3. 最近新建 / 最近编辑的正字法

不再把全部 built-in orthographies 直接铺给用户。

### 6.2 语言选择与正字法选择都改为搜索驱动

正字法工作区内至少有两类选择器：

1. 语言选择器
2. 同语言下正字法选择器

建议行为：

1. 语言选择先通过异步语言搜索命中语言 ID。
2. 命中后才加载该语言下的 built-in + user orthographies。
3. 若用户明确输入 orthography 名称，则再做二级 orthography 搜索。

建议新增服务：

1. `OrthographyCatalogSearchService.searchLanguages(query, locale)`
2. `OrthographyCatalogSearchService.listByLanguage(languageId)`
3. `OrthographyCatalogSearchService.searchOrthographies(query, filters)`

### 6.3 built-in orthography baseline 也应外部化

虽然当前 built-in orthography chunk 不像语言名那样夸张，但面板产品语义已经改变，因此 baseline 也应改为：

1. 保留最小 manifest
2. 详情按语言或类型分片
3. 只有在用户选中相关语言时才加载对应正字法清单

## 7. 正字法桥接工作区整改方案

桥接面板比正字法面板更需要按需加载，因为它天然涉及双边选择。

建议默认只提供：

1. 当前目标正字法
2. 最近用过的源正字法 / 桥接规则
3. 项目中出现过的 orthography 对

而不是直接把所有可桥接组合展开。

### 7.1 双选择器搜索流程

1. 先确定目标语言 / 目标正字法
2. 再搜索源语言 / 源正字法
3. 只在两端都明确后加载桥接候选

新增服务建议：

1. `OrthographyBridgeSearchService.searchSourceOrthographies(query, context)`
2. `OrthographyBridgeSearchService.searchTargetOrthographies(query, context)`
3. `OrthographyBridgeSearchService.listBridgeCandidates(sourceId, targetId)`

### 7.2 默认结果集控制

默认最多展示：

1. 当前目标相关桥接
2. 最近使用桥接
3. 当前项目语言对相关桥接

只有用户显式搜索时才扩大范围。

## 8. 统一 UI/交互策略

### 8.1 面板首开展示

三个面板统一遵循：

1. 不展示全量目录
2. 先展示上下文相关 subset
3. 搜索为空时不出现巨大滚动列表

### 8.2 搜索输入框行为

统一要求：

1. debounce 150-250ms
2. 结果上限 8-12 条
3. exact 命中置顶
4. 支持名称、别名、ISO 639-3、ISO 639-1、BCP 47 tag 混合检索
5. blur 后保留已选项，不因 locale 切换重置

### 8.3 结果分组建议

搜索结果建议分组：

1. `最近使用`
2. `项目使用中`
3. `精确匹配`
4. `其他结果`

这样既保留准确性，也能减少用户面对大列表时的认知压力。

## 9. 迁移步骤

### Phase 1：语言搜索运行时去全量 baseline

1. 新增 `LanguageCatalogSearchService`
2. `languageCatalogRuntimeCache` baseline 改为最小空壳 + 已访问缓存
3. `LanguageIsoInput` 切换为异步建议源
4. 保留现有 API 签名，优先兼容上层调用

### Phase 2：语言元数据面板列表收缩

1. 左侧目录改为 `当前 + 项目使用中 + 最近编辑 + 搜索结果`
2. 删除页面级默认全量目录加载
3. 搜索只拉候选，不直接拉完整目录

### Phase 3：正字法 / 桥接面板改为搜索驱动

1. built-in orthography baseline 外部化
2. 正字法列表改成按语言加载
3. 桥接源/目标都改成搜索驱动选择器

### Phase 4：索引分片深化

1. 把 `language-query-index.{locale}.json` 进一步切成 bucket shard
2. alias map 同步分片
3. 增加 LRU 内存缓存与请求去重

## 10. 验收标准

### 10.1 性能

1. 面板首开不再加载全量 700+ / 500+ 条目列表。
2. 语言名 baseline 不再直接形成超大运行时 chunk。
3. 语言输入首次键入后的网络请求量与结果渲染在可接受范围内。

### 10.2 功能

1. 输入中文名、英文名、别名、ISO 代码、BCP 47 标签都能命中对应条目。
2. 用户自定义语言和覆盖条目能在搜索结果中优先出现。
3. 正字法与桥接选择器只在必要时加载相关条目。

### 10.3 交互

1. 面板打开后默认无巨大列表压迫用户。
2. 搜索结果足够稳定，不出现输入一个字符就大量抖动重排。
3. locale 切换不会冲掉用户正在编辑的草稿。

## 11. 推荐的第一步实施项

如果只做一件最有价值的事情，建议先做：

1. 将 `LanguageIsoInput` 的建议源从同步 `langMapping` 全量扫描切换到异步搜索服务。

原因：

1. 它是三个面板共同依赖的底层能力。
2. 它能直接消除“默认全量 baseline 必须进运行时”的前提。
3. 后续语言元数据、正字法、桥接三处都能复用同一套搜索基础设施。
