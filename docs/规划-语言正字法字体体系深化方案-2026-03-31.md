# 规划：语言-正字法-字体体系深化方案

> 2026-03-31 · 解语 (Jieyu) · 面向 language / orthography / font 的平台级收口规划

---

## 〇、结论先行

当前仓库已经完成了语言-正字法-字体体系的**核心业务闭环**：

- 项目可绑定主语言与主正字法
- 层可绑定语言与正字法
- 可在创建流程中直接新建正字法、复制同语正字法、从其他语言派生正字法
- 样式层已根据语言/正字法解析脚本，并据此推荐/过滤字体

这意味着“可创建、可持久化、可在 UI 中使用”的第一阶段已经成立。

但若以 FLEx / CLDR / SLDR / ICU / CSS Fonts / W3C bidi 的成熟标准衡量，现状仍未达到“书写系统平台级完备”。当前缺口主要集中在四层：

1. **正字法元数据层偏薄**：缺少书写方向、示例字符、地区/变体、归一化与排序规则、首选字体栈等元数据
2. **转换能力尚未落地**：`conversionRules` 仍是存储预留，尚未成为可执行、可预览、可测试的转换系统
3. **字体层仍以静态推荐为主**：尚未形成覆盖率校验、回退链、复杂脚本 shaping 能力校验
4. **双向文字与互操作尚未收口**：尚无系统性的 bidi 安全策略，也未与外部 writing system 数据模型形成稳定映射

本规划的目标不是推翻现有设计，而是在现有实现之上，把“可用闭环”推进到“稳定、可扩展、可互操作”的第二阶段。

---

## 一、目标与非目标

### 1.1 目标

本轮深化要实现以下结果：

1. 定义正字法模型 v2，使其能够表达一个可独立运作的 writing system profile
2. 将 `conversionRules` 升级为真实可执行的转换能力，而不只是预留字段
3. 将字体系统从“脚本推荐表”升级为“脚本推荐 + 覆盖率判断 + 回退链”的组合策略
4. 为 RTL / 混排场景建立明确的渲染安全边界
5. 为未来的 EAF 互操作、外部数据源映射、AI 辅助生成预留稳定合同

### 1.2 非目标

本规划明确不在当前阶段做以下事项：

- 不做类似 Aegisub 的行内 override tag 体系
- 不做完整输入法/键盘安装分发平台
- 不做跨所有语言自动生成正字法的“黑箱 AI”能力
- 不在本轮引入重量级外部数据库镜像（如完整 CLDR / ScriptSource 全量离线镜像）
- 不承诺一次性解决所有历史脚本、罕见脚本的字体缺字问题

## 二、现状评估

### 2.1 已完成能力

- **数据落库已成立**：项目层已有 `primaryOrthographyId`，层已有 `orthographyId`，正字法已是独立实体
- **创建流完整**：项目创建、层创建都能直接接入正字法选择与创建
- **派生流完整**：已支持 IPA 建立、同语复制、跨语派生
- **渲染流已打通**：脚本解析、IPA 模式判断、字体预设推荐、本地字体过滤均已接入
- **设计方向正确**：遵循“层粒度样式”与“语言驱动字体”而非任意视觉样式拼装

### 2.2 当前不足

#### A. 正字法仍偏“标签对象”，不是“书写系统档案”

目前 `OrthographyDocType` 主要承载名称、语言、类型、脚本、转换规则预留与备注。它足以支撑选择与展示，但不足以支撑以下需求：

- 同一语言的地区变体区分
- 示例字符/字母表校验
- 正规化与比较规则
- 书写方向与混排处理
- 首选字体栈与行高策略
- 与外部标准模型对齐

#### B. 派生能力偏“复制”，不是“转换工作流”

当前“从其他语言派生正字法”本质上是复制源正字法元数据，再改目标语言归属。它适合快速建档，但不等于：

- 定义从 A 正字法到 B 正字法的映射规则
- 对样例文本执行转换并预览结果
- 验证转换是否可逆、是否丢信息
- 在导入/导出/AI 工作流中复用转换能力

#### C. 字体层缺少“实际可显示性”判定

现有 `SCRIPT_FONT_MAP` 已经比多数同类工具更强，但它仍然是经验映射，不等于真实可显示保障。对于低资源语言与复杂附标，缺口主要体现在：

- 未对示例字符集做覆盖校验
- 未识别字体只有 cmap 覆盖但 shaping 错误的情况
- 未建立 per-orthography 的字体栈与优先级
- 未根据脚本推荐自动联动行高/字重/字号策略

#### D. bidi 与 plain-text 安全边界未制度化

如果后续支持 Arabic / Hebrew / Urdu / Uyghur 等脚本，仅有字体还不够。必须明确：

- 文本输入框如何设置 `dir`
- 混排标签、片段预览、导出纯文本如何做 isolate
- 搜索命中、上下文片段、内联标点数字如何避免方向串扰

#### E. 互操作能力还没有“合同层”

当前模型主要服务内部 UI 与内部持久化。若未来要与 EAF、导出包、AI 任务、外部语言资源协同，则还需：

- 明确正字法的稳定 ID 策略
- 明确与 BCP 47 / ISO 15924 / ISO 639-3 的映射字段
- 明确导入导出时的降级行为

## 三、目标架构

建议将体系拆成五层，每层职责单一，避免继续把所有信息堆进一个对象里：

### 3.1 Language Profile

表示语言实体，负责：

- `languageId`（ISO 639-3）
- 多语言名称
- 默认脚本候选
- 外部元数据来源标记

### 3.2 Orthography Profile

表示某语言下的一个具体书写系统/正字法实例，负责：

- 名称、简称、所属语言
- 书写类型（phonemic / phonetic / practical / historical / other）
- 脚本标签、方向、地区/变体
- 示例字符、规范化规则、显示偏好
- 与字体栈、转换规则的绑定

### 3.3 Transform Profile

表示正字法之间的转换合同，负责：

- 源/目标正字法
- 规则定义
- 可逆性声明
- 测试样例
- 版本与状态

### 3.4 Font Profile

表示针对正字法的推荐与回退策略，负责：

- 首选字体
- 回退字体栈
- 覆盖率缓存
- 行高/字重/字号修正建议

### 3.5 Render Policy

表示 UI 侧真正用来渲染的最终策略，负责：

- 文本方向
- `font-family` 回退链
- 是否需要 bidi isolate
- 默认行高/字号/字重建议
- 平台差异化回退

核心原则：

- `Language` 不直接决定最终字体
- `Orthography` 决定大部分渲染语义
- `Transform` 独立建模，不嵌死在正字法对象里
- `Render Policy` 是运行时产物，不是所有字段都直接持久化

## 四、数据模型 v2

### 4.1 OrthographyDocType v2 建议字段

建议在现有对象上扩展，而不是推倒重建：

```ts
interface OrthographyDocType {
  id: string;
  languageId?: string;
  name: MultiLangString;
  abbreviation?: string;
  type?: 'phonemic' | 'phonetic' | 'practical' | 'historical' | 'other';
  scriptTag?: string;
  localeTag?: string;
  regionTag?: string;
  variantTag?: string;
  direction?: 'ltr' | 'rtl' | 'ttb' | 'btt';
  exemplarCharacters?: {
    main?: string[];
    auxiliary?: string[];
    numbers?: string[];
    punctuation?: string[];
    index?: string[];
  };
  normalization?: {
    form?: 'NFC' | 'NFD' | 'NFKC' | 'NFKD';
    caseSensitive?: boolean;
    stripDefaultIgnorables?: boolean;
  };
  collation?: {
    base?: string;
    customRules?: string;
  };
  fontPreferences?: {
    primary?: string[];
    fallback?: string[];
    mono?: string[];
    lineHeightScale?: number;
    sizeAdjust?: number;
  };
  inputHints?: {
    keyboardLayout?: string;
    imeId?: string;
    deadKeys?: string[];
  };
  bidiPolicy?: {
    isolateInlineRuns?: boolean;
    preferDirAttribute?: boolean;
  };
  conversionRules?: Record<string, unknown>;
  notes?: MultiLangString;
  createdAt: string;
  updatedAt?: string;
}
```

### 4.2 建议新增实体

建议新增以下文档类型，而不是继续把所有转换和字体缓存压进 `OrthographyDocType`：

#### A. OrthographyTransformDocType

用于表达一条显式转换链路：

- `id`
- `sourceOrthographyId`
- `targetOrthographyId`
- `engine`：`icu-rule` / `table-map` / `custom-js` / `manual`
- `rules`
- `isReversible`
- `inverseTransformId`
- `sampleCases`
- `status`
- `createdAt` / `updatedAt`

#### B. OrthographyRenderProfileDocType

用于缓存运行时计算出的渲染策略：

- `orthographyId`
- `resolvedScriptTag`
- `resolvedDirection`
- `fontStack`
- `monoFontStack`
- `lineHeightScale`
- `coverageSummary`
- `updatedAt`

#### C. FontCoverageCacheDocType

用于保存本地/预设字体对示例字符集的检测结果，避免重复探测：

- `fontFamily`
- `scriptTag`
- `orthographyId?`
- `coveredCharacters`
- `missingCharacters`
- `hasComplexShapingConfidence`
- `checkedAt`

### 4.3 Project / Layer 扩展建议

#### Project

- 保留 `primaryLanguageId`
- 保留 `primaryOrthographyId`
- 建议新增 `preferredRenderOrthographyId?`
- 建议新增 `defaultBidiPolicy?`

#### Layer

- 保留 `languageId`
- 保留 `orthographyId`
- 建议新增 `renderOverrides?`

`renderOverrides` 仅用于少量显示偏差，而不是替代正字法本身：

- `fontFamily?`
- `fontSize?`
- `bold?`
- `italic?`
- `lineHeightScale?`
- `direction?`

原则上：

- 默认渲染应来自 orthography render policy
- layer 只允许做少量 override
- 不允许 layer 自己演化成另一套正字法对象

## 五、能力建设路线

### 5.1 Phase A：正字法元数据加厚

**目标**：把正字法从“可选标签”升级为“可描述书写系统的档案对象”。

#### 交付项

1. 扩展 `OrthographyDocType`
2. 新增方向、示例字符、字体偏好等编辑 UI
3. 为已存在正字法提供默认迁移值
4. 在项目创建与层创建中支持更完整的预览信息

#### UI 要求

- 正字法编辑面板需显示：脚本、方向、类型、地区/变体、示例字符、首选字体栈
- 创建时只暴露必要字段；高级字段可折叠
- 预览区域至少展示一段示例文本和最终字体栈

#### 验收标准

- 已有数据迁移后不丢失现有绑定关系
- 任一正字法都可生成稳定的 render policy
- UI 在无高级字段时自动降级，不阻塞现有流程

### 5.2 Phase B：转换规则落地

**目标**：把 `conversionRules` 从“存储字段”升级为“能运行、能验证、能复用”的基础设施。

#### 交付项

1. 新增 `OrthographyTransformDocType`
2. 支持最小可用的规则引擎
3. 提供转换预览器与样例测试
4. 在“跨语言派生正字法”时允许选择是否继承转换链

#### 第一阶段引擎建议

优先级如下：

1. `table-map`：字符/字符串映射表，适合大多数 practical orthography 场景
2. `icu-rule`：规则链式转换，适合复杂附标、正规化、过滤与反向映射
3. `manual`：无法自动逆推的场景由人工确认

#### 验收标准

- 任一 transform 都可对样例文本输出确定结果
- 可标记“可逆 / 不可逆 / 部分可逆”
- 有独立测试保证不会因规则改动导致回归

### 5.3 Phase C：字体层从推荐升级到验证

**目标**：把“推荐字体”升级成“可实际显示”的策略。

#### 交付项

1. 为 orthography 生成 `fontStack`
2. 基于 `exemplarCharacters` 做字符覆盖校验
3. 允许记录本地字体覆盖缓存
4. 对高风险脚本引入 stricter fallback policy

#### 设计要求

- 预设字体只作为起点，不作为真值
- 如果首选字体缺字，应自动退到次级字体
- 如果所有候选都缺字，UI 需明确警告，而不是静默渲染乱码/豆腐块

#### 验收标准

- 给定 exemplar 字符集，可输出覆盖摘要
- 字体菜单可区分“推荐”“可显示”“风险高”三类状态
- 对 Latin + IPA、Arabic、Devanagari、Tibetan 至少各有一组验证用例

### 5.4 Phase D：bidi 与混排安全收口

**目标**：保证 RTL/LTR 混排在输入、展示、导出三条路径上行为一致。

#### 交付项

1. 正字法层补 `direction` 与 `bidiPolicy`
2. 输入组件根据 orthography 自动设置 `dir`
3. 片段预览、标签、搜索结果统一走 isolate 包裹策略
4. 纯文本导出明确是否插入 Unicode isolate 控制字符

#### 验收标准

- Arabic/Hebrew 与 Latin 数字/标点混排不出现方向串扰
- 预览、搜索、复制、导出行为一致
- 至少有一组针对混排文本的回归测试

### 5.5 Phase E：互操作与外部映射

**目标**：让本体系可与外部数据和未来功能长期兼容。

#### 交付项

1. 统一语言/脚本/地区/变体字段映射
2. 明确导入导出降级策略
3. 为 EAF / AI 任务 / 外部资源同步设计 provider-neutral 合同
4. 记录 provenance 与来源可信度

#### 验收标准

- 导出时可保留 orthography identity
- 导入未知字段时有稳定降级路径
- 内部 ID 与外部标签不混淆

## 六、实施顺序

建议按以下顺序推进，避免一开始就做过重的底层改造：

### P0：Schema 与运行时收口

优先做：

1. `OrthographyDocType` 扩展字段
2. `resolveRenderPolicyForOrthography()` 统一运行时出口
3. layer/project 渲染路径改为优先读取 render policy
4. 迁移脚本与回归测试

原因：如果没有统一的 render policy 出口，后续 font/bidi/transform 都会继续分散在多个组件和工具函数里。

### P1：转换与预览

优先做：

1. `OrthographyTransformDocType`
2. 最小转换引擎
3. 样例预览器
4. 派生流程联动 transform

原因：这一步能把“从其他语言派生”真正从复制升级为能力。

### P2：字体覆盖率与风险提示

优先做：

1. exemplar-based coverage check
2. fallback stack 生成
3. 本地字体缓存与风险提示

原因：这一步对终端用户感知最强，但依赖 P0 的 render policy 稳定输出。

### P3：bidi 与互操作

优先做：

1. 输入/展示/导出方向策略统一
2. 互操作字段落定
3. 外部导入导出降级策略

原因：它是质量收口项，不宜早于基础模型稳定之前大范围改 UI。

## 七、迁移策略

### 7.1 数据迁移原则

- 老数据必须零损迁移
- 没有高级字段的老正字法对象也必须可继续使用
- 所有新增字段都应支持延迟填充

### 7.2 迁移步骤

1. 为现有 orthography 默认填充：
   - `direction`: 从 `scriptTag` 推断，未知则 `ltr`
   - `fontPreferences.primary`: 从现有 `SCRIPT_FONT_MAP` 推导
   - `bidiPolicy`: `rtl` 时默认开启 isolate
2. 为历史 layer 保持原 `displaySettings` 不变
3. 在运行时生成 render policy 时，先取 layer override，再取 orthography，再取 script fallback

### 7.3 兼容原则

- 旧 UI 不认识的新字段不能导致崩溃
- 无 transform 的正字法仍可正常创建与选择
- 无 exemplarCharacters 的正字法仍可正常推荐字体，但不给覆盖率承诺

## 八、测试与门禁

### 8.1 单元测试

必须补齐：

- `resolveRenderPolicyForOrthography()`
- `resolveDirectionForOrthography()`
- `buildFontStackForOrthography()`
- transform engine 的正向/逆向/失败分支

### 8.2 组件测试

必须覆盖：

- 项目创建中的 orthography 高级字段展示
- 层创建时 orthography 预览与回退
- 字体菜单中的“推荐/可显示/风险”标识
- RTL 输入框与搜索结果展示

### 8.3 集成测试

至少覆盖以下主路径：

1. 创建项目 → 新建 practical orthography → 创建层 → 预览字体生效
2. 基于 IPA 正字法创建 transform → 转换样例 → 保存 → 再次加载结果一致
3. Arabic/Hebrew 正字法输入与搜索结果方向正确
4. 导出再导入后 orthography identity 不丢失

### 8.4 发布门禁

发布前至少满足：

- build 通过
- orthography / render policy / transform 相关测试全绿
- 至少完成 Latin IPA、Arabic、Devanagari、Tibetan 的手工验收

## 九、风险清单

### 9.1 模型膨胀风险

风险：正字法对象越加越大，最终失去边界。

控制方式：

- 只把“正字法自身属性”放进 `OrthographyDocType`
- 把 transform、coverage cache、render profile 拆成独立实体

### 9.2 过早接入复杂规则引擎风险

风险：一开始就做完整 ICU 规则编辑器，复杂度过高。

控制方式：

- 先落 `table-map`
- 再支持 `icu-rule`
- UI 初版只支持样例预览，不做高级调试器

### 9.3 字体检测成本风险

风险：覆盖率检测可能增加运行时开销。

控制方式：

- 仅对 exemplarCharacters 做采样检测
- 引入 cache
- 对无权限浏览器降级为静态推荐

### 9.4 bidi 改造扩散风险

风险：输入、预览、搜索、导出各自处理方向，导致策略碎片化。

控制方式：

- 统一由 render policy 输出方向策略
- 组件层禁止自发散写 `dir` 与 isolate 逻辑

---

## 十、完成定义

当且仅当以下条件同时成立，才可认为“语言-正字法-字体体系”达到第二阶段完备：

1. `OrthographyDocType` 足以表达书写系统关键元数据
2. render policy 成为唯一可信渲染入口
3. transform 能被执行、预览、测试，而非仅存储
4. 字体系统能给出回退链与覆盖风险，而非仅推荐名称
5. RTL / 混排场景在输入、展示、导出中都安全
6. 导入导出不会破坏 orthography identity

未达到上述条件之前，只能定义为：

> “已完成核心业务闭环，但尚未达到平台级完备。”

## 十一、建议的实际落地批次

建议按三个批次执行：

### 批次 A：模型与渲染统一

- 扩展 `OrthographyDocType`
- 新增 render policy 解析器
- 接管项目/层/时间轴渲染入口
- 完成迁移与回归

### 批次 B：转换基础设施

- 新增 transform doc
- 实现 table-map + 基础 icu-rule
- 接入预览器与样例测试

### 批次 C：字体与 bidi 收口

- exemplar coverage check
- fallback stack 与风险提示
- RTL 输入/展示/导出统一策略
- 补互操作字段

这样拆分的好处是：

- 批次 A 即可显著提升架构稳定性
- 批次 B 才开始增加“能力厚度”
- 批次 C 负责把用户真正感知到的显示可靠性收尾
