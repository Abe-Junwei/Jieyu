## 二、P0 — 数据正确性

### 2.1 `syncUtteranceTextToSegmentationV2` 非事务多表写入

**文件**：[LayerSegmentationV2BridgeService.ts](src/services/LayerSegmentationV2BridgeService.ts#L72-L125)

**问题**：该函数连续执行三步写入（删除旧 content → 插入 segment → 插入 content），但没有包裹 Dexie `transaction('rw', ...)`。若中途浏览器崩溃或 IndexedDB 错误，可能留下"有 segment 无 content"或"旧 content 已删新 content 未写"的不一致状态。

**影响**：此函数是所有文本保存路径的核心（`saveUtteranceText`、`saveTextTranslationForUtterance`、恢复、导入），每次用户编辑都会触发。

**建议**：将三步操作包裹在 `db.dexie.transaction('rw', db.dexie.layer_segments, db.dexie.layer_segment_contents, ...)` 中。

---

### 2.2 `layerSegmentDocSchema` 缺少 `utteranceId` 字段

**文件**：[db/index.ts](src/db/index.ts#L1088-L1107)

**问题**：TypeScript 接口 `LayerSegmentDocType` 定义了 `utteranceId?: string`，v25 迁移会回填此字段，且 Bridge 服务内部查询 (`where('utteranceId').equals(...)`) 严重依赖它。但 Zod schema `layerSegmentDocSchema` 中没有声明 `utteranceId`。

**当前影响**：由于 `insert()` 方法调用 `validate(doc)` 只做校验不用 parse 返回值，实际数据不会被 strip。但这意味着：
- Zod parse 不会对 `utteranceId` 做类型校验（如传入数字不会报错）
- 未来若有人改用 `schema.parse()` 返回值，`utteranceId` 会被静默丢弃
- 与 schema 定义不一致，阻碍基于 Zod schema 的自动化工具（如 API 文档生成、数据导出校验）

**建议**：在 `layerSegmentDocSchema` 中补 `utteranceId: z.string().min(1).optional()`。

---

### 2.3 `removeUtteranceCascadeFromSegmentationV2` 非事务级联删除

**文件**：[LayerSegmentationV2BridgeService.ts](src/services/LayerSegmentationV2BridgeService.ts#L224-L261)

**问题**：该函数依次删除 content → segment → links，三步独立执行无事务保护。cascade 删除应是原子操作，中途失败会留下孤立 segments 或 links。

**建议**：包裹在单个 Dexie `transaction('rw', ...)` 中。

## 三、P1 — 逻辑缺陷

### 3.1 EAF 导出多 segment 场景丢失精度

**文件**：[EafService.ts](src/services/EafService.ts#L226)

**问题**：当 `independent_boundary` 层对应一个 utterance 有多个 segment 时（`segArr?.length === 1` 条件不成立），代码回退到 utterance 边界，而非逐个 segment 导出。这意味着精心调整的独立边界在导出后全部丢失，回退到粗粒度的 utterance 时间范围。

**建议**：当 `segArr.length > 1` 时，为每个 segment 生成独立的 `ALIGNABLE_ANNOTATION`，而非回退到 utterance 边界。

---

### 3.2 `removeBySelector` 始终全表扫描

**文件**：[db/index.ts](src/db/index.ts#L1479-L1492)

**问题**：`removeBySelector` 使用 `this.table.filter(...)` 做 JS 端逐行匹配，即使 selector 中的字段有索引也不会利用。这在以下高频调用点都存在：
- `utterance_texts.removeBySelector({ layerId })` — 层删除路径
- `layer_links.removeBySelector({ transcriptionLayerKey })` — 层删除路径  
- `layer_links.removeBySelector({ layerId })` — 统一删除路径
- `ai_messages.removeBySelector({ conversationId })` — AI 对话清除

**影响**：功能正确但性能差，尤其在数据量大时层删除操作会明显变慢。

**建议**：`removeBySelector` 内部优先尝试 `where(indexedField).equals(value)` 路径。

---

### 3.3 `LayerTierUnifiedService.updateLayer` 与 `createLayer` 实现完全相同

**文件**：[LayerTierUnifiedService.ts](src/services/LayerTierUnifiedService.ts#L15-L19)

**问题**：`updateLayer` 和 `createLayer` 的实现完全一样（都是 `db.collections.layers.insert(layer)` + `syncLayerToTier`）。`insert` 底层是 `table.put()`（upsert），所以功能上等价。但方法名暗示"更新"通常只改变部分字段，而实际是全文档覆盖。

**影响**：调用方若只想改 `sortOrder` 但传入了过时的其他字段（如 `constraint`），会静默覆盖最新值。

**建议**：`updateLayer` 应改为接受 `Partial<LayerDocType>` 并使用 `db.collections.layers.update(id, changes)`，或至少明确注释这是 upsert 语义。

---

### 3.4 `refreshSegmentUndoSnapshot` 中 `segment_links.toArray()` 全表扫描

**文件**：[TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx#L273)

**问题**：undo 快照刷新时对 `segment_links` 执行 `.toArray()` 全表扫描，再在 JS 端 filter。此函数在每次 split/merge/delete/create 后都会被调用。

**建议**：利用 v23 添加的 `sourceLayerId`/`targetLayerId` 索引做范围查询，或按 `segmentIdSet` 用 `anyOf` 索引查询替代全表扫描。

---

### 3.5 `cleanupOrphanSegments` 中 `segment_links.toArray()` 全表扫描

**文件**：[LayerSegmentationV2BridgeService.ts](src/services/LayerSegmentationV2BridgeService.ts#L169)

**问题**：孤立 segment 清理时对 `segment_links` 做全表扫描，在 JS 端过滤。

**建议**：按 `orphanSegmentIds` 做两次索引查询 (`where('sourceSegmentId').anyOf(...)` + `where('targetSegmentId').anyOf(...)`) 替代全表扫描。

---

### 3.6 `getAllUtteranceTextsPreferV2` 双全表扫描

**文件**：[LayerSegmentationV2BridgeService.ts](src/services/LayerSegmentationV2BridgeService.ts#L351-L352)

**问题**：`layer_segments.toArray()` + `layer_segment_contents.toArray()` 两个全表扫描，用于 snapshot loader 加载时获取全部翻译文本。

**影响**：每次加载快照时全量扫描两个表，数据量大（如 1000+ utterance × 多层）时加载明显变慢。

**建议**：改为按 `textId` 索引查询（已有 `[textId+layerId]` 复合索引）。

---

### 3.7 `restoreLayerSegments`（undo 恢复）中 `segment_links.filter()` 全表扫描

**文件**：[TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx#L317-L322)

**问题**：undo 恢复 segment 时，通过 `.filter(l => segmentIds.has(...))` 对 `segment_links` 做全表扫描来找出需要清理的 stale links。

**建议**：同 3.4，用索引查询替代。

## 四、P2 — 性能隐患

### 4.1 Undo 快照深拷贝所有 utterance/translation/layer/speaker/segment 数据

**文件**：[useTranscriptionUndo.ts](src/hooks/useTranscriptionUndo.ts#L79-L93)

**问题**：每次 `pushUndo` 都对 utterances、translations、layers、layerLinks、speakers、layerSegments、layerSegmentContents、segmentLinks 做浅拷贝。对于大项目（1000+ utterance），每次编辑分配约 8 个大数组副本，50 步 undo 栈会占用大量内存。

**建议**：考虑改为增量快照（只记录 diff）或结构共享（immutable data structure），至少对未改变的集合重用引用。

---

### 4.2 `layer_utterances` 表定义但无消费者

**文件**：[db/index.ts](src/db/index.ts#L2577) — v27 migration

**问题**：v27 添加 `layer_utterances` 表并从 utterances + segments 回填数据，定义了 7 个索引。但全代码库中没有任何读写消费者（仅有 schema 定义和迁移代码）。

**影响**：
- 迁移时浪费时间回填大量数据
- 每次写入 utterance/segment 时 Dexie 需维护 7 个索引但无人使用
- 增加 DB 存储体积

**处理结果（2026-03-28）**：已采纳“正式删除”方案。当前运行时暴露面已移除，并新增 DB v32 对 `layer_utterances` 物理 drop；v27 迁移保留仅用于历史升级链。

---

### 4.3 `getUtteranceTextsByUtterancesPreferV2` N+1 查询

**文件**：[LayerSegmentationV2BridgeService.ts](src/services/LayerSegmentationV2BridgeService.ts#L397-L405)

**问题**：对每个 `utteranceId` 单独执行 `listV2UtteranceTextsByUtterance`（每次内部再做 2 个索引查询 + 1 个 bulkGet），形成 N+1 查询模式。

**建议**：改为批量查询：一次性 `where('utteranceId').anyOf(ids)` 获取所有 segments，再批量获取 contents。

## 五、P3 — 结构债务

### 5.1 Orchestrator 文件过大（4153 行）

**文件**：[TranscriptionPage.Orchestrator.tsx](src/pages/TranscriptionPage.Orchestrator.tsx)

**问题**：虽已拆分为多个子文件（Toolbar/BatchOps/Dialogs/TimelineContent 等），Orchestrator 本身仍有 ~4153 行，包含大量 `useCallback` 定义、effect 逻辑和路由分派。

**影响**：阅读和修改困难，React Fast Refresh 对超大文件的热更新性能也受影响。

**建议**：将路由操作（splitRouted/mergeRouted/deleteRouted）抽为独立 hook（如 `useRoutedSegmentOps`），进一步减小核心文件体积。

---

### 5.2 `delete layer.parentLayerId` 使用 JS `delete` 操作符

**文件**：[LayerConstraintService.ts](src/services/LayerConstraintService.ts#L201-L255)（5 处）

**问题**：约束修复函数中用 `delete layer.parentLayerId` 移除可选字段。在 `exactOptionalPropertyTypes` 下不能赋 `undefined`，但 `delete` 操作符在 TypeScript 中语义不明确：
- 对象字面量交叉类型时可能导致编译器不警告
- `Zod.parse()` 后再 `delete` 可能出现类型不一致

**建议**：统一用条件展开 (`...(parentLayerId ? { parentLayerId } : {})`) 替代 `delete`，或在修复后用 `Omit` 工具类型确保类型安全。

---

### 5.3 `DexieCollectionAdapter.findOne` 首选索引查询但静默降级为全表扫描

**文件**：[db/index.ts](src/db/index.ts#L1430-L1447)

**问题**：`findOne` 尝试用 `where(key).equals(value)` 做索引查询，但 `catch` 块静默降级为 `this.table.filter(...)` 全表扫描。调用方无感知，性能可能在某些字段上意外降低。

**建议**：至少在 `catch` 块中添加 DEV 模式下的 `console.debug` 警告，帮助开发者发现非索引查询。

---

### 5.4 `TierBackedLayerCollectionAdapter.findByIndexAnyOf` 串行循环

**文件**：[db/index.ts](src/db/index.ts#L1645-L1652)

**问题**：对每个 value 串行调用 `findByIndex`，每次内部都做一次 DB 查询。如果传入 10 个 value 就做 10 次 DB round-trip。

**建议**：合并为单次 `where(indexName).anyOf(values)` 查询。

---

### 5.5 `buildSegmentationV2BackfillRows` 中 `v22` ID 前缀已更新但仍与 `v25` 迁移解析逻辑耦合

**文件**：[db/index.ts](src/db/index.ts#L1825) vs [db/index.ts](src/db/index.ts#L2524)

**问题**：v22 迁移生成的 segment ID 前缀为 `segv22_`，v25 迁移的 `utteranceId` 回填逻辑需要知道此前缀来反推 utteranceId。如果未来有 v28+ 迁移生成新前缀（如 `segv28_`），v25 的解析逻辑不会自动兼容。

**建议**：v25 之后的所有新建 segment 都应显式写入 `utteranceId` 字段（已通过 Bridge 服务实现），此耦合仅影响历史数据迁移，但应在迁移注释中明确标记。

---

### 5.6 FlexService 导出不感知 v2 segment 边界

**文件**：[FlexService.ts](src/services/FlexService.ts#L70-L144)

**问题**：`exportToFlextext` 只接受 `utterances`/`translations`/`tokens`/`morphemes`，不接受 `layer_segments`。独立边界层的精确边界在 FLEx 导出中完全丢失（与 EAF 导出的部分支持不同，FLEx 导出完全不支持）。

**影响**：如果用户在独立边界层上标注了不同于 utterance 的时间边界，导出 FLEx 后这些边界信息无法保留。

**建议**：若需要支持，扩展 `FlexExportInput` 接受 `segments` 参数。

---

### 5.7 Snapshot loader 加载后不恢复 utteranceDrafts 与 layerSegments 的一致性

**文件**：[useTranscriptionSnapshotLoader.ts](src/hooks/useTranscriptionSnapshotLoader.ts#L149-L165)

**问题**：`loadSnapshot` 合成 `utteranceDrafts` 时，每个 utterance 查找 `defaultTrcLayer` 的翻译文本。但如果 `defaultTrcLayer` 是 `independent_boundary` 类型，翻译文本可能来自 v2 segment contents 而非直接的 `utterance_texts`，而 `getAllUtteranceTextsPreferV2` 返回的数据已经是 v2 优先的，所以内容正确。但 `utteranceDrafts` 在后续编辑中与 DB 的同步路径（通过 `useTranscriptionTranslationDraftSync`）可能因 v1/v2 路径差异而产生不一致。

**影响**：目前两条路径的最终效果一致（都指向同一文本），但增加了理解和维护的心智负担。
