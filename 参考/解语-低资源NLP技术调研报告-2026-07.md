## 1. 语音识别（ASR）

### 1.1 技术现状全景

| 模型 | 机构 | 年份 | 语言覆盖 | 参数量 | 关键特性 | 许可证 |
|------|------|------|---------|--------|---------|--------|
| **Whisper** | OpenAI | 2022 | 99 种 | 39M–1550M (tiny–large) | 68 万小时弱监督训练；零样本迁移近人类精度 | Apache-2.0 |
| **Whisper large-v3** | OpenAI | 2023 | 99 种 | 1550M | 在 v2 基础上用 100 万小时弱标签 + 400 万小时伪标签训练 | Apache-2.0 |
| **MMS (Massively Multilingual Speech)** | Meta | 2023 | **1,162 种 ASR / 4,017 种语言识别** | 1B (基座) + ~2.5M (每语言适配器) | 基于 wav2vec 2.0 预训练；适配器架构切换语言；在 FLEURS 54 种语言上 WER 减半（相比 Whisper） | CC-BY-NC-4.0 |
| **wav2vec 2.0** | Meta | 2020 | 依赖微调数据 | 300M–1B | 自监督预训练；仅 10 分钟标注数据 + 53k 小时无标签 → WER 4.8/8.2 (LibriSpeech)；MMS 的技术基石 | MIT |
| **Moonshine** | Useful Sensors | 2024 | 英语为主 | 未公开（轻量） | 专为实时转写和语音命令设计；边缘设备友好 | - |
| **LiteWhisper** | U. Washington + Kotoba | 2025 | 同 Whisper | 低秩近似压缩 | 保持精度同时大幅降低实际推理成本 | - |

### 1.2 解语最佳候选

#### 首选：MMS (facebook/mms-1b-all)

**核心优势：**
- **语言覆盖无与伦比**：1,162 种语言的 ASR 适配器，覆盖绝大多数濒危语言
- **极少数据即可微调**：每个适配器仅 ~2.5M 参数（占基座 0.25%），在 Common Voice ~4 小时数据上微调；100 步训练即可获得可用 WER（土耳其语 ~22%）
- **语言热切换**：`load_adapter("tur")` 即时切换语言，无需重新加载完整模型
- **CTC 架构**：适合流式场景，输出对齐较好

**实际数据要求：**
- 零样本：直接用预训练适配器（若目标语言在 1,162 种之列）
- 微调：推荐 1–4 小时标注语音数据；低至 10 分钟也有成效（参考 wav2vec 2.0 论文）
- 微调耗时 < 30 分钟（单 GPU）

#### 备选：Whisper large-v3

**核心优势：**
- 零样本精度最高（高资源语言）
- 生态最完善：768 个微调模型 + 187 个适配器模型（HuggingFace Hub）
- 支持 word-level timestamps (`return_timestamps="word"`)
- Apache-2.0 开源许可

**劣势（相对 MMS）：**
- 仅覆盖 99 种语言 — 对大量濒危语言无法零样本使用
- 模型体积大 (1550M)，浏览器端部署困难
- 在低资源语言上 WER 不如 MMS

### 1.3 浏览器可行性评估

| 方案 | 模型 | 大小 (q8) | 是否可行 | 备注 |
|------|------|-----------|---------|------|
| Transformers.js + WASM | whisper-tiny/base | ~50–100MB | ✅ 可行 | 精度有限但可用于预转写 |
| Transformers.js + WebGPU | whisper-small | ~300MB | ⚠️ 有条件可行 | 需 WebGPU 支持的浏览器 |
| Transformers.js + WASM | MMS adapter | ~1B → 量化后仍很大 | ❌ 目前不可行 | 基座太大；需等待蒸馏版本 |
| 服务端 API 调用 | 任意 | - | ✅ 可行 | 通过 F9 AI Provider 抽象层 |

**推荐策略：混合架构**
1. **浏览器端**：用 Transformers.js 加载 Whisper tiny/base 的 ONNX 量化版本，提供快速预转写
2. **服务端**：通过 F9 AI Provider 调用 MMS 或 Whisper large-v3，提供高精度转写
3. 用户可在「本地轻量模型」与「云端精准模型」之间灵活选择

## 2. 强制对齐（Forced Alignment）

### 2.1 技术现状

| 工具 | 机构 | 技术路线 | 语言支持 | 关键特性 |
|------|------|---------|---------|---------|
| **Montreal Forced Aligner (MFA)** | McGill U. / Montreal Corpus Tools | Kaldi 后端 + 声学模型 | 多语言（需词典 + 声学模型） | 学术标准工具；支持自定义词典和模型训练 |
| **WhisperX** | Oxford VGG | Whisper + wav2vec2 音素对齐 + VAD + pyannote 说话人分离 | 跟随 Whisper 99 种 | 70 倍实时速度（batched）；词级时间戳；说话人标签；BSD-2 许可 |
| **WebMAUS** | BAS Munich | 基于 MAUS 算法的 Web 服务 | ~40 种语言 | HTTP API 可远程调用；语言文档领域成熟 |

### 2.2 WhisperX 详细评估

**流水线架构：**
1. **VAD 预处理**（Silero-VAD 或 pyannote-VAD）→ 去除静音，减少幻觉
2. **Batched Whisper 转写** → 70 倍实时速度（faster-whisper CTranslate2 后端）
3. **wav2vec2 音素对齐** → 精确到词级的时间戳
4. **说话人分离**（pyannote speaker-diarization-community-1）→ 说话人标签

**语言支持详情：**
- 内置对齐模型：`{en, fr, de, es, it}` 通过 torchaudio pipelines
- 更多语言通过 HuggingFace wav2vec2 音素模型支持
- 社区贡献持续扩展中（项目目前 20.7k stars，108 贡献者）

**局限性：**
- 不含字母表字符的词（如 "2014."、"£13.60"）无法对齐
- 重叠语音处理欠佳
- 需要对应语言的 wav2vec2 对齐模型

### 2.3 解语集成建议（F10 功能）

**推荐方案：WhisperX (服务端) + WebMAUS (备选)**

| 对齐层级 | 推荐方案 | 实现路径 |
|---------|---------|---------|
| 句子/段级 | Whisper 自带时间戳 | 浏览器端 Transformers.js 即可 |
| 词级 | WhisperX 流水线 | 服务端 API（通过 F9） |
| 音素级 | WebMAUS | HTTP API 调用 |
| 自训练 | MFA | 用户提供词典后服务端运行 |

**最小数据要求：**
- 零样本对齐：仅需音频 + 对应文本（无需额外训练数据）
- 自定义语言对齐：需要该语言的 wav2vec2 音素模型（或 MFA 声学模型 + 发音词典）

## 3. 机器翻译（Machine Translation）

### 3.1 技术现状

| 模型 | 机构 | 年份 | 语言对数 | 参数量 | 关键特性 |
|------|------|------|---------|--------|---------|
| **NLLB (No Language Left Behind)** | Meta | 2022 | **200+ 种语言** | 600M–54B (Sparsely Gated MoE) | FLORES-200 基准；BLEU 提升 44%（vs 之前 SOTA）；开源 |
| **SeamlessM4T** | Meta | 2023 | **100 种语言** | 多模态统一模型 | **单一模型**实现语音↔语音、语音↔文本、文本↔文本翻译 + ASR；100 万小时开放语音数据训练；BLEU 提升 20%；开源 |
| **M2M-100** | Meta | 2020 | 100 种语言 | 418M–12B | 非英语中心直接翻译；Transformers.js 已支持 |
| **MarianMT** | Microsoft Translator | - | 多语言对 | 多种大小 | OPUS 数据训练；HuggingFace/Transformers.js 已支持 |

### 3.2 解语最佳候选

#### 首选：NLLB-200 (distilled-600M)

**理由：**
- 200+ 种语言是翻译领域最全面的开源覆盖
- 600M 蒸馏版在精度和大小之间取得最佳平衡
- 重点投入了低资源语言质量（"No Language Left Behind" 理念）
- Transformers.js 已支持 NLLB 架构（通过 M2M-100 兼容层）

#### 兼具 ASR 的备选：SeamlessM4T

**理由：**
- 统一模型同时处理语音和文本翻译，减少系统复杂度
- 语音→文本翻译对田野录音场景极具价值（直接从录音生成目标语言译文）

### 3.3 浏览器可行性

| 方案 | 模型 | 量化后大小 | 可行性 |
|------|------|-----------|--------|
| Transformers.js | nllb-200-distilled-600M (q4) | ~200–300MB | ⚠️ 有条件可行 |
| Transformers.js | M2M-100 (418M, q8) | ~200MB | ⚠️ 有条件可行 |
| Transformers.js | MarianMT (小模型) | ~50–150MB | ✅ 可行 |
| 服务端 API | NLLB / SeamlessM4T | - | ✅ 推荐 |

**推荐策略：**
- **高频语言对**（如中英、英法）：可考虑浏览器端加载 MarianMT 小模型
- **低资源语言翻译**：通过 F9 AI Provider 调用服务端 NLLB-200 或 SeamlessM4T
- 解语 F12（自动语法标注）可利用 NLLB 提供参考译文辅助

## 4. 文本/词元嵌入（Text & Token Embeddings）

### 4.1 技术现状

| 模型 | 机构 | 年份 | 语言覆盖 | 参数量 | 关键特性 |
|------|------|------|---------|--------|---------|
| **Glot500-m** | CIS @ LMU Munich | 2023 (ACL) | **511 种语言** | XLM-R 基础（560M） | 在 XLM-R 上对 511 种低资源语言做持续预训练；5 种任务上均优于 XLM-R 基线；开源（代码 + 数据 + 模型） |
| **ByT5** | Google | 2021 (TACL 2022) | **任意语言** | 与 T5 同架构多种大小 | **无需分词器**：直接处理 UTF-8 字节序列；对任意书写系统天然适配；对噪声/拼写错误更鲁棒 |
| **multilingual-e5-small** | Microsoft | 2023 | 100+ 种语言 | ~118M | 当前解语计划中 F28 的选型；MTEB 排行良好 |
| **XLM-R** | Meta | 2019 | 100 种语言 | 125M–560M | 业界标准多语言编码器基线 |

### 4.2 解语最佳候选

#### 语义搜索 / F19 / F28：multilingual-e5-small（维持现有选型）

- 118M 参数量化后适合浏览器端
- 100+ 语言覆盖日常需求
- Transformers.js 已完整支持 feature-extraction pipeline

#### 低资源语言覆盖扩展：Glot500-m

- 当 multilingual-e5-small 不覆盖目标语言时，Glot500-m 覆盖 511 种语言
- 可作为**分类、NER、检索**等下游任务的编码器
- 注意：Glot500-m 基于 XLM-R，体积较大（~560M），不宜浏览器直接部署

#### 未知/极端低资源语言：ByT5

- **杀手级特性：无需分词器**
- 对于没有任何 NLP 资源的语言（连分词器都没有），ByT5 直接处理原始字节
- 特别适合：
  - 非拉丁文字系统
  - 存在大量拼写/正字法变体的田野文本
  - 语言识别、相似度判断等基础任务

### 4.3 浏览器可行性

| 模型 | 量化后大小 | Transformers.js 支持 | 可行性 |
|------|-----------|---------------------|--------|
| multilingual-e5-small (q8) | ~60MB | ✅ 已支持 | ✅ 推荐 |
| Glot500-m | ~300MB+ | 需转 ONNX | ⚠️ 过大 |
| ByT5-small | ~150MB | ✅ 已支持 T5 架构 | ⚠️ 有条件可行 |

## 5. 少样本学习与 LLM 方案

### 5.1 技术现状

| 模型/方法 | 机构 | 关键特性 | 与低资源语言的关系 |
|----------|------|---------|------------------|
| **Llama 2/3/4** | Meta | 7B–405B 参数；开源；对话微调版本 | 预训练数据以高资源语言为主；低资源语言能力有限但可通过少样本提示/微调改善 |
| **Qwen3 / 3.5** | 阿里 | 多种大小；思考模式 | 中文+英文为主；Transformers.js 已支持 |
| **Gemma 3n** | Google | 专为移动端/边缘设备设计 | 多语言能力有待评估 |
| **AfroBench** | 多机构 | 64 种非洲语言, 15 种任务基准 | 揭示 LLM 在非洲语言上的巨大差距（ACL 2025） |
| **Prompt-based 少样本** | 通用方法 | 利用 LLM in-context learning | 对低资源语言有效但严重依赖语言在预训练数据中的覆盖度 |

### 5.2 AfroBench 的重要发现（ACL 2025）

AfroBench 是迄今最全面的非洲语言 LLM 评测，覆盖 64 种语言、15 种任务、22 个数据集。其核心发现对解语极具参考价值：

- **高资源语言与非洲语言之间的 LLM 性能差距巨大** — 即使是最强的闭源模型
- 这说明：**对于濒危语言，不能简单依赖通用 LLM 的零样本能力**
- 需要专门的微调/适配策略

### 5.3 解语 LLM 集成建议

**推荐策略：F9 AI Provider 多后端抽象**

```
用户请求 → F9 AI Provider 路由
  ├→ 浏览器端小模型（简单任务：分词猜测、拼写建议）
  ├→ 本地 LLM（通过 Ollama / llama.cpp 等本地服务）
  └→ 云端 API（OpenAI / Claude / 专业模型）
```

**LLM 适用场景（解语功能映射）：**

| 解语功能 | LLM 用途 | 推荐方案 |
|---------|---------|---------|
| F12 自动语法标注 | 少样本莱比锡标注 | 云端 LLM + 用户提供的几个标注范例 |
| F19 搜索/索引 | 语义搜索辅助 | 浏览器端嵌入模型（第 4 节） |
| 新功能：文化注释 | 解释田野录音文化背景 | 云端 LLM |
| 新功能：分词建议 | 对无标准正字法语言的分词建议 | ByT5 或云端 LLM |

**浏览器端 LLM 可行性：**

Transformers.js 已支持在浏览器运行文本生成模型（包括 Phi-3、Gemma、Qwen 等）。但：
- 即使量化到 q4，7B 模型仍需 ~2–4GB 下载，加载慢
- 对于解语的典型用户场景（田野工作环境、网络条件差），不推荐在浏览器端跑大型 LLM
- **建议**：小型事务（如分词、拼写校正）用浏览器端轻量模型；复杂分析任务委托给服务端

## 6. 数据采集与众包平台

### 6.1 技术现状

| 平台/项目 | 机构 | 关键特性 | 与解语的关系 |
|----------|------|---------|------------|
| **Common Voice** | Mozilla | 开源众包语音数据集；志愿者贡献录音+验证；每 3 个月发布数据；社区驱动的语言添加流程 | 录音数据可直接用于微调 MMS/Whisper |
| **Masakhane** | 社区驱动 | 非洲大陆范围的 NLP 社区研究（2000+ 种语言）；开源；ICLR 2020 AfricaNLP Workshop | 社区协作模式的典范；可参考其组织模式 |
| **CLARIN** | 欧洲 | 欧洲语言资源数字基础设施 | 存档格式标准和元数据规范的参考 |
| **Elpis** | multiple | 面向语言文档的自动 ASR 系统构建工具 | 理念相近但技术路线不同（解语更侧重浏览器端） |

### 6.2 Common Voice 对解语的直接价值

**语言众包流程（可参考）：**
1. 社区请求添加语言
2. 网站本地化翻译
3. 句子收集（供朗读）
4. 启动该语言页面
5. 志愿者贡献录音
6. 其他志愿者验证
7. 每季度发布数据集

**解语 F22（社区界面）可借鉴的模式：**
- 让语言社区成员直接贡献录音、标注和验证
- 引入"语言变体"（Variants）概念处理方言差异
- 关注"互通性"（mutual intelligibility）作为语言划分标准
- 数据开源发布建立社区信任

### 6.3 Masakhane 社区模式的启示

Masakhane 展示了一种成功的社区驱动低资源语言 NLP 研究模式：
- 语言社区成员即研究者
- 降低技术门槛，让非 NLP 专家也能贡献
- 集合大洲范围的碎片化语言数据

**对解语的启示：** 解语可以在工具层面承担类似 Masakhane 的角色 — 为语言社区提供零门槛的文档工具，使得采集到的数据能直接进入 NLP 训练流水线。

## 7. 浏览器端推理基础设施

### 7.1 Transformers.js — 核心桥梁技术

**项目状态（2026 年 7 月）：**
- 版本：v4.0.0-next.7（v4 大版本迭代中）
- GitHub Stars：15.5k | 被 6,200+ 项目依赖
- 许可证：Apache-2.0
- 维护：HuggingFace 官方

**核心架构：**
```
PyTorch/TF/JAX 模型
     ↓ (HuggingFace Optimum 转换)
ONNX 格式模型
     ↓
ONNX Runtime (WASM / WebGPU)
     ↓
浏览器内推理
```

**已支持与解语相关的任务：**

| 任务 | Pipeline 名称 | 解语场景 |
|------|-------------|---------|
| ✅ 自动语音识别 | `automatic-speech-recognition` | 预转写 |
| ✅ 特征提取 | `feature-extraction` | 句子嵌入 / 语义搜索 |
| ✅ 翻译 | `translation` | 参考翻译 |
| ✅ 文本分类 | `text-classification` | 语言识别 |
| ✅ Token 分类 | `token-classification` / `ner` | 词性标注 / NER |
| ✅ 文本生成 | `text-generation` | 标注建议 |
| ✅ 语音分类 | `audio-classification` | 语种识别 |
| ✅ TTS | `text-to-speech` | 发音范例 |

**已支持的关键模型架构：**
- ✅ Whisper (及 LiteWhisper、Moonshine)
- ✅ Wav2Vec2 (MMS 底层架构)
- ✅ MMS
- ✅ NLLB / M2M-100 / MarianMT
- ✅ T5 / mT5 / ByT5
- ✅ BERT / XLM-R / DistilBERT / ModernBERT
- ✅ Qwen / Gemma / Phi / Llama 架构
- ✅ SpeechT5 (TTS)
- ✅ PyAnnote (说话人分离)

**量化支持：**
- `fp32`（WebGPU 默认）
- `fp16`
- `q8`（WASM 默认）→ 大小约原始 1/4
- `q4` → 大小约原始 1/8，精度有所下降

**WebGPU 加速：**
```javascript
const pipe = await pipeline('automatic-speech-recognition', 'model-id', {
  device: 'webgpu',  // GPU 加速
  dtype: 'q8',       // 量化
});
```
> ⚠️ WebGPU 仍处于实验阶段，部分浏览器兼容性有限

### 7.2 解语技术栈适配性

## 8. 解语集成建议总结

### 8.1 按功能模块的技术选型

| 解语功能 | 推荐技术 | 部署位置 | 最小数据要求 | 优先级 |
|---------|---------|---------|------------|--------|
| **F9 AI Provider** | 统一抽象层 | 框架层 | - | 🔴 高 |
| **F10 强制对齐** | WhisperX (服务端) + Whisper timestamps (浏览器端) | 混合 | 仅需音频+文本 | 🔴 高 |
| **ASR 预转写** | Whisper-tiny ONNX (浏览器) + MMS/Whisper-large (服务端) | 混合 | 零样本或 1–4h 标注音频 | 🔴 高 |
| **F12 自动语法标注** | LLM 少样本 (服务端) | 服务端 | 用户提供 5–10 个标注范例 | 🟡 中 |
| **F19 语义搜索** | multilingual-e5-small ONNX | 浏览器 | 无 | 🟡 中 |
| **F28 词嵌入** | multilingual-e5-small (常规) / Glot500-m (低资源) | 混合 | 无 | 🟡 中 |
| **翻译辅助** | MarianMT (浏览器, 高频对) / NLLB-200 (服务端) | 混合 | 无 | 🟢 低 |
| **F22 社区界面** | 参考 Common Voice + Masakhane 模式 | 前端 | 需社区参与 | 🟢 低 |
| **语种识别** | MMS LID (4,017 种) | 服务端 | 无 | 🟢 低 |

### 8.3 渐进式实施路径

**第一阶段（核心基础）：**
1. 实现 F9 AI Provider 抽象层，统一管理本地/远程推理
2. 集成 Transformers.js，在 Web Worker 中加载 Whisper-tiny ONNX
3. 实现基础 ASR 预转写功能（浏览器端零依赖）

**第二阶段（精准能力）：**
4. 通过 F9 对接 MMS / Whisper-large 服务端 ASR
5. 集成 WhisperX 服务端实现词级强制对齐
6. 部署 multilingual-e5-small 实现语义搜索

**第三阶段（高级功能）：**
7. 接入 NLLB-200 提供翻译辅助
8. 利用 LLM 少样本能力实现 F12 自动语法标注
9. 探索 Glot500-m / ByT5 对极端低资源语言的覆盖
