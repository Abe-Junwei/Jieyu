# 解语详细完整实施计划（2026-03-16）

## 0. 计划定位
本计划是对“文献增量需求修订”的执行化版本，目标是把研究结论转成可交付、可验收、可追踪的研发路径。

适用范围：
- 产品：解语（Jieyu）
- 周期：6个月（24周）
- 交付方式：双周迭代（2周/迭代）
- 战略原则：本地优先、标准优先、协作优先、AI稳健增强

---

## 1. 北极星目标与成功标准

### 1.1 北极星目标
在6个月内，交付一个可用于濒危语言田野数据处理的“标准互操作 + 协作编辑 + 人机协同转写”最小可用平台。

### 1.2 成功标准（最终验收）
1. 互操作：EAF/TextGrid往返转换结构保真 >= 99%。
2. 协作：离线并发编辑后自动合并，不发生数据结构损坏（损坏率=0）。
3. AI实用性：在目标语料上，人工修订总时长较纯手工下降 >= 30%。
4. 可追溯：自动生成结果100%可追溯到模型、参数、输入版本、输出版本。
5. 稳定性：关键路径操作（导入、编辑、导出）无P1缺陷。

## 2. 总体阶段规划（24周）

### 阶段A（W1-W4）：标准化数据底座
目标：建立内部CAM（Canonical Annotation Model），确保互操作可控。

交付：
1. CAM v1 schema（tier、时间锚点、父子关系、链接、扩展字段）。
2. EAF -> CAM、TextGrid -> CAM 解析器v1。
3. CAM -> EAF、CAM -> TextGrid 导出器v1。
4. 互操作golden files测试集v1。

验收：
- 完成20个样本文件往返测试，关键字段保真 >= 99%。

### 阶段B（W5-W8）：质量保障与规则辅助
目标：把“能跑”升级为“可持续维护”。

交付：
1. 互操作CI回归管线（字段差异报告、结构差异报告）。
2. Leipzig规则提示引擎v1（非阻断提示）。
3. 数据修复建议器v1（基础规则）。

验收：
- CI对标准样本全量通过。
- Gloss编辑场景可给出准确提示并支持忽略。

### 阶段C（W9-W14）：AI作业与Provider抽象
目标：建立“可替换、可审计”的自动化能力。

交付：
1. AI任务元数据schema（任务、模型、参数、版本、置信度、审计链）。
2. Provider接口层（local/mock/remote）。
3. WebMAUS/BAS最小流程接入（导入音频+文本，返回分层标注）。
4. 失败恢复与重试机制v1。

验收：
- 可切换mock与remote，不改业务层。
- 任意自动任务可追溯与回滚。

### 阶段D（W15-W20）：协作内核PoC到可用
目标：从单人编辑升级到多人并发编辑。

交付：
1. Yjs接入PoC（utterance/token对象级）。
2. 冲突策略v1（字段级冲突规则+手动仲裁入口）。
3. 离线编辑合并测试套件。

验收：
- 典型并发冲突可自动合并。
- 离线再联机合并后结构不损坏。

### 阶段E（W21-W24）：性能优化与发布准备
目标：达到可试点交付标准。

交付：
1. 大文件与长音频性能优化。
2. 关键流程可观测性（任务耗时、失败率、导入导出成功率）。
3. 试点发布包与用户操作手册。

验收：
- 达到试点门槛，完成一轮真实用户试用反馈闭环。

## 3. 双周迭代详细计划（12个迭代）

### Iteration 1（W1-W2）
目标：CAM草案与字段清单冻结。
任务：
1. 定义CAM核心实体与必填字段。
2. 制定EAF/TextGrid映射矩阵（字段级）。
3. 建立测试样本目录结构。
产出：CAM v0.1、映射矩阵v1、样本清单。
验收：评审通过并冻结核心字段。

### Iteration 2（W3-W4）
目标：双向转换v1与基础测试。
任务：
1. 完成EAF/TextGrid到CAM解析。
2. 完成CAM到EAF/TextGrid导出。
3. 建立往返差异检测脚本。
产出：转换链路v1、差异报告v1。
验收：20个样本通过率 >= 90%。

### Iteration 3（W5-W6）
目标：回归体系上线。
任务：
1. 把往返测试接入CI。
2. 增加结构化错误分类（解析失败/映射丢失/时间冲突）。
3. 构建golden files v1。
产出：CI互操作流水线。
验收：每次PR自动输出差异报告。

### Iteration 4（W7-W8）
目标：Leipzig规则提示v1。
任务：
1. 实现缩写合法性与格式提示。
2. 实现词素对齐基础校验。
3. UI中接入非阻断提示。
产出：规则提示模块v1。
验收：典型样本提示准确率达标（由人工评估）。

### Iteration 5（W9-W10）
目标：AI任务审计模型落地。
任务：
1. 定义AI任务元数据结构。
2. 统一任务状态机（pending/running/success/failed）。
3. 记录输入输出版本链。
产出：任务审计基础设施v1。
验收：随机抽查任务可追溯到完整参数。

### Iteration 6（W11-W12）
目标：Provider接口抽象。
任务：
1. 定义provider统一接口。
2. 实现mock provider。
3. 实现local provider占位与调用流程。
产出：provider框架v1。
验收：业务层无provider分支判断。

### Iteration 7（W13-W14）

2. 增加超时、重试、结果拉取。
3. 增加授权/配额提示。
产出：remote provider v1。
验收：端到端任务可完成并回填。

### Iteration 8（W15-W16）
目标：Yjs对象级PoC。
任务：

2. token协同编辑。
3. 冲突回放测试。

验收：核心冲突场景可复现实验并通过。

### Iteration 9（W17-W18）

### Iteration 10（W19-W20）

## 6. 质量门禁（每迭代必过）

1. 单测/集成测试：无新增阻断错误。
2. 互操作回归：golden files全通过。
3. 数据安全：导入导出不丢关键字段。
4. 审计可追溯：AI任务记录完整。
5. 文档更新：变更说明、已知限制、回滚说明同步。

## 7. 风险清单与应对

1. 风险：外部服务不稳定或授权受限。
- 影响：自动任务不可用。
- 应对：provider可替换，本地fallback，配额预警。

2. 风险：标准映射复杂导致迭代延期。
- 影响：互操作质量不达标。
- 应对：先覆盖高频字段，长尾字段延期并标注。

3. 风险：协作冲突导致数据异常。
- 影响：用户信任下降。
- 应对：先对象级PoC，逐步扩大范围，加入仲裁入口。

4. 风险：AI效果不稳定。
- 影响：修订效率达不到目标。
- 应对：置信度门控 + 人在回路 + 语料分层评估。
