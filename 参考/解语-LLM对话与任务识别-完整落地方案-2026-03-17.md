# 解语 LLM 对话与任务识别完整落地方案

版本：2026-03-17
状态：可直接进入实施
适用范围：当前前端本地优先架构（Dexie + React + TypeScript）

---

## 1. 目标与边界

### 1.1 总目标

在现有解语数据底座上，新增三项能力：

1. LLM API 接入（本地 mock + 远程 provider）
2. 对话式交互（AI Sidekick 聊天面板）
3. 自动任务识别与实时建议（ProjectObserver）

### 1.2 本阶段边界

本方案优先做单机本地可运行闭环，不引入后端数据库；仅通过可替换 provider 对接外部模型。

## 3. 总体架构

### 3.1 逻辑分层

1. Data Layer
- ai_conversations
- ai_messages
- ai_recommendations（可选）
- ai_tasks（复用）
- embeddings（复用）

2. Domain Layer
- LLMProvider 接口
- ChatOrchestrator（组装上下文、路由 provider、流式输出）
- ProjectObserver（识别当前阶段并生成建议）
- RecommendationEngine（建议去重与优先级）

3. UI Layer
- AI Sidekick 面板（消息区 + 输入区 + 建议区）
- Prompt 模板选择器（MVP 可简化为固定模板）

### 3.2 数据流

1. 用户发问 -> ChatOrchestrator
2. Orchestrator 读取上下文（当前语段、覆盖率、最近编辑）
3. 调用 Provider（mock/remote）
4. 流式消息写入 ai_messages
5. Observer 周期计算项目状态并生成建议
6. 建议在面板实时展示，可一键执行跳转动作

## 5. 接口契约设计

### 5.1 Provider 接口

位置建议：src/ai/providers/LLMProvider.ts

```ts
export type ChatMessage = {
  role: 'system' | 'user' | 'assistant';
  content: string;
};

export type ChatChunk = {
  delta: string;
  done?: boolean;
  error?: string;
};

export interface LLMProvider {
  id: string;
  label: string;
  supportsStreaming: boolean;
  chat(messages: ChatMessage[], options?: {
    model?: string;
    temperature?: number;
    maxTokens?: number;
  }): AsyncGenerator<ChatChunk, void, unknown>;
}
```

### 5.2 ChatOrchestrator

位置建议：src/ai/ChatOrchestrator.ts

职责：

1. 读取对话历史
2. 拼装 system prompt + 任务上下文
3. 调用 provider 并流式落库
4. 失败重试与错误状态落库

关键方法：

- sendMessage(conversationId, userText, context)
- resumeStreaming(messageId)
- stopStreaming(messageId)

### 5.3 ProjectObserver

位置建议：src/ai/ProjectObserver.ts

输入信号：

- utterance 总数
- transcribed/translated/glossed/verified 覆盖率
- 最近编辑事件（token POS 更新、gloss 更新、备注新增）

输出：

- 阶段：collecting | transcribing | glossing | reviewing
- 建议列表（优先级排序）

## 6. 任务识别规则（MVP）

### 6.1 阶段判定

1. transcribedRate < 0.2 -> collecting
2. transcribedRate >= 0.2 且 glossedRate < 0.2 -> transcribing
3. glossedRate >= 0.2 且 verifiedRate < 0.5 -> glossing
4. verifiedRate >= 0.5 -> reviewing

### 6.2 建议规则

1. collecting
- 建议优先做分段与说话人补全

2. transcribing
- 建议补齐当前媒体未转写片段

3. glossing
- 建议优先标注高频未标注 token 的 POS/gloss

4. reviewing
- 建议处理低置信度和冲突项

1. API Key 不写入数据库明文
2. 远程 provider 参数使用环境变量或用户本地安全输入

## 12. 风险与对策

1. 风险：远程 API 不稳定
- 对策：provider 熔断 + mock 回退

2. 风险：上下文过大导致成本高
- 对策：窗口裁剪 + 摘要缓存

3. 风险：建议噪声过多
- 对策：优先级阈值 + 用户可关闭类别

4. 风险：UI 复杂度上升
- 对策：先做单面板 MVP，避免多入口分裂

---

## 13. 结论

当前解语的数据结构已经具备承载 LLM 接入与任务识别的基础条件。

要实现“对话 + 自动识别任务 + 实时建议”，关键不在扩表，而在新增四个中间层能力：

1. Provider 接口
2. Chat 编排器
3. Observer 规则引擎
4. 面板交互闭环

按本方案分 5 个里程碑实施，可在约 6 到 7 个开发日内交付 MVP。
