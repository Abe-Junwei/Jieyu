**两大风险点**：
1. **F23/F25 同步与协作**：从纯前端升级到前后端架构，是最大的架构决策点。
2. **F10 MFA Python 依赖**：MFA 是 forced alignment 最优方案（Chodroff et al. 2025），但需 Python/Kaldi 环境，纯浏览器无法运行——需本地伴侣进程或远程服务（影响 Decision Point 1）。BAS WebMAUS 作为纯远程备选仍有 CORS/配额风险。WhisperX 为服务端快速预对齐提供第三通道。
3. **社区用户可能偏好手工转写**（Prudhommeaux 2021）：ASR 辅助功能投资回报可能低于预期，需支持双轨工作流。

## 六、战略决策点

在开发过程中需要做出的关键架构决策：

### 决策点1：AI 执行环境 + 桌面分发（W12前决定，⚡ 与 DP4 合并）
- **选项A**：纯浏览器（Web Worker + WASM 模型）——零部署成本，但**无法运行 MFA/Kaldi**（forced alignment 最优方案），能力有限
- **选项B**：本地 Python 伴侣进程——可运行 MFA（Chodroff et al. 2025 实证最优）+ 完整模型支持，但增加安装复杂度
- **选项C**：远程服务（BAS WebMAUS + 自建 MFA server）——最强能力，但需要网络和运维
- **建议**：**先 A+B 混合**（浏览器端 VAD/embedding + 本地 MFA 伴侣进程处理 forced alignment），BAS/WebMAUS 作为无 Python 环境时的远程回退。Chodroff et al. 2025 表明 MFA 在低资源对齐中显著优于其他方案，本地伴侣进程是性价比最高路线
- **选项D（推荐升级）**：浏览器 + 本地伴侣 + 远程，三层渐进——基于在线调研证实 MMS adapter 和 Transformers.js v4 浏览器推理均已成熟，建议升级为此方案：浏览器端 Transformers.js v4 加载 Whisper-tiny ONNX (~50MB q8) 做预转写 + multilingual-e5-small 做语义搜索；本地伴侣进程 MFA forced alignment + 可选 MMS adapter 微调（每语言仅 ~2.5M 参数，微调 <30min）；远程服务 MMS-1b-all (精准ASR) + NLLB-200 (翻译) + WhisperX (快速对齐+说话人分离) + LLM API (F12标注辅助)
- **🆕 桌面分发集成（原 DP4 合并）**：选项 D 天然包含 Tauri 2 sidecar——W12 同步启动 Tauri 2 shell + `vite-plugin-pwa` 双通道。Tauri 负责 Python MFA sidecar 生命周期管理 + 文件系统深度访问；PWA 保留给轻量级/无 AI 场景。先 macOS 单平台 PoC 验证 sidecar 打包，再跨平台扩展（详见 DP4）

### 决策点2：后端引入时机（W37前决定）
- **选项A**：Supabase / Firebase（BaaS）——最快上线，但受限于平台
- **选项B**：CouchDB + PouchDB——与现有 IndexedDB 架构最匹配，社区验证过（Gessler 2019）
- **选项C**：自建 Node/Python 后端——最灵活，但开发运维成本最高
- **建议**：如果人力不变（单人+AI），选 B（CouchDB）最稳妥

### 决策点3：移动端 App 技术路线与接口策略（W37前决定）
- **选项A**：React Native——可复用部分 TypeScript/React 代码和类型定义，生态丰富
- **选项B**：Flutter——跨平台性能更好，但技术栈切换成本高（Dart）
- **选项C**：PWA——零安装、共享主站代码库，但离线能力和原生 API 访问受限
- **接口策略**：无论 App 技术选型如何，主站侧需在 F23（同步）引入后端时，同步设计移动端数据交换接口。核心接口：①项目/session 元数据 CRUD ②音频+粗标注批量上传 ③词表/标注模板下载 ④增量同步+冲突合并
- **建议**：如果已选 Supabase/CouchDB 后端（决策点2），移动端可直接复用同一后端；App 选型优先 React Native（复用 TypeScript 类型）或 PWA（最低成本验证）

### 决策点4：桌面分发策略（⚡ 已合并至 DP1，W12 同步启动）
- **选项A**：PWA（零成本）——加 manifest + service worker，用户从浏览器"添加到桌面"。离线基础功能可用，但无文件系统深度访问、无法捆绑 Python 伴侣进程
- **选项B**：Tauri 2（推荐）——Rust 壳 + WebView，安装包 ~10-15MB。可通过 sidecar 机制管理 Python MFA 伴侣进程生命周期，与决策点1选项B天然契合。用户获得 .dmg (macOS) / .msi (Windows) / .deb (Linux) 原生安装体验
- **选项C**：Electron——完整 Node.js 运行时，安装包 ~100-150MB，内存占用高。能力等同 Tauri 但代价大
- **建议（v2.0 更新）**：**不再"PWA 先行、Tauri 延后"，而是 W12 同步启动双通道**。理由：MFA（F10 核心对齐引擎）必须依赖 Python 环境，纯 PWA 在 Phase 3 只能依赖 WebMAUS 远程备选（CORS/配额/网络限制大），这使得 AI 闭环里程碑（M3）的完整性受制于外部服务。将 Tauri 2 shell 集成提前到 W12（AI 基础设施阶段起步时），与 `vite-plugin-pwa` 并行：**Tauri 通道**提供 sidecar 管理（Python MFA 进程生命周期）+ 文件系统深度访问 + .dmg/.msi/.deb 原生安装；**PWA 通道**保留给轻量级/无 AI 使用场景（纯手工转写 + WebMAUS 远程回退）。两者共享同一 React 代码库。**DP1 与 DP4 合并决策**：选型 D（三层渐进）天然包含 Tauri sidecar，故两个决策点合并为一个，在 W12 前统一确定。先 macOS 单平台 PoC 验证 Python sidecar 打包流程，再跨平台扩展

### 决策点5：LLM 对话 Provider 选型（W27前决定）🆕
- **选项A**：OpenAI API（GPT-4o/GPT-4o-mini）——生态最成熟、工具调用/JSON mode 最稳定、缺点是闭源+数据出境+成本较高（$2.5-15/1M tokens）
- **选项B**：Anthropic API（Claude 3.5/4）——长上下文窗口（200K tokens）最适合大型语料分析、系统提示遵循性强、缺点同为闭源+数据出境
- **选项C**：Ollama 本地部署——完全离线+数据不出本机+免费、缺点是需 GPU 支持、模型能力弱于商业 API（7B-70B 参数级别）、MacOS/Linux 原生支持（Windows 需 WSL2）
- **选项D**：Transformers.js 浏览器端轻量模型（Phi-3-mini ONNX ~2.3GB）——纯前端零依赖、但对话能力极有限、仅适合简单分类/续写任务
- **建议**：**多 Provider 并行，用户按需选择**。默认 OpenAI（开箱即用体验最佳）；隐私敏感用户切换 Ollama；F9 Provider 抽象层已支持热切换，无需提前锁定单一 Provider。API key 由用户自行提供（解语不代理/不托管），加密存储到 IndexedDB。**田野环境特殊考量**：田野调查常处于弱网/无网环境——Ollama 本地模型是刚需而非备选；Prompt 实验室的自定义指令应缓存到 IndexedDB，支持离线重放与结果暂存

### 各功能详细对标

#### 第一阶段：数据底座

**F1 CAM-Lite** — 🔧 适配移植 (~30%)
- **ELAN Annotation Format 3.0**：时间对齐多层标注模型，解语已参考其 tier/annotation 结构
- **FLEx Linguistic Computing Model**：sentence → word → morpheme → gloss 四级层次，影响 morpheme 数据设计
- **LIFT XML 1.3**：词库数据模型（lexical-unit/sense/variant/morph-type），解语 lexemes 表对标
- **Bodt (2022) 多书写系统数据模型**：单一田野语言可使用 N 种书写系统（如 Tshangla 5种：IPA/Latin/ʼUcen/Devanāgarī/Pīnyīn），FLEx 中以 BCP 47 语言+脚本子标签区分（`tsj-fonipa`、`tsj-Latn`、`tsj-Deva` 等），且可有 N 种翻译/分析语言。解语 orthographies 表 + lexemes 多脚本词形 + translation_layers 多语言翻译需对标此模型
- 四种标准定义了字段语义和层级关系，直接指导 Dexie schema 设计；具体实现（Zod 校验、跨格式映射）需自写

**F2 EAF + F3 TextGrid** — ♻️ 已完成
- 已有完整双向转换。npm 社区有 `eaf-parser`（仅解析、不支持生成）和 Python `pympi`，均不如自有实现完整
- **Bodt (2022) .flextext 往返经验**：ELAN↔FLEx 互操作需关注 (i) 6 种 linguistic type（txt/phrase/phrase-item/word-item/morph-item/morph）和对应 tier 命名模式；(ii) phrase 级 baseline 替换为 segnum 的问题——解语应在 CAM-Lite 层保持 baseline 稳定性；(iii) Custom Fields 优于 Notes（避免往返重复）

**F3b Toolbox** — 🔧 适配移植 (~20%)
- SIL Toolbox **MDF（Multi-Dictionary Formatter）标准标记集**有完整规范文档
- Python 生态有 `toolbox` 解析库，JS/TS 生态**无对等方案**
- 格式为行级文本（非 XML），解析器简单但标记映射配置需自写（~150行）

**F3c Transcriber** — 🔧 适配移植 (~40%)
- .trs 是简单 XML（Trans/Episode/Section/Turn/Sync），可复用 EafService 的 DOMParser 模式
- 结构远比 EAF 简单，预计代码量 100-150 行

**JYM/JYT 原生文件** — 🔧 适配移植 (~50%)
- 容器模式参考 **EPUB 3 (OCF)** 和 **ODF (LibreOffice)**：ZIP + mimetype 首条目不压缩 + META-INF/
- ZIP 库推荐 **fflate**（~29KB gzipped，纯 JS，性能最优，tree-shakable）优于 JSZip（~45KB，API 更友好但更重）
- 自写部分：容器规范设计、分表 JSON 序列化、内容寻址（SHA-256）、版本兼容性

#### 第二阶段：编辑能力

**F6 批量操作** — 🆕 从头开发 (~10%)
- ELAN 有跨 tier Find & Replace，FLEx 有 Bulk Edit，均为**桌面应用内嵌功能**，无可提取的 JS 代码
- 已有 searchReplaceUtils 提供文本搜索替换基础（~10% 可承继）
- 时间轴操作（批量拆分/合并/边界偏移）为领域特定逻辑，完全自写

**F7 撤销/恢复** — 🔧 适配移植 (~50%)

#### 第三阶段：AI 基础设施

**F8 任务调度** — 🔧 适配移植 (~40%)
- **Comlink** (Google, ~4KB)：Web Worker RPC，将 Worker 暴露为普通对象方法调用，消除样板代码

**F9 Provider 抽象层** — 🔧 适配移植 (~40%)
- **@huggingface/transformers** v4：浏览器端 ONNX 推理（ASR/NLP/Embedding），内部使用 ONNX Runtime Web。v4 新增支持 MMS、NLLB、ByT5、PyAnnote 等模型架构，含 q4/q8 量化 + WebGPU 加速
- 可参考 **Vercel AI SDK** 的 provider 接口设计模式
- 接口定义（submitTask/getResult）和 MockProvider/RemoteProvider 自写

**F10 MFA/WebMAUS 对齐** — 🔧 适配移植 (~60%)
- **Montreal Forced Aligner (MFA)**：Kaldi-based GMM-HMM 架构，业内 phonetic forced alignment 标杆（McAuliffe et al. 2017）。提供预训练 Global English 模型（~3700h 多方言训练数据），在低资源语言的跨语言对齐中持续表现最优（Chodroff, Ahn & Dolatian 2025）。支持从 ≥25 分钟语音自训练语言专属声学模型。**限制**：MFA 是 Python/Kaldi 栈，无法在浏览器纯前端运行——需经由本地伴侣进程或远程服务调用
- **BAS Web Services** (LMU Munich)：80+ 语言 forced alignment REST API，含多语言声学模型（Strunk et al. 2014），作为 MFA 不可用时的远程备选
- **Interlingual MFA Toolkit** (Dolatian 2023)：IPA 音素映射工具，将目标语言音素映射到英语音素集用于跨语言对齐
- **@ricky0123/vad-web** (~20K weekly)：Silero VAD WASM 封装，npm install 即用
- **rnnoise-wasm**：浏览器端降噪，npm install 即用
- **关键参数**（Chodroff et al. 2025 实证）：语言专属模型 ≥25 min 语音近天花板；输入 utterance <6s（理想 <2s）；递归对齐提升精度；同质跨语言模型（即使 100h）不如多方言 Global English 模型
- 主要风险：MFA Python 依赖（需 Decision Point 1 选型）、BAS CORS 策略

**WhisperX（服务端词级对齐，新增第三通道）**：
- Whisper + VAD Cut&Merge + wav2vec2 音素对齐 → 精确词级时间戳，12x 实时速度（batched inference）
- 支持说话人分离（pyannote），适用长录音快速预对齐 → 人工修正
- 与 MFA 互补：MFA 精度更高但速度慢，WhisperX 速度快且含说话人标签
- F10 对齐策略更新为三通道：MFA（精准）+ WhisperX（快速+分离）+ WebMAUS（远程备选）

**F28 词向量嵌入** — ♻️ 直接复用 (~70%)
- **@huggingface/transformers**：加载 multilingual-e5-small ONNX (~20MB) 到 Web Worker
- **UMAP-js** (~15KB)：嵌入降维可视化，配合 Recharts 散点图；可扩展用于语料语义聚类可视化

- **低资源覆盖扩展**：Glot500-m（服务端, 511语言, ~560M）可作为 e5-small 不覆盖时的编码器后端；ByT5（字节级, 无需分词器, 任意语言/脚本）特别适合无任何 NLP 资源的语言

#### Phase 2.2：F31 上下文引擎 + 安全加固（D4-D5）

**目标**：建立统一上下文注入管线，并补齐密钥安全底线。

**必要说明**：ShortTermContext/LongTermContext 是 F32 RAG 注入的统一入口，必须先落地；同时 API key 明文存储属于发布前高风险项，需要在本 Sprint 内完成迁移。

**实施项**：
- [ ] F31 上下文组装引擎（2 天）：
  - ShortTermContext：当前页面、选中 utterance、音频位置、最近编辑序列
  - LongTermContext：项目统计快照、词频 top-N、阶段建议摘要
  - token budget 管理：超限自动截断（优先裁剪 LongTerm，再裁历史消息）
- [ ] P1#3 API key 安全（1 天）：Web Crypto 加密存储 + 旧配置迁移（首次读取旧值即迁移）

**阶段验收**：
- 同一问题在不同页面可体现不同上下文
- 长上下文请求不出现超限崩溃
- 本地不再保存明文 key，迁移后旧配置可继续使用
