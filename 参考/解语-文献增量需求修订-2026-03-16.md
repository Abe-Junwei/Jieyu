  - FLEx 官方站点
  - BAS WebServices / WebMAUS API 帮助页

## 2. 对既有思路的增量结论（Delta）
1. 互操作优先级上调：
   - 从“支持导入导出”提升为“以标准模型为核心驱动数据层”。
   - 先保证 EAF/TextGrid 的结构保真，再做高级 AI 自动化。
2. 协作能力前置：
   - Yjs/CRDT 从“可选增强”提升为“中期主线能力”。
   - 目标是多人并发编辑的稳定性先于模型精度提升。
3. AI 引擎策略细化：
   - Elpis/ESPnet 证据支持“低资源场景可用优先”路线。
   - 采用“可插拔后端 + 置信度门控 + 人在回路”而非单模型押注。
4. 服务合规约束明确：
   - BAS/WebMAUS 存在学术授权、配额、第三方转发和数据留存窗口约束。
   - 因此默认策略应是“本地优先，云服务可替代且可禁用”。

## 3. 新增需求分层

### A. 可直接落地（短周期，建议立即进入迭代）
1. 标准化数据中间层（核心）
- 需求：建立内部 Canonical Annotation Model（CAM），字段覆盖 tier、时间锚点、父子关系、受控词表。
- 目标：EAF、TextGrid、内部存储间的无损映射。
- 验收：同一项目经 EAF -> CAM -> EAF 往返后，结构差异仅允许顺序级非语义差异。

2. 互操作回归测试集
- 需求：基于真实语料建立 golden files（EAF/TextGrid/导出结果）。
- 目标：每次改动自动验证 tiers、时间边界、annotation 数量、链接关系。
- 验收：CI 中互操作测试通过率 100%，关键字段零回归。

3. Leipzig 规则辅助校验器（非阻断）
- 需求：提供 interlinear gloss 规则提示（缩写合法性、分词对齐、分隔符建议）。
- 目标：先提示、后修复建议，不阻塞编辑。
- 验收：可对典型 gloss 行给出明确 warning/code，并支持一键忽略。

4. AI 作业元数据统一
- 需求：统一记录每次自动转写/对齐的模型、参数、时间戳、输入版本、输出置信度。
- 目标：为审计、复现、回滚提供依据。
- 验收：任意自动结果可追溯到完整任务参数和源数据快照。

5. 外部服务适配器层（BAS/WebMAUS）
- 需求：实现 provider 抽象，封装格式转换、轮询、失败重试、回填解析。
- 目标：避免业务层直接依赖单一第三方 API。
- 验收：provider 可切换（mock/local/remote），失败时不破坏主数据。

### B. 需验证后落地（中周期，先 PoC）
1. Yjs 协作内核接入
- 假设：当前数据模型可映射为可合并的 CRDT 结构。
- PoC：选择 1-2 个核心对象（utterance、token）做并发编辑冲突实验。
- 验收：离线编辑合并后无结构损坏，冲突可解释。

2. Elpis/ESPnet 可插拔 ASR 后端
- 假设：低资源模型在目标语种上可达到可用门槛。
- PoC：固定语料评估 WER/CER 与人工修订时间下降比例。
- 验收：至少一种后端在目标语料上显著优于 baseline，且部署复杂度可控。

3. 主动学习闭环
- 假设：利用修订记录做增量训练可持续提升。
- PoC：实现“高不确定片段优先复核”流程并统计收益。
- 验收：单位人工时间下的有效修订增益可量化提升。

4. FLEx/词典联动桥接
- 假设：可在不破坏现有层模型前提下映射词条与形态信息。
- PoC：最小化词条同步（lemma + gloss + morph tags）。
- 验收：跨工具交换后核心字段完整可还原。

### C. 暂缓（长期前瞻）
1. 端侧大模型深度集成（如复杂重写、端上多轮语义代理）
- 原因：资源消耗、可解释性与可靠性风险高。

2. 全自动无监督工作流替代人工校订
- 原因：低资源场景下错误成本高，不符合当前质量目标。

## 4. 路线图修订建议（对齐现有 M1+ 分期）

### 阶段 1（M1-M2）
- 保持：离线优先、媒体与标注核心体验。
- 新增：
  - CAM 数据中间层
  - 互操作 golden tests
  - AI 作业元数据统一

### 阶段 2（M3-M4）
- 强化：
  - DLx/Leipzig 对齐能力（规则提示 + 结构映射）
  - FLEx 最小桥接 PoC

### 阶段 3（M5-M6）
- 调整：
  - AI 从“功能堆叠”改为“后端可插拔 + 置信度门控 + 主动学习闭环 PoC”
  - 引入 BAS/WebMAUS provider（可替换）

### 阶段 4（M7+）
- 新增主线：
  - Yjs 协作内核（从小对象开始，逐步扩展）
  - 审计与复现体系完善（模型、参数、版本、操作轨迹）

## 5. 验收指标（建议纳入里程碑定义）
1. 互操作质量
- EAF/TextGrid 往返结构保真 >= 99%（字段级对比）。

2. 协作稳定性
- 离线并发合并后冲突导致的数据损坏率为 0。

3. AI 实用性
- 在目标语料上，人工修订时间较纯手工下降 >= 30%。

4. 可追溯性
- 自动生成结果 100% 具备可追溯元数据与可回滚记录。

## 6. 风险与控制
1. 外部服务依赖风险
- 控制：provider 抽象 + 本地 fallback + 配额/授权检查。

2. 标准映射复杂度高
- 控制：先覆盖高频字段，长尾字段走扩展命名空间。

3. 协作引入后调试复杂
- 控制：先对象级 PoC，逐步扩大 CRDT 覆盖范围。

## 7. 下一步执行清单（建议）
1. 先实现 CAM 与 EAF/TextGrid 双向映射测试骨架。
2. 建立互操作 golden files 和回归脚本。
3. 设计 provider 接口（local/mock/remote）并接入 BAS/WebMAUS 第一个最小流程。
4. 为 AI 任务落地统一元数据 schema 与审计视图。
5. 启动 Yjs 小范围并发编辑 PoC（仅 utterance/token）。
