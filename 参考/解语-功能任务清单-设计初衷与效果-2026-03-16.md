## 2026-03-29 对齐说明

本清单保留“功能全集”视角，但当前仓库的真实执行面已经发生变化：

1. **转写页是唯一开放工作台**；标注 / 分析 / 写作 / 词典页当前均为规划说明页，而非可交付页面。
2. F1-F9、F11、F28、F29、F31、语音助手等大量能力已经在转写页内部落地，后续主线不再是“从零实现这些功能”，而是“把它们从转写页大编排层中收口成稳定内核”。
3. 当前最优先的不是继续平铺更多功能编号，而是围绕转写页补齐以下能力包：
	- 编排层拆分与页面边界收口
	- 时间轴 / 波形 / layer unit 语义统一
	- 批量编辑与恢复链路闭环
	- 对齐 F10 最小产品链路
	- AI / 语音 / RAG 的转写页产品化
4. 因此，阅读本清单时请区分两层含义：
	- 下文的编号与设计初衷仍有效
	- 但当前排期优先级需以 `docs/规划-转写页剩余开发路线-2026-03-29.md` 为准

## 功能1：标准化数据中间层（CAM-Lite）

### 设计初衷
不同格式（EAF、TextGrid、内部数据）之间字段与层级不一致，直接互转会持续返工。先统一到一个最小中间模型，后续所有功能才能稳定迭代。

### 任务清单
1. 定义CAM-Lite最小字段：annotation、tier、time anchors、parent link、metadata。
2. 定义字段映射矩阵：EAF <-> CAM-Lite，TextGrid <-> CAM-Lite。
3. 定义扩展字段策略：未覆盖字段进入extensions，避免丢数。
4. 定义版本字段：为后续迁移与回滚保留schemaVersion。
5. 📚 增加 glossary/lexicon 实体：每个 annotation token 可指向 glossary entry（form + gloss + sense ID），而非仅存储字符串。（Bird 2020 §4.1, FLEx 模型）
6. 📚 增加 annotation 置信度与状态字段：`confidence: number`, `status: draft | confirmed`。支持稀疏转写中的不确定性管理。（Bird 2020 §2.4, Bird 2009）
7. 📚 增加来源追踪字段（provenance）：`createdBy`, `method: manual | auto | respeaking`, `timestamp`。人工与AI编辑统一追踪。（Bird 2009, Adams 2021）
8. 📚 支持多假设标注：同一时间区段可存储多个候选 annotation，按置信度排序，最终由人确认。（Bird 2020 §2.1.2）

### 预计实现效果
1. 导入后数据结构统一，业务逻辑简化。
2. 后续新增格式时，只需新增映射层，不改编辑核心。
3. 📚 标注不再是无语义的字符串，而是可链接到词条的结构化单元，为后续搜索、统计、词典联动打基础。
4. 📚 转写过程中的不确定性与来源可追溯，科研复现更可靠。

### 验收口径
1. 核心实体可无歧义映射。
2. 样本导入后不出现字段"无处安放"。
3. 📚 至少1个样本验证 token → glossary entry 链接的完整往返。
4. 📚 draft/confirmed 状态可切换，UI可筛选。

## 功能2：EAF导入与导出稳定链路

### 设计初衷
EAF是语言标注核心格式之一，必须保证导入可用、导出可分享，且结构可信。

### 任务清单
1. 实现EAF -> CAM-Lite解析器。
2. 实现CAM-Lite -> EAF导出器。
3. 处理tier引用关系与时间锚点边界。
4. 处理空值与异常结构容错。
5. 增加EAF往返差异报告（字段级、结构级）。

### 预计实现效果
1. 你可直接用真实EAF语料开始工作。
2. 编辑后导出的EAF可回到学术工具链使用。

### 验收口径
1. 关键字段保真率 >= 99%。
2. 无阻断性导出错误。

## 功能4：互操作回归测试体系（Golden Files）

### 设计初衷
单人开发最怕“改一处坏一片”。回归体系是防返工核心。

### 任务清单
1. 建立golden样本库（先10个，扩到20个）。
2. 建立自动对比脚本：字段完整性、层级关系、时间序列。
3. 每次改动自动生成差异报告。
4. 建立缺陷标签：解析丢失、层级断裂、时间冲突。

### 预计实现效果
1. 你每次改功能都知道是否破坏旧能力。
2. 修复效率提高，定位问题更快。

### 验收口径
1. 任一核心变更可自动触发回归。
2. 差异报告可直接定位到字段或层。

## 功能8：AI任务调度与状态管理

### 设计初衷
AI能力接入后，若没有统一任务模型，系统会出现不可控状态与“黑箱结果”。

### 任务清单
1. 定义任务状态机：pending/running/success/failed/cancelled。
2. 定义任务队列与并发策略（先串行，后可扩展）。
3. 定义超时与取消机制。
4. 定义失败重试策略（指数退避或固定重试）。
5. 📚 实现训练/推理进度实时反馈（日志输出、当前epoch/step、预估剩余时间），不只是旋转图标。（Adams 2021 §3.2 training feedback）
6. 📚 支持批量任务（如对整个语料库执行word spotting/对齐），显示整体进度与单项进度。

### 预计实现效果
1. AI调用可见、可控、可恢复。
2. 用户不会因长任务失去反馈。
3. 📚 长时间训练/推理过程中用户可判断是否正常运行、是否需要中止。

### 验收口径
1. 任意任务都可追踪状态。
2. 失败后可重试并保持数据安全。
3. 📚 训练任务有实时日志输出到UI。

## 功能9：Provider抽象层（local/mock/remote）

### 设计初衷
单一外部服务风险高，必须可替换，避免业务层被绑定。

### 任务清单
1. 定义provider统一接口（提交任务、查询状态、取结果）。
2. 实现mock provider（本地演示与回归）。
3. 实现local provider（预留本地引擎链路）。
4. 实现remote provider基础壳（后续接BAS/WebMAUS）。
5. 📚 支持预训练模型的导入/导出/选择：用户可加载不同语言的预训练模型，做 transfer learning 微调。（Adams 2021 §6.3）
6. 📚 支持根据数据特征（说话人数、数据量）自动推荐provider配置或超参数。（Adams 2021 §3.1）

### 预计实现效果
1. 切换后端不需要重写业务代码。
2. 离线开发与线上接入都可推进。
3. 📚 用户可选择或导入社区共享的预训练模型，无需从零训练。

### 验收口径
1. 三类provider可在同一业务流程切换。
2. 业务层无分叉逻辑污染。
3. 📚 模型导入后可用于推理，参数配置无需手动tuning。

## 功能10：WebMAUS/BAS最小接入

### 设计初衷
先用一个可用的外部链路快速验证AI提效，再决定是否扩展更多引擎。

### 任务清单
1. 选定1条最小可行管线（如G2P+MAUS或runPipeline）。
2. 实现输入准备（音频/文本/格式参数）。
3. 实现结果拉取与解析回填。
4. 实现授权、配额、不可用场景提示。
5. 实现失败兜底（回落到mock或手工流程）。
6. 📚 集成 VAD（语音活动检测）：自动识别录音中的语音/静默段，作为呼吸组分段的基础。（Elpis future §4.1, Bird 2020 breath groups）
7. 📚 集成说话人识别/分离（Speaker Diarization）：多说话人录音自动标注说话人。（Elpis future §4.1）
8. 📚 支持音频降噪预处理管线：田野录音噪声大，降噪后可提升后续ASR准确率。（Elpis future §4.2.3）
9. 💡 支持符号层→时间对齐层的 forced alignment 转换：将已有文本转写自动对齐到音频时间轴。（Summit Theme 2: Segmentation）

### 预计实现效果
1. AI初稿转写可用于真实工作流。
2. 你能量化外部服务在目标语料上的价值。
3. 📚 长录音可自动切分为可标注单元（呼吸组/说话人段），大幅减少手工切割时间。

### 验收口径
1. 端到端成功率 >= 90%。
2. 回填结果不破坏原有结构。
3. 📚 VAD切分结果可在时间轴界面中显示并手动修正。

## 功能12：Leipzig规则提示器与自动注释（非阻断）📚

### 设计初衷
语言学标注规范是质量底线，但强阻断会拖慢效率，先做提示型规则。进一步地，当词汇表积累到一定程度后，已见过的词应能自动填充gloss，减少重复劳动。

### 任务清单
1. 实现缩写合法性提示。
2. 实现词素分隔符一致性提示。
3. 实现行间对齐提示。
4. 支持忽略规则与例外标记。
5. 📚 实现自动interlinear glossing：当词已出现在glossary中时，自动填充gloss；多义词高亮让用户消歧。（Elpis future §4.2.5, FLEx词条链接模型）
6. 📚 频率词表与异常检测：自动生成频率排序词表，标记低频词为潜在拼写错误或新词候选。（Elpis future §3.1）
7. 📚 字符频率审计：标记低频字符为潜在录入错误，提供定位与批量修正。（Elpis future §3.1）

### 预计实现效果
1. 早期发现格式问题，减少后期清洗。
2. 不影响快速编辑流程。
3. 📚 已知词的gloss自动填充节省大量重复键入，类似FLEx体验。
4. 📚 语料质量问题（拼写不一致、罕见字符）可被系统性发现。

### 验收口径
1. 典型样本提示准确。
2. 用户可无痛忽略特例。
3. 📚 auto-gloss在已知词上准确率 >= 95%。
4. 📚 频率词表可一键生成并下载。

## 功能16：稀疏转写与覆盖率追踪 📚

### 设计初衷
Bird (2020) 的核心创新——转写不必连续。实际田野工作中，大量录音只能部分转写，工具必须支持"有多少标多少"的工作模式，而非强迫补全。同时需追踪每段录音的转写完成度，支持优先处理策略。

### 文献来源
Bird (2020) §4.1-4.2 Sparse Transcription Model; §Task C (Coverage); §Task V (View)

### 任务清单
1. 支持非连续标注（breath group内可有未标注区域，不报错不强制补全）。
2. 允许 token 只指定时间点而非时间段（便于松散定位）。
3. 自动计算每段录音/每个breath group的转写覆盖率（按时间或token计）。
4. 覆盖率视图：显示整个项目/语料库的标注深度概览（heatmap或进度条）。
5. 按覆盖率、时长、话题筛选"下一个该转写什么"。

### 预计实现效果
1. 大量历史录音可"先粗标再精化"，不必全部精标后才能使用。
2. 项目进度一目了然，不再有"一堆未标注音频不知从哪开始"的困境。
3. 与稀疏转写配合，系统引导用户将精力集中在高价值段落。

### 验收口径
1. 含未标注区域的录音可正常保存、导出、不丢已标注内容。
2. 覆盖率数值与手工统计一致。

## 功能17：词项侦测（Word Spotting / Spoken Term Detection）📚

### 设计初衷
Bird (2020) 提出的替代"全文转写"的核心能力：在未转写音频中自动定位已知词项。这解决了"已知100个词，但语料有100小时怎么办"的问题——不必全听一遍，让系统帮你找。

### 文献来源
Bird (2020) §3.4 Spoken Term Detection; §4.2 Task S (Word Spotting)

### 任务清单
1. 定义Word Spotting任务接口：输入=glossary entry的参考音频+目标音频集，输出=候选 hit 列表（位置、置信度）。
2. 通过provider抽象接入spotting引擎（初期可接WebMAUS forced alignment变体或音频指纹匹配）。
3. 结果回填为稀疏annotation（token挂到breath group + glossary entry）。
4. 用户审核界面：逐条确认/拒绝/修正 hit。
5. 确认的hit自动扩充训练素材，形成"spot → confirm → improve"正循环。

### 预计实现效果
1. 大量未转写录音中可快速定位特定词汇或表达的出现位置。
2. 词汇表越大，可自动标注的比例越高，转写工作逐步轻量化。

### 验收口径
1. 在已知词集上的召回率 >= 70%，精确率 >= 80%。
2. 确认后的结果可在concordance视图中浏览。

## 功能18：翻译优先工作流（Translate-First）📚

### 设计初衷
Bird (2020) §2.2.2 与 Samarin (1967) 的"锥形语料库"模型均强调：翻译比转写更紧迫、更容易、语料中覆盖更广。工具应允许"先翻译再转写"，而非强制"先转写再翻译"的线性流水线。Elpis future也将MT作为重要自动化方向。

### 文献来源
Bird (2020) §2.2.2, Figure 3 (Tapered Corpus); Elpis future §4.2.6

### 任务清单
1. 在tier类型中增加"translation"类型（区别于transcription tier）。
2. 支持为breath group / utterance直接添加翻译，不要求先有转写。
3. 支持语料分层标签：raw → translated → sparse → dense → glossed（锥形语料库模型）。
4. 翻译内容可作为word spotting和自动对齐的辅助输入。
5. （可选）接入简易MT provider，基于已有平行语料自动生成候选翻译。

### 预计实现效果
1. 田野工作者可先录音、立即请说话人口头翻译，时间轴对齐后即可开始使用语料。
2. 语料库不再是"要么全精标，要么全空"，而是按深度分层的渐进式资产。

### 验收口径
1. 只有翻译没有转写的录音可正常保存、展示、搜索。
2. 语料深度标签可筛选。

## 功能19：搜索、索引与Concordance视图 📚

### 设计初衷
Bird (2020) §4.2 Task V 强调：转写的价值在于让语料可搜索。当前解语没有跨录音的搜索能力。Elpis future §3.1也指出频率词表和跨文本搜索是基础能力。

### 文献来源
Bird (2020) §4.2 Task V (View); Elpis future §3.1 Text Analytics

### 任务清单
1. 实现跨项目/跨录音的全文搜索（按annotation文本、gloss、翻译）。
2. 搜索结果以concordance（上下文对齐）格式展示，每条结果可点击跳转到音频位置。
3. 支持按 glossary entry 搜索（不只是字符串匹配，解决同形异义问题）。
4. 搜索结果支持导出（CSV/JSON）用于进一步分析。
5. 基础语料统计面板：词频表、总词数、唯一词数、每说话人统计。

### 预计实现效果
1. 研究者可快速找到某个词/表达在所有录音中的全部出现。
2. 语料库从"一堆文件"变为"可查询的数据库"。

### 验收口径
1. 搜索结果与手工grep一致。
2. 点击结果可跳转到对应音频位置。

## 功能21：用户脚本与扩展API 📚

### 设计初衷
Gessler (2019) §2.3.2 指出：语言学工作的多样性意味着没有一种工具能满足所有需求。解决方案是暴露脚本接口，让高级用户自己编写批处理逻辑和自定义分析。

### 文献来源
Gessler (2019) §2.3.2 User scripts; §2.3.3 Extensions

### 任务清单
1. 暴露核心数据操作API供浏览器控制台调用（读取/修改annotation、tier、glossary）。
2. 支持用户编写简单JS脚本执行批量操作（如"将所有包含X的annotation替换为Y"）。
3. 提供脚本示例库（常见清洗、转换、统计任务）。
4. （可选）支持脚本导入/导出/分享。
5. 💡 提供 Import/Export 脚本 API：为非标准格式提供简易 JS/Python 编程接口，用户可编写自定义格式转换器，无需修改核心代码。（开发思路: Import/export API）

### 预计实现效果
1. 高级用户可处理工具未内建的特殊需求，无需等开发。
2. 社区可共享自定义脚本，形成插件生态雏形。

### 验收口径
1. 至少3个示例脚本可在控制台中正常运行并改变数据。
2. 脚本执行失败不损坏主数据（沙盒或撤销保护）。

## 功能22：社区/说话人参与界面（简化版）📚

### 设计初衷
Bird (2009, 2020) 反复强调：说话人不是"数据提供者"而是核心协作者。他们能辨认词汇、提供翻译、做respeaking，但不应被要求使用复杂的语言学工具。需要一个简化界面支持这些参与模式。

### 文献来源
Bird (2009); Bird (2020) §2.2.3, §4.3 Collaborative Workflows; Elpis (Foley 2018)

### 任务清单
1. 设计"说话人模式"最小界面：播放音频→确认/拒绝候选词→提供口头翻译。
2. 支持respeaking工作流：说话人重述录音，系统将重述与原始音频关联。
3. 支持hit review界面：左右滑动确认/拒绝word spotting结果（Bird 2020建议）。
4. 简化权限模型：说话人只能操作指定录音，不能修改项目结构。
5. 💡 支持众包/游戏化任务模式：Duolingo风格的"听音写词""听音翻译"任务，降低参与门槛。（Summit Theme 3: Crowdsourcing）
6. 💡 支持学校/社区教学场景：将转写/翻译任务嵌入课程，学生完成作为作业。（Summit Theme 3）

### 预计实现效果
1. 田野调查中说话人可直接参与标注工作，而非只是"被录的人"。
2. Respeaking产生的清晰音频可提升后续ASR准确率。

### 验收口径
1. 非技术背景用户可在5分钟说明后开始操作。
2. 说话人操作不会意外破坏项目数据。

## 功能23：离线优先与数据同步 📚

### 设计初衷
Gessler (2019) 用PouchDB/CouchDB验证了"完全离线可用+有网即同步"的架构可行性。田野语言学工作常在无网络环境下进行。Adams (2021) 也指出不是所有用户都有GPU或稳定网络。

### 文献来源
Gessler (2019) §2.1.2 PouchDB; Adams (2021) §6.4

### 任务清单
1. 确认当前 RxDB/Dexie 方案的离线完整性（已部分实现，需验证覆盖度）。
2. 确保所有核心功能（编辑、导入、导出、搜索）在断网状态下可用。
3. 实现联网时的增量同步机制（优先数据安全，冲突可解释）。
4. 大文件（音频/视频）同步策略：元数据先同步，媒体按需拉取。

### 预计实现效果
1. 田野工作者在无网环境中可完整使用解语所有核心功能。
2. 回到网络环境后数据自动同步，无需手动导出导入。

### 验收口径
1. 断网48小时后恢复网络，数据无丢失、无冲突损坏。
2. 冲突情况有明确提示和手动解决方案。

## 功能24：数据导出多格式与字典联动 📚

### 设计初衷
Elpis future §5 强调：结构化标注数据不应锁在单一工具中，应可导出为字典、出版物、教学材料等多种格式。Bird (2020) 的glossary模型天然可导出为词典。

### 文献来源
Elpis future §5; Bird (2020) Glossary model; FLEx interop

### 任务清单
1. Glossary可导出为标准词典格式（JSON-LD / TEI-Lex / CSV）。
2. 语料可导出为FLEx可导入格式（最小化：lexicon + glossed text）。
3. 支持导出为训练数据格式（Kaldi/ESPnet wav+text manifest），便于送入ASR训练。
4. 💡 Glossary自动生成社区词典：标注库本身即词典数据源，可一键发布为只读词典应用/网页，实现"标注即词典"零额外成本。（开发思路: Free dictionary）
5. 💡 FLEx 深度互操作：处理 flextext XML 的 word/morpheme 层级导入（非仅 sentence 层），支持 Send/Receive 同步协议兼容。（Summit Theme 2: FLEx interop）

### 预计实现效果
1. 解语中积累的词汇表和标注可直接流入学术工具链或社区工具。
2. 训练数据导出使解语成为ASR训练数据管理工具的一环。
3. 💡 社区可零成本获得"免费词典"，推动标注工作的社会价值回馈。
4. 💡 FLEx用户可无损来回切换，不被单一工具锁定。

### 验收口径
1. FLEx导出/导入往返核心字段无丢失（含word/morpheme层）。
2. ASR训练格式导出后可直接被ESPnet/Kaldi使用。
3. 💡 词典发布后可在浏览器中搜索、浏览。

## 功能25：协作任务管理与角色分配 💡

### 设计初衷
开发思路文档明确指出：协作功能日益成为刚需。如果真正要让社区参与语言文档，工具就不能只为独立研究者设计。需要支持任务分配（音频选择、转写指派、内容批准）与角色分工（管理者/标注者/审核者/说话人）。Summit Theme 2 也反复强调协作工作流和版本控制。

### 来源
开发思路: 协作; Summit Theme 2: Workflows; Summit Theme 3: Crowdsourcing

### 任务清单
1. 定义角色模型：admin / annotator / reviewer / speaker，各角色可操作范围不同。
2. 管理者可创建转写/标注/翻译任务，分配到具体录音+层+人。
3. 任务状态流：assigned → in-progress → submitted → approved / rejected。
4. 审批工作流：审核者可逐条查看标注并批准/打回+批注。
5. 支持多人在不同层上并行工作（如一人转写、一人翻译），避免同层冲突。
6. 任务看板视图：显示所有录音的任务分配与完成状态。

### 预计实现效果
1. 团队可协同标注同一批语料，不再是"一个人干所有事"。
2. 标注质量有审批把关，不再是"标了就算"。
3. 项目管理者可清晰掌握谁在做什么、进度如何。

### 验收口径
1. 不同角色登录后看到不同可用操作。
2. 审批工作流可走通完整流程（分配→提交→批准）。
3. 两人同时操作不同层不会数据冲突。

## 功能26：移动端支持与田野采集 💡

### 设计初衷
开发思路文档强调：许多濒危语言社区电力有限、无法使用笔记本，移动设备是主要甚至唯一的数字工具。需要在手机/平板上完成录音、简单标注、数据回传。Summit也提到Aikuma等移动采集工具的成功经验。文档中还提出了一个具体的"诱发词表采集"工作流：录音+计时器同步，按"计次"按钮标记每个词条的起止时间。

### 来源
开发思路: 移动端支持; Summit Theme 4: Mobile language apps; Aikuma

### 任务清单
1. 确保核心界面在移动端浏览器可用（响应式布局、触控友好）。
2. 移动端录音：支持在浏览器中录音并直接关联到项目。
3. 💡 诱发词表采集模式（Elicitation Timer）：录音+计时器同步启动，"计次"按钮标记每个词条的起止时间（类似秒表的lap功能），可在录制中/后修正时间点。
4. 移动端简易标注：支持在手机上对已有录音添加翻译或简单标记。
5. 移动端数据上传：录制内容在有网时自动同步到项目（结合功能23离线同步）。
6. （可选）支持蓝牙键盘输入，提升移动端转写效率。

### 预计实现效果
1. 田野调查中可直接用手机录音并标注，不依赖笔记本。
2. 诱发词表采集大幅减少后期手动切分音频的工作量。
3. 社区成员可用自己的手机参与数据采集。

### 验收口径
1. 在主流移动浏览器（Safari iOS / Chrome Android）上核心功能可用。
2. Elicitation Timer 标记的时间点准确度 <= 100ms。
3. 移动端录音可成功同步到桌面端项目。

## 功能27：元数据管理与渐进归档 💡

### 设计初衷
Summit Theme 1 & 2 反复强调：元数据管理是工作流主要瓶颈之一——格式要求复杂、录入耗时、且常被推迟到最后。归档不应是"最终产品"才做的事，而应是工作流中反复进行的中间步骤：先归档基础录音+元数据，后续逐步补充转写/标注。

### 来源
Summit Theme 1: Archiving; Summit Theme 2: Metadata; 开发思路: 渐进归档

### 任务清单
1. 定义最小元数据模型：session（日期、地点、参与者、语言、体裁）、recording（录音设备、格式、时长）、speaker（姓名/代号、年龄、性别、语言能力）。
2. 录音创建时引导填写核心元数据（必填字段最小化，可后补）。
3. 支持元数据批量编辑（如将同一session的20段录音统一设置地点/参与者）。
4. 支持元数据导出为标准格式（CMDI/IMDI/CSV），便于提交到语言档案馆。
5. 渐进归档标记：每个录音可标记"archive-ready"状态，支持筛选"已就绪/待补充"。
6. （可选）文件命名一致性检查与建议（Summit提到的常见瓶颈）。

### 预计实现效果
1. 元数据不再是"最后才补"的苦差事，而是随工作流自然积累。
2. 提交语言档案馆所需的格式化元数据可一键导出，不再手工拼凑。
3. 渐进归档确保即使项目中断，已有数据也处于可归档状态。

### 验收口径
1. 新建录音时有元数据引导，3个必填字段30秒内可完成。
2. CMDI/IMDI导出文件可通过档案馆格式验证器。
3. 渐进归档状态可筛选、可统计。

## 4. 推荐执行顺序（单人最稳路径·修订版v3）

### 第一阶段：数据底座（功能1-4）
功能1(CAM-Lite+glossary+置信度) → 功能2(EAF) → 功能3(TextGrid) → 功能4(Golden Files)

### 第二阶段：编辑能力（功能5-7）
功能5(层约束) → 功能6(批量操作) → 功能7(撤销/快照)

### 第三阶段：AI基础设施（功能8-11）
功能8(任务调度) → 功能9(Provider+预训练模型) → 功能10(WebMAUS+VAD) → 功能11(审计回滚)

### 第四阶段：语言学工作流增强（功能12, 16-19）📚
功能12(Leipzig+auto-gloss) → 功能16(稀疏转写+覆盖率) → 功能17(词项侦测) → 功能18(翻译优先) → 功能19(搜索/Concordance)

### 第五阶段：用户体验与生态（功能20-24）📚
功能20(可视化) → 功能21(脚本API+Import/Export API) → 功能22(社区界面+众包) → 功能23(离线同步) → 功能24(多格式导出+词典发布)

### 第六阶段：协作与移动（功能25-27）💡
功能25(协作任务管理) → 功能26(移动端+Elicitation Timer) → 功能27(元数据+渐进归档)

### 第七阶段：收尾（功能13-15）
功能13(诊断面板) → 功能14(性能优化) → 功能15(稳定化与文档)

**排序原则：**
1. 第一至三阶段不变——稳定底座、编辑能力、AI基础设施必须先行。
2. **第四阶段**是 Bird 2020 的核心贡献：稀疏转写、词项侦测、翻译优先、搜索索引——构成解语区别于"ELAN替代品"的核心能力。
3. **第五阶段**来自 Gessler 和 Elpis：可视化、脚本化、社区参与、离线同步、多格式导出——从"能用"到"好用"的关键。
4. **新增第六阶段**来自开发思路与Summit：协作是团队使用的前提，移动端是田野场景的刚需，元数据/归档是学术合规的保障。这些功能在核心标注与搜索就绪后再建，既有数据基础也有用户验证。
5. 收尾阶段后移至第七阶段，在所有功能基本就绪后做全局性能优化和文档更有意义。
