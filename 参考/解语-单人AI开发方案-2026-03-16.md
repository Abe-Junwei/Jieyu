# 解语单人 AI 开发方案（面向需求与功能落地）

> 2026-03-29 对齐说明：本方案原始版本适用于“从 0 到 1”的最小闭环阶段；当前仓库已越过该阶段，后续主线应从“证明可做”切换为“把转写页做成唯一可持续工作台”。以下新增章节优先生效，原始 12 周内容保留作历史参考。

## 0. 当前阶段重定义（2026-03-29）

### 0.1 当前项目现实

1. 互操作、转写编辑、AI chat、embedding/RAG、语音助手都已经在转写页内落地，不再属于“未来设想”。
2. 当前真正的瓶颈不是功能从无到有，而是转写页能力过度堆叠在单页编排层，导致维护成本、回归成本和页面边界都失真。
3. 标注 / 分析 / 写作 / 词典四个页面目前仍是规划说明页，因此单人阶段不应再按多工作台并进排期，而应优先把转写页完成到“可长期承载后续产品”的程度。

### 0.2 新目标（未来 6-8 周）

你当前的单人 + AI 开发目标应调整为：

1. 先把转写页收口成稳定产品内核：导入 -> 编辑 -> AI/语音辅助 -> 审计/恢复 -> 导出。
2. 把已经写在转写页里的 AI / 语音 / PDF / embedding 能力，从“大页内耦合实现”收敛成可抽离模块。
3. 只有在转写页编排和语义边界稳定后，才开放标注、分析、写作、词典等独立工作台。

### 0.3 新优先级

#### P0（当前必须）

1. 转写页编排层拆分：Orchestrator 继续瘦身，分离 runtime、controller、layout state。
2. 时间轴/波形/选择语义统一：dependent layer、segment-backed layer、waveform overlay、context menu 与 split/merge 使用同一套 unit 语义。
3. 高频编辑闭环补齐：批量拆分/合并、批量时间偏移/缩放、speaker / note / undo 的一致性。

#### P1（紧接其后）

1. F10 对齐最小链路：先把转写页里的“对齐任务 -> 结果回填 -> 人工修正”做成真正可用闭环。
2. AI/语音产品化：Prompt 实验、任务审计队列、语音写回与自动前进、TTS/回放辅助。
3. 转写页性能与包体治理：继续按懒加载和运行时边界降低转写主包与语音/AI 绑定深度。

#### P2（转写页收口后）

1. 从转写页中提炼 Annotation / Analysis / Writing / Lexicon 的独立工作台能力。
2. 再考虑协作、移动端和扩展生态，而不是在当前阶段前置。

### 0.4 单人阶段的新执行法

1. 每一轮只做一个“转写页能力包”，不要同时推进多个新工作台。
2. 每个能力包必须同时包含：代码改动、回归测试、计划文档更新。
3. 若一个能力仍只能写在 `TranscriptionPage.Orchestrator.tsx` 里，说明它还没达到可扩展完成态。
4. 周节奏改为：周一收敛范围，周二到周四落代码和测试，周五更新规划/发布说明，并确认下一包是否继续留在转写页或已具备抽离条件。

## 1. 方案目标（按你的实际情况重定）
你是单人借助AI开发，所以方案核心不是全量并行，而是：
1. 先做最小闭环：导入 -> 编辑 -> 导出 -> 回归验证。
2. 每次只推进一个主功能，避免战线过长。
3. 所有功能都要有可见效果，不做“只在架构层有价值”的长期铺垫。

目标周期：原始 12 周方案保留作历史参考；当前执行建议改为 **6-8 周转写页收口阶段**。

## 2. 需求优先级（单人版）

### P0（必须，先做）
1. EAF/TextGrid互操作稳定。
2. 核心标注编辑可用（分层、时间对齐、基础校验）。
3. 导入导出回归测试自动化。

### P1（强烈建议）
1. AI辅助转写最小链路（可插拔provider，先接1个后端）。
2. 任务审计和可回滚（记录模型、参数、输入输出版本）。
3. Leipzig规则提示（非阻断）。

### P2（后续）
1. 协作能力（Yjs）PoC。
2. FLEx桥接与更深的词典联动。
3. 高阶自动化与主动学习闭环。

说明：单人阶段建议把Yjs从主线移到后续，先把“单机生产力”做稳。

## 3. 功能清单与实现步骤（按效果导向）

## 功能A：互操作闭环（P0）
目标效果：你可以放心导入真实语料、编辑后导出，不担心结构损坏。

实现步骤：
1. 定义内部最小数据模型CAM-Lite。
2. 实现EAF -> CAM-Lite、TextGrid -> CAM-Lite。
3. 实现CAM-Lite -> EAF、CAM-Lite -> TextGrid。
4. 建立20个golden样本往返测试。
5. 输出结构差异报告（字段缺失、时间偏移、层关系断裂）。

验收标准：
1. 往返后关键字段保真 >= 99%。
2. 无阻断性结构错误。

---

## 功能B：编辑器可用性增强（P0）
目标效果：你在日常标注时操作更快、错误更少。

实现步骤：
1. 增加层关系即时校验（创建、删除、链接提示）。
2. 增加时间轴编辑保护（非法重叠、负时长阻止）。
3. 增加批量操作（批量拆分、合并、标签替换）。
4. 增加撤销恢复稳定性回归。

验收标准：
1. 常见编辑场景不出现数据错乱。
2. 高频操作步骤减少（你主观可感知提速）。

---

## 功能C：AI辅助转写最小链路（P1）
目标效果：AI确实节省你的修订时间，而不是增加维护负担。

实现步骤：
1. 定义provider接口（只保留最小必要方法）。
2. 先实现mock provider，跑通UI与任务状态。
3. 再接入1个真实provider（建议先WebMAUS/BAS或本地引擎二选一）。
4. 实现失败重试、超时中断、错误可读提示。
5. 在任务完成后自动写入审计记录。

验收标准：
1. 端到端任务成功率 >= 90%。
2. 人工修订总时长下降 >= 20%（单人阶段先用20%门槛）。

---

## 功能D：审计与可回滚（P1）
目标效果：每次自动处理都可追溯、可复现、可撤销。

实现步骤：
1. 建立任务记录结构（任务ID、模型、参数、输入版本、输出版本、耗时）。
2. 自动写入任务日志与变更摘要。
3. 增加一键回滚到任务前版本。

验收标准：
1. 任意自动结果可追溯到完整上下文。
2. 回滚后数据一致。

---

## 功能E：Leipzig规则提示（P1）
目标效果：减少你后期整理与发表前清洗成本。

实现步骤：
1. 做3类基础提示：缩写合法性、词素分隔一致性、行间对齐提示。
2. 提供忽略和标记已知例外。
3. 导出时附带规则提示摘要（可选）。

验收标准：
1. 提示准确且不干扰编辑流程。
2. 典型样本人工检查通过率提升。

## 4. 12周执行节奏（单人可持续）

### 第1-2周：互操作底座v1
1. CAM-Lite字段冻结。
2. EAF/TextGrid双向最小转换。
3. 建立第一批golden样本。

效果：跑通最小导入导出闭环。

### 第3-4周：互操作质量v2
1. 差异报告工具。
2. 修复高频映射问题。
3. 接入基础回归脚本。

效果：导入导出开始稳定可用。

### 第5-6周：编辑器增强
1. 层关系和时间校验。
2. 批量操作能力。
3. 撤销恢复回归。

效果：日常标注效率明显提升。

### 第7-8周：AI链路接入
1. provider抽象 + mock跑通。
2. 接入1个真实provider。
3. 失败处理与状态展示。

效果：可稳定使用AI做初稿转写。

### 第9-10周：审计与回滚
1. 任务元数据。
2. 版本记录。
3. 一键回滚。

效果：自动处理可控、可查、可复现。

### 第11-12周：规则提示与收尾
1. Leipzig基础提示。
2. 端到端压测与修复。
3. 输出试用版与操作文档。

效果：形成可长期使用的单人生产工具链。

## 5. 单人开发工作法（关键）

1. 每周只设1个主目标。
2. 每个主目标最多3个子任务。
3. 每天结束前必须有可见成果（功能、测试、文档三选一）。
4. AI只负责“加速产出”，你保留最终架构与验收判断。

建议节奏：
1. 周一：定义本周验收口径。
2. 周二到周四：编码与测试。
3. 周五：回归、记录、下周任务拆解。

---

## 6. 你最关心的“效果指标”

1. 导入导出成功率。
2. 往返保真率。
3. 单小时可完成的标注时长。
4. AI任务成功率与平均耗时。
5. 人工修订耗时下降比例。

单人阶段建议达标线：
1. 往返保真 >= 99%。
2. AI修订耗时下降 >= 20%。
3. P1缺陷保持为0。

---

## 7. 现在就可以执行的第一周任务

1. 冻结CAM-Lite最小字段（只保留当前真实使用字段）。
2. 选10个真实样本做golden文件。
3. 先把EAF->CAM-Lite->EAF链路跑通并出差异报告。
4. 把差异报告写进固定模板，后续每周复用。

第一周完成效果：
- 你将拥有一个可以持续迭代的“互操作主干”，后续所有功能都可挂在这个主干上，而不是反复返工。

## 8. 后续扩展建议（不影响当前主线）

1. 当你开始出现多人协作需求时，再启动Yjs PoC。
2. 当你积累足够语料后，再做主动学习闭环。
3. 当你有明确词典工作流时，再做FLEx深度桥接。

结论：单人阶段最优策略是先把互操作与AI生产力做成“稳态闭环”，再做协作和生态扩展。
